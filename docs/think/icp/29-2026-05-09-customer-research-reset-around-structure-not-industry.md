# 고객 리서치 리셋: 업종이 아니라 운영 구조로 다시 보기

작성일: 2026-05-09

## 이 문서의 목적

이 문서는 현재 고객 리서치를 `업종 중심`에서 `운영 구조 중심`으로 다시 리셋하기 위해 만든다.

지금까지 문서 흐름에서 얻은 중요한 교훈은 아래다.

1. `프랜차이즈`, `약국`, `식당` 같은 업종 이름만으로는 고객 적합도가 잘 안 보인다
2. 실제로 중요한 것은 업종보다 `운영 구조 pain`이다
3. 따라서 다음 리서치는 `어느 업종인가`보다 `어떤 운영 구조를 가진 조직인가`를 먼저 봐야 한다

이 문서는 `고객 리서치 재시작 기준 문서`다.

## 먼저 결론

다음 리서치는 `업종을 더 많이 찾는 리서치`가 아니라,  
`Dowin이 들어갈 수 있는 운영 구조를 가진 조직을 다시 정의하는 리서치`여야 한다.

즉 이제부터의 핵심 질문은 아래다.

`어떤 업종이냐`가 아니라  
`어떤 조직 구조에서 리더 직접 추적 pain와 shared picture 부재가 반복되는가`

## 지금까지 리서치에서 확인한 것

지금까지 문서에서 비교적 분명해진 것은 아래다.

### 1. 너무 넓은 업종 타겟은 약하다

- `비IT 운영팀`
- `운영팀 전반`
- `개별 자영업`

이런 표현은 너무 넓어서 buyer도, problem도, 도입 장면도 흐려진다.

### 2. 문제는 업종보다 운영 구조에서 반복된다

아래 구조가 있으면 업종이 달라도 비슷한 pain가 나온다.

- 리더가 직접 chase해야 한다
- 상태가 한눈에 안 보인다
- 후속조치가 여러 채널에 흩어진다
- 점검은 하지만 다음 액션으로 깔끔하게 안 이어진다
- 회의나 체크 이후 다시 정리 피로가 생긴다

### 3. 우리가 보는 핵심 고객은 `팀 운영이 있는 소규모 조직`이다

다시 말하면 아래 조건이 필요하다.

- 3~15인 안팎
- 1인 작업이 아니라 팀 작업이 있다
- 리더와 실무자 레이어가 나뉜다
- 반복 운영 루프가 있다
- 새 도구를 붙일 최소 디지털 습관이 있다

## 이제부터 고객을 어떻게 정의해야 하는가

이제부터는 업종이 아니라 아래 구조로 정의하는 편이 맞다.

## 1. 조직 크기

좋은 후보:

- 3~15인 안팎
- 너무 작지 않고, 너무 크지 않음

이유:

- shared picture 필요성이 생길 만큼 복잡해야 한다
- 동시에 도입 의사결정은 빨라야 한다

## 2. 리더 직접 추적 pain

좋은 후보:

- 대표
- 원장
- 점장
- 운영 리더

이 사람이 아래 같은 말을 할 가능성이 있어야 한다.

- `내가 직접 챙겨야 돌아간다`
- `다시 물어봐야 한다`
- `어디까지 됐는지 한 번에 안 보인다`
- `후속조치가 흐려진다`

## 3. 반복 운영 루프

좋은 후보:

- 매주 또는 거의 매일 반복되는 루프가 있다
- 같은 종류의 점검, 전달, 확인, 후속이 반복된다

예:

- 공지 전달 -> 실행 확인 -> 후속 확인
- 예약/상담 -> 처리 -> 후속 안내
- 점검 -> 개선 -> 재확인

## 4. 정보 분산

좋은 후보:

- 카톡
- 메모
- 구두 지시
- 엑셀
- 간단한 문서

이 조합이 섞여 있어서 상태가 한곳에 안 모인다.

## 5. 도구 도입 가능성

좋은 후보:

- 완전 종이 기반만은 아님
- 새 도구를 붙일 정도의 최소 디지털 습관은 있음
- IT팀처럼 이미 강한 툴셋을 쓰는 것도 아님

## 리서치 대상 조직을 판별하는 체크리스트

아래 7개 중 5개 이상이면 우선 인터뷰 후보로 볼 수 있다.

1. 직원 또는 팀원이 3명 이상 있다
2. 리더와 실무자 역할이 어느 정도 나뉜다
3. 해야 할 일과 후속조치가 반복적으로 생긴다
4. 상태 확인이 카톡, 메모, 구두 지시에 흩어진다
5. 리더가 직접 다시 묻거나 chase하는 일이 잦다
6. 이번 주 상태를 한눈에 보기 어렵다
7. 새 도구를 붙일 정도의 디지털 습관은 있다

## 이제부터의 리서치 질문

업종 질문보다 아래 질문이 먼저 와야 한다.

### 1. 이 조직은 팀 운영 조직인가

- 누가 리더인가
- 누가 실무자인가
- 인수인계나 후속조치는 어떻게 도는가

### 2. 리더 직접 추적 pain가 있는가

- 리더는 뭘 직접 챙기고 있는가
- 왜 직접 챙겨야 하는가
- 안 챙기면 어떤 문제가 생기는가

### 3. shared picture 부재가 있는가

- 지금 상태는 어디에서 보는가
- 한눈에 안 보이는 이유는 무엇인가
- 회의 직전이나 점검 직전에 뭘 다시 모아야 하는가

### 4. 반복 운영 루프가 있는가

- 매주 반복되는 운영 장면은 무엇인가
- 점검 이후 다음 액션은 어떻게 정해지는가
- 후속 확인은 누가 어떤 방식으로 하는가

### 5. 제품 전환 가능성이 있는가

- 이 문제를 한 팀 범위에서 파일럿할 수 있는가
- 파일럿 결과가 도구 위에 남을 수 있는가
- 파일럿 이후에도 같은 루프를 계속 쓸 것 같은가

## 이제부터는 어떻게 고객 가설을 세워야 하는가

기존 방식:

- `프랜차이즈 본사`
- `약국`
- `식당`
- `학원`

이런 식으로 업종부터 세웠다.

앞으로는 아래 방식이 더 맞다.

### 1단계. 구조 가설

예:

- `리더 직접 추적 의존이 강한 소규모 팀 운영 조직`
- `점검과 후속조치가 반복되는데 shared picture가 약한 조직`
- `카톡/메모/구두 지시에 흩어진 운영 상태를 다시 정리해야 하는 조직`

### 2단계. 그 구조를 가진 업종 예시 붙이기

예:

- 학원
- 소형 체인
- 의원 운영팀
- 서비스업 운영팀
- 소규모 B2B 운영팀

즉 업종은 `가설 검증용 예시`이지, 시작점이 아니다.

## 리서치 방법도 바꿔야 한다

### 1. 업종별 시장조사보다 인터뷰 후보 관찰이 먼저다

먼저 볼 것:

- 대표/원장/운영리더가 어떤 말을 하는가
- 구인 공고에 어떤 운영 책임이 적히는가
- 블로그나 커뮤니티 글에서 어떤 운영 피로가 드러나는가

### 2. 판매보다 인터뷰가 먼저다

지금 단계에서는 아래가 목적이다.

- 어떤 구조 pain가 진짜 강한지 확인
- buyer language 수집
- 제품 전환 가능한 루프 확인

### 3. 업종 1개를 깊게 보기보다 구조가 비슷한 조직 10개를 넓게 본다

지금은 vertical lock-in보다 pattern recognition이 더 중요하다.

## 다음 리서치의 실제 목표

다음 리서치는 아래 3개를 얻어야 성공이다.

### 1. 우리가 진짜 공략할 조직 정의 1문장

예:

`직원 3~15명이 함께 움직이고, 리더 직접 추적 의존이 크며, 점검과 후속조치가 카톡/메모/구두 지시에 흩어진 소규모 팀 운영 조직`

### 2. buyer language 10개

예:

- `내가 다시 물어봐야 안 빠진다`
- `어디까지 됐는지 한 번에 안 보인다`
- `다들 바쁜데 결국 내가 챙긴다`

### 3. 파일럿 가능한 루프 3개

예:

- 전달 -> 실행 확인 -> 후속 확인
- 점검 -> 개선 -> 재확인
- 예약/상담 -> 처리 -> 후속 안내

## 다음 리서치에서 당장 버릴 질문

- `어느 업종이 제일 좋지?`
- `프랜차이즈가 맞나? 약국이 맞나?`
- `식당도 될까?`

이 질문들은 너무 빨리 vertical을 정하려는 질문이다.

더 먼저 물어야 할 질문은 아래다.

- `어떤 운영 구조가 우리 제품에 가장 잘 맞지?`
- `어떤 조직에서 리더 직접 추적 pain가 가장 선명하지?`
- `어떤 반복 루프를 파일럿하면 제품으로 남기 쉽지?`

## 지금 바로 해야 할 일

1. 업종 리스트 대신 `운영 구조 기준 인터뷰 대상 조건`을 확정한다
2. 그 조건에 맞는 조직 20개를 업종 섞어서 찾는다
3. 업종별이 아니라 구조별로 인터뷰 메모를 남긴다
4. 인터뷰 10건 후에야 다시 vertical을 좁힌다

## 현재 기준 최종 문장

`다음 고객 리서치는 프랜차이즈냐 약국이냐 식당이냐를 먼저 고르는 리서치가 아니라, 리더 직접 추적 pain, shared picture 부재, 반복 운영 루프, 도구 도입 가능성을 가진 소규모 팀 운영 조직이 어디에 있는지 다시 찾는 리서치여야 한다.`
