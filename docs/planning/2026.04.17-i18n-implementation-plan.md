# 다국어(i18n) 적용 구현 계획

## 배경
현재 DOWIN는 한국어만 지원하고 있습니다. 글로벌 사용자 대응 및 확장성을 위해 다국어 지원(한국어, 영어 우선)을 구현할 필요가 있습니다.

## 목표
- 한국어(ko) 및 영어(en) 지원.
- 사용자의 언어 설정을 유지(영구 저장).
- UI 컴포넌트 및 백엔드 에러 메시지의 현지화.
- SEO 친화적인 라우팅(URL 기반 로케일 지원).

## 조사 및 최적의 실무 사례
최신 Next.js(App Router) 표준에 기반한 조사 결과:
- **프론트엔드**: `next-intl`이 App Router 프로젝트의 표준으로 자리 잡았습니다. 서버 컴포넌트(RSC)를 지원하며, 타입 안정성이 높고 라우팅 통합이 쉽습니다.
- **라우팅**: `/ko/dashboard`, `/en/dashboard`와 같은 URL 세그먼트 방식이 상태의 명확성과 SEO를 위해 권장됩니다.
- **백엔드 API**: API는 번역된 메시지를 직접 반환하거나, 프론트엔드에서 번역할 수 있는 충분한 메타데이터(ErrorCode)를 제공해야 합니다. DOWIN는 현재 두 방식을 혼용하고 있으므로, 일관성을 위해 백엔드에서도 다국어 메시지 조회를 지원하는 것이 좋습니다.

## 현재 구현 상태 (2026-04-27 기준)

### 1. 라우팅 및 번역 리소스
- `next-intl` 기반 locale 세그먼트 라우팅이 적용되어 있다.
- 주요 화면은 `src/app/[locale]` 하위에서 렌더링된다.
- 번역 리소스는 `src/messages/ko.json`, `src/messages/en.json`을 사용한다.

### 2. 현재 locale 결정 순서
- 서버와 미들웨어는 아래 우선순위로 locale을 결정한다.
  1. `X-DOWIN-Locale` 헤더
  2. `NEXT_LOCALE` 쿠키
  3. `Accept-Language`
  4. 기본값 `ko`
- 루트 fallback(`/`)도 위 규칙과 동일한 helper를 사용한다.
- 로그인/회원가입/프로필 언어 변경 시 `NEXT_LOCALE` 쿠키를 동기화한다.

### 3. 로그인 사용자 초기 진입 처리
- 로그인 사용자가 저장된 locale과 다른 locale 경로로 진입하면, 미들웨어가 같은 경로의 locale prefix만 교정한다.
- 이 방식으로 보호 화면에서 영어가 먼저 보였다가 한국어로 바뀌는 클라이언트 플래시를 제거했다.
- 따라서 locale 교정은 클라이언트 `router.replace`가 아니라 가능한 한 첫 서버 응답 전에 처리하는 것을 기준으로 삼는다.

## 제안된 아키텍처

### 1. 프론트엔드: next-intl 통합
- **미들웨어(Middleware)**: URL, `NEXT_LOCALE` 쿠키 또는 `Accept-Language` 헤더에서 로케일을 감지합니다. `/dashboard` 접속 시 `/ko/dashboard` 또는 `/en/dashboard`로 리다이렉트합니다.
- **디렉토리 구조**: `src/app`의 모든 루트를 `src/app/[locale]` 하위로 이동합니다.
- **번역 리소스**: `messages/ko.json` 및 `messages/en.json`에 저장합니다.
- **공통 컴포넌트**: `useTranslations`(클라이언트) 또는 `getTranslations`(서버)를 사용합니다.

### 2. 백엔드: 로케일 대응 API 응답
- **감지**: `X-DOWIN-Locale` 헤더, `NEXT_LOCALE` 쿠키, `Accept-Language` 순서로 로케일을 추출하는 헬퍼를 사용합니다.
- **에러 메시지**: `src/lib/server/api-response.ts`를 수정하여 다국어 룩업 테이블을 지원하도록 변경합니다.
- **저장**: `users` 테이블에 `locale` 컬럼을 추가합니다.

### 3. 데이터베이스 변경 사항
```sql
ALTER TABLE users ADD COLUMN locale TEXT NOT NULL DEFAULT 'ko';
```
