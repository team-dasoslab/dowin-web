# DOWIN Agent Orchestration 설계

## 1. 배경 및 맥락 (Background & Context)

- DOWIN는 이미 작업 유형별 스킬을 갖고 있다.
  - `dowin-planning`
  - `dowin-backend`
  - `dowin-frontend`
  - `dowin-quality-check`
  - `dowin-performance-check`
  - `dowin-security-check`
- 따라서 지금 필요한 것은 새 범용 프레임워크 도입이 아니라, 이 스킬들이 한 작업 안에서 어떻게 이어지고 실패 시 어디로 돌아가야 하는지 명시하는 것이다.
- 목표는 superpowers 같은 범용 개발 방법론을 복제하는 것이 아니라, DOWIN의 현재 문서 중심 구조를 더 일관되고 반복 가능하게 만드는 것이다.
- 이 문서에서 말하는 "오케스트레이터"는 별도 서버나 제품 기능이 아니라, 요청을 먼저 분류하고 다음 작업 단계를 정하는 상위 작업 주체를 뜻한다.

## 2. 설계 목표 (Goals)

- 현재 작업 체인을 저장소 기본 운영 모델로 명시한다.
- 각 단계 출력에 공통 상태 형식을 부여한다.
- 검토 단계 실패 시 복귀 지점을 명확히 한다.
- 같은 실패 반복을 줄이기 위해 실패 카테고리를 표준화한다.
- 새 규칙이 기존 DOWIN 문서 읽기 순서와 충돌하지 않게 한다.

## 3. 비목표 (Non-Goals)

- worktree 기반 병렬 브랜치 운영 강제
- 전 작업 TDD 강제
- 자동 커밋/자동 머지
- 제품 사용자 플로우에 AI 생성 루프 도입
- 저장소 외부에서도 바로 통하는 범용 에이전트 프레임워크 제작

## 4. 핵심 개념 (Core Concepts)

### 4.1. 오케스트레이터

- 사용자 요청을 먼저 해석한다.
- 문제를 planning/backend/frontend/review 계열 중 어느 경로로 보낼지 정한다.
- 하위 단계에 넘길 맥락을 압축해서 전달한다.
- 각 단계 결과를 읽고 다음 단계 진행, 재시도, 복귀를 결정한다.

### 4.2. 작업자

- planning, backend, frontend, quality, performance, security 중 특정 단계의 실제 실행 주체다.
- 맡은 범위 안에서 파일을 읽고, 수정하거나, 검토하거나, 검증한다.
- 전체 작업을 다시 분해하는 책임보다는 맡은 단계의 산출물을 명확히 남기는 책임이 크다.

### 4.3. 단일 세션과 멀티 세션

- 단일 세션:
  - 하나의 에이전트가 오케스트레이터와 작업자 역할을 함께 수행한다.
  - 간단한 버그 수정, 문서 수정, 단일 도메인 변경에 우선 적용한다.
- 멀티 세션:
  - 상위 오케스트레이터가 별도 작업 세션을 열어 planning/backend/frontend 등을 분리한다.
  - 요구사항이 모호하거나, 백엔드와 프론트가 연쇄적으로 바뀌거나, 검토 단계까지 handoff가 필요한 작업에 적용한다.

원칙:

- 세션을 많이 여는 것 자체는 목표가 아니다.
- 컨텍스트 오염을 줄이고 책임 분리를 명확히 할 때만 멀티 세션을 쓴다.
- 단일 세션으로 충분한 작업을 억지로 planning -> backend -> frontend 체인으로 쪼개지 않는다.

## 5. DOWIN 기본 작업 체인 (Default Flow)

기본 순서는 아래로 둔다.

1. `planning`
2. `backend`
3. `frontend`
4. `quality-check`
5. `performance-check`
6. `security-check`

예외:

- 백엔드만 바뀌는 작업은 `frontend`를 생략할 수 있다.
- 프론트 단독 작업이어도 API 계약 확인이 필요하면 planning에서 그 범위를 먼저 고정한다.
- 성능/보안 검토가 명백히 무관한 사소한 문서 작업에는 후반 검토 단계를 축소할 수 있다.

## 6. 오케스트레이터 동작 규칙 (Orchestrator Behavior)

### 6.1. 첫 분류

오케스트레이터는 요청을 받으면 먼저 아래를 분류한다.

- 이 요청이 문서/기획 정리인지
- 백엔드 구현이 중심인지
- 프론트 연동이 중심인지
- 검토나 품질 게이트가 중심인지
- 단일 세션으로 끝낼 수 있는지
- 멀티 세션 handoff가 필요한지

### 6.2. 하위 단계 호출 기준

- `planning`을 먼저 거는 경우:
  - 요구사항이 모호함
  - MVP/Post-MVP 경계가 중요함
  - 백엔드와 프론트 변경 범위를 먼저 고정해야 함
- `backend`를 먼저 거는 경우:
  - API/도메인/DB 변경이 작업의 중심임
  - 프론트는 백엔드 결과를 받아 이어붙이는 성격임
- `frontend`를 먼저 거는 경우:
  - 서버 계약 변경 없이 UI/상태 처리 개선이 중심임
- review 계열을 거는 경우:
  - 구현 완료 후 회귀, 성능, 보안, 배포 위험을 판정해야 함

### 6.3. 하위 단계에 넘기는 정보

오케스트레이터는 전체 대화를 그대로 넘기지 않고 아래만 압축해서 넘긴다.

- 문제 요약
- 기대 동작
- 현재 확인된 사실
- 수정 범위
- 수정하면 안 되는 범위
- 통과 기준
- 이전 단계 산출물 중 다음 단계에 필요한 정보

## 7. 단계별 책임 (Stage Responsibilities)

### 7.1. planning

- 문제 정의
- 현재 구현과 문서의 차이 확인
- MVP/Post-MVP 구분
- 영향 도메인과 파일 범위 정리
- 성공/실패 기준 정리

### 7.2. backend

- OpenAPI 계약 반영
- Zod validation
- service/storage/route 구현
- 도메인 테스트 반영
- 필요 시 schema 설계

### 7.3. frontend

- 페이지/도메인 훅/공통 UI 연결
- loading/empty/error 상태 처리
- mutation, invalidation, toast, rollback 정리
- 모바일 동작 점검

### 7.4. quality-check

- 비즈니스 규칙 회귀
- 테스트 누락
- 응답 규격 불일치
- 문서/구현 불일치
- 배포 전 최소 검증 기준 점검

### 7.5. performance-check

- aggregation hot path
- 반복 스캔
- N+1 가능성
- 불필요한 데이터 로드
- 인덱스/쿼리 폭 위험

### 7.6. security-check

- auth/session
- authorization
- ownership filtering
- validation coverage
- secret/sensitive data exposure

## 8. 공통 출력 형식 (Shared Output Contract)

각 스킬은 작업 종료 시 아래 형식을 기준으로 결과를 남기는 것을 기본으로 한다.

```text
stage: planning|backend|frontend|quality|performance|security
status: pass|needs_revision|fail
summary: 한두 문장 요약
findings:
- ...
- ...
failure_categories:
- ...
- ...
return_to: planning|backend|frontend|none
next_step: 다음에 수행할 단계 또는 검증
```

규칙:

- `pass`
  - 다음 단계 진행 가능
- `needs_revision`
  - 같은 작업 맥락 안에서 수정 후 재검토 필요
- `fail`
  - 현재 단계 결과를 받아들이지 않고 명시한 단계로 복귀

`findings`가 없으면 빈 배열 대신 `- none`을 쓸 수 있다.

## 9. 세션 간 전달 형식 (Handoff Contract)

멀티 세션에서 오케스트레이터는 최소 아래 형식으로 다음 작업자에게 전달한다.

```text
task: 무엇을 해야 하는가
why_now: 왜 지금 이 단계를 수행하는가
scope:
- 수정해야 하는 범위
non_goals:
- 건드리면 안 되는 범위
known_facts:
- 이미 확인된 사실
artifacts_from_previous_stage:
- 이전 단계 산출물
done_criteria:
- 이 단계가 끝났다고 볼 조건
```

### 9.1. planning -> backend 전달 예시

```text
task: workspace 멤버 강퇴 API 구현
why_now: planning에서 MVP 범위와 응답 규격을 고정했음
scope:
- openapi
- route/service/storage
- backend tests
non_goals:
- frontend 멤버 관리 UI 변경
known_facts:
- ADMIN만 실행 가능
- 자기 자신 강퇴 불가
artifacts_from_previous_stage:
- 에러 응답 규격 초안
- 영향 파일 목록
done_criteria:
- 계약, 구현, 테스트가 일치
```

### 9.2. backend -> frontend 전달 예시

```text
task: 멤버 관리 화면에 강퇴 액션 연결
why_now: backend 구현과 테스트가 완료됨
scope:
- generated api hook 사용
- mutation/invalidation/toast 반영
non_goals:
- 멤버 카드 레이아웃 전면 개편
known_facts:
- 403/409 에러 케이스 존재
- 성공 시 멤버 목록 재조회 필요
artifacts_from_previous_stage:
- API path
- request/response shape
- 에러 코드 목록
done_criteria:
- 실제 API 연동
- loading/error/success 상태 처리 완료
```

## 10. 실패 카테고리 표준 (Failure Categories)

DOWIN에서는 아래 카테고리를 우선 표준으로 사용한다.

- `scope_gap`
- `mvp_boundary_gap`
- `api_contract_mismatch`
- `schema_design_gap`
- `missing_validation`
- `missing_test`
- `state_handling_gap`
- `rollback_gap`
- `performance_scan_risk`
- `n_plus_one_risk`
- `query_width_risk`
- `auth_gap`
- `ownership_gap`
- `secret_exposure_risk`
- `doc_impl_drift`

원칙:

- 자유 텍스트 설명은 허용하지만, 가능한 한 위 카테고리 중 하나 이상을 붙인다.
- 같은 실패가 반복될 때는 카테고리를 유지해 재작업 패턴을 드러낸다.

## 11. 복귀 규칙 (Return Rules)

### 11.1. planning으로 복귀하는 경우

- 문제 정의 자체가 모호함
- 범위가 과도하거나 누락됨
- MVP/Post-MVP 구분 부재
- 구현보다 기획 결정이 먼저 필요한 상태

### 11.2. backend로 복귀하는 경우

- API 계약 불일치
- validation 누락
- business rule 위반
- auth/ownership 누락
- storage/query 구조 문제
- 테스트 누락이 백엔드 동작과 직접 연결됨

### 11.3. frontend로 복귀하는 경우

- 상태 처리 누락
- API 연동 실수
- invalidation/rollback 누락
- 모바일 UI 회귀
- protected action의 노출/가드 문제

기본 원칙:

- 가능하면 가장 가까운 수정 가능 단계로 복귀한다.
- 모든 실패를 planning으로 올리지 않는다.

## 12. 재시도 정책 (Retry Policy)

- 동일 작업 안에서 자동 재시도는 최대 2회까지만 허용한다.
- 같은 `failure_categories` 조합이 2회 반복되면 사람 판단으로 전환한다.
- 재시도 프롬프트에는 반드시 아래가 포함돼야 한다.
  - 실패 단계
  - 실패 카테고리
  - 수정 범위
  - 건드리면 안 되는 범위
  - 통과 기준

예시:

```text
stage: quality
status: needs_revision
failure_categories:
- missing_test
- api_contract_mismatch
return_to: backend

수정 범위:
- /api/push/send-weekly-focus 응답 요약 구조
- 해당 route/service 테스트

금지:
- unrelated refactor
- frontend copy 변경

통과 기준:
- OpenAPI와 route 응답 shape 일치
- 실패 케이스 테스트 추가
```

## 13. 회귀 분류 규칙 (Regression Triage)

오케스트레이터는 "프론트가 깨졌다" 같은 증상을 바로 백엔드 책임으로 보내지 않는다.

### 13.1. 1차 분류 질문

- 요청이 실제로 나갔는가
- 응답 코드가 비정상인가
- 응답 shape가 계약과 다른가
- 응답은 정상인데 렌더링이나 상태 처리만 깨졌는가

### 13.2. 분류 기준

- 요청 자체가 안 나감:
  - 우선 프론트 상태/이벤트/가드 문제로 본다.
- 5xx 또는 명백한 서버 예외:
  - 우선 백엔드 회귀 가능성으로 본다.
- 2xx지만 계약과 다른 응답:
  - API 계약 회귀로 분류하고 backend로 보낸다.
- 2xx이고 계약도 맞지만 UI만 깨짐:
  - frontend로 보낸다.

### 13.3. 전달 원칙

- 백엔드에는 "프론트가 깨졌음"이 아니라 "어떤 요청/응답/계약 차이가 확인됐는지"를 보낸다.
- 프론트에는 "API 이상함"이 아니라 "응답은 정상이고 상태 처리/렌더링 경로를 보라"는 식으로 보낸다.
- 근거 없이 책임을 밀어넣지 않는다.

## 14. DOWIN 적용 규칙 (Repository Application)

### 10.1. 스킬 적용 순서

- planning 문맥이면 `dowin-planning`
- 백엔드 구현이면 `dowin-backend`
- 프론트 구현이면 `dowin-frontend`
- 구현 후 검증은 순서대로:
  - `dowin-quality-check`
  - `dowin-performance-check`
  - `dowin-security-check`

### 10.2. 검토 스킬의 역할 제한

- quality/performance/security 스킬은 기본적으로 판정과 복귀 지시를 담당한다.
- 직접 구현 수정까지 떠안는 것을 기본값으로 삼지 않는다.
- 즉, 검토 스킬은 “해결사”보다 “게이트”에 가깝다.

### 10.3. 문서 우선 원칙 유지

- 기존 AGENTS/온보딩의 읽기 순서는 유지한다.
- 오케스트레이션은 읽기 순서를 대체하는 것이 아니라, 작업 단계 간 연결 규칙을 보강하는 것이다.

## 15. 단계별 종료 정의 (Definition of Done Per Stage)

### 11.1. planning done

- 문제와 범위가 명시됨
- 영향 도메인과 파일 범위가 정리됨
- 구현 가능한 action item이 있음

### 11.2. backend done

- 관련 계약/검증/구현/테스트가 반영됨
- 최소 검증 명령이 통과하거나 잔여 리스크가 명시됨

### 11.3. frontend done

- UI 연동과 상태 처리가 완료됨
- 관련 invalidation/rollback/toast가 정리됨
- 모바일 영향이 검토됨

### 11.4. quality/performance/security done

- 결과가 `pass`이거나
- `needs_revision`/`fail`이면 복귀 지점과 핵심 findings가 명시됨

## 16. 도입 순서 (Rollout)

### 16.1. 1차 도입

- 본 문서 추가
- 각 DOWIN 스킬에 공통 출력 형식과 복귀 규칙 반영

### 16.2. 2차 도입

- 온보딩에 새 운영 모델 요약 추가
- 리뷰 응답 템플릿 정리

### 16.3. 3차 도입

- 실제 작업 세션에서 반복 사용
- 자주 나오는 failure category와 복귀 패턴 보정

## 17. 구현 방향 제안 (Implementation Direction)

초기 구현은 거창한 멀티 에이전트 런타임보다 아래 순서가 낫다.

1. 단일 세션 오케스트레이션을 기본값으로 둔다.
2. 각 스킬 출력 형식과 복귀 규칙을 먼저 맞춘다.
3. handoff 형식이 안정되면 planning/backend/frontend를 멀티 세션으로 분리한다.

이유:

- 지금 DOWIN는 저장소 특화 스킬이 강점이지, 세션 관리 인프라가 강점은 아니다.
- 출력 계약 없이 멀티 세션만 도입하면 전달 품질이 들쭉날쭉해진다.
- 먼저 단일 세션에서 사람이 읽어도 일관된 handoff 문장을 만들 수 있어야 한다.

## 18. 기대 효과 (Expected Impact)

- 현재 방식의 장점인 저장소 특화 규칙은 유지한다.
- 반면, 단계 간 손off가 자연어 감에만 의존하는 문제를 줄일 수 있다.
- 검토 스킬이 “좋아 보임/별로임”이 아니라 실제 게이트로 작동하게 된다.
- 같은 실패가 반복될 때 재시도 프롬프트 품질이 올라간다.
