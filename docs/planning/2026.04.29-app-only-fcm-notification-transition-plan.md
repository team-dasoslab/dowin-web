# 앱 전용 FCM 알림 전환 계획

## 1. 배경 및 맥락 (Background & Context)

- 현재 Dowin 알림은 PWA(Web Push + VAPID) 기반으로 설계되어 있다.
- 그러나 제품 방향은 브라우저 웹 알림을 유지하는 것이 아니라, 앱(WebView) 사용자에게만 알림을 제공하는 쪽으로 바뀌었다.
- 저장소에도 이 흔적이 남아 있다.
  - 웹 푸시 구독 UI: `src/components/PushSubscriptionManager.tsx`
  - Web Push 구독 API: `src/app/api/push/subscribe/route.ts`
  - Web Push 발송 경로: `src/domain/notification/services/web-push.ts`
  - 서비스워커 및 manifest: `src/app/sw.ts`, `public/manifest.json`
- 반면 WebView bridge는 이미 앱 전용 전환에 필요한 최소 계약을 일부 갖고 있다.
  - `requestNotificationPermission()`
  - `getPushToken()`

## 2. 문제 정의 (Problem Statement)

- 현재 알림 구조는 브라우저 `Notification`, `PushManager`, 서비스워커, VAPID 키를 전제로 한다.
- 이 구조는 앱 전용 알림 정책과 맞지 않는다.
- 특히 프로필 알림 토글은 브리지 기반 네이티브 권한/토큰 흐름이 아니라 브라우저 웹 푸시 구독 흐름에 직접 묶여 있다.
- 따라서 단순히 발송기만 FCM으로 바꾸면 안 되고, 저장 모델, 등록 API, 프론트 흐름, 문서, 법률 문구까지 함께 정리해야 한다.

## 3. 목표 (Goals)

### 3.1. MVP 목표

- 브라우저 웹 알림 지원을 중단한다.
- 알림 수신 채널을 앱(WebView) 로그인 사용자 전용으로 제한한다.
- 서버는 Web Push subscription 대신 FCM 디바이스 토큰을 저장한다.
- 프로필 알림 설정은 네이티브 권한 요청 + FCM 토큰 등록 흐름으로 전환한다.
- 기존 일일 리마인드 시간 설정은 유지한다.

### 3.2. Post-MVP 목표

- 복수 디바이스 토큰 관리 고도화
- 토큰별 앱 버전/플랫폼 세분화 정책
- 세션 종료/로그아웃/장기 미접속 토큰 정리 자동화
- 리더용 알림 및 유료 자동 리마인드의 앱 전용 채널 확장

## 4. 비목표 (Non-Goals)

- 브라우저 웹 알림 유지
- PWA 설치 UX 개선
- 이메일/카카오/슬랙 등 멀티채널 알림 확장
- 네이티브 앱 저장소 내부의 FCM SDK 구현 세부
- 알림 카피 전면 개편

## 5. 정책 결정 (Product Decision)

이번 전환에서 아래를 제품 정책으로 고정한다.

- 웹 브라우저에서는 알림을 지원하지 않는다.
- 알림은 앱(WebView) 환경에서만 켤 수 있다.
- 권한 요청, 디바이스 토큰 발급, 실제 푸시 수신은 네이티브 앱 책임이다.
- 웹 프론트는 브리지로 권한 상태를 읽고, 서버에 토큰 등록/해제 요청만 수행한다.
- 웹 브라우저에서 프로필 알림 섹션에 접근하면 `앱에서만 지원` 안내를 노출한다.
- 로그아웃 시 현재 기기 FCM 토큰은 서버에서 비활성화하거나 제거해야 한다.

## 6. 현재 구조와 목표 구조 비교 (Current vs Target)

### 6.1. 현재 구조

```text
[브라우저/PWA]
  └─ Notification.requestPermission()
  └─ Service Worker 등록
  └─ PushManager.subscribe()
  └─ POST /api/push/subscribe

[서버]
  └─ push_subscriptions(endpoint, p256dh, auth) 저장
  └─ VAPID 기반 Web Push 발송
```

### 6.2. 목표 구조

```text
[앱 WebView]
  └─ bridge.requestNotificationPermission()
  └─ bridge.getPushToken()
  └─ POST /api/notifications/devices

[서버]
  └─ device_push_tokens(token, platform, provider, lastSeenAt, disabledAt) 저장
  └─ FCM 기반 발송
```

## 7. 범위 정의 (Scope)

### 7.1. 이번 전환에서 다루는 것

- Notification 저장 모델 전환
- 토큰 등록/해제 API 추가
- 프로필 알림 설정 UI 전환
- 로그인 직후 알림 권한 요청 흐름 재정리
- 일일/주간 알림 발송기 FCM 전환
- PWA/Web Push 관련 자산 제거

### 7.2. 이번 전환에서 다루지 않는 것

- 네이티브 앱 내부 FCM 토큰 갱신 전략의 세부 구현
- 마케팅 푸시, 세그먼트 푸시, 인앱 메시지
- 브라우저 fallback 알림

## 8. 데이터 모델 전환안 (Data Model)

### 8.1. 유지할 것

- `user_notification_settings`
  - `dailyReminderEnabled`
  - `dailyReminderHour`
  - `dailyReminderMinute`
  - `timezone`

이 설정은 채널과 별개인 사용자 리마인드 정책이므로 유지한다.

### 8.2. 교체할 것

- 기존 `push_subscriptions`
  - `endpoint`
  - `p256dh`
  - `auth`

이 구조는 Web Push 전용이므로 앱 전용 FCM 토큰 모델과 맞지 않는다.

### 8.3. 신규 저장 구조 제안

테이블명 예시: `device_push_tokens`

권장 컬럼:

- `id`
- `user_id`
- `provider`
  - 초기값은 `FCM`
- `platform`
  - `IOS` | `ANDROID`
- `token`
- `app_version`
- `notification_enabled`
- `created_at`
- `updated_at`
- `last_seen_at`
- `disabled_at`

권장 제약:

- `token` unique
- `user_id` index
- `disabled_at` nullable

### 8.4. 마이그레이션 원칙

- 기존 Web Push 구독 데이터는 신규 테이블로 변환하지 않는다.
- 기존 `push_subscriptions`는 전환 완료 전까지 읽기만 유지하거나, 충분한 배포 확인 후 제거한다.
- `user_notification_settings`는 그대로 유지하여 리마인드 시간 설정을 보존한다.

## 9. API 계약 전환안 (API Contract)

### 9.1. 폐기 대상

- `POST /api/push/subscribe`
- `DELETE /api/push/subscribe`

이 API는 payload shape와 의미 모두 Web Push 구독에 강하게 묶여 있다.

### 9.2. 신규 API 제안

#### POST `/api/notifications/devices`

용도:

- 앱에서 현재 사용자 디바이스 토큰을 등록하거나 갱신한다.

예시 request:

```json
{
  "provider": "FCM",
  "platform": "IOS",
  "token": "fcm_device_token",
  "appVersion": "1.2.3",
  "notificationEnabled": true
}
```

동작:

- 세션 사용자 기준으로 토큰 upsert
- 첫 등록 시 필요하면 기본 알림 설정 생성
- `lastSeenAt` 갱신

#### DELETE `/api/notifications/devices`

용도:

- 현재 사용자 디바이스 토큰을 비활성화한다.

예시 request:

```json
{
  "token": "fcm_device_token"
}
```

동작:

- 현재 사용자 소유 토큰만 disable 처리
- hard delete보다 soft disable을 우선 검토

### 9.2.1. 로그아웃 연계 규칙

- 앱 사용자가 로그아웃할 때는 세션 종료와 별개로 현재 기기의 FCM 토큰 정리도 함께 수행한다.
- 권장 순서:
  1. 현재 기기 토큰 확인
  2. `DELETE /api/notifications/devices` 또는 동등한 disable API 호출
  3. `POST /api/auth/logout` 호출
- 토큰 정리 API가 실패하더라도 로그아웃 자체는 막지 않되, 다음 로그인 시 토큰 상태를 재동기화할 수 있어야 한다.
- 서버는 로그아웃 API 내부에서 토큰까지 강제 정리할지, 클라이언트가 명시적으로 분리 호출할지는 구현 시 선택할 수 있다.
- MVP 기준 최소 요구사항은 `로그아웃 후 해당 기기로 더 이상 푸시가 발송되지 않도록` 서버 상태를 정리하는 것이다.

### 9.3. 기존 설정 API

- `GET /api/push/settings`
- `PUT /api/push/settings`

이 API는 당장 재사용 가능하지만, 이름이 채널에 종속적이므로 추후 `/api/notifications/settings`로 정리할지 검토한다.

MVP에서는 아래 중 하나를 택한다.

- 선택지 A: 기존 경로 유지, 내부 의미만 앱 전용 알림으로 해석
- 선택지 B: 새 경로 추가 후 프론트 전환, 구 경로는 deprecated 처리

현재는 A가 더 현실적이다.

## 10. 프론트엔드 전환안 (Frontend)

### 10.1. 프로필 알림 설정

현재:

- `PushSubscriptionManager`가 브라우저 권한, 서비스워커, PushManager, `/api/push/subscribe`를 직접 다룬다.

변경 후:

- 브라우저 웹:
  - 토글 비활성화
  - `앱에서만 알림을 받을 수 있습니다` 안내 노출
- 앱 WebView:
  - 토글 ON 시
    1. `requestNotificationPermission()`
    2. `getPushToken()`
    3. 디바이스 등록 API 호출
  - 토글 OFF 시
    1. 현재 토큰 확인
    2. 디바이스 disable API 호출
  - 로그아웃 시
    1. 현재 토큰 확인
    2. 디바이스 disable API 호출
    3. 세션 로그아웃 호출

### 10.2. 브리지 사용 원칙

- `src/lib/bridge.ts`에서 브라우저 `Notification.requestPermission()` fallback 제거
- `getPushToken()`의 브라우저 fallback 빈 문자열 반환도 제거 또는 명시적 no-op 처리
- 브라우저에서 알림 관련 액션을 눌렀을 때는 실패가 아니라 정책 안내로 처리

### 10.3. 로그인 직후 권한 요청

- 앱 환경에서만 실행한다.
- 로그인 성공 후 첫 `dashboard/my` 진입 시 1회만 요청한다.
- 이미 한 번 요청한 세션 또는 기기 상태에서는 같은 로그인 흐름 안에서 반복 요청하지 않는다.
- `권한 요청`과 `토큰 등록`을 분리한다.
- 권한 허용 직후 서버 등록을 자동으로 할지, 프로필 토글에서 명시적으로 켤지 결정이 필요하다.

현재 권장:

- 로그인 직후 첫 진입 시에는 권한만 1회 묻고
- 실제 토큰 등록은 프로필 알림 토글 ON 시 수행

이유:

- 권한 허용과 리마인드 opt-in을 같은 의미로 취급하면 정책 경계가 흐려진다.
- iOS/Android 모두 알림 prompt를 반복 노출하면 거부율과 피로도가 커진다.
- 로그인 후 첫 핵심 화면(`dashboard/my`)에서만 1회 요청해야 제품 흐름이 예측 가능하다.

## 11. 발송 파이프라인 전환안 (Delivery)

### 11.1. 유지 가능한 것

- 일일 리마인드 대상 계산
- 주간 집중 리마인드 대상 계산
- 사용자 locale/timezone 기반 필터링

즉, `누구에게 어떤 메시지를 보낼지` 계산 계층은 상당 부분 재사용 가능하다.

### 11.2. 교체할 것

- `sendWebPushMessages`
- VAPID key 의존
- Web Push endpoint/p256dh/auth payload

### 11.3. 신규 발송 추상화 제안

- `sendFcmMessages`
- 입력 payload 예시
  - `token`
  - `title`
  - `body`
  - `url`
  - `pushType`
  - `campaignId`

### 11.4. 실패 처리 원칙

- FCM invalid token 응답은 soft disable 또는 삭제
- 일시적 실패와 영구 실패를 분리 집계
- 배치 결과에 `success`, `failed`, `disabledTokens`를 포함

## 16. 오픈 쟁점 (Open Questions)

### 16.1. 설정 API 경로 rename 시점

- 지금 즉시 `/api/notifications/settings`로 옮길지
- 아니면 먼저 토큰 등록 구조만 바꾸고 settings 경로는 유지할지

현재는 후자가 현실적이다.

### 16.2. 토글 OFF의 의미

- 서버 토큰 soft disable만 할지
- 네이티브에서 OS-level 권한 해제를 유도할지

MVP에서는 soft disable이 맞다.

### 16.4. 로그아웃 시 토큰 정리 위치

- 선택지 A: 클라이언트가 `DELETE /api/notifications/devices` 후 `POST /api/auth/logout`
  - 장점: 현재 기기 기준 정리가 명확함
  - 단점: 호출 수가 늘어남
- 선택지 B: `POST /api/auth/logout` 내부에서 현재 기기 토큰도 함께 disable
  - 장점: 호출이 단순함
  - 단점: 현재 기기 토큰 식별자를 auth logout 경로에 어떻게 넘길지 계약이 추가로 필요함

현재는 A가 더 명확하다.

### 16.3. 자동 등록 시점

- 로그인 직후 권한 허용만 받아도 자동 등록할지
- 프로필 토글 ON에서만 등록할지

현재는 프로필 토글 ON에서만 등록하는 쪽이 더 명확하다.

## 17. 액션 아이템 (Action Items)

- [ ] 일일/주간 알림 발송기를 FCM sender로 교체
- [ ] PWA/Web Push 자산 제거
- [ ] 문서/법률 문구 갱신

## 18. 성공 및 실패 판단 기준 (Success & Failure Criteria)

### 성공

- 브라우저 웹에서 더 이상 알림 구독이나 PWA 설치를 전제하지 않는다.
- 앱 WebView에서만 알림 토글과 리마인드 설정이 유효하게 동작한다.
- 서버는 Web Push subscription이 아니라 FCM 디바이스 토큰을 사용한다.
- 기존 리마인드 시간 설정은 보존된다.

### 실패

- 프론트는 앱 전용이라고 보이지만 서버 발송은 여전히 VAPID/Web Push에 묶여 있다.
- 브라우저 fallback이 남아 정책과 구현이 어긋난다.
- 토큰 무효화/재설치/중복 등록 케이스를 고려하지 않아 운영 중 장애가 생긴다.
