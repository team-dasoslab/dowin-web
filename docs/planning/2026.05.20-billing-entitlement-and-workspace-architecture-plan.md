# Billing Entitlement 중앙화 + Workspace 아키텍처 변경 계획

## 1. 배경 및 맥락

- 현재 Dowin의 billing 구조는 `FREE | STANDARD` 단일 플랜과 workspace 월정액을 전제로 구현되어 있다.
- 현재 구현은 checkout, webhook, billing projection, workspace 멤버 제한, profile/pricing UI가 모두 이 전제를 공유한다.
- 동시에 workspace 도메인은 `사용자 1명당 1개 워크스페이스`를 강하게 가정한다.
- 이번 변경의 목적은 아래 3가지를 바로 한 번에 구현하는 것이 아니라, 앞으로 이 3가지를 바꿀 수 있도록 공통 전제를 먼저 중앙화하는 것이다.
  - 결제 없는 유료 예외 워크스페이스를 위한 `entitlementSource`
  - 사용자 다중 워크스페이스 참여/소유
  - seat 기반 자동 과금

## 2. 현재 문제 정의

### 2.1. billing/권한 판단이 여러 층에 흩어져 있다

- 현재 권한 판단은 `planCode`, `billingStatus`, `canManageBilling`, 멤버 수 제한, portal 노출 조건 등으로 여러 파일에 흩어져 있다.
- 이 구조에서는 `결제된 STANDARD`와 `운영자가 부여한 STANDARD`를 동일하게 취급하기 쉽다.
- 결과적으로 `무료 예외 entitlement`를 붙이려 해도 billing service, admin override, profile billing UI, pricing UI를 함께 수정해야 한다.

### 2.2. billing 모델이 `플랜 코드` 중심이다

- 현재 시스템의 주 질문은 `이 워크스페이스가 FREE인가 STANDARD인가`다.
- seat 기반 과금으로 가려면 질문이 `이 워크스페이스에 몇 좌석 entitlement가 있는가`, `누가 billable member인가`, `좌석 변경을 언제 반영하는가`로 바뀐다.
- 즉 단순 가격 수정이 아니라 권한 모델 자체가 바뀌는 일이다.

### 2.3. workspace 컨텍스트가 단일 membership 전제에 묶여 있다

- 현재 많은 서비스가 `findUserWorkspace(userId)`와 `findMembershipByUserId(userId)`를 사용한다.
- 이 구조는 `현재 사용자에게 workspace가 하나만 있다`는 가정에 기대고 있다.
- 다중 워크스페이스를 열려면 `현재 활성 workspace`를 해석하는 공통 컨텍스트 계층이 먼저 필요하다.

## 3. 핵심 판단

### 3.1. 세 변경의 성격은 다르다

- `entitlementSource`
  - 권한 판단 중앙화가 잘 되어 있으면 비교적 작게 끝낼 수 있는 변경이다.
- `per-seat billing`
  - 모듈화가 좋아도 여러 계층을 반드시 바꿔야 하는 도메인 모델 변경이다.
- `다중 workspace`
  - 모듈화가 좋아도 앱 전역의 workspace context를 바꾸는 아키텍처 변경이다.

### 3.2. 따라서 구현 순서를 바꿔야 한다

- 바로 `per-seat`나 `다중 workspace`에 들어가지 않는다.
- 먼저 `권한 판단`과 `workspace 선택`을 공통 계층으로 빼고, 이후 각 기능 변경을 순차적으로 태운다.

## 4. 목표

### 4.1. 이번 planning의 목표

- 이후 변경을 받기 쉬운 공통 구조를 먼저 정의한다.
- 중앙화 없이 바로 큰 기능 변경에 들어가며 생길 재작업을 줄인다.
- `manual entitlement`, `active workspace`, `seat-aware entitlement`가 서로 충돌하지 않게 순서를 고정한다.

### 4.2. 이번 planning의 비목표

- 즉시 seat 기반 billing 구현
- 즉시 다중 workspace 사용자 노출
- 즉시 플랜 가격/정책 변경 확정

## 5. 중앙화 선행 변경안

### 5.1. Entitlement를 billing 상태와 분리한다

- 현재:
  - `billingStatus`와 `planCode`를 조합해 권한을 추론한다.
- 변경 목표:
  - `billing`은 외부 결제 상태
  - `entitlement`는 실제 제품 권한 상태
  - `source`는 entitlement가 왜 열려 있는지

권장 개념:

- `planCode`
  - 사용자/운영자에게 보이는 권한 레벨
- `billingStatus`
  - 외부 결제 상태 (`NONE`, `ACTIVE`, `CANCELED`, `EXPIRED`, `REVOKED`)
- `entitlementSource`
  - `POLAR`, `MANUAL_GRANT`, `PARTNER`, `INTERNAL_TEST`
- `entitlementStatus`
  - `ACTIVE`, `SCHEDULED_END`, `INACTIVE`
- `entitlementEndsAt`
  - 기간성 예외 권한 종료 시점

핵심 원칙:

- portal 진입 가능 여부는 `billingStatus`만이 아니라 `entitlementSource`까지 보고 판단한다.
- 제품 기능 게이트는 최종적으로 `entitlement`를 기준으로 읽는다.
- `POLAR`가 아닌 entitlement는 결제 관리 UI와 명시적으로 분리한다.

### 5.1.1. Workspace operational access 정책

워크스페이스 URL 접근은 두 단계로 나눈다.

1. `workspace access`
   - 사용자가 해당 워크스페이스의 멤버인지 확인한다.
   - 실패하면 워크스페이스 정보 자체를 노출하지 않는다.
2. `workspace operational access`
   - 멤버십이 있고, Basic 사용권이 활성인지 확인한다.
   - 운영 화면과 운영 API는 이 조건을 만족해야 한다.

운영 접근 허용 기준:

- `planCode`가 `BASIC` 또는 과거 호환값 `STANDARD`
- `billingStatus`가 `ACTIVE` 또는 `CANCELED`
- 최근 billing risk 기준으로 수동 검토가 필요하지 않음

`CANCELED`는 현재 결제 기간이 남아 있는 상태로 취급해 운영 접근을 허용한다. `NONE`, `EXPIRED`, `REVOKED`, 수동 검토 필요 상태는 운영 접근을 차단한다.

결제 비활성 상태에서도 허용하는 표면:

- 워크스페이스 목록/전환
- `/{workspaceId}/subscription-required`
- `/{workspaceId}/profile`
- `/{workspaceId}/profile/billing`
- `/{workspaceId}/profile/contact`
- `/{workspaceId}/profile/password`
- `/{workspaceId}/profile/avatar`
- `/{workspaceId}/profile/delete-account`
- `/{workspaceId}/workspace/settings`
- `/{workspaceId}/profile/members`
- `/{workspaceId}/profile/invites`

결제 비활성 상태에서 차단하는 표면:

- `/{workspaceId}/dashboard`
- `/{workspaceId}/dashboard/my`
- `/{workspaceId}/setup`
- `/{workspaceId}/scoreboards`
- `/{workspaceId}/report`
- `/{workspaceId}/profile/export`
- 점수판, 액션 아이템, 기록, 팀 메모, 리포트, CSV export 같은 운영 API

프론트 라우트는 결제 비활성 운영 화면 접근을 `/{workspaceId}/subscription-required`로 유도한다. API와 서비스는 URL 직접 접근이나 오래된 클라이언트를 막기 위해 `BASIC_SUBSCRIPTION_REQUIRED`와 `WORKSPACE_SEAT_LIMIT_EXCEEDED`를 계속 반환한다.

#### 운영 API gate 점검표

2026-06-01 기준 운영 API coverage는 아래 기준으로 본다.

| --- | --- | --- |

### 5.2. Workspace Access Context를 공통화한다

- 현재:
  - 대부분의 서비스가 `findUserWorkspace(userId)`로 바로 workspace를 결정한다.
- 변경 목표:
  - 사용자에게 여러 membership이 있어도 `현재 어떤 workspace를 보고 있는지`를 공통 계층이 결정하게 한다.

권장 개념:

- `ActiveWorkspaceResolver`
  - request, session, persisted preference 중 하나를 기준으로 현재 workspace를 결정
  - **(결정사항)** 전역 미들웨어가 아닌, 워크스페이스 맥락이 필요한 API 라우트나 서버 액션 내부에서만 명시적으로(Opt-in) 호출(`requireWorkspaceAccess`)하여 불필요한 성능 낭비를 방지한다.
- `WorkspaceAccessContext`
  - `workspaceId`, `role`, `membershipId`, `planCode`, entitlement 요약을 같이 제공
  - **(결정사항)** Cloudflare D1 환경 특성상(HTTP 기반 통신) N+1 쿼리 지연이 치명적이므로, 이 Context를 생성할 때 결제 상태와 워크스페이스 정보를 단일 쿼리(또는 Batch)로 한 번에 가져오도록 강제한다.

핵심 원칙:

- 도메인 서비스는 `userId -> workspace 단일 조회`를 직접 하지 않는다.
- 가능한 한 `workspaceId` 또는 `WorkspaceAccessContext`를 입력으로 받는다.
- authz helper도 `workspaceId`를 명시 인자로 받는 쪽으로 수렴한다.
- 워크스페이스 무관 API(프로필 수정 등)에서는 이 Context를 생성하지 않는다.

### 5.3. Member Capacity를 pricing과 분리한다

- 현재:
  - 멤버 수 제한이 `planCode -> memberLimit`로 바로 이어진다.
- 변경 목표:
  - 멤버 수용량(capacity) 판단을 entitlement 계층으로 분리한다.

권장 개념:

- `CapacityPolicy`
  - 현재 workspace가 허용하는 최대 멤버 수 계산
- `SeatEntitlement`
  - 후속 per-seat에서 `purchasedSeatCount`, `graceSeatCount`, `overagePolicy` 등을 담을 수 있게 설계

핵심 원칙:

- invite/create/join 시점의 제한 판단은 공통 capacity policy를 통해서만 수행한다.
- `FREE=10`, `STANDARD=30` 같은 현재 규칙은 초기 구현값일 뿐, 상위 서비스에서 직접 해석하지 않는다.

## 6. 3개 변경의 순차 적용안

### 6.1. 1단계: `entitlementSource` 도입

목표:

- 결제 기반 entitlement와 운영자 부여 entitlement를 분리한다.

필수 변경:

1. `workspace_billing_state` 또는 별도 entitlement projection에 source 관련 필드 추가
2. admin manual override가 `MANUAL_GRANT`를 명시적으로 기록
3. profile billing/pricing 화면에서 source별 문구와 portal 노출 조건 분리
4. 제품 기능 게이트가 `planCode + entitlementSource + entitlementStatus`를 읽도록 정리

기대 효과:

- 특정 workspace만 유료 기능을 무료 예외로 쓰게 하는 요구를 운영자 친화적으로 처리할 수 있다.
- 이후 partner/free test/grace workspace 같은 예외 정책도 같은 구조로 흡수 가능하다.

### 6.2. 2단계: 다중 workspace를 위한 active workspace 도입

목표:

- 사용자 1명이 여러 workspace에 참여할 수 있도록 전제를 바꾼다.

필수 변경:

1. `workspace_members.user_id` unique 제거
2. 대신 `(workspace_id, user_id)` unique 추가
3. `listUserWorkspaces`, `findMembership(workspaceId, userId)` 중심으로 storage/service 전환
4. `GET /api/workspaces/me` 등 단일 응답 구조를 다중 응답 구조로 재설계
   - `GET /workspaces/me` -> `GET /workspaces` (목록) 및 `GET /workspaces/[workspaceId]` (단건)으로 분리
   - `PUT /workspaces/current` -> 완전 폐기 (URL 전환으로 대체)
   - `POST /workspaces/join` -> `POST /workspaces/[workspaceId]/join`
5. **(결정사항)** Client URL 및 API 경로를 `/[workspaceId]/dashboard/...` 형태로 전면 개편
   - 쿠키(`dowin_workspace`) 기반의 암묵적 상태 의존을 제거하여 멀티 탭 Race Condition을 방지한다.
   - 쿠키는 최상위 경로(`/`) 접속 시 마지막 접속 워크스페이스로 리다이렉트해주는 용도로만 제한적으로 사용한다.
6. 대시보드, billing, profile, analytics, contact 등에서 URL의 `workspaceId`를 기반으로 Context 명시화

기대 효과:

- billing이 원래 workspace 단위이기 때문에, 다중 workspace를 허용하면 과금 모델과 실제 사용자 구조가 오히려 더 잘 맞는다.

주의:

- 이 단계는 entitlementSource보다 영향 범위가 넓다.
- 반드시 active workspace 계층을 먼저 넣고, 이후 화면/API 전환을 단계적으로 태워야 한다.

### 6.3. 3단계: per-seat billing

목표:

- workspace 월정액 중심 모델에서 seat-aware entitlement 모델로 확장한다.

필수 변경:

1. billable seat 정의
   - **(결정사항)** '초대(Invite)'를 발송하는 즉시 1 Seat을 점유하는 것으로 간주한다. (어뷰징 방지 및 Notion 표준 적용)
   - **(결정사항)** 현재 역할(ADMIN, MEMBER) 구별 없이 워크스페이스 내 모든 사용자는 동일하게 1 Seat을 차지한다. (무료 Guest는 고려하지 않음)
   - 공식: `Used Seats = count(workspace_members) + count(pending_invites)`
2. seat entitlement 저장 모델 추가
3. capacity policy를 purchased seat 기준으로 변경
4. Polar subscription quantity/update 또는 seat-pack 전략 확정
5. proration/upgrade timing/downgrade timing 정책 결정
6. webhook projection이 plan뿐 아니라 seat count까지 반영
7. billing UI에 현재 seat 상태, 증감 정책, 초과 상태 노출

기대 효과:

- 작은 팀과 큰 팀 사이 가격 공정성을 높일 수 있다.

주의:

- 이 단계는 중앙화가 되어 있어도 작지 않다.
- 중앙화의 효과는 "한 군데 수정으로 끝난다"가 아니라 "영향 범위를 예측 가능한 모듈로 제한한다"는 데 있다.

## 7. 권장 구현 순서

1. entitlement 중앙화
   - `EntitlementService` 또는 동등한 공통 판정 계층 도입
2. capacity policy 중앙화
   - invite/join/member limit 판단을 한 곳으로 수렴
3. `entitlementSource` 도입
4. active workspace resolver 도입
5. 다중 workspace 허용
6. seat entitlement 모델 추가
7. Polar per-seat 또는 tiered hybrid 전략 확정 후 적용

## 8. 설계 권장안

### 8.1. 서비스 경계

- `BillingService`
  - 외부 결제 흐름, checkout, portal, webhook projection 갱신 담당
- `EntitlementService`
  - 현재 workspace의 최종 사용 가능 권한 판단 담당
- `WorkspaceAccessService` 또는 `ActiveWorkspaceResolver`
  - 현재 사용자의 workspace context 선택 담당
- `CapacityPolicyService`
  - 멤버 수용량 및 join/invite 가능 여부 판단 담당

### 8.2. 피해야 할 것

- 화면에서 `planCode === "STANDARD"` 같은 분기 재증식
- service마다 독자적으로 `billingStatus`를 해석하는 구조

## 13. 구현 감사 결과 (2026-05-28)

### 13.0. 현재 플랜 운영 전제

- 현재 제품 운영 전제는 **모든 워크스페이스가 Basic 플랜**이라는 것이다.
- Basic은 무료 플랜이 아니라 **가입과 동시에 결제해야 하는 seat 기반 기본 유료 플랜**이다.
- 구현 코드와 DB enum에는 아직 과거 명칭인 `FREE | STANDARD`가 남아 있다.
- 따라서 앞으로의 목표 모델은 `FREE -> BASIC` 단순 rename이 아니라, `FREE` 무료 운영 전제를 제거하고 `BASIC` paid seat entitlement를 기본값으로 만드는 것이다.
- 사용자는 결제 완료 전에는 정상 가입 완료 상태가 아니며, 워크스페이스와 세션은 결제 성공 검증 이후에만 활성화되어야 한다.
- 기존 `STANDARD` 업그레이드 checkout 흐름은 목표 모델과 맞지 않는다. Basic seat checkout이 가입 플로우의 기본 경로가 되고, `STANDARD`는 제거하거나 후속 상위 플랜으로 다시 정의해야 한다.

### 13.1. 이미 반영된 것

- `entitlementSource`는 `workspace_billing_state`에 추가됐고, OpenAPI/Orval 응답 타입에도 노출되어 있다.
- Polar webhook projection은 결제 기반 권한을 `entitlementSource=POLAR`로 기록한다.
- 운영자 manual override는 `STANDARD` 전환 시 기본적으로 `MANUAL_GRANT`를 기록한다.
- profile billing/pricing UI는 `POLAR`이 아닌 entitlement에서 billing portal CTA를 분리한다.
- `WorkspaceAccessContext`와 `requireWorkspaceAccess`가 도입되어 일부 workspace-scoped API가 workspace, membership, billing state를 한 번에 해석한다.
- `workspace_members.user_id` 단일 unique 제약은 제거됐고, `(workspace_id, user_id)` unique 제약으로 바뀌었다.

### 13.2. 아직 남은 구조적 빈틈

- 별도 `EntitlementService`는 아직 없고, 제품 기능 접근 여부는 `planCode === "STANDARD"`에 가깝게 판정된다.
- `entitlementStatus`, `entitlementEndsAt` 같은 기간성 권한 상태는 아직 없다.
- `CapacityPolicyService`가 없어서 join/invite/member limit 판단이 `planCode -> memberLimit`에 직접 묶여 있다.
- seat entitlement 저장 모델이 없다. `purchasedSeatCount`, `graceSeatCount`, `usedSeatCount` 같은 값을 저장하거나 계산하는 경계가 아직 없다.
- Polar checkout과 webhook projection은 quantity 또는 seat-pack 정보를 다루지 않는다.
- `findUserWorkspace(userId)`, `findMembershipByUserId(userId)` 기반의 단일 workspace fallback이 일부 active code path에 남아 있다.
- `POST /api/auth/signup`은 현재 사용자 생성, 복원코드 생성, 세션 발급까지 즉시 완료한다. 가입 즉시 결제 모델에서는 이 흐름이 부적합하다.
- `POST /api/workspaces`는 현재 결제 없이 워크스페이스를 생성한다. 목표 모델에서는 결제 성공 전 워크스페이스가 활성 생성되면 안 된다.

## 16. 결제 비활성 워크스페이스의 Basic 재시작 흐름 (2026-06-01)

### 16.1. 문제 정의

워크스페이스 생성 시 Basic seat checkout을 요구하더라도, 운영 중인 워크스페이스의 결제 상태가 이후에 비활성화될 수 있다.

- 사용자가 결제를 완료하지 않고 이탈해 `NONE` 상태가 남는 경우
- 구독 기간이 종료되어 `EXPIRED`가 된 경우
- 결제 실패 복구 기간을 지나 혜택이 종료된 경우
- 수동 검토나 환불/취소 이슈로 `REVOKED`가 된 경우

이미 구현한 운영 접근 gate는 대시보드, 설정, 점수판, 리포트, export 같은 운영 표면을 차단하는 데 충분하다. 다만 차단 화면에서 사용자가 결제 문제를 해결할 수 있는 복구 경로가 아직 명확하지 않다.

### 16.2. Polar portal과 새 checkout의 역할 분리

Polar Customer Portal은 기존 고객이 자기 결제 관계를 관리하는 표면이다. Dowin에서는 아래 용도로 본다.

- 활성 구독의 결제수단 변경
- 결제 실패 후 카드 갱신
- invoice/receipt 확인
- 활성 구독 취소
- 설정이 허용하는 범위의 seat/plan 관리

반면 새 Basic checkout은 `현재 복구할 활성 구독이 없거나, portal만으로 복구 가능하다고 확정할 수 없는 상태`에서 새 구독 결제를 시작하는 표면이다.

따라서 `portal 하나로 모든 재구독을 해결한다`는 전제를 두지 않는다. 결제 비활성 워크스페이스에는 상태별로 portal과 checkout을 분리해서 노출한다.

### 16.3. 상태별 UX 결정

| 상태 | 운영 접근 | 기본 CTA | 보조 CTA | 비고 |
| --- | --- | --- | --- | --- |
| `ACTIVE` | 허용 | 결제 관리 열기 | 없음 | 정상 상태 |
| `CANCELED` + 기간 남음 | 허용 | 결제 관리 열기 | 없음 | 현재 기간 종료 전까지 운영 접근 허용 |
| `PAST_DUE` | 정책 결정 전까지 차단 또는 경고 | 결제 관리 열기 | 없음 | Polar상 복구 가능한 상태다. 현재 enum에는 없으므로 후속으로 추가 여부 결정 |
| `NONE` | 차단 | Basic 구독 시작하기 | 결제 관리 열기 가능 시 노출 | 구독이 없으므로 새 checkout이 기본 |
| `EXPIRED` | 차단 | Basic 구독 다시 시작하기 | 결제 관리 열기 | portal 복구 가능성을 열어두되 checkout을 기본 복구 경로로 제공 |
| `REVOKED` | 차단 | 문의하기 | 운영자 판단 시 checkout | 환불/악용/권한 회수 가능성이 있으므로 일반 self-serve 재개는 보수적으로 제한 |

현재 구현 enum은 `NONE | ACTIVE | CANCELED | EXPIRED | REVOKED`만 가진다. Polar의 `past_due` 이벤트는 복구 가능한 결제 실패 상태이므로, 후속 구현에서 `PAST_DUE`를 별도 `billingStatus`로 추가하는 것을 우선 검토한다. 추가 전까지는 `subscription.past_due`를 `EXPIRED`나 `REVOKED`로 섞어버리지 말고, projection 정책을 확정한 뒤 반영한다.

### 16.4. API 설계안

기존 워크스페이스 생성 checkout과 별도로, 이미 존재하는 워크스페이스의 Basic 결제를 다시 시작하는 API를 둔다.

- `POST /api/workspaces/{workspaceId}/billing/checkout`
- 관리자만 호출 가능
- `requireWorkspaceAccess`로 멤버십과 role을 확인한다.
- `billingStatus`가 `NONE` 또는 `EXPIRED`일 때 self-serve checkout을 허용한다.
- `REVOKED`는 기본적으로 `CONTACT_REQUIRED` 계열 에러를 반환하고, 운영자 수동 override 이후에만 허용하는 방향을 검토한다.
- `ACTIVE`, `CANCELED`는 새 checkout 대신 portal을 사용하게 한다.
- checkout success redirect는 사용자 안내와 후속 polling/재조회에만 사용하고, 권한 활성화는 반드시 Polar webhook projection을 기준으로 확정한다.

요청/응답 초안:

```text
POST /api/workspaces/{workspaceId}/billing/checkout

request:
- idempotencyKey: string
- seatCount?: number

response:
- checkoutUrl: string
- checkoutId?: string
```

`seatCount` 기본값은 현재 사용 좌석 수다. seat 계산은 기존 결정처럼 `workspace_members + pending_invites`를 기준으로 하되, 최소 1 seat을 보장한다.

### 16.4.1. Polar checkout seat 고정 정책

Polar 공식 `Create Checkout Session` API는 seat-based pricing 전용 필드로 `seats`, `min_seats`, `max_seats`를 제공한다. 문서상 `seats`는 checkout의 사전 지정 좌석 수이고, `min_seats`/`max_seats`는 checkout에서 허용하는 최소/최대 좌석 수다.

참조:

- [Polar Create Checkout Session](https://polar.sh/docs/api-reference/checkouts/create-session)
- [Polar Checkout Links](https://polar.sh/docs/features/checkout/links)

Dowin는 static Checkout Link가 아니라 Checkout API로 session을 생성하므로, Basic seat 수 제한을 서버에서 정해 checkout session에 함께 전송한다.

워크스페이스 최초 생성 checkout은 아직 기존 멤버가 없고 사용자가 Dowin에서 seat 수를 선택하는 단계이므로, 선택한 seat 수로 고정한다.

```json
{
  "seats": 5,
  "min_seats": 5,
  "max_seats": 5
}
```

정책:

- 워크스페이스 최초 생성 checkout은 사용자가 Dowin에서 입력한 seat 수로 고정한다.
- 결제 비활성 워크스페이스의 Basic 재시작 checkout은 현재 멤버 수보다 낮게 결제할 수 없게 `min_seats=usedSeatCount`로 생성한다.
- Basic 재시작 checkout의 `seats`는 `max(requestedSeatCount, usedSeatCount)`로 계산한다.
- Basic 재시작 checkout의 `max_seats`는 현재 API 입력 상한과 같은 999로 둔다.
- 따라서 기존 워크스페이스 복구 checkout에서는 다운그레이드는 막고, checkout 화면에서 추가 seat 구매는 허용한다.
- Customer Portal에서 seat 변경을 허용할지는 checkout 고정 정책과 별개다. Portal 변경이 켜져 있으면 webhook의 실제 subscription seats를 최종 source of truth로 삼아 Dowin seat entitlement를 갱신한다.

구현 보강:

- `PolarBillingClient.createCheckoutSession`은 `minSeats`, `maxSeats`를 받아 `min_seats`, `max_seats`로 전송한다.
- workspace setup checkout은 `seats = minSeats = maxSeats`로 생성한다.
- resubscribe checkout은 `seats = max(requestedSeatCount, usedSeatCount)`, `minSeats = usedSeatCount`, `maxSeats = 999`로 생성한다.
- workspace setup checkout 완료 검증 시 Polar checkout detail의 실제 `seats`를 읽어 `purchasedSeatCount`로 저장한다. webhook이 workspace 생성보다 먼저 도착해도 pending 입력값만으로 seat 수가 고정되는 빈틈을 줄이기 위함이다.

### 16.4.2. Basic 환불/해지 정책

Basic seat 구독은 결제 직후 워크스페이스 대시보드, 기록 조회, 운영 기능 접근권이 열린다. 따라서 사용자가 결제와 동시에 서비스를 이용할 수 있는 구조이고, 이미 결제된 월 이용 기간은 원칙적으로 환불하지 않는다.

정책:

- 일반 해지는 환불이 아니라 `현재 결제 기간 종료 후 자동 갱신 중단`으로 처리한다.
- 결제 후 한 달 뒤 종료되도록 미리 해지 예약하는 것은 허용한다.
- 해지 예약 상태에서는 `current_period_end`까지 Basic 권한과 seat entitlement를 유지한다.
- 기간 말 해지가 실제로 종료되면 `EXPIRED` 또는 동등한 비활성 상태로 전환하고 운영 접근 gate를 적용한다.
- 법령상 환불 권리, 중복 결제, 명백한 오결제, 결제 직후 Dowin 귀책 기술 오류, Polar/결제 네트워크의 chargeback 예방 환불 등은 예외로 별도 검토한다.
- 예외 환불 또는 즉시 revoke가 확정되면 Basic 권한은 즉시 회수한다.

Polar 공식 문서 기준:

- Customer Portal은 항상 제공되며 고객은 활성 구독을 직접 취소할 수 있다: https://polar.sh/docs/features/customer-portal
- Polar는 `cancel_at_period_end=true` 방식의 기간 말 해지를 지원하며, 이 경우 현재 기간 종료일까지 benefit이 유지된다: https://polar.sh/docs/guides/subscription-upgrades
- Polar는 merchant가 `no refunds` 정책을 두더라도 chargeback 예방 목적으로 구매 후 60일 내 자체 판단 환불을 할 수 있다: https://polar.sh/docs/features/refunds

### 16.4.3. 환불 반복 악용 방지

Polar는 Dowin 환불 정책과 별개로 chargeback 예방 목적의 환불을 처리할 수 있다. 따라서 Dowin가 환불 자체를 완전히 막는 구조는 불가능하며, 방어선은 `권한 회수`, `재결제 차단`, `수동 검토`에 둔다.

위험 이력 집계 기준:

- `workspaceId`: 같은 워크스페이스에서 반복 환불 또는 즉시 revoke가 있었는지 확인한다.
- `customerKey`: 같은 Polar customer가 다른 워크스페이스에서도 반복 환불 또는 즉시 revoke를 만들었는지 확인한다.
- `billingCustomerExternalRef`: Polar customer key가 아직 없거나 바뀐 경우에도 같은 external customer 기준 이력을 본다.
- `billingOwnerUserId`: 같은 Dowin 결제 관리자 계정이 새 워크스페이스로 우회하는 패턴을 본다.

이벤트 처리 기준:

- 전액 `order.refunded`: Basic 권한을 즉시 회수하고 risk count에 포함한다.
- 부분 `order.refunded`: 자동 권한 회수는 하지 않지만 risk count에는 포함한다.
- 즉시 효력 상실 `subscription.revoked`: risk count에 포함한다.
- 기간 말 해지 후 정상 종료로 발생한 revoke/ended는 risk count에 포함하지 않는다.

차단 기준:

- 최근 30일 위험 이벤트 합계가 2회 이상이면 새 Basic checkout을 `BILLING_REVIEW_REQUIRED`로 막는다.
- 차단 판단은 위 4개 축 중 하나라도 임계치를 넘으면 적용한다.
- `REVOKED` 상태인 워크스페이스는 기본적으로 self-serve 재결제를 막고 운영자 수동 검토로 보낸다.
- 정상적인 법정 환불권 행사 자체를 벌점으로 설명하지 않는다. 제품 문구는 `악의적 결제, 사기 의심, 반복 남용 패턴`에 대한 수동 검토와 향후 거래 거부 가능성으로 표현한다.
