# STANDARD 단일 상품 + 내부 멤버 상한 결제 모델 변경안

> 현재 상태 (2026-06-09): 이 문서는 `STANDARD` 단일 상품과 내부 멤버 상한을 다루는 과거 결제 모델이다. 현재 제품 정책은 **Basic-only seat 결제**이며, 신규 구현 기준은 `docs/planning/2026.05.20-billing-entitlement-and-workspace-architecture-plan.md`와 `docs/planning/2026.06.02-polar-seat-count-correction-plan.md`를 따른다. `FREE`/`STANDARD`는 신규 요금제가 아니라 legacy 호환값으로만 본다.

## 1. 배경 및 맥락 (Background & Context)

- 기존 `2026.04.18-polar-billing-mvp-plan.md`에서는 워크스페이스 단위 월정액 `STANDARD`를 첫 결제 MVP로 잡았다.
- 이후 수익성 확보를 위해 참여 인원수 기반 과금과 `STANDARD_10`, `STANDARD_20`, `STANDARD_30` 같은 10명 단위 상품 티어를 검토했다.
- 하지만 seat 기반 자동 과금은 멤버 추가/삭제 시점마다 Polar 구독 수량 동기화, webhook 실패 재시도, 중간 변경 시 proration 정책이 필요하다.
- 반대로 상품 티어를 여러 개 만들면 seat sync는 피할 수 있지만, Polar 상품 생성/가격 변경/상품 ID 환경 변수/checkout 매핑/webhook 검증/테스트 케이스가 티어 수만큼 늘어난다.
- 따라서 MVP에서는 **Polar 상품은 `STANDARD` 하나만 유지하고, Dowin 내부 정책으로 유료 플랜의 멤버 상한을 둔다.**

## 2. 왜 지금 필요한가 (Priority & Rationale)

- 현재 코드와 OpenAPI는 `FREE | STANDARD` 단일 유료 플랜을 강하게 가정하고 있다.
- 결제 MVP의 1차 목적은 매출 최적화가 아니라 `FREE -> STANDARD` 전환, webhook 기반 권한 반영, 환불/해지 시 권한 회수의 정합성 검증이다.
- 다중 상품 티어를 먼저 열면 실제 수요가 검증되기 전에 billing/product 운영 복잡도가 커진다.
- `STANDARD`에 내부 멤버 하드캡을 두면 무제한 사용 리스크는 줄이면서도 DB enum, Orval 타입, 프론트 문구, Polar product mapping 변경을 최소화할 수 있다.

## 3. 결정 (Decision)

- MVP 플랜 코드는 그대로 `FREE`, `STANDARD`만 사용한다.
- `FREE`는 기존처럼 10명 제한을 유지한다.
- `STANDARD`는 단일 월정액 상품으로 운영하되, 내부 멤버 상한을 둔다.
  - 가격: 월 29,000원
  - 멤버 상한: 30명
- `STANDARD` 상한에 도달하면 신규 멤버 초대/참가를 차단하고, 더 큰 팀은 문의 CTA로 넘긴다.
- `STANDARD_10`, `STANDARD_20`, `STANDARD_30` 같은 다중 Polar 상품은 MVP에서 만들지 않는다.

## 4. 범위와 비범위 (Scope / Non-Goals)

### 4.1. MVP 범위 (Goals)

- 단일 Polar 상품: `STANDARD`
- 가격: 월 29,000원
- Polar product ID는 환경 변수가 아니라 DB의 `billing_provider_products` 매핑에서 조회
- 내부 용량 정책:
  - `FREE`: 10명
  - `STANDARD`: 30명까지 허용
- 플랜별 멤버 상한은 코드 상수가 아니라 DB의 `billing_plan_limits`에서 조회
- 멤버 초대/참가 경로에서 플랜별 하드 리밋 검증
- `STANDARD` 상한 도달 시 신규 초대 차단 + 문의 CTA 노출
- 기존 `FREE | STANDARD` API contract와 제품 문구 최대한 유지

### 4.2. MVP 비범위 (Non-Goals)

- 1인당 자동 과금
- Polar subscription quantity 변경
- 다중 Polar 상품 티어
- 티어 간 upgrade/downgrade checkout
- proration 계산 또는 차액 환불
- `STANDARD`보다 큰 팀을 위한 셀프서브 결제

## 5. 정책 상세

### 5.1. 멤버 상한

- `FREE` 상한: 10명
- `STANDARD` 상한: 30명
- `STANDARD` 가격: 월 29,000원
- 상한은 `planCode` 문자열에 묻지 않고, 서버의 플랜 제한 정책에서 계산한다.
- MVP seed 기준은 `billing_plan_limits`에 `FREE=10`, `STANDARD=30`으로 저장한다.
- 이후 더 큰 팀 수요가 확인되면 아래 중 하나로 확장한다.
  - 수동 계약/문의 기반 `BUSINESS`
  - `STANDARD_50` 같은 제한적 다중 상품
  - 실제 seat/usage 기반 billing

### 5.2. 상한 초과 상태

- 기존 멤버와 데이터는 유지한다.
- 신규 멤버 초대와 초대코드 참가만 차단한다.
- 운영 쓰기 전체를 잠글지는 `FREE` 한도 초과 정책과 별도로 결정한다.
  - `FREE` 회수 후 10명 초과 상태는 현재처럼 운영 쓰기 제한을 둘 수 있다.
  - `STANDARD`에서 유료 상한을 초과한 상태는 우선 신규 멤버 유입 차단에 집중한다.

### 5.3. 업셀 동선

- `FREE`가 10명에 도달하면 `STANDARD 업그레이드` CTA를 노출한다.
- `STANDARD`가 유료 상한에 도달하면 `문의하기` CTA를 노출한다.
- MVP에서는 `STANDARD -> 더 높은 셀프서브 티어` checkout을 제공하지 않는다.

### 5.4. 가격 표시 원칙

- 메인 가격 표시는 `월 29,000원`으로 둔다.
- 메인 카피는 seat 기반 과금처럼 보이지 않도록 팀 단위 정액을 강조한다.
  - 예: `월 29,000원으로 30명까지 팀 실행 관리`
- `30명 기준 1인당 약 1,000원`은 보조 설명으로만 사용한다.
- 결제 상품명, billing 화면, 랜딩 pricing 문구는 모두 `STANDARD` 단일 상품으로 맞춘다.
- Polar product ID는 가격 표시 원천으로 사용하지 않고 checkout 생성을 위한 provider 매핑으로만 사용한다.

## 6. 구현 Action Items

1. **플랜 제한 정책 공통화**
   - `FREE_PLAN_MEMBER_LIMIT` 같은 코드 상수 대신 `billing_plan_limits`에서 `member_limit`을 조회한다.
   - `FREE=10`, `STANDARD=30`을 migration seed로 관리한다.
2. **멤버 초대/참가 검증 수정**
   - 현재 `FREE`만 검사하는 로직을 플랜별 상한 검사로 변경한다.
   - 에러 코드는 기존 `WORKSPACE_MEMBER_LIMIT_REACHED`를 재사용할지, 유료 상한 전용 코드를 둘지 결정한다.
3. **워크스페이스 응답 확장 검토**
   - 프론트가 현재 인원/상한/초과 여부를 일관되게 표시할 수 있도록 `memberLimit`, `isOverMemberLimit` 같은 일반화된 필드를 검토한다.
   - 기존 `freeMemberLimit`, `isOverFreeMemberLimit`는 호환 필요성을 보고 유지/대체한다.
4. **UI 문구 수정**
   - `FREE` 한도 도달: `STANDARD 업그레이드`
   - `STANDARD` 한도 도달: `문의하기`
   - pricing 노출: `월 29,000원으로 30명까지 팀 실행 관리`
5. **운영 정책 문서 업데이트**
   - `STANDARD`는 무제한 좌석 상품이 아니라 정해진 멤버 상한 안에서 쓰는 워크스페이스 월정액임을 명확히 적는다.
6. **Polar product 매핑 관리**
   - `billing_provider_products`에 `POLAR`, `sandbox|production`, `STANDARD`, product ID를 저장한다.
   - checkout 요청 body에서 product ID를 받지 않고 서버가 active 매핑을 조회한다.

## 7. Post-MVP 확장 조건

아래 신호가 확인되기 전까지는 다중 Polar 상품을 만들지 않는다.

- 30명 이상 팀의 실제 가입/도입 문의가 반복된다.
- `STANDARD` 상한 때문에 전환이 막히는 사례가 2건 이상 관찰된다.
- 가격/용량별 전환율을 실험할 만큼 결제 유입이 충분하다.
- 상품 ID, checkout 매핑, webhook projection, 고객지원 문구를 티어별로 운영할 여력이 생긴다.

Post-MVP 후보:

- `STANDARD` 단일 상품 유지 + 수동 `BUSINESS` 계약
- `STANDARD_50` 하나만 추가
- `STANDARD_10/30/50` 같은 제한적 tiered pricing
- 실제 seat/usage 기반 billing

## 8. 결론

MVP에서는 `STANDARD_10/20/30` 같은 다중 상품 티어를 만들지 않는다. 단일 `STANDARD` 상품과 내부 멤버 상한으로 시작하면 현재 구현 구조를 크게 흔들지 않으면서도 무제한 사용 리스크를 줄일 수 있다. 실제 큰 팀 수요가 확인된 뒤에만 다중 상품 또는 custom plan으로 확장한다.
