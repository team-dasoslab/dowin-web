# Dowin GA4 로깅 계획

## 1. 배경 및 맥락 (Background & Context)

- 현재 DOWIN의 단기 우선순위는 유료화보다 `범용 서비스화 + 무료 가치 강화`다.
- 이 단계에서 필요한 것은 단순 페이지뷰 집계가 아니라, 사용자가 핵심 여정 어디에서 멈추는지 확인할 수 있는 제품 이벤트 계측이다.
- 특히 현재 로드맵에서 남은 핵심 축은 `설정 → 기록 → 리뷰` 여정 완결과 `무료 가치 측정 지표` 정의다.
- 따라서 분석 도입도 마케팅 리포트 중심이 아니라, 제품 판단과 파일럿 관찰에 직접 쓰이는 구조여야 한다.

## 2. 우선순위 및 당위성 (Priority & Rationale)

- 지금 로깅이 필요한 이유는 아래 판단을 감이 아니라 데이터로 보려는 것이다.
  - 가입 후 실제 세팅까지 도달하는가
  - 첫 기록이 일어나는가
  - 주간 리뷰까지 닫히는가
  - 푸시 알림이 실제 재방문으로 이어지는가
- 기존 무료 가치 로드맵에서 정의한 `활성 사용자`, `재방문`, `주간 리뷰 완료율`은 이벤트 기반 계측 없이는 일관되게 판단하기 어렵다.
- 또한 PWA 푸시 알림은 이미 구현되어 있으므로, 발송 여부만이 아니라 `클릭`과 `랜딩 이후 행동`까지 봐야 제품 가치를 검증할 수 있다.

## 3. 기대 효과 (Impact: Before & After)

- 이전:
  - 기능이 사용되는지 여부를 토스트 반응, 체감, 수동 관찰에 의존한다.
  - 핵심 여정 이탈 지점을 구조적으로 판단하기 어렵다.
  - 푸시 알림의 실효성을 열람 수가 아니라 추측으로만 본다.
- 이후:
  - 핵심 여정 퍼널을 이벤트 기준으로 확인할 수 있다.
  - 회고 기능 도입 전후의 사용 변화도 같은 체계 안에서 비교할 수 있다.
  - 파일럿 팀별 무료 가치 신호를 같은 기준으로 측정할 수 있다.

## 4. 목표와 비목표 (Goals & Non-Goals)

### 4.1. 목표

- GA4를 사용해 핵심 제품 이벤트를 표준화한다.
- `설정 → 기록 → 리뷰` 여정의 퍼널과 완료율을 볼 수 있게 한다.
- PWA 푸시 클릭을 감지하고, 클릭 이후 실제 랜딩과 후속 행동을 함께 측정한다.
- 개인정보를 보내지 않는 안전한 이벤트 구조를 정한다.

### 4.2. 비목표

- 마케팅 어트리뷰션 고도화
- 광고 전환 최적화
- 자유 메모/회고 본문 전문 분석
- 전사 데이터 웨어하우스 구축
- 처음부터 모든 버튼 클릭을 계측하는 과도한 이벤트 남발

## 5. 도구 및 기본 원칙 (Tools & Principles)

### 5.1. 도구 선택

- 1차 도구는 `GA4`로 둔다.
- 서버 운영 로그는 별도 문제로 분리하며, 본 문서는 제품 이벤트 로깅에 집중한다.
- PWA 푸시 클릭은 서비스 워커 `notificationclick` 이벤트를 기준으로 감지한다.

### 5.2. 계측 원칙

- 이벤트는 `핵심 의사결정에 필요한 최소 개수`만 먼저 넣는다.
- 이벤트명은 제품 행위를 기준으로 짧고 안정적으로 유지한다.
- 이벤트 파라미터는 구조화된 메타만 보내고, 자유 텍스트 본문은 보내지 않는다.
- 이메일, 닉네임, 초대코드 원문, 메모 본문 등 PII 또는 민감 텍스트는 전송하지 않는다.
- 내부 식별자는 원본 값 대신 익명화 키 또는 해시를 우선 검토한다.
- 이벤트 스키마는 클라이언트/서비스 워커/서버에서 일관되게 재사용 가능해야 한다.

## 7. 이벤트 정의 (Event Definitions)

### 7.1. 퍼널 중심 이벤트

#### `sign_up_completed`

- 의미:
  - 공개 회원가입이 정상 완료된 시점
- 주요 파라미터:
  - `signup_method`: `self_signup`

#### `workspace_created`

- 의미:
  - 사용자가 새 워크스페이스 생성을 완료한 시점
- 주요 파라미터:
  - `workspace_id_hash`

#### `workspace_joined`

- 의미:
  - 사용자가 초대코드 등으로 워크스페이스 참가를 완료한 시점
- 주요 파라미터:
  - `workspace_id_hash`
  - `join_method`: `invite_code`

#### `scoreboard_created`

- 의미:
  - 활성 점수판 생성 완료
- 주요 파라미터:
  - `workspace_id_hash`
  - `scoreboard_id_hash`
  - `lead_measure_count`

#### `lead_measure_created`

- 의미:
  - 선행지표 생성 완료
- 주요 파라미터:
  - `scoreboard_id_hash`
  - `period_type`: `WEEKLY` | `MONTHLY`

#### `daily_log_checked`

- 의미:
  - 사용자가 특정 날짜의 O/X 체크를 변경해 저장 성공한 시점
- 주요 파라미터:
  - `scoreboard_id_hash`
  - `lead_measure_id_hash`
  - `log_date`
  - `checked_value`: `done` | `undone`
  - `view_type`: `week` | `month`

#### `weekly_review_started`

- 의미:
  - 사용자가 특정 주차 리뷰 작성 흐름을 연 시점
- 주요 파라미터:
  - `week_start`
  - `workspace_id_hash`
  - `has_active_scoreboard`

#### `weekly_review_completed`

- 의미:
  - 특정 주차 리뷰를 완료 상태로 저장한 시점
- 주요 파라미터:
  - `week_start`
  - `workspace_id_hash`
  - `review_status`: `completed`
  - `has_next_focus`: `true` | `false`

### 7.2. 조회/재방문 보조 이벤트

#### `dashboard_my_viewed`

- 의미:
  - 사용자가 내 대시보드 화면을 유효하게 연 시점
- 주요 파라미터:
  - `view_type`: `week` | `month`
  - `has_active_scoreboard`

#### `dashboard_team_viewed`

- 의미:
  - 사용자가 팀 대시보드를 연 시점
- 주요 파라미터:
  - `workspace_id_hash`
  - `member_count_bucket`: `1` | `2_5` | `6_15` | `16_plus`

## 8. PWA 푸시 로깅 계획

### 8.1. 왜 따로 본다

- 푸시 알림은 단순 발송 수보다 `실제 반응`과 `반응 후 행동`이 중요하다.
- 따라서 수신, 클릭, 목표 화면 오픈, 후속 행동을 최소한으로 나눠서 봐야 한다.

### 8.2. 권장 이벤트 흐름

1. 서버 또는 발송 로직에서 실제 사용자 디바이스 수신 여부를 직접 판단하지는 않는다.
2. 서비스 워커가 푸시를 수신해 표시할 때 `push_notification_shown` 또는 내부 로그를 남긴다.
3. 사용자가 알림을 누르면 서비스 워커 `notificationclick`에서 `push_notification_clicked`를 기록한다.
4. 앱이 열리거나 기존 탭이 포커스되면 `push_notification_opened_target`를 기록한다.
5. 이후 같은 세션 안에서 `daily_log_checked` 또는 `weekly_review_started`가 발생하는지 연결해 본다.

### 8.3. 푸시 이벤트 파라미터

- `push_type`
  - 예: `daily_reminder`, `weekly_focus`
- `campaign_id`
  - 발송 종류를 안정적으로 식별하는 내부 키
- `target_path`
  - 예: `/dashboard/my`
- `delivery_channel`
  - `web_push`

### 8.4. 구현 원칙

- 클릭 감지는 서비스 워커에서 처리한다.
- 클릭 직후 GA4를 직접 호출하는 방식보다, 내부 API 또는 열린 클라이언트 페이지를 통해 전달하는 방식을 우선 검토한다.
- 이유:
  - 서비스 워커 환경에서 직접 분석 호출이 누락될 수 있다.
  - 페이지가 아직 열리지 않은 시점에도 클릭 로그를 안정적으로 남겨야 한다.
- 권장 순서:
  - 서비스 워커 `notificationclick`
  - 내부 로깅 API 호출
  - 필요 시 서버에서 GA4 Measurement Protocol 또는 후속 분석 경로로 전달

## 9. 핵심 지표 정의 (Metrics)

### 9.1. 활성화 지표

- 가입 후 워크스페이스 진입율
  - `sign_up_completed` 대비 `workspace_created | workspace_joined`
- 점수판 생성율
  - 워크스페이스 진입 사용자 대비 `scoreboard_created`
- 첫 기록 전환율
  - 점수판 생성 사용자 대비 `daily_log_checked`

### 9.2. 무료 가치 지표

- 주간 리뷰 완료율
  - 활성 사용자 대비 `weekly_review_completed`
- 2주 연속 사용 비율
  - 2개 연속 주차에서 `daily_log_checked` 또는 `weekly_review_completed`가 존재한 사용자 비율
- 푸시 후 행동 전환율
  - `push_notification_clicked` 이후 24시간 내 `daily_log_checked` 발생 비율

### 9.3. 해석 원칙

- 단일 이벤트 수보다 `다음 단계 전환율`을 우선 본다.
- 회고 기능이 붙은 이후에는 `weekly_review_completed`가 핵심 north-star 후보 지표다.
- 초기 파일럿에서는 절대 수치보다 팀별 추세와 이탈 구간 차이를 본다.

## 10. 일일 운영 리포트 확장안 (Daily Discord Report Extension)

### 10.1. 왜 별도 운영 리포트가 필요한가

- GA4 화면만으로는 매일 제품 판단을 하기 어렵다.
- 현재 필요한 것은 분석 툴 접속 자체가 아니라, 운영자가 매일 같은 질문을 같은 형식으로 확인하는 루틴이다.
- 따라서 `GA4 원천 이벤트 -> 일일 요약 리포트 -> Discord 웹훅 발송` 구조를 추가해, 무료 가치와 리텐션 신호를 빠르게 읽는 운영 레이어를 둔다.
- 실행 주체는 앱 Worker 내부 cron이 아니라 `별도 스케줄러 워커`를 기본값으로 둔다.

### 10.2. 일일 리포트가 반드시 답해야 하는 질문

- 어제 신규 사용자는 실제 설정까지 갔는가
- 어제 기록을 시작한 사용자가 늘었는가 줄었는가
- 최근 7일 기준 반복 사용 신호가 좋아지고 있는가
- 푸시 또는 리더 개입이 실제 후속 행동으로 이어졌는가
- 특정 단계에서 비정상적으로 이탈이 커졌는가

### 10.3. Discord 일일 리포트 MVP 섹션

- `Acquisition / Activation`
  - `sign_up_completed`
  - `workspace_created | workspace_joined`
  - `scoreboard_created`
  - 첫 `daily_log_checked`
- `Engagement`
  - DAU 유사치: 어제 `dashboard_my_viewed` 또는 `daily_log_checked`가 발생한 고유 사용자 수
  - 기록 사용자 수
  - 2주 연속 사용 후보 수
- `Retention Signal`
  - 최근 7일 내 2회 이상 기록한 사용자 수
  - 최근 14일 내 2개 이상 서로 다른 날짜에 기록한 사용자 수
- `Intervention`
  - `push_notification_clicked`
  - 클릭 후 24시간 내 `daily_log_checked`
  - 추후 `leader_report_*`, `leader_reminder_*`
- `Alert`
  - 주요 전환율 급락
  - 이벤트 수집량 비정상 하락
  - 발송 실패 또는 리포트 생성 실패

### 10.4. Discord 리포트는 무엇을 보여주지 말아야 하는가

- 원시 이벤트 나열
- PII가 섞인 사용자 목록
- 단일 날짜 절대값만 보여주는 단편 숫자
- 아직 표본이 적은 상태에서의 과도한 인과 해석

- `leader_report_published` ⏳ **보류** (리더 리포트 기능 미개발)
  - 이유: 리포트 발행 실패와 사용자 무반응을 구분해야 한다.
- `leader_report_clicked` ⏳ **보류** (리더 리포트 기능 미개발)
  - 이유: 리포트의 실질적 관심도를 봐야 한다.
- `leader_reminder_sent` ⏳ **보류** (리마인드 기능 미개발)
  - 이유: 개입 시도량 자체를 별도로 집계해야 한다.
- `leader_reminder_followup_converted` ⏳ **보류** (리마인드 기능 미개발)
  - 이유: Basic paid value 가설의 핵심 검증 지표다.

### 11.3. GA4 외에 별도로 남겨야 하는 운영 로그

- Discord 웹훅 발송 성공/실패
- 일일 리포트 생성 시각과 기준 기간
- 보고서 버전(`report_version`)
- 집계 쿼리 기준 시간대(`Asia/Seoul`)
- 알림/리포트별 `campaign_id`

원칙:

- 제품 행동 로그는 GA4에 둔다.
- 운영 파이프라인 성공/실패 로그는 스케줄러 워커 로그 또는 내부 저장 경로에 둔다.
- 둘을 섞으면 장애 원인과 사용자 행동 해석이 모두 흐려진다.

## 12. 데이터 수집 기준 (What To Collect And Why)

### 12.1. 사용자/워크스페이스 단위 식별자

- `user_id_hash`
- `workspace_id_hash`
- `scoreboard_id_hash`
- `lead_measure_id_hash`

왜 필요한가:

- 개인 raw ID 없이도 사용자/팀/점수판 단위 전환을 볼 수 있다.
- Discord 리포트에서 팀 단위 추세와 사용자 재방문 패턴을 안전하게 집계할 수 있다.

### 12.2. 행동 맥락 파라미터

- `view_type`
- `period_type`
- `log_date`
- `week_start`
- `checked_value`
- `join_method`
- `signup_method`
- `has_active_scoreboard`
- `lead_measure_count`

왜 필요한가:

- 단순 이벤트 발생 여부를 넘어서, 어느 화면/주기/상황에서 전환이 일어났는지 읽을 수 있다.
- 예를 들어 `scoreboard_created`는 올랐는데 `lead_measure_count`가 낮다면 설정 품질 문제가 의심된다.

### 12.3. 개입/알림 파라미터

- `push_type`
- `campaign_id`
- `target_path`
- `delivery_channel`
- 추후 `report_type`, `report_version`

왜 필요한가:

- 같은 클릭이라도 어떤 알림/리포트가 실제 기록 재개를 만들었는지 구분할 수 있다.
- 실험군/대조군 비교 시 운영 메시지 차이를 안정적으로 묶을 수 있다.

### 12.4. 실험 해석용 집계값

- `D0 activation`: 가입 당일 `workspace_created|joined -> scoreboard_created -> first daily_log_checked`
- `D1/D7 return`: 다음날, 7일 내 재방문 또는 재기록
- `push followup conversion`: 클릭 후 24시간 내 기록
- `review completion rate`: 리뷰 시작 대비 완료
- `intervention conversion`: 리더 개입 후 24시간/7일 내 기록 재개

왜 필요한가:

- 실험은 이벤트 자체보다 `가설이 기대한 중간 결과`를 봐야 한다.
- 따라서 Discord 리포트도 raw count보다 위 집계값을 우선 노출해야 한다.

## 13. 분석 프레임 (How To Analyze)

### 13.1. 기본 원칙

- 절대 수치보다 `전환율`, `전주 대비 변화`, `7일 이동 평균`을 우선 본다.
- 일일 리포트는 정밀 분석 보고서가 아니라 `이상 징후 탐지 + 후속 질문 생성` 용도다.
- 데이터가 적은 초기 파일럿에서는 유저 수보다 `팀별 패턴 차이`를 같이 본다.

### 13.2. 바로 도출 가능한 유의미한 결과

- `가입은 되는데 점수판 생성이 낮다`
  - 온보딩/설정 UX 문제 가능성
- `점수판 생성은 되는데 첫 기록이 낮다`
  - 설정 직후 행동 유도 부족 가능성
- `푸시 클릭은 높은데 기록 재개가 낮다`
  - 메시지 카피보다 랜딩 후 UX 또는 타이밍 문제 가능성
- `리더 리포트 클릭은 높은데 리마인드/후속 기록이 낮다`
  - 상태 파악은 되지만 개입 동기가 약한 상태
- `기록 사용자는 유지되는데 리뷰 완료율이 낮다`
  - 제품이 기록 도구에는 가깝지만 회고/운영 루프는 아직 닫히지 않음

### 13.3. Discord 리포트에 같이 넣을 해석 규칙

- 전환율이 3일 연속 하락하면 `watch`
- 7일 평균 대비 30% 이상 급락하면 `alert`
- 신규 기능 릴리스 후 핵심 지표가 개선되지 않으면 `hypothesis not supported`
- 클릭은 늘고 후속 행동이 줄면 `message/landing mismatch`

## 14. 가설 검증 루프 운영안 (Experiment Iteration Loop)

### 14.1. 한 번의 가설은 한 단계만 건드린다

- 예: 회원가입 이후 첫 기록 전환 개선
- 예: 푸시 클릭 후 기록 재개율 개선
- 예: 리더 리포트 클릭 후 팀 대시보드 진입율 개선

나쁜 예:

- 카피, 타이밍, UI, 리포트 구조를 한 번에 모두 바꾸는 것

### 14.2. 가설 템플릿

- 가설:
  - `설정 완료 직후 첫 기록 CTA를 강화하면 D0 first daily_log_checked 전환율이 오른다`
- 선행지표:
  - `scoreboard_created -> first daily_log_checked` 전환율
- 보조지표:
  - `dashboard_my_viewed`, 이탈률 추정, 7일 내 재기록
- 실패 기준:
  - 전환 개선 없음 또는 재방문 악화

### 14.3. Discord 리포트 활용 방식

- 매일:
  - 이상치 확인
  - 전날 릴리스/실험 영향 체크
- 매주:
  - 동일 가설의 7일 추세 확인
  - 실험 지속/중단/수정 결정
- 기능 배포 후:
  - 배포 전 7일 vs 배포 후 7일 비교

### 14.4. 운영 규칙

- 리포트에는 `수치`, `변화`, `해석 후보`, `다음 확인 질문`이 함께 있어야 한다.
- 숫자만 보내면 행동으로 이어지지 않는다.
- 해석만 보내면 과신하게 된다.
- 따라서 `What happened / Why it may matter / What to check next` 구조를 유지한다.

## 15. 데이터 안전 및 프라이버시 원칙

- GA4에는 이메일, 닉네임, 자유 메모, 회고 본문을 보내지 않는다.
- URL query에 사용자 식별 가능 데이터가 들어가면 정제 후 전송한다.
- 내부 ID는 해시 또는 분석 전용 익명 키로 치환하는 것을 기본값으로 둔다.
- 이벤트 파라미터는 사전에 허용 목록을 정의하고, 임의 객체 전체를 전송하지 않는다.

## 17. 성공 및 실패 판단 기준 (Success & Failure Criteria)

- 성공:
  - 핵심 퍼널 이벤트가 누락 없이 수집된다.
  - 사용자가 어디에서 멈추는지 퍼널로 설명할 수 있다.
  - 푸시 클릭 이후 실제 기록 행동 여부를 추적할 수 있다.
  - 주간 리뷰 기능 도입 시 완료율을 동일 체계로 바로 비교할 수 있다.
  - Discord 일일 리포트만으로도 전날 운영 이상 징후와 다음 확인 질문을 정리할 수 있다.
- 실패:
  - 버튼 클릭 이벤트만 과도하게 쌓이고 제품 판단에 쓰이지 않는다.
  - PII가 이벤트 파라미터로 섞여 들어간다.
  - 서비스 워커 클릭 로그와 실제 후속 행동이 분리되어 해석이 불가능하다.
  - Discord 리포트가 raw count 나열에 그쳐 실험 판단에 쓰이지 않는다.
