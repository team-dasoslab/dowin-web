# DOWIN Audit Logs MVP Schema Plan

## 1. 배경 및 맥락 (Background & Context)

- 현재 DOWIN는 핵심 비즈니스 액션이 서비스 레이어에서 비교적 명확하게 분리돼 있다.
  - `scoreboard archive/reactivate`
  - `lead measure archive/reactivate/delete`
  - `workspace member remove/leave`
  - `workspace delete`
  - `daily log write/delete`
- 반면, 운영 관점에서 “누가 언제 무엇을 바꿨는지” 남는 공통 로그 테이블은 없다.
- 이 상태에서는 장애 후 아래를 파악하기 어렵다.
  - 특정 삭제가 실수였는지 의도였는지
  - 어떤 엔티티가 언제 사라졌는지
  - 부분 복구 대상이 무엇인지

따라서 지금 필요한 것은 “전 변경 이력 전체 저장”이 아니라, 운영 복구와 사고 추적에 직접 도움이 되는 최소 audit log다.

## 2. MVP 목표 (What This MVP Must Solve)

이번 MVP는 아래만 풀면 된다.

1. 핵심 삭제/상태전이 action 추적
2. actor, entity, 시각, workspace 범위 추적
3. 부분 복구 시 대상 식별 보조
4. 사용자 문의나 장애 postmortem 시 최소 설명 근거 확보

이번 MVP에서 안 푸는 것:

- 모든 필드 변경 full snapshot
- 일반 조회용 활동 피드
- BI/analytics 목적 이벤트 로그 대체
- DB trigger 기반 전 테이블 자동 캡처

## 3. 왜 서비스 레이어 기록이 맞는가

DOWIN는 현재 `route -> service -> storage` 구조가 분명하다.

이 구조에선 service 레이어에서 기록하는 편이 낫다.

이유:

- 비즈니스 의미가 있는 action 이름을 남길 수 있음
- 단일 사용자 요청 안의 다중 SQL을 하나의 action으로 묶을 수 있음
- D1/SQLite trigger에 비해 구현/테스트/이해가 단순함
- “강퇴”, “보관”, “재활성화” 같은 제품 언어를 그대로 남길 수 있음

즉, 이번 MVP는 low-level mutation log보다 business action log가 맞다.

## 4. 제안 테이블 스키마

테이블명: `audit_logs`

### 4.1. 컬럼 제안

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `id` | integer PK | 로그 PK |
| `actor_type` | text | `USER` 또는 `ADMIN` |
| `actor_user_id` | integer nullable | 서비스 사용자 actor |
| `actor_admin_user_id` | integer nullable | 운영자 actor |
| `workspace_id` | integer nullable | 관련 워크스페이스 |
| `entity_type` | text | 대상 엔티티 타입 |
| `entity_id` | integer nullable | 대상 엔티티 PK |
| `action_type` | text | 수행 action |
| `metadata` | text(JSON) nullable | 복구/추적용 보조 정보 |
| `created_at` | timestamp integer | 기록 시각 |

운영 원칙:

- `actor_type = USER`면 `actor_user_id`만 채운다.
- `actor_type = ADMIN`이면 `actor_admin_user_id`만 채운다.
- 둘 다 동시에 채우지 않는다.
- 현재 DOWIN 운영 콘솔은 서비스 사용자와 별도 운영 계정 체계를 전제로 하므로, actor 구분은 선택이 아니라 필수다.

### 4.2. enum 후보

`actor_type`

- `USER`
- `ADMIN`

`entity_type`

- `WORKSPACE`
- `WORKSPACE_MEMBER`
- `WORKSPACE_INVITE`
- `WORKSPACE_TAG`
- `SCOREBOARD`
- `LEAD_MEASURE`
- `DAILY_LOG`
- `TEAM_MEMO`
- `USER`
- `CONTACT_INQUIRY`
- `ADMIN_USER`
- `ADMIN_ROLE_GRANT`

`action_type`

- `CREATE`
- `UPDATE`
- `DELETE`
- `ARCHIVE`
- `REACTIVATE`
- `RESTORE`
- `REMOVE_MEMBER`
- `LEAVE_WORKSPACE`
- `TRANSFER_ADMIN`
- `STATUS_CHANGE`
- `GRANT_ROLE`
- `REVOKE_ROLE`

MVP에선 모든 enum을 처음부터 다 열 필요는 없고, 실제 기록 액션 기준으로 좁게 시작해도 된다.

## 5. metadata에 무엇을 넣을까

`metadata`는 JSON text로 두는 편이 현실적이다.

MVP 원칙:

- 복구와 추적에 직접 필요한 것만 남긴다
- 민감정보 원문은 최대한 넣지 않는다
- full diff 대신 핵심 식별 정보 위주로 둔다

예시:

### scoreboard archive

```json
{
  "userId": 12,
  "previousStatus": "ACTIVE",
  "nextStatus": "ARCHIVED",
  "goalName": "운동 주 5회"
}
```

### contact inquiry status change

```json
{
  "previousStatus": "RECEIVED",
  "nextStatus": "IN_PROGRESS",
  "answerSummaryChanged": false,
  "changeReason": "초기 분류 완료"
}
```

### admin role grant

```json
{
  "targetAdminUserId": 7,
  "role": "SUPPORT_ADMIN",
  "grantReason": "문의 처리 담당자 온보딩"
}
```

### workspace member remove

```json
{
  "membershipId": 44,
  "targetUserId": 18,
  "role": "MEMBER"
}
```

### daily log delete

```json
{
  "leadMeasureId": 31,
  "logDate": "2026-04-19",
  "status": "DONE"
}
```

## 6. MVP에서 반드시 기록할 액션

우선순위는 운영 복구 가치 기준으로 잡는다.

### 1순위

- scoreboard archive
- scoreboard reactivate
- lead measure archive
- lead measure reactivate
- lead measure delete
- workspace member remove
- workspace leave
- workspace delete

### 2순위

- daily log delete
- team memo delete
- workspace tag delete
- contact inquiry status change / answer summary update
- billing manual override
- admin role grant / revoke
- soft-delete restore

### 후순위

- create/update 전체
- 로그인/로그아웃
- 세션 생성/만료

즉, 처음에는 “사라지거나 상태가 크게 바뀌는 액션” 중심으로만 남긴다.

## 7. 구현 원칙

### 7.1. service에서 명시적으로 기록

예:

- `ScoreboardService.archiveScoreboard`
- `LeadMeasureService.deleteLeadMeasure`
- `WorkspaceService.removeMember`

이런 메서드 성공 직후 기록한다.

### 7.2. 비즈니스 액션 성공 후 기록

- 실패한 action까지 기본 audit log에 남기면 노이즈가 커진다
- MVP는 성공한 mutation만 기록

실패 로그가 필요하면 나중에 보안/감사 축에서 별도 검토한다.

### 7.3. 본 작업 트랜잭션과 같이 묶을지

이상적으로는 같은 트랜잭션에 묶는 것이 맞다.  
다만 D1와 현재 구조에서 구현 복잡도가 올라가면, MVP는 아래 순서도 가능하다.

1. 핵심 mutation 성공
2. audit log insert

이 경우 audit log 누락 가능성이 있으므로, 중기 목표는 트랜잭션 일관성 확보로 둔다.

## 8. 인덱스 제안

조회 패턴은 운영 추적 중심일 가능성이 크다.

최소 인덱스:

- `(workspace_id, created_at)`
- `(entity_type, entity_id, created_at)`
- `(actor_user_id, created_at)`
- `(actor_admin_user_id, created_at)`
- `(actor_type, created_at)`

## 9. 예상되는 질의 패턴

예:

- “어제 workspace 3에서 무슨 삭제가 있었나?”
- “scoreboard 12는 누가 archive했나?”
- “user 18이 최근에 어떤 destructive action을 했나?”
- “어떤 운영자가 문의 상태를 해결됨으로 바꿨나?”
- “최근 7일간 운영자 수동 권한 보정이 몇 번 있었나?”

따라서 관리자/운영 기준 조회는 최근순 정렬이 기본일 가능성이 높다.

## 10. DOWIN MVP 제안안

가장 현실적인 1차 도입안:

1. `audit_logs` 테이블 추가
2. storage에 `createAuditLog` 추가
3. service 레이어에서 아래만 기록
   - scoreboard archive/reactivate
   - lead measure archive/reactivate/delete
   - workspace member remove/leave
   - workspace delete
   - contact inquiry status change / answer summary update
   - billing manual override
   - admin role grant / revoke
   - soft-delete restore
4. metadata는 최소 JSON으로 시작
5. 운영 조회는 우선 SQL 직접 조회 또는 간단한 내부 스크립트 수준으로 시작

이렇게 하면 복구와 추적 가치가 크면서도 도입 범위가 과하지 않다.

단, 운영 콘솔(`/admin`)을 실제로 도입하는 시점에는 아래를 함께 맞춰야 한다.

1. `actor_type` + actor ID 구분 반영
2. 운영자 계정 스키마(`admin_users`, `admin_sessions`, `admin_role_grants`)와 연결
3. 문의 처리, billing 수동 보정, 운영 권한 변경, 복구 액션을 운영 audit log 대상에 포함

## 11. 열어둘 질문

- `entity_id`가 text를 필요로 하는 경우가 있는가
- `metadata`에 before/after를 얼마나 넣을 것인가
- `workspace delete`처럼 cascade가 큰 action은 영향 엔티티 개수를 같이 남길 것인가
- 운영 조회 API를 만들지, 당분간 SQL 조회로 버틸지
- 운영자 actor를 기존 user actor와 한 컬럼에 합칠지, `actor_type + actor_id` 또는 이원 컬럼으로 분리할지
  - 현재 운영 콘솔 분리 방향 기준으로는 이원 컬럼 방식을 채택하는 쪽으로 기울어 있다
