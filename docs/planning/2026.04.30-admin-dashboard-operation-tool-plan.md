# DOWIN 어드민 대시보드(운영 도구) 기획안

## 1. 배경 및 맥락 (Background & Context)

- **상황**: DOWIN는 현재 폐쇄형 베타를 지나 공개 회원가입 및 유료 플랜(`STANDARD`) 도입 단계에 있음.
- **문제**: 결제 오류, 데이터 실수 삭제 등 운영 이슈 발생 시 개발자가 매번 CLI(Wrangler)나 DB 도구를 직접 조작해야 함. 이는 대응 속도가 느리고 휴먼 에러의 위험이 큼.
- **해결**: 운영자(또는 개발자)가 제품 UI 상에서 안전하게 상태를 조회하고 필수적인 복구 액션을 수행할 수 있는 전용 도구가 필요함.

## 2. 우선순위 및 당위성 (Priority & Rationale)

- **왜 지금인가?**: 실제 유료 결제가 시작되면 "결제 성공 후 권한 미반영" 이슈는 1순위 긴급 대응 대상임. 또한 사용자가 직접 데이터를 관리하기 시작하면 실수 삭제에 대한 복구 요청이 빈번해짐.
- **추가 배경**: 2026-05-01 기준 서비스 내부 문의하기 MVP가 구현되었고, 사용자는 이미 앱 안에서 자기 문의의 상태와 결과 요약을 확인할 수 있다. 이제 운영자도 같은 데이터를 제품 안에서 처리할 수 있어야 한다.
- **핵심 가치**:
  - **문의 처리 연속성**: Discord 알림에만 머무르지 않고, 제품 DB의 `contact_inquiries`를 기준으로 상태와 결과 요약을 직접 관리할 수 있어야 한다.
  - **결제 신뢰성**: 결제 원장 조회를 통해 사용자 문의에 즉각 답변 가능.
  - **데이터 안전성**: Time Travel과 Audit Log를 결합해 데이터 손실 리스크 최소화.

## 3. 상세 설계 (Detailed Design)

### 3.1. 진입 및 보안 (Auth & Access)

- **도메인 분리**: 운영 콘솔은 서비스 도메인 `dowin.dasoslab.com` 아래 숨김 경로로 두지 않고, 별도 운영 도메인 `dowin-admin.dasoslab.com`으로 분리한다.
- **경로**: 운영 콘솔 내부 라우트는 `dowin-admin.dasoslab.com` 기준 `/`, `/inquiries`, `/billing`, `/recovery` 등으로 구성한다.
- **계정 체계 분리**: 운영자 계정은 서비스 사용자(`users`)와 별도 테이블/세션 체계로 관리한다.
- **네트워크 제어**: 운영 도메인은 지정된 고정 IP 또는 사내/VPN egress IP만 접근 가능하도록 allowlist를 건다.
- **권한**: 운영 도메인 접근은 서비스 세션(`dowin_sid`)이 아니라 별도 운영 세션으로만 허용한다.
- **운영 계정 모델 제안**:
  - `admin_users`
  - `admin_sessions`
  - `admin_role_grants` 또는 이에 준하는 운영 권한 테이블
- **운영 권한 예시**:
  - `SUPPORT_ADMIN`: 문의 처리, billing 조회
  - `RECOVERY_ADMIN`: 복구/복원 액션
  - `BILLING_ADMIN`: billing 조회, 수동 보정
  - `SYSTEM_ADMIN`: 운영 계정/권한 관리 및 최고 권한
- **보안**:
  - 운영 도메인은 앱 레벨 인증 전에도 Cloudflare 또는 동등한 프록시 계층에서 IP allowlist로 1차 차단한다.
  - 운영 세션은 서비스 세션보다 더 짧은 TTL을 사용한다.
  - 운영자 로그인에는 후속 단계에서 MFA, IP 제한, 강제 재인증 같은 보안 강화 수단을 고려한다.
  - 어드민이 수행하는 모든 '쓰기/복구' 액션은 `audit_logs`에 별도 기록 (Actor: Admin ID, Actor Type: `ADMIN`).

### 3.1.1. 운영 계정 스키마 초안

운영 계정은 서비스 사용자 스키마를 재사용하지 않고, 최소 아래 3개 테이블로 시작하는 편이 맞다.

#### `admin_users`

- 목적: 운영 콘솔 로그인 주체를 서비스 사용자와 분리
- 컬럼 초안:
  - `id`
  - `loginId`
    - 운영자 로그인 식별자
    - 서비스 사용자 `customId`와 별개
  - `passwordHash`
  - `displayName`
  - `status`
    - `ACTIVE | SUSPENDED | DISABLED`
  - `mfaEnabled`
  - `lastLoginAt`
  - `createdAt`
  - `updatedAt`

중요:

- 운영자 계정은 서비스 사용자와 1:1 연결을 강제하지 않는다.
- 동일인이 서비스 사용자이기도 할 수 있지만, 연결이 필요하면 후속으로 `linkedUserId` nullable 필드를 검토한다.
- MVP에서는 운영자 계정을 별도 발급/관리하는 방향이 더 안전하다.

#### `admin_sessions`

- 목적: `dowin-admin.dasoslab.com` 전용 인증 세션 관리
- 컬럼 초안:
  - `id`
  - `adminUserId`
  - `sessionTokenHash`
  - `expiresAt`
  - `lastAccessedAt`
  - `ipAddress`
  - `userAgent`
  - `createdAt`

원칙:

- 서비스 세션 쿠키(`dowin_sid`)와 이름/저장소/만료 정책을 공유하지 않는다.
- 쿠키 scope도 서비스 도메인과 분리한다.
- 세션 TTL은 서비스 세션보다 짧게 잡는다.
- 강제 로그아웃이나 비정상 접근 추적을 위해 최근 접근 메타를 남긴다.
- 세션 원문 토큰은 DB에 그대로 저장하지 않고 hash로 저장하는 쪽을 기본안으로 둔다.

#### `admin_role_grants`

- 목적: 운영 계정에 역할을 부여/회수하고 이력을 남김
- 컬럼 초안:
  - `id`
  - `adminUserId`
  - `role`
    - `SUPPORT_ADMIN | BILLING_ADMIN | RECOVERY_ADMIN | SYSTEM_ADMIN`
  - `grantedByAdminUserId`
  - `grantReason`
  - `grantedAt`
  - `revokedAt`
  - `revokedByAdminUserId`
  - `revokeReason`

원칙:

- 운영 권한은 `admin_users`의 단일 enum 컬럼보다 grant 테이블이 낫다.
- 그래야 복수 역할, 회수 이력, 임시 권한 부여, 최소 권한 원칙을 다룰 수 있다.
- 권한 체크는 "revoked 되지 않은 최신 grant 존재 여부" 기준으로 판단한다.

### 3.1.2. 최소 권한 매핑

- `SUPPORT_ADMIN`
  - 문의 리스트/상세 조회
  - 문의 상태 변경
  - `answerSummary` 작성/수정
  - billing 상태 조회
- `BILLING_ADMIN`
  - billing event ledger 조회
  - 수동 권한 보정
  - Polar customer / subscription 식별자 조회
- `RECOVERY_ADMIN`
  - audit log 조회
  - soft-delete 복구
  - 복구 관련 운영 액션 실행
- `SYSTEM_ADMIN`
  - 운영 계정 생성/정지
  - 운영 권한 부여/회수
  - 모든 운영 메뉴 접근 가능

MVP 제안:

- 초기에는 `SUPPORT_ADMIN`, `RECOVERY_ADMIN`, `SYSTEM_ADMIN` 3개만 먼저 연다.
- 이 단계에서 `SUPPORT_ADMIN`은 billing 조회까지 맡고, billing 쓰기 액션(수동 권한 보정)은 `SYSTEM_ADMIN`만 수행한다.
- `BILLING_ADMIN`은 billing 읽기/쓰기 운영량이 실제로 분리 필요해질 때 도입한다.

### 3.1.3. 네트워크 및 세션 정책 초안

#### 운영 도메인 접근 제어

- 운영 콘솔은 `dowin-admin.dasoslab.com` 에만 노출한다.
- `dowin.dasoslab.com` 쪽에는 운영 콘솔 진입 링크나 리다이렉트 경로를 두지 않는다.
- `dowin-admin.dasoslab.com` 는 Cloudflare 또는 동등한 프록시 계층에서 아래 순서로 보호한다.
  - 1차: IP allowlist
  - 2차: 운영자 로그인 세션
  - 3차: 후속 단계에서 MFA 또는 Cloudflare Access 검토

#### IP allowlist 운영 원칙

- 허용 대상은 아래 중 하나로 제한한다.
  - 고정 사무실 IP
  - 운영자 VPN egress IP
  - 긴급 대응용 고정 백업 IP
- 개인 주거지 변동 IP를 직접 allowlist에 계속 추가하는 방식은 운영 피로가 커서 기본 정책으로 삼지 않는다.
- allowlist 변경은 `SYSTEM_ADMIN` 또는 동등 권한자만 수행 가능해야 한다.
- 긴급 장애 대응 시 임시 IP 허용이 필요하면 만료 시각과 사유를 운영 로그에 남긴다.

#### `admin_sessions` 쿠키 정책 초안

- 쿠키 이름은 서비스 세션과 구분되는 전용 이름을 쓴다.
  - 예: `dowin_admin_sid`
- scope:
  - `Domain=dowin-admin.dasoslab.com`
  - 서비스 도메인과 공유하지 않음
- 필수 보안 속성:
  - `HttpOnly`
  - `Secure`
  - `Path=/`
- `SameSite`:
  - 기본은 `Strict` 우선 검토
  - 외부 SSO/Access 연동 플로우가 필요하면 `Lax`로 낮출지 별도 검토
- TTL:
  - 절대 만료는 서비스 세션보다 훨씬 짧게 둔다.
  - 운영 콘솔은 장시간 로그인 유지보다 탈취 시 노출 시간을 줄이는 쪽이 우선이다.
  - 초기 제안은 1시간에서 4시간 범위 안에서 확정한다.
- sliding 재발급:
  - 완전 무제한 연장은 피하고, 마지막 활동 기준 짧은 범위 내에서만 연장한다.
  - 민감 액션 직전에는 sliding 유효 세션이 있더라도 재인증을 요구할 수 있다.
- 민감 액션 보호:
  - 복구, 수동 권한 보정, 운영 권한 부여/회수는 최근 재인증을 요구할 수 있다.

#### 권장 추가 방어선

- 운영자 로그인 rate limit
- 로그인 실패 누적 시 계정 일시 잠금
- 최근 로그인 IP / user agent 이력 보관
- 운영자 세션 강제 만료 기능
- Cloudflare WAF 또는 Access 이벤트와 앱 audit log의 상호 참조 가능성 확보

### 3.1.4. D1 / Drizzle 구현 스키마 초안

현재 `src/db/schema.ts` 패턴을 따르면 다음 기준이 자연스럽다.

- PK는 서비스 `users`/`workspaces`와 같은 `integer` auto increment 기준을 우선한다.
- 세션 토큰 자체는 `text` unique 또는 PK로 둔다.
- 시간 컬럼은 `integer(..., { mode: "timestamp" })`를 사용한다.
- enum 성격 컬럼은 SQLite text + Drizzle enum 제한으로 맞춘다.

#### `admin_users` 제안

- 테이블명: `admin_users`
- 컬럼:
  - `id`: `integer` PK auto increment
  - `login_id`: `text` not null unique
  - `password_hash`: `text` not null
  - `display_name`: `text` not null
  - `status`: `text` enum `ACTIVE | SUSPENDED | DISABLED`, not null, default `ACTIVE`
  - `mfa_enabled`: `integer` boolean, not null, default `false`
  - `last_login_at`: timestamp nullable
  - `created_at`: timestamp not null default now
  - `updated_at`: timestamp not null default now
- 인덱스:
  - `admin_users_login_id_unique`
  - 필요 시 `admin_users_status_idx`

메모:

- `login_id`는 이메일 강제가 아니라 운영용 식별자면 충분하다.
- 이메일을 붙일지 여부는 운영자 초대/비밀번호 재설정 UX가 생길 때 결정해도 늦지 않다.
- `updated_at` 자동 갱신은 애플리케이션 레이어에서 명시적으로 관리한다.

#### `admin_sessions` 제안

- 테이블명: `admin_sessions`
- 컬럼:
  - `id`: `text` PK
    - 세션 row 식별용 랜덤 id
  - `admin_user_id`: `integer` not null, `admin_users.id` FK, onDelete `cascade`
  - `session_token_hash`: `text` not null unique
  - `expires_at`: timestamp not null
  - `created_at`: timestamp not null default now
  - `last_accessed_at`: timestamp not null default now
  - `ip_address`: `text` nullable
  - `user_agent`: `text` nullable
- 인덱스:
  - `admin_sessions_admin_user_idx`
  - `admin_sessions_expires_at_idx`
  - `admin_sessions_token_hash_unique`

메모:

- 운영 세션은 서비스 세션보다 민감하므로, DB에는 원문 토큰 대신 `session_token_hash`를 저장한다.
- 쿠키에는 원문 토큰을 넣고, 서버는 요청마다 hash 후 `session_token_hash`로 조회한다.
- 이 문서 기준 MVP 구현안은 `id 직접 저장`이 아니라 `token hash 저장`으로 확정한다.

#### `admin_role_grants` 제안

- 테이블명: `admin_role_grants`
- 컬럼:
  - `id`: `integer` PK auto increment
  - `admin_user_id`: `integer` not null, `admin_users.id` FK, onDelete `cascade`
  - `role`: `text` enum `SUPPORT_ADMIN | BILLING_ADMIN | RECOVERY_ADMIN | SYSTEM_ADMIN`, not null
  - `granted_by_admin_user_id`: `integer` nullable, `admin_users.id` FK, onDelete `set null`
  - `grant_reason`: `text` nullable
  - `granted_at`: timestamp not null default now
  - `revoked_at`: timestamp nullable
  - `revoked_by_admin_user_id`: `integer` nullable, `admin_users.id` FK, onDelete `set null`
  - `revoke_reason`: `text` nullable
- 인덱스:
  - `admin_role_grants_admin_user_idx`
  - `admin_role_grants_role_idx`
  - `admin_role_grants_active_lookup_idx` on (`admin_user_id`, `role`, `revoked_at`)

메모:

- SQLite partial index를 당장 쓰지 않는다면 `revoked_at is null` 조건은 앱 쿼리에서 처리한다.
- 동일 role의 중복 활성 grant를 막고 싶으면 앱 레벨에서 "active grant already exists" 검증을 먼저 둔다.
- 나중에 임시 권한 만료가 필요하면 `expires_at`을 추가하는 방향이 자연스럽다.

#### 관계 / 삭제 규칙

- `admin_users -> admin_sessions`: cascade
- `admin_users -> admin_role_grants.admin_user_id`: cascade
- `admin_users -> admin_role_grants.granted_by_admin_user_id`: set null
- `admin_users -> admin_role_grants.revoked_by_admin_user_id`: set null

권장 이유:

- 운영 계정 삭제 또는 비활성화 시 세션과 대상 grant는 함께 정리돼야 한다.
- 하지만 "누가 권한을 부여/회수했는가" 기록은 actor 계정 삭제 후에도 row 자체는 남는 편이 낫다.

#### bootstrap 규칙

- 첫 `SYSTEM_ADMIN`은 수동 migration seed 또는 1회성 내부 스크립트로 생성한다.
- 제품 UI 안에서 "첫 운영자 가입" 흐름은 두지 않는다.
- bootstrap 이후의 운영 계정 생성/정지/권한 부여는 운영 콘솔 내부 기능으로 이관한다.
- 로컬 bootstrap 실행 절차는 [Admin User Bootstrap Runbook](/docs/dev/operations/2026.05.01-admin-user-bootstrap-runbook.md) 에 정리한다.

#### 쿼리 기준

- 로그인:
  - `admin_users.login_id` 조회
  - `status = ACTIVE` 확인
  - 비밀번호 검증 후 원문 세션 토큰 생성
  - `session_token_hash`를 계산해 `admin_sessions` insert
- 권한 체크:
  - `admin_role_grants`에서 `admin_user_id`, `role`, `revoked_at is null` 조건 조회
- 세션 검증:
  - 쿠키의 원문 세션 토큰을 hash
  - `admin_sessions.session_token_hash` 기준 조회
  - `expires_at` 확인
  - 필요 시 `last_accessed_at` 갱신

#### MVP에서 아직 미루는 것

- 운영자 비밀번호 재설정 UX
- 운영자 이메일 인증
- 운영자 초대 토큰 플로우
- role grant 만료 시각 자동 소멸
- audit log와 Cloudflare Access 로그 자동 조인

### 3.2. 결제 지원 센터 (Billing Support)

워크스페이스 단위의 결제 현황을 추적하고 문제를 해결한다.

- **조회 기능**:
  - 워크스페이스 ID 또는 이름으로 검색.
  - 현재 플랜(`FREE` / `STANDARD`) 및 상태(`ACTIVE`, `CANCELED`, `REVOKED` 등) 표시.
  - Polar Customer ID 및 Subscription ID 링크 제공.
- **결제 원장(Ledger) 뷰어**:
  - `billing_events` 테이블 데이터를 타임라인 형태로 노출.
  - 웹훅 수신 시각, 이벤트 타입(`subscription.active` 등), 처리 결과(성공/실패 사유) 표시.
- **운영 액션**:
  - **수동 권한 보정 (Manual Override)**: 웹훅 누락 시 관리자가 강제로 `STANDARD` 상태로 갱신하는 기능 (사유 입력 필수).
  - **Customer Portal 링크 생성**: 사용자가 결제 수단 변경 등을 요청할 때 전달할 Polar Portal URL 확인.

### 3.3. 문의 처리 센터 (Contact Support)

서비스 내부 문의하기로 접수된 `contact_inquiries`를 운영자가 조회하고 상태를 갱신한다.

- **조회 기능**:
  - 문의 ID, 제목, 사용자 ID, 워크스페이스 ID, 답변 이메일 기준 검색.
  - 상태(`RECEIVED`, `IN_PROGRESS`, `RESOLVED`) 및 유형(`GENERAL`, `BILLING`, `BUG_OR_ACCOUNT`) 필터.
  - 최신 접수순 기본 정렬, `RECEIVED` 우선 정렬 옵션 제공.
- **문의 리스트 뷰**:
  - 문의 유형, 제목, 현재 상태, 접수 시각, reply email, user/workspace 맥락을 테이블로 노출.
  - billing 문의는 가능하면 관련 워크스페이스 billing 상태 요약으로 빠르게 점프할 수 있어야 함.
- **문의 상세 뷰**:
  - 원문, 접수 시점 메타데이터(locale, source, userId, workspaceId), 현재 `answerSummary`, `answeredAt` 표시.
  - Discord 알림 재확인용 최소 요약 정보 노출. 단, Discord 자체를 진실 원천으로 취급하지는 않음.
- **운영 액션**:
  - 상태 변경: `RECEIVED -> IN_PROGRESS -> RESOLVED`
  - 필요 시 `IN_PROGRESS -> RECEIVED` 되돌리기 허용 여부는 후속 검토. MVP 기본 흐름은 단방향 전이를 우선한다.
  - 사용자에게 노출될 `answerSummary` 입력/수정
  - `RESOLVED` 전환 시 `answerSummary`를 사실상 필수로 간주
  - 모든 변경은 사유 또는 변경 메모와 함께 `audit_logs`에 기록

### 3.4. 데이터 복구 센터 (Recovery Center)

실수 삭제 대응 및 시스템 장애 복구를 보조한다.

- **감사 로그 검색 (Audit Log Search)**:
  - `docs/planning/2026.04.20-audit-logs-mvp-schema-plan.md` 기반.
  - 특정 워크스페이스에서 발생한 `DELETE`, `ARCHIVE` 액션만 필터링해서 조회.
  - 삭제된 엔티티의 ID와 당시 메타데이터(JSON) 확인.
- **소프트 딜리트 복구 (Soft-delete Restore)**:
  - `deleted_at`이 찍힌 엔티티(점수판, 선행지표 등) 리스트 노출.
  - [복구] 버튼 클릭 시 `deleted_at = NULL`로 업데이트.
- **D1 Time Travel 보조**:
  - 현재 복구 가능한 시점(Time Travel Bookmarks) 정보 노출.
  - 대규모 장애 시 "언제로 되돌려야 하는지" 판단 근거 제공 (현재 데이터량 및 마지막 업데이트 시점 확인).

## 5. 기술적 고려사항 (Technical Considerations)

- **Cloudflare D1**: 어드민용 조회 쿼리가 프로덕션 성능에 영향을 주지 않도록 인덱스(`workspace_id`, `created_at`) 최적화 확인.
- **Contact Inquiry Model**: 현재 사용자용 문의 화면이 `status`, `answerSummary`를 그대로 노출하므로, 운영자 도구의 상태 전이 규칙과 입력 검증은 사용자 화면 계약과 어긋나면 안 된다.
- **Auth Boundary**: 운영 콘솔은 서비스 로그인 체계와 인증 쿠키를 공유하지 않는다. 동일 인물이 서비스 사용자이자 운영자일 수는 있어도, 인증 사실은 별도 세션으로 다시 확인해야 한다.
- **Network Boundary**: 운영 콘솔은 경로 기반 숨김이 아니라 별도 서브도메인 + IP allowlist로 1차 노출면을 줄인다. 단, IP 제한만으로 충분하다고 가정하지 않고 운영자 로그인과 감사 로그를 함께 둔다.
- **Schema Boundary**:
  - 서비스 사용자 데이터: `users`, `sessions`, `workspace_members`
  - 운영자 데이터: `admin_users`, `admin_sessions`, `admin_role_grants`
  - 감사 로그 actor는 서비스 사용자와 운영자를 구분할 수 있어야 한다.
- **Deployment Boundary**: 가능하면 운영 콘솔은 서비스 앱과 배포 단위도 분리하거나, 최소한 별도 hostname/routing rule과 환경 변수 세트를 사용한다.
- **Audit Schema Follow-up**: 기존 `audit_logs` MVP 문서는 `actor_user_id` 중심인데, 운영 콘솔을 도입하면 최소 아래 중 하나로 확장해야 한다.
  - `actor_type` + `actor_user_id` + `actor_admin_user_id`
  - 또는 `actor_type` + `actor_id`
  현재 요구를 보면 `actor_type`과 actor ID 분리는 필수다.
- **Polar SDK**: 어드민에서 Polar API를 직접 호출해야 하는 경우(예: 포털 링크 생성) Secret Key 관리 주의.
- **Audit Logging**: 어드민의 액션은 `audit_logs`의 `metadata`에 `isAdminAction: true` 플래그를 포함하여 기록.
  - 문의 상태 변경 시 이전 상태/이후 상태, answerSummary 변경 여부, actor admin ID를 남긴다.
  - billing 수동 권한 보정과 복구 액션도 동일한 감사 로그 규칙을 따른다.

## 7. 성공 및 실패 판단 기준 (Success & Failure Criteria)

- **성공**:
  - 문의 접수 후 운영자가 개발자 도움 없이 1분 내에 문의 내용을 확인하고 상태를 `IN_PROGRESS`로 전환할 수 있다.
  - 운영자가 답변 요약을 저장하면 사용자가 앱 안에서 같은 결과를 다시 확인할 수 있다.
  - 결제 문의 시 개발자 도움 없이 운영자가 1분 내에 상태 파악 가능.
  - 실수 삭제 문의 시 5분 내에 대상 데이터 확인 및 복구 완료.
- **실패**:
  - 문의 상태/답변 처리가 Discord나 SQL 수작업에 계속 의존할 때.
  - 어드민 도구가 있음에도 불구하고 여전히 직접 SQL을 날려야만 해결 가능한 케이스가 대부분일 때.
  - 어드민 액션에 대한 추적 로그가 남지 않아 운영 책임 소재가 불분명해질 때.
