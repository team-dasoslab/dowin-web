# AI 코드 리뷰 인지 부하 완화 및 품질 강제화 방안

AI가 코드를 작성하는 속도가 인간의 리뷰 속도를 압도하면서 발생하는 **"인지 과부하(Cognitive Overload)"**와 **"묻지마 승인(Rubber-Stamping)"** 문제를 해결하기 위해, 사람의 눈에 의존하는 대신 **'기계적인 방어망(Mechanical Net)'**을 구축합니다.

이 문서는 [AI 작업 품질 평가 기준](./2026.07.14-ai-work-evaluation-plan.md)을 기계적이고 시스템적으로 강제하기 위한 3가지 핵심 방법론을 정의합니다.

## 1. 테스트의 함정 극복: 커버리지 대신 "뮤테이션 테스트(Mutation Testing)"

AI는 문법적으로 완벽하고 테스트 커버리지를 100%로 만드는 테스트 코드를 아주 잘 짭니다. 하지만 연구에 따르면 AI가 짠 테스트의 상당수가 **"내용물 없는 껍데기(Tautological test)"**일 확률이 높습니다. (예: 실행만 하고 핵심 비즈니스 로직 검증(Assertion)은 빼먹는 경우)

- **해결책 (Mutation Testing):** 코드를 의도적으로 망가뜨려 봅니다(예: `>`를 `>=`로, `+`를 `-`로 변경). 이때 AI가 작성한 테스트 코드가 실패(에러)를 뿜어내지 않고 통과해 버린다면, 그 테스트는 가짜입니다.
- **실무 적용:** CI 파이프라인에 Stryker 같은 돌연변이 테스트 도구를 연동하여, AI가 짠 로직이 진짜로 튼튼한지 기계가 검증하게 만듭니다.

## 2. 인간의 인지 부하를 줄이는 "구조적 제약 (Structural Habits)"

사람의 작업 기억력(Working Memory)은 한계가 있기 때문에, 컨텍스트가 방대한 코드를 던져주면 "그럴싸해 보이네" 하고 넘어가버리는 Rubber-Stamping(묻지마 승인) 현상이 발생합니다.

- **작업 크기 강제 제한 (Scope Constraint):** AI에게 한 번에 너무 많은 코드를 짜게 하거나, 여러 책임을 가진 클래스를 만들게 하면 안 됩니다. PR(Pull Request) 사이즈가 일정 수준을 넘어가면 아예 리뷰 대상에서 제외하는 하드 리밋(Hard Limit)을 둡니다.
- **설명 강제 (Intent Verification):** AI가 코드를 짤 때 "어떤 코드를 바꿨는지"가 아니라 **"왜 이런 아키텍처적 결정을 내렸는지(Why)"**를 강제로 요약하게 만듭니다. 사람은 코드가 아니라 AI의 **의도(Intent)**를 먼저 리뷰합니다.

## 3. 방어적 파이프라인 구축 (Layered Defense)

연구자들은 AI 코드를 '경력이 없는 주니어'가 짠 코드로 취급하라고 권고합니다. 철저히 **"신뢰하지 말고 검증하라(Verify, Don't Trust)"**는 원칙을 적용해야 합니다.

- **정적 분석과 린트의 극대화:** AI의 코드는 사람이 짠 코드보다 복잡도(Cyclomatic Complexity)가 높은 경우가 많습니다. SonarQube, ESLint의 복잡도 룰을 매우 깐깐하게 설정하여, 복잡한 코드는 아예 CI에서 빌드가 안 되게 막아버립니다.
- **AI-Review-AI (LLM-as-a-Judge) 및 집중 리뷰 영역 안내:** 인간이 코드를 보기 전에, 리뷰 전용 AI가 파이프라인에서 1차로 리뷰 노이즈를 걷어냅니다. 1차 리뷰를 마친 AI는 인간 리뷰어에게 **"핵심 변경 사항 요약"**과 함께 **"중점적으로 봐야 할 파일(Focus List)과 가볍게 넘겨도 될 파일"**을 분류해 주어, 사람의 인지 에너지를 가장 중요한 곳에만 쓰도록 유도합니다.
