# AI 코드 리뷰 인지 부하 완화 및 품질 강제화 방안

AI가 코드를 작성하는 속도가 인간의 리뷰 속도를 압도하면서 발생하는 **"인지 과부하(Cognitive Overload)"**와 **"묻지마 승인(Rubber-Stamping)"** 문제를 해결하기 위해, 사람의 눈에 의존하는 대신 **'기계적인 방어망(Mechanical Net)'**을 구축합니다.

이 문서는 [AI 작업 품질 평가 기준](./2026.07.14-ai-work-evaluation-plan.md)을 현재 Dowin에서 바로 적용 가능한 방식으로 운영하기 위한 원칙을 정의합니다.

## 1. 테스트의 함정 극복: 커버리지보다 의미 있는 검증

AI는 문법적으로 완벽하고 테스트 커버리지를 높이는 테스트 코드를 잘 만들 수 있습니다. 하지만 실행만 하고 핵심 비즈니스 로직 검증(Assertion)을 빼먹는 테스트는 실제 회귀를 막지 못합니다.

- 테스트는 커버리지 수치보다 실패 조건과 핵심 assertion을 우선 확인합니다.
- 새 비즈니스 규칙이나 예외 처리가 생기면 happy path뿐 아니라 fail case를 함께 요구합니다.
- 현재 단계에서는 별도 mutation testing 도구를 도입하지 않고, 리뷰 체크리스트와 기존 Vitest 검증으로 시작합니다.

## 2. 인간의 인지 부하를 줄이는 "구조적 제약 (Structural Habits)"

사람의 작업 기억력(Working Memory)은 한계가 있기 때문에, 컨텍스트가 방대한 코드를 던져주면 "그럴싸해 보이네" 하고 넘어가버리는 Rubber-Stamping(묻지마 승인) 현상이 발생합니다.

- **작업 크기 강제 제한 (Scope Constraint):** AI에게 한 번에 너무 많은 코드를 짜게 하거나, 여러 책임을 가진 클래스를 만들게 하면 안 됩니다. PR(Pull Request) 사이즈가 일정 수준을 넘어가면 아예 리뷰 대상에서 제외하는 하드 리밋(Hard Limit)을 둡니다.
- **설명 강제 (Intent Verification):** AI가 코드를 짤 때 "어떤 코드를 바꿨는지"가 아니라 **"왜 이런 아키텍처적 결정을 내렸는지(Why)"**를 강제로 요약하게 만듭니다. 사람은 코드가 아니라 AI의 **의도(Intent)**를 먼저 리뷰합니다.

## 3. 방어적 파이프라인 운영 (Layered Defense)

AI가 만든 결과는 신뢰를 전제로 merge하지 않고, 현재 저장소에 있는 로컬 검증과 스킬 기반 리뷰 게이트로 확인합니다.

- **기존 로컬 검증 우선:** 작업 범위에 맞춰 `yarn lint`, `yarn tsc --noEmit`, `yarn test:frontend`, `yarn test:backend`, `yarn test --run`을 선택적으로 실행합니다.
- **리뷰 게이트 분리:** 구현 변경은 `dowin-quality-check`, 보안 경계 변경은 `dowin-security-check`, 하네스 변경은 `dowin-harness-security-check`로 분리합니다.
- **집중 리뷰 영역 안내:** 작업 완료 보고에는 핵심 변경 사항과 `focus_list`를 포함해 사람이 봐야 할 파일과 가볍게 넘겨도 되는 파일을 구분합니다.
- **외부 리뷰 API 보류:** 현재 단계에서는 외부 LLM/API 기반 PR 코드 리뷰나 자동 inline comment bot을 도입하지 않습니다.
