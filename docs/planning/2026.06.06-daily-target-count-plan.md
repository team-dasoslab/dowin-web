# `하루 n회` 액션 아이템 기획안

## 1. 배경 및 맥락 (Background & Context)

- 현재 Dowin의 액션 아이템은 사용자가 `주 n회` 또는 `월 n회` 목표를 설정하고, 날짜별 기록판에서 O/X에 가까운 Boolean 기록을 남기는 구조다.
- 이 구조는 `주 3회 운동`, `월 12회 콘텐츠 작성`처럼 "며칠에 걸쳐 몇 번 했는가"를 보기에는 충분하다.
- 하지만 `하루 물 4잔`, `하루 영업 DM 10건`, `하루 스트레칭 3세트`처럼 하루 안에서 여러 번 수행해야 하는 행동은 현재 O/X 기록만으로 표현할 수 없다.
- DB와 도메인 enum에는 이미 `DAILY` period가 존재하지만, `하루 n회`를 곧바로 `매일 해야 하는 목표`로 해석하면 사용자가 원하는 범위를 좁힐 수 있다.
- 예를 들어 `월수금마다 3개`, `주 3일만 하루 10개`, `월 8일만 하루 2콜` 같은 목표는 "매일"이 아니라 "하는 날에는 n개"에 가깝다.
- 더 중요한 제약은 `daily_logs`가 `(lead_measure_id, log_date)`별 Boolean 1개만 저장한다는 점이다. 따라서 하루 안의 수행 횟수를 표현하려면 period 추가보다 날짜별 count 기록 방식이 먼저 필요하다.

## 2. 우선순위 및 당위성 (Priority & Rationale)

- `하루 n회`는 개인 루틴과 팀 운영 행동 모두에 자연스럽게 등장하는 기록 방식이다.
- 이 기능은 목표 종류를 늘리는 문제가 아니라, 액션 아이템의 실행 단위를 더 정확하게 표현하게 하는 기본 모델 확장이다.
- 다만 기존 `주 n회` 기록 경험을 복잡하게 만들면 핵심 점수판 UX가 무거워진다.
- 따라서 MVP는 `DAILY` period를 사용자에게 새로 노출하기보다, 기존 `WEEKLY/MONTHLY` 기간 목표에 count형 기록 방식을 얹는 방향이 적절하다.

## 3. 제품 결정 (Product Decision)

액션 아이템은 `기간 목표`와 `기록 방식`을 분리해서 해석한다.

| 개념 | 필드 초안 | 의미 |
| --- | --- | --- |
| 기간 목표 | `period` + `targetValue` | 주/월 안에서 며칠을 달성해야 하는가 |
| 기록 방식 | `trackingMode` | 날짜 셀이 O/X인지 count 입력인지 |
| 하는 날의 목표량 | `dailyTargetCount` | 그날을 달성 처리하기 위한 최소 count |

예시:

| 사용자 목표 | period | targetValue | trackingMode | dailyTargetCount | 집계 의미 |
| --- | --- | ---: | --- | ---: | --- |
| 주 3회 운동 | `WEEKLY` | 3 | `BOOLEAN` | 1 | O 기록 3일이면 달성 |
| 주 3일, 하루 DM 10개 | `WEEKLY` | 3 | `COUNT` | 10 | count가 10 이상인 날짜 3일이면 달성 |
| 매일 물 4잔 | `WEEKLY` | 7 | `COUNT` | 4 | count가 4 이상인 날짜 7일이면 달성 |
| 월 8일, 하루 2콜 | `MONTHLY` | 8 | `COUNT` | 2 | count가 2 이상인 날짜 8일이면 달성 |

핵심 원칙:

- `period`는 집계 기간을 의미한다.
- `trackingMode`는 날짜별 입력 방식을 의미한다.
- `targetValue`는 기간 안에 달성해야 하는 날짜 수다.
- `dailyTargetCount`는 한 날짜가 달성으로 인정되기 위한 최소 수행 횟수다.
- count가 `dailyTargetCount` 이상인 날짜만 기간 집계에 `+1`로 반영한다.
- 하루에 여러 번 많이 했더라도 기간 집계는 해당 날짜 `+1`까지만 인정한다.
- 요일 지정은 MVP에서 필수로 요구하지 않는다. 사용자는 주/월 안에서 자유롭게 채울 수 있다.

## 4. MVP 범위 (MVP Scope)

### 4.1. Setup

- 액션 아이템 주기 선택은 우선 `매주 / 매월`을 유지한다.
- 기록 방식 선택을 추가한다.
  - `완료형`: 하는 날 O/X
  - `횟수형`: 하는 날 n개
- `완료형` 선택 시 기존처럼 `주 n회`, `월 n회`로 표시한다.
- `횟수형` 선택 시 `주 n일, 하는 날마다 m회` 또는 `월 n일, 하는 날마다 m회`로 표시한다.
- 기본 생성값은 기존 사용자 경험을 유지하기 위해 `WEEKLY 3회`로 둔다.
- `dailyTargetCount`의 최대값은 MVP에서 보수적으로 제한한다.
  - 초안: 최대 20회
  - 이유: stepper UI와 점수판 셀 가독성을 유지하기 위함

### 4.2. Daily Log 입력

- `trackingMode=COUNT` 액션 아이템의 날짜 셀은 Boolean 체크 버튼이 아니라 count 입력 컨트롤로 표시한다.
- 기본 UI는 작은 stepper를 사용한다.
  - `-`
  - `현재 count / dailyTargetCount`
  - `+`
- count가 0이면 미기록 상태로 해석한다.
- count가 `dailyTargetCount` 이상이면 해당 날짜는 달성으로 해석한다.
- count가 target을 초과할 수 있는지는 MVP에서 제한한다.
  - 초안: `dailyTargetCount`까지만 clamp
  - 이유: 점수판은 성과량 총합보다 목표 달성 여부를 빠르게 확인하는 표면이기 때문

### 4.3. 주간 보기

- `trackingMode=COUNT` 액션 아이템도 주간 점수판에 함께 노출한다.
- 각 날짜 셀에는 `count/dailyTargetCount`를 표시한다.
- 주간 달성 수는 `count >= dailyTargetCount`를 만족한 날짜의 수로 계산한다.
- 주간 목표 총량은 `targetValue`로 계산한다.
- 주간 achievement rate는 `achievedDays / targetValue` 기준으로 계산한다.

### 4.4. 월간 보기

- `trackingMode=COUNT` 액션 아이템은 월간 보기에서도 노출한다.
- 월간 목표 총량은 `targetValue`로 계산한다.
- 월간 달성 수는 `count >= dailyTargetCount`를 만족한 날짜의 수로 계산한다.
- 월간 achievement rate는 `achievedDays / targetValue` 기준으로 계산한다.

### 4.5. API 및 데이터 모델

- `daily_logs`는 Boolean `value`만으로는 하루 안의 count를 표현할 수 없으므로 count 저장 필드가 필요하다.
- 후보:
  - `count integer not null default 1` 추가
  - 또는 `value`를 유지하고 `count`를 nullable로 추가
- MVP에서는 기존 O/X 기록과의 호환을 위해 `value`를 유지하고 `count`를 추가하는 방향을 우선 검토한다.
- `lead_measures`에는 기록 방식과 하는 날의 목표량을 표현하는 필드가 필요하다.
  - `tracking_mode`: `BOOLEAN | COUNT`
  - `daily_target_count`: integer, 기본값 1
- API 입력은 기존 `{ value: true }`와 신규 `{ count: number }`를 `trackingMode`에 따라 분리한다.
- `COUNT` 지표는 count 입력만 허용하고, `BOOLEAN` 지표는 기존 Boolean 입력을 유지한다.

## 5. API 계약 설계안 (API Contract)

### 5.1. 공통 원칙

- 기존 daily log 쓰기 경로는 유지한다.
  - `PUT /workspaces/{workspaceId}/lead-measures/{leadMeasureId}/logs/{date}`
  - `DELETE /workspaces/{workspaceId}/lead-measures/{leadMeasureId}/logs/{date}`
- `PUT`은 기록이 있는 상태를 만든다.
- `DELETE`는 기록이 없는 상태로 되돌린다.
- `COUNT` 지표의 `0회`는 별도 row로 저장하지 않고 `DELETE`로 표현한다.
- `BOOLEAN` 지표는 기존 `{ value: true }` 계약을 유지한다.
- `COUNT` 지표는 신규 `{ count: number }` 계약만 허용한다.
- tracking mode와 body 형태가 맞지 않으면 `422 VALIDATION_ERROR`를 반환한다.

### 5.2. Daily Log 쓰기 요청

OpenAPI 컴포넌트 초안:

```yaml
DailyLogUpsertRequest:
  oneOf:
    - $ref: "#/components/schemas/DailyLogBooleanUpsertRequest"
    - $ref: "#/components/schemas/DailyLogCountUpsertRequest"

DailyLogBooleanUpsertRequest:
  type: object
  required: [value]
  additionalProperties: false
  properties:
    value:
      type: boolean
      enum: [true]

DailyLogCountUpsertRequest:
  type: object
  required: [count]
  additionalProperties: false
  properties:
    count:
      type: integer
      minimum: 1
      maximum: 20
```

의미:

- `{ value: true }`
  - `trackingMode=BOOLEAN` 액션 아이템 전용
  - 기존 O 기록 생성/수정
  - 서버 저장 시 `value=true`, `count=1`로 정규화
- `{ count: 1..20 }`
  - `trackingMode=COUNT` 액션 아이템 전용
  - 해당 날짜 수행 횟수 저장
  - 서버 저장 시 `value=true`, `count=count`로 정규화
- `count=0`
  - `PUT`으로 받지 않는다
  - 클라이언트는 `DELETE`를 호출한다

### 5.3. Daily Log 응답

기존 `DailyLog` 응답은 count와 achieved를 추가한다.

```yaml
DailyLog:
  type: object
  required: [id, leadMeasureId, logDate, value, count, achieved, createdAt]
  properties:
    id: { type: integer }
    leadMeasureId: { type: integer }
    logDate: { type: string, format: date }
    value: { type: boolean }
    count: { type: integer, minimum: 1 }
    achieved: { type: boolean }
    createdAt: { type: string, format: date-time }
```

의미:

- `value`
  - 기존 O/X 호환 필드
  - 저장 row가 있으면 항상 `true`
- `count`
  - `COUNT`는 실제 수행 횟수
  - `BOOLEAN`은 항상 `1`
- `achieved`
  - `COUNT`: `count >= dailyTargetCount`
  - `BOOLEAN`: `value === true`

### 5.4. 주간/월간 로그 값 타입

기존 weekly/monthly/team 응답의 `logs`는 `boolean | null` 맵이다.
count를 섞으면 boolean과 number가 같은 필드에 섞이므로, MVP에서는 명시 객체로 확장한다.

```yaml
DailyLogCell:
  type: object
  nullable: true
  required: [value, count, achieved]
  properties:
    value: { type: boolean }
    count: { type: integer, minimum: 0 }
    achieved: { type: boolean }
```

`WeeklyLogItem.logs`, `MonthlyLogItem.logs`, `TeamDashboardMemberMeasure.logs`는 아래처럼 바꾼다.

```yaml
logs:
  type: object
  additionalProperties:
    $ref: "#/components/schemas/DailyLogCell"
```

함께 바로잡을 기존 계약 drift:

- `WeeklyLogItem`에 `period: { type: string, enum: [DAILY, WEEKLY, MONTHLY] }`를 추가한다.
- `WeeklyLogItem`, `MonthlyLogItem`, `TeamDashboardMemberMeasure`에 `trackingMode`, `dailyTargetCount`를 추가한다.
- 기존 DB enum과의 호환 때문에 `DAILY` period는 계약에 남길 수 있지만, MVP 사용자 노출에서는 `WEEKLY/MONTHLY + trackingMode=COUNT`를 우선한다.
- `MonthlyLogItem.period` enum은 구현 호환상 `[DAILY, WEEKLY, MONTHLY]`를 유지하거나, 구현 정리 시 `[WEEKLY, MONTHLY]`로 축소한다.
- `TeamDashboardMemberMeasure.period`는 이미 `[DAILY, WEEKLY, MONTHLY]`를 허용하므로 유지한다.

응답 예시:

```json
{
  "id": 10,
  "name": "물 마시기",
  "period": "WEEKLY",
  "targetValue": 7,
  "trackingMode": "COUNT",
  "dailyTargetCount": 4,
  "logs": {
    "2026-06-01": { "value": true, "count": 3, "achieved": false },
    "2026-06-02": { "value": true, "count": 4, "achieved": true },
    "2026-06-03": null
  },
  "achieved": 1,
  "total": 7,
  "achievementRate": 14.3
}
```

기존 `WEEKLY/MONTHLY`도 같은 응답 구조를 쓴다.

```json
{
  "period": "WEEKLY",
  "targetValue": 3,
  "trackingMode": "BOOLEAN",
  "dailyTargetCount": 1,
  "logs": {
    "2026-06-01": { "value": true, "count": 1, "achieved": true },
    "2026-06-02": null
  }
}
```

### 5.5. 집계 필드

`WeeklyLogItem`, `MonthlyLogItem`, `TeamDashboardMemberMeasure`는 `total`을 명시적으로 추가한다.

```yaml
achieved: { type: integer }
total: { type: integer }
achievementRate: { type: number }
```

tracking mode별 의미:

- `BOOLEAN` weekly
  - `achieved = O 기록 수`
  - `total = targetValue`
- `BOOLEAN` monthly, period가 `WEEKLY`인 경우
  - `achieved = 주차별 min(O 기록 수, targetValue)의 합`
  - `total = targetValue * monthWeekCount`
- `BOOLEAN` monthly, period가 `MONTHLY`인 경우
  - `achieved = O 기록 수`
  - `total = targetValue`
- `COUNT` weekly
  - `achieved = count >= dailyTargetCount인 날짜 수`
  - `total = targetValue`
- `COUNT` monthly
  - `achieved = count >= dailyTargetCount인 날짜 수`
  - `total = targetValue`

### 5.6. Analytics export 계약

`AnalyticsExportDailyRow`는 count 기반 지표를 표현할 수 있도록 count 필드를 추가한다.

```yaml
AnalyticsExportDailyRow:
  type: object
  required: [date, leadMeasureId, leadMeasureName, period, status, count]
  properties:
    date: { type: string, format: date }
    leadMeasureId: { type: integer }
    leadMeasureName: { type: string }
    period: { type: string, enum: [DAILY, WEEKLY, MONTHLY] }
    trackingMode: { type: string, enum: [BOOLEAN, COUNT] }
    dailyTargetCount: { type: integer, minimum: 1, maximum: 20 }
    status: { type: string, enum: [ACHIEVED, MISSED, NOT_RECORDED] }
    count: { type: integer, minimum: 0 }
```

의미:

- `COUNT`
  - `count`: 해당 날짜 수행 횟수
  - `ACHIEVED`: `count >= dailyTargetCount`
  - `MISSED`: `0 < count < dailyTargetCount`
  - `NOT_RECORDED`: row 없음
- `BOOLEAN`
  - `count`: O 기록이 있으면 `1`, 없으면 `0`
  - `ACHIEVED`: O 기록 있음
  - `NOT_RECORDED`: row 없음
  - `MISSED`: 사용하지 않음

### 5.7. 검증 및 에러 규칙

| 조건                                          | 응답                          |
| --------------------------------------------- | ----------------------------- |
| `COUNT` 지표에 `{ value: true }` 입력         | `422 VALIDATION_ERROR`        |
| `BOOLEAN` 지표에 `{ count: n }` 입력          | `422 VALIDATION_ERROR`        |
| `count < 1`                                   | `422 VALIDATION_ERROR`        |
| `count > DAILY_TARGET_COUNT_MAX`              | `422 VALIDATION_ERROR`        |
| `dailyTargetCount > DAILY_TARGET_COUNT_MAX`   | `422 VALIDATION_ERROR`        |
| 보관된 액션 아이템에 기록                     | 기존처럼 `403`                |
| 지난주 이전 날짜 수정                         | 기존처럼 `403`                |
| 접근 권한 없는 workspace/scoreboard/measure   | 기존 auth/ownership 규칙 유지 |

## 6. MVP 제외 범위 (Non-Goals)

- 하루 안의 수행 시각 기록
- 수행 단위 커스터마이징
  - 예: 잔, 분, 세트, 건
- 요일 지정
  - 예: 월수금만 추천/강조/리마인드
- 하루에 목표치를 초과한 수행량을 기간 목표 여러 날로 환산하는 기능
- 목표 초과분을 보너스 점수로 반영하는 기능
- count 변경 히스토리나 undo stack
- 팀원이 하루 중 언제 몇 번 했는지 보는 상세 감사 로그
- AI 기반 목표 횟수 추천

## 7. 사용자 경험 원칙 (UX Principles)

- 기존 O/X 점수판의 빠른 기록 경험을 해치지 않는다.
- `COUNT` 셀만 count UI로 바꾸고, `BOOLEAN`은 기존 체크 경험을 유지한다.
- 사용자가 `주 n일`과 `하는 날마다 n회`의 차이를 setup 단계에서 바로 이해해야 한다.
- count UI는 모바일에서 오입력이 적어야 한다.
- 목표 초과분을 과시하기보다 "오늘 목표를 채웠는가"를 빠르게 읽게 한다.

## 9. 성공 및 실패 기준 (Success & Failure Criteria)

### 성공

- 사용자가 setup에서 `주 n일`과 `하는 날마다 n회`의 차이를 자연스럽게 이해한다.
- count형 지표는 하루 안에서 0회부터 하는 날 목표 횟수까지 기록할 수 있다.
- 기존 완료형 O/X 기록 경험은 변하지 않는다.
- 주간/월간 달성률이 `count >= dailyTargetCount`인 날짜 수 기준으로 계산된다.
- 팀 대시보드와 export에서 count가 Boolean 기록처럼 잘못 축약되지 않는다.

### 실패

- count형 지표인데 날짜별로 0/1만 기록된다.
- `WEEKLY` 사용자가 하루에 여러 번 눌러 주간 목표를 채울 수 있다고 오해한다.
- count UI가 점수판을 과하게 복잡하게 만들어 빠른 기록성이 떨어진다.
- 개인 대시보드와 팀 대시보드의 달성률 계산 기준이 달라진다.
- 기존 로그 데이터의 O/X 의미가 마이그레이션 이후 불명확해진다.
