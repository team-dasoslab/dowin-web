# GitHub PR & Action Item Integration Plan

## Background and Context

Dowin 사용자는 개발 업무의 완료 신호를 GitHub PR merge에서 만들 때가 많다.
이 기능은 PR title, PR body, 또는 PR에 포함된 commit message에 액션 아이템 코드를 적으면, PR merge 시 Dowin의 해당 액션 아이템 daily log를 자동으로 기록하고 어떤 PR이 기록을 만들었는지 내 홈에서 확인할 수 있게 한다.

이전 설계는 GitHub App installation을 Dowin workspace에 직접 연결하고 모든 액션 아이템 코드를 `#DOW-1`처럼 고정 prefix로 처리했다.
하지만 이 방식은 사용자가 workspace마다 GitHub App을 반복 설치해야 하고, 계정 단위로 GitHub를 쓰는 사용자 경험과 맞지 않는다.

따라서 새 설계는 아래 원칙으로 바꾼다.

- GitHub App installation은 Dowin 사용자 계정 단위로 저장한다.
- GitHub repository는 Dowin workspace에 연결한다.
- 같은 GitHub repository는 활성 상태로 하나의 Dowin workspace에만 연결한다.
- 액션 아이템 코드는 workspace별 prefix와 workspace별 sequence로 만든다.
- GitHub 매칭 문법은 `#ABC-1`처럼 `#`가 있는 token만 허용한다.

## Product Decision

MVP의 핵심 결정은 `계정 단위 GitHub 설치 + repository 단위 workspace 연결`이다.

사용자 관점:

1. 사용자는 GitHub App을 자기 GitHub 계정 또는 organization에 한 번 설치한다.
2. Dowin 각 workspace 설정에서 연결할 repository를 선택한다.
3. 액션 아이템에는 `#ABC-1` 같은 코드가 표시된다.
4. 사용자는 PR title/body/commit message에 해당 코드를 넣는다.
5. PR이 merge되면 Dowin이 해당 날짜 daily log를 자동으로 기록한다.

시스템 관점:

1. Webhook의 `installation.id + repository.id`로 repository row를 찾는다.
2. repository row의 활성 workspace mapping을 찾는다.
3. 그 workspace의 `actionItemPrefix + displaySequence` 기준으로 액션 아이템을 찾는다.
4. PR link를 upsert한다.
5. PR이 merge된 경우 daily log를 idempotent하게 반영한다.

## Why Not Workspace-Level Installation

워크스페이스 단위 installation은 구현은 단순하지만 UX 비용이 크다.

- 같은 GitHub 계정을 여러 Dowin workspace에서 쓰려면 매번 설치 flow를 반복해야 한다.
- GitHub App 설치가 실제로는 GitHub account/org 자원인데 Dowin workspace 자원처럼 보인다.
- 나중에 계정 단위 설치 재사용으로 바꾸려면 migration과 UI 의미 변경이 더 커진다.

그래서 MVP부터 installation과 workspace mapping을 분리한다.

## Identifier Decision

### Display Format

액션 아이템 코드는 아래 형식으로 표시하고 매칭한다.

```text
#ABC-1
```

- `#`: 필수.
- `ABC`: workspace의 action item prefix.
- `1`: workspace 안에서 증가하는 action item sequence.

`ABC-1`처럼 `#`가 없는 문자열은 매칭하지 않는다.

### Prefix Rules

workspace는 action item prefix를 가진다.

- 컬럼명: `workspaces.actionItemPrefix`
- SQL 컬럼명: `action_item_prefix`
- 허용 문자: `A-Z`, `0-9`
- 길이: 3자
- 기본 생성: workspace 생성 시 workspace 이름에서 영문/숫자를 추출해 3자 prefix를 제안한다.
- 이름에서 만들 수 없으면 `DOW`를 기본 후보로 사용한다.
- 충돌 시 `D01`, `D02` 같은 fallback을 사용한다.

MVP에서는 prefix를 Dowin 전체에서 unique로 강제하지 않는다.
이유는 GitHub 매칭 전에 repository가 workspace를 먼저 결정하기 때문이다.

다만 같은 workspace 안에서는 prefix가 하나만 존재하고, 액션 아이템 sequence는 workspace 안에서 unique해야 한다.

### Why Prefix Collision Is Acceptable

아래 상태는 허용한다.

```text
workspace A: prefix APP, action item #APP-1
workspace B: prefix APP, action item #APP-1
workspace C: prefix APP, action item #APP-1
```

이것이 안전한 이유는 webhook 처리 시 `#APP-1`만으로 액션 아이템을 찾지 않기 때문이다.
항상 먼저 GitHub repository mapping으로 workspace를 확정하고, 그 workspace 안에서만 `#APP-1`을 해석한다.

### The Required Safety Constraint

아래 상태는 MVP에서 금지한다.

```text
repo X -> workspace A
repo X -> workspace B
```

같은 repo를 여러 workspace에 동시에 연결하면 `#APP-1`이 어느 workspace의 액션 아이템인지 모호해질 수 있다.
따라서 `github_workspace_repository_links`에는 활성 repository mapping이 repository당 하나만 존재해야 한다.

Post-MVP에서 같은 repo를 여러 workspace에 연결하려면 workspace별 전역 unique prefix 또는 더 긴 식별자 문법을 다시 도입해야 한다.

## Scope

### MVP

- workspace에 3자 action item prefix 추가.
- 기존 workspace prefix 백필.
- action item public ID 발급 체계 추가.
- GitHub App installation을 Dowin user 계정 단위로 저장.
- GitHub installation repository 목록을 동기화.
- workspace별 repository 연결/해제 API 추가.
- 같은 repository의 활성 workspace 연결은 하나만 허용.
- PR title/body/PR commit message에서 `#ABC-1` 형식 코드 추출.
- PR merge 시 daily log 자동 기록.
- 내 홈(`dashboard/my`)에 action item code와 연결된 PR 링크 표시.
- workspace 설정에서 GitHub 계정 연결 상태와 repository 연결 상태 표시.

### Out Of Scope

- 같은 GitHub repository를 여러 workspace에 동시에 연결하는 1:N mapping.
- workspace prefix를 사용자가 직접 변경하는 UI.
- GitHub issue, branch, standalone push event 연동.
- PR comment 자동 작성.
- GitHub repository access 변경 webhook 기반 자동 reconciliation.
- Jira, Linear 등 다른 외부 도구 연동.
- 기존 PR link의 완전한 reconciliation.

## Data Model

### `workspaces` 변경

| 컬럼               | 설명                                          |
| ------------------ | --------------------------------------------- |
| `actionItemPrefix` | action item code prefix. 3자 대문자 영문/숫자 |

제약:

- `NOT NULL`
- format 검증은 application validation에서 수행한다.
- DB unique는 MVP에서 걸지 않는다.

백필:

- 기존 workspace마다 prefix를 생성한다.
- 이름에서 영문/숫자만 추출해 대문자 3자로 만든다.
- 3자 미만이면 `DOW`, `D01`, `D02` 순서로 보완한다.
- MVP에서는 전역 unique가 필수는 아니지만, 사람이 보기 좋은 기본값을 위해 가능한 한 중복을 피한다.

### `githubUserInstallations`

GitHub App installation을 Dowin 사용자 계정 단위로 저장한다.

| 컬럼             | 설명                                  |
| ---------------- | ------------------------------------- |
| `id`             | PK                                    |
| `userId`         | Dowin 사용자                          |
| `installationId` | GitHub App installation ID            |
| `accountLogin`   | GitHub user/org login snapshot        |

| `createdAt`      | 생성 시각                             |
| `updatedAt`      | 수정 시각                             |

제약:

### `workspaceGithubRepositoryLinks`

Dowin workspace와 GitHub repository의 활성 연결을 저장한다.

| 컬럼                | 설명                         |
| ------------------- | ---------------------------- |
| `id`                | PK                           |
| `workspaceId`       | Dowin workspace              |
| `repositoryId`      | 내부 `githubRepositories.id` |
| `connectedByUserId` | 연결한 Dowin 사용자          |
| `status`            | `ACTIVE`, `DISCONNECTED`     |
| `createdAt`         | 생성 시각                    |
| `updatedAt`         | 수정 시각                    |

제약:

- `workspaceId -> workspaces.id` cascade
- `repositoryId -> githubRepositories.id` cascade
- `connectedByUserId -> users.id` set null
- `(workspaceId, repositoryId)` unique
- 활성 repository mapping은 repository당 하나만 허용한다.

D1 partial unique index 제약을 단순하게 가져가기 위해 MVP 구현은 아래 중 하나로 고정한다.

1. `repositoryId` unique를 걸고, 연결 해제 후 재연결은 기존 row를 `ACTIVE`로 되살린다.
2. history 보존이 필요하면 `activeRepositoryLocks(repositoryId unique, workspaceRepositoryLinkId)` 보조 테이블을 둔다.

MVP 기본안은 1번이다.
연결 해제 이력보다 충돌 방지가 더 중요하다.

## Backend Flow

### GitHub Account Connect Flow

1. 사용자가 workspace 설정에서 GitHub 연결 버튼을 누른다.
2. 백엔드는 Dowin 세션과 workspace ADMIN 권한을 확인한다.
3. 백엔드는 `githubInstallationStates`에 `state`를 만든다.
4. 프론트는 `https://github.com/apps/{GITHUB_APP_SLUG}/installations/new?state={state}`로 이동한다.
5. GitHub가 setup URL로 `installation_id`, `setup_action`, `state`를 전달한다.
6. setup route는 state를 확인하고 installation id를 임시 저장한다.
7. setup route는 GitHub OAuth authorize URL로 redirect한다.
8. OAuth callback은 `code`를 user access token으로 교환한다.
9. user token으로 `GET /user/installations`를 호출해 installation 접근 권한을 검증한다.
10. App JWT로 installation과 repository 목록을 재조회한다.
11. `githubUserInstallations`, `githubRepositories`를 upsert한다.
12. state를 `COMPLETED`로 바꾸고 workspace GitHub 설정 화면으로 redirect한다.

이 시점에는 GitHub 계정/App 설치가 연결된 것이며, repository가 workspace에 자동 연결되는 것은 아니다.
다만 사용자가 설치 과정에서 선택한 repository 목록은 workspace 설정 화면에서 바로 선택할 수 있게 보여준다.

### Workspace Repository Connect Flow

1. 사용자가 GitHub 설정 화면에서 repository를 선택한다.
2. 백엔드는 workspace ADMIN 권한을 확인한다.
3. 해당 repository가 현재 사용자에게 연결된 GitHub installation의 repository인지 확인한다.
4. repository가 다른 workspace에 ACTIVE로 연결되어 있으면 실패한다.
5. `workspaceGithubRepositoryLinks`를 ACTIVE로 upsert한다.

### Workspace Repository Disconnect Flow

1. 사용자가 workspace에서 repository 연결을 해제한다.
2. `workspaceGithubRepositoryLinks.status`를 `DISCONNECTED`로 변경한다.
3. GitHub App installation 자체는 제거하지 않는다.
4. 기존 `githubPrLinks`는 보존한다.

### Webhook Handling Flow

1. raw body를 읽고 `X-Hub-Signature-256`을 검증한다.
2. `X-GitHub-Delivery` 기준으로 멱등성을 확인한다.
3. `pull_request` 이벤트만 처리한다.
4. payload에서 `installation.id`, `repository.id`, PR 메타데이터를 검증한다.
5. `installation.id + repository.id`로 내부 repository를 찾는다.
6. repository의 ACTIVE workspace link를 찾는다.
7. PR title/body에서 `#([A-Z0-9]{3})-(\d+)` 후보를 추출한다.
8. PR commit 목록을 조회하고 commit message에서도 같은 후보를 추출한다.
9. 후보 prefix가 workspace의 `actionItemPrefix`와 정확히 같지 않으면 무시한다.
10. 후보 sequence가 workspace의 `actionItemPublicIds.displaySequence`와 일치하면 액션 아이템을 찾는다.
11. PR link를 upsert한다.
12. PR이 `MERGED`이면 merge 날짜 KST 기준 daily log를 반영한다.

## Daily Log Rules

- daily log 자동 기록은 PR이 `MERGED`일 때만 수행한다.
- 날짜는 `pull_request.merged_at`을 KST `YYYY-MM-DD`로 변환한다.
- `merged_at`이 없으면 webhook 수신 시각의 KST 날짜를 fallback으로 사용한다.
- `BOOLEAN` 액션 아이템은 `value=true`, `count=1`로 upsert한다.
- `COUNT` 액션 아이템은 해당 날짜 count를 `+1` 한다.
- 같은 PR이 이미 daily log에 반영됐다면 다시 반영하지 않는다.
- 같은 날짜에 서로 다른 PR이 같은 `COUNT` 액션 아이템으로 merge되면 count는 PR 수만큼 증가한다.

## API Contract Draft

### GitHub Account

- `GET /api/integrations/github`
  - 현재 사용자의 GitHub App installation 목록과 repository 목록 조회.
- `POST /api/integrations/github/install-url`
  - GitHub App 설치 URL 생성.
  - request: `{ workspaceId: string, locale: "ko" | "en" }`
- `POST /api/integrations/github/disconnect`
  - Dowin 내부에서 특정 installation을 DISCONNECTED 처리.

### Workspace Repository Mapping

- `GET /api/workspaces/{workspaceId}/integrations/github`
  - workspace의 GitHub repository 연결 상태 조회.
  - 사용자가 가진 GitHub repository 중 연결 가능한 목록도 함께 반환한다.
- `POST /api/workspaces/{workspaceId}/integrations/github/repositories`
  - workspace에 repository 연결.
  - request: `{ repositoryId: number }`
- `DELETE /api/workspaces/{workspaceId}/integrations/github/repositories/{repositoryLinkId}`
  - workspace repository 연결 해제.

### GitHub Callback/Webhook

- `GET /api/integrations/github/setup`
- `GET /api/integrations/github/oauth/callback`
- `POST /api/webhooks/github`

필수 환경 변수:

- `GITHUB_APP_SLUG`

- `GITHUB_APP_PRIVATE_KEY`
- `GITHUB_APP_CLIENT_ID`
- `GITHUB_APP_CLIENT_SECRET`
- `GITHUB_WEBHOOK_SECRET`
- `APP_BASE_URL`

GitHub App 설정:

- Homepage URL: Dowin production URL
- Setup URL: `https://{APP_BASE_URL}/api/integrations/github/setup`
- Callback URL: `https://{APP_BASE_URL}/api/integrations/github/oauth/callback`
- Webhook URL: `https://{APP_BASE_URL}/api/webhooks/github`
- Webhook: active
- Webhook secret: `GITHUB_WEBHOOK_SECRET`
- Request user authorization during installation: off
- Redirect on update: off for MVP
- Expire user authorization tokens: on
- Device flow: off
- Repository permissions:
  - Metadata: Read-only
  - Pull requests: Read-only
- Subscribe to events:
  - Pull request
- Visibility:
  - production 외부 사용자 대상이면 public
  - 내부 검증 단계이면 private 가능

## Backfill and Migration Plan

### Migration Principles

- schema 변경은 `src/db/schema.ts`를 먼저 수정한다.
- migration은 `yarn mig:local`로 생성/적용한다.
- generated migration SQL에 백필 SQL을 손으로 끼워 넣지 않는다.
- 기존 데이터 백필은 service/script로 별도 실행한다.

### Workspace Prefix Backfill

- 모든 기존 workspace에 3자 `actionItemPrefix`를 부여한다.
- dry-run에서 workspace별 기존 이름과 생성될 prefix를 출력한다.
- execute는 `--execute` flag가 있을 때만 수행한다.

### Action Item Public ID Backfill

- 기존 action item은 workspace별로 `createdAt asc, id asc` 정렬한다.
- workspace별 1부터 `displaySequence`를 부여한다.
- 이미 public ID가 있으면 건너뛴다.
- 완료 후 `(workspaceId, displaySequence)` 중복과 `leadMeasureId` 중복을 검증한다.

## Frontend Scope

- workspace 설정에 GitHub integration 진입점을 추가한다.
- GitHub 설정 화면은 두 층을 보여준다.
  - GitHub 계정/App installation 연결 상태.
  - 현재 workspace에 연결된 repository 목록과 연결 가능한 repository 목록.
- repository 연결/해제는 workspace ADMIN만 수행한다.
- 내 홈(`dashboard/my`) 액션 아이템 행에 `#ABC-1` 코드와 PR 링크를 표시한다.
- 팀 대시보드에는 MVP에서 PR 링크를 표시하지 않는다.
- 사용자 문구는 “PR 상태 표시”가 아니라 “PR merge 시 daily log 자동 기록”을 기준으로 쓴다.

## Success Criteria

- 사용자는 GitHub App을 한 번 설치하고 여러 workspace에서 repository를 선택할 수 있다.
- 같은 repository는 활성 상태로 하나의 workspace에만 연결된다.
- `#ABC-1`은 repository가 연결된 workspace 안에서만 해석된다.
- 다른 workspace에 같은 `#ABC-1`이 있어도 오연결되지 않는다.
- PR title/body/commit message에 코드가 있으면 PR link가 생성된다.
- PR merge 시 daily log가 자동 기록된다.
- `COUNT` 액션 아이템은 merge된 PR 수만큼 count가 증가한다.
- 같은 PR 재전송은 daily log count를 중복 증가시키지 않는다.
- GitHub token/private key/raw payload는 DB나 로그에 저장하지 않는다.

## Failure Criteria

- GitHub App installation을 workspace에 직접 귀속시킨다.
- `#ABC-1`만 보고 workspace를 찾는다.
- 같은 repository를 여러 workspace에 ACTIVE로 연결할 수 있다.
- `ABC-1`처럼 `#` 없는 문자열을 매칭한다.
- prefix가 workspace prefix와 다른 코드를 매칭한다.
- PR merge 후 daily log를 기록하지 않고 PR 링크만 저장한다.
- setup callback의 `installation_id`를 GitHub API 검증 없이 신뢰한다.
- generated migration SQL에 백필 SQL을 손으로 추가한다.
- OpenAPI/Orval 갱신 없이 frontend에서 임시 타입으로 구현한다.

## Resolved Decisions

- **Workspace Prefix**: MVP에서는 자동 생성만 지원하고 사용자 편집은 막는다. UI/UX 복잡도와 prefix 충돌 처리 로직을 줄이기 위함이다.
- **Private Repo Exposure**: GitHub App 설치 시 사용자가 명시적으로 권한을 허용한 repository이므로, public/private 상관없이 전체 이름(owner/name)을 그대로 노출한다.
- **App Disconnect Behavior**: Dowin 내부 상태만 `DISCONNECTED`로 변경하되, 프론트엔드에서 GitHub 자체 uninstall 안내 링크(GitHub 세팅 페이지)를 함께 제공하여 사용자가 원할 경우 앱 삭제를 쉽게 할 수 있도록 돕는다.
