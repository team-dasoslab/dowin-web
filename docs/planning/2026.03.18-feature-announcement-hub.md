# Dowin 기능 공지 허브 기획

## 1. 배경 및 맥락 (Background & Context)

- 지금의 DOWIN는 점수판, 대시보드, 프로필 등 핵심 기능이 빠르게 늘고 있지만, 사용자는 "무엇이 새로 생겼는지"를 서비스 안에서 자연스럽게 알기 어렵다.
- 새 기능이 추가되어도 사용자가 직접 눌러보지 않으면 존재를 놓치기 쉽고, 이미 만든 기능의 체감 가치는 실제 완성도보다 낮게 인식될 수 있다.
- 특히 DOWIN는 매일 자주 들어오는 서비스이므로, 새 기능을 알리는 방식도 외부 공지보다 인앱 경험 안에서 매끄럽게 이어져야 한다.
- 단순 텍스트 공지보다 "우리 이런 기능 만들었어, 바로 써봐"라는 제품 톤에 맞는 카드형 안내와 업데이트 모아보기 허브가 필요하다.

## 2. 우선순위 및 당위성 (Priority & Rationale)

- 왜 지금 해야 하는가:
  - 최근 `dashboard/my` 확장, export, avatar, confetti 등 사용자 체감이 큰 기능 후보가 계속 쌓이고 있다.
  - 기능 출시 속도 대비 안내 채널이 없으면, 개발 투자 대비 사용 전환율이 낮아진다.
- 왜 다른 것이 아닌 이것을 먼저 해야 하는가:
  - 푸시 알림이나 이메일보다 인앱 공지는 구현 범위가 작고, 제품 경험을 해치지 않으면서도 가장 직접적으로 기능 발견을 유도할 수 있다.
  - 대시보드는 사용자가 가장 자주 들어오는 화면이므로, 여기에 예쁜 공지 카드와 업데이트 허브 진입점을 두면 재방문 동선과 잘 맞는다.
- 4DX 관점의 당위:
  - 점수판은 "보이는 것이 행동을 만든다"는 전제를 갖는다.
  - 새 기능 안내도 같은 원리로, 사용자가 가치 있는 변화를 즉시 인지하고 행동으로 이어지게 해야 한다.

## 3. 기대 효과 (Impact: Before & After)

- 이전:
  - 새 기능이 배포되어도 사용자는 뒤늦게 알거나 아예 모른다.
  - "기능은 있는데 안 쓰는 상태"가 반복된다.
  - 제품이 계속 좋아지고 있다는 감각이 약하다.
- 이후:
  - 사용자는 로그인 후 자연스럽게 최신 기능을 본다.
  - 공지 카드에서 바로 해당 기능 화면으로 이동해 사용을 시작할 수 있다.
  - 별도 허브에서 최근 추가 기능을 한 번에 훑어보며 DOWIN의 발전 흐름을 이해할 수 있다.
  - 제품이 살아 움직이고 있다는 인상이 강화된다.

## 4. 구체적 실행 방안 (Action Items)

### 4.1. 문제 정의

- DOWIN에는 현재 "출시된 기능을 사용자에게 다시 소개하는 제품 레이어"가 없다.
- 토스트는 일회성이고, 푸시 알림은 권한과 타이밍 제약이 크며, 외부 공지 채널은 서비스 안의 전환으로 바로 이어지지 않는다.
- 따라서 다음 2개를 묶은 단일 기능으로 정의한다.
  - 대시보드 내 인앱 공지 카드
  - 전체 업데이트를 모아보는 "새 기능 모아보기" 허브

### 4.2. MVP 범위

- 포함:
  - 보호 영역 진입 후 가장 최근의 주요 업데이트 1건을 카드형으로 노출
  - 카드에 제목, 한 줄 설명, 강조 비주얼, CTA 버튼 제공
  - 최신순 업데이트 목록을 볼 수 있는 전용 페이지 제공
  - 각 업데이트 항목에 기능 요약, 출시일, 태그, CTA 링크 제공
  - `publishedAt` 기준 최근 14일 이내 항목에 `NEW` 강조 적용
- 제외:
  - 브라우저 푸시/이메일 발송
  - 사용자 세그먼트별 맞춤 노출
  - 실험(A/B test) 기반 문구 최적화
  - 관리자용 웹 CMS
  - 댓글, 반응, 좋아요 같은 소셜 피드화
  - 사용자별 읽음 상태 저장

### 4.3. 사용자 경험 설계

#### A. 인앱 공지 카드

- 노출 위치:
  - 1순위는 `dashboard/my` 상단 요약 영역 아래
  - 필요 시 팀 대시보드 또는 프로필에도 재사용 가능하되, MVP는 `dashboard/my` 우선
- 카드 구성:
  - eyebrow: `NEW FEATURE`, `IMPROVEMENT`, `JUST SHIPPED` 중 하나
  - title: 기능명
  - body: 왜 좋은지 중심의 짧은 설명
  - CTA: `바로 써보기`로 통일
  - secondary action: 우측 상단 `닫기(X)` 버튼
- 노출 원칙:
  - 한 번에 1장만 보여준다.
  - `isMajor === true`인 항목 중 가장 최근 1건만 기본 노출한다.
  - 사용자가 닫으면 같은 기기/브라우저에서 같은 `id`의 업데이트는 계속 숨긴다.
  - 새 major 업데이트가 나오면 이전 dismiss 상태와 무관하게 다시 노출한다.

#### B. 새 기능 모아보기 허브

- 페이지 목적:
  - 최신 변경사항을 제품 안에서 스크롤 기반으로 빠르게 확인
  - 놓친 기능을 다시 발견
  - CTA를 통해 실제 기능 화면으로 이동
- 기본 명칭:
  - 프로필 메뉴명: `새 기능 모아보기`
  - 페이지 타이틀: `새 기능 모아보기`
  - 공지 카드 보조 링크: `모든 새 기능 보기`
- 목록 구성:
  - 최신순 정렬
  - 각 아이템에 태그(`Dashboard`, `Profile`, `Setup`, `Quality of Life` 등) 노출
  - 최근 14일 이내 항목에만 `NEW` 배지 제공
- 진입 경로:
  - 공지 카드 하단 링크
  - 헤더/프로필 메뉴의 `새 기능 모아보기`
  - 릴리스 직후 토스트에서 보조 링크로 재사용 가능

#### C. 메뉴 IA 제안

- 프로필 화면은 현재 `계정 설정`, `데이터`, `알림 설정`, `앱 설치 가이드`, `세션`처럼 기능성 메뉴를 그룹으로 보여주는 구조다.
- 따라서 `공지사항`이라는 별도 운영 섹션보다, 제품 탐색 성격의 메뉴인 `새 기능 모아보기`를 독립 메뉴 아이템으로 추가하는 편이 자연스럽다.
- 권장 위치:
  - `데이터` 아래에 새 그룹을 만들기보다
  - `앱 정보` 또는 `둘러보기` 성격의 소규모 그룹을 새로 두고 그 안에 배치
- 추천 그룹 예시:
  - `앱 둘러보기`
  - `새 기능 모아보기`
  - `iPhone 앱 설치 가이드`
- 이유:
  - 사용자는 프로필에서 "내 설정"과 "앱 자체 정보"를 분리해서 인식하는 편이 자연스럽다.
  - `CSV 다운로드`와 같은 데이터 기능 옆에 두면 성격이 섞여 보인다.

### 4.4. MVP 제품 원칙

- 제품 공지이지 운영 공지 게시판이 아니다.
- 긴 문장보다 "무엇이 좋아졌는지"와 "어디로 가면 되는지"를 빠르게 전달해야 한다.
- 공지 카드의 목적은 읽는 것보다 클릭 후 사용 전환이다.
- 허브의 목적은 아카이브보다 탐색과 재발견이다.
- 대시보드의 4DX 집중감을 해치지 않도록 카드 수와 시각 밀도를 제한한다.

### 4.5. 콘텐츠 운영 방식

- MVP에서는 별도 CMS 없이 레포 안의 구조화된 TS 데이터 파일로 업데이트 항목을 관리한다.
- 구현 형태:
  - `src/content/product-updates.ts`를 단일 진실 원천으로 사용한다.
  - 항목 추가 시 객체 하나를 템플릿에 맞춰 복사해 채우는 방식으로 운영한다.
  - 사람이 직접 수정하든 LLM이 수정하든 같은 필드 구조를 따르도록 한다.
- 각 항목은 최소 아래 정보를 가진다.
  - `id`
  - `slug`
  - `title`
  - `summary`
  - `publishedAt`
  - `ctaLabel`
  - `ctaHref`
  - `isMajor`
  - `tag`
- 장점:
  - 배포와 함께 관리 가능
  - 디자인과 카피를 코드 리뷰 안에서 함께 품질 관리 가능
  - 템플릿 구조가 고정되어 있어 LLM에게 "새 항목 1개 추가"를 시키기 쉽다
- 단점:
  - 비개발자 단독 운영은 어렵다.
- 결론:
  - 현재 팀 규모와 운영 방식상 MVP로 가장 현실적이다.

### 4.6. 최신 기준 정의

- `최신`은 영구 분류가 아니라 강조 노출 상태로 정의한다.
- 기본 규칙:
  - `publishedAt` 기준 최근 14일 이내면 `NEW`
  - 14일이 지나면 `NEW` 배지는 제거되지만 `/updates` 목록에는 계속 남긴다
- 대시보드 카드 규칙:
  - `isMajor === true`인 항목 중 가장 최근 1건을 노출
  - 읽음 상태는 두지 않는다.
  - dismiss는 localStorage 기반으로 같은 기기/브라우저에서 같은 `updateId`에만 적용한다.
- 운영 유의:
  - 업데이트 기록은 계속 누적한다
  - 강조 기간만 14일로 제한한다

### 4.7. 정보 구조 제안

- 신규 페이지:
  - `/updates`
- 상세 페이지 여부:
  - MVP에서는 목록형만 우선 제공
  - 설명량이 늘어나면 `/updates/[slug]` 상세 페이지를 Post-MVP로 확장
- CTA 링크 대상 예시:
  - `dashboard/my` 기간 탐색
  - export 패널
  - profile avatar 설정
  - setup 개선 영역

### 4.7.a. `/updates` 페이지 실제 구성안

- 상단 헤더:
  - 제목: `새 기능 모아보기`
  - 보조 문구: `최근 추가되거나 좋아진 기능을 한 번에 확인해보세요`
- 최상단 pinned 영역:
  - 가장 최근 `major` 업데이트 1건을 넓은 hero card로 노출
  - 강한 제목, 한 줄 요약, 대표 CTA, 출시일 표시
- 그 아래 목록 영역:
  - 최신순 세로 카드 리스트
  - 카드 필드: 태그, 제목, 요약, 출시일, CTA
- 상태 배지:
  - 최근 14일 이내: `NEW`
  - 14일 초과: 일반 항목으로 유지
- 빈 상태:
  - 아직 등록된 업데이트가 없으면 `새 기능이 추가되면 여기에서 가장 먼저 알려드릴게요`
- 모바일 기준:
  - 한 줄 요약은 2~3줄 이내
  - CTA는 카드 하단 고정이 아니라 본문 아래 자연 배치
  - 필터/탭은 MVP에서 제외하고 단순 스크롤 우선

### 4.7.b. 대시보드 카드 실제 구성안

- 위치:
  - `dashboard/my`의 Dowin 카드 아래, 주간/월간 전환 영역 위
- 이유:
  - 점수판의 주 정보는 먼저 보여주되, 탐색 UI보다 앞에서 새 기능 진입을 유도할 수 있다.
- 카드 레이아웃:
  - 좌측: 작은 아이콘 또는 accent block
  - 본문: eyebrow, 제목, 요약
  - 우측 또는 하단: `바로 써보기` 버튼
  - 보조 링크: `모든 새 기능 보기`
  - 우측 상단: `X` dismiss 버튼
- 노출 규칙:
  - `isMajor === true`인 항목 중 최신 1건만 노출
  - minor 업데이트는 `/updates` 허브에서만 노출해도 된다

### 4.7.c. 카드 카피 가이드 예시

- title 예시:
  - `내 기록을 CSV로 내려받을 수 있어요`
  - `프로필 아이콘을 직접 골라보세요`
  - `이제 지난 주와 지난 달도 바로 볼 수 있어요`
- summary 예시:
  - `회고나 공유가 필요할 때, 선택한 기간의 데이터를 바로 저장할 수 있습니다.`
  - `준비된 아바타 중 하나를 선택해 팀 대시보드에서 더 쉽게 구분할 수 있습니다.`
- CTA 예시:
  - `바로 써보기`
  - `설정하러 가기`
  - `자세히 보기`

### 4.8. 우선 구현 시나리오

1. 사용자가 `dashboard/my`에 들어온다.
2. 최신 major 업데이트가 있으면 공지 카드가 보인다.
3. 사용자가 `바로 써보기`를 누른다.
4. 대상 기능 화면으로 이동한다.
5. 사용자는 필요 시 `새 기능 모아보기`로 이동해 최근 릴리스를 더 본다.

### 4.9. MVP 이후 확장 (Post-MVP)

- 사용자 역할/도메인별 맞춤 노출
- `이번 달 바뀐 점` 요약 배치
- release note 상세 페이지
- 스크린샷/애니메이션/GIF 첨부
- 푸시 알림 또는 이메일과 인앱 공지 연동
- 실험 기반 카드 문구/CTA 최적화
- "다음에 볼 기능" 북마크

- [ ] **[문서]** 온보딩 및 관련 도메인 문서에 업데이트 허브 위치 반영
- [ ] **[검증]** 모바일에서 카드 높이, CTA 터치 영역, 허브 스크롤 경험 점검

## 5. 예상 비용 및 기간 (Cost & Estimation)

- 기획/디자인 초안: 0.5~1일
- 프론트엔드 MVP 구현: 1~2일
- 총합:
  - 정적 콘텐츠 기반 MVP: 약 1.5~2.5일

## 6. 성공 및 실패 판단 기준 (Success & Failure Criteria)

- 성공:
  - 새 기능 출시 후 사용자가 첫 방문에서 변경사항을 인지한다.
  - 공지 카드 CTR이 유의미하게 발생한다.
  - 허브 페이지를 통해 과거 릴리스의 재발견이 가능하다.
  - 사용자가 "기능이 생긴 줄 몰랐다"는 상황이 줄어든다.
- 실패:
  - 공지 카드가 대시보드 집중을 방해하는 배너처럼 느껴진다.
  - 허브가 단순 아카이브로만 남고 실제 기능 사용 전환으로 이어지지 않는다.

## 7. 도메인 영향 범위

- `dashboard`
  - 공지 카드의 첫 노출 지면
- `notification`
  - MVP에서는 직접 발송보다 인앱 공지 중심이지만, Post-MVP 연동 후보
- `profile`
  - 메뉴 진입점 또는 설정 내 "새 기능 모아보기" 링크 재사용 가능
- `analytics`
  - 추후 CTR, 기능 전환율 측정 대상

## 8. 권장 구현 방향 요약

- 이번 기능은 운영 공지 게시판이 아니라 제품 내 release communication layer로 정의한다.
- MVP는 "정적 TS 콘텐츠 + 대시보드 공지 카드 + `/updates` 허브" 조합으로 간다.
- 업데이트 기록은 파일 기반으로 정적으로 관리한다.
- 사용자별 읽음 상태는 MVP에서 도입하지 않는다.
- 대신 피로도 방지를 위해 대시보드 카드에 한해 `localStorage` 기반 dismiss를 허용한다.
- `NEW` 강조 기준은 `publishedAt` 기준 14일로 고정한다.
- 구현 우선순위는 아래 순서를 권장한다.
  1. 정적 업데이트 데이터 모델 정의
  2. `NEW 14일` helper 정의
  3. `dashboard/my` 공지 카드 구현
  4. `/updates` 허브 구현

## 9. 오픈 질문

- `major` 업데이트의 기준을 기능 추가만으로 볼지, UX 개선도 포함할지 결정이 필요하다.
- 허브를 로그인 사용자만 볼지, 추후 마케팅 랜딩처럼 외부 공개할지 아직 미정이다.
