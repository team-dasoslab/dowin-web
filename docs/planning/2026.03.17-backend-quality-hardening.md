# Dowin 백엔드 품질 보강 및 게이트 강화 계획

## 1. 배경 및 맥락 (Background & Context)

- 현재 Dowin 백엔드는 `route -> service -> storage` 구조와 도메인별 테스트 배치가 비교적 잘 잡혀 있다.
- 반면, 새로 합류한 개발자가 바로 신뢰하기 어려운 지점도 남아 있다.
- 대표적으로 아래 문제가 확인된다.
  - 일부 이전 작업 가이드 참조가 실제 문서 경로와 어긋난다.
  - 인증/비즈니스 에러 처리 일부가 문자열 비교에 의존한다.
  - `403`과 `404`의 사용 기준이 도메인별로 명확히 문서화되어 있지 않다.
  - 날짜/주간 경계 계산이 서비스 내부 로컬 시간 계산에 강하게 결합되어 있다.
  - 저장소 전역 `yarn tsc --noEmit`, `yarn lint`가 현재 기본 품질 게이트로 안정적으로 동작하지 않는다.
- 이 상태는 "구조는 괜찮지만, 품질 게이트가 팀의 신뢰 기반이 되기엔 아직 느슨한 상태"로 볼 수 있다.

## 2. 우선순위 및 당위성 (Priority & Rationale)

- 지금 이 작업이 필요한 이유는 기능 추가 속도보다 회귀 방지 비용이 더 중요해지는 구간에 들어왔기 때문이다.
- 백엔드 개발자가 늘어나거나 작업 빈도가 올라가면, 문서와 규약의 불일치, 느슨한 에러 체계, 깨진 전역 검증은 바로 생산성 저하와 회귀로 이어진다.
- 특히 DOWIN의 핵심 가치는 기록 데이터의 신뢰도와 점수판 규칙의 일관성에 있다.
  - `ACTIVE` 점수판 단일성
  - 미래 날짜 기록 금지
  - 워크스페이스/소유권 접근 제어
  - 주간/월간 집계 정확성
- 따라서 이번 계획은 기능 확장이 아니라 "앞으로 기능을 안전하게 늘리기 위한 기반 정비"에 해당한다.

## 3. 기대 효과 (Impact: Before & After)

- 이전
  - 신규 개발자가 문서와 실제 구조 차이부터 직접 보정해야 한다.
  - 라우트별 에러 응답 기준이 암묵적이어서 일관성이 깨질 수 있다.
  - 날짜/경계 로직 회귀 위험이 크지만 테스트 축이 명시적으로 고정돼 있지 않다.
  - `tsc`, `lint`가 깨져 있어 배포 전 품질 체크가 부분 수동 점검에 의존한다.
- 이후
  - 문서와 코드 경로가 일치해 온보딩 비용이 낮아진다.
  - 에러 모델과 권한 응답 기준이 명확해져 API 계약이 단단해진다.
  - 핵심 날짜/집계 규칙이 유틸과 테스트로 고정되어 회귀 가능성이 낮아진다.
  - 저장소 수준 품질 게이트가 복구되어 PR과 배포 전 검증 신뢰도가 올라간다.

## 4. 구체적 실행 방안 (Action Items)

### 4.1. 문서와 규약 정합성 복구

- [ ] `.agents/skills/backend/SKILL.md`와 reference 문서의 공통 문서 경로를 실제 위치 기준으로 정정한다.
  - `docs/dev/common/2026.03.12-api-conventions.md`
  - `docs/dev/common/2026.03.12-quality-strategy.md`
- [ ] `docs/onboarding.md`에 "현재 신뢰 가능한 검증 명령"과 "깨져 있는 전역 게이트"를 더 분명히 분리해 적는다.
- [ ] 도메인 구현 문서에서 실제 구현과 어긋난 상태 설명이 있으면 현재 코드 기준으로 갱신한다.

### 4.2. 에러 모델 표준화

- [ ] 서비스 레이어에서 문자열 메시지 비교를 제거하고 공통 커스텀 에러 기반으로 통일한다.
- [ ] 인증 실패, 소유권 불일치, 상태 전이 실패를 각각 어떤 에러 코드로 노출할지 규칙을 문서화한다.
- [ ] `403`과 `404`의 사용 기준을 API 컨벤션 문서에 명시한다.
  - 예: 리소스 존재 은닉이 필요한 경우 `404`
  - 예: 역할 부족, 상태 전이 금지처럼 존재가 전제된 정책 위반은 `403`
- [ ] 라우트별 예외 매핑이 서비스 에러 모델과 직접 연결되도록 정리한다.

### 4.3. 날짜/집계 로직 안정화

- [ ] `daily-log`, `dashboard`의 주간/월간 계산 로직을 공통 유틸로 분리한다.
- [ ] 시간대 기준을 문서로 고정한다.
  - 현재 사용자 기준은 KST
  - 서버 계산 시 어떤 기준 날짜 문자열을 신뢰할지 명확히 정의
- [ ] 아래 경계 테스트를 추가한다.
  - 월말/월초
  - 연말/연초
  - 일요일/월요일 주간 경계
  - 미래 날짜
  - 점수판 기간과 월간 집계가 교차하는 경우

### 4.4. 품질 게이트 재정의 및 복구

- [ ] "현재 저장소에서 반드시 통과해야 하는 게이트"와 "복구 중인 게이트"를 분리 정의한다.
- [ ] 단기 기본 게이트를 아래처럼 운영한다.
  - 변경 도메인 단위 `yarn test --run <target>`
  - 변경 파일 단위 `yarn eslint <files>`
  - 필요 시 `yarn gen:api`
- [ ] 중기 목표로 저장소 전역 게이트를 복구한다.
  - `yarn tsc --noEmit`
  - `yarn lint`
  - `yarn test --run`
  - 필요 시 `yarn test:frontend` / `yarn test:server` 분리 실행
- [ ] CI 또는 PR 체크 기준을 문서로 고정한다.
  - 최소: 변경 도메인 테스트 + 변경 파일 lint
  - 목표: 전역 typecheck + 전역 lint + 전체 test

### 4.5. 테스트 전략 보강

- [ ] 도메인별 핵심 회귀 테스트 목록을 "항상 지켜야 하는 규칙" 중심으로 다시 정리한다.
  - Auth: 세션/비밀번호/관리자 생성
  - Workspace: 단일 소속, 권한
  - Scoreboard: 단일 ACTIVE, archive/reactivate
  - Daily Log: 미래 날짜 금지, archived lead measure 금지, 주간/월간 집계
- [ ] 서비스 테스트와 라우트 테스트의 역할을 더 선명히 구분한다.
  - 서비스 테스트는 비즈니스 규칙 고정
  - 라우트 테스트는 HTTP 계약, 쿠키, 응답 코드 고정
- [ ] 회귀 위험이 큰 API는 happy path 외 대표 error case를 라우트 레벨에서도 명시적으로 고정한다.

### 4.6. 운영 기준 정리

- [ ] 새 백엔드 API 작업의 기본 순서를 강제한다.
  - 도메인 문서 확인
  - OpenAPI 수정
  - 실패 테스트 작성
  - 구현
  - 도메인 단위 검증
- [ ] "머지 가능"의 의미를 명확히 한다.
  - 관련 테스트 통과
  - 관련 lint 통과
  - 에러 코드/응답 규격 일치
  - 기존 문서와 구현 불일치가 생기지 않음

## 5. 예상 비용 및 기간 (Cost & Estimation)

- 1차 문서 및 규약 정비: 반나절 ~ 1일
- 2차 에러 모델 통일 및 날짜 유틸 분리: 1 ~ 2일
- 3차 테스트 보강 및 전역 게이트 복구 착수: 1 ~ 3일
- 전체적으로는 기능 개발과 병행 가능한 1주 내외의 기술부채 정리 트랙으로 운영하는 것이 적절하다.

## 6. 성공 및 실패 판단 기준 (Success & Failure Criteria)

- 성공
  - 새 개발자가 문서만 읽고도 실제 백엔드 진입 경로와 검증 명령을 혼동하지 않는다.
  - 인증/권한/상태 전이 에러가 문자열 비교 없이 일관된 코드 체계로 처리된다.
  - 날짜 경계 관련 핵심 회귀 테스트가 고정된다.
  - PR 기준 최소 품질 게이트가 팀 내에서 명확하고 반복 가능하다.
  - 저장소 전역 게이트 복구 일정과 책임 범위가 분명하다.
- 실패
  - 문서는 추가됐지만 실제 스킬/검증 기준 정리는 충분히 반영되지 않아 현장에서 참조성이 떨어진다.
  - `tsc`, `lint` 문제를 "나중에"로 남겨두고 계속 우회만 하게 된다.
  - 에러 처리와 권한 응답 규칙이 도메인별로 계속 달라진다.

## 7. 권장 실행 순서 (Recommended Rollout)

1. 문서 경로와 검증 명령부터 바로 정리한다.
2. 에러 모델 표준화와 `403/404` 기준을 먼저 고정한다.
3. 날짜/집계 유틸 분리와 경계 테스트를 붙인다.
4. 마지막으로 전역 `tsc`, `lint` 복구를 별도 배치로 진행한다.
