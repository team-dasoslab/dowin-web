# 공개 전환 체크리스트

> 상태 메모 (2026-03-30): 현재 DOWIN는 이미 공개 회원가입(Self-signup)과 복원코드 기반 비밀번호 재설정을 지원한다. 아래 체크리스트는 구현 이전 가정이 섞여 있으므로, 지금은 abuse/rate limit/migration/운영 정책 보강용 후속 체크리스트로 읽는 편이 맞다.

## 1. 배경 및 맥락 (Background & Context)

- 작성 당시 DOWIN는 `관리자가 직접 계정을 생성`하는 폐쇄형 운영 모델을 사용했다.
- 이 방식은 당시 사용자 규모와 운영 현실에는 잘 맞았지만, 공개 회원가입을 도입하는 순간 인증 정책과 운영 리스크가 크게 달라진다.
- 따라서 지금 필요한 것은 공개 기능을 바로 구현하는 것이 아니라, 공개 전환 시 필요한 요구사항을 미리 분리해서 관리하는 일이다.

## 2. 우선순위 및 당위성 (Priority & Rationale)

- 공개 전환은 지금 당장 구현할 MVP 범위가 아니다.
- 다만 관련 요구사항을 문서로 분리하지 않으면, 향후 개발 중 self-signup, 비밀번호 재설정, abuse 대응 같은 항목이 즉흥적으로 섞일 위험이 크다.
- 폐쇄형 운영을 유지하더라도, 전환 조건과 준비 항목을 체크리스트로 관리해두면 이후 설계 변경 비용을 줄일 수 있다.

## 3. 기대 효과 (Impact: Before & After)

- 이전:
  - 공개 전환 관련 요구사항이 산발적으로 논의된다.
  - 어떤 항목이 MVP 범위이고 어떤 항목이 Post-MVP인지 흐려진다.
  - 인증 정책 변경 시 놓치는 보안/운영 요구사항이 생길 수 있다.
- 이후:
  - 공개 전환 시 필요한 항목이 별도 문서에서 관리된다.
  - 폐쇄형 유지와 공개 준비가 동시에 가능해진다.
  - 실제 구현 전 우선순위와 리스크가 명확해진다.

## 4. 구체적 실행 방안 (Action Items)

### 4.1. 계정 생성 정책

- [x] self-signup 도입 및 기본 운영 경로 반영
- [ ] 공개 가입 허용 대상 정의
  - 누구나 가입 가능
  - 초대 기반 공개 베타
  - 특정 도메인/코드 기반 제한 공개
- [ ] 기존 `관리자 생성 계정`과 `self-signup 계정`이 공존할 때의 계정 생성 정책 정의

### 4.2. 인증/복구 플로우

- [x] 복원코드 기반 비밀번호 재설정 경로 구현
- [ ] 추가 비밀번호 재설정 방식 필요 여부 정의
  - 이메일 링크
  - 관리자 초기화
  - 기타 대체 수단
- [ ] 최초 로그인 플로우와 공개 가입 직후 온보딩 플로우를 구분할지 결정
- [ ] 세션 만료, 중복 로그인, 장기 미접속 계정 처리 정책 정의

### 4.3. abuse / spam / rate limit

- [ ] 로그인 시도 rate limiting 정책 정의
- [ ] 회원가입 요청 rate limiting 정책 정의
- [ ] 봇/스팸 가입 대응 방식 정의
  - captcha 도입 여부
  - IP / device fingerprint 기반 제한 여부
- [ ] 워크스페이스 생성 남용 방지 정책 정의

### 4.4. 워크스페이스 정책

- [ ] 공개 가입 사용자가 워크스페이스를 바로 만들 수 있는지 정의
- [ ] `1인 1워크스페이스` 정책을 공개 전환 후에도 유지할지 결정
- [ ] 워크스페이스 참가 방식 정의
  - 직접 생성
  - 초대 링크
  - 승인 기반 참가

### 4.5. 데이터 migration / 공존 전략

- [ ] 기존 폐쇄형 사용자와 공개 가입 사용자의 공존 모델 정의
- [ ] 기존 계정의 식별자 체계(`customId`)를 그대로 유지할지 확정
- [ ] 기존 워크스페이스와 공개 유입 사용자가 섞일 때 role 부여 규칙 정의
- [ ] 과거 운영 데이터와 새 가입 사용자 흐름이 충돌하지 않도록 migration 시나리오 작성

### 4.6. 운영/지원 준비

- [ ] 계정 잠금, 복구, 수동 가입 해제 등 최소 운영 절차 정리
- [ ] 공개 전환 후 필요한 운영 지표 정의
  - 가입 수
  - 로그인 실패율
  - 비밀번호 재설정 요청 수
  - 스팸/차단 이벤트 수
- [ ] 전역 관리자 콘솔이 필요한 시점을 별도 판단 기준으로 정리

## 5. 예상 비용 및 기간 (Cost & Estimation)

- 문서 정리 자체는 짧게 끝낼 수 있지만, 실제 공개 전환 구현은 auth / workspace / onboarding / 운영 정책을 함께 건드리는 중간 규모 작업이다.
- 구현 단계로 넘어가면 최소 1개 이상의 별도 마일스톤으로 관리하는 것이 적절하다.

## 6. 성공 및 실패 판단 기준 (Success & Failure Criteria)

- 성공:
  - 폐쇄형 운영과 공개 전환 준비가 같은 문맥 안에서 충돌하지 않는다.
  - 공개 전환 요구사항이 즉흥적인 기능 요청이 아니라 별도 체크리스트로 관리된다.
  - 실제 self-signup 구현에 들어갈 때 필요한 선결 조건이 문서상 명확하다.
- 실패:
  - 공개 전환 관련 논의가 다시 메인 MVP 범위에 섞여 들어온다.
  - 인증/운영/보안 요구사항이 분리되지 않아 구현 중 재설계가 반복된다.
