# WIG Audit Log / Soft Delete 도입 계획

## 1. 배경 및 맥락 (Background & Context)

- WIG 운영 문서 정리 과정에서 가장 큰 복구 리스크는 `A 시점 데이터 손실을 B 시점에 늦게 알았을 때` 부분 복구 난이도가 높다는 점으로 드러났다.
- 현재 WIG 스키마는 핵심 엔티티 대부분이 hard delete 중심이고, “누가 언제 무엇을 지웠는지”를 남기는 audit trail도 거의 없다.
- 이 상태에서는 아래 비용이 커진다.
  - 무엇이 사라졌는지 찾는 비용
  - 실수 삭제와 정상 삭제를 구분하는 비용
  - 특정 사용자/워크스페이스 단위 부분 복구 비용
  - 장애 후 원인 설명 비용

이 문서의 목적은 “전 테이블에 무조건 soft delete를 붙인다”가 아니라, WIG에서 운영상 의미가 큰 엔티티부터 최소한으로 도입하는 계획을 세우는 것이다.

## 2. 왜 지금 필요한가 (Priority & Rationale)

- 출시 직전 운영 기준에서 가장 위험한 것은 “복구는 가능한데 너무 비싸거나 느린 상태”다.
- soft delete와 audit log는 기능 자체보다 운영 안전장치에 가깝다.
- 특히 WIG는 아래 데이터가 핵심 가치다.
  - 워크스페이스 소속
  - 활성/보관 점수판 상태
  - 선행지표 구조
  - 날짜별 daily log 기록
- 이 축에서 실수 삭제가 나면 사용자 신뢰에 직접 타격이 간다.

## 3. 두 장치의 역할 분리 (Soft Delete vs Audit Log)

### 3.1. Soft Delete

soft delete는 row를 바로 지우지 않고 `deleted_at` 같은 마커를 남기는 방식이다.

효과:

- 실수 삭제 즉시 복구가 쉬움
- “현재는 숨기되 복원 가능” 상태를 만들 수 있음
- 부분 복구 범위를 줄일 수 있음

한계:

- 누가 지웠는지, 어떤 값이 바뀌었는지까지는 부족함
- 조회 조건이 늘어나고 구현 복잡도가 올라감

### 3.2. Audit Log

audit log는 누가 어떤 엔티티에 어떤 동작을 했는지 남기는 기록이다.

효과:

- 삭제/수정 주체 추적
- 사고 타임라인 파악
- 부분 복구 대상 식별
- 운영 설명 책임 강화

한계:

- audit log만으로는 즉시 복원되지 않음
- 로그 설계와 보존 정책이 필요함

정리하면:

- soft delete는 “복구 가능한 현재 상태”
- audit log는 “무슨 일이 있었는지의 이력”

둘은 대체재가 아니라 보완재다.

## 4. WIG에선 어디부터 적용해야 하나

모든 테이블에 일괄 적용하는 것은 과하다.  
운영상 손실 비용과 관계 복잡도를 기준으로 우선순위를 나눈다.

### 4.1. 1순위 soft delete 후보

- `workspaces`
- `workspace_members`
- `scoreboards`
- `lead_measures`

이유:

- 사용자 경험과 운영 데이터 구조에 직접 영향
- 실수 삭제 시 복구 가치가 큼
- 관계 단위 복구의 기준점 역할을 함

### 4.2. 2순위 soft delete 후보

- `daily_logs`
- `workspace_invites`
- `team_memos`

이유:

- 중요 데이터이긴 하지만 volume과 조회 영향이 더 큼
- 특히 `daily_logs`는 양이 많아 soft delete 적용 전 조회/집계 비용을 따로 검토해야 한다

### 4.3. soft delete 비우선 후보

- `sessions`
- `auth_recovery_codes`
- `push_subscriptions`

이유:

- 운영 복구 가치보다 만료/재생성 경로가 더 중요함
- 보안/수명주기 특성상 hard delete 또는 자연 만료가 더 적합할 수 있다

## 5. Audit Log는 어디까지 남겨야 하나

최소 MVP는 “모든 값 변화 전체 스냅샷”까지 갈 필요는 없다.

우선 아래 정도면 충분하다.

- actor user id
- action type
  - `CREATE`
  - `UPDATE`
  - `DELETE`
  - `RESTORE`
- entity type
  - `WORKSPACE`
  - `WORKSPACE_MEMBER`
  - `SCOREBOARD`
  - `LEAD_MEASURE`
  - `DAILY_LOG`
- entity id
- workspace id nullable
- occurred at
- optional metadata JSON
  - 변경 사유
  - 주요 필드 diff 요약

## 6. 추천 설계 원칙

### 6.1. soft delete는 `deleted_at` 우선

- boolean `is_deleted`보다 `deleted_at`이 낫다
- 이유:
  - 삭제 시점 추적 가능
  - 복구/정렬/비교에 유리

### 6.2. soft delete 대상은 기본 조회에서 제외

- 서비스/스토리지 레이어에서 기본 쿼리는 `deleted_at is null`
- 운영 복구나 관리자 조회에서만 포함

### 6.3. restore도 명시적 action으로 남긴다

- soft delete만 도입하고 restore 기록이 없으면 운영 추적이 반쪽이 된다

### 6.4. audit log는 business action 기준으로 남긴다

- 모든 SQL mutation을 저수준으로 남기기보다
- 서비스 레이어에서 의미 있는 action 기준으로 남기는 편이 WIG 규모에 맞다

예:

- workspace member 강퇴
- scoreboard archive
- lead measure 삭제
- daily log 대량 정리

## 7. WIG에서 예상되는 구현 비용과 리스크

### soft delete 리스크

- 조회 누락 버그
  - `deleted_at is null` 조건 누락 시 숨겨야 할 데이터가 보일 수 있음
- unique 제약 재검토
  - soft delete row가 남아 있으면 새 row 생성과 충돌 가능
- 집계 영향
  - `daily_logs` 같은 테이블은 집계 쿼리에 조건이 빠지면 수치가 틀어질 수 있음

### audit log 리스크

- 로그는 남았는데 복구용 정보가 부족할 수 있음
- 메타데이터를 너무 많이 남기면 관리 비용이 커짐
- 개인정보 또는 민감 필드가 로그에 과도하게 복제될 위험이 있음

## 8. 권장 도입 순서 (Recommended Rollout)

### 배치 1. 최소 audit log

- 공통 `audit_logs` 테이블 설계
- 서비스 레이어에서 핵심 삭제/복구/상태전이 action 기록
- 대상:
  - workspace member
  - scoreboard
  - lead measure

### 배치 2. soft delete 1차

- `scoreboards`, `lead_measures`에 `deleted_at` 도입 검토
- 관련 조회/집계/수정 흐름 정리
- restore 가능 여부 정의

### 배치 3. workspace 계층 확장

- `workspace_members`, `workspaces` soft delete 또는 deactivate 전략 정리
- 멤버 강퇴/탈퇴/삭제의 의미 분리

### 배치 4. daily log 판단

- `daily_logs`는 volume과 집계 특성 때문에 별도 검토
- soft delete보다 audit log + export 보관이 더 현실적인지 비교 필요

## 9. MVP 제안

지금 WIG 규모에서 가장 현실적인 최소안은 아래다.

1. 전 테이블 soft delete는 하지 않는다
2. 먼저 `audit_logs`를 도입한다
3. soft delete는 `scoreboards`, `lead_measures`, `workspace_members`부터 본다
4. `daily_logs`는 바로 soft delete하지 말고 성능/집계 복잡도 검토 후 결정한다

이렇게 가는 이유:

- audit log는 운영 가치를 빠르게 준다
- soft delete는 도메인별 조회 규칙 수정이 커서 범위를 좁히는 편이 안전하다

## 10. 성공 및 실패 기준 (Success & Failure Criteria)

- 성공
  - 운영자가 “누가 무엇을 지웠는지”를 추적할 수 있다
  - 핵심 엔티티 삭제가 즉시 irrecoverable 상태가 아니다
  - 부분 복구 시 대상 식별 비용이 줄어든다
- 실패
  - 전 테이블 일괄 soft delete로 복잡도만 늘고 운영 기준은 불명확하다
  - audit log가 남지만 실제 복구 판단에 도움 되는 정보가 없다
  - 집계/조회 쿼리에 soft delete 조건 누락이 반복된다

- `daily_logs`는 soft delete 도입 전 성능/집계 리스크 검토
