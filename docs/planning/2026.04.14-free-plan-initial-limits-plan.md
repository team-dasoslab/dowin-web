# Free 플랜 출시 정책 v1

## 1. 배경 및 맥락 (Background & Context)

- 현재 WIG는 결제 모듈이 없지만, 공개 운영과 리텐션 검증을 위해서는 플랜 구조를 먼저 깔아둘 필요가 있다.
- 지금 단계의 핵심은 돈을 받는 것이 아니라, `무엇에 돈을 받을 수 있는지`를 검증하는 것이다.
- 따라서 결제 흐름 없이도 모든 워크스페이스를 `Free 플랜 사용 중`으로 간주하고, 향후 유료화 포인트가 될 경계를 미리 정의하는 것이 적절하다.

## 2. 문제 정의 (Problem Statement)

- 플랜 개념이 없으면 나중에 유료화 시 데이터 모델, 권한, 기능 게이트를 한꺼번에 뜯어고쳐야 한다.
- 반대로 지금부터 Free limit가 너무 강하면 핵심 경험이 막혀 공개 초기 리텐션 검증 자체가 어렵다.
- 따라서 현재 필요한 것은 `핵심 경험은 열어두고, 확장 경험에만 경계를 두는 Free 플랜`이다.

## 3. 목표 (Goals)

### 3.1. MVP 목표

- 모든 워크스페이스에 `FREE` 플랜 상태를 기본 부여한다.
- 핵심 팀 실행 경험은 Free에서도 막지 않는다.
- 향후 유료화 포인트가 될 수 있는 데이터/기능 제한만 최소한으로 둔다.
- 결제 모듈 없이도 제품 내부에서 `현재 Free 플랜 사용 중` 상태를 명시할 수 있게 한다.

### 3.2. Post-MVP 목표

- `STANDARD` 또는 동등한 유료 플랜 추가
- 업그레이드 CTA
- 결제/청구/인보이스 도입
- 플랜별 기능 게이트 정교화

## 4. 제품 원칙 (Product Principles)

- Free 플랜은 제품을 제대로 써볼 수 있어야 한다.
- 핵심 가치 기능은 Free에서 막지 않는다.
- 제한은 `확장성`, `운영 깊이`, `보관 범위`에 우선 적용한다.
- 제한은 사용자를 벌주는 방식이 아니라, 향후 업그레이드 이유가 되는 방식이어야 한다.
- 결제 모듈이 없더라도 플랜 상태는 제품 내부에서 일관되게 보여야 한다.
- 무료/유료 차이는 `보는 범위`보다 `리더가 팀을 실제로 움직일 수 있는가`에서 가장 선명해야 한다.

## 4.1. Free / STANDARD 한 줄 가치 정의

- `FREE`
  - 팀이 실행 상태를 함께 보고 기록하는 기본 운영판
- `STANDARD`
  - 리더가 팀의 막힘을 더 빨리 파악하고, 바로 개입하고, 후속 변화를 확인하는 운영 도구

이 정의를 기준으로 보면:

- Free는 제품의 핵심 경험을 체험하게 해야 한다.
- Standard는 단순 조회 확장이 아니라 `리더 개입 효율`을 높여야 한다.
- 조회 기간, 보관 기간, export는 보조 가치일 수 있지만 유료의 전면 메리트가 되어서는 안 된다.

## 5. 초기 플랜 구조 (Initial Plan Structure)

### 5.1. 플랜 타입

- `FREE`
- 향후 예약:
  - `STANDARD`

### 5.2. 현재 운영 정책

- 모든 신규 워크스페이스는 기본적으로 `FREE`
- 기존 워크스페이스도 마이그레이션 시 `FREE`
- 결제/청구 관련 필드는 MVP에서 비활성 또는 placeholder 상태로 둔다

## 6. Free에서 열어둘 핵심 경험 (Must Remain Open)

- 회원가입/로그인
- 워크스페이스 생성/참가
- 점수판 생성
- 선행지표 생성/기록
- 개인 대시보드 조회
- 팀 대시보드 조회
- 리더용 주간 운영 리포트 기본 사용

이 항목들은 막히면 공개 초기 가치 검증이 거의 불가능해진다.

## 7. Free 초기 제한 정책 (Initial Limits)

출시 v1 원칙:

- 제한은 한 번에 많이 넣지 않는다.
- 제품/데이터 경계를 심는 데 꼭 필요한 제한만 먼저 적용한다.
- v1에서는 `핵심 경험을 막지 않는 3개 제한`만 우선 고정한다.

### 7.1. 멤버 수 제한

- 의미:
  - Free는 소규모 팀 검증용으로 유지
- 적용값:
  - 워크스페이스당 최대 10명
- 장점:
  - 향후 팀 확장 과금 근거가 명확함
- 주의:
  - 10명 제한으로도 초기 팀 유입이 막히는지 운영 데이터로 확인 필요

### 7.2. 리더 리마인드 플랜 게이트

- 의미:
  - 리더가 팀원에게 보내는 개입형 리마인드는 `STANDARD`의 핵심 가치 기능으로 본다
- 적용값:
  - `FREE`: 사용 불가
  - `STANDARD`: 사용 가능
- 장점:
  - 무료 공개 운영에서는 핵심 기록/조회 경험에 집중할 수 있다
  - 리더 개입 도구를 가장 명확한 업그레이드 이유로 남길 수 있다

### 7.3. 리더 리포트 보관 기간 제한

- 의미:
  - 최신 몇 주 리포트만 조회 가능
- 적용값:
  - 최근 4주
- 장점:
  - 운영 히스토리 축적 가치가 유료 포인트가 될 수 있음
- 현재 판단:
  - 출시 v1 즉시 적용 대상은 아니다
  - 리더 리포트 기능 자체가 아직 초기라, 보관 기간 제한까지 동시에 넣으면 운영 가치 검증이 어려워질 수 있다

### 7.4. 데이터 보관 기간 제한

- 의미:
  - 기록 자체를 막지 않고, 장기 히스토리 조회/분석 범위를 제한
- 적용값:
  - 기록 조회는 최근 6개월
- 장점:
  - 무료 체험 가치를 해치지 않으면서 장기 데이터 가치를 분리 가능
- 현재 판단:
  - 출시 v1 즉시 적용 대상은 아니다
  - 현재 제품 단계에서는 장기 히스토리보다 핵심 루프 검증이 우선이므로, 조회 제한은 후속 유료화 실험 단계로 미룬다

### 7.5. 고급 분석 제한

- 의미:
  - 기본 대시보드는 열어두고, 향후 심화 리포트나 비교 분석만 제한
- 장점:
  - 핵심 경험을 해치지 않음

### 7.6. 데이터 export 플랜 게이트

- 의미:
  - CSV export는 기본 조회 경험이 아니라 운영/보고용 보조 확장 기능으로 본다
- 적용값:
  - `FREE`: 사용 불가
  - `STANDARD`: 사용 가능
- 장점:
  - 앱 내 기본 조회는 열어두면서, 외부 반출/가공 수요는 보조 유료 포인트로 분리할 수 있다

## 7.7. 리더 개입/후속 확인 기능 묶음

- 의미:
  - `STANDARD`의 진짜 메리트는 특정 기능 하나가 아니라 `파악 -> 개입 -> 후속 확인` 흐름 전체다
- 포함 후보:
  - 팀원별 리더 리마인드 발송
  - 개입 이력 조회
  - 리포트 내 개입 우선순위 강조
  - 리마인드 이후 기록 재개 확인
- 원칙:
  - 가능하면 이 기능들은 같은 플랜 tier 안에서 묶여 보여야 한다
  - 유료 메리트가 `한 개 버튼 해금`처럼 보이지 않게 설계해야 한다

## 8. 초기 비추천 제한 (What Not To Limit Now)

- 점수판 생성 자체 제한
- 기본 기록 기능 제한
- 팀 대시보드 접근 차단
- 리더 리포트 기능 전체 차단
- 첫 워크스페이스 생성/참가 제한

이 항목을 막으면 Free 플랜이 아니라 `제품 가치가 성립하지 않는 데모 모드`가 될 가능성이 높다.

## 9. 출시 적용안 (Release Policy v1)

### 9.1. 추천안

- 모든 워크스페이스 기본 플랜: `FREE`
- 멤버 수 제한: 최대 10명
- 리더 리마인드: `STANDARD` 전용
- 데이터 export: `STANDARD` 전용
- 리더 리포트 보관 기간 제한: 설계만 유지, 적용은 후순위
- 기록 조회 제한: 설계만 유지, 적용은 후순위
- 고급 분석: 추후 유료 플랜 전용 후보로만 유지
- 후속 유료 플랜 코드: `STANDARD`

### 9.2. 왜 이 조합이 맞는가

- 핵심 실행 경험은 보존된다.
- 리더 리포트까지는 무료로 운영 가치를 보여줄 수 있다.
- v1부터 구현 복잡도를 과도하게 올리지 않는다.
- 나중에 결제 도입 시 업그레이드 이유를 설명하기 쉽다.
- 무엇보다 `STANDARD를 쓰면 팀을 더 잘 움직일 수 있다`는 메시지를 만들 수 있다.

### 9.3. 제한 도달 시 해석

- 멤버 수 10명 초과:
  - 신규 멤버 초대/참가를 막고 현재 인원은 유지한다.
- 리더 리마인드 사용 시도:
  - Free에서는 기능 자체를 노출하지 않거나, `STANDARD` 업그레이드 안내로 연결한다.
- 데이터 export 사용 시도:
  - Free에서는 export 진입을 막거나, `STANDARD` 업그레이드 안내로 연결한다.
- 기록 조회 6개월 초과:
  - 출시 v1에서는 아직 적용하지 않는다.
- 리더 리포트 4주 초과:
  - 출시 v1에서는 아직 적용하지 않는다.

## 10. UI/카피 원칙 (UI & Messaging)

- 워크스페이스 설정 또는 프로필에서 `현재 Free 플랜 사용 중` 표시
- 결제 버튼은 MVP에서 노출하지 않거나, `추후 제공 예정` 수준으로 유지
- 제한 도달 시 문구는 차단보다 안내 중심
  - 예: `Free 플랜에서는 최근 6개월 기록까지 확인할 수 있어요`
  - 예: `Free 플랜에서는 최근 4주 리포트까지 확인할 수 있어요`
  - 예: `팀원 리마인드는 STANDARD 플랜에서 제공될 예정이에요`
  - 예: `CSV export는 STANDARD 플랜에서 제공될 예정이에요`

## 11. 유료 매력도 판단 (Paid Value Attractiveness)

### 11.1. 현재 기준 핵심 결제 사유 후보

- 리더 리마인드
  - 팀을 실제로 움직이게 만드는 직접 개입 기능이라 유료 포인트로 설득력이 있다
- 개입 이후 후속 확인
  - 리더 입장에서 중요한 것은 보내는 것 자체보다 `그래서 다시 움직였는가`를 보는 것이므로, 이 흐름이 붙어야 유료 가치가 강해진다
- 리포트 내 개입 우선순위
  - 단순 주간 요약보다 `누구를 먼저 봐야 하는지`가 드러날 때 운영 도구로서 유료 가치가 선명해진다
- 데이터 export
  - 보고, 공유, 2차 가공 수요가 분명해 보조 유료 포인트로 설명하기 쉽다

### 11.2. 현재 기준 보조 가치 후보

- 기록 조회 6개월 초과
- 리더 리포트 4주 초과
- 향후 고급 분석

이 항목들은 있으면 분명 좋아지지만, 단독 결제 사유로는 상대적으로 약하다.

### 11.3. 현재 판단 요약

- `STANDARD`의 중심 세일즈 포인트는 `긴 조회기간`이 아니라 `리더 개입`과 `개입 이후 후속 확인`이어야 한다.
- `export`는 보조 가치로 두되, 전면 세일즈 포인트로 쓰지 않는다.
- 히스토리 확장은 결제를 정당화하는 보조 요소로 두고, 플랜의 전면 가치는 아니다.
- 따라서 랜딩/가격 정책/후속 구현 우선순위 모두 아래 기준을 따른다.
  - 핵심 가치: 리더가 팀을 더 잘 움직이게 해준다
  - 보조 가치: 데이터를 더 오래 보고, 밖으로 꺼내고, 더 깊게 분석할 수 있다

## 13. 도메인 영향 범위 (Domain Impact)

- workspace
  - 워크스페이스별 플랜 상태 저장 필요
- profile/settings
  - 현재 플랜 표시 필요
- dashboard
  - 기록 조회 6개월 제한 연동 가능
  - 리포트 보관 기간/히스토리 조회 제한 연동 가능
- analytics/export
  - CSV export의 플랜 게이트 연동 필요
- notification
  - 리더 리마인드의 플랜 게이트 연동 필요
  - 워크스페이스 단위 리더 리포트 설정과 연동 가능
- leader operations
  - 리더 개입 이력, 후속 기록 재개 확인 같은 `STANDARD` 가치 흐름 설계 필요
- analytics
  - Free limit 도달률, 제한 이후 이탈/잔존 추적 가능
- billing(후속)
  - 현재는 미구현, 향후 플랜 업그레이드와 연결

## 14. 실행 순서 (Recommended Sequence)

1. 워크스페이스 플랜 상태 모델 추가
2. 모든 워크스페이스를 `FREE`로 운영
3. 제품 UI에 Free 플랜 표시
4. 멤버 수 제한, 리더 리마인드 플랜 게이트, 데이터 export 게이트 적용

## 16. 성공 및 실패 판단 기준 (Success & Failure Criteria)

### 성공

- 결제 없이도 플랜 구조가 제품 내부에서 일관되게 보인다.
- Free 플랜만으로도 공개 운영과 리텐션 검증이 가능하다.
- 제한이 핵심 경험을 해치지 않으면서 향후 유료화 경계를 만들어준다.
- 무료와 유료의 차이가 `리더 개입 가능 여부` 기준으로 사용자에게 설명 가능하다.

### 실패

- 제한이 너무 강해 제품 핵심 가치 체험이 막힌다.
- 제한이 너무 약해 향후 유료화 포인트가 전혀 남지 않는다.
- 플랜 상태는 생겼지만 실제 기능/데이터 경계와 연결되지 않는다.
- 사용자가 `왜 STANDARD를 써야 하는지`를 한 문장으로 이해하지 못한다.
