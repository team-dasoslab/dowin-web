# AI Engineering 적용 계획

## 1. 배경

Dowin 저장소는 이미 `AGENTS.md`, `codex.md`, `.agents/skills/*`를 통해 AI Agent가 읽어야 할 문서, 작업 분류, 검증 루틴, 스킬별 책임을 제공한다.

현재 부족한 것은 새로운 AI 모델을 도입하는 것이 아니라, AI가 만든 결과를 평가하고, 실패를 축적하고, 다음 작업 품질을 개선하는 운영 루프다.

이 문서는 AI Engineering 관련 개념을 Dowin의 개발 운영에 맞게 해석하고, 실제로 적용할 작업을 정리한다.

## 2. 핵심 개념 정리

### 2.1. Prompt Engineering

Prompt Engineering은 좋은 지시문을 작성해 AI의 응답 품질을 높이는 방법이다.

핵심은 역할 부여, few-shot 예시, 출력 형식 지정, 제약 조건 명시다.

Dowin 적용 관점:

- 각 스킬의 `SKILL.md`가 저장소 전용 prompt 역할을 한다.
- 현재는 작업 유형별 지시가 있지만, 실패 사례를 반영해 prompt를 체계적으로 개선하는 루프는 약하다.

### 2.2. Context Engineering

Context Engineering은 프롬프트 하나가 아니라, AI가 판단할 때 참고할 정보 환경 전체를 설계하는 일이다.

핵심은 어떤 문서, 코드, 이전 대화, 도구 실행 결과를 언제 얼마나 넣을지 결정하는 것이다.

Dowin 적용 관점:

- `README.md`, `docs/onboarding.md`, `AGENTS.md`, `codex.md`, `.agents/skills/*`가 기본 context layer다.
- 하지만 작업별 context packet을 자동으로 만드는 도구는 아직 없다.

### 2.3. Agent Engineering

Agent Engineering은 AI가 한 번의 응답으로 끝나지 않고, 계획, 도구 호출, 결과 확인, 재판단을 반복하도록 실행 환경을 설계하는 일이다.

Dowin 적용 관점:

- `docs/planning/2026.04.09-dowin-agent-orchestration.md`가 agent workflow의 출발점이다.
- 현재는 단일 세션 중심 운영이고, 자동 handoff나 리뷰 자동화는 아직 약하다.

### 2.4. Tool Calling

Tool Calling은 LLM이 직접 외부 작업을 수행하는 것이 아니라, 호출할 함수와 인자를 구조화해 제안하고, 실제 실행은 외부 시스템이 담당하는 방식이다.

Dowin 적용 관점:

- Codex가 `rg`, `yarn test`, `apply_patch` 같은 도구를 사용하는 것이 개발 harness의 tool calling 사례다.
- 제품 내부 AI 기능을 만들 경우에도 LLM이 직접 DB를 수정하지 않고, 검증된 service/API를 호출하도록 경계를 둬야 한다.

### 2.5. MCP

MCP(Model Context Protocol)는 AI가 파일시스템, DB, 외부 API 같은 도구와 데이터 소스에 통일된 방식으로 연결되도록 만든 프로토콜이다.

Dowin 적용 관점:

- 현재 저장소에는 Dowin 전용 MCP 운영 표준이 없다.
- 외부 도구를 붙이기 전에 허용 도구, 권한 경계, 승인 기준, 결과 검증 기준을 먼저 정의해야 한다.

### 2.6. Harness

Harness는 모델이 도구를 호출하고, 결과를 받고, 다시 판단하는 실행 루프와 환경 전체를 뜻한다.

Dowin 적용 관점:

- Codex, Cursor, Claude Code 같은 도구는 각각 다른 harness다.
- Dowin은 `AGENTS.md`, `codex.md`, `.agents/skills/*`로 Codex harness에 맞는 작업 환경을 만들고 있다.
- 다만 harness 보안과 품질 개선 루프는 더 강화해야 한다.

### 2.7. RAG

RAG(Retrieval-Augmented Generation)는 답변 생성 전에 관련 문서를 검색해 context에 주입하는 방식이다.

Dowin 적용 관점:

- 현재는 사람이 문서 읽기 순서를 지정하는 방식에 가깝다.
- 향후에는 변경 파일이나 작업 유형을 기준으로 관련 문서를 자동 검색해 Agent에게 제공할 수 있다.

### 2.8. Context Management

Context Management는 context window가 유한하다는 제약 안에서 무엇을 넣고 뺄지, 언제 요약할지, 어떤 정보를 장기 기억으로 남길지 결정하는 전략이다.

Dowin 적용 관점:

- 작업이 길어질수록 읽은 파일, 명령 결과, 사용자 요구가 섞여 context가 커진다.
- 단계별 요약, handoff contract, failure category를 더 엄격히 쓰면 context 오염을 줄일 수 있다.

### 2.9. Memory

Memory 관리는 세션이 끝나도 남길 정보를 저장하고, 다음 작업에서 필요한 시점에 불러오는 구조다.

Dowin 적용 관점:

- 현재 long-term memory 역할은 문서와 스킬이 담당한다.
- 실패 사례, 자주 쓰는 context packet, 리뷰 결과는 별도 memory artifact로 남길 필요가 있다.

### 2.10. Evaluation

Evaluation은 특정 태스크에서 AI 출력이 기준에 맞는지 측정하는 체계다.

Dowin 적용 관점:

- 현재는 코드 테스트와 린트는 있지만 AI 산출물 자체를 평가하는 eval은 없다.
- Golden case, rubric, failure category를 먼저 만들어야 한다.

### 2.11. Benchmark

Benchmark는 표준화된 데이터셋과 메트릭으로 모델, 프롬프트, 설정을 비교하는 방식이다.

Dowin 적용 관점:

- 외부 benchmark보다 Dowin 작업에 맞는 내부 golden dataset이 더 중요하다.
- 예: API 계약 변경, 프론트 API 연동, WebView bridge 변경, 결제 권한 변경.

### 2.12. Prompt 버전 관리

Prompt 버전 관리는 프롬프트와 스킬을 코드처럼 관리하고, 변경 이유와 결과를 추적하는 방식이다.

Dowin 적용 관점:

- `.agents/skills/*`는 git으로 관리되지만, 변경 이유와 성능 변화 기록은 약하다.
- 스킬 변경 changelog와 eval 결과 연결이 필요하다.

### 2.13. Review Automation

Review Automation은 PR 변경 범위를 분석해 자동으로 위험을 태깅하거나 리뷰 코멘트를 남기는 방식이다.

Dowin 적용 관점:

- 현재 CI는 lint, type check, test, build, preview smoke 중심이다.
- 변경 파일 기반으로 quality/security/performance/harness 리뷰가 자동으로 떠오르는 구조가 필요하다.

### 2.14. LLM 동작 원리

#### Tokenization

Tokenization은 텍스트를 모델이 처리할 수 있는 토큰 단위로 나누는 과정이다.

Dowin 적용 관점:

- 한국어 문서와 긴 코드 context는 토큰 비용을 빠르게 키운다.
- 작업별 context를 선별하지 않으면 비용과 지연 시간이 늘고, 중요한 정보가 묻힐 수 있다.

#### Context Window

Context Window는 모델이 한 번에 참고할 수 있는 최대 토큰 범위다.

Dowin 적용 관점:

- 모든 문서를 다 넣는 방식보다 작업별 필요한 문서만 넣는 방식이 안정적이다.

#### KV Cache / Prompt Caching

KV Cache는 이전 토큰의 attention 계산 결과를 재사용해 다음 토큰 생성을 빠르게 하는 방식이다.

Dowin 적용 관점:

- 자주 반복되는 시스템 지시, 스킬, 저장소 규칙은 안정적인 앞쪽 context로 유지하고, 매번 바뀌는 사용자 요청과 변경 파일 정보는 뒤쪽에 배치하는 전략이 유리하다.

#### Decoding / Reasoning

Decoding은 모델이 다음 토큰을 선택하는 방식이며, reasoning은 복잡한 문제를 단계적으로 풀도록 하는 모델 또는 harness의 동작 방식이다.

Dowin 적용 관점:

- 코드 수정, 보안 검토, DB 권한 변경은 낮은 무작위성과 충분한 검증 루프가 필요하다.
- 카피, 랜딩, 문구 실험은 더 넓은 탐색이 유효할 수 있다.

## 3. Dowin 현재 상태 진단

### 3.1. 강점

- AI Agent용 진입 문서가 있다.
- 작업 유형별 로컬 스킬이 있다.
- 백엔드, 프론트엔드, WebView, 품질, 성능, 보안, 하네스 보안이 분리되어 있다.
- OpenAPI, Orval, Zod, 테스트, CI가 있어 rule-based 검증 기반이 있다.
- 하네스 변경 시 별도 보안 점검을 요구하는 규칙이 있다.

### 3.2. 부족한 점

- AI 산출물 전용 eval이 없다.
- Golden case와 benchmark dataset이 없다.
- 스킬/prompt 변경 이력과 결과 품질이 연결되어 있지 않다.
- PR 변경 범위 기반 리뷰 자동화가 없다.
- Context packet 자동 생성이 없다.
- MCP/tool 운영 표준이 없다.
- 제품 내부 AI 기능의 우선순위가 현재 Basic paid value와 다시 정렬되어 있지 않다.

### 3.3. 적용 원칙

- 우선순위는 제품 내부 AI 기능보다 개발 운영 보조에 둔다.
- 가능한 작업은 로컬 모델, 로컬 스크립트, 규칙 기반 분석으로 먼저 처리한다.
- PR 코드 리뷰처럼 외부 LLM/API 호출이 필요한 기능은 당장 도입하지 않는다.
- 자동화는 처음부터 PR comment bot으로 가지 않고, CI summary 또는 로컬 리포트 출력부터 시작한다.
- AI 판단이 들어가는 경우에도 최종 merge, 배포, DB 변경, 보안 판단 책임은 사람에게 둔다.

## 4. 적용 작업 목록

### 4.1. P0: AI 작업 품질 Eval 만들기

목표:

- AI 작업 결과를 사람이 감으로만 판단하지 않고 반복 가능한 기준으로 평가한다.

작업:

- `docs/dev/common/` 또는 `docs/planning/`에 AI 작업 eval 기준 문서를 만든다.
- 대표 golden case 5개를 정의한다.
  - API 계약 변경
  - 프론트 API 연동
  - WebView bridge 변경
  - 결제/권한 변경
  - 문서/온보딩 정리
- 각 case마다 성공 기준을 적는다.
  - 요구사항 충족
  - 변경 범위 준수
  - 테스트 추가 여부
  - 문서 갱신 여부
  - 보안/권한 누락 여부
- 실패 카테고리를 기존 오케스트레이션 문서의 `failure_categories`와 연결한다.

산출물:

- `docs/planning/YYYY.MM.DD-ai-work-evaluation-plan.md`
- Golden case 목록
- Eval rubric

### 4.2. P0: `quality-check` 스킬에 AI 산출물 확인 항목 추가

목표:

- 품질 점검이 코드 테스트뿐 아니라 AI 작업 특유의 실패도 잡도록 한다.

작업:

- `.agents/skills/quality-check/SKILL.md`에 AI 산출물 검토 항목을 추가한다.
- `references/quality-rules.md`에 다음 항목을 넣는다.
  - 요구사항 누락
  - 과잉 수정
  - 관련 없는 리팩터링
  - 문서-구현 불일치
  - 테스트 없이 동작 변경
  - OpenAPI와 구현 불일치
- 리뷰 결과 보고 형식에 `ai_output_risk` 카테고리를 추가할지 검토한다.

산출물:

- 갱신된 `dowin-quality-check`
- AI 작업 검토 체크리스트

### 4.3. P1: 스킬/프롬프트 변경 이력 관리

목표:

- 스킬 변경이 왜 일어났고 어떤 실패를 줄이기 위한 것인지 추적한다.

작업:

- `.agents/skills/CHANGELOG.md` 또는 `docs/planning/ai-skill-change-log.md` 위치를 정한다.
- 스킬 변경 시 아래 항목을 기록한다.
  - 변경 날짜
  - 대상 스킬
  - 변경 이유
  - 기대 효과
  - 연결된 실패 카테고리
  - 후속 eval 필요 여부
- 하네스 관련 변경은 `dowin-harness-security-check` 대상임을 명시한다.

산출물:

- 스킬 변경 로그
- prompt 버전 관리 규칙

### 4.4. P1: AI 작업 실패 사례 로그 만들기

목표:

- 반복 실패를 개인 기억이 아니라 프로젝트 자산으로 남긴다.

작업:

- 실패 사례를 저장할 문서 위치를 정한다.
- 최소 기록 형식을 정의한다.
  - 작업 요약
  - 실패 증상
  - 원인
  - 실패 카테고리
  - 재발 방지 스킬/문서 수정
  - 관련 테스트
- 실패 사례가 3회 이상 반복되면 스킬이나 onboarding 문서를 수정하는 규칙을 둔다.

산출물:

- AI 작업 실패 사례 로그
- 반복 실패 대응 규칙

### 4.5. P2: 로컬/규칙 기반 PR 위험 태깅 자동화

목표:

- 외부 AI 코드 리뷰 API 없이, 변경 파일에 따라 필요한 리뷰가 자동으로 드러나게 한다.

범위:

- 포함:
  - 변경 파일 패턴 분석
  - 필요한 스킬/검증 명령 추천
  - CI summary 또는 로컬 리포트 출력
  - 보안/성능/하네스 변경 감지
- 제외:
  - 외부 LLM API 기반 코드 리뷰
  - PR inline comment bot
  - 자동 승인/자동 merge
  - 코드 의미를 LLM이 판정하는 고비용 리뷰

작업:

- 파일 패턴별 리뷰 게이트를 정의한다.
  - `src/app/api/**`, `src/domain/**`: backend, quality, security
  - `src/api-spec/openapi.yaml`: backend, frontend, quality
  - `src/lib/server/**`: backend, security
  - `src/app/**`, `src/components/**`, `src/hooks/**`: frontend, quality
  - `src/lib/bridge.ts`, `src/types/bridge.ts`, app webview docs: frontend-webview, security
  - `.agents/**`, `AGENTS.md`, `codex.md`: harness-security
  - storage/query/aggregation 변경: performance
- GitHub Actions에서 변경 파일 목록을 읽어 체크리스트를 출력하는 workflow를 검토한다.
- 처음에는 comment bot보다 CI summary 출력부터 시작한다.
- 로컬에서 같은 결과를 볼 수 있도록 `yarn` script 또는 작은 Node 스크립트를 검토한다.

산출물:

- 변경 파일 패턴별 review gate mapping
- 로컬 review checklist 스크립트 초안
- PR review checklist workflow 초안

### 4.6. P2: Context Packet 표준화

목표:

- Agent가 매번 탐색 비용을 반복하지 않도록 작업별 필수 context를 묶는다. 무거운 KI(Knowledge Item) 시스템 도입 전, 가벼운 "Working Set" 문서로 시작한다.

작업:

- 작업 유형별 context packet을 정의한다.
  - **Backend API Change Packet**
    - 먼저 볼 파일: `src/api-spec/openapi.yaml`, `src/db/schema.ts`, 해당 도메인 로직
    - 금지 사항: OpenAPI 갱신 없이 라우터 먼저 수정 금지, `drizzle-kit push` 로컬 수동 적용 금지
    - 검증 명령: `yarn gen:api`, `yarn test:backend`
  - **Frontend API Integration Packet**
    - 먼저 볼 파일: `src/api/generated/` 훅, 해당 도메인의 `_hooks/`, UI 컴포넌트
    - 금지 사항: 페이지 컴포넌트 내부에 복잡한 상태/폼 로직 혼재 금지
    - 검증 명령: `yarn test:frontend`, `yarn tsc --noEmit`
  - **Billing Entitlement Packet**
    - 먼저 볼 파일: 결제 도메인 로직, 스키마, 권한 체크 서비스
    - 금지 사항: 권한 우회 가능성 있는 느슨한 체크 금지, 보안 승인 전 배포 금지
    - 검증 명령: `yarn test:billing` (또는 관련 테스트), `dowin-security-check`
  - **WebView Bridge Packet**
    - 먼저 볼 파일: `src/lib/bridge.ts`, `src/types/bridge.ts`, `docs/dev/app-webview/`
    - 금지 사항: 모바일 fallback 검증 없이 bridge 호출 금지
  - **Product Update Packet**
    - 먼저 볼 파일: `src/content/product-updates.ts`, `src/lib/product-updates.ts`, 업데이트 페이지 또는 대시보드 카드 영향 범위
  - **Harness/Security Packet**
    - 먼저 볼 파일: `AGENTS.md`, `.agents/skills/*`, `codex.md`
- 각 packet 문서를 `docs/planning/` 하위 또는 별도 `docs/dev/common/context-packets.md`에 명문화한다.
- `dowin-orchestrator`가 작업 분류 후 가장 먼저 이 packet을 로드하여 탐색(rg) 범위를 좁히도록 JIT 탐색 규칙에 추가한다.

산출물:

- Context packet 상세 문서 (`docs/dev/common/context-packets.md` 등)
- Orchestrator 스킬 및 도메인 스킬 내 JIT 규칙 갱신 완료

## 5. 권장 실행 순서

1. AI 작업 eval 기준과 golden case를 먼저 만든다.
2. `quality-check` 스킬에 AI 산출물 검토 항목을 추가한다.
3. 스킬 변경 로그와 실패 사례 로그를 만든다.
4. 외부 API 없는 변경 파일 기반 PR 위험 태깅을 도입한다.
5. Context packet을 표준화한다.
6. MCP/tool 운영 기준을 정리한다.
7. 제품 내부 AI 기능은 마지막에 현재 유료 가치와 맞춰 다시 기획한다.

## 6. 운영 원칙

Dowin에서 AI를 활용한다는 것은 단순히 Codex나 Cursor 같은 도구를 쓰는 것이 아니다.

현재는 저장소 문서, 스킬, 검증 명령을 통해 AI가 일할 수 있는 context와 harness를 만들고 있다. 다음 단계는 AI 산출물을 eval로 측정하고, 실패 사례를 skill/prompt 개선에 되먹임하며, PR 단계에서 필요한 리뷰 체크리스트를 로컬/규칙 기반으로 자동화하는 것이다.

즉, 목표는 AI에게 모든 작업을 맡기는 것이 아니라, 사람이 목표와 품질 기준을 설계하고 AI가 반복 작업을 안정적으로 수행하도록 개발 환경을 만드는 것이다.

외부 LLM/API 기반 PR 코드 리뷰는 비용, 노이즈, 보안, 판단 책임 문제가 있으므로 현재 단계에서는 보류한다. 먼저 로컬에서 재현 가능한 규칙 기반 자동화와 문서화된 eval 기준을 갖춘 뒤 필요성을 다시 판단한다.

## 7. 완료 기준

- AI 작업 eval 기준 문서가 존재한다.
- Golden case가 최소 5개 정의되어 있다.
- `dowin-quality-check`가 AI 산출물 특화 실패를 점검한다.
- 스킬 변경 이력과 실패 사례 로그가 운영된다.
- PR 변경 범위 기반 리뷰 게이트가 문서화되어 있다.
- MCP/tool 운영 기준이 하네스 보안 기준과 연결되어 있다.
- 제품 내부 AI 기능 후보가 현재 Basic paid value 기준으로 재정렬되어 있다.
