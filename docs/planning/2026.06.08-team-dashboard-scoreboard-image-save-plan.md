# 팀 대시보드 내 점수판 이미지 저장 기능 설계

## 1. 배경 및 맥락 (Background & Context)

- 현재 사용자는 팀 대시보드에서 자신의 점수판을 외부 채널에 공유하려면 직접 화면을 캡처한다.
- 수동 캡처는 화면 크기, 스크롤 위치, 메모 버튼, 브라우저 UI 노출 여부에 따라 결과물이 매번 달라진다.
- 이 기능의 1차 목적은 공유 채널을 직접 붙이는 것이 아니라, 사용자가 Slack, 카카오톡, 노션, 회고 문서 등에 바로 올릴 수 있는 정돈된 주간 점수판 이미지를 만드는 것이다.
- Dowin의 제품 가치는 `기록` 자체보다 `상태 파악 -> 개입 -> 후속 확인`의 운영 루프에 있으므로, 이미지 저장 결과물은 개인 성과 과시보다 주간 상태를 빠르게 전달하는 산출물이어야 한다.

## 2. 문제 정의 (Problem)

팀원이 자신의 주간 실행 현황을 팀 회의나 채팅방에 공유하고 싶을 때, 현재는 임의 캡처에 의존한다. 이로 인해 다음 문제가 생긴다.

- 캡처 품질이 사용자마다 다르고, Dowin 점수판처럼 보이지 않는다.
- 모바일에서는 필요한 영역만 깔끔하게 캡처하기 어렵다.
- 메모 버튼, 여백, 스크롤 상태처럼 공유에 필요 없는 UI가 함께 노출된다.
- 팀원이 공유할 때 어떤 주차의 어떤 점수판인지 맥락이 부족하다.

## 3. 목표 (Goals)

- 팀 대시보드에서 현재 로그인 사용자의 점수판만 1클릭으로 PNG 저장할 수 있다.
- 저장 이미지는 화면 캡처가 아니라 이미지 전용 레이아웃으로 생성한다.
- 저장 이미지에는 주차, 사용자, 핵심 목표, 성공 기준, 주간/월간 달성률, 액션 아이템별 기록 상태가 들어간다.
- 팀 메모, 다른 팀원 정보, 관리 버튼 등 공유에 불필요하거나 민감한 정보는 이미지에 포함하지 않는다.
- API 변경 없이 현재 `GET /api/workspaces/{workspaceId}/dashboard/team` 응답과 팀 대시보드 상태만 사용한다.

## 4. MVP 범위 (MVP Scope)

### 4.1. 진입 위치

- 대상 화면: `src/app/[locale]/(protected)/[workspaceId]/dashboard/page.tsx`
- 버튼 위치: `팀 주간 점수판` 섹션 헤더 오른쪽 또는 `내 점수판` 카드/테이블 헤더 오른쪽
- 버튼 라벨:
  - ko: `이미지 저장`
  - en: `Save image`
- 버튼 아이콘: `DowinIcon`의 `action-download` 또는 Lucide `Download`

버튼은 현재 사용자에게 활성 점수판이 있을 때만 노출한다.

### 4.2. 저장 대상

- 현재 로그인 사용자(`myUserId`)와 일치하는 `dashboard.members` 항목만 저장 대상이다.
- `member.hasScoreboard !== true` 또는 `member.leadMeasures.length === 0`이면 버튼을 숨기거나 비활성화한다.
- 선택 주차는 현재 팀 대시보드의 `weekDates`, `weekLabel`, `dashboard.weekStart`를 그대로 따른다.

### 4.3. 저장 결과

- 파일 형식: PNG
- 파일명:

```text
dowin-{nickname}-{weekStart}-scoreboard.png
```

- `nickname`은 파일명 안전화를 적용한다.
  - 공백은 `-`
  - 한글은 유지 가능
  - `/`, `\`, `:`, `*`, `?`, `"`, `<`, `>`, `|`는 제거 또는 `-`로 치환
- `weekStart`는 `YYYY-MM-DD` 그대로 사용한다.

## 5. MVP 제외 범위 (Non-Goals)

- 타인 점수판 이미지 저장
- 팀 전체 점수판 한 장으로 저장
- 팀 메모 포함 이미지
- 공개 링크 생성
- 서버 저장 또는 서버 렌더링 이미지 생성
- 카카오톡, Slack, Discord 직접 공유 연동
- Web Share API 기반 파일 공유
- 월간 점수판 이미지 저장
- 과거 주차 여러 개를 한 번에 저장

## 6. 현재 구현 기준 (Current Implementation Baseline)

관련 구현 파일:

- `src/app/[locale]/(protected)/[workspaceId]/dashboard/page.tsx`
  - 팀 대시보드 페이지
  - `useTeamDashboard`로 `dashboard`, `weekDates`, `weekLabel` 조회
  - `useGetUsersMe`로 현재 사용자 정보를 조회
  - `membersWithScoreboard`를 계산해 `WeeklyTable` 렌더링
- `src/app/[locale]/(protected)/[workspaceId]/dashboard/_hooks/useTeamDashboard.ts`
  - 선택 주차 상태와 팀 대시보드 조회를 관리
  - `selectedDate`, `weekDates`, `weekLabel`, `movePeriod`, `resetToToday` 제공
- `src/app/[locale]/(protected)/[workspaceId]/dashboard/_components/WeeklyTable.tsx`
  - 멤버별 주간 점수판 UI 렌더링
  - 데스크톱 테이블과 모바일 카드 레이아웃을 모두 포함
  - 메모 버튼과 메모 레일도 함께 포함하므로 이미지 저장 대상 컴포넌트로 직접 캡처하지 않는다.
- `src/messages/ko.json`, `src/messages/en.json`
  - `Dashboard` namespace에 버튼, 토스트, 에러 문구를 추가한다.

## 7. 구현 전략 (Implementation Strategy)

### 7.1. DOM 캡처가 아니라 이미지 전용 컴포넌트 사용

`WeeklyTable`을 그대로 캡처하지 않는다.

이유:

- 화면용 컴포넌트에는 메모 버튼, 레일 상태, 모바일/데스크톱 분기, overflow 처리, 현재 viewport 영향이 섞여 있다.
- 공유 이미지는 항상 같은 비율과 폭을 가져야 한다.
- 긴 텍스트, count 지표, 주차 라벨이 이미지 안에서 예측 가능하게 배치돼야 한다.

새 컴포넌트를 추가한다.

```text
src/app/[locale]/(protected)/[workspaceId]/dashboard/_components/ScoreboardImageCard.tsx
```

역할:

- 저장 이미지에 들어갈 순수 프레젠테이션 컴포넌트
- 버튼, 입력, 메모, 링크, hover 상태 없음
- 고정 폭 기반 레이아웃 사용
- 같은 데이터로 브라우저 화면과 이미지 생성 라이브러리 양쪽에서 안정적으로 렌더링

권장 props:

```ts
import type { TeamDashboardMember } from "@/api/generated/dowin.schemas";

type ScoreboardImageCardProps = {
  workspaceName: string;
  member: TeamDashboardMember;
  weekDates: string[];
  weekLabel: string;
  generatedAtLabel: string;
};
```

### 7.2. 이미지 저장 훅 분리

새 훅을 추가한다.

```text
src/app/[locale]/(protected)/[workspaceId]/dashboard/_hooks/useScoreboardImageExport.ts
```

역할:

- DOM node ref를 PNG로 변환
- 파일명 생성
- 다운로드 anchor 생성 및 정리
- pending/error 상태 관리
- toast 호출은 페이지 또는 훅 중 한 곳으로 일관되게 둔다.

권장 반환값:

```ts
type UseScoreboardImageExportResult = {
  exportRef: React.RefObject<HTMLDivElement | null>;
  isExporting: boolean;
  saveImage: () => Promise<void>;
};
```

훅 입력:

```ts
type UseScoreboardImageExportInput = {
  member: TeamDashboardMember | null;
  weekStart: string | null;
  onSuccess?: () => void;
  onError?: () => void;
};
```

### 7.3. 이미지 생성 라이브러리 검토

MVP는 클라이언트에서 DOM을 PNG로 변환한다.

검토 후보:

| 후보 | 방식 | 장점 | 리스크 | 판단 |
|------|------|------|--------|------|
| `html-to-image` | DOM clone -> SVG `foreignObject` -> canvas/data URL | React 예제가 단순하고 `toPng`, `toBlob`, `pixelRatio`, `backgroundColor`, `cacheBust` 옵션을 바로 쓸 수 있다. `dom-to-image` fork라 유지보수성과 기능 확장을 기대할 수 있다. | Safari/WebView 계열에서는 `foreignObject` 보안 정책과 스타일/폰트 embedding 이슈를 실제 확인해야 한다. | MVP 1순위 후보 |
| `modern-screenshot` | DOM clone -> SVG/canvas, `html-to-image` fork 계열 | 최근 릴리스와 TypeScript 친화 API가 있고, worker/context 재사용 옵션이 있다. 반복 캡처나 복잡한 이미지 저장이 커지면 유리할 수 있다. | MVP는 1회 저장 중심이라 worker/context 장점이 크지 않다. `html-to-image`보다 프로젝트 내 사용 사례가 더 낯설 수 있다. | 2순위 후보. `html-to-image`가 Tailwind 4/WebView에서 깨지면 대체 검토 |
| `html2canvas` | DOM을 직접 해석해 canvas에 재구성 | 오래된 라이브러리이고 브라우저 호환 범위가 넓다. SVG `foreignObject` 의존이 낮다. | 실제 스크린샷이 아니라 DOM 속성을 읽어 재구성하므로 CSS 지원 범위 밖 스타일은 틀어질 수 있다. Tailwind 4의 최신 CSS 표현과 복잡한 스타일에서 검증 부담이 크다. | 이번 MVP에는 비권장 |
| `dom-to-image` | DOM clone -> SVG `foreignObject` -> canvas/data URL | 원형 라이브러리이고 API가 단순하다. | 원본 프로젝트는 오래된 API/브라우저 기준에 가깝고, README에도 Safari 제약이 명시돼 있다. | 비권장 |
| `dom-to-image-more` | `dom-to-image` fork | 원본보다 개선된 fork 후보. | 타입/유지보수 상태를 구현 전에 별도 검증해야 한다. 현재 문서 기준으로 1순위로 둘 근거가 부족하다. | 보류 |
| 서버 렌더링 이미지 생성 (`satori`, `sharp`, headless browser 등) | 서버에서 이미지 생성 | 결과물 통제가 가장 좋고 WebView 다운로드 문제를 우회할 수 있다. | Cloudflare Worker 런타임, bundle size, CPU time, 폰트/이미지 자원 관리, API 계약 추가가 필요하다. MVP 범위보다 크다. | Post-MVP |

MVP 권장 후보:

- `html-to-image`

선택 이유:

- 브라우저 DOM을 PNG data URL로 변환하기 쉽다.
- 서버 API를 추가하지 않아도 된다.
- 현재 기능 범위에서는 복잡한 서버 렌더링 이미지 생성보다 구현 비용이 낮다.
- `ScoreboardImageCard`를 이미지 전용 DOM으로 설계하면 `html2canvas`보다 CSS 해석 차이를 줄이고, 서버 렌더링보다 빠르게 MVP를 검증할 수 있다.

구현 전 확인:

- `yarn add html-to-image`로 의존성을 추가한다.
- Next.js client component에서만 import한다.
- 빌드 시 SSR 경로에서 `window` 접근 문제가 생기지 않도록 훅 내부 또는 동적 import로 사용한다.
- 설치 후 실제 브라우저에서 Tailwind 4 스타일, 폰트, `UserAvatar`, 긴 텍스트, COUNT 지표를 캡처해 확인한다.

권장 사용 방식:

```ts
const { toPng } = await import("html-to-image");
const dataUrl = await toPng(node, {
  cacheBust: true,
  pixelRatio: 2,
  backgroundColor: "#f4f4f5",
});
```

주의:

- `pixelRatio`는 2를 기본값으로 둔다.
- 이미지 카드의 루트 배경색을 명시한다.
- 외부 이미지 URL을 쓰지 않는다. `UserAvatar`가 내부 CSS/문자 기반 fallback을 안정적으로 렌더링하는지 확인한다.
- 폰트 로딩 직후 저장하면 깨질 수 있으므로 `document.fonts?.ready`를 기다린 뒤 변환한다.
- WebView/Safari에서 `html-to-image` 결과가 깨지면 1차 대체 후보는 `modern-screenshot`, 2차 대체 후보는 `html2canvas`다.

## 8. UI 설계 (UI Design)

### 8.1. 버튼 배치

`DashboardPage`에서 현재 사용자 멤버를 계산한다.

```ts
const myMember =
  membersWithScoreboard.find((member) => member.userId === myUserId) ?? null;
```

`팀 주간 점수판` 헤더 근처에 저장 버튼을 배치한다.

- 버튼은 `myMember != null`일 때만 노출한다.
- `isExporting` 중에는 disabled 처리한다.
- pending 라벨:
  - ko: `저장 중...`
  - en: `Saving...`

권장 구조:

```tsx
<div className="flex items-center justify-between gap-3 px-1">
  <SectionHeader title={t("teamWeeklyScoreboard")} />
  {myMember ? (
    <Button type="button" onClick={saveImage} disabled={isExporting}>
      <DowinIcon name="action-download" size="14px" />
      {isExporting ? t("savingImage") : t("saveScoreboardImage")}
    </Button>
  ) : null}
</div>
```

### 8.2. 숨김 렌더링 영역

이미지 전용 컴포넌트는 화면에 보이지 않게 렌더링하되, DOM 측정과 스타일 계산이 가능해야 한다.

권장 방식:

```tsx
<div
  aria-hidden="true"
  className="pointer-events-none fixed left-[-10000px] top-0"
>
  <div ref={exportRef}>
    {myMember ? (
      <ScoreboardImageCard
        workspaceName={dashboard.workspaceName ?? "Dowin"}
        member={myMember}
        weekDates={weekDates}
        weekLabel={weekLabel}
        generatedAtLabel={t("imageGeneratedAt")}
      />
    ) : null}
  </div>
</div>
```

주의:

- `display: none`을 쓰면 캡처 라이브러리가 실제 크기와 스타일을 계산하지 못할 수 있다.
- 숨김 영역은 viewport 바깥에 fixed 배치한다.
- 이미지 카드의 최상위 폭은 고정한다.

## 9. 이미지 카드 레이아웃 명세

### 9.1. 캔버스 크기

- 기본 폭: `1080px`
- 권장 최소 높이: 콘텐츠 기반 자동 높이
- 루트 padding: `48px`
- 배경: `#f4f4f5` 또는 현재 대시보드 배경과 맞는 zinc 계열
- 내부 카드 radius: `24px` 이하

Tailwind 예시:

```tsx
<div className="w-[1080px] bg-zinc-100 p-12 text-zinc-950">
  <div className="rounded-[24px] bg-white p-10">...</div>
</div>
```

### 9.2. 포함 정보

상단:

- Dowin 로고 또는 `Dowin` 텍스트
- `팀 주간 점수판`
- 워크스페이스명
- 주차 라벨

사용자/목표 요약:

- 사용자 아바타
- 닉네임
- `나` 배지
- 핵심 목표(`member.goalName`)
- 성공 기준(`member.lagMeasure`)

성과 요약:

- 주간 달성률: `member.weeklyAchievementRate` 또는 API 응답의 현재 필드명
- 월간 달성률: `member.monthlyAchievementRate` 또는 API 응답의 현재 필드명
- 달성률 필드명이 nullable이면 `0` 또는 `-` 표기 규칙을 명시적으로 적용한다.

액션 아이템 표:

- 컬럼: `액션 아이템`, `월`, `화`, `수`, `목`, `금`, `토`, `일`, `달성`
- `BOOLEAN` 기록:
  - 달성: `○`
  - 미달성/미기록: 빈 칸
- `COUNT` 기록:
  - count가 있으면 `{count}/{dailyTargetCount}`
  - count가 없으면 빈 칸
  - achieved이면 녹색 계열, 일부 기록만 있으면 zinc 계열
- 달성 컬럼:
  - `{achieved}/{total}`
  - 작은 progress bar

하단:

- `Dowin`
- 생성 시각 또는 저장 주차 맥락
- 예: `2026.06.08 저장`

### 9.3. 긴 텍스트 처리

- 핵심 목표와 성공 기준은 최대 2줄까지 허용한다.
- 액션 아이템명은 최대 2줄까지 허용한다.
- 줄임 처리만으로 의미가 사라지지 않도록 이미지 카드에서는 화면 테이블보다 더 넓은 첫 컬럼을 사용한다.
- 긴 단어가 레이아웃을 깨지 않도록 `break-words`를 적용한다.

### 9.4. 색상 원칙

- 화면과 같은 dominant zinc/white 기반을 유지하되, 달성 상태만 green/primary로 강조한다.
- 단일 보라/파랑 계열로만 읽히는 이미지가 되지 않게 배경, 카드, 텍스트, 성공 색의 대비를 분리한다.
- 이미지 저장물은 외부 채팅방에서 볼 때도 읽혀야 하므로 작은 텍스트는 `text-zinc-600`보다 흐리게 두지 않는다.

## 10. 데이터 매핑 규칙

계산 helper는 컴포넌트 내부에 흩뿌리지 말고 별도 lib로 분리한다.

```text

```

권장 helper:

```ts
export function getSafeImageFilename(input: {

export function getLeadMeasureProgress(
```

테스트 대상:

- `total`이 0일 때 rate가 0이 되는지
- `achieved > total`인 비정상 데이터가 들어와도 progress width가 100%를 넘지 않는지

## 12. 상태 및 에러 처리

### 12.1. 성공

- PNG 다운로드를 시작한다.
- 성공 toast:
  - ko: `점수판 이미지를 저장했습니다.`
  - en: `Scoreboard image saved.`

### 12.2. 실패

실패 케이스:

- `myMember`가 없다.
- export ref가 아직 없다.
- 이미지 변환 라이브러리 호출 실패
- 다운로드 anchor 동작 실패
- 폰트 또는 스타일 계산 실패

처리:

- 버튼 pending 해제
- 실패 toast 표시
- 콘솔에는 원인 로깅 가능
- API 재호출은 하지 않는다.

### 12.3. 중복 클릭 방지

- `isExporting` 중 버튼 disabled
- 같은 요청이 중복 실행되지 않게 `saveImage` 초반에 guard를 둔다.

## 13. WebView 및 브라우저 처리

MVP는 브라우저 다운로드 anchor를 기본으로 한다.

```ts
const link = document.createElement("a");
link.href = dataUrl;
link.download = filename;
link.click();
```

주의:

- 앱 WebView에서 `download` attribute가 무시될 수 있다.
- 이 MVP에서는 직접 네이티브 공유를 붙이지 않는다.
- 구현 후 앱 환경에서 다운로드가 동작하지 않으면 `frontend-webview` 단계에서 별도 bridge 메서드 또는 Web Share file fallback을 후속 검토한다.

후속 후보:

- data URL을 Blob/File로 변환한 뒤 `navigator.share({ files })` 지원 브라우저에서 공유
- native bridge에 이미지 저장/공유 메서드 추가

단, 후속 후보는 MVP 구현 범위가 아니다.

## 14. 접근성 및 UX 세부 규칙

- 버튼은 `button` 요소로 제공한다.
- 아이콘만 쓰지 말고 텍스트 라벨을 함께 노출한다.
- pending 상태에서는 버튼 텍스트가 `저장 중...`으로 바뀐다.
- 숨김 이미지 카드 영역은 `aria-hidden="true"`로 둔다.
- 이미지 저장 컴포넌트 안의 텍스트는 실제 화면 독해 대상이 아니므로 focusable 요소를 넣지 않는다.

## 16. 검증 계획 (Verification)

필수:

```bash
yarn tsc --noEmit
yarn lint
yarn test --run src/app/[locale]/(protected)/[workspaceId]/dashboard/_lib/scoreboard-image.test.ts
```

수동 확인:

- 현재 사용자에게 활성 점수판이 있을 때 `이미지 저장` 버튼이 보인다.
- 현재 사용자에게 점수판이 없으면 버튼이 보이지 않는다.
- 저장된 PNG에 다른 팀원의 정보와 메모가 포함되지 않는다.
- 데스크톱 브라우저에서 PNG가 다운로드된다.
- 모바일 viewport에서도 버튼 텍스트가 깨지지 않는다.
- 긴 핵심 목표, 긴 성공 기준, 긴 액션 아이템명이 이미지 레이아웃을 깨지 않는다.
- `BOOLEAN` 지표와 `COUNT` 지표가 모두 올바르게 표시된다.
- 과거 주차로 이동한 뒤 저장하면 파일명과 이미지 주차가 해당 주차로 나온다.

선택:

```bash
yarn test --run
```

이미지 저장은 브라우저 동작 의존성이 있으므로, 가능하면 실제 브라우저에서 한 번 저장 결과물을 확인한다.

## 17. 성공 및 실패 기준 (Success & Failure Criteria)

### 성공

- 사용자가 팀 대시보드에서 자신의 점수판 이미지를 1클릭으로 저장한다.
- 저장 이미지는 캡처처럼 보이지 않고, Dowin 주간 점수판 산출물처럼 읽힌다.
- 이미지에 주차와 사용자 맥락이 명확하다.
- 메모, 버튼, 타인 정보가 이미지에 섞이지 않는다.
- 구현이 API 변경 없이 프론트엔드 범위에서 끝난다.

### 실패

- 화면용 `WeeklyTable` 캡처에 의존해 viewport마다 이미지 품질이 달라진다.
- 저장 이미지에 다른 팀원 정보나 메모가 들어간다.
- WebView에서 동작하지 않는 공유 기능까지 MVP에 포함해 범위가 커진다.
- 이미지 생성 실패 시 버튼이 pending 상태에 갇힌다.
- 긴 텍스트나 count 지표 때문에 이미지 레이아웃이 깨진다.
