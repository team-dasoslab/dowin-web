# 선행지표 태그 확장안

## 1. 배경 및 문제 정의

- 현재 선행지표는 이름, 주기, 목표 횟수만으로 관리되어 비슷한 성격의 실행 항목을 묶어 보기 어렵다.
- 사용자가 점수판을 오래 운영할수록 "운동", "집중", "회고", "건강"처럼 행동 카테고리 단위로 정리하고 싶은 요구가 생긴다.
- 태그가 없으면 회고, 필터, export, 향후 분석 기능이 모두 지표명 문자열에 의존하게 되어 확장성이 떨어진다.

## 2. 왜 지금 필요한가

- 현재 `dashboard/my` 기간 탐색, export, 실행 가이드 같은 기능이 이미 선행지표 단위 읽기 경험을 넓히고 있다.
- 이 시점에 태그를 붙이면 이후 분석/필터 기능을 문자열 파싱 없이 메타데이터 기반으로 확장할 수 있다.

## 3. 범위

### 3.1. MVP

- 태그는 `선행지표별 복수 선택 가능 메타데이터`로 정의한다.
- 태그는 우선 사용자 자유입력이 아니라 `워크스페이스 공용 preset + 필요 시 신규 추가` 구조로 제한한다.
- 선행지표 생성/수정 화면에서 태그를 붙이고, `dashboard/my`와 export에서 태그를 읽기 전용으로 노출한다.
- MVP에서는 태그 기반 필터/집계까지 한 번에 넣지 않는다. 먼저 "붙일 수 있고, 볼 수 있고, export에 실린다"까지만 포함한다.

### 3.2. Post-MVP

- 태그별 필터링, 태그별 달성률 집계, 태그 기반 실행 가이드 추천, 태그 정렬/관리 화면은 후속 범위로 둔다.
- 태그 색상 커스터마이징, 팀 공용 태그 정책, 태그 병합/삭제 도구도 후속으로 미룬다.

## 4. 제품 원칙

- 태그는 4DX의 본질인 "행동 정의"를 흐리지 않아야 하므로, 지표명보다 앞서는 주 식별자가 되면 안 된다.
- 한 지표에 너무 많은 태그를 붙이면 분류보다 장식이 되므로 MVP에서는 최대 3개 제한을 기본 가정으로 둔다.
- 태그는 개인 메모가 아니라 향후 필터/분석에 쓰일 구조화 데이터이므로, 중복/동의어 난립을 막는 쪽으로 설계한다.

## 5. 영향 도메인

- `lead-measure`: 태그 저장, 생성/수정 입력 규칙
- `scoreboard/setup`: 태그 선택 UX
- `dashboard`: 태그 읽기 노출
- `analytics/export`: 태그 포함 export

## 6. 설계 메모

- 현재 Lead Measure 도메인 문서는 `name`, `target_value`, `period`, `status`만 다루고 있어 태그 메타데이터 확장이 필요하다.
- DB는 `lead_measure_tags` 같은 join table 분리가 가장 무난하다. 태그를 단순 CSV 문자열로 넣으면 정규화, 필터, rename 대응이 약해진다.
- 다만 태그 사전까지 별도 엔티티로 둘지, `workspace_tag_presets`를 둘지는 MVP 백엔드 설계 단계에서 확정한다.

## 7. 액션 아이템

- [ ] **[기획]** 선행지표 태그 MVP 규칙 확정
  - 태그 입력 방식을 자유입력으로 열지, 워크스페이스 preset 중심으로 제한할지 결정
  - 태그 최대 개수, 중복 방지, 기본 추천 태그 세트 여부를 정의
  - 태그가 개인 메타인지 워크스페이스 공용 분류인지 소유권 규칙을 확정
- [ ] **[백엔드/API]** 선행지표 태그 계약 추가
  - Lead Measure 생성/수정/조회 응답에 `tags` 필드 추가
  - 태그 validation, workspace 범위 검증, rename/delete 정책을 OpenAPI 기준으로 설계
  - 필요 시 workspace tag preset 조회/생성 API 추가 여부 판단
- [ ] **[프론트엔드]** Setup 및 선행지표 편집 화면에 태그 선택 UX 추가
  - 지표명 입력보다 보조 레벨의 메타데이터로 배치
  - 모바일에서 칩 선택/해제와 신규 태그 추가 흐름이 과하지 않도록 최소화
- [ ] **[프론트엔드]** `dashboard/my` 및 export 화면에 태그 읽기 노출 추가
  - 카드/테이블에서 지표명 아래 보조 정보로 노출
  - export에는 태그를 배열 또는 구분자 문자열로 일관되게 포함
- [ ] **[검증]** 선행지표 태그 회귀 기준 정리
  - 태그 없는 기존 지표가 깨지지 않는지 확인
  - 중복 태그, 최대 개수 초과, 워크스페이스 외부 태그 주입 차단 케이스 확인

## 8. 성공 기준

- 사용자는 선행지표 생성/수정 시 태그를 0개 이상 선택해 지표 성격을 구조적으로 구분할 수 있다.
- 태그가 없어도 기존 점수판/대시보드/기록 흐름은 그대로 동작한다.
- 태그는 `dashboard/my`와 export에서 일관되게 읽히며, 이후 필터/분석 기능이 붙을 수 있는 형태로 저장된다.
- 태그 체계가 지표명 자체를 대체하지 않고, 분류 보조 메타데이터로 안정적으로 동작한다.
