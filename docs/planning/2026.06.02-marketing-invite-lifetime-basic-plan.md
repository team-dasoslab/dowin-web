# 영구 무료 혜택 초대코드 기반 Lifetime Basic 기획

## 1. 배경

Dowin의 다음 마케팅 목표는 곧바로 일반 유료 전환을 넓히는 것이 아니라, 실제 팀 리더/운영자가 제품을 충분히 써보고 높은 밀도의 피드백을 주는 초기 사용군을 확보하는 것이다.

현재 제품 방향은 모든 워크스페이스가 Basic 사용권을 가져야 운영 기능에 접근하는 구조다. 따라서 피드백 모집용 혜택도 사용자 계정 전체가 아니라 **워크스페이스 1개에 한정된 Basic 영구 사용권**으로 설계해야 한다.

## 2. 문제 정의

- 일반 결제 플로우만 열어두면 초기 피드백을 받을 충분한 사용량이 생기기 어렵다.
- 무제한 무료 공개를 열면 제품 가치 검증과 결제 전환 기준이 흐려진다.
- 임의 수동 권한 보정만으로 운영하면 누가 어떤 경로로 들어왔고 얼마나 피드백을 주는지 추적하기 어렵다.
- 초대코드가 외부에 무분별하게 퍼지면 비용, 지원 부하, 데이터 품질 문제가 생긴다.

## 3. 목표

- 운영자가 어드민 콘솔에서 영구 무료 혜택 초대코드를 생성하고 관리한다.
- 초대코드를 사용한 사용자는 결제 없이 워크스페이스 1개를 만들 수 있다.
- 생성된 워크스페이스에는 Basic 기능을 기간 제한 없이 제공한다.
- 초대코드는 사용 횟수 제한과 비활성화 상태를 가진다.
- 각 코드의 사용, 전환, 피드백 상태를 운영자가 추적할 수 있다.

## 4. 비목표

- 사용자 계정 전체에 영구 Basic을 부여하지 않는다.
- 한 초대코드 사용자가 여러 워크스페이스를 무료로 만들 수 있게 하지 않는다.
- 기존 워크스페이스 멤버 초대코드(`workspace_invites`)와 같은 엔티티로 합치지 않는다.
- 공개 랜딩에 무제한 무료 플랜처럼 노출하지 않는다.
- 이 단계에서 추천인/바이럴 리워드 시스템까지 만들지 않는다.

## 5. 핵심 결정

### 5.1. 대상 단위

영구 무료 혜택 초대코드는 **워크스페이스 1개**에만 적용한다.

- 사용자는 코드 1개로 새 워크스페이스 1개를 생성한다.
- 해당 워크스페이스에 `Basic lifetime entitlement`가 부여된다.
- 같은 사용자가 다른 워크스페이스를 만들거나 참여하는 경우, 그 워크스페이스에는 이 혜택이 자동 전파되지 않는다.

이 기준은 현재 Dowin의 권한 모델이 workspace 중심이라는 점과 맞다.

### 5.2. 혜택 정의

사용자-facing 표현은 `Basic 영구 무료 혜택` 또는 `Basic 영구 무료 사용권`으로 둔다.

내부 정의:

- `planCode`: `BASIC`
- `billingStatus`: `ACTIVE`
- `entitlementSource`: `BETA_PROMOTIONAL_GRANT`
- `entitlementEndsAt`: 없음
- `billing portal`: 노출하지 않음
- 결제 갱신/해지/환불 문구: 노출하지 않음

고객에게 보여줄 원칙:

- “마케팅 캠페인으로 들어왔다”보다 “영구 무료 혜택을 받고 있다”는 감각이 먼저 와야 한다.
- billing 화면에는 결제 관리 CTA 대신 혜택 상태와 seat 한도를 보여준다.
- 추천 문구:
  - `Basic 영구 무료 혜택이 적용되어 있어요`
  - `이 워크스페이스는 결제 없이 Basic 기능을 계속 사용할 수 있어요`
  - `최대 10명까지 함께 사용할 수 있어요`

주의:

- “영구 무제한”은 기간 제한이 없다는 뜻으로 제한한다.
- seat, 악용 방지, 약관 위반, 운영 중단 같은 정책 예외까지 무조건 무제한으로 보장한다는 의미로 쓰지 않는다.

### 5.2.1. Entitlement source 명명

내부 source 이름은 `BETA_PROMOTIONAL_GRANT`로 둔다.

이유:

- 이번 혜택의 목적은 일반 할인/쿠폰이 아니라 베타 성격의 초기 피드백 모집이다.
- `MARKETING_GRANT`는 내부적으로 틀리지는 않지만, “마케팅팀이 준 권한”처럼 넓고 모호하게 읽힐 수 있다.
- `PROMOTIONAL_GRANT`는 이벤트/할인/프로모션 전체를 포괄하기 좋지만, 이번 기능의 핵심인 베타 피드백 모집 의도가 약해진다.
- `BETA_PROMOTIONAL_GRANT`는 베타 모집과 프로모션성 영구 혜택을 동시에 표현한다.

주의:

- 이 값은 내부 enum, 분석, source 추적용이다.
- 고객-facing 문구에는 `BETA`, `PROMOTIONAL`, `MARKETING` 같은 내부 용어를 노출하지 않는다.
- 고객에게는 `Basic 영구 무료 혜택`으로만 설명한다.

### 5.3. Seat 정책

초기 정책은 코드별로 `grantedSeatCount`를 가진다.

- 기본값: 10 seats
- 최대값: 10 seats
- 운영자가 코드 생성 시 조정 가능
- 해당 워크스페이스의 `workspace_seat_entitlements.purchased_seat_count`에 반영
- `seat_source`: `BETA_PROMOTIONAL_GRANT`

변경 가능성:

- 코드 생성 전에는 운영자가 `grantedSeatCount`를 수정할 수 있다.
- 코드 사용 후 생성된 워크스페이스의 seat 수는 코드 값을 자동 추종하지 않는다.
- 이미 생성된 워크스페이스의 seat 수를 바꾸려면 어드민 billing/manual entitlement 보정 화면에서 워크스페이스별로 조정한다.
- 코드 자체의 `grantedSeatCount`를 나중에 바꾸더라도 기존 redemption에는 소급 적용하지 않는다.

이유:

- Basic이 seat 기반 권한으로 이동 중이므로, lifetime 권한도 seat cap 없이 열면 나중에 비용과 멤버 제한 정책이 무너진다.
- 고객에게는 “무료이지만 작은 팀 파일럿에 충분한 최대 10명”으로 설명할 수 있다.
- seat 변경을 코드 전체에 소급하면 캠페인 운영과 고객 기대가 흔들리므로, 사용된 워크스페이스별 보정으로 분리한다.

## 6. 제품 흐름

### 6.1. 사용자 흐름

1. 사용자가 영구 무료 혜택 초대코드를 받는다.
2. 회원가입 또는 로그인한다.
3. 워크스페이스 없음 상태에서 `초대코드로 Basic 시작하기` 또는 동등한 진입을 선택한다.
4. 코드와 워크스페이스 이름을 입력한다.
5. 서버가 코드 사용 가능 여부를 검증한다.
6. 새 워크스페이스를 생성하고, 해당 워크스페이스에 Basic lifetime entitlement를 부여한다.
7. 사용자는 결제 없이 대시보드/설정/기록 등 운영 기능에 접근한다.
8. 제품 안 또는 운영 메시지로 피드백 요청을 받는다.

### 6.1.1. 진입 방식 후보

사용자 흐름은 아직 논의가 필요하다. 후보는 세 가지다.

#### 후보 A. 회원가입 후 워크스페이스 생성 단계에서 코드 입력

흐름:

1. 회원가입/로그인
2. 워크스페이스 없음 화면
3. `Basic 결제 시작`과 별도로 `초대코드로 영구 무료 혜택 받기` 선택
4. 코드 입력 + 워크스페이스 생성

장점:

- 기존 auth와 workspace creation 흐름을 크게 흔들지 않는다.
- 계정 생성 실패와 코드 redemption 실패를 분리할 수 있다.
- 이미 로그인한 사용자도 같은 흐름을 쓸 수 있다.

단점:

- 사용자가 코드를 들고 왔다는 맥락이 회원가입 화면에서 바로 보이지 않는다.

권장도:

- MVP 1순위.

1순위로 보는 이유:

- 이번 혜택은 사용자 계정이 아니라 워크스페이스 1개에 적용된다.
- 따라서 코드 검증과 entitlement 부여는 계정 생성보다 워크스페이스 생성 시점에 붙는 것이 도메인 책임에 맞다.
- 회원가입 단계에서 코드를 처리하면 auth 도메인이 workspace entitlement 정책을 알아야 하고, 가입 성공 후 workspace 생성 실패 같은 중간 상태가 복잡해진다.
- 워크스페이스 생성 단계에서 처리하면 계정 생성은 계정 생성대로 완료되고, 코드 실패 시에도 사용자는 다른 코드 입력, 일반 Basic 결제, 문의로 복구할 수 있다.
- 이미 가입한 사용자도 동일한 흐름을 사용할 수 있어 중복 구현이 줄어든다.

#### 후보 B. 회원가입 단계에서 코드 입력

흐름:

1. 회원가입 화면에서 초대코드 입력
2. 계정 생성
3. 워크스페이스 생성 시 자동 적용

장점:

- 사용자가 “나는 특별 혜택으로 들어왔다”는 확신을 가입 단계부터 가진다.

단점:

- auth와 workspace entitlement 책임이 섞인다.
- 가입은 성공했지만 workspace 생성은 실패하는 중간 상태를 더 세심하게 처리해야 한다.
- 이미 계정이 있는 사용자의 redemption 흐름을 별도로 또 만들어야 한다.

권장도:

- Post-MVP 또는 링크 기반 진입이 안정화된 뒤 검토.

#### 후보 C. 초대코드가 포함된 링크로 진입

흐름:

1. 사용자가 `/?promoCode=...` 또는 `/invite/lifetime-basic/{code}` 링크로 접근
2. 로그인/회원가입
3. 가입 후 워크스페이스 생성 단계에 코드가 자동으로 채워짐
4. 사용자가 확인 후 적용

장점:

- 마케팅 배포에 가장 좋다.
- 코드 오타가 줄고 캠페인별 attribution이 쉬워진다.
- 회원가입 전후 경험을 이어줄 수 있다.

단점:

- 코드가 URL로 유출되기 쉽다.
- URL 저장 위치, 만료, 로그인 redirect 이후 복원 처리가 필요하다.

권장도:

- Post-MVP.
- MVP에서는 워크스페이스 생성 단계에서 직접 코드를 입력하게 하므로 별도 초대 링크를 두지 않는다.
- 캠페인 운영 중 코드 입력 마찰이나 오타 문제가 실제로 확인되면 링크 prefill을 추가한다.

### 6.1.2. 권장 MVP 사용자 흐름

MVP는 후보 A만 구현한다.

권장 흐름:

1. 사용자가 초대코드를 받는다.
2. 로그인/회원가입 후 워크스페이스 없음 화면으로 이동한다.
3. `영구 무료 혜택 코드로 시작` CTA를 누른다.
4. 초대코드와 워크스페이스 이름을 입력한다.
5. 서버가 새 워크스페이스 1개에 Basic 영구 무료 혜택을 부여한다.

이 방식은 auth 도메인을 깨지 않고 workspace creation 단계에서만 entitlement를 처리한다.

### 6.2. 운영자 흐름

운영자는 어드민 콘솔에서 아래를 수행한다.

- 영구 무료 혜택 초대코드 생성
- 코드 이름/캠페인/메모 입력
- 사용 횟수 제한 설정
- 부여 seat 수 설정
- 활성/비활성 전환
- 사용된 워크스페이스 목록 확인
- 피드백 요청 상태 또는 후속 메모 기록

## 7. 코드 정책

### 7.1. 필수 필드

영구 무료 혜택 초대코드 엔티티는 기존 `workspace_invites`와 분리한다.

권장 테이블명:

- `marketing_invite_codes`

필드 초안:

- `id`
- `code`
- `campaignName`
- `description`
- `maxUses`
- `usedCount`
- `grantedSeatCount`
- `entitlementSource`: `BETA_PROMOTIONAL_GRANT`
- `status`: `ACTIVE | INACTIVE`
- `createdByAdminUserId`
- `createdAt`
- `updatedAt`

사용 이력 테이블:

- `marketing_invite_redemptions`

필드 초안:

- `id`
- `marketingInviteCodeId`
- `redeemedByUserId`
- `workspaceId`
- `redeemedAt`
- `feedbackStatus`: `NOT_REQUESTED | REQUESTED | RECEIVED | DROPPED`
- `acquisitionSource`: `MARKETING_INVITE`
- `campaignName`
- `operatorNote`

### 7.2. 사용 제한

- 코드별 `maxUses`를 초과하면 더 이상 사용할 수 없다.
- `INACTIVE` 상태 코드는 사용할 수 없다.
- 같은 사용자가 같은 마케팅 초대코드를 여러 번 사용할 수 없다.
- 한 redemption은 반드시 하나의 workspace와 연결된다.
- 코드 사용과 workspace 생성, entitlement 부여, redemption 기록은 하나의 트랜잭션으로 처리한다.

### 7.3. 남용 방지

MVP에서 최소로 막아야 할 것:

- 사용 완료된 코드 재사용
- 비활성 코드 사용
- 사용 횟수 초과 코드 사용
- 같은 사용자 중복 사용
- 이미 결제/권한이 있는 워크스페이스에 영구 무료 혜택 코드를 덮어쓰기
- `grantedSeatCount > 10` 코드 생성

Post-MVP 후보:

- 코드 만료 시각
- 허용 이메일 도메인
- 캠페인별 전체 seat 한도
- 운영자 승인 후 활성화
- 의심 사용 패턴 알림

## 8. 기존 시스템과의 관계

### 8.1. 기존 워크스페이스 초대코드와 분리

기존 `workspace_invites`는 이미 존재하는 워크스페이스에 멤버를 합류시키는 코드다.

마케팅 초대코드는 새 워크스페이스 생성과 Basic entitlement 부여를 동시에 처리한다.

따라서 두 코드는 이름이 비슷해도 도메인 책임이 다르다.

- `workspace_invites`: 멤버십 참가
- `marketing_invite_codes`: acquisition + entitlement grant

### 8.2. Billing entitlement와 연결

이 기능은 `docs/planning/2026.05.20-billing-entitlement-and-workspace-architecture-plan.md`의 `entitlementSource` 분리 방향과 맞춰 구현한다.

결정:

- `entitlementSource`는 `BETA_PROMOTIONAL_GRANT`로 한다.
- `seat_source`도 `BETA_PROMOTIONAL_GRANT`로 맞춘다.
- 마케팅 권한 워크스페이스에는 billing portal을 노출하지 않는다.

이유:

- `MANUAL_GRANT`는 운영자가 개별 워크스페이스를 수동 보정한 의미가 강하다.
- `PARTNER`는 제휴/외부 채널 의미가 강하다.
- 이번 기능은 캠페인성 영구 무료 혜택과 베타 피드백 모집이 핵심이므로 `BETA_PROMOTIONAL_GRANT`가 가장 정확하다.

## 9. 어드민 콘솔 범위

### 9.1. MVP

- 목록: 코드, 캠페인명, 상태, 사용량, 부여 seat, 생성자, 생성일
- 생성: 캠페인명, 최대 사용 횟수, seat 수, 설명
- 상세: 사용 워크스페이스, 사용자, 사용일, 피드백 상태, 운영 메모
- 액션: 활성화/비활성화, 코드 기본 seat 수 변경

주의:

- 코드 기본 seat 수 변경은 이후 redemption에만 적용한다.
- 기존 redemption workspace의 seat 수 변경은 워크스페이스별 manual seat adjustment로만 처리한다.

### 9.2. Post-MVP

- 코드 만료일 설정
- 피드백 요청 발송/기록 자동화
- 캠페인별 전환/활성 사용률 대시보드
- code leak 의심 알림
- 특정 코드 redemption 취소 또는 권한 회수

## 10. 피드백 루프

영구 무료 혜택 초대코드의 목적은 무료 사용자가 아니라 피드백 가능한 초기 사용군 확보다.

따라서 redemption 후 아래 이벤트를 추적해야 한다.

- 워크스페이스 생성 완료
- 첫 핵심 목표 생성
- 첫 액션 아이템 생성
- 첫 일일 기록
- 첫 팀원 초대
- 7일 내 재방문
- 피드백 요청 발송
- 피드백 수신

운영자가 봐야 할 최소 상태:

- `초대됨`
- `워크스페이스 생성`
- `핵심 흐름 시작`
- `피드백 요청됨`
- `피드백 수신`
- `이탈 의심`

### 10.1. 마케팅 가입자 이벤트 추적

현재 제품은 클라이언트 `trackEvent` 기반 GA4 이벤트를 이미 사용한다. 마케팅 초대코드 사용자는 별도 이벤트 시스템을 새로 만들기보다, redemption 데이터를 서버 진실 원천으로 두고 기존 이벤트에 acquisition 속성을 붙인다.

서버 진실 원천:

- `marketing_invite_redemptions.workspaceId`
- `marketing_invite_redemptions.redeemedByUserId`
- `marketing_invite_codes.campaignName`
- `marketing_invite_codes.id`

클라이언트 이벤트 공통 속성:

- `acquisition_source`: `MARKETING_INVITE`
- `entitlement_source`: `BETA_PROMOTIONAL_GRANT`
- `marketing_invite_code_id`
- `campaign_name`
- `workspace_id`

추적 대상 이벤트:

- `marketing_invite_redeemed`
- `marketing_workspace_created`
- `marketing_setup_started`
- `marketing_scoreboard_created`
- `marketing_action_item_created`
- `marketing_daily_log_checked`
- `marketing_member_invited`
- `marketing_feedback_requested`
- `marketing_feedback_received`

주의:

- GA4만으로 권한/캠페인 attribution의 진실을 판단하지 않는다.
- 운영자 콘솔의 캠페인별 상태는 DB redemption 테이블 기준으로 계산한다.
- GA4는 퍼널과 행동 분석용으로만 사용한다.

### 10.2. 피드백 수집 방식

피드백 수집은 MVP에서 운영자 직접 수집으로 결정한다.

#### 운영자 직접 수집

장점:

- 초기 고객의 말투, 맥락, 구매 가능성을 깊게 확인할 수 있다.
- 제품 안에서 긴 폼을 만들기 전에 어떤 질문이 유효한지 배울 수 있다.

단점:

- 운영자가 직접 추적해야 하므로 누락 가능성이 있다.

MVP 적용:

- 어드민 콘솔에서 `feedbackStatus`와 `operatorNote`를 수동으로 관리한다.
- 운영자가 직접 받은 피드백 요약을 어드민 콘솔 상세 화면의 `operatorNote`에 기록한다.
- `feedbackStatus`는 운영자가 `NOT_REQUESTED -> REQUESTED -> RECEIVED` 또는 `DROPPED`로 수동 변경한다.
- 피드백 접촉 채널은 운영자가 캠페인별로 관리한다.

#### 시스템 내 수집

장점:

- 누락이 줄고, 피드백 데이터가 제품 DB에 남는다.
- 사용자가 제품 경험 직후 짧게 남기기 쉽다.

단점:

- 질문 설계가 약하면 낮은 품질의 응답만 쌓일 수 있다.
- 초기에는 구현 범위가 늘어난다.

Post-MVP 적용:

- 대시보드 또는 문의 화면에 `초기 사용자 피드백 남기기` 진입점을 추가한다.
- 피드백 응답을 `marketing_invite_redemptions` 또는 별도 `marketing_feedback_responses`와 연결한다.

권장:

- 첫 캠페인은 운영자 직접 수집으로 시작한다.
- 제품 내에는 과한 설문 대신 `피드백 요청됨/수신됨` 상태와 내부 문의 연결 정도만 둔다.
- 5~10개 팀 피드백을 받은 뒤 시스템 내 피드백 질문을 고정한다.

## 12. MVP / Post-MVP 경계

### MVP

- 워크스페이스 1개 lifetime Basic entitlement
- 코드별 사용 횟수 제한
- 코드 비활성화
- 코드별 seat 수 부여, 최대 10 seats
- 어드민 콘솔 생성/목록/상세
- redemption 이력 추적
- 피드백 상태 수동 기록
- 가입/로그인 이후 워크스페이스 생성 단계에서 코드 입력

### Post-MVP

- 코드 만료
- 캠페인 분석 대시보드
- 피드백 요청 자동화
- 초대 링크 기반 코드 prefill
- 이메일/도메인 제한
- 사용 취소/권한 회수 자동화
- 제휴사/크리에이터별 코드 발급

## 13. 성공 기준

- 운영자가 개발자 도움 없이 마케팅 초대코드를 생성하고 비활성화할 수 있다.
- 초대코드가 사용되면 결제 없이 워크스페이스 1개가 Basic 상태로 열린다.
- 코드별 사용 횟수 제한이 정확히 적용된다.
- 마케팅 코드로 생성된 워크스페이스에서는 결제 portal CTA가 보이지 않는다.
- billing 화면에서 `Basic 영구 무료 혜택`과 최대 10명 제한이 명확히 보인다.
- 운영자가 코드별 사용 워크스페이스와 피드백 상태를 확인할 수 있다.
- 초대코드 사용자의 핵심 흐름 시작률과 피드백 수신률을 추적할 수 있다.

## 14. 실패 기준

- 마케팅 코드가 사용자 계정 전체 entitlement처럼 동작한다.
- 코드가 퍼졌을 때 사용량을 즉시 멈출 수 없다.
- 마케팅 코드 워크스페이스가 일반 결제 워크스페이스와 UI/권한 설명에서 구분되지 않는다.
- 최대 10 seats 제한 없이 무제한 멤버 초대가 가능해진다.
- 피드백 수집 없이 무료 워크스페이스만 늘어난다.
