# `/dailylog` 기반 채팅 데일리 로그 입력 기획

> 상태 메모 (2026-04-18): 현재는 **보류(parked)** 상태다. 구현 착수 문서가 아니라, 추후 기록 마찰이 핵심 문제로 확인될 때 다시 검토할 후보안으로 유지한다.

## 1. 배경 및 맥락 (Background & Context)

- 현재 DOWIN의 핵심 조회 경험은 `dashboard/my`, `dashboard`에 있다.
- 점수판 집계, 주간/월간 달성률, 팀 현황 같은 읽기 중심 경험은 이미 웹 대시보드가 담당하고 있다.
- 이번 논의의 핵심은 `집계 조회를 메신저로 옮기는 것`이 아니라, **데일리 로그 입력만 Discord와 Slack에서 더 가볍게 처리하게 하는 것**이다.
- 즉 방향은 아래처럼 정리된다.
  - 조회/리뷰/집계: 웹 대시보드 유지
  - 데일리 로그 입력: Discord/Slack 체크인 스레드에서 처리

## 2. 문제 정의 (Problem Statement)

- 사용자는 매일 DOWIN 대시보드에 들어가서 기록해야 하므로, 기록 시작 마찰이 있다.
- 실제 팀 운영에서는 Discord나 Slack에 이미 머무는 시간이 더 길 수 있다.
- 따라서 `기록 시작`만 채팅 안에서 줄여주면, 웹 대시보드의 장점은 유지하면서 입력 마찰만 낮출 수 있다.

## 3. 핵심 결론 (Recommendation)

### 3.1. 이 아이디어는 가능한가

- 가능하다.
- 다만 처음부터 자유 텍스트 자연어 입력으로 열면 복잡도가 빠르게 커진다.

### 3.1.1. 1차 채널 결정

- **1차 MVP는 Discord부터 시작한다.**
- 이유:
  - 실제 사용자가 Discord를 더 자주 쓰므로 초기 검증 속도가 빠르다.
  - 먼저 자주 쓰는 채널에서 기록 마찰을 줄이는 것이 제품 실험 가치가 크다.
  - Slack까지 동시에 열면 account linking, 이벤트 검증, 운영 로그가 두 세트로 늘어난다.
- 따라서 이 문서의 구현 순서는 `Discord 선행 -> Slack 확장`으로 둔다.
- 다만 최종 목표는 Slack을 버리는 것이 아니라, **같은 입력 모델을 Slack에도 확장 가능하게 설계하는 것**이다.

### 3.2. 복잡도를 낮추는 MVP 원칙

- 자유 텍스트 자연어 파싱은 하지 않는다.
- 아무 채널/아무 스레드에서 입력받지 않는다.
- **사용자가 `/dailylog`로 연 입력 세션에 대한 답장만 허용**한다.
- 기록 날짜는 기본적으로 **오늘만 허용**한다.
- 사용자는 선행지표명을 완전히 입력하지 않는다.
- **봇이 보여준 번호를 입력하면 그 번호를 O로 처리**한다.

### 3.3. 선행지표명 완전 일치 입력은 왜 비추천인가

- 파싱은 쉬워지지만 UX가 거칠다.
- 선행지표명이 길면 입력 피로가 커진다.
- 띄어쓰기, 문장부호, 약간의 표기 차이로 실패가 잦아진다.
- 모바일 Discord/Slack에서 반복 입력하기 불편하다.

따라서 인증 방식은 `완전 동일 텍스트 입력`보다 **번호 입력 방식**이 더 적합하다.

## 4. MVP 사용자 경험 (MVP UX)

### 4.1. 기본 흐름

1. 사용자가 Discord/Slack에서 `/dailylog`를 실행한다.
2. 봇은 오늘 기록 가능한 선행지표 목록을 번호와 함께 보여준다.
3. 사용자는 그 세션의 스레드 또는 지정된 답장 맥락에 번호를 입력한다.
4. 서버는 입력된 번호를 오늘의 완료 항목으로 해석한다.
5. 저장 후 봇이 처리 결과를 답장한다.
6. 집계 결과나 상세 리뷰는 기존처럼 웹 대시보드에서 본다.

### 4.2. 예시

```text
오늘 기록
1. 독서 30분
2. 러닝 5km
3. 영어 회화 20분

입력 예시: 1 3
```

사용자가 `1 3`을 입력하면:

- `1`, `3`은 `O`
- `2`는 `미기록`

## 5. 토글 모델 (Toggle Model)

### 5.1. 권장 방식

- 채팅에서는 버튼식 토글보다 **최신 메시지 전체 덮어쓰기** 모델이 낫다.
- 즉 사용자의 새 입력 한 줄이 `오늘 상태의 최종본`이 된다.

예:

```text
초기 입력: 1 3
결과: 1=O, 2=미기록, 3=O
```

이후 같은 사용자가 같은 `/dailylog` 세션에 다시

```text
1 2 3
```

을 입력하면 결과는 아래처럼 갱신된다.

- `1=O`
- `2=O`
- `3=O`

### 5.2. 이 방식이 나은 이유

- 사용자가 이해하기 쉽다.
- 서버가 현재 상태를 재계산하면 된다.
- 개별 toggle 이벤트를 따로 추적하지 않아도 된다.
- 재입력 시 수정 UX가 단순하다.

## 6. 상태 모델 제안 (State Model)

### 6.1. MVP 권장안

- MVP에서는 **`O / 미기록`만 지원**하는 것이 낫다.
- 입력된 번호는 `O`
- 입력되지 않은 번호는 `미기록`

### 6.2. 왜 `X`는 나중이 나은가

- 웹 대시보드는 `O/X/미기록`을 가질 수 있어도, 채팅 문법까지 `X`를 열면 복잡도가 올라간다.
- `1o 2x 3o` 같은 문법은 가능하지만 초기 학습 비용이 생긴다.
- 먼저 `했으면 번호 입력, 아니면 비워둠` 정도로 시작하는 편이 낫다.

Post-MVP 확장 가능 예시:

- `1o 2x 3o`
- `1+ 2- 3+`

하지만 첫 버전은 열지 않는 편이 좋다.

## 7. 입력 문법 (Input Syntax)

### 7.1. MVP 문법

- 공백 구분 번호 입력
- 예:
  - `1 3`
  - `2`
  - `1 2 3`

### 7.2. 허용 규칙

- 중복 번호는 dedupe
- 범위 밖 번호는 에러
- 숫자가 아닌 값은 에러
- 빈 입력은 허용하지 않거나, 명시적으로 `전체 미기록으로 변경` 여부를 추후 결정

현재는 아래 정책이 더 안전하다.

- 빈 메시지는 저장하지 않음
- 전체 해제는 `none` 같은 별도 키워드로 나중에 열 수 있음

## 8. 문맥 식별 방식 (Context Identification)

### 8.1. 핵심 원칙

- **아무 채널 입력을 파싱하지 않는다.**
- `/dailylog`로 열린 세션에 연결된 메시지만 처리한다.

### 8.2. 이유

- 이 세션이 어느 워크스페이스의 어느 사용자 기록 입력인지 명확해진다.
- 별도 자연어 문맥 추론이 필요 없다.
- 잘못된 채널 오작동을 줄일 수 있다.

### 8.3. 서버가 알아야 하는 것

- provider (`DISCORD` | `SLACK`)
- thread/channel/message 식별자
- 이 입력 세션이 연결된 `workspaceId`
- 이 입력 세션의 대상 `userId`
- 해당 날짜
- 그날 유효한 선행지표 번호 매핑

즉 메시지 본문만 보고 해석하는 것이 아니라, **`/dailylog`로 열린 세션 문맥 위에서 번호를 해석**해야 한다.

## 9. 인증/권한 모델 (Auth / Authorization)

### 9.1. 왜 사용자 연결이 필요한가

- Discord/Slack 사용자와 DOWIN 사용자를 정확히 연결해야 한다.
- 그래야 다른 사람이 대신 기록하거나, 다른 워크스페이스 데이터에 쓰는 문제를 막을 수 있다.

### 9.2. 기본 원칙

- 외부 메신저 사용자와 DOWIN 사용자는 반드시 account linking 해야 한다.
- 메시지 작성자와 기록 대상 사용자가 일치해야 한다.
- 같은 워크스페이스 소속인지도 서버에서 다시 검사해야 한다.

### 9.3. 초기 정책

- 자기 기록만 허용
- 리더가 타인 기록을 대신 입력하는 기능은 MVP 제외

## 10. 현재 DOWIN 규칙과의 연결 (Domain Fit)

- DOWIN의 Daily Log는 이미 날짜/권한/미래 날짜/과거 수정에 대한 규칙이 있다.
- 채팅 입력은 새로운 도메인이 아니라, **기존 Daily Log 규칙 위에 새로운 입력 채널을 하나 더 여는 것**으로 봐야 한다.

따라서 채팅 입력도 아래를 그대로 따라야 한다.

- 미래 날짜 기록 불가
- 허용 날짜 범위 외 수정 불가
- 활성 점수판과 활성 선행지표 기준으로만 처리
- 권한과 워크스페이스 소속 검증 유지

## 11. 필요한 새 개념 (New Concepts)

### 11.1. Daily Log Session

- 하루 단위, 사용자 단위, provider 단위의 기록 입력 문맥

예상 필드:

- `id`
- `provider`
- `externalThreadId`
- `externalChannelId`
- `workspaceId`
- `userId`
- `logDate`
- `status`
- `createdAt`

### 11.2. Daily Log Item Mapping

- 해당 기록 세션에서 번호와 선행지표를 매핑하는 테이블 또는 payload

예상 필드:

- `checkInSessionId`
- `itemNumber`
- `leadMeasureId`
- `labelSnapshot`

이 매핑이 있어야 사용자가 `1 3`만 보내도 어떤 지표인지 정확히 처리할 수 있다.

## 12. 서버 처리 방식 (Processing)

### 12.1. 처리 절차

1. Discord/Slack 이벤트 수신
2. 서명 검증
3. 작성자 외부 ID 조회
4. linked DOWIN user 조회
5. 해당 메시지가 `/dailylog` 세션 답장인지 확인
6. 기록 세션 조회
7. 번호 문법 검증
8. 번호 -> 선행지표 ID 매핑
9. 오늘 상태 전체 재계산
10. 기존 Daily Log 규칙에 맞게 저장
11. 결과 메시지 응답

### 12.2. 저장 방식

- 개별 메시지를 toggle 이벤트로 저장하지 않는다.
- **최신 입력을 최종 상태로 해석**하고, 그 결과를 Daily Log에 반영한다.

## 13. 실패 처리 UX (Failure Handling)

사용자가 실패 원인을 바로 이해할 수 있어야 한다.

예시:

- `4번은 없는 항목이에요. 1~3 사이 번호만 입력해주세요.`
- `이 세션에서는 오늘 기록만 입력할 수 있어요.`
- `DOWIN 계정 연결이 필요해요. /dowin link 를 먼저 진행해주세요.`
- `기록 가능 시간이 지났어요. 자세한 수정은 대시보드에서 해주세요.`

## 14. 복잡도 평가 (Complexity Assessment)

### 14.1. 어디서 복잡해지는가

- 사용자 연결(account linking)
- Discord/Slack 이벤트 서명 검증
- 기록 세션/번호 매핑 저장
- 기존 Daily Log 규칙과의 정확한 결합
- 중복 메시지/재입력 처리

### 14.2. 그래도 할 만한 이유

- 자연어 파싱을 버리면 난이도가 크게 줄어든다.
- 읽기 모델은 웹 대시보드에 그대로 두므로 범위가 제한된다.
- 입력 채널만 추가하는 구조라서 제품 실험 가치가 있다.

### 14.3. 하지 말아야 할 범위

- 자유 자연어 입력
- 아무 채널/아무 스레드 입력 허용
- 타인 대신 기록
- 과거 날짜 자유 수정
- Slack/Discord 동시 풀스코프 구현
- 첫 버전부터 `O/X/미기록` 모두 지원

## 15. 권장 MVP 순서 (Recommended Sequence)

1. Discord account linking 구현
2. Discord `/dailylog` 세션 생성 흐름 구현
3. 번호 입력 파서 구현
4. `O / 미기록` 저장 모델 구현
5. 실패 메시지/성공 메시지 UX 정리
6. 웹 대시보드와 결과 일치 검증
7. 같은 모델을 Slack으로 확장

현재 기준으로는 **Discord부터 먼저 열고, 이후 같은 모델을 Slack에 붙이는 것이 맞다.**

## 17. 결론 (Summary)

- `대시보드 조회는 웹`, `데일리 로그 입력만 채팅`이라는 방향은 타당하다.
- 다만 자유 텍스트 자연어 방식으로 가면 빠르게 복잡해진다.
- 따라서 첫 버전은 **DOWIN `/dailylog` 세션 + 번호 입력 + 최신 메시지 전체 덮어쓰기 + O/미기록만 지원**으로 자르는 것이 맞다.
- 개발 순서는 Discord를 먼저 두되, 데이터 모델과 처리 흐름은 Slack까지 확장 가능한 형태로 잡는 것이 맞다.
- 이 범위라면 복잡도는 있지만, 충분히 통제 가능한 MVP다.
- 다만 현재 시점에서는 구현 우선순위로 올리지 않고, **후속 실험 후보**로만 유지한다.
