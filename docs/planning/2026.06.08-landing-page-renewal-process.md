# 랜딩 페이지 리뉴얼 제작 순서

## 1. 문서의 역할

이 문서는 Dowin 랜딩 페이지를 기존 화면의 부분 수정이 아니라 전면 리뉴얼 대상으로 보고, 전략 기획부터 런칭 이후 운영까지 어떤 순서로 진행할지 정리한 기준 문서다.

리뉴얼 과정에서는 랜딩을 예쁜 소개 페이지로 만드는 것보다, 처음 방문한 팀 리더가 Dowin가 어떤 문제를 해결하는 제품인지 빠르게 이해하고 다음 행동으로 이동하게 만드는 것을 우선한다.

## 2. 리뉴얼 기본 원칙

- Dowin는 팀의 목표 실행과 주간 운영을 관리하는 제품으로 설명한다.
- 랜딩의 1차 독자는 3~15인 팀의 리더, 대표, 팀장, 운영 책임자다.
- 제품을 외부 방법론이나 도서 기반 서비스처럼 설명하지 않는다.
- 개인 투두 앱, 자기계발 앱, 단순 목표 기록 앱처럼 읽히는 표현을 피한다.
- 핵심 메시지는 기록 자체가 아니라 상태 파악, 개입, 후속 확인이 이어지는 운영 루프에 둔다.
- 사용자 노출 용어는 핵심 목표, 성공 기준, 액션 아이템, 점수판, 주간 점검을 기준으로 맞춘다.
- 섹션별 설명은 짧게 유지하고, 긴 설명보다 제품 화면 조각과 운영 장면으로 이해시키는 방향을 우선한다.

## 3. 전체 제작 순서

### 3.1. 전략 기획

목표 설정, 타깃 고객, 경쟁 구도, 전환 행동을 먼저 확정한다.

주요 작업:

- 랜딩의 1차 목표를 정한다.
- 회원가입, 로그인, 문의, 파일럿 신청, 결제 전환 중 우선 CTA를 결정한다.
- 1차 타깃 고객을 구체화한다.
- 타깃 고객이 랜딩에서 해결되기를 기대하는 질문을 정리한다.
- 직접 경쟁 제품과 대체 도구를 나눠 분석한다.
- Dowin가 기존 도구와 다르게 보여줘야 할 차별점을 정의한다.

산출물:

- 랜딩 목표 1개
- 보조 목표 1~2개
- 핵심 타깃 고객 정의
- 주요 경쟁사 및 대체 도구 목록
- CTA 우선순위
- 리뉴얼 성공 기준

### 3.2. 콘텐츠 설계

전략 기획에서 정한 타깃과 CTA를 기준으로 메시지와 페이지 흐름을 설계한다.

주요 작업:

- 핵심 메시지를 한 문장으로 정리한다.
- 히어로 헤드라인과 보조 카피 방향을 정한다.
- 문제 제기, 솔루션, 제품 흐름, 사회적 증거, CTA 섹션의 순서를 설계한다.
- 각 섹션에서 답해야 할 사용자 질문을 정한다.
- 필요한 제품 이미지, 화면 조각, 영상, 그래픽 요소를 목록화한다.
- 한국어와 영어 카피가 모두 필요한지 결정한다.

기본 섹션 흐름:

1. Hero
2. Problem
3. Solution
4. Product Flow
5. Social Proof 또는 Product Proof
6. Final CTA

Dowin에 맞춘 섹션 판단 기준:

- Hero는 제품 정체성과 타깃을 즉시 드러낸다.
- Problem은 팀 실행이 흐려지는 운영 장면을 보여준다.
- Solution은 Dowin의 구조를 기능 나열이 아니라 실행 운영 흐름으로 설명한다.
- Product Flow는 핵심 목표 설정, 액션 아이템 기록, 점수판 확인, 주간 점검으로 이어지는 실제 사용 흐름을 보여준다.
- Social Proof가 아직 부족하면 고객 로고나 과장 수치 대신 제품 화면 기반의 Product Proof를 우선 사용한다.

산출물:

- 섹션 구성안
- 섹션별 메시지
- 1차 카피 초안
- 필요한 시각 자산 목록
- 카피 검토 체크리스트

### 3.3. 와이어프레임

콘텐츠 설계안을 화면 구조로 옮긴다.

주요 작업:

- 데스크톱과 모바일의 섹션 배치를 설계한다.
- 첫 화면에서 브랜드, 제품 정체성, CTA, 제품 비주얼이 모두 보이는지 확인한다.
- 섹션별 정보 밀도를 조절한다.
- CTA 위치와 반복 빈도를 결정한다.
- 실제 제품 화면 조각을 어디에 배치할지 정한다.

산출물:

- 데스크톱 와이어프레임
- 모바일 와이어프레임
- 섹션별 레이아웃 메모
- CTA 배치안

### 3.4. 디자인

와이어프레임에 브랜드 톤, 컬러, 폰트, UI 스타일을 적용한다.

주요 작업:

- 기존 Dowin 제품 UI와 연결되는 시각 언어를 정한다.
- 카드, 버튼, 배지, 제품 프리뷰, 섹션 배경의 스타일을 정리한다.
- 랜딩이 SaaS 운영 도구처럼 보이도록 정보 밀도와 여백을 조절한다.
- 모바일에서 텍스트와 CTA가 겹치지 않는지 확인한다.
- 디자인 리뷰 후 수정안을 반영한다.

디자인 기준:

- 지나치게 감성적인 히어로나 장식 중심 구성은 피한다.
- 실제 제품 화면 또는 제품 상태를 보여주는 비주얼을 우선한다.
- 한 섹션에는 하나의 메시지와 하나의 시각적 중심만 둔다.
- 반복 카드가 많아져 설명 페이지처럼 보이지 않게 한다.

산출물:

- 데스크톱 디자인 시안
- 모바일 디자인 시안
- 컴포넌트 스타일 기준
- 사용 이미지 및 제품 화면 자산 목록

### 3.5. 개발

확정된 디자인을 Next.js와 Tailwind 기반으로 구현한다.

주요 작업:

- 랜딩 컴포넌트 구조를 정한다.
- 반응형 레이아웃을 구현한다.
- CTA 버튼과 로그인, 회원가입, 문의, 결제 등 실제 이동 경로를 연결한다.
- 다국어 메시지가 필요한 경우 메시지 파일을 함께 수정한다.
- 기존 루트 라우팅과 로그인 사용자 리다이렉트가 깨지지 않게 확인한다.

Dowin 개발 기준:

- 기존 프론트엔드 패턴과 컴포넌트 구조를 우선 재사용한다.
- 라우팅, locale, 로그인 사용자 처리 규칙을 현재 구현과 맞춘다.
- 사용자 노출 카피는 제품 용어 기준과 일치시킨다.
- 랜딩 전용 구현이 앱 내부 화면 규칙을 불필요하게 흔들지 않게 한다.

산출물:

- 구현된 랜딩 페이지
- 반응형 UI
- CTA 연결
- 필요한 메시지 파일 변경

### 3.6. 연동 및 설정

랜딩이 실제 전환과 측정에 연결되도록 설정한다.

주요 작업:

- GA4 또는 GTM 적용 필요 여부를 확인한다.
- 랜딩 조회, CTA 클릭, 회원가입 시작, 가입 완료, 문의 제출 같은 전환 이벤트를 정의한다.
- SEO 기본 메타태그를 점검한다.
- OG 이미지와 공유 문구를 정리한다.
- 검색 노출보다 직접 유입과 전환 측정이 우선인지 판단한다.

산출물:

- 이벤트 정의표
- 전환 경로 목록
- SEO 메타 정보
- OG 정보

### 3.7. QA 및 테스트

기능, 시각, 반응형, 성능, 전환 경로를 점검한다.

주요 작업:

- 데스크톱과 모바일 주요 뷰포트를 확인한다.
- CTA 이동이 의도한 경로로 연결되는지 확인한다.
- 로그인 사용자와 비로그인 사용자의 루트 접근 동작을 확인한다.
- 폼이 있다면 제출과 에러 상태를 확인한다.
- 이미지 로딩, 레이아웃 흔들림, 텍스트 겹침을 확인한다.
- 필요한 최소 테스트, 타입 체크, 린트를 실행한다.

산출물:

- QA 체크리스트
- 발견 이슈 목록
- 수정 완료 내역
- 최종 검증 명령 결과

### 3.8. 런칭 및 운영

배포 후 실제 유입과 전환 데이터를 보면서 개선 사이클을 운영한다.

주요 작업:

- 배포 경로와 배포 브랜치를 확인한다.
- 마케팅 채널, 초대 링크, 콘텐츠 유입 경로와 연결한다.
- 전환 이벤트가 정상 수집되는지 확인한다.
- 초기 방문자 반응과 이탈 구간을 관찰한다.
- A/B 테스트가 필요한 항목을 정한다.

산출물:

- 런칭 체크리스트
- 운영 지표 대시보드 기준
- 후속 개선 후보
- A/B 테스트 후보

## 4. 1단계 전략 기획 체크리스트

다음 작업은 이 체크리스트를 채우는 것이다.

### 4.1. 목표 설정

결정해야 할 질문:

- 이번 랜딩 리뉴얼의 1차 목표는 무엇인가?
- 지금 가장 중요한 전환은 회원가입, 로그인, 문의, 파일럿 신청, 결제 중 무엇인가?
- 보조 CTA가 필요하다면 무엇을 맡길 것인가?
- CTA 문구는 사용자가 실제로 다음에 하는 행동과 일치하는가?

초안:

- 1차 목표: 3~15인 팀 리더가 Dowin를 팀 실행 관리 제품으로 이해하고 가입 또는 파일럿 문의로 이동하게 만든다.
- 1차 CTA 후보: 시작하기 또는 회원가입
- 보조 CTA 후보: 제품 흐름 보기 또는 문의하기
- 전환 기준 후보: 랜딩 CTA 클릭률, 회원가입 시작률, 회원가입 완료율, 문의 제출률

### 4.2. 타깃 고객 정의

결정해야 할 질문:

- 랜딩의 기본 독자는 개인 사용자인가, 팀 리더인가?
- 팀 규모는 어느 정도로 볼 것인가?
- 타깃이 지금 쓰고 있는 대체 도구는 무엇인가?
- 타깃이 Dowin를 처음 봤을 때 가장 먼저 확인하고 싶은 것은 무엇인가?

초안:

- 1차 타깃: 3~15인 팀의 리더, 대표, 팀장, 운영 책임자
- 주요 상황: 팀이 목표를 세우지만 주간 실행 상태가 흐려지고, 회의 때마다 감과 기억에 의존해 진행 상황을 판단한다.
- 기존 대체 도구: Notion, 스프레드시트, Slack 또는 Discord 공지, 회의록, 개인 투두 앱
- 핵심 기대: 이번 주에 누가 어떤 액션을 실행했고, 목표 달성 흐름이 좋은지 빠르게 보고 싶다.

### 4.3. 경쟁사 및 대체 도구 분석

결정해야 할 질문:

- 직접 경쟁 제품은 무엇인가?
- 사용자가 실제로 Dowin 대신 쓰는 대체 도구는 무엇인가?
- Dowin가 기능이 아니라 사용 장면에서 다르게 보여줘야 할 점은 무엇인가?
- 경쟁사 분석을 랜딩 카피에 직접 반영할지, 내부 판단 기준으로만 쓸지 결정한다.

분석 대상 후보:

- 목표 관리 도구
- OKR 관리 도구
- 프로젝트 관리 도구
- 팀 대시보드 도구
- Notion 템플릿
- 스프레드시트 기반 주간 점검표

비교 축 후보:

- 목표와 실행 행동이 연결되는가
- 주간 단위로 상태를 빠르게 볼 수 있는가
- 리더가 개입이 필요한 지점을 찾기 쉬운가
- 팀원이 기록하기 쉬운가
- 회의 준비 노동을 줄이는가

### 4.4. 리뉴얼 성공 기준

정량 기준 후보:

- 랜딩 CTA 클릭률
- 회원가입 시작률
- 회원가입 완료율
- 문의 제출률
- Hero 영역 이탈률
- CTA 클릭 이후 로그인 또는 가입 완료까지의 전환율

정성 기준 후보:

- 처음 보는 사용자가 Dowin를 팀 실행 관리 제품으로 설명할 수 있다.
- 개인 투두 앱이나 자기계발 앱으로 오해하지 않는다.
- 핵심 목표, 액션 아이템, 점수판이 제품 구조로 이해된다.
- 랜딩 문장이 추상적인 동기부여 문구가 아니라 팀 운영 장면으로 읽힌다.

## 6. 현재 열린 질문

- 이번 리뉴얼의 1차 CTA를 회원가입으로 둘지, 파일럿 문의로 둘지 결정해야 한다.
- 현재 유료화 방향과 맞춰 가격 또는 Basic 플랜 메시지를 랜딩에 포함할지 결정해야 한다.
- 사회적 증거가 부족한 경우 제품 화면 기반 Product Proof로 대체할지 결정해야 한다.
- 경쟁사 분석을 실제 외부 조사까지 진행할지, 내부 대체 도구 분석으로 먼저 시작할지 결정해야 한다.
