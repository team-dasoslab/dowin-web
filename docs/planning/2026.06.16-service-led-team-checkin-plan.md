# 체크인 기획

## 0. 기획 전제

이 문서는 기존 리더 리포트 메뉴나 기존 `report` 프로토타입을 살리는 계획이 아니다. 기존 리더 리포트 기능은 현재 비활성화된 상태이며, 신규 제품 방향에서는 제거 대상으로 본다.

따라서 이 기능은 기존 리포트 화면을 확장하거나 이름만 바꾸는 작업이 아니라, `리더가 반복적으로 직접 개입하지 않아도 팀원이 합의한 액션 아이템을 다시 실행하게 만드는 서비스 주도 체크인`을 처음부터 설계하는 greenfield 기획이다.

기존 리더 리포트 관련 문서는 히스토리와 참고 자료로만 본다. 신규 구현의 기준은 이 문서다.

## 1. 배경 및 맥락

현재 Dowin의 핵심 가치는 `기록` 자체가 아니라 `상태 파악 -> 개입 -> 후속 확인`이 이어지는 운영 루프다.

다만 팀 운영에서 리더가 매번 팀원에게 직접 확인하고 독려해야 한다면, Dowin은 리더의 일을 줄이는 도구가 아니라 리더가 해야 할 일을 더 잘 보이게 하는 도구에 머문다. 고객 피드백이 뚜렷하지 않은 상황에서도 이 지점은 제품 가치의 중요한 가설이다.

새 기능의 방향은 리더가 모든 체크인을 직접 수행하는 것이 아니라, 리더는 목표와 액션 아이템의 합의에 집중하고 Dowin이 반복적인 실행 푸시와 후속 확인을 담당하는 것이다.

참고 문서:

- `docs/planning/2026.04.14-leader-report-reminder-plan.md` (기존 리더 리포트/자동 체크인 히스토리 참고용)
- `docs/planning/2026.05.20-billing-entitlement-and-workspace-architecture-plan.md`
- `docs/dev/common/2026.05.09-product-positioning-and-writing-rules.md`

## 2. 문제 정의

- 리더가 팀원의 실행 상태를 직접 확인하고 메시지를 보내는 루틴은 오래 지속되기 어렵다.
- 팀원 입장에서는 리더가 반복해서 묻는 경험이 압박, 감시, 평가로 읽힐 수 있다.
- 현재 개인 리마인드는 사용자 본인 중심이고, 팀 운영 관점의 합의된 액션 아이템 실행을 다시 움직이게 하는 기능은 약하다.
- 팀이 목표와 액션 아이템을 합의해도, 실제 주중에는 그 합의가 잊히거나 밀리는 순간이 생긴다.
- 리더가 매번 개입하면 팀 운영 비용이 줄지 않고, 리더가 개입하지 않으면 실행 루프가 끊긴다.

## 3. 제품 가설

리더가 목표와 액션 아이템의 합의만 잘 만들고, 이후의 반복 체크인과 실행 재개 유도는 Dowin이 맡으면 팀 운영 비용은 줄고 실행 지속성은 높아진다.

검증할 핵심 가설:

- 리더는 직접 메시지를 보내지 않아도 팀 실행 상태가 다시 움직이는 것을 보고 싶어 한다.
- 팀원은 리더의 직접 추궁보다 서비스가 보내는 짧고 중립적인 체크인을 덜 방어적으로 받아들인다.
- 체크인은 발송 자체보다 팀원의 1탭 반응과 후속 기록 재개까지 연결될 때 제품 가치가 생긴다.
- 팀원이 `막힘 있음`이나 `목표 조정 필요`를 보냈을 때, 리더가 말로만 피드백하는 것보다 바로 적용 가능한 구조화된 조정안을 제안하면 실행 재개 가능성이 높아진다.
- Basic seat 결제의 유료 가치는 조회 기능보다 `리더의 반복 운영 부담을 줄이는 서비스 주도 실행 루프`에서 더 선명해진다.

## 4. 역할 분담 원칙

### 4.1. 리더의 역할

- 팀원과 핵심 목표, 성공 기준, 액션 아이템을 함께 정한다.
- 이번 주 무엇을 하면 이긴 것으로 볼지 합의한다.
- Dowin이 어떤 기준으로 체크인을 보낼지 워크스페이스 정책을 설정한다.
- 체크인 결과 리포트를 보고, 서비스 자동 체크인만으로 해결되지 않는 경우에만 직접 개입한다.
- 목표나 액션 아이템이 현실과 맞지 않으면 조정 대화를 연다.
- 팀원의 요청에 대해 말로만 답하지 않고, 목표 횟수 수정, 액션 아이템 보관, 대체 액션 아이템 생성 같은 구조화된 조정안을 제안한다.

리더의 기본 역할은 `반복 푸시 담당자`가 아니라 `운영 기준과 조정의 책임자`다.

### 4.2. Dowin의 역할

- 액션 아이템 실행이 멈춘 신호를 감지한다.
- 합의된 목표와 액션 아이템 기준으로 팀원에게 개인 체크인을 보낸다.
- 팀원이 `지금 기록하기`, `오늘 나중에 하기`, `막힘 있음`, `목표 조정 필요` 같은 낮은 마찰의 반응을 남기게 한다.
- 팀원 반응 후 같은 체크인을 과도하게 반복하지 않는다.
- 체크인 이후 실제 기록 재개, 달성률 변화, 무응답 지속 여부를 추적한다.
- 리더에게 발송 로그가 아니라 `서비스가 무엇을 했고 결과가 어땠는지`를 보고한다.
- 리더가 선택할 수 있는 조정 액션을 정해진 타입으로 제공하고, 팀원이 수락하면 해당 변경을 안전하게 적용한다.

Dowin의 기본 역할은 `알림 발송기`가 아니라 `합의된 실행 루프를 대신 굴리는 운영 보조자`다.
Dowin은 리더의 자유 코멘트를 분석해 임의로 목표나 액션 아이템을 바꾸지 않는다. 변경은 리더가 명시적으로 선택한 구조화된 제안과 팀원의 수락을 통해서만 적용한다.

### 4.3. 팀원의 역할

- 체크인을 받으면 긴 보고 없이 1탭 또는 짧은 입력으로 현재 상태를 알린다.
- 바로 할 수 있으면 기록으로 이어진다.
- 지금 하기 어렵다면 나중에 하겠다는 신호를 남긴다.
- 막혔거나 목표가 과하면 리더에게 직접 설명하기 전에 제품 안에서 신호를 남긴다.
- 리더가 제안한 조정안을 확인하고, 수락 또는 거절한다.
- 수락한 조정안은 버튼 한 번으로 적용되며, 별도 설정 화면을 직접 찾아갈 필요가 없다.

팀원 경험은 `혼남`이 아니라 `이번 주 실행을 다시 잡는 개인 리셋`이어야 한다.

## 5. MVP 범위

### 5.1. 포함

- 워크스페이스 ADMIN이 서비스 주도 체크인 정책을 켜고 끌 수 있다.
- 리더 본인을 팀 점수 집계 및 액션 아이템 실행 대상(체크인 수신 포함)에 포함할지, 아니면 순수 관리자로만 활동할지 설정할 수 있다.
- 체크인 발송 기준은 MVP에서 `주간 목표 빈도 대비 실행 시작 지연` 하나로 제한한다.
- 체크인 대상 액션 아이템은 생성 후 168시간(`7 * 24h`)이 지난 경우에만 포함한다.
- 체크인 대상은 팀원 1명과 액션 아이템 1개 조합이다. (단, 리더 역할의 사용자에게는 팀원용 체크인 알림을 발송하지 않는다.)
- 체크인 템플릿은 고정 문구로 시작한다.
- 팀원 반응은 MVP에서 최소 2개, 가능하면 4개까지 둔다.
  - `지금 기록하기`
  - `오늘 나중에 하기`
  - `막힘 있음`
  - `목표 조정 필요`
- `오늘 나중에 하기`는 당일 동일 체크인 재발송을 막는다.
- `막힘 있음`과 `목표 조정 필요`는 리더에게 직접 추궁 대상이 아니라 조정 신호로 보고한다.
- 리더는 결과 리포트에서 발송 이유, 팀원 반응, 후속 기록 재개 여부를 확인한다.
- 리더는 `막힘 있음`, `목표 조정 필요` 신호에 대해 구조화된 조정안을 제안할 수 있다.
- 팀원은 리더의 조정안을 1탭으로 수락하거나 거절할 수 있다.
- 팀원이 수락한 조정안은 제품 상태에 바로 반영된다.
- 기본 채널은 현재 앱 푸시/알림 구조를 우선 활용한다.

### 5.2. 제외

- AI가 자유롭게 문구를 생성하는 체크인
- 리더가 팀원별로 강한 압박 문구를 커스텀하는 기능
- 공개 랭킹, 공개 경고, 팀원 비교
- 반복 미응답에 따른 자동 escalation
- Slack, Discord, KakaoWork, 이메일 같은 외부 채널 발송
- 목표/액션 아이템 자동 생성
- 리더 코멘트 분석 기반 자동 변경
- 팀원 동의 없는 목표/액션 아이템 변경
- 리더가 자유 텍스트만 보내고 제품 상태는 바뀌지 않는 피드백 흐름
- 성격, 의지, 태도 추정

## 6. 체크인 트리거

### 6.1. 체크인 대상 자격

조건:

- 팀원이 활성 점수판과 액션 아이템을 가지고 있다.
- 대상 액션 아이템은 `WEEKLY` 주기의 활성 액션 아이템이다.
- 대상 액션 아이템은 생성 시각(`lead_measures.created_at`) 기준으로 168시간(`7 * 24h`) 이상 지난 상태여야 한다.
- 생성 후 168시간이 지나지 않은 액션 아이템은 이번 주 기록이 없더라도 체크인 대상에서 제외한다.
- 동일 팀원/액션 아이템에 대한 체크인이 당일 suppress 조건에 걸려 있지 않다.

의도:

- 새로 만든 액션 아이템을 즉시 압박하지 않고, 최소 1주일의 적응 기간을 둔다.
- 체크인은 오래된 합의가 느슨해졌을 때만 개입한다.

예시:

- 금요일 11:00에 만든 액션 아이템은 다음 금요일 11:00 전까지 체크인 대상이 아니다.
- 월요일 09:00에 만든 액션 아이템은 다음 월요일 09:00 이후부터 체크인 대상이 될 수 있다.
- 대상 자격을 통과해도 실제 발송은 워크스페이스 리더가 설정한 체크인 발송 시간대에만 이뤄진다.

### 6.2. 체크인 발송 시간

MVP 기준:

- 기본 발송 시간은 멤버 현지 시간 기준 16:00이다.
- 리더는 워크스페이스 체크인 설정에서 발송 시간을 1시간 단위로 변경할 수 있다.
- 실제 배치 cron은 매시 실행되지만, 체크인 delivery 생성은 멤버 현지 시간의 `sendHour`와 일치하는 실행에서만 허용한다.
- 23:00처럼 업무 맥락에서 늦은 시간대에 자동 체크인이 새로 생성되지 않도록, 발송 시간 조건은 push 전송 이전의 delivery 생성 단계에서 적용한다.
- 여러 시간대를 쓰는 팀에서는 각 멤버의 알림 timezone 기준으로 발송 시간을 해석한다.

의도:

- 체크인은 기록 압박이 아니라 업무 중 상태 확인과 개입 시작을 돕는 운영 알림이어야 한다.
- 리더가 팀 운영 리듬에 맞는 시간대를 선택할 수 있어야 한다.

### 6.3. 주간 목표 빈도 대비 실행 시작 지연

현재 구현에서는 `SLOW_WEEKLY_START` 발송을 사용하지 않는다. 늦은 시간 발송과 과도한 자동 개입을 피하기 위해, MVP 운영에서는 `NO_WEEKLY_LOG` 중심으로 체크인을 보낸다.

조건 예시:

- 주간 목표 1회: 금요일 오전까지 기록 0건
- 주간 목표 2회: 수요일 오전까지 기록 0건
- 주간 목표 3회 이상: 수요일 오전까지 기록 1건 이하

의도:

- 목표 주기가 끝나기 전에 실행을 다시 시작할 시간을 남긴다.

문구 예시:

- `이번 주 {액션아이템명} 목표까지 아직 여유가 있지만, 시작이 늦어지고 있어요. 오늘 한 번만 기록해 볼까요?`
- `{액션아이템명} 실행 리듬이 이번 주 목표보다 늦습니다. 가능한 작은 단위로 먼저 재개해 주세요.`

## 7. 팀원 반응 모델

| 반응               | 제품 해석                             | 리더에게 보이는 신호 | 후속 정책                           |
| ------------------ | ------------------------------------- | -------------------- | ----------------------------------- |
| `지금 기록하기`    | 실행 재개 의사                        | `기록으로 이동`      | 해당 체크인 종료                    |
| `오늘 나중에 하기` | 의도는 있으나 현재 타이밍 부적합      | `오늘 처리 예정`     | 당일 동일 체크인 재발송 금지        |
| `막힘 있음`        | 실행 장애물 존재                      | `막힘 신호`          | 리더 리포트에 조정 후보로 표시      |
| `목표 조정 필요`   | 목표/액션 아이템 난이도 불일치 가능성 | `목표 조정 후보`     | 반복 체크인보다 리더 조정 제안 우선 |

MVP 구현 부담을 줄여야 한다면 1차는 `지금 기록하기`, `오늘 나중에 하기` 2개로 시작하고, 파일럿에서 `막힘 있음`, `목표 조정 필요`를 추가한다.

## 8. 리더 조정 액션

팀원이 `막힘 있음`이나 `목표 조정 필요`를 선택하면, 리더는 단순 코멘트만 남기는 것이 아니라 제품 상태를 바꿀 수 있는 조정안을 제안해야 한다.

핵심 원칙:

- 리더가 선택한 구조화된 액션만 제안으로 보낸다.
- Dowin은 리더 코멘트를 분석해 자체적으로 목표나 액션 아이템을 변경하지 않는다.
- 팀원이 수락해야 실제 변경이 적용된다.
- 팀원이 거절하거나 응답하지 않으면 기존 목표/액션 아이템은 유지된다.
- 모든 조정 제안은 누가, 언제, 어떤 이유로 제안했고, 팀원이 어떻게 반응했는지 이력으로 남긴다.

### 8.1. MVP 조정 액션 타입

| 액션 타입 | 리더가 정하는 값 | 팀원에게 보이는 버튼 | 적용 결과 | MVP 포함 여부 |
| --- | --- | --- | --- | --- |
| `CHANGE_TARGET_COUNT` | 새 목표 횟수 | `{n}건으로 수정하기` | 현재 액션 아이템의 주간/월간 목표 횟수를 변경 | 포함 |
| `ARCHIVE_ACTION_ITEM` | 보관할 액션 아이템 | `이 액션 아이템 보관하기` | 현재 액션 아이템을 운영 대상에서 제외 | 포함 |
| `REPLACE_ACTION_ITEM` | 보관할 기존 액션 아이템 + 새 액션 아이템 이름/목표 횟수 | `새 액션 아이템으로 바꾸기` | 기존 액션 아이템 보관 후 대체 액션 아이템 생성 | 포함 |
| `PAUSE_CHECKIN` | 체크인 유예 기간 | `{n}일 동안 체크인 쉬기` | 해당 액션 아이템의 자동 체크인을 일시 중지 | Post-MVP |
| `REQUEST_CONTEXT` | 리더의 짧은 질문 | `상황 답변하기` | 팀원이 짧은 메모로 맥락 제공 | Post-MVP |

MVP는 `CHANGE_TARGET_COUNT`, `ARCHIVE_ACTION_ITEM`, `REPLACE_ACTION_ITEM` 3개만 우선 다룬다. 이 3개는 팀원이 "지금 못 하고 있다"는 신호를 보냈을 때 실제 실행 조건을 바꾸는 액션이기 때문이다.

### 8.2. 조정 제안 플로우

1. 팀원이 체크인에서 `막힘 있음` 또는 `목표 조정 필요`를 선택한다.
2. 리더 결과 화면에 `조정 필요 신호`가 생성된다.
3. 리더가 해당 신호를 열고 조정 액션 타입을 선택한다.
4. 리더가 필요한 값을 입력한다.
   - 목표 횟수 변경: 새 목표 횟수
   - 액션 아이템 보관: 보관 사유 선택 또는 짧은 코멘트
   - 대체 액션 아이템 생성: 새 액션 아이템 이름, 목표 주기, 목표 횟수
5. Dowin이 팀원에게 구조화된 제안을 보낸다.
6. 팀원이 `수락` 또는 `거절`을 선택한다.
7. 팀원이 수락하면 Dowin이 변경을 적용하고, 이후 체크인/집계 기준도 새 상태를 따른다.
8. 리더 결과 화면에는 제안 상태가 `제안됨`, `수락됨`, `거절됨`, `만료됨`으로 표시된다.

### 8.3. 팀원 수락 화면 원칙

팀원에게는 리더의 코멘트보다 실제 바뀌는 내용을 먼저 보여준다.

예시:

- `이번 주 목표를 5건에서 3건으로 바꿀까요?`
  - 버튼: `3건으로 수정하기`
  - 보조 버튼: `그대로 유지하기`
- `현재 액션 아이템을 보관하고, "잠재 고객 3명에게 짧은 메시지 보내기"로 바꿀까요?`
  - 버튼: `새 액션 아이템으로 바꾸기`
  - 보조 버튼: `그대로 유지하기`
- `이 액션 아이템을 이번 주 운영 대상에서 제외할까요?`
  - 버튼: `보관하기`
  - 보조 버튼: `그대로 유지하기`

팀원 화면에서 금지할 것:

- 리더 코멘트만 보여주고 실제 변경 버튼이 없는 상태
- `리더가 이렇게 하라고 했습니다`처럼 명령형으로 보이는 문구
- 팀원 수락 없이 즉시 변경되는 흐름
- 여러 변경을 한 번에 묶어 팀원이 무엇을 수락하는지 모호한 흐름

### 8.4. 상태 모델 초안

조정 제안은 체크인 응답과 별도 엔티티로 다룬다.

필수 상태:

- `PROPOSED`: 리더가 제안했고 팀원이 아직 응답하지 않음
- `ACCEPTED`: 팀원이 수락했고 변경 적용 완료
- `DECLINED`: 팀원이 거절함
- `EXPIRED`: 응답 기한이 지나 적용하지 않음
- `CANCELED`: 리더가 제안을 철회함

필수 필드:

- `workspaceId`
- `leaderUserId`
- `memberUserId`
- `sourceCheckInId`
- `actionType`
- `payload`
- `leaderNote`
- `status`
- `expiresAt`
- `acceptedAt`
- `appliedAt`

`payload`는 action type별로 Zod schema를 분리한다. 자유 문자열 payload 하나로 처리하지 않는다.

## 9. 리더 결과 리포트

리더에게 보여줄 핵심은 누가 잘못했는지가 아니라, 서비스가 반복 운영을 얼마나 대신 처리했고 어디에 사람의 조정이 필요한지다.

리포트의 최소 질문:

- 이번 주 Dowin이 어떤 체크인을 보냈는가
- 팀원은 어떻게 반응했는가
- 반응 후 실제 기록이 재개됐는가
- 자동 체크인보다 목표/액션 아이템 조정이 필요한 대상은 누구인가
- 리더가 제안한 조정안 중 수락/거절/만료된 것은 무엇인가
- 리더가 직접 개입해야 하는 대상은 최소 누구인가

리포트 최소 구성:

- 총 체크인 발송 수
- 체크인 대상 팀원 수
- 1탭 반응률
- 24시간 내 기록 재개율
- `막힘 있음` 또는 `목표 조정 필요` 신호 수
- 조정 제안 수락률
- 조정 제안 후 기록 재개율
- 리더 확인 필요 대상 목록

리더 화면의 기본 카피 방향:

- `반복 푸시보다 목표와 액션 아이템 조정이 필요한 신호가 있습니다.`
- `팀원이 수락하면 변경이 바로 적용됩니다.`

### 9.1. UI/UX 참고 원칙

현재 프로토타입은 백엔드 계약의 기준이 아니라 UX 탐색 결과로만 본다. 백엔드는 mock 데이터의 필드명, 상태명, 화면 배치에 맞추지 않고 이 문서의 제품 의미와 도메인 상태 모델을 기준으로 설계한다.

팀원 화면의 핵심은 `개인에게 온 체크인을 빠르게 이해하고 1탭으로 반응하는 것`이다.

- 화면 형태는 Inbox, 알림함, 카드 상세, 바텀시트 중 무엇이든 가능하다.
- 백엔드가 보장해야 할 것은 목록 조회, 상세 조회, 읽음 처리, 반응 제출, 조정 제안 수락/거절이다.
- UI가 대화형 타임라인처럼 보이더라도 백엔드 엔티티는 자유 대화 메시지가 아니라 `체크인`, `체크인 반응`, `조정 제안`으로 분리한다.
- 팀원이 `막힘 있음`, `목표 조정 필요`를 선택할 때 짧은 메모를 남길 수는 있지만, 그 메모는 상태 변경의 근거가 아니라 리더가 조정안을 만들 때 참고하는 맥락이다.

리더 화면의 핵심은 `발송 로그 열람`이 아니라 `서비스 자동 체크인으로 해결되지 않은 조정 필요 신호를 처리하는 것`이다.

- 리더 화면은 주간 기간 기준으로 요약 지표, 리더 확인 필요 대상, 최근 활동을 조회할 수 있어야 한다.
- 리더 확인 필요 대상은 체크인 반응이 `BLOCKED` 또는 `ADJUSTMENT_REQUESTED`인 상태에서 아직 처리되지 않은 항목이다.
- 리더가 남기는 자유 코멘트는 조정 제안의 보조 설명일 뿐이며, 실제 제품 상태 변경은 `adjustmentProposal.actionType`과 `payload`로만 결정한다.
- 프로토타입에 있는 `pause_checkin` 성격의 흐름은 Post-MVP로 두고 MVP 백엔드 계약에는 넣지 않는다.

## 10. 가드레일

- 체크인은 개인 푸시 또는 개인 알림으로만 보낸다.
- 리더 본인에게는 팀원용 실행 독려 체크인 알림을 발송하지 않는다. (리더는 대시보드 리포트를 통해 확인)
- 공개 비교, 공개 경고, 팀원 순위는 제공하지 않는다.
- 문구는 관찰 가능한 실행 상태와 다음 행동만 다룬다.
- 사람의 성향, 의지, 태도를 추정하지 않는다.
- 동일 팀원/액션 아이템 기준 하루 1회 이하로 제한한다.
- 동일 팀원 기준 하루 전체 체크인 수를 제한한다.
- 워크스페이스 전체 일일 발송량 상한을 둔다.
- 팀원이 반응하면 같은 날 같은 체크인을 반복하지 않는다.
- 반복 미응답을 이유로 자동으로 더 강한 문구를 보내지 않는다.
- 리더에게는 발송량을 늘리기보다 목표/액션 아이템 조정을 먼저 제안한다.
- 리더 조정 액션은 팀원의 수락 전까지 적용하지 않는다.
- 조정 제안은 한 번에 하나의 변경만 담는다.
- 팀원이 조정 제안을 거절해도 동일 제안을 반복 푸시하지 않는다.

## 11. 측정 지표

### 11.1. 제품 가치 지표

- 서비스 주도 체크인 설정률
- 체크인 발송 성공률
- 체크인 수신자 1탭 반응률
- 체크인 수신 후 24시간 내 기록 재개율
- 체크인 수신 후 7일 내 액션 아이템 달성률 변화
- 리더 결과 리포트 조회율
- 리더 직접 개입 필요 대상 감소 여부
- 리더 조정 제안 생성률
- 조정 제안 수락률
- 조정 제안 수락 후 7일 내 기록 재개율

### 11.2. 수용성 지표

- 체크인 수신 후 알림 비활성화율
- 체크인 수신 후 기록 중단률
- 동일 팀원 반복 발송 비율
- `막힘 있음`, `목표 조정 필요` 선택 후 리더 조정 발생률
- 조정 제안 거절률
- 조정 제안 수신 후 알림 비활성화율
- 파일럿 인터뷰에서 `감시`, `압박`, `잔소리` 표현 등장 빈도

성공 판단은 발송 수로 하지 않는다. 발송은 적더라도 기록 재개와 목표 조정 신호가 선명하면 성공에 가깝고, 발송은 많지만 반응률이 낮거나 알림 opt-out이 늘면 실패에 가깝다.

## 12. MVP / Post-MVP 경계

### 12.1. MVP

- 워크스페이스 단위 체크인 정책 on/off
- 트리거 2개
- 고정 템플릿
- 팀원 1탭 반응
- 당일 suppress/cooldown
- 발송/반응/후속 기록 재개 이벤트 저장
- 리더 결과 리포트 최소 지표
- 리더 조정 액션 3종
  - 목표 횟수 변경 제안
  - 액션 아이템 보관 제안
  - 기존 액션 아이템 보관 후 대체 액션 아이템 생성 제안
- 팀원 조정 제안 수락/거절
- 수락된 조정 제안 적용 이력
- Basic 유료 가치 후보로 검증

### 12.2. Post-MVP

- 팀원별 선호 시간 반영
- 더 세밀한 목표 조정 제안
- 인앱 inbox
- 외부 메신저/이메일 채널
- 리더용 정책 추천
- 액션 아이템 난이도 자동 감지
- 팀별 체크인 톤 세분화
- 리더에게 목표/액션 아이템 재합의 워크플로 제안
- 체크인 유예 기간 제안
- 팀원 맥락 질문/답변
- 조정 제안 템플릿 추천

## 13. 구현 영향 범위

예상 도메인:

- notification
- dashboard/report
- scoreboard
- lead-measure/action item
- daily-log
- analytics/event log
- billing/entitlement gate
- webview/app push

예상 저장 모델:

- workspace check-in settings
- check-in delivery
- check-in response
- check-in follow-up status
- check-in cooldown/suppress state
- leader adjustment proposal
- leader adjustment proposal audit/apply log
- Basic usage/value event ledger

구현 시 유의할 것:

- 사용자 노출 용어는 `액션 아이템`을 사용하고, 구현 모델명은 기존 `lead_measure`를 유지한다.
- 앱 푸시/FCM은 기존 `notification` 도메인의 device token/FCM 발송 경로를 재사용한다.
- 결제 비활성 워크스페이스에서는 체크인 운영 API를 차단한다.
- 이벤트 저장은 체크인 도메인의 usage/value event로 시작하고, billing usage ledger가 생기면 합류한다.
- 목표 횟수 변경, 액션 아이템 보관, 대체 액션 아이템 생성은 기존 `lead-measure` service/storage 경계를 통해 적용한다.
- 조정 제안 수락 시 과거 기록은 삭제하거나 재작성하지 않는다. 변경 시점 이후의 집계/체크인 기준만 새 액션 아이템 상태를 따른다.

## 14. 백엔드 구현 계약

이 섹션은 백엔드 개발자가 이 문서만 보고 1차 구현을 시작할 수 있도록 고정하는 계약이다. 실제 구현 시 API 계약 변경은 반드시 `src/api-spec/openapi.yaml`을 먼저 수정하고 `yarn gen:api`를 실행한다.

### 14.1. 도메인 경계

새 도메인은 `team-checkin`으로 둔다.

- 새 코드 위치:
  - `src/domain/team-checkin/validation.ts`
  - `src/domain/team-checkin/services/team-checkin.service.ts`
  - `src/domain/team-checkin/storage/team-checkin.storage.ts`
  - `src/app/api/workspaces/[workspaceId]/team-checkins/**`
- 기존 `notification` 도메인은 FCM 발송 어댑터와 디바이스 토큰 저장을 계속 담당한다.
- 기존 `dashboard/report` 도메인은 읽기 집계가 중심이므로 체크인 상태 변경 책임을 갖지 않는다.
- 기존 `lead-measure` 도메인은 조정 제안 수락 시 실제 액션 아이템 변경을 적용하는 하위 도메인으로 호출된다.
- 기존 `daily-log` 도메인은 후속 기록 재개 여부 계산에 필요한 기록 조회만 제공한다.

### 14.2. 권한 및 entitlement 규칙

- 모든 사용자 API는 `dowin_sid` 세션이 필요하다.
- 모든 API는 `workspaceId` uid를 내부 numeric id로 resolve하고, 해당 워크스페이스 멤버십을 확인한다.
- 정책 조회/수정:
  - 조회는 같은 워크스페이스 멤버에게 허용한다.
  - 수정은 워크스페이스 `ADMIN`만 허용한다.
- 리더 리포트 조회와 조정 제안 생성/취소는 워크스페이스 `ADMIN`만 허용한다.
- 팀원 체크인 목록/상세/반응/조정 제안 수락·거절은 해당 `memberUserId` 본인만 허용한다.
- 체크인 발송 배치와 내부 재실행 API는 `Authorization: Bearer <CRON_SECRET>`로 보호한다.
- 결제 비활성 워크스페이스에서는 체크인 정책, 발송 배치, 리더 리포트, 팀원 응답 API 모두 운영 API로 보고 차단한다. 결제 문제 해결용 최소 화면에 필요한 billing/profile/contact API 예외에는 포함하지 않는다.
- MVP에서는 Basic entitlement가 있는 워크스페이스만 기능을 사용 가능하게 둔다. legacy `FREE`/`STANDARD`는 신규 기능 기준값으로 사용하지 않는다.

| `updated_at` | timestamp | 수정 시각 |

권장 제약/인덱스:

- unique: `(workspace_id, member_user_id, lead_measure_id, reason_code, period_start)`
- index: `(workspace_id, period_start)`
- index: `(member_user_id, created_at)`
- index: `(lead_measure_id, created_at)`

발송 판단 규칙:

- 대상은 같은 워크스페이스의 활성 멤버 중 활성 점수판과 활성 액션 아이템을 가진 사용자다.
- `lead_measures.status='ACTIVE'`만 대상이다.
- `MONTHLY` 액션 아이템은 MVP 자동 체크인 대상에서 제외한다. MVP 트리거는 주간 실행 리듬 검증에 한정한다.
- 액션 아이템 생성 후 168시간(`7 * 24h`)이 지나지 않았으면 체크인 후보에서 제외한다.
- `NO_WEEKLY_LOG`는 스키마/응답 호환을 위해 남겨둘 수 있지만, MVP 배치 판단에서는 발송 reason으로 사용하지 않는다. 단순히 이번 주 기록이 없다는 이유만으로 월요일부터 체크인을 보내지 않는다.
- `SLOW_WEEKLY_START`:
  - `targetValue=1`: 금요일 10:00 KST 이후 달성 기록 0개
  - `targetValue=2`: 수요일 10:00 KST 이후 달성 기록 0개
  - `targetValue>=3`: 수요일 10:00 KST 이후 달성 기록 1개 이하
- 같은 `workspace/member/leadMeasure/reason/periodStart` 조합은 중복 생성하지 않는다.
- 팀원이 `SNOOZE_TODAY`, `BLOCKED`, `ADJUSTMENT_REQUESTED`로 반응한 뒤에는 같은 날 같은 액션 아이템으로 새 체크인을 만들지 않는다.
- 활성 디바이스 토큰이 없으면 delivery row는 `SKIPPED` + `skipReason=NO_ACTIVE_DEVICE_TOKEN`으로 남긴다. 이것도 발송 성공률 계산의 분모에는 포함하되, 수신자 반응률 분모에서는 제외한다.

### 14.5. 체크인 반응 모델

`team_checkin_responses`

| 컬럼 | 타입/값 | 설명 |
| --- | --- | --- |
| `id` | integer PK | 내부 PK |
| `delivery_id` | integer FK unique | 하나의 체크인에 대한 최종 반응 |
| `workspace_id` | integer FK | 조회/권한 필터용 중복 저장 |
| `member_user_id` | integer FK | 반응 사용자 |
| `response_type` | `LOG_NOW` \| `SNOOZE_TODAY` \| `BLOCKED` \| `ADJUSTMENT_REQUESTED` | 팀원 반응 |
| `note` | text nullable | 막힘/조정 요청의 짧은 맥락. 최대 500자 |
| `created_at` | timestamp | 반응 시각 |

반응 의미:

- `LOG_NOW`: 기록 의사. 서버가 daily log를 자동 생성하지 않는다. 응답 저장 후 프론트가 기존 기록 화면으로 이동한다.
- `SNOOZE_TODAY`: 당일 동일 액션 아이템 체크인을 suppress한다.
- `BLOCKED`: 실행 장애물 신호. 리더 확인 필요 대상에 오른다.
- `ADJUSTMENT_REQUESTED`: 목표 또는 액션 아이템 조정 신호. 리더 확인 필요 대상에 오른다.

검증 규칙:

- 이미 응답된 delivery에는 다시 응답할 수 없다. 필요하면 409 `TEAM_CHECKIN_ALREADY_RESPONDED`.
- 응답 사용자는 `delivery.memberUserId`와 같아야 한다.
- `BLOCKED`, `ADJUSTMENT_REQUESTED`의 `note`는 선택값이지만 빈 문자열은 null로 저장한다.
- `note`는 상태 변경 로직에 사용하지 않는다.

### 14.6. 후속 기록 재개 판정

후속 기록 재개는 별도 유저 액션이 아니라 집계 가능한 상태다.

- 기준: `team_checkin_responses.created_at` 이후 24시간 안에 같은 `lead_measure_id`에 달성 처리된 daily log가 생기면 `resumedWithin24h=true`.
- `COUNT` 액션 아이템은 `daily_logs.count >= lead_measures.daily_target_count`인 날짜만 달성 처리한다.
- `BOOLEAN` 액션 아이템은 `daily_logs.value=true`인 날짜만 달성 처리한다.
- 계산은 리포트 조회 시 동적으로 계산해도 되지만, 성능상 필요하면 `team_checkin_followups` projection을 둔다.

`team_checkin_followups`를 둘 경우:

| 컬럼 | 타입/값 | 설명 |
| --- | --- | --- |
| `id` | integer PK | 내부 PK |
| `delivery_id` | integer FK unique | 체크인 |
| `checked_at` | timestamp | 마지막 판정 시각 |
| `resumed_within_24h` | boolean | 24시간 내 기록 재개 여부 |
| `first_resumed_log_id` | integer nullable | 최초 후속 기록 |

MVP에서는 리포트 조회 부하가 커지면 projection을 추가하고, 그렇지 않으면 동적 계산으로 시작한다.

| `leader_user_id` | integer FK | 제안 생성자 |
| `member_user_id` | integer FK | 제안 수신자 |
| `lead_measure_id` | integer FK | 현재 대상 액션 아이템 |
| `action_type` | `CHANGE_TARGET_COUNT` \| `ARCHIVE_ACTION_ITEM` \| `REPLACE_ACTION_ITEM` | 제안 타입 |
| `payload_json` | text | 타입별 payload JSON |
| `leader_note` | text nullable | 선택 설명. 최대 500자 |
| `status` | `PROPOSED` \| `ACCEPTED` \| `DECLINED` \| `EXPIRED` \| `CANCELED` \| `APPLY_FAILED` | 상태 |

| `responded_at` | timestamp nullable | 팀원 수락/거절 시각 |
| `applied_at` | timestamp nullable | 실제 변경 완료 시각 |
| `apply_error_code` | text nullable | 적용 실패 코드 |
| `created_at` | timestamp | 생성 시각 |
| `updated_at` | timestamp | 수정 시각 |

상태 전이:

- `PROPOSED -> ACCEPTED`: 팀원이 수락했고 적용까지 성공
- `PROPOSED -> APPLY_FAILED`: 팀원이 수락했지만 적용 중 대상 상태가 바뀌어 실패
- `PROPOSED -> DECLINED`: 팀원이 거절
- `PROPOSED -> EXPIRED`: 만료 배치 또는 응답 시점 검증에서 만료 처리
- `PROPOSED -> CANCELED`: 리더가 철회
- terminal 상태(`ACCEPTED`, `DECLINED`, `EXPIRED`, `CANCELED`, `APPLY_FAILED`)는 다시 변경하지 않는다.

동시성/중복 규칙:

- 같은 `source_response_id`에 대해 `PROPOSED` 상태 제안은 1개만 허용한다.
- 같은 액션 아이템에 미응답 제안이 있으면 새 제안을 막고 409 `TEAM_CHECKIN_PROPOSAL_ALREADY_OPEN`을 반환한다.
- 팀원이 응답하는 시점에 대상 액션 아이템이 이미 보관됐거나 삭제됐으면 `APPLY_FAILED`로 기록하고 기존 상태를 임의 복구하지 않는다.

### 14.8. 조정 제안 payload

payload는 자유 문자열이 아니라 action type별 Zod schema로 검증한다.

`CHANGE_TARGET_COUNT`

```json
{
  "newTargetValue": 3
}
```

검증:

- 현재 액션 아이템의 `period`를 유지한다.
- `WEEKLY`는 `1..7`, `MONTHLY`는 `1..31` 범위다.
- MVP 조정 제안은 `WEEKLY` 액션 아이템만 허용한다. `MONTHLY` 액션 아이템에 대한 자동 체크인과 조정 제안은 Post-MVP로 둔다.

`ARCHIVE_ACTION_ITEM`

```json
{
  "reason": "NOT_RELEVANT_THIS_WEEK"
}
```

검증:

- `reason`: `BLOCKED_BY_EXTERNAL_DEPENDENCY` | `TOO_LARGE` | `NOT_RELEVANT_THIS_WEEK` | `OTHER`
- 실제 적용은 기존 `lead-measure` 보관 로직을 사용한다.
- 기존 daily log는 삭제하지 않는다.

`REPLACE_ACTION_ITEM`

```json
{
  "replacementName": "잠재 고객 3명에게 짧은 메시지 보내기",
  "replacementPeriod": "WEEKLY",
  "replacementTargetValue": 3,
  "replacementTrackingMode": "BOOLEAN",
  "replacementDailyTargetCount": 1
}
```

검증:

- 기존 액션 아이템을 보관하고 같은 점수판에 새 액션 아이템을 생성한다.
- `replacementPeriod`는 MVP에서 `WEEKLY`만 허용한다.
- `replacementTargetValue`는 `1..7`.
- `replacementTrackingMode`는 `BOOLEAN` 또는 `COUNT`.
- `replacementDailyTargetCount`는 `1..20`, `BOOLEAN`이면 `1`.
- 기존 액션 아이템의 태그는 자동 복사하지 않는다. 태그 복사는 Post-MVP로 둔다.

### 14.11. 표준 에러 코드

공통 코드 외에 아래 코드를 추가한다.

| 코드 | HTTP | 의미 |
| --- | --- | --- |
| `TEAM_CHECKIN_DISABLED` | 403 | 워크스페이스 체크인 정책이 꺼져 있음 |
| `TEAM_CHECKIN_NOT_FOUND` | 404 | 체크인 없음 또는 접근 권한 없음 |
| `TEAM_CHECKIN_ALREADY_RESPONDED` | 409 | 이미 반응한 체크인 |
| `TEAM_CHECKIN_RESPONSE_REQUIRED` | 400 | 조정 제안의 근거가 될 반응이 없음 |
| `TEAM_CHECKIN_RESPONSE_NOT_ACTIONABLE` | 409 | `BLOCKED`/`ADJUSTMENT_REQUESTED`가 아닌 반응에 조정 제안 생성 시도 |
| `TEAM_CHECKIN_PROPOSAL_NOT_FOUND` | 404 | 조정 제안 없음 또는 접근 권한 없음 |
| `TEAM_CHECKIN_PROPOSAL_ALREADY_OPEN` | 409 | 같은 대상에 미응답 제안이 이미 있음 |
| `TEAM_CHECKIN_PROPOSAL_NOT_OPEN` | 409 | 이미 종료된 제안에 응답 시도 |
| `TEAM_CHECKIN_PROPOSAL_EXPIRED` | 409 | 만료된 제안에 응답 시도 |
| `TEAM_CHECKIN_PROPOSAL_APPLY_FAILED` | 409 | 팀원 수락 후 실제 변경 적용 실패 |

### 14.12. 발송 채널과 deep link

MVP 채널은 앱 WebView FCM이다.

- FCM `pushType`: `team_checkin`
- FCM `campaignId`: `team_checkin:{deliveryUid}`
- 체크인 deep link: `/{locale}/{workspaceId}/dashboard/my?checkinId={deliveryUid}`
- 조정 제안 deep link: `/{locale}/{workspaceId}/dashboard/my?proposalId={proposalUid}`
- 웹 브라우저 환경에서는 FCM 수신이 없을 수 있으므로, 팀원 inbox API가 앱 내 미수신/미클릭 상태를 보완한다.

발송 실패 처리:

- FCM invalid token은 기존 notification storage의 token disable 경로를 재사용한다.
- 일부 토큰 실패가 있어도 delivery 자체는 사용자 단위로 `SENT` 또는 `FAILED`를 기록한다. 여러 디바이스 중 하나라도 성공하면 `SENT`로 본다.

### 14.13. Basic usage/value event ledger

체크인은 Basic 유료 가치 검증과 환불/운영 판단에 쓰일 가능성이 있으므로 usage event를 저장한다.

별도 `basic_usage_events` 테이블을 추가하거나 기존 billing usage ledger가 먼저 생기면 그쪽에 합류한다. 1차 구현에서는 체크인 전용 이벤트 저장을 우선 허용한다.

필수 event type:

- `TEAM_CHECKIN_ENABLED`
- `TEAM_CHECKIN_DELIVERY_SENT`
- `TEAM_CHECKIN_RESPONSE_SUBMITTED`
- `TEAM_CHECKIN_FOLLOWUP_RESUMED_24H`
- `TEAM_CHECKIN_REPORT_VIEWED`
- `TEAM_CHECKIN_ADJUSTMENT_PROPOSED`
- `TEAM_CHECKIN_ADJUSTMENT_ACCEPTED`
- `TEAM_CHECKIN_ADJUSTMENT_APPLIED`

저장 원칙:

- event에는 `workspaceId`, `actorUserId`, `targetUserId`, `leadMeasureId`, `eventType`, `occurredAt`, `metadataJson`을 둔다.
- `metadataJson`에는 원문 note나 leaderNote를 저장하지 않는다. 민감한 운영 맥락은 proposal/audit 쪽에만 최소 저장한다.

### 14.14. 테스트 기준

백엔드 1차 구현의 필수 테스트:

- validation:
  - settings schema 범위
  - response type과 note 길이
  - action type별 payload schema
- service:
  - 정책 off면 배치가 후보를 만들지 않음
  - ADMIN 제외 설정이 발송 대상에서 ADMIN을 제외함
  - `SLOW_WEEKLY_START` 후보 탐지
  - 주간 목표 2회 액션 아이템이 월요일 10:00 KST에 기록 0개여도 후보가 되지 않음
  - 생성 후 168시간이 지나지 않은 액션 아이템은 `SLOW_WEEKLY_START` 조건을 만족해도 후보가 되지 않음
  - daily/member/workspace limit과 중복 delivery 방지
  - `SNOOZE_TODAY` 이후 당일 동일 액션 아이템 suppress
  - `BLOCKED`/`ADJUSTMENT_REQUESTED`만 리더 attention item에 포함
  - proposal 생성/수락/거절/취소/만료 상태 전이
  - `CHANGE_TARGET_COUNT`, `ARCHIVE_ACTION_ITEM`, `REPLACE_ACTION_ITEM` 적용
  - 적용 실패 시 `APPLY_FAILED`와 audit 기록
- route:
  - 미인증 401
  - 워크스페이스 비멤버 404 또는 403
  - MEMBER가 settings 수정/report/proposal 생성 시 403
  - 다른 팀원의 inbox/checkin/proposal 응답 접근 차단
  - internal run API의 `CRON_SECRET` 검증
- storage:
  - 주요 unique/index 전제에 맞는 upsert/중복 방지
  - workspace ownership 필터가 query 조건에 포함됨

최종 검증:

```bash
yarn gen:api
yarn test:backend
yarn tsc --noEmit
yarn lint
```

스키마 변경이 있으므로 로컬 마이그레이션은 `yarn mig:local`로만 생성/적용한다. 원격 마이그레이션은 별도 요청 없이는 실행하지 않는다.

## 16. 성공 및 실패 기준

### 성공

- 리더가 목표와 액션 아이템 합의 이후 반복 체크인을 직접 보내지 않아도 된다.
- 팀원이 체크인을 받은 뒤 1탭으로 반응하거나 기록을 재개한다.
- 리더 결과 리포트에서 직접 개입이 필요한 대상이 명확히 줄어든다.
- `막힘 있음`, `목표 조정 필요` 신호가 목표/액션 아이템 재조정으로 이어진다.
- 리더가 구조화된 조정안을 제안하고, 팀원이 수락해 제품 상태가 실제로 바뀐다.
- 조정안 수락 후 기록 재개율이 단순 체크인만 받은 경우보다 높다.
- 팀원 알림 opt-out이나 부정 피드백이 증가하지 않는다.
- Basic 유료 가치 설명이 `조회`보다 `반복 운영 부담 감소`로 더 선명해진다.

### 실패

- 체크인이 감시성 알림이나 자동 잔소리로 받아들여진다.
- 팀원이 체크인을 받은 뒤 기록 대신 알림을 끈다.
- 리더가 여전히 대부분의 팀원에게 직접 메시지를 보내야 한다.
- 리더가 코멘트만 남기고 제품 상태를 바꾸는 액션을 만들지 못한다.
- Dowin이 리더/팀원 동의 없이 목표나 액션 아이템을 바꾸는 것처럼 보인다.
- 팀원이 조정 제안을 수락해도 설정 화면에서 직접 다시 수정해야 한다.
- 후속 기록 재개나 목표 조정 신호가 거의 없다.
- 발송 조건이 넓어져 팀원별 체감 빈도가 과도해진다.
- 결과 리포트가 발송 로그에 머물고 리더 의사결정을 줄이지 못한다.
