# 알림 스케줄 커스터마이즈 기획

## 1. 배경 및 맥락 (Background & Context)

- 현재 DOWIN의 알림은 `매일 밤 9시 기록 리마인드`, `매주 목요일 오후 3시 집중 리마인드`로 고정되어 있다.
- 이 정책은 초기 MVP로는 단순하지만, 실제 사용자의 생활 패턴이나 리더의 팀 운영 리듬과 맞지 않을 가능성이 크다.
- 최근 논의된 `리더용 주간 운영 리포트`와 `팀원별 선행지표 리마인드`도 결국 알림 시점이 맞아야 운영 도구로서 의미가 생긴다.

## 2. 문제 정의 (Problem Statement)

- 제품이 정한 고정 시간은 사용자에게 의미 있는 시간이 아닐 수 있다.
- 알림 시간이 맞지 않으면 알림 자체가 가치 신호가 아니라 소음으로 받아들여질 수 있다.
- 특히 리더용 주간 운영 리포트는 `원하는 요일/시간`에 와야 관리 도구가 되는데, 현재 알림 구조는 그 요구를 직접 반영하지 못한다.

## 3. 목표 (Goals)

### 3.1. MVP 목표

- 기존 개인 기록 리마인드가 고정 시각이 아니라 사용자 설정 기반으로 확장될 수 있는 구조를 만든다.
- 리더용 주간 운영 리포트는 최소한 발행 요일을 커스터마이즈할 수 있게 한다.
- 알림 스케줄 저장 구조를 통합 관점에서 정리해, 개인 알림과 리더 알림이 제각각 따로 자라지 않게 한다.

### 3.2. Post-MVP 목표

- 시간 단위 세밀한 커스터마이즈
- 시간대(Timezone)별 처리
- 평일/주말 분리
- 채널별(푸시/이메일 등) 별도 스케줄
- 달성률 임계치 기반 자동 리마인드 유료 기능 추가

## 4. 범위 정의 (Scope)

### 4.1. 이번 문서가 다루는 것

- 개인 기록 리마인드 시간 커스터마이즈 방향
- 리더 주간 리포트 발행 요일/시간 정책
- 스케줄 저장 구조 초안
- notification 도메인과 profile/settings 도메인 영향

### 4.2. 이번 문서가 다루지 않는 것

- 알림 메시지 카피 자체
- 리더 리마인드 발송 UX 상세
- 이메일/슬랙 등 멀티채널 설계
- AI 기반 최적 발송 시각 추천
- 유료 자동 리마인드의 과금 정책 확정

## 5. 현재 정책 요약 (Current State)

- 개인 기록 리마인드:
  - 매일 밤 9시 KST
- 개인 집중 리마인드:
  - 매주 목요일 오후 3시 KST
- 리더용 알림:
  - 아직 별도 구현 없음

현재 구조는 단순하지만, 아래 문제가 있다.

- 사용자가 기록하는 실제 시간대와 어긋날 수 있다.
- 리더는 팀 운영 요일에 맞는 리포트를 받지 못한다.
- 추후 기능이 늘수록 알림 종류마다 고정 시간이 늘어나 일관성이 깨질 수 있다.

## 6. 제품 원칙 (Product Principles)

- 알림은 제품이 정한 시각보다 사용자가 의미 있다고 느끼는 시각에 와야 한다.
- 개인 알림과 리더 알림은 각각 역할이 다르지만, 저장 구조는 한 방향으로 정리해야 한다.
- MVP에서는 과한 자유도보다 `핵심 커스터마이즈 한두 개`를 우선 제공한다.
- 커스터마이즈가 늘어나도 발송 로직이 복잡도 폭발을 일으키지 않도록 스케줄 모델을 먼저 정리해야 한다.

## 7. MVP 정책 제안 (MVP Proposal)

### 7.1. 개인 기록 리마인드

- 사용자가 기록 리마인드 시간을 직접 설정할 수 있게 한다.
- 기본값은 현재 정책을 유지한다.
  - 기본: 매일 21:00 KST
- MVP에서는 1일 1회만 허용한다.
- `주간 집중 리마인드`는 MVP에서 그대로 유지하거나 비활성화 가능성을 함께 검토한다.
  - 이유: 개인 기록 리마인드와 별개로 두 번째 고정 알림이 계속 남으면 정책 복잡도가 다시 올라간다.

### 7.2. 리더 주간 운영 리포트

- 리더는 발행 요일 1개를 선택할 수 있어야 한다.
- 시간은 MVP에서 아래 둘 중 하나로 정한다.
  - 고정 추천 시간 사용
  - 제한된 시간대 선택 제공
- 최소 기준은 `요일 커스터마이즈 가능`이다.

### 7.3. 팀원별 리더 리마인드

- 수동 발송 액션이므로 예약 발송은 MVP 제외
- 다만 추후 예약 발송으로 확장될 수 있도록 저장 구조를 염두에 둔다.

### 7.4. 유료 자동 리마인드 (Post-MVP)

- 유료 기능으로 `달성률이 n% 이하인 선행지표 자동 리마인드`를 추가할 수 있다.
- 사용자는 아래 3가지를 직접 설정한다.
  - 임계 달성률
  - 발송 주기
  - 발송 시간
- 발송 대상은 활성 선행지표 중 현재 달성률이 기준 이하인 항목으로 제한한다.
- 동일 지표에 대한 과도한 반복 발송을 피하기 위해 최소 cooldown 또는 주간 상한이 필요하다.
- 이 기능은 기본 무료 리마인드와 분리된 `Paid automation`으로 포지셔닝하는 편이 자연스럽다.

## 8. 저장 구조 초안 (Schedule Model Draft)

### 8.1. 개인 알림 설정 예시

```json
{
  "dailyReminderEnabled": true,
  "dailyReminderTime": "21:00",
  "weeklyFocusReminderEnabled": false,
  "weeklyFocusDayOfWeek": 4,
  "weeklyFocusTime": "15:00",
  "timezone": "Asia/Seoul"
}
```

### 8.2. 리더 리포트 설정 예시

```json
{
  "leaderWeeklyReportEnabled": true,
  "leaderWeeklyReportDayOfWeek": 1,
  "leaderWeeklyReportTime": "09:00",
  "timezone": "Asia/Seoul"
}
```

### 8.3. 유료 자동 리마인드 설정 예시

```json
{
  "autoReminderEnabled": true,
  "triggerBelowPercent": 60,
  "sendFrequency": "TWICE_A_WEEK",
  "sendTime": "20:00",
  "timezone": "Asia/Seoul"
}
```

### 8.3. 설계 원칙

- 시간은 문자열 `HH:mm` 형태로 단순 저장
- 요일은 정수 또는 enum으로 저장
- timezone 필드는 MVP에서도 구조상 포함하는 쪽이 안전하다
  - 현재는 KST만 써도, 나중에 모델을 깨지 않게 하기 위함

## 9. 도메인 영향 범위 (Domain Impact)

- notification
  - 고정 시각 크론 발송에서 사용자 설정 기반 발송으로 확장 필요
  - 발송 대상 필터링 시 시간 조건이 추가됨
- profile/settings
  - 개인 알림 시간 설정 UI 필요
- 리더 설정 화면 또는 워크스페이스 설정
  - 리더 주간 리포트 요일/시간 설정 진입점 필요
- analytics
  - 설정률, 설정 변경률, 시간대별 효과 측정 가능

## 10. 구현 전 쟁점 (Open Design Questions)

### 10.1. weekly focus 알림 유지 여부

- 선택지 A: 그대로 유지
  - 장점: 현재 기능 유지
  - 단점: 커스터마이즈가 반쪽이 될 수 있음
- 선택지 B: 개인 daily 알림만 먼저 커스터마이즈
  - 장점: MVP 단순화
  - 단점: 기존 목요일 알림이 계속 고정 정책으로 남음

현재는 B가 더 현실적이다.

### 10.2. 리더 리포트 시간 자유도

- 선택지 A: 요일만 선택, 시간은 고정
  - 장점: 구현 단순
  - 단점: 리더 운영 리듬 반영이 부족
- 선택지 B: 요일 + 제한된 시간대 선택
  - 장점: 실사용 가치가 더 큼
  - 단점: 발송 로직 복잡도 증가

현재는 B가 더 낫지만, 구현 복잡도가 높으면 A로 시작 가능하다.

### 10.3. 설정 저장 위치

- 개인 알림: profile/settings 문맥이 자연스럽다.
- 리더 리포트: workspace/settings 또는 별도 notification settings가 자연스럽다.
- MVP에서는 저장 구조를 통합하되, UI 진입점은 분리될 수 있다.

## 11. 권장 실행 순서 (Recommended Sequence)

1. 개인 기록 리마인드 시간 설정 모델 확정
2. 리더 주간 리포트 요일/시간 정책 확정
3. notification 발송 로직을 설정 기반으로 전환
4. profile/settings 및 리더 설정 UI 추가
5. 효과 측정 이벤트 추가

## 13. 성공 및 실패 판단 기준 (Success & Failure Criteria)

### 성공

- 알림이 더 이상 전원 공통 고정 시각이 아니라 사용자 설정 기반으로 동작한다.
- 리더 주간 리포트가 팀 운영 리듬에 맞게 발행된다.
- 알림 피로를 늘리지 않으면서 실질적인 반응률 개선 신호를 얻는다.

### 실패

- 알림 종류마다 설정 방식이 제각각이라 정책 일관성이 무너진다.
- 커스터마이즈는 생겼지만 실제 발송 로직은 여전히 고정 시간 가정에 묶여 있다.
- 설정 UI만 늘고 실제 반응률이나 기록 재개율 개선은 없다.
