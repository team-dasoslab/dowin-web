# Dowin 수익화 프레임워크 v0

> 상태 메모 (2026-05-19): 이 문서는 Dowin 수익화 사고를 구조화하기 위한 초기 프레임워크 초안이다. 현재 기준으로는 일부 섹션이 실제 제품 우선순위보다 앞서가 있으므로, 실행 기준 문서로 바로 쓰지 않는다. 현재 Dowin 적용판은 `docs/planning/2026.05.19-dowin-monetization-strategy-v1.md`를 우선 참고한다.

## 1. 이 문서의 역할

- 이 문서는 `수익화 전략 방법론 정리`의 후속 문서다.
- 일반적인 가격 전략 원칙을 Dowin에 바로 대입할 수 있도록, 실제 판단 축과 후속 조사 항목을 정리한다.
- 아직 최종 가격표나 최종 플랜을 확정하는 문서는 아니다.
- 현재 제품 우선순위와 정확히 맞물린 실행안이라기보다, 장기 pricing/package 사고를 정리한 프레임 문서에 가깝다.
- 대신 이후 아래 작업의 기준 문서로 사용한다.
  - 플랜 구조 설계
  - 가격 페이지 카피
  - 업그레이드 유도 문구
  - 체험 구조 설계
  - 인터뷰/설문/실험 설계

## 2. Dowin 수익화에서 먼저 고정해야 하는 전제

### 2.1. 제품 정체성 전제

- Dowin은 기록 앱이 아니라 `상태 파악 -> 개입 -> 후속 확인` 운영 루프를 줄여주는 제품으로 봐야 한다.
- 따라서 Dowin의 수익화도 `기록 저장`에 과금하는 구조보다 `운영 문제 해결`에 과금하는 구조가 더 맞다.

### 2.2. 가치 설명 전제

- 고객이 돈을 내는 이유는 아래 중 하나 이상이어야 한다.
  - 팀 상태를 더 빨리 파악할 수 있다
  - 리더의 수기 확인 시간이 줄어든다
  - 개입 시점을 놓치지 않게 된다
  - 후속 확인이 자동화된다
- 즉, Dowin은 `더 예쁜 목표 관리`보다 `더 빠른 운영 판단`을 팔아야 한다.

### 2.3. 패키징 전제

- 플랜 차이는 `기록 가능 여부`보다 `운영 루프의 깊이`에서 나와야 한다.
- 상위 플랜은 저장 공간이 아니라 아래를 더 강하게 제공해야 한다.
  - 더 빠른 가시성
  - 더 쉬운 개입
  - 더 자동화된 후속 확인

## 3. Dowin 수익화 프레임워크의 6개 축

### 3.1. 축 A. 결제자 정의

#### 핵심 질문

- 실제로 Dowin 비용을 내는 사람은 누구인가
- 제품을 매일 여는 사람과 결제 승인자가 같은가
- 비용을 승인할 때 어떤 언어로 납득하는가

#### 현재 Dowin 가설

- 개인 사용자는 진입 채널 역할은 가능하지만, 핵심 결제자는 아닐 가능성이 높다.
- 가장 유력한 결제자는 아래 중 하나다.
  - 소규모 팀 리더
  - 운영 담당자
  - 스터디/프로젝트 운영자

#### 이 축에서 더 조사할 것

- 현재 파일럿 또는 잠재 사용자 중 누가 실제 예산을 결정하는가
- 회사 카드/개인 카드/팀 비용 처리 중 어떤 패턴이 자연스러운가
- 리더가 비용을 설명할 때 쓰는 언어가 `생산성`, `운영`, `관리`, `리포트` 중 무엇에 가까운가

#### 산출물

- 결제자 1차 우선순위 표
- 사용자 vs 결제자 vs 승인자 구분표
- 결제자별 핵심 메시지 1문장씩

### 3.2. 축 B. 가치 지표(Value Metric) 정의

#### 핵심 질문

- 고객이 더 큰 가치를 얻을수록 함께 커지는 단위는 무엇인가
- 그 단위가 고객에게 이해 가능하고 예측 가능한가
- 그 단위가 실제 예산과 연결되는가

#### Dowin 후보 지표

- 멤버 수
- 활성 멤버 수
- 워크스페이스 수
- 체크인/자동화 실행 수
- 리더 기능 묶음 수준
- 운영 리포트/분석 수준

#### 후보별 1차 해석

##### 멤버 수

- 장점:
  - 설명이 쉽다
  - 팀 제품처럼 보이기 좋다
- 한계:
  - Dowin의 핵심 가치가 단순 협업보다 `운영 개입`이라면 설명력이 약할 수 있다
  - 사용은 적은데 멤버만 많은 팀에서 저항이 생길 수 있다

##### 활성 멤버 수

- 장점:
  - 실제 사용 가치와 더 가까울 수 있다
  - 작은 팀 입장에서 공정하게 느껴질 가능성이 있다
- 한계:
  - 계산 방식 설명이 필요할 수 있다
  - 운영상 측정 기준을 명확히 정의해야 한다

##### 자동화 실행 수

- 장점:
  - `운영 자동화` 가치와 직접 연결될 수 있다
  - 상위 플랜 차별화에 유리하다
- 한계:
  - 초기에 너무 복잡해질 수 있다
  - 고객이 쉽게 예측하지 못하면 불안감이 생긴다

##### 리더 기능 수준

- 장점:
  - Dowin의 핵심 구매 이유와 직접 연결되기 쉽다
  - `운영 깊이` 기반 패키징에 적합하다
- 한계:
  - 순수 기능 티어형이라 확장 매출이 자연 증가하지 않을 수 있다

#### 현재 판단

- Dowin은 단일 가치 지표 하나만으로 닫히기보다 아래 조합이 유력하다.
  - 기본 플랜 구조는 `운영 단계형 티어`
  - 확장 기준은 `팀 규모` 또는 `활성 멤버`
  - 상위 가치 차별화는 `운영 자동화`와 `리더 도구`

#### 산출물

- 가치 지표 후보 비교표
- 가치 지표 선택 원칙
- 측정 방식 초안

### 3.3. 축 C. 세그먼트 정의

#### 핵심 질문

- 누구의 어떤 운영 상황을 각 플랜이 대표하는가
- 세그먼트별 지불 의사 차이는 어디에서 발생하는가

#### Dowin 1차 세그먼트 가설

##### 세그먼트 1. 개인 실행 회복형

- 상황:
  - 목표는 자주 세우지만 흐려진다
- 반응 가치:
  - 개인 실행 리듬 유지
- 한계:
  - 지불 의사가 가장 약할 가능성이 높다
- 현재 역할 메모:
  - 핵심 유료 세그먼트라기보다 `무료 진입 채널` 또는 보조 세그먼트로 보는 편이 맞다

##### 세그먼트 2. 소규모 팀 리더형

- 상황:
  - 3~10명 내외 팀을 운영한다
  - 주간 상태를 빨리 파악하고 싶다
- 반응 가치:
  - 회의 준비 시간 감소
  - 멤버 상태 가시성

##### 세그먼트 3. 운영 강도 높은 팀형

- 상황:
  - 단순 조회보다 리더 개입과 후속 확인이 중요하다
- 반응 가치:
  - 자동 체크인
  - 위험 신호 감지
  - 리더 브리핑

#### 현재 판단

- Dowin의 핵심 유료 세그먼트는 `개인`보다 `리더/운영자`일 가능성이 높다.
- 따라서 가격표는 소비자 앱처럼 `개인 취향` 기준이 아니라 `팀 운영 강도` 기준으로 설계하는 편이 맞다.

#### 산출물

- 세그먼트별 JTBD 요약
- 세그먼트별 지불 의사 가설
- 세그먼트별 대표 메시지

### 3.4. 축 D. 업그레이드 트리거 정의

#### 핵심 질문

- 사용자가 어떤 순간에 `돈 내야겠다`고 느끼는가
- 그 순간이 제품 가치 체험 이후에 오는가

#### Dowin 업그레이드 트리거 후보

- 팀원을 더 많이 초대하려는 순간
- 팀 전체 상태를 한 화면에서 보고 싶은 순간
- 회의 전에 요약이 필요한 순간
- 누가 막혔는지 자동으로 알고 싶은 순간
- 자동 체크인/리마인드/후속 보고가 필요한 순간
- 관리자/권한/보안 기능이 필요한 순간

#### 좋은 트리거 조건

- 고객이 이미 Dowin의 기본 가치를 경험한 이후여야 한다
- 돈을 내면 해결되는 문제가 분명해야 한다
- 억지 제한처럼 느껴지지 않아야 한다

#### 나쁜 트리거 예시

- 핵심 가치를 보기 전에 너무 빨리 막히는 경우
- 왜 제한되는지 설명이 안 되는 경우
- 사용량은 적은데 가격 압박부터 오는 경우

#### 산출물

- 플랜별 업그레이드 트리거 맵
- 제품 내 노출 위치 초안
- 트리거별 카피 방향

### 3.5. 축 E. 체험 구조 정의

#### 핵심 질문

- Dowin은 무료 없음, 무료 체험, 제한형 체험 중 무엇이 맞는가
- 고객이 가치 체험에 필요한 최소 시간은 얼마인가

#### Dowin에서 검토할 선택지

##### 선택지 1. 영구 무료

- 장점:
  - 진입 장벽이 낮다
- 한계:
  - 유료 전환 긴장감이 약해질 수 있다
  - 운영 지원 비용이 커질 수 있다

##### 선택지 2. 기간 제한 무료 체험

- 장점:
  - 전면 유료 구조와 잘 맞는다
  - 체험 종료 시 전환 논리가 명확하다
- 한계:
  - Dowin의 가치가 3일 안에 안 보이면 전환이 약할 수 있다

##### 선택지 3. 특정 운영 기능 체험

- 예:
  - 첫 주 자동 체크인 체험
  - 리더 브리핑 1회 체험
  - 팀 대시보드 전체 열람 체험
- 장점:
  - 유료 핵심 가치만 체험시킬 수 있다
- 한계:
  - 설계가 섬세해야 한다

#### 현재 판단

- Dowin은 `영구 무료` 여부보다 `유료 핵심 운영 가치를 충분히 체험하게 만들 것인가`가 더 중요하다.
- 현재 방향상 가장 유력한 후보는 아래 둘이다.
  - 전체 기능 기간 제한 체험
  - 상위 운영 기능 제한 체험

#### 산출물

- 체험 구조 비교표
- 추천 체험 구조 1안
- 체험 종료 후 전환 카피 초안

### 3.6. 축 F. 검증 설계

#### 핵심 질문

- 어떤 데이터가 모이면 이 수익화 구조가 맞다고 볼 것인가
- 설문과 실제 행동 데이터를 어떻게 같이 볼 것인가

#### Dowin에서 필요한 검증 방식

##### 인터뷰

- 목표:
  - 구매 언어 확인
  - 예산 승인 구조 확인
  - 가장 큰 운영 문제 확인

##### 가격 설문

- 목표:
  - 허용 가격 범위 확인
  - 플랜별 반응 차이 확인

##### 패키징 설문 또는 Conjoint

- 목표:
  - 어떤 혜택 조합이 실제 전환 의사를 높이는지 확인

##### 랜딩/제품 실험

- 목표:
  - 실제 클릭과 결제 의사 확인

#### Dowin 검증 핵심 지표 후보

- 가격 페이지 CTA 클릭률
- 체험 시작률
- 체험 완료 후 결제 전환율
- 플랜별 선택 비중
- 업그레이드율
- 업그레이드까지 걸리는 시간
- 플랜별 이탈률

#### 산출물

- 인터뷰 질문지
- 가격 설문안
- 랜딩 실험안
- 성공 판단 기준표

## 4. Dowin 수익화 프레임워크를 실제 플랜 설계에 연결하는 법

### 4.1. 플랜 숫자를 바로 정하지 말고 의미를 먼저 붙인다

- `5 / 10 / 20 / 30`이 있다면 먼저 아래를 붙여야 한다.
  - 어떤 결제자를 위한 플랜인가
  - 어떤 운영 문제를 해결하는가
  - 어떤 업그레이드 트리거를 반영하는가

### 4.2. 플랜 차이는 기능 수보다 운영 단계 차이여야 한다

- 예시 축:
  - 개인/소규모 운영 시작
  - 팀 상태 가시성
  - 리더 개입 도구
  - 운영 자동화

### 4.3. 가격표는 비교표가 아니라 선택표여야 한다

- 사용자는 `기능 27개 비교`보다 `우리 팀은 어느 단계인가`를 더 빨리 이해해야 한다.

## 5. 현재 기준의 권장 다음 조사 순서

1. 결제자 인터뷰 가설 정리
2. 가치 지표 후보 비교표 작성
3. 세그먼트별 메시지 초안 작성
4. 업그레이드 트리거 맵 작성
5. 체험 구조 2안 비교
6. 가격/패키징 검증 질문지 설계

## 6. 지금 단계의 잠정 결론

- Dowin 수익화 프레임워크의 핵심은 `얼마를 받을까`보다 `누가 왜 어떤 순간에 업그레이드하는가`다.
- Dowin은 기록 도구보다 운영 도구로 설명할 때 수익화 논리가 더 강하다.
- 따라서 이후 플랜 설계는 아래 세 가지를 중심으로 진행하는 편이 맞다.
  - 결제자는 누구인가
  - 가치 지표는 무엇인가
  - 상위 플랜 전환 이유는 무엇인가
