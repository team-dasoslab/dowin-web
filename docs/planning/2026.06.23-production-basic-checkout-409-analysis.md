# Production Basic Checkout 409 분석 메모

## 1. 현상

프로덕션에서 사용자가 `Basic 구독하기`를 누르면 아래 요청이 실패한다.

```text
POST https://dowin.app/api/workspaces/vSIs8wxbEX5D/billing/checkout
Status: 409 Conflict
```

이 엔드포인트는 기존 워크스페이스가 `NONE` 또는 `EXPIRED` billing 상태일 때 새 Basic checkout을 시작하는 복구/재구독 경로다.

## 2. 현재 코드 기준 결론

이 409는 일반적인 서버 장애라기보다 서버가 checkout을 열 수 없는 상태라고 판단했을 때 의도적으로 반환하는 응답이다.

관련 코드:

- `src/app/api/workspaces/[workspaceId]/billing/checkout/route.ts`
  - 세션 확인
  - `Idempotency-Key` 검증
  - `createPolarBillingClient(env)` 생성
  - `BillingService.startBasicCheckout()` 호출
- `src/domain/billing/services/billing.service.ts`
  - billing 상태, 수동 검토 여부, Polar 설정, provider product 매핑, Polar checkout API 결과를 확인
  - 실패 조건에서 `ConflictError("BILLING_NOT_READY")` 또는 `ConflictError("BILLING_REVIEW_REQUIRED")`를 던짐

따라서 브라우저 Network response body의 `error.code`를 먼저 확인해야 한다.

```json
{
  "error": {
    "code": "BILLING_NOT_READY",
    "message": "..."
  }
}
```

또는:

```json
{
  "error": {
    "code": "BILLING_REVIEW_REQUIRED",
    "message": "..."
  }
}
```

## 3. 409 발생 조건

`BillingService.startBasicCheckout()` 기준으로 409가 나는 조건은 아래와 같다.

### 3.1. `BILLING_REVIEW_REQUIRED`

아래 중 하나면 checkout이 막힌다.

- 최근 30일 이내 refund/revoked 위험 이벤트 합계가 기준치 이상
- 현재 billing 상태가 `REVOKED`

확인 대상:

- `billing_events`
- `workspace_billing_state`
- 관리자 billing 화면 또는 직접 DB 조회

### 3.2. `BILLING_NOT_READY`

아래 중 하나면 checkout이 막힌다.

- billing 상태가 `NONE` 또는 `EXPIRED`가 아님
  - `ACTIVE`, `CANCELED`는 새 checkout 대신 portal 중심으로 처리해야 한다.
- `REVOKED` 상태라 새 checkout 대신 수동 검토가 필요함
- 프로덕션 Polar client 생성 실패
  - `POLAR_ENV`
  - `APP_BASE_URL`
  - `POLAR_CHECKOUT_TOKEN` 또는 `POLAR_ACCESS_TOKEN`
- 프로덕션 DB에 active Basic provider product 매핑이 없음
  - `provider = POLAR`
  - `environment = production`
  - `plan_code = BASIC`
  - `is_active = true`
- Polar checkout API가 recoverable error를 반환
  - 400, 401, 403, 404, 409, 422
  - 예: product ID가 sandbox/live 환경과 맞지 않음, token scope 문제, 잘못된 product ID

## 4. 가장 의심되는 원인

프로덕션에서만 발생한다면 우선순위는 다음과 같다.

1. `billing_provider_products`에 production Basic product 매핑이 없거나 비활성이다.
2. Cloudflare production secret에 Polar 설정이 빠져 있거나 `POLAR_ENV`가 실제 product 매핑과 다르다.
3. production DB에는 매핑이 있지만 provider product ID가 sandbox product ID이거나 Polar live product ID와 맞지 않는다.
4. 해당 워크스페이스가 `REVOKED` 또는 수동 검토가 필요한 billing 상태다.

## 5. 확인 순서

### 5.1. 브라우저 응답 body 확인

Network 탭에서 실패 요청의 response body를 확인한다.

- `BILLING_REVIEW_REQUIRED`
  - billing risk 또는 `REVOKED` 상태 문제다.
- `BILLING_NOT_READY`
  - 상태, 설정, provider product, Polar API 실패 중 하나다.

### 5.2. 해당 워크스페이스 billing 상태 확인

워크스페이스 public uid가 `vSIs8wxbEX5D`라면 내부 workspace id를 먼저 찾는다.

```sql
select id, uid, name, plan_code, billing_customer_external_ref, billing_owner_user_id
from workspaces
where uid = 'vSIs8wxbEX5D';
```

그 다음 billing projection을 확인한다.

```sql
select
  workspace_id,
  provider,
  billing_status,
  plan_code,
  entitlement_source,
  customer_key,
  subscription_key,
  current_period_end,
  cancel_at_period_end,
  billing_owner_user_id,
  last_event_occurred_at,
  updated_at
from workspace_billing_state
where workspace_id = <workspace_id>;
```

판단:

- row가 없으면 `billingStatus = NONE`으로 취급된다.
- `billing_status`가 `NONE` 또는 `EXPIRED`여야 checkout 가능하다.
- `NONE` 또는 `EXPIRED`이면 마지막 `entitlement_source`가 `POLAR`, `BETA_PROMOTIONAL_GRANT`, `MANUAL_GRANT` 등 무엇이든 새 Basic checkout을 열 수 있다.
- `REVOKED`면 수동 검토 흐름이다.

### 5.3. production Basic provider product 매핑 확인

```sql
select provider, environment, plan_code, provider_product_id, is_active, updated_at
from billing_provider_products
where provider = 'POLAR'
  and environment = 'production'
  and plan_code = 'BASIC';
```

기대값:

```text
provider = POLAR
environment = production
plan_code = BASIC
provider_product_id = <Polar live Basic product id>
is_active = 1
```

주의:

- `environment = sandbox` 매핑만 있으면 프로덕션 checkout은 실패한다.
- production row가 있어도 `provider_product_id`가 sandbox product id면 Polar API 단계에서 실패할 수 있다.

### 5.4. Cloudflare production secret 확인

프로덕션 Worker에 아래 값이 있어야 한다.

```text
POLAR_ENV=production
APP_BASE_URL=https://dowin.app
POLAR_CHECKOUT_TOKEN=<live checkout token>
```

또는 checkout token 대신 access token을 쓴다면:

```text
POLAR_ACCESS_TOKEN=<live access token>
```

주의:

- `wrangler.jsonc`에는 Polar 값이 vars로 고정되어 있지 않다.
- 프로덕션에서는 Cloudflare secrets 또는 배포 환경 변수에 의존한다.
- `POLAR_ENV=sandbox`인데 DB에는 production product만 있거나, 반대로 `POLAR_ENV=production`인데 DB에는 sandbox product만 있으면 checkout product 조회가 실패한다.

### 5.5. Polar API 실패 로그 확인

현재 checkout 경로는 Polar recoverable error를 `BILLING_NOT_READY`로 감싼다.

필요하면 Cloudflare logs에서 checkout 요청 시점 주변의 Polar API 실패 로그를 확인한다.

중점:

- Polar status code
- sanitized Polar response body
- product ID 환경 불일치
- token 인증 또는 scope 오류

## 6. 즉시 조치 후보

### 6.1. production provider product 매핑이 없다면

관리자 API 또는 관리자 화면으로 production Basic product mapping을 등록한다.

요청 형태:

```json
{
  "provider": "POLAR",
  "environment": "production",
  "planCode": "BASIC",
  "providerProductId": "<Polar live Basic product id>",
  "isActive": true,
  "changeReason": "Production Basic checkout 활성화를 위한 Polar product 매핑 등록"
}
```

대상 API:

```text
POST /api/admin/billing/provider-products
```

필요 권한:

```text
SYSTEM_ADMIN
```

### 6.2. Polar secret이 없다면

Cloudflare production secret에 값을 추가한 뒤 재배포 또는 Worker secret 반영 상태를 확인한다.

필수:

```text
POLAR_ENV=production
APP_BASE_URL=https://dowin.app
POLAR_CHECKOUT_TOKEN=<live checkout token>
```

portal까지 안정화하려면 `customer_sessions:write` scope가 있는 `POLAR_ACCESS_TOKEN`도 별도로 필요할 수 있다. 다만 이번 checkout 409의 1차 필수값은 checkout 생성에 필요한 token이다.

### 6.3. 워크스페이스 상태가 checkout 불가 상태라면

- `ACTIVE` 또는 `CANCELED`
  - 새 checkout이 아니라 portal 진입이 맞다.
- `REVOKED` 또는 `BILLING_REVIEW_REQUIRED`
  - 운영자 검토 후 manual correction 또는 support 대응이 필요하다.
- non-Polar entitlement
  - Polar checkout을 새로 여는 대신 해당 entitlement source 정책에 맞게 처리해야 한다.

## 7. 후속 개선 제안

현재 `BILLING_NOT_READY`가 너무 넓다. 같은 409라도 운영 원인이 다르기 때문에 프로덕션 디버깅이 어렵다.

개선 후보:

1. 서버 로그에 checkout 실패 reason을 구조화해서 남긴다.
   - `POLAR_CLIENT_NOT_CONFIGURED`
   - `BASIC_PROVIDER_PRODUCT_MISSING`
   - `UNSUPPORTED_BILLING_STATUS`
   - `NON_POLAR_ENTITLEMENT`
   - `POLAR_CHECKOUT_CREATE_FAILED`
2. 클라이언트에는 기존처럼 넓은 메시지를 유지하되, 운영 로그에는 workspace uid/id, billing status, entitlement source, Polar environment를 남긴다.
3. 관리자 billing 화면에서 provider product 매핑 상태와 production readiness를 한 번에 볼 수 있게 한다.

## 8. 다음 세션 시작 액션

1. 브라우저 Network response body에서 `error.code` 확인
2. production DB에서 `billing_provider_products` production Basic row 확인
3. Cloudflare production secret의 `POLAR_ENV`, `APP_BASE_URL`, `POLAR_CHECKOUT_TOKEN` 확인
4. 해당 workspace의 `workspace_billing_state` 확인
5. 필요하면 `BILLING_NOT_READY` 세부 reason 로깅 개선 작업으로 전환
