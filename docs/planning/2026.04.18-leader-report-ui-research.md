# 리더 리포트 UI 리서치

## 1. 왜 별도 리서치가 필요한가

- 현재 `src/app/[locale]/(protected)/report/page.tsx` 프로토타입은 요약, 규칙, 액션, 대상자 설명이 같은 무게로 경쟁해 리더가 어디부터 봐야 하는지 흐려진다.
- 이 화면은 "예쁜 대시보드"보다 "짧은 상태 브리프"에 가깝게 설계해야 한다.
- 따라서 일반적인 analytics dashboard보다, 실제로 `상태 요약 -> 위험 신호 -> 다음 확인 대상` 흐름이 검증된 제품 UI를 참고하는 편이 맞다.

## 2. 이번 리서치에서 찾는 기준

이번 화면은 아래 질문 3개에 빨리 답해야 한다.

1. 팀이 지금 이기고 있는가
2. 실행이 실제로 시작됐는가
3. 누구를 먼저 확인해야 하는가

레퍼런스 선정 기준:

- 숫자와 상태가 첫 화면에서 바로 읽히는가
- 카드 수가 많아도 시선의 우선순위가 명확한가
- 긴 설명 없이도 다음 액션 대상이 드러나는가
- 리더/관리자용 브리프처럼 읽히는가

## 3. 우선 참고할 제품

### 3.1. Linear Project Overview

출처:

- https://linear.app/docs/project-overview

왜 참고하는가:

- 개요 화면에서 설명보다 핵심 상태와 프로젝트 속성을 우선 보여준다.
- 복잡한 시각 요소보다 밀도 있는 정보 배치로 빠른 스캔에 강하다.
- 상세 설명은 뒤로 보내고, 상단에는 현재 상태를 파악하는 데 필요한 정보만 둔다.

DOWIN에 가져올 점:

- 리더 리포트도 hero 영역에서 감성 카피보다 상태 요약을 먼저 보여준다.
- 대상자 목록은 "카드 묶음"보다 조밀한 list/table 성격으로 간다.
- 불필요한 장식보다 정렬, 간격, 라벨 계층으로 우선순위를 만든다.

가져오지 않을 점:

- 너무 제품 관리 도구 같은 무채색/업무툴 톤을 그대로 복제하지는 않는다.
- 상세 메타데이터를 많이 쌓아 복잡하게 만드는 방식은 피한다.

### 3.2. Asana Status Update / Project Overview

출처:

- https://help.asana.com/s/article/project-progress-and-status-updates?language=en_US
- https://asana.com/features/project-management/status-updates

왜 참고하는가:

- 상태 업데이트를 작성할 때 `Summary`, `Next steps`, `Highlights`처럼 반복 가능한 블록 구조를 강제한다.
- 한 번에 모든 데이터를 보여주기보다, 상태와 핵심 하이라이트를 먼저 읽게 만든다.
- 주간 업데이트 리마인드와 히스토리 개념이 있어, DOWIN의 리더 리포트와 문제 구조가 가깝다.

DOWIN에 가져올 점:

- 리포트는 자유형 랜딩 페이지가 아니라 고정된 블록 구조여야 한다.
- `요약 1개 + 핵심 수치 + 상태 하이라이트 + 확인 대상`처럼 읽기 순서가 고정된 구조를 쓴다.
- 리포트 텍스트는 길게 쓰기보다 섹션 제목 자체가 의미를 가져야 한다.

가져오지 않을 점:

- 차트 삽입, milestone 드래그, 풍부한 편집기 같은 작성 기능은 MVP 범위가 아니다.
- Dowin 리더 리포트는 status authoring 도구가 아니라 수신 브리프에 더 가깝다.

### 3.3. Jira Project Updates Summary

출처:

- https://support.atlassian.com/jira-cloud-android/docs/view-project-updates-in-jira-mobile/

왜 참고하는가:

- 모바일 요약에서 `읽지 않은 업데이트 수`, `On track / At risk / Off track` 개수를 먼저 보여주는 식으로 집계 우선순위가 분명하다.
- 상태 분포를 먼저 보고, 개별 업데이트는 뒤에서 확인하는 흐름이 선명하다.

DOWIN에 가져올 점:

- 개별 팀원 설명 전에 `팀 전체 상태 분포`를 먼저 보여준다.
- 상세 사유를 모두 펼치지 말고, 먼저 "몇 명이 문제 상태인지"를 집계로 보여준다.
- 개입 대상 목록은 전체 분포 아래에 두는 게 맞다.

가져오지 않을 점:

- 상태 카테고리를 너무 많이 늘리지 않는다.
- reaction/comment 같은 협업 피드는 MVP에 필요 없다.

### 3.4. monday.com Dashboard / Project Health

출처:

- https://monday.com/features/dashboards
- https://support.monday.com/hc/en-us/articles/22598441769746-Project-boards-on-monday-com

왜 참고하는가:

- 숫자 위젯과 health summary가 "현재 어디가 위험한지"를 빠르게 보여주는 패턴을 잘 갖고 있다.
- 한눈에 보이는 집계 블록 자체는 참고할 가치가 있다.

DOWIN에 가져올 점:

- 상단 숫자 블록은 3개 정도로 제한하고, 제목보다 숫자가 먼저 읽히게 만든다.
- 상태 신호는 컬러를 남발하지 말고 제한된 health semantics만 쓴다.

가져오지 않을 점:

- 위젯이 많은 dashboard 스타일은 이번 화면에 맞지 않는다.
- 리더 리포트는 조립형 dashboard가 아니라 고정 브리프여야 한다.

## 4. HR 제품 공통 패턴

리더/매니저용 HR 제품은 도메인이 달라도 거의 같은 구조를 반복한다.

- `지금 상태`
  - 현재 점수, 현재 참여율, 현재 위험 신호
- `방향`
  - 지난 기간 대비 좋아지는지, 멈추는지, 악화되는지
- `분포`
  - 누가 상위권인지보다 어떤 상태군에 사람이 몰리는지
- `행동`
  - 지금 누구에게 어떤 조치를 해야 하는지

제품별로 다르게 보이지만 실제 핵심은 아래와 같다.

- Lattice:
  - 팀 상태 요약 + real-time insights + adoption/participation
- 15Five:
  - 성과/몰입/이탈 위험을 한 화면에서 보고 action plan으로 연결
- Culture Amp:
  - completion, feedback progress, manager action progress
- Leapsome:
  - goals/performance/engagement를 연결해서 기간별 방향을 읽게 함

DOWIN에 번역하면 아래 구조가 된다.

- 상태:
  - 이번 주 Win/Loss, 실행 시작률, 즉시 확인 필요 인원
- 방향:
  - 최근 3주 Win 인원 추세
  - 최근 3주 실행 시작률 추세
  - 최근 3주 점수판 보유율 추세
- 분포:
  - 이기는 중 / 밀리는 중 / 아직 시작 안 함 / 점수판 없음
- 행동:
  - 지금 개입할 3명과 다음 액션

즉, 리더 리포트는 "현재 상태표"가 아니라 `건강도 + 이동 방향 + 행동 큐`여야 한다.

## 5. 결론: 어떤 UI를 참고해서 만들 것인가

우선순위는 아래 순서로 둔다.

1. **Linear**
   - 전체 정보 밀도와 목록형 정리 방식 참고
2. **Asana**
   - 상태 브리프의 블록 구조와 요약 방식 참고
3. **Jira**
   - 상태 분포를 먼저 보여주는 집계 우선순위 참고
4. **monday.com**
   - 상단 숫자 위젯의 시선 처리만 제한적으로 참고

즉, 이번 화면은 아래처럼 조합하는 것이 맞다.

- 골격: Linear
- 섹션 구조: Asana
- 상단 집계 사고방식: Jira
- 숫자 카드 톤: monday.com 일부

## 6. Dowin 리더 리포트에 적용할 구체 규칙

### 5.1. 레이아웃

- 상단 hero는 큰 카피 블록이 아니라 상태 요약 블록으로 쓴다.
- 상단 첫 화면은 "지금 상태"에 집중하고, 그 아래에서 "왜 그런지"와 "누가 그런지"를 푼다.
- 개입 대상은 카드 3개보다 리스트 1개가 우선이지만, 그 전에 상태 분포 층이 있어야 한다.

권장 순서:

1. 이번 주 팀 요약 문장
2. 핵심 숫자 3개
3. 방향 요약
4. 상태 분해
5. 팀 상태 분포 보드
6. 먼저 확인할 팀원 리스트
7. 변화 요약
8. 보조 CTA

### 5.2. 숫자 카드

- 4개 이상 두지 않는다.
- `미기록`, `점수판 없음`을 상단에서 따로 찢지 말고 `즉시 확인 필요`로 합친다.
- 카드 제목보다 숫자가 먼저 보여야 한다.

### 5.2.a. 상태 분해 블록

- 숫자 카드 바로 아래에는 "왜 즉시 확인이 필요한지"를 짧게 푸는 블록이 필요하다.
- 예:
  - `Win 3명 / Loss 2명`
  - `기록 시작 4명 / 미시작 1명`
  - `점수판 없음 1명`
- 차트보다 compact row 또는 짧은 summary sentence가 맞다.

### 6.2.a. 방향 블록

- 리더가 가장 먼저 알아야 하는 건 현재 수치 자체보다 "좋아지고 있는지"다.
- 따라서 숫자 카드 다음에는 작은 추세 블록이 온다.

예:

- `Win 인원 2 → 3 → 3`
- `실행 시작률 40% → 60% → 80%`
- `점수판 보유율 80% → 80% → 100%`

표현 원칙:

- 정교한 차트보다 짧은 스파크라인, 단계 요약, 화살표 표현이 맞다.
- 초점은 정확한 통계 해석보다 방향 감각이다.
- 지난 3주 정도면 충분하다.

### 6.2.b. 상태군 보드

- 리더는 상위 3명보다 팀의 현재 분포를 먼저 봐야 한다.
- 따라서 사람을 순위로 나열하기보다 상태군으로 묶어 보여준다.

권장 상태군:

- `이기는 중`
- `시작했지만 밀리는 중`
- `아직 시작 안 함`
- `점수판 없음`

표시 원칙:

- 각 군집마다 인원 수를 크게 보여준다.
- 대표 이름 1~2명을 붙여 실제 사람 감각은 살린다.
- 위젯보다는 compact panel 성격이 맞다.

### 6.3. 대상자 목록

- 한 사람당 정보는 한 줄 또는 두 줄 이내로 제한한다.
- 이름, 상태 라벨, 현재 점수, 다음 액션 한 줄만 둔다.
- `왜 이 사람이 위험한지`를 장문 문단으로 쓰지 않는다.

### 6.3.a. 변화 요약

- 화면을 더 매력적으로 만드는 건 정적 상태보다 "변화"다.
- 가능하면 아래 종류를 짧게 붙인다.
  - `새로 시작한 사람`
  - `회복한 사람`
  - `멈춘 사람`
- 현재 API로 확정 안 되면 mock/secondary tone으로만 둔다.

### 6.4. 카피 톤

- 평가보다 운영 톤을 유지한다.
- `낮다`, `부진하다`, `문제 인물` 같은 사람 중심 표현은 피한다.
- `이번 주 시작 확인`, `설정 필요`, `재조정 논의`처럼 행동 중심 표현을 쓴다.

### 6.5. 시각 톤

- 상태 색은 3개 이내로 제한한다.
- 강한 배경색 카드 여러 개를 병렬로 두지 않는다.
- 강조는 배경색보다 숫자 크기, 정렬, 굵기 차이로 만든다.

## 7. 만들면 안 되는 방향

- 갤러리형 analytics dashboard처럼 카드와 위젯을 계속 늘리는 방식
- 설명 카드와 규칙 카드를 본문과 같은 무게로 배치하는 방식
- 팀원별 큰 컬러 카드에 장문 설명을 넣는 방식
- 숫자 3개만 남겨 앱 화면을 푸시 본문처럼 만들어버리는 방식
- 현재 수치만 있고 방향성이 없는 정적 현황판으로 끝나는 방식
- 현재 API로 확정되지 않은 상태를 이미 계산된 진실처럼 보여주는 방식

## 8. 바로 구현에 넘길 화면 계약

```text
[이번 주 팀은 5명 중 3명이 이기고 있어요]
아직 시작하지 않은 1명과 점수판이 없는 1명이 있어 먼저 확인이 필요합니다.

[이기는 중 3/5] [실행 시작 4/5] [즉시 확인 필요 2명]

방향
- Win 인원 2 → 3 → 3
- 실행 시작률 40% → 60% → 80%
- 점수판 보유율 80% → 80% → 100%

상태 분해
- Win 3명 / Loss 2명
- 기록 시작 4명 / 미시작 1명
- 점수판 없음 1명

팀 상태 분포
- 이기는 중 3명 | 유진, 태호
- 시작했지만 밀리는 중 1명 | 지수
- 아직 시작 안 함 1명 | 민수
- 점수판 없음 1명 | 현우

먼저 확인할 팀원
- 민수 | 이번 주 미기록 | 0/5 | 이번 주 시작 가능한 행동부터 확인
- 현우 | 점수판 없음 | 설정 전 | DOWIN와 선행지표 설정부터 열어주기
- 지수 | 2주 연속 Loss | 2/5 | 선행지표 난이도 재조정 논의

이번 주 변화
- 새로 시작한 사람 1명
- 회복한 사람 1명
- 멈춘 사람 1명

[팀 대시보드 보기] [리포트 설정]
```

## 10. 정리

리더 리포트는 dashboard를 예쁘게 만드는 작업이 아니다.  
`Linear처럼 조밀하게`, `Asana처럼 블록을 제한해서`, `Jira처럼 상태 분포를 먼저 보여주고`, `HR 제품들처럼 방향성을 먼저 읽게 하는 방식`이 현재 Dowin 목표와 가장 잘 맞는다.
