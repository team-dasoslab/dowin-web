# AI 작업 품질 평가 기준 (Eval Plan)

## 1. 목적

AI(Agent)가 수행한 작업이 실제 병합(Merge) 및 배포될 수 있는 품질인지 일관되게 평가하기 위함입니다. 코드가 문법적으로 작동하는지(CI/CD)를 넘어서, "AI가 지시사항을 안전하고 정확하게 따랐는지"를 사람이 확인하기 위한 채점표입니다.

## 2. 공통 평가 기준 (General Rubric: O/X Checklist)

모든 AI 작업물은 다음 증거 기반(Evidence-based)의 체크리스트를 작업 범위에 맞춰 `O/X/N/A`로 평가합니다.
해당 작업에 적용되는 항목에서 위반(X)이 발견될 경우 작업을 반려(Fail)하고 `failure_categories`를 태깅하여 재시도하도록 합니다.
작업 범위와 무관한 항목은 `N/A`로 표시하되, 왜 적용 대상이 아닌지 짧게 설명합니다.

### 2.1. 요구사항 및 실행 증명 (Functional & Executable)

- [ ] **기능 증명:** PR 내용에 지시한 기능을 검증하는 신규 Unit/Integration Test 코드가 포함되었거나, 수동 검증(Walkthrough 결과 등) 스크린샷/로그가 명확히 첨부되었는가?
- [ ] **의미 있는 테스트:** 추가된 테스트 코드가 단순히 커버리지만 높이는 껍데기(Tautological test)가 아니며, 핵심 로직이 깨졌을 때 실패할 수 있는 의미 있는 검증(Assertion)을 수행하는가?
- [ ] **할루시네이션(Hallucination) 없음:** `package.json`에 없는 외부 패키지를 `import` 했거나, 존재하지 않는 사내 UI 컴포넌트/API를 임의로 지어내어 호출한 내역이 없는가?
- [ ] **엣지 케이스 처리:** 정상적인 동작뿐만 아니라 에러 발생 상황(API 실패, 빈 배열, null 등)에 대한 예외 처리가 포함되어 있는가?

### 2.2. 변경 범위 및 오버 엔지니어링 (Scope & Over-engineering)

- [ ] **과잉 수정 금지:** 지시한 범위 외의 파일(예: 관련 없는 파일의 줄바꿈 정리, 사용하지 않는 변수 임의 삭제 등)이 변경 목록(Git diff)에 단 1개라도 섞여있지 않은가?
- [ ] **비-목표(Non-goals) 준수:** 프롬프트나 작업 지시서에서 명시적으로 금지한 사항(예: "DB 스키마는 건드리지 말 것")을 철저히 지켰는가?
- [ ] **불필요한 추상화 금지:** 단순한 로직을 불필요하게 여러 파일/함수로 쪼개거나, 오버 엔지니어링된 디자인 패턴을 억지로 도입하지 않았는가?

### 2.3. 코드 품질 및 유지보수성 (Code Quality & Maintainability)

- [ ] **디버깅 잔재 제거:** 임시로 작성한 `console.log`, 주석 처리된(죽은) 코드, 임의로 남긴 `TODO` 주석 등이 최종 작업물에 그대로 남아있지 않은가?
- [ ] **공통 컴포넌트 우선 사용:** 새로운 컴포넌트나 함수를 무작정 만들지 않고 기존 공통 컴포넌트/유틸리티를 사용했는가? (기존 컴포넌트를 적용하기 어려운 명확한 사유가 있을 때만 새로운 생성을 허용함)
- [ ] **가독성:** 변수명과 함수명이 역할을 명확히 설명하고 있으며, 너무 축약되거나 의미 불명인 네이밍(`data1`, `temp` 등)을 피했는가?
- [ ] **종속성 최소화:** 자바스크립트 내장 함수나 기존 유틸리티로 해결할 수 있는 문제에 대해 새로운 외부 의존성(npm 패키지)을 함부로 추가하지 않았는가?

### 2.4. 문서 및 규칙 정합성 (Documentation & Rules)

- [ ] **도메인 규칙 준수:** Dowin 저장소의 컨벤션(`yarn` 패키지 매니저 사용, `dowin_sid` 세션 검증 패턴 등)을 어기지 않고 그대로 따랐는가?
- [ ] **스펙 동기화:** 로직이나 API 핸들러가 수정되었을 때, 이에 대응하는 OpenAPI 스펙(`openapi.yaml`) 및 관련 기획 문서(`docs/planning/*`)가 함께 업데이트되었는가?
- [ ] **리뷰 가이드라인 제공:** 작업 종료 또는 PR 생성 시, 인간 리뷰어를 위해 "핵심 변경 사항"과 "집중해서 봐야 할 파일(Focus List)"을 명시적으로 정리하여 제공했는가?

### 2.5. 보안 및 성능 (Security & Performance)

- [ ] **권한 필터링:** 리소스 조회/수정 시 현재 사용자의 권한(Auth) 및 소유권(Ownership) 필터링 로직이 누락 없이 명시적으로 코드에 구현되어 있는가?
- [ ] **하드코딩 금지:** 비밀번호, API 키, 토큰 등 민감한 정보가 코드에 평문(Hardcoded)으로 노출되지 않았는가?
- [ ] **입력값 검증:** 외부에서 들어오는 모든 입력값(API Payload, Query 등)에 대해 Zod 등을 이용한 엄격한 유효성 검사가 적용되었는가?
- [ ] **성능 핫스팟 우려 없음:** 반복문 내부에서 불필요하게 무거운 연산을 하거나 N+1 형태의 비효율적인 DB 쿼리를 유발하는 코드가 없는가?

## 3. 대표 사례 (Golden Cases)

### 3.1. API 계약 변경 (API Contract Modification)

- **상황:** 백엔드 API의 요청/응답 규격이나 파라미터를 변경하는 작업.
- **Pass 기준:**
  - `src/api-spec/openapi.yaml`을 먼저 갱신하고 `yarn gen:api`를 실행했는가.
  - Zod 스키마와 핸들러 응답이 OpenAPI 스펙과 정확히 일치하는가.
  - 기존 통신 클라이언트(프론트엔드)가 깨지지 않게 하위 호환성을 유지했거나 필요한 곳을 모두 수정했는가.

### 3.2. 프론트 API 연동 (Frontend API Integration)

- **상황:** 생성된 Orval API 훅을 사용하여 UI 컴포넌트에 데이터를 연결하는 작업.
- **Pass 기준:**
  - Orval로 자동 생성된 최신 React Query 훅(`useQuery`, `useMutation`)을 적절히 사용하는가.
  - 로딩 상태와 에러 처리(예: `withErrorHandler` 패턴)가 누락되지 않았는가.
  - API 스펙에 없는 필드를 `as TYPE` 등으로 강제 타입 단언하지 않았는가.

### 3.3. WebView bridge 변경 (WebView Bridge Update)

- **상황:** 네이티브 앱과 웹 브라우저 간의 통신 규격(`src/lib/bridge.ts`)을 변경하는 작업.
- **Pass 기준:**
  - 브리지 추가 시 `docs/dev/app-webview/`의 관련 문서가 업데이트 되었는가.
  - 네이티브 앱 컨텍스트(App-shell)와 일반 웹 브라우저 컨텍스트의 Fallback 처리가 명확히 분리되었는가.

### 3.4. 결제/권한 변경 (Billing/Entitlement Modification)

- **상황:** Workspace 멤버 권한, 요금제, 한도 등 비즈니스 규칙 변경 작업.
- **Pass 기준:**
  - `dowin_sid` 세션을 통한 Auth/Ownership 검증이 모든 관련 엔드포인트에 적용되었는가.
  - 비즈니스 룰 회귀를 방지하기 위해 엄격한 실패 케이스(Fail case) 테스트가 추가되었는가.

### 3.5. 문서/온보딩 정리 (Documentation & Onboarding)

- **상황:** 시스템 아키텍처나 온보딩 문서를 갱신하는 작업.
- **Pass 기준:**
  - 코드를 읽고 실제 구현과 문서 내용 간의 불일치를 해소했는가.
  - "Dowin"의 도메인 룰(서적/프레임워크 기반이 아님)을 어기지 않았는가.

## 4. 실패 카테고리 (Failure Categories)

평가 시 발견된 문제는 정해진 카테고리로 분류하여 피드백합니다.
상세한 카테고리 목록과 설명은 [`CHANGELOG.md`](/.agents/skills/CHANGELOG.md) 파일 상단에 표 형태로 정리되어 있으니 이를 참고하여 분류해 주세요.

## 5. 작업 종료 후 필수 검증 기준 (Post-Work Verification)

AI 에이전트가 단일 태스크나 세션을 종료하기 전에, 작업 범위에 맞는 검증 명령을 실행하거나 실행하지 못한 이유를 명시해야 합니다.
검증은 `AGENTS.md`의 Verification Defaults를 우선 기준으로 삼고, 이 문서는 AI 작업 품질 평가 관점에서 보조 기준을 제공합니다.

### 5.1. 프론트엔드 구현 변경

- `yarn lint`
- `yarn tsc --noEmit`
- `yarn test:frontend`

대상:

- app logic
- UI behavior
- routing
- hooks
- generated API usage
- shared UI components
- user-visible state

### 5.2. 백엔드/API/domain 변경

- `yarn lint`
- `yarn tsc --noEmit`
- `yarn test:backend`

API 계약 변경이면 추가로:

- `yarn gen:api`

### 5.3. 교차 영역 또는 광범위 변경

- `yarn lint`
- `yarn tsc --noEmit`
- `yarn test --run`

### 5.4. 문서/기획/프롬프트/스킬 변경

앱 로직을 함께 수정하지 않은 문서-only, planning-only, prompt/skill instruction-only 변경은 frontend/backend 테스트 게이트를 요구하지 않습니다.

다만 다음 파일처럼 AI 운영 레이어를 건드린 경우에는 완료 전 하네스 보안 점검을 추가합니다.

- `AGENTS.md`
- `codex.md`
- `.agents/skills/**`
- agent permission, approval, or automation guidance

### 5.5. 실패 처리

작업 범위에 맞는 필수 검증 명령이 실패하면 해당 AI 작업물은 병합 조건에 미달한 것으로 간주합니다.
검증을 실행하지 않은 경우에는 결과를 통과로 주장하지 말고, 미실행 사유와 잔여 리스크를 보고해야 합니다.
