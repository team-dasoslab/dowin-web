# Dowin 운영 사고 대응 준비 계획

## 1. 배경 및 맥락 (Background & Context)

- DOWIN는 이제 “기능이 돌아간다”를 넘어서 “장애가 나도 복구할 수 있다”를 기준으로 운영 준비를 해야 하는 단계다.
- 특히 현재 배포 구조는 전통적인 VM 서버가 아니라 `Cloudflare Workers + D1 + OpenNext` 조합이므로, 일반적인 서버 운영 문서를 그대로 가져오면 실제 사고 대응에 맞지 않는 부분이 생긴다.
- 운영 초기에 가장 위험한 것은 아래 4가지다.
  - DB 데이터가 대량 삭제되거나 손상되는 사고
  - 스키마 변경 또는 migration 실수로 컬럼/테이블이 사라지는 사고
  - 서비스 URL은 살아 있지만 앱이 실제로 응답하지 않거나 로그인/API가 실패하는 사고
  - Dowin 코드 문제가 아니라 Cloudflare 플랫폼 장애로 서비스가 영향을 받는 사고

이 문서는 각 사고를 “보통 서비스에서는 어떻게 준비하고 대응하는가”를 기준으로 정리하되, DOWIN의 현재 현실에 맞게 바로 실행 가능한 체크리스트까지 남기는 것을 목표로 한다.

## 2. 우선순위 및 당위성 (Priority & Rationale)

- 운영 초기에 가장 치명적인 리스크는 “데이터 복구 불가”다.
- 그 다음은 “배포는 롤백했는데 DB 스키마는 이미 깨져서 앱이 계속 죽는 상황”이다.
- 전통적인 서버 장애와 달리 DOWIN는 Cloudflare 플랫폼 의존도가 높기 때문에, 플랫폼 장애 시 무엇을 할 수 있고 무엇은 못 하는지를 미리 구분해야 한다.
- 따라서 대응 문서는 아래 순서로 읽히고 실행되어야 한다.

1. 공통 incident 운영 체계
2. 데이터 손실 대응
3. 스키마 변경 사고 대응
4. 서비스 접속 불가 대응
5. Cloudflare 장애 대응

## 3. 공통 운영 원칙 (Common Incident Operating Model)

보통 서비스 운영에서 사고 대응은 기술 조치보다 먼저 “운영 체계”를 고정한다.  
Google SRE 가이드도 incident response의 핵심을 역할 분리와 지속적인 커뮤니케이션으로 둔다.

### 3.1. 공통 역할

- Incident Commander
  - 현재 상황을 한 줄로 정의하고 우선순위를 결정한다.
  - 누가 무엇을 하고 있는지 관리한다.
- Ops Lead
  - 실제 복구 작업을 수행한다.
  - 프로덕션 변경은 이 역할만 수행한다.
- Communications Lead
  - 사용자 공지, 내부 공지, 상태 공유를 담당한다.
  - 운영 중 기술 담당자가 설명 역할까지 동시에 맡지 않도록 분리한다.

초기 소규모 팀이라면 1명이 2개 역할을 겸할 수는 있지만, “복구 담당”과 “상태 공유 담당”은 가능한 한 분리하는 편이 낫다.

### 3.2. 공통 단계

1. 감지
   - 장애를 모니터링, 로그, 사용자 제보로 인지한다.
2. 분류
   - `P1` 전체 서비스 불가
   - `P2` 핵심 기능 일부 불가
   - `P3` 우회 가능하나 기능 저하
3. 안정화
   - 추가 배포 중단
   - 자동화 작업 중단
   - 원인 추정 전이라도 피해 확산을 먼저 막는다.
4. 복구
   - 롤백, restore, failover, 임시 read-only 전환 같은 조치를 한다.
5. 사후 조치
   - postmortem 작성
   - 재발 방지 액션 아이템 등록

### 3.3. 공통 운영 산출물

- 사고 전용 채널 또는 문서 1개
- 타임라인
- 현재 영향 범위
- 임시 우회책
- 최종 원인
- 후속 액션

## 4. 사고 유형 1: DB 데이터가 날아갔을 때

### 4.1. 일반 서비스에서의 표준 대응

보통 서비스는 아래 3단계를 기본으로 둔다.

1. 예방
   - 자동 백업
   - 장기 보관 백업
   - 복구 리허설
   - 위험한 쓰기 작업 전 수동 백업
2. 탐지
   - 삭제/수정 건수 급증 알림
   - 핵심 테이블 row count 이상 감지
3. 복구
   - 가장 최근 정상 시점으로 point-in-time restore
   - 필요 시 특정 테이블만 별도 복원
   - 복구 후 데이터 정합성 검증

핵심은 “백업이 있다”가 아니라 “특정 시점으로 되돌릴 수 있고, 실제로 해본 적 있다”다.

### 4.2. Cloudflare D1 기준으로 볼 점

- D1 production storage는 Time Travel 기반 point-in-time recovery를 지원한다.
- Cloudflare 공식 문서 기준으로 Time Travel은 기본 활성화되어 있고, 별도 설정 없이 복원 시점을 지정할 수 있다.
- 복원은 destructive operation이라 현재 DB를 그 시점 상태로 덮어쓴다.
- restore 시 in-flight query와 transaction이 취소될 수 있다.
- Cloudflare 계정/제품 플랜에 따라 Time Travel 복원 가능 기간이 달라질 수 있으므로, Dowin 운영 플랜의 실제 보존 기간을 배포 전 확인한다.
- 30일보다 긴 보관이 필요하면 D1 export를 R2 같은 외부 저장소로 주기 저장하는 별도 백업 체계가 필요하다.

### 4.3. Dowin 권장 운영 방안

#### 예방

- `wig-db`가 `version: production`인지 먼저 확인한다.
- 운영 전 필수로 아래를 갖춘다.
  - D1 Time Travel 사용 가능 여부 확인
  - 주기적 export 보관 정책
  - 복구 연습 절차
- 위험 작업 전에는 아래 중 하나를 실행한다.
  - `wrangler d1 time-travel info wig-db`로 현재 bookmark 확보
  - `wrangler d1 export wig-db --remote --output=...`로 수동 export 생성

#### 탐지

- 핵심 API 실패율 급증 감지
  - 로그인
  - `GET /api/workspaces/me`
  - 대시보드 조회
  - daily log 쓰기
- Workers Logs에서 SQL 에러, uncaught exception, 응답 코드 급증을 본다.
- 가능하면 주요 테이블 row count 또는 최근 1시간 write volume 이상 탐지를 추가한다.

#### 즉시 대응

1. 추가 배포 중단
2. migration 실행 중이었다면 즉시 중단
3. 관리자 write 작업 중지
4. 가장 최근 정상 시점을 정한다
5. Time Travel restore 또는 export 기반 복원 여부를 결정한다

#### 복구

- DB 전체 손상이라면 Time Travel restore를 우선 검토한다.
- 특정 사용자/워크스페이스 일부만 손상됐다면 전체 restore보다 export/수동 보정이 더 안전할 수 있다.
- 복구 직후 아래를 확인한다.
  - 로그인 가능
  - 워크스페이스 조회 가능
  - 활성 점수판 조회 가능
  - 최근 기록 조회 가능
  - 신규 기록 쓰기 가능

#### 사후 조치

- 어떤 SQL 또는 어떤 코드 경로가 데이터 손실을 일으켰는지 기록
- restore bookmark와 복구 완료 시각 기록
- “전체 restore”와 “부분 보정” 선택 근거 기록

### 4.4. Dowin 출시 전 최소 액션

- [ ] `wrangler d1 info wig-db`로 storage version 확인
- [ ] Time Travel restore를 staging 또는 비운영 DB에서 1회 리허설
- [ ] 장기 보관용 D1 export 백업 전략 결정
- [ ] “데이터 손실 사고 시 15분 내 할 일” 체크리스트 확정

## 5. 사고 유형 2: 스키마 업데이트하다가 컬럼/테이블이 사라졌을 때

### 5.1. 일반 서비스에서의 표준 대응

보통 서비스는 schema migration 사고를 “코드 롤백 문제”가 아니라 “데이터 계약 변경 문제”로 본다.

실무 원칙은 아래가 기본이다.

1. 파괴적 변경은 한 번에 하지 않는다.
2. 먼저 확장하고, 나중에 제거한다.
3. 앱 코드와 스키마를 동시에 되돌릴 수 있는지 확인한다.
4. migration 전후 검증 쿼리를 둔다.

즉, `add -> backfill -> dual read/write -> cutover -> drop` 같은 expand/contract 방식이 기본이다.

### 5.2. Cloudflare D1 기준으로 볼 점

- D1 migration은 SQL 파일 단위로 버전 관리되고, 적용 이력은 `d1_migrations` 테이블에 남는다.
- migration은 단순히 “남은 파일 적용” 모델에 가깝기 때문에, 잘못 적용한 파괴적 변경을 자동으로 안전하게 되돌려주지 않는다.
- Worker code rollback은 가능하지만, Cloudflare 공식 문서상 rollback은 연결된 리소스 상태를 함께 되돌리지 않는다.
- 즉, 코드만 이전 버전으로 되돌리고 DB는 이미 새 스키마로 깨져 있으면 앱이 계속 죽을 수 있다.

### 5.3. Dowin 권장 운영 방안

#### 예방

- 절대 `drop column`, `drop table`, 대규모 rename을 단일 배포에서 바로 실행하지 않는다.
- migration 전에 필수 체크:
  - 현재 DB export 또는 bookmark 확보
  - 영향받는 API/쿼리 목록 확인
  - rollback 시 code/db 조합이 호환되는지 확인
- 스키마 제거는 2단계 이상으로 나눈다.
  - 1차: 새 컬럼/새 테이블 추가, 코드 dual-write 또는 dual-read
  - 2차: 앱이 새 구조만 사용하도록 전환
  - 3차: 일정 기간 뒤 제거

#### 탐지

- 배포 직후 아래 에러를 집중 확인한다.
  - `no such column`
  - `no such table`
  - `constraint failed`
  - 500 응답 급증
- 신규 배포 후 핵심 smoke test를 바로 돌린다.

#### 즉시 대응

1. 추가 migration 중단
2. 현재 앱 버전 신규 배포 중단
3. 사고가 코드만의 문제인지, 스키마 파괴까지 갔는지 분리한다
4. 필요 시 Worker code rollback을 하되, DB와 호환되는지 먼저 확인한다

#### 복구

- 컬럼/테이블이 실제로 삭제된 경우:
  - 가장 빠른 복구는 보통 “정정 migration” 또는 “Time Travel restore” 중 하나다.
- 아래 기준으로 고른다.
  - 다른 정상 쓰기 데이터가 거의 없다면: restore가 빠를 수 있다.
  - 사고 후 이미 신규 정상 데이터가 쌓였다면: 정정 migration + 수동 데이터 보정이 더 현실적일 수 있다.
- 단순 code rollback만으로 해결되는지 먼저 착각하지 않도록 주의한다.

#### 사후 조치

- 파괴적 migration을 어떤 승인 체계 없이 실행했는지 기록
- expand/contract를 어긴 이유 기록
- “migration 전 snapshot/bookmark 확보” 절차 미준수 여부 기록

### 5.4. Dowin 출시 전 최소 액션

- [ ] destructive schema change 금지 원칙 문서화
- [ ] migration 실행 전 수동 bookmark 확보 절차 추가
- [ ] migration 후 smoke test 체크리스트 작성
- [ ] “코드 rollback과 DB restore는 별도 판단” 원칙 팀 내 고정

## 6. 사고 유형 3: 서버 접속이 안 될 때

### 6.1. 일반 서비스에서의 표준 대응

전통적 운영에서는 “서버 접속 안 됨”을 아래로 쪼갠다.

- DNS 문제
- TLS/도메인 문제
- 앱 프로세스 다운
- DB 연결 실패
- 특정 API만 죽음
- 외부 의존성 장애

즉, 사용자가 “서비스 안 돼요”라고 말해도 내부적으로는 계층별로 분리해서 확인한다.

### 6.2. Dowin 현재 인프라 기준 분해

DOWIN는 전통적인 SSH 접속형 서버가 아니므로, 아래 순서가 현실적이다.

1. 배포 라우팅/도메인 문제인가
2. Worker 자체가 에러를 내는가
3. D1 접근이 실패하는가
4. 특정 API만 실패하는가
5. Cloudflare 플랫폼 전체 이슈인가

### 6.3. Dowin 권장 운영 방안

#### 예방

- 핵심 사용자 여정을 smoke test로 고정한다.
  - 랜딩 또는 로그인 페이지 접근
  - 로그인
  - `GET /api/workspaces/me`
  - 내 대시보드 조회
  - daily log 1건 쓰기
- 배포 직후 수동 확인이 아니라 최소 1개 자동 체크를 둔다.
- `wrangler.jsonc`의 `observability.enabled`는 이미 켜져 있으므로 Workers Logs 기반 모니터링을 운영에 연결한다.

#### 탐지

- Workers Logs에서 5xx 급증, uncaught exception 확인
- 배포 직후 특정 버전 이후부터 오류가 시작됐는지 확인
- 로그인/API/정적 페이지 중 어디까지 죽는지 범위를 좁힌다

#### 즉시 대응

1. 최근 배포가 원인으로 의심되면 Worker rollback 가능 여부 확인
2. rollback 전에 DB 스키마 호환성 확인
3. 장애가 쓰기 경로에만 있으면 임시 read-only 안내를 검토
4. 사용자가 반복 실패하는 핵심 경로를 우선 복구

#### 복구

- 최근 배포 문제라면 Worker rollback을 우선 검토한다.
- Cloudflare 공식 문서 기준으로 `wrangler rollback` 또는 Dashboard rollback이 가능하다.
- 다만 rollback은 Worker에 연결된 리소스를 함께 되돌리지 않으므로 D1 schema/data 상태는 별도 확인해야 한다.

#### 사후 조치

- 어떤 배포 버전부터 실패했는지 기록
- 사전에 잡았어야 할 smoke test가 무엇이었는지 기록
- “배포는 성공했지만 사용자 흐름은 실패”한 유형인지 분류

### 6.4. Dowin 출시 전 최소 액션

- [ ] 배포 직후 5분 내 실행할 smoke checklist 작성
- [ ] 최근 배포 rollback 절차 문서화
- [ ] Worker 버전 확인/rollback 담당자 권한 정리
- [ ] 읽기 가능 / 쓰기 불가 / 전체 불가 3단계 공지 문구 초안 작성

## 7. 사고 유형 4: Cloudflare 장애가 났을 때

### 7.1. 일반 서비스에서의 표준 대응

서드파티 플랫폼 장애는 보통 아래 원칙으로 대응한다.

1. 내 코드 문제인지 공급자 문제인지 빠르게 분리
2. 공급자 장애면 무리한 코드 변경보다 현황 파악과 사용자 공지가 우선
3. 우회 가능한 기능만 제한적으로 제공
4. 복구 후 의존도 축소 또는 대체 경로를 검토

### 7.2. Cloudflare 기준으로 볼 점

- Cloudflare는 `cloudflarestatus.com` 상태 페이지를 제공한다.
- Incident Alert notification을 email, webhook, PagerDuty로 받을 수 있다.
- Notifications 기능은 플랜별 차이가 있으므로 현재 계정 플랜 기준으로 사용 가능한 전달 채널을 먼저 확인해야 한다.
- DOWIN는 Workers와 D1 모두 Cloudflare에 묶여 있으므로, 플랫폼 장애 시 애플리케이션 레벨에서 할 수 있는 일이 제한적일 수 있다.

### 7.3. Dowin 권장 운영 방안

#### 예방

- 최소한 아래는 출시 전에 연결한다.
  - Cloudflare Incident Alert email
  - 내부 운영 채널 연동 가능하면 webhook 또는 PagerDuty
- 장애 시 사용자에게 노출할 상태 공지 채널을 정한다.
  - 별도 상태 페이지
  - 공지용 노션/문서
  - SNS 또는 메일

#### 탐지

- 장애 신고가 들어오면 즉시 아래를 같이 본다.
  - Dowin 내부 로그
  - Cloudflare status
  - 최근 배포 여부
- 내부 로그도 죽고 Cloudflare status에 incident가 뜨면 공급자 장애 가능성이 높다.

#### 즉시 대응

1. 최근 Dowin 배포가 원인인지 먼저 배제
2. Cloudflare status 확인
3. 공급자 장애로 판단되면 불필요한 배포/마이그레이션 중단
4. 사용자 공지 발행
5. 장애 범위가 제한적이면 읽기 전용 공지 또는 일부 기능 제한 공지

#### 복구

- Cloudflare 복구 이후 Dowin 핵심 흐름을 다시 smoke test 한다.
- 장애 중 실행하려다 실패한 migration, export, batch가 있으면 재실행 여부를 따로 판단한다.
- “장애 도중 반쯤 끝난 작업”이 없는지 확인한다.

#### 사후 조치

- DOWIN가 Cloudflare 단일 의존성으로 인해 받을 수밖에 없는 리스크와, 줄일 수 있는 리스크를 분리해서 기록한다.
- 예:
  - 줄이기 어려운 것: Workers/D1 플랫폼 자체 outage
  - 줄일 수 있는 것: 백업 장기 보관 부재, 공지 채널 부재, smoke test 부재

### 7.4. Dowin 출시 전 최소 액션

- [ ] Cloudflare Incident Alert 수신 채널 연결
- [ ] 외부 공지 채널 1개 확보
- [ ] 공급자 장애 시 내부 판단 플로우 작성
- [ ] 장애 복구 후 smoke test 체크리스트 작성

## 9. 성공 및 실패 판단 기준 (Success & Failure Criteria)

- 성공
  - 팀이 “무슨 일이 나면 누구가 무엇을 먼저 하는지”를 안다.
  - 데이터 손실과 스키마 사고를 코드 rollback 문제와 구분해서 다룬다.
  - Cloudflare 장애 시 무의미한 코드 변경을 반복하지 않는다.
  - 최소 1회 restore와 1회 rollback을 실제로 연습했다.
- 실패
  - 백업/복구가 “있을 것”이라고만 믿고 실제로 검증하지 않는다.
  - 파괴적 migration을 여전히 단일 배포에 섞는다.
  - 장애가 나면 로그, 상태 페이지, 최근 배포를 체계적으로 보지 못한다.
