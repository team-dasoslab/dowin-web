# 리더용 주간 운영 리포트 + 자동 운영 체크인 기획 요약

> 문서 역할: 이 문서는 의사결정과 범위 확인을 위한 요약본이다. 상세 배경, 참고 근거, 확장 논의는 `docs/planning/2026.04.14-leader-report-reminder-plan.md`를 기준 문서로 본다.

> 상태 메모 (2026-04-18): 리더 리포트 MVP는 `팀 승패 + 실행 시작 여부 + 개입 필요 대상`만 우선 다룬다. 과한 퍼널, 랭킹, 장문 해석은 범위에서 제외한다.
>
> 추가 메모 (2026-04-22): STANDARD의 대표 가치는 단순 리포트나 CSV export가 아니라 `서비스가 팀원이 멈춘 순간을 감지하고, 운영 체크인을 자동 발송해 실행 재개를 돕는 도구`다.
>
> 추가 메모 (2026-04-24): 유료 가치는 `체크인을 보냈다`에서 끝나지 않는다. 리더는 `왜 누구에게 체크인이 갔는지`, `팀원이 어떻게 반응했는지`, `그 뒤 실제로 다시 움직였는지`까지 보고받아야 한다. 따라서 STANDARD의 핵심 묶음은 `자동 운영 체크인 + 체크인 결과 리포트`다.

## 1. 한눈에 보기

### 1.1. 문제

- 지금의 팀 대시보드는 상태를 보여주지만, 리더에게 `누구를`, `왜`, `어떻게` 먼저 봐야 하는지는 밀어주지 못한다.
- 리더가 매주 직접 팀을 훑고 따로 체크인해야 해서 운영 비용이 남아 있다.
- 자동 알림이 있어도 결과 추적이 없으면 단순 발송 기능처럼 보인다.

### 1.2. 핵심 결정

- `FREE`는 상태 파악까지 제공한다.
- `STANDARD`는 `개입 + 결과 보고 + 후속 확인`까지 제공한다.
- 유료의 핵심은 `긴 조회기간`보다 `팀이 다시 움직이게 만드는 운영 자동화`다.

### 1.3. STANDARD 핵심 루프

1. 위험 신호 감지
2. 자동 운영 체크인 발송
3. 팀원 1탭 반응
4. 체크인 결과 보고
5. 후속 변화 확인

리더가 받아야 하는 것은 단순 발송 로그가 아니라 `개입 결과 보고`다.

## 2. 플랜 역할 구분

### 2.1. FREE

- 리더가 팀 상태를 기본적으로 파악할 수 있다
- 기본 주간 요약 리포트를 받을 수 있다
- 팀 대시보드에서 누가 시작했고 누가 멈췄는지 볼 수 있다
- 이번 주 기본 위험 신호를 확인할 수 있다

### 2.2. STANDARD

- 서비스가 특정 팀원에게 자동 운영 체크인을 발송할 수 있다
- 개입 우선순위를 더 선명하게 받을 수 있다
- 팀원의 반응을 확인할 수 있다
- 자동 체크인 이후 실제 기록 재개 여부를 확인할 수 있다
- 어떤 체크인이 왜 발송됐고 어떤 결과로 이어졌는지 리포트로 볼 수 있다

### 2.3. FREE / STANDARD 기능표

| 기능                        | FREE | STANDARD | 비고                                                 |
| --------------------------- | ---- | -------- | ---------------------------------------------------- |
| 팀 상태 파악                | 가능 | 가능     | 팀 대시보드, 기본 주간 리포트                        |
| 이번 주 기본 위험 신호 확인 | 가능 | 가능     | FREE는 기본 신호, STANDARD는 더 선명한 우선순위 제공 |
| 주간 운영 리포트 수신       | 가능 | 가능     | FREE도 기본 리포트는 열어둔다                        |
| 자동 운영 체크인 발송       | 불가 | 가능     | 핵심 유료 기능                                       |
| 팀원 1탭 반응 수집          | 불가 | 가능     | `LOG_NOW`, `LATER_TODAY`                             |
| 체크인 결과 보고            | 불가 | 가능     | 발송 이유, 반응, 후속 변화 포함                      |
| 기록 재개/회복 추적         | 불가 | 가능     | 24시간/7일 회복 확인                                 |
| 리더 개입 이력 누적 조회    | 불가 | 가능     | 운영 도구로서의 유료 가치                            |

판단 원칙:

- `FREE`는 본다.
- `STANDARD`는 움직인다.
- 유료 차이는 `리포트 조회 가능 여부`보다 `개입 + 결과 보고 + 후속 확인`에서 가장 선명해야 한다.

## 3. 제품 원칙

- 이 기능의 1차 사용자는 팀 리더다.
- 리포트와 자동 체크인은 `평가`보다 `운영`, `개입`, `코칭`을 위한 기능이어야 한다.
- 팀원 입장에서는 감시가 아니라 `지금 중요한 선행지표를 다시 보게 하는 체크인`으로 느껴져야 한다.
- 리더를 완전히 대체하지는 않지만, 리더가 매건 직접 발송하지 않도록 해야 한다.
- 개입 문구는 사람의 성향이나 태도를 추정하지 않고, 관찰 가능한 실행 데이터와 다음 행동에만 근거해야 한다.
- 자동 체크인은 많이 보내는 것보다 적고 분명하게 보내는 편이 낫다.
- 결과 리포트는 사람 평가표가 아니라 운영 상태판이어야 한다.

## 4. MVP 범위

### 4.1. 기능 A: 리더용 주간 운영 리포트

목표:

- 리더가 앱에 들어오기 전부터 이번 주 어디를 봐야 하는지 알게 한다.

MVP 포함:

- 워크스페이스 단위 설정 1개
- 주 1회 발송
- 발행 요일 1개 선택
- 웹 푸시를 구독한 ADMIN 대상 발송
- 팀 요약, 실행 시작 여부, 개입 필요 대상 요약
- 팀 대시보드 딥링크

리포트 핵심 질문:

1. 팀 전체가 지금 이기고 있는가
2. 실행이 이번 주 실제로 시작됐는가
3. 지금 바로 확인할 대상이 누구인가

리포트 핵심 블록:

- 상단 한 줄 요약
- 핵심 숫자 3개
  - `이기는 중`
  - `실행 시작`
  - `즉시 확인 필요`
- 개입 대상 목록 최대 3명
- 팀 대시보드 보기 CTA

MVP 제외:

- 장문 해석
- 팀원 랭킹
- 사람 평가용 비교표
- 과한 퍼널 지표
- AI 자동 해석

### 4.2. 기능 B: 자동 운영 체크인

목표:

- 리더가 매건 직접 챙기지 않아도 팀이 다시 움직이게 만드는 기본 운영 루프를 서비스가 맡는다.

MVP 포함:

- `STANDARD` 플랜 전용
- 워크스페이스 ADMIN이 정책 설정
- 서비스가 조건 기반으로 자동 발송
- 발송 대상은 팀원 1명 + 선행지표 1개
- 트리거 2개
  - `이번 주 미기록`
  - `점수판 없음`
- 템플릿 1종
  - `CHECK_IN`
- 팀원 반응 2종
  - `지금 기록하기`
  - `오늘 나중에 할게요`
- 웹 푸시 우선

MVP 제외:

- 대량 일괄 발송
- 리더 매건 승인 발송
- AI 자동 꾸짖기
- 공개 피드백
- 무응답자 자동 escalation
- 다중 채널 발송

### 4.3. 기능 C: 체크인 결과 리포트

목표:

- 자동 체크인이 실제로 팀을 움직였는지 리더가 판단할 수 있게 한다.

리더가 확인해야 하는 최소 질문:

1. 어떤 이유로 누구에게 체크인이 발송됐는가
2. 팀원은 그 체크인에 어떻게 반응했는가
3. 그 뒤 실제 기록 재개나 회복이 있었는가

MVP 카드/지표:

- `이번 주 자동 체크인`
  - 총 발송 수, 발송 대상 수
- `반응`
  - 즉시 기록, 나중에 하겠다고 응답, 무응답 수
- `회복`
  - 24시간 내 기록 재개 수
  - 7일 내 기록 재개 수
- `직접 확인 필요`
  - 자동 체크인 이후에도 여전히 멈춘 팀원 수

MVP 대상 목록:

- 이름
- 발송 이유
- 마지막 반응
- 현재 후속 변화 상태
- 리더 직접 확인 필요 라벨

## 5. 팀원 경험 원칙

- 팀원을 평가하는 점수, 순위, 적색 경고를 보여주지 않는다.
- `리더가 확인하고 있습니다` 같은 감시성 문구를 쓰지 않는다.
- 메시지의 첫 목적은 반성이 아니라 다음 행동이다.
- 팀원이 반응하면 같은 날 동일 체크인을 재발송하지 않는다.
- `막힘`이나 `나중에 하겠다`는 실패가 아니라 운영 신호로 다룬다.

문구 방향 예시:

- `이번 주 {선행지표명} 기록이 아직 비어 있어요. 오늘 안에 한 번만 업데이트해 주세요.`
- `{선행지표명}이 아직 시작되지 않았습니다. 이번 주 가능한 가장 작은 실행부터 확인 부탁드려요.`

금지:

- `의지가 부족해 보여요`
- `관심이 없어 보입니다`
- `왜 아직 안 했나요`

## 6. 데이터 및 API 초안

상세 엔지니어링 설계는 구현 단계에서 `docs/dev/`로 넘기고, 이 문서에는 MVP 판단에 필요한 수준만 남긴다.

### 6.1. 워크스페이스 설정

권장 저장 단위:

- `workspace_notification_settings`

핵심 필드:

- `workspace_id`
- `leader_weekly_report_enabled`
- `leader_weekly_report_day_of_week`
- `leader_weekly_report_hour`
- `leader_weekly_report_minute`
- `leader_checkin_enabled`
- `leader_checkin_mode`
- `leader_checkin_timezone`
- `updated_by_user_id`

핵심 규칙:

- 워크스페이스당 1 row
- ADMIN만 조회/수정 가능
- FREE는 주간 리포트 설정 가능
- `leader_checkin_enabled`는 STANDARD에서만 활성화 가능

예상 계약:

- `GET /api/push/workspace-settings`
- `PUT /api/push/workspace-settings`

### 6.2. 자동 운영 체크인 이력

권장 저장 단위:

- `leader_action_deliveries`
- `leader_action_delivery_responses`

핵심 필드:

- `workspace_id`
- `target_user_id`
- `scoreboard_id`
- `lead_measure_id`
- `action_type`
- `reason_code`
- `template_key`
- `delivery_status`
- `sent_at`
- `followup_status`

허용 enum 초안:

- `action_type`
  - `CHECK_IN`
- `reason_code`
  - `WEEKLY_NOT_STARTED`
  - `NO_SCOREBOARD`
- `response_type`
  - `LOG_NOW`
  - `LATER_TODAY`

예상 계약:

- `POST /api/push/leader-actions`
- `GET /api/push/leader-actions`
- `POST /api/push/leader-actions/{id}/responses`

### 6.3. 내부 발송

- 주간 리포트와 자동 체크인 모두 내부 크론 실행 경로가 필요하다.
- 네임스페이스는 우선 기존 `/api/push/*`를 유지한다.

예상 계약:

- `GET /api/push/send-leader-weekly-report`
- `GET /api/push/send-leader-check-ins`

## 7. 측정 지표

핵심 지표:

- 주간 리포트 설정률
- 주간 리포트 발송 성공률
- 리포트 딥링크 클릭률
- 자동 운영 체크인 발송률
- 자동 운영 체크인 수신자 응답률
- 자동 운영 체크인 수신 후 24시간 내 기록 재개율
- 자동 운영 체크인 수신 후 7일 내 기록 재개율
- 직접 확인 필요 대상 비율

팀원 수용성 지표:

- 체크인 수신 후 `LOG_NOW` 비율
- 체크인 수신 후 `LATER_TODAY` 비율
- 체크인 수신 후 푸시 opt-out 증가 여부
- 동일 팀원 반복 발송 비율

성공 판단 원칙:

- 발송 수만으로 성공 판단하지 않는다.
- 발송은 많지만 응답률이 낮거나 opt-out이 증가하면 실패에 가깝다.

## 8. 리스크 및 가드레일

### 8.1. 감시 도구처럼 보일 위험

- 리포트 문구는 평가/통제 톤을 피한다.
- 팀원 랭킹이나 공개 비교는 제외한다.
- 자동 체크인은 공개가 아니라 개인 푸시로 보낸다.

### 8.2. 알림 피로

- 주간 리포트는 주 1회만 허용한다.
- 자동 운영 체크인은 동일 대상/지표 기준 빈도 제한을 둔다.
- `오늘 나중에 할게요` 응답 후 같은 날 재발송하지 않는다.

### 8.3. 잘못된 개입 유도

- 사람 자체가 아니라 특정 선행지표와 이번 주 실행 상태에 초점을 맞춘다.
- 자동 운영 체크인은 서비스가 발송하되, 시스템이 자동으로 꾸짖거나 압박하지 않는다.

## 9. 권장 실행 순서

1. 리더 주간 운영 리포트 설정/발송 MVP
2. 자동 운영 체크인 정책/발송 MVP
3. 팀원 1탭 반응 UX
4. 체크인 결과 리포트 MVP
5. 측정 지표 수집
6. 필요 시 멀티채널 fallback 검토

### 9.1. 권장 배치 전략

- 1차 배치:
  - FREE/STANDARD 공통 기반으로 `리더 주간 운영 리포트`를 먼저 안정화한다
  - 이유:
    - 리더가 상태를 파악하는 기본 루프가 먼저 있어야 이후 개입 결과도 설명된다
- 2차 배치:
  - `STANDARD` 전용 자동 운영 체크인 발송과 팀원 1탭 반응을 붙인다
  - 이유:
    - 여기서부터 실제 결제 이유가 생긴다
- 3차 배치:
  - 체크인 결과 리포트와 후속 변화 확인을 붙여 유료 루프를 닫는다
  - 이유:
    - 발송 기능만으로는 운영 도구가 아니라 단순 알림 기능처럼 보이기 쉽다

### 9.2. 구현 우선순위 세분화

1. `FREE/STANDARD 공통 기반`
   - 워크스페이스 리포트 설정
   - 리포트 집계 데이터 스펙
   - 리포트 발송 크론
2. `STANDARD 개입 엔진`
   - 자동 체크인 정책 설정
   - 발송 대상 선정 규칙
   - 발송 이력 저장
3. `팀원 반응 루프`
   - 푸시 딥링크
   - `LOG_NOW`, `LATER_TODAY` 응답 저장
   - suppress 규칙
4. `결과 보고 루프`
   - 반응/무응답 집계
   - 24시간/7일 기록 재개 계산
   - 직접 확인 필요 대상 노출

## 11. MVP / Post-MVP 경계

### 11.1. MVP

- 주간 리포트 주 1회
- 워크스페이스당 리포트 설정 1개
- 발행 요일 선택 1개
- 팀원 1명 + 선행지표 1개 기준 자동 체크인
- 템플릿 2종
- 팀원 반응 2종
- 최소 발송/응답/후속 상태 저장
- 체크인 결과 리포트 기본 카드
- PWA 웹 푸시 우선

### 11.2. Post-MVP

- 다중 채널 발송
- 다중 요일 발송
- 세분화된 자동 추천 액션
- 반응 옵션 확장
- 인앱 inbox fallback
- AI 기반 문구 추천

## 12. 성공 기준

성공:

- 리더가 주간 리포트만 보고도 이번 주 우선 확인 대상을 빠르게 정할 수 있다.
- 서비스가 자동 운영 체크인을 발송할 수 있다.
- 팀원이 1탭 반응하거나 실제 기록을 재개한다.
- 리더가 `왜 발송됐는지`, `어떻게 반응했는지`, `실제로 회복됐는지`를 확인할 수 있다.

실패:

- 리포트가 다시 길고 추상적으로 흘러 실제 개입으로 이어지지 않는다.
- 자동 체크인이 감시성 알림처럼 느껴진다.
- 발송은 많지만 응답과 기록 재개가 낮다.
- 체크인 결과 리포트가 단순 발송 로그 수준에 머문다.
