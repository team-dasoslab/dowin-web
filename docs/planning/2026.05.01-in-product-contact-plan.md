# 서비스 내 문의하기 전환안

## 1. 배경 및 맥락 (Background & Context)

- 현재 `/contact`는 서비스 내부 진입점은 있지만, 실제 제출은 `Tally` 외부 폼으로 이동한다.
- 이 구조는 빠른 임시 운영에는 유리했지만, 아래 문제가 남아 있다.
  - 사용자가 서비스 밖으로 이탈한다.
  - `Tally` 기준 개인정보/국외이전 동의 문구를 별도 관리해야 한다.
  - 문의 접수와 운영 알림이 제품 내부 상태와 분리되어 있다.
  - 결제/환불/워크스페이스 이슈 같은 운영 문의를 사용자 세션과 연결해 보기 어렵다.
- 이미 DOWIN에는 Discord 웹훅 발송 포맷과 운영 알림 경험이 일부 존재하므로, 문의 접수도 같은 방향의 운영 채널로 묶는 것이 맞다.

## 2. 문제 정의 (Problem Statement)

- 문의하기가 제품 기능이 아니라 외부 위임 링크처럼 동작해 신뢰감과 연속성이 약하다.
- 운영자는 새 문의를 실시간으로 확인하기 어렵고, 별도 외부 도구를 열어야 한다.
- 현재 동의 문구와 법률 문서 일부가 `Tally` 전제를 갖고 있어, 장기적으로는 실제 운영 구조와 문서가 어긋난다.

## 3. 왜 지금 해야 하는가 (Priority & Rationale)

- 결제/환불 정책, billing 화면, 프로필의 문의 진입점이 이미 제품 안에 있다.
- 앞으로 `STANDARD` 결제 운영과 수동 검토가 늘면, 문의 접수와 응답 맥락을 서비스 사용자/워크스페이스 기준으로 연결할 필요가 커진다.
- 문의 접수는 별도 고급 운영도구보다 먼저 닫아야 하는 기본 운영 기능이다.

## 4. 목표와 비목표 (Goals / Non-goals)

### 4.1. MVP 목표

- 서비스 내부에서 문의를 작성하고 제출할 수 있다.
- 접수 직후 백엔드가 Discord 웹훅으로 운영 채널에 새 문의 알림을 보낸다.
- 로그인 사용자의 경우 문의에 `userId`, `workspaceId`, email 같은 운영용 식별 맥락을 함께 남길 수 있다.
- 문의하기는 로그인 사용자만 사용할 수 있다.
- 일반 문의와 결제/환불 문의를 같은 접수 경로로 받되, 유형은 명시적으로 구분한다.
- `Tally` 의존 문구와 UX를 제거하고, 자체 수집/보관 기준에 맞는 안내로 바꾼다.

### 4.2. MVP 비목표

- 운영자용 문의 관리 대시보드
- 첨부파일 업로드
- 사용자와 운영자 간 스레드형 답변 UI
- Discord에서 바로 상태 변경/답변 처리하는 양방향 봇
- 자동 분류, 자동 응답, AI 요약

## 5. MVP 범위 제안 (Recommendation)

### 5.1. 사용자 경험

- `/contact`는 더 이상 외부 폼 동의 게이트가 아니라 실제 문의 작성 페이지가 된다.
- 필수 입력은 아래 정도로 제한한다.
  - 문의 유형
    - `GENERAL`
    - `BILLING`
    - `BUG_OR_ACCOUNT`
  - 답변 받을 이메일
  - 제목
  - 내용
  - 개인정보 수집·이용 동의
- 로그인 사용자는 프로필 진입점과 billing 화면 진입점을 그대로 재사용한다.
- 비로그인 사용자는 `/contact` 진입 시 로그인으로 유도하거나 접근 자체를 막는다.
- 로그인 사용자는 기본 이메일과 사용자 맥락을 자동 채운다.

### 5.2. 운영 흐름

1. 사용자가 문의를 제출한다.
2. API는 입력값과 동의 여부를 검증한다.
3. DB에 문의 원문과 운영 메타데이터를 저장한다.
4. 저장 성공 후 Discord 웹훅으로 새 문의 요약 메시지를 발송한다.
5. 사용자는 접수 완료 화면 또는 토스트를 본다.

중요:

- Discord 발송 실패가 문의 자체를 유실시키면 안 된다.
- 따라서 진실 원천은 Discord가 아니라 DOWIN DB여야 한다.
- MVP에서도 `DB 저장 성공 -> Discord 발송 시도` 순서를 유지하는 편이 맞다.

## 6. 데이터 및 상태 모델 (Data Model)

### 6.1. `contact_inquiries` 신규 테이블 제안

- `id`
- `category`
  - `GENERAL | BILLING | BUG_OR_ACCOUNT`
- `status`
  - MVP 기본값 `RECEIVED`
  - Post-MVP 확장: `IN_PROGRESS | RESOLVED | CLOSED`
- `subject`
- `message`
- `replyEmail`
- `consentedAt`
- `locale`
- `source`
  - `CONTACT_PAGE`
  - 추후 `BILLING_PAGE`, `RISK_REVIEW_BANNER` 같은 세부 유입값 확장 가능
- `userId`
  - not null
- `workspaceId`
  - nullable, 현재 세션에 활성 워크스페이스가 없을 수 있음
- `discordDeliveryStatus`
  - MVP 기본값 `PENDING`
  - 발송 성공 시 `SENT`
  - 실패 시 `FAILED`
- `discordDeliveredAt`
- `createdAt`
- `updatedAt`

### 6.2. 왜 저장이 먼저여야 하는가

- Discord 웹훅은 운영 채널일 뿐 영속 저장소가 아니다.
- 향후 문의 내역 조회, 분쟁 대응, 보관기간 적용을 생각하면 앱 DB 저장이 먼저여야 한다.
- 운영자 대시보드가 아직 없어도, 최소한 백엔드에서 재발송/재검토 가능한 상태는 남겨야 한다.

## 7. API 및 백엔드 경계 (Backend Scope)

### 7.1. API 계약

- `POST /api/contact-inquiries`
  - 인증 세션 필수 API로 둔다.
  - body 예시:
    - `category`
    - `replyEmail`
    - `subject`
    - `message`
    - `privacyConsent`

### 7.2. 처리 원칙

- Zod로 입력 검증
- 세션이 없으면 즉시 인증 오류를 반환한다
- 사용자/워크스페이스 맥락을 세션 기준으로 자동 연결한다
- API 응답은 저장소 표준인 `apiSuccess`, `apiError`, `withErrorHandler` 패턴 사용
- Discord 발송은 route handler 안에 직접 쓰기보다 서비스 계층으로 분리
- Discord 실패는 API 전체 실패로 바로 번역하지 않고, 문의는 접수 성공 처리하되 운영 재처리 가능 상태를 남긴다

### 7.3. Discord 웹훅 payload 원칙

- 운영자가 바로 triage 가능한 요약만 보낸다.
- 권장 포함값:
  - 문의 ID
  - 문의 유형
  - 제목
  - reply email
  - userId / workspaceId
  - 제출 시각
  - 본문 요약 또는 일부 본문
- Discord에는 민감정보 전체 원문을 길게 그대로 보내지 않는다.
- 길이 제한과 마스킹 규칙을 서비스 레벨에서 둔다.

## 8. 프론트엔드 범위 (Frontend Scope)

### 8.1. 현재 페이지 전환 방향

- 기존 `ContactConsentPage`는 `Tally` 외부 동의 화면이라 이름과 책임이 바뀌어야 한다.
- 새 페이지는 아래 상태를 가져야 한다.
  - 기본 입력 상태
  - 제출 중 상태
  - 성공 상태
  - 실패 상태

### 8.2. 카피 전환

- `외부 문의 폼 안내`, `Continue to Tally Contact Form` 같은 문구는 제거
- 자체 접수 기준 카피로 교체
- billing 화면, profile 화면, footer의 `/contact` 링크는 유지

## 9. 법률/정책 영향 (Legal & Policy Impact)

- `Tally` 기반 국외이전 동의는 자체 문의 저장 구조로 바뀌면 기본 문의 흐름에서 제거 가능성이 높다.
- 대신 자체 문의 접수에 맞는 개인정보 수집·이용 안내는 유지해야 한다.
- 결제/환불 정책과 개인정보처리방침의 `in-product contact path` 표현은 유지 가능하지만, 구체 문구는 실제 저장/보관 방식에 맞게 다시 맞춰야 한다.
- 보관기간은 현재 문구를 기준으로 아래 원칙을 유지하는 쪽이 자연스럽다.
  - 일반 문의: 처리 완료 후 12개월
  - 결제/환불/계약 해지/분쟁 관련 문의: 관련 법령 기준 최대 3년

중요:

- 법률 문구는 구현 전에 최종 운영 방침과 일치해야 한다.
- 특히 첨부파일 미지원이면 관련 문구에서도 첨부파일을 기본 수집항목처럼 쓰지 않는 편이 맞다.

## 10. 보안 및 악용 방지 (Security & Abuse)

- 로그인 사용자만 허용하더라도 반복 제출, 너무 긴 본문, 링크 도배 같은 기본 abuse 가드는 필요하다.
- Discord 웹훅 URL은 앱 코드에 두지 않고 secret/env로만 관리한다.
- 문의 본문은 운영 목적 외 다른 경로로 재노출하지 않는다.
- 로그에는 본문 전체를 남기지 않는 것이 안전하다.

## 11. MVP 이후 (Post-MVP)

- 운영자 문의 리스트/상태 변경 화면
- 문의 상세 조회와 Discord 발송 재시도
- billing 문의의 관련 주문/워크스페이스 자동 연결
- 첨부파일 업로드
- 신고/버그/환불 등 유형별 추가 입력 필드
- 관리자 메모와 처리 이력
- 자동 분류 태그

## 12. 구현 순서 제안 (Action Items)

1. 기획/정책
   - `/contact`를 자체 제출형으로 전환하는 카피와 동의 문구 확정
   - 보관기간, 수집항목, 로그인 필수 정책 최종 확정
2. API 계약
   - `src/api-spec/openapi.yaml`에 `POST /api/contact-inquiries` 추가
   - 요청/응답 스키마와 오류 케이스 정의
3. 백엔드
   - D1 schema / migration 추가
   - route handler, validation, service, Discord notifier 구현
   - Discord 실패 저장 규칙과 테스트 추가
4. 프론트엔드
   - `/contact` 페이지를 실제 입력 폼으로 교체
   - 로그인 사용자 기본값 주입, 성공/실패 상태 처리
5. 문서/정책
   - 개인정보처리방침, 결제/환불 정책, 메시지 카피 정렬
   - `docs/onboarding.md`와 관련 도메인 문서 상태 갱신

## 13. 성공 및 실패 기준 (Success / Failure Criteria)

### 성공

- 사용자가 서비스 내부에서 문의를 제출하고 이탈 없이 접수를 완료한다.
- 운영자가 Discord에서 새 문의를 1분 내 확인할 수 있다.
- 문의 원문과 운영 식별 맥락이 DB에 남아 후속 대응이 가능하다.

### 실패

- Discord 발송 실패 시 문의가 유실된다.
- 법률 문구는 자체 저장이라고 쓰지만 실제로는 외부 폼/Tally 의존이 남아 있다.
- billing/support 문의가 들어와도 어떤 사용자/워크스페이스 이슈인지 역추적이 어렵다.

## 14. 결론

- 이 작업은 단순 UI 교체가 아니라 `운영 접점의 내부화`다.
- MVP에서는 "문의 저장 + Discord 운영 알림 + 최소 정책 정렬"까지만 닫고, 운영자 대시보드는 후속으로 미루는 것이 적절하다.
- 다음 구현 단계는 `dowin-backend`로 넘겨 API 계약과 저장 모델부터 확정하는 순서가 가장 안전하다.
