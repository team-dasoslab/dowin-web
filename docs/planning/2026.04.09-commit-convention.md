# DOWIN 커밋 컨벤션 정리

## 1. 배경 및 맥락 (Background & Context)

- 현재 DOWIN 저장소에는 별도의 공통 커밋 메시지 규칙 문서가 없다.
- 다만 실제 이력에는 반복되는 패턴이 충분히 쌓여 있어, 관행을 문서화할 수 있는 상태다.
- 이 문서의 목적은 새로운 규칙을 과도하게 도입하는 것이 아니라, 이미 사용 중인 패턴을 기준으로 리뷰와 추적이 쉬운 표준을 정리하는 것이다.

## 2. 현재 이력에서 확인된 패턴 (Observed History)

최근 커밋 이력 기준으로 가장 많이 쓰인 타입은 아래와 같다.

- `fix`
- `feature`
- `docs`
- `chore`
- `style`
- `refactor`
- `feat`

이력상 특징은 다음과 같다.

- 타입 접두어를 붙이는 형식은 이미 사실상 정착되어 있다.
- 메시지 본문은 대부분 짧은 한국어 설명이다.
- `feature`와 `feat`가 혼용되고 있다.
- 일부 커밋은 PR 번호를 제목 끝에 함께 남긴다.
- 백엔드 스킬에는 멀티 커밋 시 `API spec -> tests -> implementation -> docs` 순서 가이드가 이미 있다.

## 3. 권장 표준 (Recommended Convention)

### 3.1. 기본 형식

권장 기본 형식은 아래로 통일한다.

```text
<type>: <변경 요약>
```

예:

```text
feat: 프로필 탈퇴 전용 화면 추가
fix: 비밀번호 변경 페이지 분리
docs: onboarding 문서 정리
```

### 3.2. 타입 규칙

앞으로는 아래 타입만 우선 사용한다.

- `feat`
  - 사용자 가치가 생기는 기능 추가나 확장
- `fix`
  - 버그 수정, 회귀 수정, 잘못된 동작 보정
- `docs`
  - README, onboarding, planning, dev docs, skill 문서 변경
- `chore`
  - 설정, 스크립트, 빌드, CI, 생성물, 리포지토리 관리 작업
- `refactor`
  - 동작 변화 없이 구조를 개선하는 코드 재구성
- `style`
  - 시각 스타일 조정이나 포맷 성격의 UI 변경

### 3.3. `feature` 대신 `feat` 사용

- 실제 이력에는 `feature:`가 많이 남아 있지만, 앞으로의 표준은 `feat:`로 통일하는 것이 낫다.
- 따라서 새 커밋에서는 `feature:`를 추가로 늘리지 않는 것을 권장한다.

## 4. 제목 작성 규칙 (Title Rules)

- 제목은 한 줄로 끝나는 짧은 요약을 쓴다.
- 기본 언어는 현재 이력과 동일하게 한국어를 우선 사용한다.
- 구현 수단보다 사용자 영향이나 변경 결과를 먼저 적는다.
- 가능한 한 한 커밋에는 한 가지 의도만 담는다.
- 마침표는 붙이지 않는다.

좋은 예:

- `feat: 최근 4주 달성률 차트 추가`
- `fix: 팀 대시보드 정렬 기준을 나로 통일`
- `docs: searchParams suspense 규칙 추가`

피하는 예:

- `feat: 이것저것 수정`
- `fix: QA 및 추가 기능 개발`
- `chore: update`

## 5. PR 번호 표기 규칙 (PR Number Handling)

- 이력에는 `(#27)` 같은 PR 번호가 붙은 커밋과 붙지 않은 커밋이 혼재한다.
- 로컬에서 직접 만드는 커밋은 PR 번호 없이 `type: summary` 형식을 우선한다.
- PR 번호는 플랫폼이 자동으로 붙이는 경우만 허용하고, 수동 규칙으로 채택하지 않는다.

## 6. 커밋 분할 원칙 (Commit Splitting)

- 한 커밋에는 하나의 명확한 의도를 담는다.
- 문서만 바뀌면 `docs`
- 생성물만 갱신되면 `chore`
- 구조 개선만 있으면 `refactor`
- 사용자 동작 변화가 있으면 `feat` 또는 `fix`

백엔드처럼 변경이 여러 단계로 쪼개질 때는 기존 스킬 규칙을 그대로 따른다.

1. API spec
2. tests
3. implementation
4. docs

프론트엔드 변경을 여러 커밋으로 나눌 때는 아래 순서를 권장한다.

1. `gen:api`
2. UI
3. API 연동
4. docs

여기서 `gen:api`는 API 계약 변경이 실제로 있을 때만 해당한다. 계약 변경이 없다면 UI 또는 API 연동부터 시작하면 된다.

## 7. 적용 기준 (How To Apply)

- 기존 `feature:` 이력은 과거 관행으로 둔다.
- 새 커밋부터 `feat:`를 표준으로 사용한다.
- 의미가 섞인 커밋은 가능하면 `feat`와 `fix`, 또는 구현과 문서로 분리한다.
- 자동 생성물만 갱신한 경우에는 `chore`로 둔다.

## 8. 권장 예시 (Examples)

- `feat: 워크스페이스 초대코드 참가 화면 추가`
- `fix: 일일 기록 낙관적 업데이트 롤백 보정`
- `docs: 프론트엔드 스킬에 공통 UI 재사용 규칙 명시`
- `chore: CI에 타입 체크 단계 추가`
- `refactor: 대시보드 로직을 domain hook으로 분리`
- `style: 모바일 점수판 카드 간격 조정`

## 9. 결론

- DOWIN는 이미 `type: summary` 형태의 커밋 문화를 갖고 있다.
- 다만 `feature`와 `feat` 혼용, 넓은 범위의 모호한 제목, PR 번호 표기 혼재는 정리할 필요가 있다.
- 앞으로의 공통 표준은 `feat|fix|docs|chore|refactor|style` 중심의 짧은 한국어 제목으로 두는 것이 현재 저장소 이력과 가장 잘 맞는다.
