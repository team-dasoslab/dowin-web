# Basic seat 결제 구매 기록 보존 리서치

## 1. 결론

Dowin는 결제수단 원문을 저장하면 안 된다. 저장해야 하는 것은 카드 정보가 아니라 `누가, 어떤 워크스페이스에, 몇 seat를, 어떤 조건으로, 얼마에 구매했고, 어떤 결제/환불/해지/세금 처리가 있었는지`를 입증하는 구매 기록이다.

정책 기본값:

- Basic seat 결제/구독/환불/seat entitlement/세금 증빙 아카이브는 `10년` 보존을 기본값으로 둔다.
- 한국만 먼저 판매해도 EU/영국 OSS 또는 해외 세무 확장을 고려해 10년 기준으로 설계한다.
- 계정 삭제 또는 개인정보 삭제 요청이 있어도 법정 보존 대상 구매기록은 운영 데이터와 분리 보관한다.
- 보존기간 만료 후 개인 식별자는 삭제 또는 익명화한다.
- CVC/CVV, 카드 전체번호, track data, PIN, 사용자가 입력한 결제수단 원문은 Dowin DB, 로그, analytics, error tracker에 저장하지 않는다.

## 3. Dowin에 저장해야 하는 구매 기록

`billing ledger`와 별도 `billing retention archive` 또는 동등한 보존 테이블에는 최소 아래 정보를 남긴다.

| 범주        | 필수 항목                                                                                                                                                                           |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 구매 주체   | `workspaceId`, 결제 관리자 `userId`, billing email, billing name/company, billing address, country, tax ID/VAT number가 있으면 그 값                                                |
| 계약/구독   | plan code는 신규 기준 `BASIC`, product code, billing interval, seat count, unit price, currency, subscription start/end, current period start/end, trial 여부, terms/policy version |
| 결제        | Polar customer ID, subscription ID, checkout ID, order/invoice/payment ID, payment status, amount, tax amount, discount, paid_at, failed_at                                         |
| 세금/지역   | tax jurisdiction, VAT/GST/sales tax rate, reverse charge 여부, 면세/사업자 검증 증빙, invoice/receipt/credit note 식별자                                                            |
| 변경 이력   | seat 증감, proration, credit note, refund, chargeback/dispute, revoke, risk review 결과                                                                                             |
| 동의/고지   | checkout 고지 버전, 자동갱신 동의 timestamp, IP country evidence, user agent, cancellation request timestamp                                                                        |
| 원본 스냅샷 | Polar webhook event ID, event type, sanitized payload snapshot, invoice/receipt PDF URL 또는 보관 키                                                                                |

주의:

- `sanitized payload snapshot`은 원본 webhook을 그대로 저장한다는 뜻이 아니다.
- 저장 전 카드 원문, 민감 인증값, 불필요한 주소/전화번호/메타데이터를 제거하는 정규화 계층을 둔다.
- Polar가 이미 invoice/receipt를 보관하더라도 Dowin는 감사 대응을 위해 최소 식별자와 주요 금액/세율 snapshot을 자체 보존한다.

## 4. 저장하면 안 되는 결제 데이터

Dowin 애플리케이션, DB, 로그, 에러 리포트, analytics에는 아래 값을 저장하지 않는다.

- CVC/CVV/CID/CAV2
- 카드 전체번호(PAN)
- 카드 track data, PIN, PIN block
- 사용자가 직접 입력한 결제수단 원문
- 결제 화면 iframe/postMessage 등에서 전달될 수 있는 PSP 내부 민감값

허용 가능한 최소 표시 정보:

- payment method provider ID/token
- card brand
- last4
- expiry month/year
- wallet type 또는 funding type이 제공되는 경우 그 분류값

## 5. 삭제/탈퇴 처리 규칙

계정 또는 워크스페이스 삭제 시:

- 운영 데이터 삭제/익명화와 구매 기록 보존을 분리한다.
- 제품 사용 데이터는 기존 삭제 정책에 따라 처리한다.
- 결제/세무 아카이브는 `legal_retention_until` 전까지 삭제하지 않는다.
- 보존 아카이브에는 화면 표시용 닉네임 같은 비필수 개인정보를 남기지 않는다.
- 보존 목적상 필요한 billing name, email, address, tax ID는 암호화 또는 강한 접근통제 대상이다.
- 개인정보처리방침에는 보존 항목, 근거, 기간, 분리보관 원칙을 항목별로 고지한다.

## 6. 구현 액션

MVP:

1. `billing_retention_records` 또는 기존 billing ledger 확장안 결정
2. Polar webhook 처리 시 저장할 normalized payment snapshot 스키마 정의
3. checkout 생성 시 terms/policy version, seat count, country/tax evidence를 metadata 또는 내부 pending record에 저장
4. 계정 삭제/워크스페이스 삭제 플로우에서 billing retention archive를 분리 보존하도록 정책 반영
5. `src/content/legal-documents.ts` 개인정보처리방침에 항목별 보유기간과 결제/세무 보존 근거 반영
6. 결제 화면에 가격, 주기, 자동갱신, 다음 결제일 또는 계산 기준, 해지 방법, 환불/청약철회 조건, 즉시 제공 여부를 명시
7. 로그/에러 리포트에서 카드 원문 및 CVC/CVV가 저장되지 않는지 보안 점검

Post-MVP:

1. EU/영국 판매 전 OSS/VAT 처리 방식 확정
2. 미국/캐나다 판매 전 tax registration, nexus, GST/HST 판단을 외부 세무 도구 또는 전문가 검토로 확정
3. billing archive export tool 추가
4. legal hold flag와 retention 만료 삭제 job 추가
5. 국가별 개인정보처리방침/환불정책 문구 분기 검토

## 7. 남은 법무/세무 확인 질문

- 한국에서 Basic seat 구독을 디지털콘텐츠/용역 중 어떤 성격으로 고지할지
- 통신판매업 신고, 사업자 정보, 결제 화면 필수 표시 항목의 최종 문안
- 미성년자 가입/결제 허용 여부
- EU/영국 판매를 Polar merchant of record 구조로 처리할 수 있는지, 아니면 Dowin 직접 VAT/OSS 의무가 생기는지
- 미국/캐나다 판매에서 Polar가 merchant of record로 세금 신고 책임을 어디까지 부담하는지
- B2B 고객의 세금계산서/인보이스 요청 대응 범위
