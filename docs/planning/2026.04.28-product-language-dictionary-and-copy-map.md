# 제품 용어 사전 + 화면별 카피 적용표

## 1. 이 문서의 역할

- 이 문서 하나로 `왜 바꾸는지`, `어디까지 바꾸는지`, `무슨 용어를 쓸지`, `화면별로 어떻게 적용할지`를 함께 본다.
- 목표는 방향 문서와 실행 문서를 나누지 않고, 실제 카피 작업의 단일 기준점을 만드는 것이다.

## 1.1. 현재 반영 상태 (2026-04-28)

- 핵심 용어는 `핵심 목표 / 성공 기준 / 액션 아이템 / 점수판`으로 확정했다.
- `src/messages/ko.json`, `src/messages/en.json` 기준으로 Setup, 대시보드, 리포트, 랜딩 핵심 카피 1차 반영을 완료했다.
- 다만 법률 문서, 업데이트 히스토리, 일부 에러/보조 카피까지 전면 정리된 상태는 아니다.
- 랜딩은 용어 반영과 Hero variant 비교 프리뷰까지 들어갔지만, 최종 Hero 선택과 본문 카피 재작성은 별도 작업으로 남아 있다.

## 2. 배경 및 문제 정의

### 2.1. 배경

- DOWIN는 4DX 책의 문제의식과 구조를 제품으로 옮긴 서비스다.
- 현재 제품에는 `가중목`, `선행지표`, `후행지표`, `점수판`, `책무` 같은 책 중심 용어가 핵심 객체명과 UX 문구에 직접 노출된다.
- 이 구조는 책을 이미 아는 사용자에게는 강점이지만, 책을 읽지 않은 신규 사용자에게는 진입 장벽이 된다.

### 2.2. 지금 해결해야 하는 문제

- 현재 제품은 기능 자체보다 용어 해석이 먼저 필요한 구간이 존재한다.
- 이 문제는 랜딩 카피 수준에 그치지 않고, 설정 화면, 대시보드, 알림, 팀 운영 문맥, 빈 상태 문구까지 제품 전반에 걸쳐 반복된다.
- 가장 위험한 결과는 `제품이 쉬워지긴 했지만 평범한 할 일 앱처럼 보이는 것`이다.

### 2.3. 이번 작업의 핵심 원칙

- 내부 모델은 유지하고 사용자 노출 언어만 정리한다.
- 책 용어를 완전히 삭제하지 않고 보조 설명으로 내린다.
- 같은 개념은 핵심 화면에서 반드시 같은 기본 명칭으로 유지한다.
- 용어는 쉽게 바꾸되, 4DX 구조는 더 강하게 보여줘야 한다.

## 3. 현재 상태 진단

### 3.1. 가장 큰 문제

- 현재 제품은 같은 개념을 화면마다 다른 온도로 부른다.
- 예를 들어 Setup과 대시보드에는 `가중목`, `후행지표`, `선행지표`, `점수판`이 강하게 남아 있는데, 일부 문구와 랜딩에서는 이미 `최우선 목표`, `액션`, `행동` 같은 범용 표현이 섞여 있다.
- 이 상태는 쉬워지는 중간 단계가 아니라, 제품 학습 비용이 더 커지는 혼용 상태다.

### 3.2. 실제 혼용 사례

- `src/messages/ko.json`
  - `Setup.dowinShort`: `가중목`
  - `Setup.leadMeasureShort`: `핵심 행동`
  - `Dashboard.dowinLabel`: `가중목`
  - 랜딩 `Landing.Mockup.stats.dowinTitle`: `최우선 목표`
  - 랜딩 `Landing.Mockup.stats.leadTitle`: `선행지표: 이번 주 액션`
- 즉, 한 제품 안에 `가중목`, `최우선 목표`, `핵심 행동`, `선행지표`, `액션`이 동시에 공존한다.

## 4. 최종 용어 전략

### 4.1. 기본 원칙

- 기본 라벨은 범용 언어를 쓴다.
- 4DX 용어는 보조 설명, 코치마크, 도움말에서만 노출한다.
- 같은 개념은 핵심 화면에서 반드시 같은 기본 명칭으로 유지한다.

### 4.2. 최종 확정 용어

| 내부 개념           | 사용자 기본 노출명 | 보조 설명                 | 이번 단계 결정 |
| ------------------- | ------------------ | ------------------------- | -------------- |
| 가중목 / Dowin      | 핵심 목표          | 4DX의 가중목              | 확정           |
| 후행지표            | 성공 기준          | 목표 결과, 4DX의 후행지표 | 확정           |
| 선행지표            | 액션 아이템        | 반복 실행 항목, 4DX의 선행지표 | 확정           |
| 점수판 / Scoreboard | 점수판             | 핵심 목표와 액션 아이템 진행 상황을 보는 보드 | 유지 확정 |
| 책무                | 주간 점검          | 실행 점검, 약속 확인      | 확정           |

### 4.3. 왜 이 조합인가

- `핵심 목표`
  - `가장 중요한 목표`의 의미를 유지하면서도 화면 라벨로 쓰기 쉽다.
- `성공 기준`
  - 사용자가 `이걸 왜 입력하지?`에 바로 답이 된다.
  - `후행지표`보다 설명 비용이 낮다.
- `액션 아이템`
  - 사용자가 즉시 이해하기 쉽다.
  - 다만 일반 투두처럼 읽힐 수 있으므로 `반복해서 실행할 액션 아이템` 같은 보조 설명이 필요하다.
- `점수판`
  - 이미 제품 메타포로 충분히 자연스럽다.
  - `성과 현황`처럼 결과 조회 화면으로 좁아지지 않는다.
- `주간 점검`
  - `책무`의 긴장감은 남기되 감시/통제 뉘앙스는 줄인다.

### 4.4. 가장 경계할 실패

- 용어는 쉬워졌지만 기존 4DX 구조 의미가 사라져 제품이 평범한 할 일 앱처럼 보이는 것
- 특히 `액션 아이템`이 일반 투두처럼 읽히는 것
- `점수판`을 과도하게 일반화해서 운영 보드의 메타포를 잃는 것

## 5. 금지 표현 / 허용 표현

### 5.1. 금지 표현

- `가중목 -> 목표`
  - 너무 평평해서 `집중` 의미가 사라진다.
- `선행지표 -> 할 일`
  - 일반 투두처럼 들려 제품 구조가 무너진다.
- `점수판 -> 대시보드`
  - 이미 대시보드라는 상위 화면이 있어 개념 충돌이 생긴다.
- `점수판 -> 성과 현황`
  - 결과 보고서처럼 들려 운영 보드의 의미가 약해진다.
- `책무 -> 감시`
  - 리더 중심 운영 가치를 해친다.

### 5.2. 허용 표현

- 첫 진입 설명 문구에서만
  - `핵심 목표(4DX의 가중목)`
  - `액션 아이템(4DX의 선행지표)`
  - `성공 기준(4DX의 후행지표)`
- 보조 문맥에서는
  - `이번 기간의 핵심 목표`
  - `결과를 바꾸는 액션 아이템`
  - `이번 주 점수판`

## 6. 화면별 적용 기준

### 6.1. 랜딩

#### 현재 문제

- 랜딩은 상대적으로 쉬운 언어를 쓰고 있지만 `선행지표`, `점수판`이 여전히 섞여 있다.
- 제품을 처음 보는 사용자는 랜딩에서 쉬운 말을 보고 들어왔다가 Setup에서 갑자기 책 용어를 만나게 된다.

#### 적용 원칙

- 랜딩은 끝까지 범용 언어만 유지한다.
- 책 용어는 랜딩 본문에 직접 넣지 않는다.

#### 추천 치환

| 현재 표현                   | 변경 표현                     |
| --------------------------- | ----------------------------- |
| 최우선 목표                 | 핵심 목표                     |
| 선행지표: 이번 주 액션      | 액션 아이템: 이번 주 실행     |
| 팀 전체의 점수판            | 팀 전체의 점수판              |
| 핑계 없는 점수판, 주간 회고 | 한눈에 보는 점수판, 주간 점검 |
| 선행지표 쌓아가기           | 액션 아이템 쌓아가기          |

### 6.2. Setup

#### 현재 문제

- 가장 중요한 생성 플로우에서 가장 많은 책 용어가 직접 노출된다.
- 다만 의미 설명 자체는 이미 꽤 잘 되어 있어, 라벨만 바꿔도 체감 개선이 크다.

#### 적용 원칙

- Setup은 가장 먼저 범용화해야 한다.
- 라벨은 쉬워져야 하지만, 코치마크와 helper text를 통해 4DX 의미는 유지해야 한다.

#### 추천 치환

| 현재 표현          | 변경 표현        |
| ------------------ | ---------------- |
| 점수판 설정        | 점수판 설정      |
| 새 점수판 생성하기 | 새 점수판 만들기 |
| 가중목             | 핵심 목표        |
| 후행지표           | 성공 기준        |
| 선행지표           | 액션 아이템      |
| 선행지표 추가      | 액션 아이템 추가 |
| 점수판 보관        | 점수판 보관      |

#### 유지할 보조 설명

- `핵심 목표`
  - helper text 또는 코치마크에서 `4DX의 가중목` 설명 유지
- `성공 기준`
  - `특정 일까지 X에서 Y로` 형식 유지
- `액션 아이템`
  - `결과를 바꾸는 반복 행동` 설명 유지

### 6.3. 내 대시보드

#### 현재 문제

- 헤더/카드/섹션 타이틀에는 여전히 `가중목`, `후행지표`, `선행지표`, `점수판`이 남아 있다.
- 사용자는 기록 단계로 들어온 뒤에도 책 용어를 계속 해석해야 한다.

#### 적용 원칙

- 대시보드는 행동과 상태를 이해시키는 화면이므로, 개념 정의보다 `지금 무엇을 보면 되는지`가 드러나야 한다.

#### 추천 치환

| 현재 표현     | 변경 표현     |
| ------------- | ------------- |
| 가중목        | 핵심 목표     |
| 후행지표      | 성공 기준     |
| 선행지표      | 액션 아이템   |
| 점수판 보관함 | 점수판 보관함 |
| 점수판 관리   | 점수판 관리   |
| 월간 기록판   | 월간 점수판   |

### 6.4. 팀 대시보드

#### 현재 문제

- 팀 뷰는 리더가 보는 화면인데도 `점수판` 중심 표현이 강하다.
- 리더에게 필요한 것은 용어가 아니라 `누가 어디서 막히는지`다.

#### 적용 원칙

- 팀 뷰는 `운영`, `점검`, `개입` 언어를 우선 사용한다.
- 팀원이 평가 대상처럼 보이지 않도록 `지원`, `점검`, `흐름` 표현을 유지한다.

#### 추천 치환

| 현재 표현           | 변경 표현           |
| ------------------- | ------------------- |
| 팀 주간 점수판      | 팀 주간 점수판      |
| 선행지표 달성 현황  | 액션 아이템 실행 현황 |
| 점수판 없음         | 점수판 없음         |
| 점수판부터 열어주기 | 점수판부터 만들기   |

### 6.5. 프로필 Export / 보관함 / 부가 화면

#### 적용 원칙

- 부가 화면도 주 용어를 따라야 한다.
- 특히 empty state와 CTA는 핵심 플로우 용어와 정확히 맞아야 한다.

#### 추천 치환

| 현재 표현            | 변경 표현               |
| -------------------- | ----------------------- |
| 아직 가중목이 없어요 | 아직 핵심 목표가 없어요 |
| 새 점수판 만들기     | 새 점수판 만들기        |
| 선행지표 선택        | 액션 아이템 선택        |
| 점수판 보관함        | 점수판 보관함           |

## 7. 코치마크 / 도움말 기준

### 7.1. 코치마크는 유지하되 역할을 바꾼다

- 지금의 코치마크는 책 용어를 설명하는 구조다.
- 변경 후 코치마크는 `범용 용어를 먼저 보여주고`, 그 아래에서 `4DX 용어 대응 관계`를 알려주는 구조여야 한다.

### 7.2. 예시

- `핵심 목표`
  - 보조 설명: `4DX에서는 이것을 가중목이라고 부릅니다. 이번 기간에 가장 우선순위가 높은 목표 1개에 집중하세요.`
- `성공 기준`
  - 보조 설명: `4DX의 후행지표입니다. 목표가 달성됐는지 판단할 수 있게 X에서 Y로 형태로 적어주세요.`
- `액션 아이템`
  - 보조 설명: `4DX의 선행지표입니다. 결과를 바꾸는 반복 행동만 남기는 것이 중요합니다.`

## 8. 적용 우선순위

### 8.1. 1차 적용

- `src/messages/ko.json`
  - `Setup`
  - `Dashboard`
  - `Common`
- 대상 화면
  - 랜딩
  - Setup
  - `dashboard/my`
  - 팀 대시보드

### 7.2. 2차 적용

- 보관함
- Export
- 업데이트 카드
- 알림 문구

### 7.3. 3차 적용

- 법률 문서, 외부 카피, 마케팅 문구, 영문 번역 동기화

## 9. 구현 시 주의사항

- 번역 키 이름은 당장 바꾸지 않아도 된다.
  - 예: `dowinShort`, `lagMeasureLabel`, `leadMeasureHead` 키는 유지하고 값만 우선 변경 가능
- 다만 신규 카피 추가 시에는 점진적으로 범용 키명으로 옮기는 것이 좋다.
  - 예: `primaryGoalLabel`, `successCriteriaLabel`, `keyActionLabel`
- 프론트 구현 단계에서 가장 먼저 해야 할 일은 `값 치환`이지 `키명 리네이밍`이 아니다.

## 10. 성공 기준

- 사용자가 Setup에서 책 용어를 몰라도 입력을 시작할 수 있다.
- 랜딩에서 본 용어와 실제 제품 안 용어가 이어진다.
- 팀 리더가 대시보드를 볼 때 `운영 도구`로 읽히고 `개념 공부 화면`처럼 느껴지지 않는다.
- 제품 전체에서 `핵심 목표 / 성공 기준 / 액션 아이템 / 점수판` 조합이 안정적으로 유지된다.
