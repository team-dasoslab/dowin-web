# Polar Seat Count 수정 플로우 정리

## 1. 배경

Dowin의 Basic 워크스페이스 생성 checkout은 사용자가 입력한 `seatCount`를 Polar checkout에 전달한다.

현재 워크스페이스 생성 checkout은 아래 정책을 사용한다.

- 신규 워크스페이스 생성 checkout
  - `seats = seatCount`
  - `minSeats = seatCount`
  - `maxSeats = seatCount`
- 기존 Basic 재시작 checkout
  - `seats = max(requestedSeatCount, usedSeatCount)`
  - `minSeats = usedSeatCount`
  - `maxSeats = 999`
- webhook 또는 checkout 완료 검증에서 받은 실제 `seats`를 최종적으로 `workspace_seat_entitlements.purchasedSeatCount`에 반영한다.

문제는 사용자가 checkout 전에 멤버 한도를 잘못 입력하고 결제를 완료할 수 있다는 점이다. 이 경우 결제 직후 Dowin에서 멤버 한도를 어떻게 수정할 수 있는지 정책과 구현 경로가 필요하다.

## 2. 현재 확인한 Polar 동작

Polar의 seat-based pricing은 결제 이후에도 seat 수 관리를 지원한다.

확인한 공식 문서 기준:

- seat-based pricing에서는 billing manager가 Customer Portal 또는 API로 seats를 관리할 수 있다.
- Customer Portal 설정에서 seat count 변경을 허용할 수 있다.
- Portal 변경을 비활성화하면 앱이 직접 Update Subscription endpoint를 통해 변경을 처리해야 한다.
- Update Subscription API는 seat-based subscription의 seat 수를 업데이트할 수 있다.

참고:

- https://polar.sh/docs/features/seat-based-pricing
- https://polar.sh/docs/features/customer-portal/settings
- https://polar.sh/docs/guides/subscription-upgrades
- https://polar.sh/docs/api-reference/customer-portal/subscriptions/update

## 3. Dowin에서 가능한 수정 방식

### 3.1. Customer Portal 수정

가장 단순한 방식은 Polar Customer Portal에서 seat count 변경을 허용하는 것이다.

장점:

- Dowin이 별도 수정 UI/API를 만들지 않아도 된다.
- 결제 수단, 구독 상태, seat 변경이 Polar 쪽 표준 흐름으로 처리된다.
- Dowin은 webhook의 실제 subscription seats를 받아 `workspace_seat_entitlements`만 갱신하면 된다.

단점:

- 사용자가 Portal에서 변경하는 순간을 Dowin UI가 직접 통제하기 어렵다.
- Dowin의 멤버 수 정책, over-limit 안내, 변경 전 확인 문구를 강하게 넣기 어렵다.
- Portal 설정에 따라 사용자가 현재 멤버 수보다 낮은 seat 수를 선택할 수 있는지 별도 확인이 필요하다.

### 3.2. Dowin 서버에서 Polar subscription update 호출

Dowin에 “멤버 한도 변경” API를 만들고, 서버가 Polar Update Subscription API를 호출하는 방식이다.

권장 정책:

- 워크스페이스 ADMIN만 변경 가능
- `requestedSeatCount >= currentMemberCount`만 허용
- `requestedSeatCount` 상한은 현재 checkout 정책과 동일하게 999
- 현재 구매 seat보다 큰 요청은 즉시 반영
- 현재 구매 seat보다 작은 요청은 다음 결제주기부터 반영
- 변경 성공 후 즉시 Dowin projection을 낙관적으로 바꾸기보다 Polar webhook 또는 subscription 조회 결과를 기준으로 최종 반영
- 실패 시 `BILLING_NOT_READY` 또는 별도 billing update error로 사용자에게 안내

장점:

- Dowin UI에서 현재 멤버 수와 변경 후 과금 영향을 함께 안내할 수 있다.
- 현재 멤버 수보다 낮은 seat 수를 서버에서 확실히 막을 수 있다.
- 추후 admin console에서 운영자 수동 보정과 구분하기 쉽다.

단점:

- Polar subscription update client 구현이 필요하다.
- 변경 즉시 과금/차액/proration이 어떻게 적용되는지 사용자 안내가 필요하다.
- webhook 반영 전후의 일시적 상태를 다뤄야 한다.

## 4. 현재 발견한 seatCount 미반영 원인

사용자가 멤버 한도를 `1`로 입력했는데 Polar 결제창에 반영되지 않는 현상은 Polar API 자체보다 Dowin의 pending checkout 재사용 가능성이 더 높다.

기존 `prepareWorkspaceCheckout` 흐름에는 같은 사용자의 활성 pending checkout이 있으면 새 요청의 `seatCount`를 무시하고 기존 checkout URL을 재사용하는 로직이 있었다.

이 경우:

1. 사용자가 seat 수 `5`로 checkout을 생성한다.
2. 결제를 완료하지 않고 돌아온다.
3. seat 수 `1`로 다시 checkout을 시도한다.
4. 서버가 새 checkout을 만들지 않고 기존 checkout URL을 반환한다.
5. Polar 결제창에는 이전 seat 수가 남아 보인다.

이번 백엔드 변경 방향은 다음과 같다.

- 같은 `Idempotency-Key` 요청만 기존 checkout을 재사용한다.
- 새 `Idempotency-Key` 요청은 같은 사용자라도 새 pending checkout을 생성한다.
- 따라서 사용자가 seat 수를 바꿔 다시 시도하면 새 seat 수가 Polar checkout 생성 요청에 반영된다.

## 5. MVP 권장안

MVP에서는 두 단계를 분리한다.

### 5.1. 즉시 적용

- 신규 checkout에서 기존 pending checkout URL 재사용 제거
- checkout 생성 시 `seats = minSeats = maxSeats = seatCount` 유지
- checkout 완료 후 Polar checkout detail 또는 webhook의 `seats`를 `purchasedSeatCount`로 반영

이 단계는 결제 전 seat 수 변경이 제대로 반영되도록 하는 수정이다.

### 5.2. 후속 구현

별도 “멤버 한도 변경” 플로우를 만든다.

필요 API 후보:

- `PATCH /workspaces/{workspaceId}/billing/seats`
- request
  - `seatCount: number`
- validation
  - workspace ADMIN
  - active Basic billing
  - `entitlementSource = POLAR`
  - `seatCount >= currentMemberCount`
  - `seatCount <= 999`
- behavior
  - Polar subscription update 호출
  - 현재 구매 seat보다 큰 요청은 `proration_behavior = "prorate"`로 즉시 반영
  - 현재 구매 seat보다 작은 요청은 `proration_behavior = "next_period"`로 다음 결제주기 적용 예약
  - 성공 응답은 변경 요청 접수 상태를 반환
  - 최종 seat 수는 webhook 또는 subscription 조회 기준으로 반영

### 5.3. 구현 반영 현황

2026-06-05 기준 “멤버 한도 변경” 플로우는 Dowin API 직접 호출 방식으로 구현되어 있다.

- `PATCH /api/workspaces/{workspaceId}/billing/seats`
  - ADMIN만 호출 가능
  - active Basic + Polar entitlement만 허용
  - 요청 seat가 현재 워크스페이스 멤버 수보다 작으면 `BILLING_SEAT_COUNT_BELOW_MEMBER_COUNT`로 거절
  - 현재 구매 seat보다 큰 요청은 Polar `proration_behavior = "prorate"`로 즉시 반영 요청
  - 현재 구매 seat보다 작은 요청은 Polar `proration_behavior = "next_period"`로 다음 결제주기 적용 예약
- `GET /api/workspaces/{workspaceId}/billing/me`
  - `workspace_seat_entitlements.purchasedSeatCount`를 현재 적용 한도로 반환
  - Polar subscription의 `pending_update.seats`를 조회해 `pendingSeatCount`로 반환
  - `pendingSeatCount`가 있으면 `workspace_billing_state.current_period_end`를 `pendingSeatEffectiveAt`로 반환
  - Polar 조회 실패/권한 부족 같은 recoverable 오류는 billing overview 조회 자체를 깨지 않도록 pending 정보를 `null`로 처리

`pendingSeatCount`는 현재 DB에 저장하지 않는다. 현재 적용 한도는 기존처럼 `workspace_seat_entitlements` projection을 사용하고, 예약 한도는 Polar subscription 조회 결과를 read-through로 노출한다. 다음 결제주기 도래 후 webhook 또는 subscription 조회 결과가 실제 `seats`를 반영하면 `workspace_seat_entitlements.purchasedSeatCount`가 최종 갱신된다.

## 6. 사용자 안내 문구 방향

결제 전:

- `멤버 한도는 결제 후에도 변경할 수 있습니다. 단, 현재 멤버 수보다 낮게 줄일 수는 없습니다.`

결제 후 seat 변경:

- `멤버 한도는 현재 남아 있는 멤버 수 이상으로만 줄일 수 있습니다.`
- `예: 10명에서 5명으로 줄이려면 먼저 멤버를 5명까지 내보낸 뒤 한도를 5명으로 변경할 수 있습니다.`
- `한도를 낮춘 경우 다음 결제주기부터 적용되고, 한도를 높인 경우 결제 상태 갱신 후 반영됩니다.`
- 예약된 다운그레이드가 있으면 billing 화면에서 “적용 예정 한도”를 별도 행으로 보여준다.
  - 적용일이 있으면 `{date}부터 {count}명`
  - 적용일을 알 수 없으면 `다음 결제주기부터 {count}명`

## 7. 정책 결정 및 미해결 질문

정책 결정:

- seat 증가: 즉시 반영한다. 운영상 새 멤버 초대/접근이 바로 필요하기 때문이다.
- seat 감소: 다음 결제주기부터 반영한다. 이미 결제한 기간의 조건을 유지해 크레딧/환불 혼란을 줄이기 위함이다.
- Dowin API 직접 호출 시 증가 요청은 Polar `proration_behavior = "prorate"`, 감소 요청은 `proration_behavior = "next_period"`를 명시한다.

미해결 질문:

- Polar Customer Portal에서 seat count 변경을 켤지, Dowin 자체 UI/API로만 처리할지 결정이 필요하다.
- 증가 요청의 `prorate`, 감소 요청의 `next_period` 결과가 실제 sandbox에서 어떤 invoice/portal 문구로 표시되는지 확인이 필요하다.
- seat 감소 요청이 현재 사용 중인 seat보다 낮을 때 Dowin은 서버에서 먼저 막는다. Polar 자체 차단 여부와 무관하게 Dowin 정책을 우선 적용한다.
- 감소 예약은 `GET /api/workspaces/{workspaceId}/billing/me`에서 Polar `pending_update.seats`를 조회해 UI에 표시한다. 별도 DB 저장은 하지 않는다.
- 다음 결제주기 도래 후 webhook 또는 subscription 조회 결과로 `purchasedSeatCount`를 최종 갱신하는 경로는 계속 검증해야 한다.
- 사용자가 감소 예약을 다시 변경하거나 취소할 수 있는지 정책을 정해야 한다.

## 9. 활성 구독 워크스페이스 삭제 이슈

현재 구현 기준으로 `DELETE /api/workspaces/{workspaceId}`는 ADMIN 권한만 확인한 뒤 Dowin DB의 워크스페이스 row를 삭제한다.

현재 삭제 흐름에는 아래 처리가 없다.

- `workspace_billing_state` 조회
- 활성 Polar 구독 여부 확인
- Polar subscription cancel 호출
- Customer Portal 유도
- 삭제 전 billing 데이터 보존 또는 운영 보정 이벤트 기록

현재 DB 스키마도 이 문제를 키운다. `billing_events`, `billing_checkout_events`, `workspace_billing_state`, `workspace_seat_entitlements`는 모두 `workspace_id -> workspaces.id` cascade 관계다. 따라서 워크스페이스를 hard delete하면 점수판/멤버 데이터뿐 아니라 Dowin 로컬 결제 원장, checkout 이력, 현재 billing projection, seat entitlement까지 같이 사라진다.

따라서 활성 Basic 구독 워크스페이스를 삭제하면 Dowin 로컬 데이터는 삭제되지만 Polar 구독은 계속 유지될 수 있다. 이 경우 사용자는 Dowin 안에서 해당 구독의 결제 관리 화면으로 다시 진입하기 어렵고, 이후 Polar webhook이 도착해도 삭제된 workspace를 매칭하지 못해 무시될 수 있다. Polar 대시보드에는 결제/고객 정보가 남아 있어도, Dowin에서 "어떤 워크스페이스의 어떤 상태였는지"를 재구성하기 어렵다.

이 문제는 활성 구독뿐 아니라 과거 결제 이력이 있는 워크스페이스에도 적용된다. 환불, chargeback, 반복 결제 악용, 고객지원 문의, 세무/운영 확인을 위해서는 Dowin 내부의 workspace 식별자와 provider 식별자를 함께 보존해야 한다.

임시 정책:

- 결제 이력이 있는 Polar 워크스페이스는 hard delete를 막아야 한다.
- 최소 즉시 차단 범위는 `billingStatus = ACTIVE | CANCELED`이고 `entitlementSource = POLAR`인 워크스페이스다.
- 더 안전한 정책은 `workspace_billing_state.provider = POLAR`, `billing_events` 존재, `billing_checkout_events` 존재, `billingCustomerExternalRef` 존재 중 하나라도 해당하면 hard delete를 금지하는 것이다.
- 사용자에게는 먼저 결제 관리에서 구독을 취소하거나 고객지원으로 문의하도록 안내한다.
- 결제 이력이 없는 무료/테스트 워크스페이스만 기존 hard delete를 허용한다.

후속 구현 후보:

- `DELETE /api/workspaces/{workspaceId}`에서 `workspace_billing_state`를 조회한다.
- `ACTIVE` 또는 아직 기간이 남은 `CANCELED` Polar 구독이면 삭제를 거절한다.
- billing 이력 존재 여부를 확인하는 storage 메서드를 추가한다.
- 거절 에러는 기존 `BILLING_NOT_READY`보다 명확한 별도 코드가 낫다.
  - 후보: `WORKSPACE_ACTIVE_SUBSCRIPTION_DELETE_FORBIDDEN`
- 프론트 삭제 UI도 billing 상태를 보고 활성 구독이면 삭제 CTA 대신 결제 관리/문의 CTA를 우선 노출한다.
- 장기적으로는 workspace deletion과 subscription cancellation을 분리한다.
- 결제 이력이 있는 워크스페이스는 soft delete 또는 archive 상태로 숨기고, billing 원장과 provider 식별자는 보존한다.
- hard delete가 필요한 경우에는 운영자 전용 절차로만 허용하고, 삭제 전 별도 billing archive/snapshot을 남긴다.

### 9.1. 권장 삭제 UX / API 정책

사용자 관점의 `워크스페이스 삭제`는 실제 DB hard delete가 아니라 `deleted_at` 기반 soft delete로 정의한다. 사용자는 삭제된 것으로 보지만, Dowin은 billing 원장과 provider 식별자를 보존한다.

권장 흐름:

1. 사용자가 워크스페이스 삭제를 요청한다.
2. 서버가 현재 billing 상태와 billing 이력 존재 여부를 확인한다.
3. `ACTIVE` 또는 기간이 남은 `CANCELED` Polar 구독이면 삭제를 거절한다.
   - 안내: 먼저 결제 관리에서 구독을 취소하고, 구독 종료 또는 운영자 확인 후 삭제할 수 있다.
4. 활성 구독이 없고 삭제 가능 상태이면 `workspaces.deleted_at`을 기록한다.
5. 일반 사용자 조회에서는 `deleted_at is null` 워크스페이스만 반환한다.
6. 삭제된 워크스페이스의 dashboard/setup/report 등 운영 화면 접근은 `404` 또는 삭제 안내 화면으로 막는다.
7. 운영자/고객지원 도구에서는 삭제된 워크스페이스와 billing 식별자를 조회할 수 있게 둔다.

구현 순서:

1. `workspaces.deleted_at` migration 추가
2. `WorkspaceStorage.softDeleteWorkspace` 추가
3. 기본 workspace 조회 경로에 `deleted_at is null` 적용
   - `resolveIdByUid`
   - `findWorkspaceById`
   - `findUserWorkspace`
   - `listUserWorkspaces`
   - `findAccessibleWorkspaceByUid`
   - `getAccessContextData`
4. `DELETE /api/workspaces/{workspaceId}`에서 billing guard 추가
5. 활성/기간 유지 구독이면 삭제 거절 에러 반환
6. 삭제 가능하면 hard delete 대신 soft delete 수행
7. 프론트 삭제 UI에서 활성 구독 상태면 삭제 버튼 대신 결제 관리 CTA와 안내 문구 노출

주의:

- `workspaces.deleted_at`만 추가하고 조회 필터를 빠뜨리면 삭제된 워크스페이스가 계속 노출될 수 있다.
- soft delete된 워크스페이스에 webhook이 도착하면 billing 원장은 계속 수신하되, 운영 접근권은 복구하지 않는다.
- hard delete는 결제 이력이 없는 워크스페이스 또는 운영자 전용 정리 작업으로 제한한다.
