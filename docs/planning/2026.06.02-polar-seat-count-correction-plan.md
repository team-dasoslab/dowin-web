# Polar Seat Count 수정 플로우 정리

## 1. 배경

Dowin의 Basic 워크스페이스 생성 checkout은 사용자가 입력한 `seatCount`를 Polar checkout에 전달한다.

현재 워크스페이스 생성 checkout은 아래 정책을 사용한다.

- 신규 워크스페이스 생성 checkout
  - `seats = seatCount`
  - `minSeats = seatCount`
  - `maxSeats = seatCount`
- 기존 Basic 재시작 checkout
  - `seats = max(requestedSeatCount, usedSeatCount)`
  - `minSeats = usedSeatCount`
  - `maxSeats = 999`
- webhook 또는 checkout 완료 검증에서 받은 실제 `seats`를 최종적으로 `workspace_seat_entitlements.purchasedSeatCount`에 반영한다.

문제는 사용자가 checkout 전에 멤버 한도를 잘못 입력하고 결제를 완료할 수 있다는 점이다. 이 경우 결제 직후 Dowin에서 멤버 한도를 어떻게 수정할 수 있는지 정책과 구현 경로가 필요하다.

## 2. 현재 확인한 Polar 동작

Polar의 seat-based pricing은 결제 이후에도 seat 수 관리를 지원한다.

확인한 공식 문서 기준:

- seat-based pricing에서는 billing manager가 Customer Portal 또는 API로 seats를 관리할 수 있다.
- Customer Portal 설정에서 seat count 변경을 허용할 수 있다.
- Portal 변경을 비활성화하면 앱이 직접 Update Subscription endpoint를 통해 변경을 처리해야 한다.
- Update Subscription API는 seat-based subscription의 seat 수를 업데이트할 수 있다.

참고:

- https://polar.sh/docs/features/seat-based-pricing
- https://polar.sh/docs/features/customer-portal/settings
- https://polar.sh/docs/guides/subscription-upgrades
- https://polar.sh/docs/api-reference/customer-portal/subscriptions/update

## 3. Dowin에서 가능한 수정 방식

### 3.1. Customer Portal 수정

가장 단순한 방식은 Polar Customer Portal에서 seat count 변경을 허용하는 것이다.

장점:

- Dowin이 별도 수정 UI/API를 만들지 않아도 된다.
- 결제 수단, 구독 상태, seat 변경이 Polar 쪽 표준 흐름으로 처리된다.
- Dowin은 webhook의 실제 subscription seats를 받아 `workspace_seat_entitlements`만 갱신하면 된다.

단점:

- 사용자가 Portal에서 변경하는 순간을 Dowin UI가 직접 통제하기 어렵다.
- Dowin의 멤버 수 정책, over-limit 안내, 변경 전 확인 문구를 강하게 넣기 어렵다.
- Portal 설정에 따라 사용자가 현재 멤버 수보다 낮은 seat 수를 선택할 수 있는지 별도 확인이 필요하다.

### 3.2. Dowin 서버에서 Polar subscription update 호출

Dowin에 “멤버 한도 변경” API를 만들고, 서버가 Polar Update Subscription API를 호출하는 방식이다.

권장 정책:

- 워크스페이스 ADMIN만 변경 가능
- `requestedSeatCount >= currentMemberCount`만 허용
- `requestedSeatCount` 상한은 현재 checkout 정책과 동일하게 999
- 현재 구매 seat보다 큰 요청은 즉시 반영
- 현재 구매 seat보다 작은 요청은 다음 결제주기부터 반영
- 변경 성공 후 즉시 Dowin projection을 낙관적으로 바꾸기보다 Polar webhook 또는 subscription 조회 결과를 기준으로 최종 반영
- 실패 시 `BILLING_NOT_READY` 또는 별도 billing update error로 사용자에게 안내

장점:

- Dowin UI에서 현재 멤버 수와 변경 후 과금 영향을 함께 안내할 수 있다.
- 현재 멤버 수보다 낮은 seat 수를 서버에서 확실히 막을 수 있다.
- 추후 admin console에서 운영자 수동 보정과 구분하기 쉽다.

단점:

- Polar subscription update client 구현이 필요하다.
- 변경 즉시 과금/차액/proration이 어떻게 적용되는지 사용자 안내가 필요하다.
- webhook 반영 전후의 일시적 상태를 다뤄야 한다.

## 4. 현재 발견한 seatCount 미반영 원인

사용자가 멤버 한도를 `1`로 입력했는데 Polar 결제창에 반영되지 않는 현상은 Polar API 자체보다 Dowin의 pending checkout 재사용 가능성이 더 높다.

기존 `prepareWorkspaceCheckout` 흐름에는 같은 사용자의 활성 pending checkout이 있으면 새 요청의 `seatCount`를 무시하고 기존 checkout URL을 재사용하는 로직이 있었다.

이 경우:

1. 사용자가 seat 수 `5`로 checkout을 생성한다.
2. 결제를 완료하지 않고 돌아온다.
3. seat 수 `1`로 다시 checkout을 시도한다.
4. 서버가 새 checkout을 만들지 않고 기존 checkout URL을 반환한다.
5. Polar 결제창에는 이전 seat 수가 남아 보인다.

이번 백엔드 변경 방향은 다음과 같다.

- 같은 `Idempotency-Key` 요청만 기존 checkout을 재사용한다.
- 새 `Idempotency-Key` 요청은 같은 사용자라도 새 pending checkout을 생성한다.
- 따라서 사용자가 seat 수를 바꿔 다시 시도하면 새 seat 수가 Polar checkout 생성 요청에 반영된다.

## 5. MVP 권장안

MVP에서는 두 단계를 분리한다.

### 5.1. 즉시 적용

- 신규 checkout에서 기존 pending checkout URL 재사용 제거
- checkout 생성 시 `seats = minSeats = maxSeats = seatCount` 유지
- checkout 완료 후 Polar checkout detail 또는 webhook의 `seats`를 `purchasedSeatCount`로 반영

이 단계는 결제 전 seat 수 변경이 제대로 반영되도록 하는 수정이다.

### 5.2. 후속 구현

별도 “멤버 한도 변경” 플로우를 만든다.

필요 API 후보:

- `PATCH /workspaces/{workspaceId}/billing/seats`
- request
  - `seatCount: number`
- validation
  - workspace ADMIN
  - active Basic billing
  - `entitlementSource = POLAR`
  - `seatCount >= currentMemberCount`
  - `seatCount <= 999`
- behavior
  - Polar subscription update 호출
  - 현재 구매 seat보다 큰 요청은 `proration_behavior = "prorate"`로 즉시 반영
  - 현재 구매 seat보다 작은 요청은 `proration_behavior = "next_period"`로 다음 결제주기 적용 예약
  - 성공 응답은 변경 요청 접수 상태를 반환
  - 최종 seat 수는 webhook 또는 subscription 조회 기준으로 반영

## 6. 사용자 안내 문구 방향

결제 전:

- `멤버 한도는 결제 후에도 변경할 수 있습니다. 단, 현재 멤버 수보다 낮게 줄일 수는 없습니다.`

결제 후 seat 변경:

- `현재 멤버 수보다 낮은 멤버 한도는 설정할 수 없습니다. 먼저 멤버를 정리한 뒤 다시 시도해주세요.`
- `멤버 한도 변경은 Polar 결제 정보에 반영됩니다. 실제 과금 변경 시점은 Polar 결제 정책을 따릅니다.`

## 7. 정책 결정 및 미해결 질문

정책 결정:

- seat 증가: 즉시 반영한다. 운영상 새 멤버 초대/접근이 바로 필요하기 때문이다.
- seat 감소: 다음 결제주기부터 반영한다. 이미 결제한 기간의 조건을 유지해 크레딧/환불 혼란을 줄이기 위함이다.
- Dowin API 직접 호출 시 증가 요청은 Polar `proration_behavior = "prorate"`, 감소 요청은 `proration_behavior = "next_period"`를 명시한다.

미해결 질문:

- Polar Customer Portal에서 seat count 변경을 켤지, Dowin 자체 UI/API로만 처리할지 결정이 필요하다.
- 증가 요청의 `prorate`, 감소 요청의 `next_period` 결과가 실제 sandbox에서 어떤 invoice/portal 문구로 표시되는지 확인이 필요하다.
- seat 감소 요청이 현재 사용 중인 seat보다 낮을 때 Polar가 자체적으로 막는지, Dowin에서만 막아야 하는지 확인이 필요하다.
- webhook만 기다릴지, subscription update 성공 후 즉시 subscription 조회를 한 번 더 할지 결정이 필요하다.
