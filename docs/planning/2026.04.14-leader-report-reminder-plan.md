# 리더용 주간 운영 리포트 + 팀원 리마인드 기획

## 1. 배경 및 맥락 (Background & Context)

- 현재 WIG는 개인/팀 점수판과 기본 푸시 리마인드 흐름을 갖추고 있지만, 리더가 팀 전체 상태를 빠르게 파악하고 개입하는 장면은 아직 약하다.
- 기존 논의에서 `주간 회의 모드`를 우선순위로 두는 안도 있었지만, 실제 리더 가치 관점에서는 별도 회의 화면보다 `주간 운영 리포트`와 `즉시 개입 액션`이 더 직접적이다.
- 특히 리더는 매번 앱에 직접 들어와 팀 전체를 훑기보다, 먼저 `이번 주 어디를 봐야 하는지`를 받고 필요한 팀원에게 바로 개입할 수 있어야 한다.

## 2. 문제 정의 (Problem Statement)

- 지금의 팀 대시보드는 정보를 보여주지만, 리더에게 `누구를`, `왜`, `어떻게` 개입해야 하는지 먼저 밀어주지는 않는다.
- 리더는 매주 팀 상태를 스스로 스캔해야 하므로 제품이 주는 운영 효율이 제한적이다.
- 푸시 인프라는 존재하지만, 현재는 개인 자기관리 리마인드 중심이라 리더의 팀 운영 액션과 직접 연결되지 않는다.

## 3. 목표 (Goals)

### 3.1. MVP 목표

- 워크스페이스 단위로 정한 요일과 시간에 팀 주간 운영 리포트를 발행한다.
- 리더가 리포트나 팀 대시보드를 보고 특정 팀원/선행지표에 대해 직접 리마인드를 보낼 수 있다.
- 리더가 팀원 평가보다 `실행 상태 파악 -> 개입 -> 후속 확인` 흐름으로 제품을 사용하게 만든다.
- 고정 시각 알림이 아니라, 리더/사용자 운영 패턴에 맞는 알림 스케줄 커스터마이즈 방향을 연다.

현재 플랜 전제:

- 리더용 주간 운영 리포트는 Free에서도 운영 가치 검증용으로 열어둘 수 있다.
- 팀원별 리더 리마인드는 출시 v1 기준 `STANDARD` 전용 기능으로 본다.

### 3.2. Post-MVP 목표

- 리포트 내용 개인화
- 개입 우선순위 자동 추천
- 리마인드 템플릿 고도화 및 성과 분석
- 이메일/카카오워크/슬랙 등 채널 확장

## 4. 핵심 원칙 (Product Principles)

- 이 기능의 1차 사용자는 팀 리더다.
- 리포트와 리마인드는 `평가`보다 `운영`, `개입`, `코칭`을 위한 기능이어야 한다.
- 리더는 팀 전체를 빠르게 스캔할 수 있어야 하고, 필요한 한 명에게 즉시 액션할 수 있어야 한다.
- 팀원 입장에서는 감시가 아니라 `지금 중요한 선행지표를 다시 보게 하는 체크인`으로 느껴져야 한다.
- 알림은 많을수록 좋은 것이 아니라, 적고 분명할수록 좋다.
- 이 기능 묶음의 유료 메리트는 `보는 것`이 아니라 `움직이게 하는 것`이어야 한다.

## 4.1. Free / STANDARD 역할 구분

- `FREE`
  - 리더가 팀 상태를 기본적으로 파악할 수 있다
  - 기본 주간 요약 리포트를 받을 수 있다
- `STANDARD`
  - 리더가 특정 팀원에게 바로 개입할 수 있다
  - 개입 우선순위를 더 선명하게 받을 수 있다
  - 개입 이후 실제 기록 재개 여부를 확인할 수 있다

즉 이 문서의 기능 구분은 아래 원칙을 따른다.

- Free는 `상태 파악`
- Standard는 `개입 + 후속 확인`

## 5. 기능 1: 리더용 주간 운영 리포트 (Weekly Leader Report)

### 5.1. 해결하려는 문제

- 리더가 매주 앱에 들어와 팀 상태를 직접 훑어야 한다.
- 팀원별 기록 누락, 반복 미달성, 점수판 부재 같은 문제를 먼저 알려주지 못한다.
- 결과적으로 제품이 `항상 열어보는 운영 도구`가 아니라 `필요할 때 들어가는 조회 화면`에 머문다.

### 5.2. 사용자 가치

- 리더는 앱에 들어오기 전부터 이번 주 운영 포인트를 안다.
- 팀 전체를 1분 내로 훑고, 바로 개입 우선순위를 정할 수 있다.
- 회의 문화가 없어도 주간 단위 관리 습관을 만들 수 있다.
- 다만 Free 리포트만으로는 `파악`까지가 기본이고, 실제 `개입`은 Standard에서 더 선명해져야 한다.

### 5.3. MVP 범위

- 워크스페이스 ADMIN이 관리하는 리포트 설정 제공
- 설정 주체:
  - 워크스페이스당 리더 리포트 설정은 1개만 둔다
  - ADMIN이 여러 명이어도 같은 요일/시간 설정을 공유한다
  - 발송 대상은 해당 워크스페이스의 모든 ADMIN이다
- 발행 요일 선택:
  - 예: 매주 월요일, 화요일, 수요일 중 1개
  - 향후 다중 요일/격주 발행은 Post-MVP
- 발행 시각:
  - MVP에서는 기본 추천 시각 제공 + 커스텀 시간 선택 가능 여부를 함께 검토
  - 최소한 `요일은 커스터마이즈 가능`해야 한다
- 전달 채널은 기존 푸시 인프라를 우선 활용
- 리포트 내용은 요약 중심으로 짧게 제공
- Free/Standard 구분:
  - `FREE`: 기본 팀 요약 + 팀 대시보드 딥링크
  - `STANDARD`: 후속 단계에서 개입 우선순위 강조, 개입 이력 연결, 후속 변화 확인까지 확장

### 5.3.1. 채널 전제와 한계

- 현재 WIG의 푸시는 웹 푸시 opt-in 기반이므로, 모든 ADMIN이 반드시 받는 채널은 아니다.
- 따라서 MVP에서는 `웹 푸시를 구독한 ADMIN만 수신`으로 제한한다.
- 웹 푸시 미구독 ADMIN을 위한 인앱 fallback/inbox는 Post-MVP로 둔다.
- 성공 판단도 이 제약을 전제로 잡아야 한다.

### 5.4. 리포트 포함 항목

- 워크스페이스명
- 기준 주차
- 팀 요약
  - 활성 점수판 보유 인원 수
  - 이번 주 기록 시작 인원 수
  - 기록 누락 인원 수
  - Win/Loss 인원 수
- 개입 필요 항목
  - 이번 주 아직 기록이 없는 팀원
  - 반복적으로 미달성 중인 팀원 또는 지표
  - 점수판이 없는 팀원
- 딥링크
  - `/dashboard` 팀 대시보드

MVP 메모:

- Free 리포트는 `누구를 먼저 개입해야 하는지`를 과도하게 계산하기보다 기본 운영 요약에 집중한다.
- 대신 Standard로 갈수록 `개입 우선순위`와 `후속 변화`가 더 선명하게 붙어야 한다.

### 5.5. 리포트 문구 방향

- 요약이 길면 안 된다.
- 사람을 평가하는 언어보다 운영 포인트를 알려주는 언어를 사용한다.
- 예시 방향:
  - `이번 주 확인이 필요한 팀원 2명이 있어요`
  - `아직 기록을 시작하지 않은 팀원이 있습니다`
  - `반복적으로 막히는 선행지표가 보입니다`

### 5.6. MVP 제외 범위

- 장문 이메일 리포트
- AI 자동 해석
- 팀원 개별 사유 분석
- 비교 랭킹/서열화

## 6. 기능 2: 팀원별 선행지표 리마인드 (Leader Reminder)

### 6.1. 해결하려는 문제

- 리더가 팀 대시보드에서 문제를 발견해도 바로 개입할 수단이 부족하다.
- 현재 푸시는 개인 자기관리용이며, 리더의 코칭 액션과 연결되지 않는다.
- 메모만으로는 즉시 행동 유도까지 이어지지 않는다.

### 6.2. 사용자 가치

- 리더는 특정 팀원에게 특정 선행지표를 기준으로 바로 체크인을 보낼 수 있다.
- 팀원은 `무엇을 다시 봐야 하는지`가 분명한 리마인드를 받는다.
- 리더의 개입이 추상적인 관리가 아니라 구체적 액션으로 남는다.
- 이 기능은 `STANDARD`의 대표 메리트로 보여야 한다.

### 6.3. MVP 범위

- 플랜 게이트:
  - 출시 v1 기준 `STANDARD` 플랜 전용
- 권한:
  - 워크스페이스 ADMIN만 발송 가능
- 발송 시작점:
  - 팀 대시보드 멤버 컨텍스트
  - 향후 주간 리포트 딥링크에서 바로 진입 가능
- 발송 대상:
  - 같은 워크스페이스의 특정 팀원 1명
- 발송 단위:
  - 해당 팀원의 특정 선행지표 1개
- 메시지:
  - 기본 템플릿 2~3개 중 선택
  - 짧은 자유 입력 한 줄 허용 여부는 MVP에서 선택 사항
- 전달 채널:
  - 기존 웹 푸시 우선

플랜 메시지 원칙:

- Free 사용자가 느껴야 하는 메시지:
  - `팀 상태는 볼 수 있지만, 팀원을 바로 움직이는 개입 도구는 STANDARD에서 제공된다`
- Standard 사용자가 느껴야 하는 메시지:
  - `문제를 보는 데서 끝나지 않고, 바로 체크인하고 이후 변화를 확인할 수 있다`

### 6.3.1. 스케줄 커스터마이즈 연계

- 현재 개인 기록 알림은 매일 밤 9시, 주간 집중 알림은 매주 목요일 오후 3시 고정 정책이다.
- 리더용 리포트/리마인드를 도입하면서, 기존 알림 정책도 장기적으로는 커스터마이즈 가능한 구조로 재설계해야 한다.
- 최소 요구:
  - 리더 주간 리포트는 요일 선택 가능
  - 개인 기록 리마인드는 향후 시간 커스터마이즈 확장 여지를 남기는 저장 구조여야 함

### 6.4. 리마인드 발송 플로우

1. 리더가 팀원 선택
2. 해당 팀원의 활성 선행지표 목록 확인
3. 리마인드 보낼 선행지표 선택
4. 템플릿 선택 또는 짧은 메시지 작성
5. 발송
6. 발송 성공 시 이력 저장

### 6.5. 기본 템플릿 예시 방향

- `이번 주 {선행지표명} 체크해보세요`
- `{선행지표명} 진행 상황 한번 확인 부탁드려요`
- `이번 주 목표 흐름을 위해 {선행지표명} 다시 봐주세요`

### 6.6. 남용 방지 규칙

- 동일 리더가 동일 팀원/동일 지표에 너무 자주 보내지 못하게 제한
  - 예: 하루 1회 또는 주 2회
- 발송 이력을 리더가 확인 가능해야 함
- 발송 이후 후속 기록 재개 여부를 확인 가능해야 함
- 팀원은 왜 이 알림을 받았는지 이해할 수 있어야 함

### 6.7. MVP 제외 범위

- 대량 일괄 발송
- 자동 발송 추천
- 감정 점수/압박 점수화
- 공개 피드백/공개 지적 기능
- Free 플랜 제공

## 7. 도메인 영향 범위 (Domain Impact)

- dashboard
  - 팀 대시보드에서 리더 개입 시작점 제공
  - 리포트 딥링크 도착지 정리
- notification
  - 리더용 주간 리포트 스케줄 발송
  - 리더가 발송하는 팀원 리마인드 푸시 추가
  - 현재 고정 시각 알림 정책을 사용자 설정형으로 확장할 수 있는 구조 검토
- workspace
  - ADMIN 권한 기준 발송/설정 권한 적용
- analytics
  - 리포트 발행률, 리마인드 발송률, 딥링크 클릭률, 후속 기록률 측정
- profile/settings
  - 워크스페이스 리더 리포트 설정 진입점 필요

## 8. 데이터 및 API 초안 (Draft)

### 8.1. 주간 리포트 설정

- 설정 항목 예시:
  - `enabled`
  - `sendDayOfWeek`
  - `sendTime`
  - `workspaceId`

추가 규칙:

- 리더 리포트 설정은 워크스페이스 단위로 1개만 가진다.
- ADMIN이 여러 명이면 동일한 설정을 공유하고, 웹 푸시를 구독한 ADMIN 모두에게 발송한다.

예상 계약:

- MVP에서는 기존 체계를 유지해 `/api/push/*` 하위로 확장한다.
- 이유:
  - 현재 개인 구독/설정/cron 경로가 이미 `/api/push/*`에 모여 있다.
  - 이번 단계의 목표는 도메인 재명명보다 기능 구현 착수다.
  - `/api/notifications/*` 승격은 채널 확장이나 inbox 도입 시점에 다시 검토한다.
- 예시:
  - `GET /api/push/workspace-settings`
  - `PUT /api/push/workspace-settings`

### 8.2. 리더 리마인드 발송

요청 예시:

```json
{
  "targetUserId": 12,
  "leadMeasureId": 34,
  "templateKey": "check_progress",
  "customMessage": "오늘 이 지표 한번 체크해주세요"
}
```

예상 계약:

- MVP에서는 `/api/push/*` 하위로 확장한다.
- 계약 책임:
  - 리더가 팀원/선행지표 기준으로 1건 발송 가능
  - 발송 이력 조회 가능
  - ADMIN 권한 검증 포함
- 예시:
  - `POST /api/push/leader-reminders`
  - `GET /api/push/leader-reminders`

### 8.3. 내부 발송 크론

- 내부 크론 전용 엔드포인트 필요
- 각 워크스페이스 설정에 맞는 주간 리포트 발송
- 발송 시점에는 해당 워크스페이스의 ADMIN 목록을 조회해 웹 푸시 구독자에게 전달
- 네임스페이스는 기존 `/api/push/*`를 유지한다
- 예시:
  - `GET /api/push/send-leader-weekly-report`

### 8.4. 이벤트 로그 전제

- 현재 제품의 주요 계측은 클라이언트 `gtag` 이벤트 중심이다.
- 하지만 이 기능의 핵심 지표는 서버 발송과 후속 행동 연결이 필요하므로, 최소한 아래 이벤트를 별도 정의해야 한다.
  - 리더 리포트 발행
  - 리더 리포트 클릭
  - 리더 리마인드 발송
  - 리마인드 대상자의 후속 기록 재개
- 이 이벤트들은 단순 분석이 아니라 `STANDARD가 팀을 더 움직이게 하는가`를 증명하는 핵심 근거다.
- 구현 전 analytics/notification 경계에서 이벤트 저장 주체를 명시해야 한다.

## 10. 리스크 및 가드레일 (Risks & Guardrails)

### 10.1. 감시 도구처럼 보일 위험

- 리포트 문구는 평가/통제 톤을 피한다.
- 팀원 랭킹이나 공개 비교는 MVP에서 제외한다.
- 리마인드는 공개가 아니라 개인 푸시로 보낸다.

### 10.2. 알림 피로

- 주간 리포트는 주 1회만 허용한다.
- 리더 리마인드는 동일 대상/지표 기준 빈도 제한을 둔다.

### 10.3. 잘못된 개입 유도

- 리더가 개인을 압박하기보다, 특정 선행지표를 기준으로 체크인하도록 UI와 카피를 설계한다.
- 사람 자체가 아니라 `이번 주 중요한 행동`에 초점을 맞춘다.

## 13. MVP / Post-MVP 경계 (Boundary)

### 13.1. MVP

- 리더 주간 리포트 주 1회 발행
- 워크스페이스당 리더 리포트 설정 1개
- 발행 요일 선택 1개
- 리더 리포트 시간 또는 기본 시간대 정책 명시
- 팀원 1명 + 선행지표 1개 기준 리마인드 발송
- 기본 템플릿 기반 푸시
- 최소 발송 이력
- 기존 개인 알림 스케줄 커스터마이즈를 위한 저장 구조 초안

### 13.2. Post-MVP

- 다중 채널 발송
- 다중 요일 발송
- 개인/리더 알림 시간 완전 커스터마이즈
- 자동 추천 리마인드
- AI 요약/코칭
- 리더용 우선순위 보드 자동 생성

## 14. 기대 효과 (Expected Impact)

- WIG의 리더 가치가 `보는 대시보드`에서 `개입하는 운영 도구`로 확장된다.
- 회의 문화가 강하지 않은 팀에서도 주간 운영 습관을 만들 수 있다.
- 무료와 유료의 차이가 `정보 제공`과 `행동 유도`의 차이로 더 선명해진다.
- 이후 유료화 논의도 `관리 편의`가 아니라 `리더의 파악 속도`, `개입 효율`, `팀 실행 지속성` 같은 더 명확한 가치로 연결된다.

## 15. 성공 및 실패 판단 기준 (Success & Failure Criteria)

### 성공

- 웹 푸시를 구독한 ADMIN에게 주 1회 리더 리포트가 정상 발송된다.
- 리더가 앱에 직접 들어오기 전부터 이번 주 관리 포인트를 받는다.
- 리포트에서 팀 대시보드 딥링크 클릭이 실제로 발생한다.
- 리더가 특정 팀원/선행지표에 즉시 개입할 수 있다.
- 리마인드 이후 팀원의 실제 기록 재개가 관찰된다.

### 실패

- 리포트가 길고 추상적이어서 실제 개입으로 이어지지 않는다.
- 웹 푸시 미구독 ADMIN 비중이 높아 리포트 기능의 도달률 자체가 너무 낮다.
- 리마인드가 감시성 알림으로 받아들여져 팀원 반감만 만든다.
- 발송 후 기록 재개나 운영 개선 신호가 거의 없다.
