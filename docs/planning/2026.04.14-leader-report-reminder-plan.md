## 1. 배경 및 맥락 (Background & Context)

- 현재 WIG는 개인/팀 점수판과 기본 푸시 리마인드 흐름을 갖추고 있지만, 리더가 팀 전체 상태를 빠르게 파악하고 개입하는 장면은 아직 약하다.
- 기존 논의에서 `주간 회의 모드`를 우선순위로 두는 안도 있었지만, 실제 리더 가치 관점에서는 별도 회의 화면보다 `주간 운영 리포트`와 `즉시 개입 액션`이 더 직접적이다.
- 특히 리더는 매번 앱에 직접 들어와 팀 전체를 훑기보다, 먼저 `이번 주 어디를 봐야 하는지`를 받고 필요한 팀원에게 바로 개입할 수 있어야 한다.

## 2. 문제 정의 (Problem Statement)

- 지금의 팀 대시보드는 정보를 보여주지만, 리더에게 `누구를`, `왜`, `어떻게` 개입해야 하는지 먼저 밀어주지는 않는다.
- 리더는 매주 팀 상태를 스스로 스캔해야 하므로 제품이 주는 운영 효율이 제한적이다.
- 푸시 인프라는 존재하지만, 현재는 개인 자기관리 리마인드 중심이라 리더의 팀 운영 액션과 직접 연결되지 않는다.

## 3. 목표 (Goals)

### 3.1. MVP 목표

- 워크스페이스 단위로 정한 요일과 시간에 팀 주간 운영 리포트를 발행한다.
- 리더가 리포트나 팀 대시보드를 보고 특정 팀원/선행지표에 대해 직접 리마인드를 보낼 수 있다.
- 리더가 팀원 평가보다 `실행 상태 파악 -> 개입 -> 후속 확인` 흐름으로 제품을 사용하게 만든다.
- 고정 시각 알림이 아니라, 리더/사용자 운영 패턴에 맞는 알림 스케줄 커스터마이즈 방향을 연다.

현재 플랜 전제:

- 리더용 주간 운영 리포트는 Free에서도 운영 가치 검증용으로 열어둘 수 있다.
- 팀원별 리더 리마인드는 출시 v1 기준 `STANDARD` 전용 기능으로 본다.

### 3.2. Post-MVP 목표

- 리포트 내용 개인화
- 개입 우선순위 자동 추천
- 리마인드 템플릿 고도화 및 성과 분석
- 이메일/카카오워크/슬랙 등 채널 확장

## 4. 핵심 원칙 (Product Principles)

- 이 기능의 1차 사용자는 팀 리더다.
- 리포트와 리마인드는 `평가`보다 `운영`, `개입`, `코칭`을 위한 기능이어야 한다.
- 리더는 팀 전체를 빠르게 스캔할 수 있어야 하고, 필요한 한 명에게 즉시 액션할 수 있어야 한다.
- 팀원 입장에서는 감시가 아니라 `지금 중요한 선행지표를 다시 보게 하는 체크인`으로 느껴져야 한다.
- 알림은 많을수록 좋은 것이 아니라, 적고 분명할수록 좋다.
- 이 기능 묶음의 유료 메리트는 `보는 것`이 아니라 `움직이게 하는 것`이어야 한다.

## 4.1. Free / STANDARD 역할 구분

- `FREE`
  - 리더가 팀 상태를 기본적으로 파악할 수 있다
  - 기본 주간 요약 리포트를 받을 수 있다
- `STANDARD`
  - 리더가 특정 팀원에게 바로 개입할 수 있다
  - 개입 우선순위를 더 선명하게 받을 수 있다
  - 개입 이후 실제 기록 재개 여부를 확인할 수 있다

즉 이 문서의 기능 구분은 아래 원칙을 따른다.

- Free는 `상태 파악`
- Standard는 `개입 + 후속 확인`

## 5. 기능 1: 리더용 주간 운영 리포트 (Weekly Leader Report)

### 5.1. 해결하려는 문제

- 리더가 매주 앱에 들어와 팀 상태를 직접 훑어야 한다.
- 팀원별 기록 누락, 반복 미달성, 점수판 부재 같은 문제를 먼저 알려주지 못한다.
- 결과적으로 제품이 `항상 열어보는 운영 도구`가 아니라 `필요할 때 들어가는 조회 화면`에 머문다.

### 5.2. 사용자 가치

- 리더는 앱에 들어오기 전부터 이번 주 운영 포인트를 안다.
- 팀 전체를 1분 내로 훑고, 바로 개입 우선순위를 정할 수 있다.
- 회의 문화가 없어도 주간 단위 관리 습관을 만들 수 있다.
- 다만 Free 리포트만으로는 `파악`까지가 기본이고, 실제 `개입`은 Standard에서 더 선명해져야 한다.

### 5.3. MVP 범위

- 워크스페이스 ADMIN이 관리하는 리포트 설정 제공
- 설정 주체:
  - 워크스페이스당 리더 리포트 설정은 1개만 둔다
  - ADMIN이 여러 명이어도 같은 요일/시간 설정을 공유한다
  - 발송 대상은 해당 워크스페이스의 모든 ADMIN이다
- 발행 요일 선택:
  - 예: 매주 월요일, 화요일, 수요일 중 1개
  - 향후 다중 요일/격주 발행은 Post-MVP
- 발행 시각:
  - MVP에서는 기본 추천 시각 제공 + 커스텀 시간 선택 가능 여부를 함께 검토
  - 최소한 `요일은 커스터마이즈 가능`해야 한다
- 전달 채널은 기존 푸시 인프라를 우선 활용
- 리포트 내용은 요약 중심으로 짧게 제공
- Free/Standard 구분:
  - `FREE`: 기본 팀 요약 + 팀 대시보드 딥링크
  - `STANDARD`: 후속 단계에서 개입 우선순위 강조, 개입 이력 연결, 후속 변화 확인까지 확장

### 5.3.1. 채널 전제와 한계

- 현재 WIG의 푸시는 웹 푸시 opt-in 기반이므로, 모든 ADMIN이 반드시 받는 채널은 아니다.
- 따라서 MVP에서는 `웹 푸시를 구독한 ADMIN만 수신`으로 제한한다.
- 웹 푸시 미구독 ADMIN을 위한 인앱 fallback/inbox는 Post-MVP로 둔다.
- 성공 판단도 이 제약을 전제로 잡아야 한다.

### 5.4. 리포트 포함 항목

- 워크스페이스명
- 기준 주차
- 팀 승패 요약
  - `isWinning=true` 인원 수 / ACTIVE 점수판 보유 인원 수
  - 목적: 리더가 팀이 지금 이기고 있는지 한 줄로 파악
- 실행 시작 요약
  - 이번 주 최소 1회 이상 기록한 인원 수 / ACTIVE 점수판 보유 인원 수
  - 목적: 팀이 선행지표 실행을 실제로 시작했는지 확인
- 개입 필요 대상 요약
  - 이번 주 아직 기록이 없는 팀원 수
  - 점수판이 없는 팀원 수
  - 최근 2주 연속 `isWinning=false` 또는 반복 미기록 상태인 팀원 수
- 개입 필요 대상 목록
  - 이번 주 아직 기록이 없는 팀원 1~3명
  - 2주 연속 저조 상태인 팀원 1~3명
- 딥링크
  - `/dashboard` 팀 대시보드

MVP 메모:

- 책 기준으로 리더에게 가장 중요한 질문은 아래 3가지다.
  - 지금 이기고 있는가
  - 선행지표 실행이 실제로 시작됐는가
  - 어디에 개입해야 하는가
- 따라서 MVP는 이 3가지에 직접 답하는 지표만 노출한다.
- Free 리포트는 `누구를 먼저 개입해야 하는지`를 과도하게 계산하기보다 기본 운영 요약에 집중한다.
- 대신 Standard로 갈수록 `개입 우선순위`와 `후속 변화`가 더 선명하게 붙어야 한다.

### 5.4.1. 책 기준으로 제외하는 항목

- 팀원 랭킹
- 사람별 상세 비교표
- 과한 퍼널 지표 나열
- 장문 AI 해석
- 감정 점수화

이유:

- 책의 핵심은 `평가`보다 `점수판을 통한 실행 촉진`이다.
- 리더는 누구를 낮게 평가할지보다, 어디를 먼저 움직여야 할지를 알아야 한다.
- 따라서 MVP 리포트는 서열화보다 개입 포인트를 먼저 보여줘야 한다.

### 5.4.2. 현재 구현과의 매핑

- `GET /api/dashboard/team` 응답의 아래 필드를 우선 활용한다.
  - `hasScoreboard`
  - `achieved`
  - `total`
  - `achievementRate`
  - `isWinning`
- 여기서 바로 만들 수 있는 값:
  - ACTIVE 점수판 보유 인원 수
  - 팀 Win 인원 수
  - 팀 Loss 인원 수
  - 기록 시작 인원 수 (`achieved > 0`)
  - 미기록 인원 수 (`achieved = 0`)
- 반복 저조 인원은 주차별 스냅샷 또는 지난주 집계 비교가 추가로 필요하므로, 구현 시 비용을 보고 MVP 포함 여부를 확정한다.

### 5.4.3. 현재 프로토타입 진단 (2026-04-18)

현재 `src/app/[locale]/(protected)/report/page.tsx` 프로토타입 기준 문제는 아래와 같다.

- 헤드라인, TL;DR, 우측 규칙 카드가 사실상 같은 메시지를 반복한다.
- `팀 승패`, `기록 시작`, `미기록`, `점수판 없음`이 모두 같은 시각적 무게로 배치돼 우선순위가 보이지 않는다.
- `먼저 봐야 할 팀원`, `이번 주 리더 액션`, `이 리포트는 3가지만 답합니다`가 동시에 강하게 노출되어 초점이 분산된다.
- 팀원 카드마다 `왜`, `다음 액션`을 장문으로 적어 스캔보다 읽기를 요구한다.
- 현재 API로 확정적으로 보여줄 수 있는 수치와, 아직 추가 집계가 필요한 수치가 한 화면에서 같은 확실도로 보인다.

정리하면 현재 화면의 문제는 "정보가 부족"해서가 아니라 "한 번에 너무 많은 결론을 같은 톤으로 말한다"는 데 있다.

### 5.4.4. 리더 리포트 MVP 정보 우선순위

리더 리포트 화면은 아래 순서로만 읽히게 설계한다.

1. 팀 전체가 지금 이기고 있는가
2. 실행이 이번 주 실제로 시작됐는가
3. 지금 바로 확인할 대상이 누구인가

이 순서를 깨는 요소는 MVP에서 제거하거나 약화한다.

- 규칙 설명 카드
- 긴 해석 문단
- 중복 TL;DR 카드
- 별도 "다음 액션" 카드
- 과한 배지/아이콘/배경 톤 차이

### 5.4.5. 리더 리포트 MVP 화면 계약

MVP 화면은 짧아야 하지만, 앱 안의 리포트 화면인 만큼 "개입 판단"까지는 가능해야 한다.
따라서 `푸시 본문` 수준의 초압축 요약과 `인앱 리포트 화면`은 구분한다.

- 푸시 본문:
  - 팀 승패 + 즉시 확인 필요 요약만 전달
- 인앱 리포트 화면:
  - 요약
  - 상태 분해
  - 상태군 보드
  - 개입 대상
  - 변화 요약

즉, MVP 화면은 아래 5개 블록으로 본다.

#### A. 상단 한 줄 요약

- 한 문장으로 팀 상태를 요약한다.
- 형식 예시:
  - `이번 주 팀은 5명 중 3명이 이기고 있어요.`
- 보조 문장은 1문장만 허용한다.
  - 예: `아직 기록을 시작하지 않은 1명과 점수판이 없는 1명이 있어 먼저 확인이 필요합니다.`

핵심 원칙:

- 첫 문장은 팀 승패만 말한다.
- 둘째 문장은 실행 시작/개입 필요 대상을 함께 묶는다.
- 그 아래에 장문 해석이나 원칙 설명을 붙이지 않는다.

#### B. 핵심 숫자 3개

상단 요약 바로 아래에는 아래 숫자만 노출한다.

- `이기는 중`
  - `isWinning=true` 인원 수 / ACTIVE 점수판 인원 수
- `실행 시작`
  - 이번 주 `achieved > 0` 인원 수 / ACTIVE 점수판 인원 수
- `즉시 확인 필요`
  - `achieved = 0` 또는 `hasScoreboard = false` 인원 수

표시 원칙:

- 4개 이상으로 늘리지 않는다.
- `미기록`과 `점수판 없음`은 초기 화면에서 분리하지 않고 `즉시 확인 필요`로 합친다.
- 세부 분해는 아래 개입 대상 목록에서 라벨로 풀어준다.

#### C. 상태 분해 (왜 그런지)

핵심 숫자 아래에는 팀 상태를 한 단계 더 분해하는 짧은 요약 블록이 필요하다.

- `Win 3명 / Loss 2명`
- `기록 시작 4명 / 미시작 1명`
- `점수판 없음 1명`

역할:

- 상단 숫자를 다시 설명하는 층
- 리더가 "즉시 확인 필요 2명"의 구성 이유를 바로 이해하게 하는 층

표현 원칙:

- 차트보다 짧은 문장 또는 compact row가 우선이다.
- 여기서도 사람별 상세는 풀지 않는다.

#### D. 상태군 보드

리더 화면의 매력은 단순 합계보다 "팀이 어떻게 퍼져 있는지"를 보여줄 때 살아난다.
따라서 팀원을 랭킹하지 말고 상태군으로 묶는다.

권장 상태군:

- `이기는 중`
- `시작했지만 밀리는 중`
- `아직 시작 안 함`
- `점수판 없음`

표시 원칙:

- 각 상태군에 인원 수와 대표 이름 1~2명을 붙인다.
- "누가 상위권인가"보다 "어느 상태에 사람이 몰려 있는가"가 먼저 보이게 한다.
- 색상은 상태군 차이를 보조하는 수준으로만 쓴다.

#### E. 개입 대상 목록 3명 이내

- 진짜 핵심 블록은 이 목록이다.
- 최대 3명까지만 노출한다.
- 각 행은 아래 4개 정보만 가진다.
  - 이름
  - 상태 라벨
    - `이번 주 미기록`
    - `점수판 없음`
    - `2주 연속 Loss`
  - 현재 점수 요약
    - 예: `0 / 5`
    - 점수판 없음이면 `설정 전`
  - 리더 다음 액션 한 줄
    - 예: `이번 주 시작 가능한 가장 작은 행동부터 확인`

행동 원칙:

- `왜`와 `다음 액션`을 두 문단으로 나누지 않는다.
- 카드형 박스 여러 개보다 표 또는 조밀한 리스트에 가깝게 설계한다.
- 강한 배경색은 1개 상태에만 쓰지 말고, 라벨 색 정도로만 차이를 둔다.

#### F. 변화 요약

가능하면 이 블록을 붙여야 화면이 "살아있는 리포트"처럼 보인다.

예시:

- `이번 주 새로 시작한 사람 1명`
- `지난주보다 회복한 사람 1명`
- `이번 주 멈춘 사람 1명`

원칙:

- MVP에서는 mock 또는 보조 정보로 둘 수 있다.
- 현재 API로 확정 불가하면 secondary tone으로만 노출한다.
- 리더가 이번 주 변화 포인트를 감각적으로 느끼게 하는 역할이다.

#### G. 하단 보조 액션

- 하단에는 CTA 1~2개만 둔다.
  - `팀 대시보드 보기`
  - `리포트 설정`
- 별도의 설명 카드나 원칙 카드는 두지 않는다.

### 5.4.6. MVP 와이어프레임 텍스트 초안

```text
[이번 주 팀은 5명 중 3명이 이기고 있어요]
아직 시작하지 않은 1명과 점수판이 없는 1명이 있어 먼저 확인이 필요합니다.

[이기는 중 3/5] [실행 시작 4/5] [즉시 확인 필요 2명]

상태 분해
- Win 3명 / Loss 2명
- 기록 시작 4명 / 미시작 1명
- 점수판 없음 1명

팀 상태 분포
- 이기는 중 3명 | 유진, 태호
- 시작했지만 밀리는 중 1명 | 지수
- 아직 시작 안 함 1명 | 민수
- 점수판 없음 1명 | 현우

먼저 확인할 팀원
- 민수 | 이번 주 미기록 | 0/5 | 이번 주 시작 가능한 행동부터 확인
- 현우 | 점수판 없음 | 설정 전 | WIG와 선행지표 설정부터 열어주기
- 지수 | 2주 연속 Loss | 2/5 | 선행지표 난이도 재조정 논의

이번 주 변화
- 새로 시작한 사람 1명
- 회복한 사람 1명
- 멈춘 사람 1명

[팀 대시보드 보기] [리포트 설정]
```

### 5.4.7. 데이터 확실도에 따른 표현 규칙

- 현재 `GET /api/dashboard/team`만으로 확정 가능한 값
  - 팀 승패
  - 실행 시작 여부
  - 미기록 여부
  - 점수판 없음 여부
- 추가 주차 스냅샷 또는 비교 집계가 있어야 확정 가능한 값
  - `2주 연속 Loss`
  - `반복 미기록`
  - 개입 우선순위 자동 추천

따라서 프로토타입 단계에서는 아래 원칙을 지킨다.

- 현재 API로 확정 가능한 값은 기본 화면에 사용한다.
- 추가 집계가 필요한 값은 secondary badge 또는 mock 주석 수준으로만 둔다.
- `2주 연속 Loss`가 구현 전이라면 첫 프로토타입에서는 과감히 제외해도 된다.

### 5.5. 리포트 문구 방향

- 요약이 길면 안 된다.
- 사람을 평가하는 언어보다 운영 포인트를 알려주는 언어를 사용한다.
- 예시 방향:
  - `이번 주 팀은 5명 중 3명이 이기고 있어요`
  - `아직 기록을 시작하지 않은 팀원이 2명 있어요`
  - `먼저 확인이 필요한 팀원이 있습니다`

### 5.6. MVP 제외 범위

- 장문 이메일 리포트
- AI 자동 해석
- 팀원 개별 사유 분석
- 비교 랭킹/서열화

## 6. 기능 2: 팀원별 선행지표 리마인드 (Leader Reminder)

### 6.1. 해결하려는 문제

- 리더가 팀 대시보드에서 문제를 발견해도 바로 개입할 수단이 부족하다.
- 현재 푸시는 개인 자기관리용이며, 리더의 코칭 액션과 연결되지 않는다.
- 메모만으로는 즉시 행동 유도까지 이어지지 않는다.

### 6.2. 사용자 가치

- 리더는 특정 팀원에게 특정 선행지표를 기준으로 바로 체크인을 보낼 수 있다.
- 팀원은 `무엇을 다시 봐야 하는지`가 분명한 리마인드를 받는다.
- 리더의 개입이 추상적인 관리가 아니라 구체적 액션으로 남는다.
- 이 기능은 `STANDARD`의 대표 메리트로 보여야 한다.

### 6.3. MVP 범위

- 플랜 게이트:
  - 출시 v1 기준 `STANDARD` 플랜 전용
- 권한:
  - 워크스페이스 ADMIN만 발송 가능
- 발송 시작점:
  - 팀 대시보드 멤버 컨텍스트
  - 향후 주간 리포트 딥링크에서 바로 진입 가능
- 발송 대상:
  - 같은 워크스페이스의 특정 팀원 1명
- 발송 단위:
  - 해당 팀원의 특정 선행지표 1개
- 메시지:
  - 기본 템플릿 2~3개 중 선택
  - 짧은 자유 입력 한 줄 허용 여부는 MVP에서 선택 사항
- 전달 채널:
  - 기존 웹 푸시 우선

플랜 메시지 원칙:

- Free 사용자가 느껴야 하는 메시지:
  - `팀 상태는 볼 수 있지만, 팀원을 바로 움직이는 개입 도구는 STANDARD에서 제공된다`
- Standard 사용자가 느껴야 하는 메시지:
  - `문제를 보는 데서 끝나지 않고, 바로 체크인하고 이후 변화를 확인할 수 있다`

### 6.3.1. 스케줄 커스터마이즈 연계

- 현재 개인 기록 알림은 매일 밤 9시, 주간 집중 알림은 매주 목요일 오후 3시 고정 정책이다.
- 리더용 리포트/리마인드를 도입하면서, 기존 알림 정책도 장기적으로는 커스터마이즈 가능한 구조로 재설계해야 한다.
- 최소 요구:
  - 리더 주간 리포트는 요일 선택 가능
  - 개인 기록 리마인드는 향후 시간 커스터마이즈 확장 여지를 남기는 저장 구조여야 함

### 6.4. 리마인드 발송 플로우

1. 리더가 팀원 선택
2. 해당 팀원의 활성 선행지표 목록 확인
3. 리마인드 보낼 선행지표 선택
4. 템플릿 선택 또는 짧은 메시지 작성
5. 발송
6. 발송 성공 시 이력 저장

### 6.5. 기본 템플릿 예시 방향

- `이번 주 {선행지표명} 체크해보세요`
- `{선행지표명} 진행 상황 한번 확인 부탁드려요`
- `이번 주 목표 흐름을 위해 {선행지표명} 다시 봐주세요`

### 6.6. 남용 방지 규칙

- 동일 리더가 동일 팀원/동일 지표에 너무 자주 보내지 못하게 제한
  - 예: 하루 1회 또는 주 2회
- 발송 이력을 리더가 확인 가능해야 함
- 발송 이후 후속 기록 재개 여부를 확인 가능해야 함
- 팀원은 왜 이 알림을 받았는지 이해할 수 있어야 함

### 6.7. MVP 제외 범위

- 대량 일괄 발송
- 자동 발송 추천
- 감정 점수/압박 점수화
- 공개 피드백/공개 지적 기능
- Free 플랜 제공

## 7. 도메인 영향 범위 (Domain Impact)

- dashboard
  - 팀 대시보드에서 리더 개입 시작점 제공
  - 리포트 딥링크 도착지 정리
- notification
  - 리더용 주간 리포트 스케줄 발송
  - 리더가 발송하는 팀원 리마인드 푸시 추가
  - 현재 고정 시각 알림 정책을 사용자 설정형으로 확장할 수 있는 구조 검토
- workspace
  - ADMIN 권한 기준 발송/설정 권한 적용
- analytics
  - 리포트 발행률, 리마인드 발송률, 딥링크 클릭률, 후속 기록률 측정
- profile/settings
  - 워크스페이스 리더 리포트 설정 진입점 필요

## 8. 데이터 및 API 초안 (Draft)

### 8.1. 주간 리포트 설정

- 설정 항목 예시:
  - `enabled`
  - `sendDayOfWeek`
  - `sendTime`
  - `workspaceId`

추가 규칙:

- 리더 리포트 설정은 워크스페이스 단위로 1개만 가진다.
- ADMIN이 여러 명이면 동일한 설정을 공유하고, 웹 푸시를 구독한 ADMIN 모두에게 발송한다.
- 선행 조건:
  - `push_subscriptions.user_id`는 `users.id` FK 기준으로 정리돼 있어야 한다
  - MVP에서는 `timezone`을 `Asia/Seoul` 기본값으로 저장하되 외부 입력으로는 아직 열지 않는다

예상 계약:

- MVP에서는 기존 체계를 유지해 `/api/push/*` 하위로 확장한다.
- 이유:
  - 현재 개인 구독/설정/cron 경로가 이미 `/api/push/*`에 모여 있다.
  - 이번 단계의 목표는 도메인 재명명보다 기능 구현 착수다.
  - `/api/notifications/*` 승격은 채널 확장이나 inbox 도입 시점에 다시 검토한다.
- 예시:
  - `GET /api/push/workspace-settings`
  - `PUT /api/push/workspace-settings`

### 8.2. 리더 리마인드 발송

요청 예시:

```json
{
  "targetUserId": 12,
  "leadMeasureId": 34,
  "templateKey": "check_progress",
  "customMessage": "오늘 이 지표 한번 체크해주세요"
}
```

예상 계약:

- MVP에서는 `/api/push/*` 하위로 확장한다.
- 계약 책임:
  - 리더가 팀원/선행지표 기준으로 1건 발송 가능
  - 발송 이력 조회 가능
  - ADMIN 권한 검증 포함
- 예시:
  - `POST /api/push/leader-reminders`
  - `GET /api/push/leader-reminders`

저장 모델 원칙:

- API 입력은 `targetUserId + leadMeasureId` 형태를 유지할 수 있다
- 다만 persisted row는 `target_user_id`와 `workspace_id`를 중복 저장하지 않고 `scoreboard_id + lead_measure_id` 기준으로 남긴다
- 이유:
  - 대상 사용자와 워크스페이스는 활성 점수판에서 유도 가능하다
  - 중복 컬럼을 줄여 발송 이력과 실제 소유 관계가 어긋날 가능성을 줄인다
- 서비스 필수 검증:
  - 발송자가 해당 워크스페이스 ADMIN인지 확인
  - `lead_measure_id`가 `scoreboard_id` 소속인지 확인
  - `scoreboard_id` 상태가 `ACTIVE`인지 확인

### 8.3. 내부 발송 크론

- 내부 크론 전용 엔드포인트 필요
- 각 워크스페이스 설정에 맞는 주간 리포트 발송
- 발송 시점에는 해당 워크스페이스의 ADMIN 목록을 조회해 웹 푸시 구독자에게 전달
- 네임스페이스는 기존 `/api/push/*`를 유지한다
- 예시:
  - `GET /api/push/send-leader-weekly-report`

### 8.4. 이벤트 로그 전제

- 현재 제품의 주요 계측은 클라이언트 `gtag` 이벤트 중심이다.
- 하지만 이 기능의 핵심 지표는 서버 발송과 후속 행동 연결이 필요하므로, 최소한 아래 이벤트를 별도 정의해야 한다.
  - 리더 리포트 발행
  - 리더 리포트 클릭
  - 리더 리마인드 발송
  - 리마인드 대상자의 후속 기록 재개
- 이 이벤트들은 단순 분석이 아니라 `STANDARD가 팀을 더 움직이게 하는가`를 증명하는 핵심 근거다.
- 구현 전 analytics/notification 경계에서 이벤트 저장 주체를 명시해야 한다.

- 주간 리포트 발송 성공률

- 리더 리마인드 발송률

## 10. 리스크 및 가드레일 (Risks & Guardrails)

### 10.1. 감시 도구처럼 보일 위험

- 리포트 문구는 평가/통제 톤을 피한다.
- 팀원 랭킹이나 공개 비교는 MVP에서 제외한다.
- 리마인드는 공개가 아니라 개인 푸시로 보낸다.

### 10.2. 알림 피로

- 주간 리포트는 주 1회만 허용한다.
- 리더 리마인드는 동일 대상/지표 기준 빈도 제한을 둔다.

### 10.3. 잘못된 개입 유도

- 리더가 개인을 압박하기보다, 특정 선행지표를 기준으로 체크인하도록 UI와 카피를 설계한다.
- 사람 자체가 아니라 `이번 주 중요한 행동`에 초점을 맞춘다.

## 11. 권장 실행 순서 (Recommended Sequence)

## 13. MVP / Post-MVP 경계 (Boundary)

- 워크스페이스당 리더 리포트 설정 1개
- 발행 요일 선택 1개
- 리더 리포트 시간 또는 기본 시간대 정책 명시
- 팀원 1명 + 선행지표 1개 기준 리마인드 발송
- 기본 템플릿 기반 푸시
- 최소 발송 이력
- 기존 개인 알림 스케줄 커스터마이즈를 위한 저장 구조 초안

### 13.2. Post-MVP

- 다중 채널 발송
- 다중 요일 발송
- 개인/리더 알림 시간 완전 커스터마이즈
- 자동 추천 리마인드
- AI 요약/코칭
- 리더용 우선순위 보드 자동 생성

## 14. 기대 효과 (Expected Impact)

- WIG의 리더 가치가 `보는 대시보드`에서 `개입하는 운영 도구`로 확장된다.
- 회의 문화가 강하지 않은 팀에서도 주간 운영 습관을 만들 수 있다.
- 무료와 유료의 차이가 `정보 제공`과 `행동 유도`의 차이로 더 선명해진다.
- 이후 유료화 논의도 `관리 편의`가 아니라 `리더의 파악 속도`, `개입 효율`, `팀 실행 지속성` 같은 더 명확한 가치로 연결된다.

## 15. 성공 및 실패 판단 기준 (Success & Failure Criteria)

### 성공

- 웹 푸시를 구독한 ADMIN에게 주 1회 리더 리포트가 정상 발송된다.
- 리더가 앱에 직접 들어오기 전부터 이번 주 관리 포인트를 받는다.
- 리포트에서 팀 대시보드 딥링크 클릭이 실제로 발생한다.
- 리더가 특정 팀원/선행지표에 즉시 개입할 수 있다.
- 리마인드 이후 팀원의 실제 기록 재개가 관찰된다.

### 실패

- 리포트가 길고 추상적이어서 실제 개입으로 이어지지 않는다.
- 웹 푸시 미구독 ADMIN 비중이 높아 리포트 기능의 도달률 자체가 너무 낮다.
- 리마인드가 감시성 알림으로 받아들여져 팀원 반감만 만든다.
- 발송 후 기록 재개나 운영 개선 신호가 거의 없다.
