# 문서 웹 뷰어 설계안

## 1. 배경 및 문제

- 현재 `docs/` 하위 문서 수가 많아졌고, VS Code 탐색기 + Markdown 미리보기만으로는 문서 간 관계와 우선순위를 빠르게 파악하기 어렵다.
- 특히 `docs/planning`은 파일 수가 많고, 문서명이 날짜 기반이라 "무슨 주제인지", "지금 봐야 하는지", "관련 문서가 무엇인지"를 한 번에 읽기 어렵다.
- 목적은 외부 공개용 문서 사이트가 아니라, 개발자가 `기획 -> 구현 -> 검토` 흐름에서 문서를 더 빨리 찾고 비교하도록 돕는 것이다.

## 2. 왜 지금 필요한가

- 현재 온보딩 문서가 안내 역할은 하고 있지만, 실제 탐색은 여전히 폴더와 파일명 기억에 의존한다.
- planning 문서가 누적되면서 "최신 의사결정", "관련 후속 문서", "이미 구현된 문서"를 분간하는 비용이 커졌다.
- 따라서 첫 버전은 범용 문서 포털보다 `planning 문서 읽기 비용 절감`에 집중하는 편이 맞다.

## 3. 목표

### 3.1. 핵심 목표

- `docs/planning` 문서를 한 화면에서 훑을 수 있게 한다.
- 문서를 "파일명"이 아니라 "주제, 상태, 연결 관계" 기준으로 읽게 만든다.
- 내부 도구 성격을 유지하되, 별도 로컬 전용 가드 없이 빠르게 붙일 수 있어야 한다.

### 3.2. 비목표

- 외부 사용자 공개 문서 사이트
- 완전한 검색 엔진 구축
- MDX 기반 인터랙티브 문서 플랫폼
- 모든 문서를 처음부터 완벽하게 구조화하는 대규모 메타데이터 정비

## 4. 사용자와 핵심 사용 장면

### 4.1. 주요 사용자

- Dowin 저장소에 들어온 개발자
- 문서 먼저 읽고 작업 범위를 정리해야 하는 AI 에이전트
- 기획/문서 정합성을 빠르게 확인하려는 운영자

### 4.2. 가장 중요한 사용 장면

1. `planning` 문서만 빠르게 훑고 지금 살아 있는 기획 스레드를 찾는다.
2. 특정 주제의 planning 문서와 관련 dev 문서를 이어서 읽는다.
3. 같은 주제의 오래된 기획과 최신 기획을 비교한다.
4. 온보딩에서 언급된 핵심 문서를 UI에서 바로 연다.

## 5. 제품 원칙

- 내부 문서 도구답게 빠르고 가벼워야 한다.
- 첫 화면은 `planning` 중심이어야 한다. 전체 문서 균등 노출은 오히려 약점이다.
- 폴더 구조를 그대로 복제하지 말고, 사람이 읽는 단위로 재구성해야 한다.
- 문서 본문보다 먼저 "이 문서가 왜 중요한지"를 보여줘야 한다.
- 파일명 날짜는 유지하되, 제목과 상태 해석을 보조 UI가 대신해줘야 한다.

## 6. 제안 구조

### 6.1. 권장 접근 방식

- 기존 Next.js 앱 안에 일반 라우트로 붙인다.
- 경로 예시: `/docs-viewer`
- 문서 소스는 기존 `docs/**` 파일 시스템을 그대로 사용한다.

이 방향이 맞는 이유는 다음과 같다.

- 새로운 별도 앱보다 유지비가 낮다.
- 기존 UI 토큰과 레이아웃 패턴을 재사용할 수 있다.
- `api-docs`처럼 별도 접근 제한 없이도 일관된 내부 도구 경험을 만들 수 있다.

### 6.2. 정보구조

상위 탭은 아래 5개로 제한한다.

- `Planning`
- `Dev`
- `Operations`
- `Reference`
- `All Docs`

첫 진입 기본 탭은 반드시 `Planning`이다.

### 6.3. Planning 탭 내부 구조

`Planning` 탭은 단순 파일 리스트가 아니라 아래 4개 레이어로 보인다.

1. `Overview`
- planning 문서 총 개수
- 최근 14일 문서
- 최근 업데이트 주제
- 아직 구현 후속이 없는 문서 수

2. `Topic Lanes`
- `온보딩/진입`
- `대시보드/리포트`
- `수익화/플랜`
- `운영/관리도구`
- `알림/리마인드`
- `정책/법률`
- `기타`

3. `Status Lenses`
- `지금 읽을 것`
- `최근 업데이트`
- `핵심 기준 문서`
- `히스토리 참고`

4. `Document Feed`
- 선택한 토픽/상태 기준으로 문서 카드 리스트 노출

핵심은 폴더명보다 `토픽 + 상태` 조합이 먼저 보여야 한다는 점이다.

## 7. 화면 설계

### 7.1. 전체 레이아웃

권장 레이아웃은 데스크톱 기준 3패널이다.

- 좌측: 문서 렌즈와 토픽 네비게이션
- 중앙: 문서 카드 리스트 또는 비교 리스트
- 우측: 선택 문서 메타 + TOC + 관련 문서

모바일/좁은 폭에서는 1패널 스택으로 바꾼다.

- 상단 sticky 필터 바
- 문서 리스트
- 문서 상세

### 7.2. 좌측 패널

좌측은 "어디 폴더를 볼지"보다 "어떤 관점으로 볼지"를 담당한다.

구성:

- 탭 스위처: Planning / Dev / Operations / Reference / All Docs
- Planning quick filters:
  - `최근 7일`
  - `핵심 기준`
  - `미구현 후속`
  - `문서 많은 주제`
- 토픽 목록
- 즐겨찾기 문서 pin

### 7.3. 중앙 패널

중앙은 리스트가 아니라 카드형 문서 피드가 기본이다.

카드에는 최소한 아래 정보가 보여야 한다.

- 문서 제목
- 파일 경로 일부
- 작성일
- 1줄 요약
- 토픽
- 상태 배지
- 관련 문서 수

카드 액션:

- `바로 보기`
- `나란히 비교`
- `관련 문서 보기`

파일명만 보이는 테이블형 리스트는 보조 뷰로만 둔다.

### 7.4. 우측 패널

우측은 문서 본문 전체보다 "읽기 전에 필요한 맥락"을 먼저 준다.

순서:

1. 문서 메타
- 제목
- 경로
- 작성일
- 추정 토픽

2. 읽기 가이드
- 이 문서를 언제 읽는지
- 먼저 읽을 문서
- 다음에 읽을 문서

3. 본문 TOC

4. 관련 문서 그래프
- 같은 키워드
- 같은 planning 스레드
- 연결된 dev 문서

### 7.5. 문서 상세 화면

상세는 단순 Markdown 렌더링으로 끝내지 않는다.

본문 상단에 다음 보조 블록을 둔다.

- `왜 이 문서를 봐야 하는가`
- `현재성 판단`
- `관련 구현/후속 문서`

이 블록은 frontmatter + 파일명 + 경로 규칙 기반으로 자동 생성하고, 필요한 경우 수동 메타데이터로 보강한다.

## 8. Planning 문서를 한눈에 보이게 하는 UX 핵심

### 8.1. 첫 화면을 Planning Dashboard로 고정

첫 진입부터 `docs/planning`만 대상으로 한 대시보드를 보여준다.

표시 항목:

- 최근 planning 문서 8개
- 가장 많이 연결된 planning 문서
- 온보딩에서 직접 참조되는 planning 문서
- 최근 30일 내 새로 생긴 planning 주제

### 8.2. 날짜 대신 주제 제목을 전면 배치

현재 파일명은 날짜가 앞에 와서 인지 비용이 높다.

따라서 카드 제목 규칙은 아래처럼 간다.

- 1행: 문서 제목
- 2행: 날짜 + 경로 + 파일명

즉, 날짜는 메타고 주제는 헤드라인이어야 한다.

### 8.3. 상태 배지 도입

초기에는 자동 추론 기반의 가벼운 상태 배지만 붙인다.

- `기준 문서`
- `최근 문서`
- `후속 연결 있음`
- `히스토리`

판정 기준 예시:

- `기준 문서`: 온보딩 또는 다른 문서에서 반복 참조됨
- `최근 문서`: 최근 14일 작성
- `후속 연결 있음`: 본문 내 관련 문서 링크가 일정 수 이상
- `히스토리`: 오래됐고 최신 문서가 이를 대체하는 링크가 있음

### 8.4. 관련 문서 묶음 보기

planning에서 가장 불편한 지점은 "이 문서 다음에 뭘 봐야 하지?"다.

따라서 각 문서에서 아래 묶음을 항상 보인다.

- `같이 읽기`
- `이후 결정`
- `구현 반영 문서`

이 3개 블록만 잘 보여도 VS Code보다 훨씬 빨라진다.

### 8.5. 나란히 비교 모드

planning 문서는 이전안과 최신안을 비교해야 할 때가 많다.

그래서 비교 모드는 MVP부터 넣는 편이 좋다.

- 좌측 문서 선택
- 우측 문서 선택
- 제목, 메타, 섹션 단위 비교

처음부터 diff 엔진까지 갈 필요는 없다.
두 문서를 나란히 읽는 것만으로도 가치가 크다.

## 9. 탐색 방식

### 9.1. 기본 탐색

- 토픽 선택
- 상태 렌즈 선택
- 리스트 훑기
- 카드 클릭

### 9.2. 검색

검색은 전체 텍스트보다 아래 우선순위가 맞다.

1. 제목
2. 파일명
3. 경로
4. 본문 헤더
5. 본문 전체

이유는 planning 탐색의 대부분이 "정확한 본문 내용 검색"보다 "찾던 문서 재발견"이기 때문이다.

### 9.3. 점프 액션

상단 command bar를 둔다.

- 문서 제목 검색
- `planning`만 보기
- 최근 문서로 점프
- 온보딩 핵심 문서로 점프

## 10. 메타데이터 전략

### 10.1. MVP

기존 문서를 대규모 수정하지 않고 아래 정보만 자동 추출한다.

- 경로
- 파일명
- frontmatter
- 첫 번째 H1
- 첫 문단 요약
- 문서 내 링크

### 10.2. Post-MVP

필요한 planning 문서부터 선택적으로 메타데이터를 추가한다.

예시:

- `category`
- `status`
- `related`
- `supersedes`
- `implemented_by`

이 단계는 문서 UX 가치가 검증된 뒤에 가는 편이 맞다.

## 11. MVP 범위

### 11.1. 포함

- `/local/docs` 로컬 전용 라우트
- `Planning` 기본 대시보드
- 토픽 필터
- 상태 렌즈
- 문서 카드 리스트
- Markdown 상세 보기
- TOC
- 관련 문서 자동 추출
- 2문서 나란히 보기

### 11.2. 제외

- 편집 기능
- 실시간 파일 감시 기반 고급 인덱싱
- 태그 수동 편집 UI
- 공개 배포
- 문서별 코멘트/주석 기능

## 12. Post-MVP

- Dev 문서와 planning 문서 간 관계 시각화
- 히스토리 타임라인 보기
- "온보딩 추천 순서" 자동 플레이리스트
- 본문 내 링크 hover preview
- 구현 파일 연결 보기
- 즐겨찾기/최근 본 문서 저장

## 13. 기술 방향 제안

### 13.1. 라우팅

- Next App Router 내부 일반 페이지
- 예시 경로:
  - `/docs-viewer`
  - `/docs-viewer/[section]`
  - `/docs-viewer/view?path=...`
  - `/docs-viewer/compare?a=...&b=...`

### 13.2. 데이터 수집

- 서버 컴포넌트 또는 서버 유틸에서 `docs/**` 파일 스캔
- Markdown 파싱
- 헤더/링크/요약 추출
- 간단한 메모리 인덱스 생성

### 13.3. 렌더링

- 문서 리스트: 서버에서 인덱스 생성 후 전달
- 상세 보기: 선택 문서 Markdown 렌더링
- 비교 보기: 두 문서 본문 동시 렌더링

### 13.4. 접근 제한

- 별도 접근 제한은 두지 않는다.
- 다만 사용자-facing 서비스 탐색과 섞이지 않도록 메인 내비게이션에서는 숨기고, 직접 URL 진입 또는 개발자용 진입점으로만 노출하는 편이 맞다.

## 14. 화면별 추천 카피

### 14.1. Planning 홈

- 제목: `Planning 문서 한눈에 보기`
- 설명: `지금 읽을 기획, 최근 바뀐 기준, 구현으로 이어지는 문서를 빠르게 찾습니다.`

### 14.2. 문서 상세 보조 영역

- `이 문서는 이런 상황에서 먼저 봅니다`
- `함께 읽으면 좋은 문서`
- `이후 결정 또는 구현 문서`

### 14.3. 빈 상태

- `조건에 맞는 planning 문서가 없습니다`
- `필터를 줄이거나 전체 planning 문서를 확인하세요`

## 15. 우선순위 제안

1. `Planning` 탭 단독 MVP부터 만든다.
2. 토픽 분류와 관련 문서 묶음이 실제로 유용한지 먼저 본다.
3. 이후 `Dev`와 `Operations`를 같은 프레임으로 확장한다.

범용 문서 포털을 먼저 만들면 구조는 커지지만 핵심 문제인 planning 가독성은 늦게 해결된다.
이번 요청 기준으로는 그 순서가 비효율적이다.

## 16. 실행 액션 아이템

- [ ] `docs/**` 스캔 기준과 인덱싱 필드 확정
- [ ] `planning` 토픽 분류 규칙 정의
- [ ] `/docs-viewer` IA wireframe 작성
- [ ] 문서 카드 정보 구조 시안 작성
- [ ] Markdown 렌더러 및 TOC 추출 방식 결정
- [ ] 개발자 진입 경로 방식 결정
- [ ] MVP 구현 후 실제 `docs/planning` 문서로 탐색 테스트 수행

## 17. 성공 기준

- planning 문서 찾는 시간이 VS Code 탐색 대비 줄어든다.
- 온보딩에서 자주 언급하는 planning 문서를 UI에서 2클릭 이내로 열 수 있다.
- 특정 주제의 old/new planning 문서를 비교하는 흐름이 자연스럽다.
