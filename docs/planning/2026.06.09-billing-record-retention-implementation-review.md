# Basic seat 결제 구매 기록 보존 구현 검토

## 1. 검토 기준

기준 문서:

- `docs/planning/2026.06.09-billing-record-retention-research.md`

검토 대상:

- DB schema: `src/db/schema.ts`
- Polar checkout/client: `src/domain/billing/polar.ts`
- Basic checkout: `src/domain/workspace/services/workspace-checkout.service.ts`, `src/domain/billing/services/billing.service.ts`
- Polar webhook: `src/app/api/webhooks/polar/route.ts`, `src/domain/billing/services/polar-webhook.service.ts`
- Billing storage/admin override: `src/domain/billing/storage/billing.storage.ts`, `src/domain/billing/services/admin-billing.service.ts`
- 계정/워크스페이스 삭제: `src/domain/profile/*`, `src/domain/workspace/*`
- 법률 문구: `src/content/legal-documents.ts`

리서치 문서상 MVP 필수 기준:

1. `billing_retention_records` 또는 동등한 보존 아카이브 결정
2. Polar webhook 저장 전 normalized/sanitized payment snapshot 정의
3. checkout 생성 시 terms/policy version, seat count, country/tax evidence 저장
4. 계정/워크스페이스 삭제 시 billing retention archive 분리 보존
5. 개인정보처리방침에 결제/세무 보존 항목과 기간 반영
6. 결제 화면에 가격, 주기, 자동갱신, 다음 결제일/계산 기준, 해지 방법, 환불 조건, 즉시 제공 여부 명시
7. 카드 원문/CVC/CVV가 DB, 로그, 에러 리포트에 저장되지 않는지 점검

## 2. 종합 판단

현재 구현은 **결제 권한 반영용 billing ledger**로는 동작하지만, 리서치 문서가 요구한 **법정 보존용 구매 기록 아카이브**로는 아직 부족하다.

부분 충족:

- Polar hosted checkout을 사용하므로 Dowin 서버가 카드번호/CVC/CVV를 직접 입력받는 흐름은 보이지 않는다.
- Basic checkout과 webhook은 `BASIC`, seat count, customer/subscription key, billing status, refund/revoke 일부 이력을 저장한다.
- active Polar subscription이 있는 workspace 삭제는 막는다.
- 개인정보처리방침에는 결제 식별자와 checkout 요청 이력 보유 가능성이 일반 문구로 들어가 있다.

주요 미충족:

- 10년 보존용 별도 archive 또는 `legal_retention_until` 필드가 없다.
- webhook 원문 payload를 그대로 `billing_events.payload_json`에 저장한다.
- checkout 시점의 약관/환불정책 버전, 자동갱신 동의 시각, country/tax evidence, billing email/name/address/tax ID snapshot이 없다.
- 계정 삭제는 `users` row를 hard delete하며, 일부 billing 관련 FK가 cascade 또는 set null로 처리되어 구매자 입증력이 약해질 수 있다.
- 개인정보처리방침의 보유기간은 `필요한 기간` 수준이고, 한국 5년/EU OSS 10년 같은 항목별 기준이 아직 반영되지 않았다.

## 4. 주요 발견 사항

### 4.1. 높음: webhook 원문 payload를 그대로 저장한다

근거:

- `src/app/api/webhooks/polar/route.ts`에서 request body 전체를 `payloadJson`으로 넘긴다.
- `src/domain/billing/services/polar-webhook.service.ts`는 `appendBillingEvent` 호출 시 `payloadJson: input.payloadJson`을 그대로 저장한다.
- `src/db/schema.ts`의 `billing_events.payload_json`은 일반 text 필드다.

리스크:

- 리서치 문서의 `sanitized payload snapshot` 원칙과 충돌한다.
- Polar webhook payload에 결제수단 상세, billing address, email, tax data, provider 내부 metadata가 포함될 경우 필요 이상의 개인정보가 원장에 장기 보관될 수 있다.
- CVC/CVV 저장 가능성은 Polar payload 정책에 따라 낮을 수 있지만, 현재 코드는 민감 키 제거 방어층이 없다.

필요 조치:

- `normalizePolarBillingEventPayload(payload)` 같은 정규화 계층을 만든다.
- allowlist 기반으로 `event id/type`, `customer id`, `subscription id`, `checkout/order/invoice/payment id`, 금액, 통화, 세금, seat count, period, refund/dispute status만 남긴다.
- denylist로 `card`, `payment_method`, `cvc`, `cvv`, `pan`, `number`, `billing_address` 중 불필요 상세, provider 내부 토큰을 제거한다.
- 원문 payload 저장이 꼭 필요하면 별도 암호화 저장소와 제한된 retention으로 분리해야 한다.

### 4.2. 높음: 법정 보존용 archive가 없다

근거:

- `src/db/schema.ts`에는 `billing_events`, `billing_checkout_events`, `workspace_billing_state`, `workspace_seat_entitlements`만 있다.
- `billing_events`와 `workspace_billing_state`는 `workspace_id -> workspaces.id` FK가 `onDelete: "cascade"`다.
- `workspace_seat_entitlements`도 `workspace_id -> workspaces.id` cascade다.
- 별도 `billing_retention_records`, `legal_retention_until`, `legal_hold`, `retention_expires_at` 필드는 없다.

리스크:

- 현재 billing table은 권한 projection/운영 원장 성격이지 세무/분쟁 대응용 장기 보존 archive가 아니다.
- 향후 hard delete, cleanup, migration, workspace purge가 들어가면 구매 기록도 같이 사라질 수 있다.
- 계정/워크스페이스 삭제 요청과 법정 보존을 분리한다는 리서치 문서 요구를 구현으로 보장하지 못한다.

필요 조치:

- `billing_retention_records`를 별도 테이블로 둔다.
- `workspace_id`, `user_id`는 nullable reference 또는 snapshot id로 두고, 삭제 이후에도 구매자/거래/세무 증빙이 유지되게 한다.
- 최소 필드:
  - `workspace_id_snapshot`, `billing_owner_user_id_snapshot`, `billing_email`, `billing_name`, `billing_country`, `tax_id`
  - `plan_code`, `seat_count`, `unit_price`, `currency`, `amount`, `tax_amount`
  - `provider`, `customer_key`, `subscription_key`, `checkout_id`, `order_id`, `invoice_id`, `payment_id`
  - `paid_at`, `refunded_at`, `canceled_at`, `current_period_start/end`
  - `terms_version`, `privacy_policy_version`, `billing_policy_version`
  - `legal_retention_until`, `legal_hold`, `created_at`

### 4.3. 높음: 계정 삭제가 결제 구매자 입증력을 약화시킬 수 있다

근거:

- `ProfileStorage.deleteUser`는 `users` row를 hard delete한다.
- `billing_checkout_events.requested_by_user_id`는 `users.id`에 `onDelete: "cascade"`로 연결되어 있다.
- `workspace_billing_state.billing_owner_user_id`는 `onDelete: "set null"`이다.
- 별도 billing owner snapshot은 없다.

리스크:

- 구매자가 계정을 삭제하면 checkout 요청 이력은 cascade 삭제될 수 있고, billing owner는 null이 된다.
- `누가 구매했는지`, `누가 자동갱신/환불/해지 요청을 했는지`를 나중에 입증하기 어렵다.
- 리서치 문서의 `계정 삭제 또는 개인정보 삭제 요청이 있어도 법정 보존 대상 구매기록은 운영 데이터와 분리 보관` 기준을 충족하지 못한다.

필요 조치:

- 계정 삭제 전에 billing retention archive에 구매자 snapshot을 남긴다.
- `billing_checkout_events.requested_by_user_id` cascade를 장기적으로 재검토한다. 최소한 archive는 FK cascade 밖에 있어야 한다.
- 사용자 삭제 후에도 필요한 billing email/name/company/tax data만 암호화/접근통제로 보존한다.

### 4.4. 중간: checkout 시점의 정책 버전과 동의 증거가 없다

근거:

- workspace setup checkout metadata는 `flow`, `workspaceCheckoutId`, `userId`, `targetPlanCode`, `requestedSeatCount`만 보낸다.
- 기존 workspace 재구독 checkout metadata도 `flow`, `workspaceId`, `workspaceUid`, `requestedByUserId`, `targetPlanCode`, `requestedSeatCount`만 보낸다.
- `pending_workspace_checkouts`에는 locale, workspace name, requested seat count, product id, checkout id/url/status만 저장된다.

리스크:

- 사용자가 어떤 약관/환불정책/자동갱신 고지를 보고 결제했는지 사후 입증하기 어렵다.
- 해외 판매 시 tax evidence, IP country evidence, billing country snapshot이 없다.

필요 조치:

- checkout 생성 시 내부 pending record 또는 metadata에 아래를 저장한다.
  - `termsVersion`, `privacyPolicyVersion`, `billingPolicyVersion`
  - `autoRenewalNoticeAcceptedAt`
  - `checkoutNoticeVersion`
  - `ipCountry`, `billingCountry`, `taxEvidenceSource`
  - `displayedPrice`, `currency`, `billingInterval`

### 4.5. 중간: invoice/payment/tax snapshot이 없다

근거:

- schema와 service에서 `invoice_id`, `receipt_url`, `payment_id`, `order_id`, `tax_amount`, `tax_rate`, `tax_jurisdiction`, `credit_note` 저장 필드를 찾지 못했다.
- webhook projection은 subscription/customer/status/period/seat/refund revoke 중심이다.

리스크:

- 정산/세무/환불/chargeback 증빙을 provider dashboard 접근에 과도하게 의존하게 된다.
- EU/영국/캐나다/미국 판매 시 거래별 세금 계산 근거를 자체 export하기 어렵다.

필요 조치:

- Polar webhook 중 order/invoice/payment/refund 관련 이벤트에서 보존할 normalized snapshot을 정의한다.
- provider가 제공하는 invoice/receipt URL 또는 file key를 저장하되, 접근 권한과 만료 가능성을 고려해 최소 식별자와 금액/세율 snapshot을 자체 보존한다.

### 4.6. 중간: 법률 문구가 현재 리서치 기준보다 느슨하다

근거:

- `src/content/legal-documents.ts`는 결제 식별자, 결제 이벤트, checkout 요청 이력을 `정산, 환불, 분쟁 대응, 부정 결제 확인에 필요한 기간` 동안 보유할 수 있다고만 설명한다.
- 한국어/영어 개인정보처리방침 모두 결제/세무 보존의 구체 기간이 없다.
- 문의 기록은 12개월/3년으로 명시되어 있으나, 계약/대금결제/세무/OSS 10년 기준은 반영되지 않았다.

리스크:

- 실제 보존 정책을 10년으로 설계하더라도 사용자 고지와 불일치한다.
- 한국 판매만 해도 전자상거래법/세법상 항목별 보존기간과 개인정보 분리보관 문구가 더 구체적이어야 한다.

필요 조치:

- 개인정보처리방침에 `계약/청약철회/대금결제/공급/소비자 분쟁/세무 증빙` 항목별 기간을 추가한다.
- 해외 판매 전 EU/UK/Canada/US 보존기간과 merchant of record 책임 범위를 법무/세무 검토 후 반영한다.

### 4.7. 낮음: 결제수단 원문 미수신 구조는 방향이 맞다

근거:

- `createPolarBillingClient.createCheckoutSession`은 Polar `/checkouts` API로 hosted checkout URL을 생성한다.
- Dowin checkout 입력값은 product, external customer id, seats, success/return URL, metadata 중심이다.
- 현재 Dowin API에서 카드번호/CVC/CVV를 받는 경로는 확인되지 않았다.

남은 확인:

- Polar API 에러 body를 `console.error`에 남기는 경로가 있다. 현재 1,000자로 truncate하지만, provider error body에 민감값이 들어올 가능성은 낮더라도 로그 정규화가 더 안전하다.
- webhook payload sanitizer를 만들 때 로그 sanitizer와 같은 키 정책을 공유하는 편이 낫다.

## 5. 권장 구현 순서

### Phase 0. 즉시 방어

1. Polar webhook 저장 전에 sanitizer를 적용한다.
2. `billing_events.payload_json`에 원문이 아니라 normalized snapshot만 저장하도록 테스트를 추가한다.
3. `card`, `cvc`, `cvv`, `pan`, `number`, `payment_method_details`, `billing_address` 같은 민감/과다 키가 저장되지 않는 회귀 테스트를 만든다.
4. Polar API error log도 같은 sanitizer를 통과시킨다.

### Phase 1. 보존 archive

1. `billing_retention_records` 스키마를 설계한다.
2. checkout completion, subscription active/update/canceled/ended/revoked, order refunded 이벤트에서 archive upsert/append 정책을 정한다.
3. `legal_retention_until = occurred_at + 10 years`를 기본값으로 둔다.
4. `legal_hold` flag를 추가해 dispute/tax/fraud review 중 자동 삭제를 막는다.
5. workspace/user 삭제와 무관하게 archive row가 유지되도록 FK 전략을 분리한다.

### Phase 2. checkout evidence

1. 정책 버전 상수를 둔다.
   - `TERMS_VERSION`
   - `PRIVACY_POLICY_VERSION`
   - `BILLING_POLICY_VERSION`
   - `CHECKOUT_NOTICE_VERSION`
2. checkout 생성 요청 시 seat count, displayed price/currency/interval, locale, country evidence, policy versions를 pending record에 저장한다.
3. 결제 완료 후 archive에 checkout evidence를 복사한다.

### Phase 3. 법률 문구 정렬

1. `src/content/legal-documents.ts`의 개인정보처리방침 보유기간을 항목별로 구체화한다.
2. 결제 및 환불 정책에는 Basic 즉시 제공, 월 구독, 자동갱신, 해지 경로, 환불 예외, 반복 남용 수동 검토를 현재 checkout UI와 일치시킨다.
3. 해외 판매 전 merchant of record, VAT/OSS, sales tax/GST/HST 책임 범위를 확정한다.

## 6. MVP 수용 기준

MVP 판매 전 최소 수용 기준:

- [ ] webhook payload 저장은 sanitizer allowlist를 통과한다.
- [ ] 카드 원문/CVC/CVV/track data/PIN 관련 키가 DB와 로그에 저장되지 않는 테스트가 있다.
- [ ] `billing_retention_records` 또는 동등 archive에 checkout/order/subscription/refund/tax snapshot이 저장된다.
- [ ] archive row는 user/workspace 삭제와 독립적으로 `legal_retention_until` 전까지 유지된다.
- [ ] 계정 삭제/워크스페이스 삭제 플로우가 billing archive 분리보관을 명시적으로 호출하거나 보장한다.
- [ ] 개인정보처리방침에 결제/세무/분쟁 보존 항목과 기간이 반영된다.
- [ ] checkout 화면과 metadata/archive가 같은 policy version을 사용한다.

## 7. 결론

현재 구현은 Basic entitlement, seat count, refund/revoke risk review 같은 제품 권한 운영에는 꽤 가까워졌다. 그러나 리서치 문서의 핵심인 `구매 기록 보존` 관점에서는 아직 append-only billing event 저장을 법정 보존 아카이브로 대체해 사용할 수 없다.

가장 먼저 고쳐야 할 것은 webhook 원문 저장이다. 그 다음으로 별도 retention archive와 삭제 분리보관을 붙여야 한국 판매와 해외 확장 모두에서 나중에 다시 갈아엎지 않는다.
