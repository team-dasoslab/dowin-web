# `dashboard/my` 기간·지표 선택형 Export 확장안

## 1. 배경 및 맥락 (Background & Context)

- 사용자가 대시보드에서 보는 기간별 지표를 회고나 회의 자료로 저장하려면 지금은 수작업 복사가 필요하다.
- `dashboard/my`가 기간 탐색을 지원하게 되면, 선택한 기간의 데이터를 외부에서 재사용할 수 있어야 사용자 가치가 완성된다.
- 다만 Cloudflare Workers Free 플랜 제약상 서버에서 직접 `xlsx`를 생성하는 방식은 CPU time 측면에서 위험하다.

## 2. 우선순위 및 당위성 (Priority & Rationale)

- export는 단순 다운로드 기능이 아니라, 사용자가 대시보드의 기간 탐색 결과를 실제 회고 자료로 전환하는 다리 역할을 한다.
- 기간 탐색이 도입된 이후에는 선택 결과를 밖으로 꺼내는 기능이 자연스러운 다음 단계다.
- 서버 비용과 복잡도를 낮추기 위해 파일 생성 책임은 프론트에 두는 구조가 MVP에 더 적합하다.

## 3. MVP 방향 (MVP Direction)

- 서버는 파일을 만들지 않고, 선택 조건에 맞는 원본/집계 데이터를 반환한다.
- 프론트엔드는 서버 응답을 기반으로 `csv` 또는 `xlsx` 파일을 생성해 다운로드한다.
- export 대상은 `dashboard/my`에서 사용자가 직접 선택한 선행지표와 기간(`from`, `to`)으로 한정한다.
- 팀 일괄 export는 개인정보 및 권한 정책 확정 전까지 제외한다.

## 4. 입력 UX 원칙 (Input UX)

- 기간 입력은 `from` / `to` 날짜를 필수로 받는다.
- `from <= to` 검증이 필요하다.
- 기간 길이는 최대 92일(약 3개월)까지만 허용한다.
- 지표 입력은 `전체 선행지표` 또는 `선택 지표`(복수 선택)를 지원한다.
- 기간 기본값은 현재 보기 단위 기준으로 자동 채운다. 주간이면 해당 주, 월간이면 해당 월을 사용한다.

## 5. 데이터 반환 원칙 (Server Response Principles)

- 화면 장식용 문자열이 아니라 파일 재가공 가능한 구조화 데이터를 제공한다.
- 결과에는 최소 `periodMeta`, `summary`, `leadMeasureBreakdown`, `dailyRows`를 포함한다.
- 과도한 대용량 응답을 피하기 위해 최대 기간 92일 제한을 둔다.
- 필요하면 후속 단계에서 페이지네이션 또는 분할 다운로드 전략을 검토한다.

## 6. 파일 생성 원칙 (Frontend File Generation)

- 기본 포맷은 `csv`를 우선한다.
- `xlsx`는 후속 옵션으로 둔다.
- 파일명은 `dowin-my-export-YYYYMMDD-YYYYMMDD.csv` 패턴처럼 검색 조건이 드러나야 한다.
- 생성 실패 시 재시도와 토스트 에러를 제공하고, API 재호출 없이 동일 payload로 재생성 가능해야 한다.

## 7. MVP 제외 범위 (Non-Goals)

- 팀 단위 일괄 export
- 서버 측 `xlsx` 직접 생성
- 무제한 기간 다운로드
- 대용량 비동기 export 잡 처리

## 8. 성공 및 실패 기준 (Success & Failure Criteria)

### 성공

- 사용자가 현재 대시보드 보기와 일치하는 기간 데이터를 바로 다운로드할 수 있다.
- 서버와 프론트의 책임이 단순하게 분리된다.
- Free 플랜 환경에서도 무리 없는 비용 구조를 유지한다.

### 실패

- 다운로드 결과와 화면에 보이는 기간/지표가 어긋난다.
- 서버에서 파일 생성을 맡아 런타임 부담이 커진다.
- 응답 구조가 화면 전용이라 재가공 가능한 export 데이터로 쓰기 어렵다.
