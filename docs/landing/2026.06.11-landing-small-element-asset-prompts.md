# Dowin 랜딩 작은 요소 에셋 프롬프트

## 1. 문서의 역할

이 문서는 랜딩 섹션 전체를 덮는 대표 이미지가 아니라, 레퍼런스처럼 화면 곳곳에 배치할 수 있는 작은 시각 요소를 만들기 위한 프롬프트 모음이다.

대상은 아래 같은 에셋이다.

- floating 3D 오브젝트
- 말풍선, 칩, 토큰, 배지
- 미니 UI 카드
- 점수판 셀, 체크 셀, 진행률 조각
- 추상 아이콘형 product proof
- 섹션 배경에 흩어 놓을 작은 그래픽 요소

기존 문서 `docs/landing/2026.06.11-landing-asset-image-prompts.md`가 섹션 대표 이미지 중심이라면, 이 문서는 배치 가능한 소형 오브젝트 중심이다.

## 2. 최종 산출물 규칙

담당자는 이 문서만 보고 아래 산출물을 만든다.

### 2.1. 저장 위치

최종 선택된 에셋은 모두 아래 경로에 저장한다.

```text
public/assets/landing/
```

소형 에셋은 기존 대표 이미지와 구분하기 위해 파일명에 `element-` prefix를 붙인다.

예:

```text
public/assets/landing/element-square-check-cell.png
public/assets/landing/element-mini-scoreboard-card.png
public/assets/landing/element-action-item-chip-stack.png
```

### 2.2. 파일 규칙

- 최종 파일 형식: `png`
- 배경: pure white background 우선
- 권장 크기: 1024x1024
- 가로형 카드/말풍선만 1536x1024 또는 1200x800 허용
- 파일명은 kebab-case로 쓴다.
- 최종 적용 후보만 `public/assets/landing/`에 저장한다.
- 탈락안, 원본 후보, 실험 이미지는 `public/assets/landing/`에 넣지 않는다.

### 2.3. 최소 최종 산출물

1차 완료 기준은 아래 9개 파일이다.


## 3. 참고 레퍼런스에서 가져올 점

### 3.1. `temp/c.jpg`

- 큰 대표 이미지보다 말풍선, 아바타, 작은 일러스트, 카드 썸네일이 섹션 리듬을 만든다.
- 검은 섹션 안에서 컬러 말풍선이 강한 대비를 만든다.
- 에셋은 독립적으로 떠 있지만, 문맥은 사용자의 고민과 해결 흐름에 붙어 있다.

Dowin 적용:

- Problem 섹션에는 고민 말풍선, 흐릿한 액션 카드, 작은 팀원 토큰을 배치한다.
- Solution 섹션에는 큰 인물 일러스트보다 목표/액션/점수판 조각을 배치한다.

### 3.2. `temp/l.jpg`

- 추상 원형 오브젝트, 미니 서비스 카드, 문서/자동화 일러스트, 지표 카드가 반복된다.
- 흰 배경과 딥 블루 배경을 오가며 작은 요소의 대비를 만든다.
- B2B SaaS답게 제품 기능을 작은 proof 조각으로 보여준다.

Dowin 적용:

- `핵심 목표`, `성공 기준`, `액션 아이템`, `점수판`을 각각 작은 카드형 요소로 만든다.
- primary blue와 black을 함께 써서 흰 섹션에서도 힘이 빠지지 않게 한다.

### 3.3. `temp/m.jpg`

- 동전처럼 반복 가능한 3D 오브젝트가 hero와 섹션 배경에 리듬을 만든다.
- 앱 아이콘, 카드, 휴대폰, QR 같은 작은 요소가 제품의 사용 맥락을 만든다.
- 강한 파란 배경과 흰 카드의 대비가 선명하다.

Dowin 적용:

- Dowin에서는 동전 대신 `네모 체크 셀`, `목표 토큰`, `점수판 셀`, `월/화/수 상태 칩` 같은 반복 오브젝트를 쓴다.
- 모바일 기록은 큰 앱 화면보다 작은 phone/check-in card 수준으로 제한한다.

### 3.4. `temp/p.jpg`

- 기하학적 브랜드 조각, 수치 카드, 클립보드 일러스트, 후기 카드가 섹션별로 역할을 나눈다.
- 밝은 배경 위에 민트/핑크/블루 같은 포인트가 있으나, 섹션마다 주색이 명확하다.

Dowin 적용:

- 과한 다색 대신 `#3A64C7` primary, black, white, zinc를 기본으로 하고 보조색은 아주 적게 쓴다.
- 수치 카드 느낌은 Pricing/Proof에만 제한적으로 사용한다.

## 4. Dowin 소형 에셋 공통 가이드

### 4.0. 기준 에셋

앞으로 새로 생성하는 에셋은 아래 4개와 같은 질감으로 맞춘다.

```text
public/assets/landing/element-goal-target-orb.jpeg
public/assets/landing/element-mini-scoreboard-card.jpeg
public/assets/landing/element-problem-chat-bubbles.jpeg
public/assets/landing/element-weekly-check-cells.jpeg
```

공통 질감은 `matte soft clay/plastic product render`다. 흰 배경, 둥근 bevel, 두께감 있는 3D 형태, raised UI 디테일, 파랑/흰색/회색/검정 제한 팔레트를 유지한다. 생성 이미지 자체에는 그림자를 넣지 않는다.

### 4.1. 스타일 원칙

- Toss처럼 밝고 부드러운 3D/product graphic 느낌을 살린다.
- 선택된 4개 기준 에셋과 같은 matte clay/plastic 질감으로 맞춘다.
- 유리, 금속, glossy toy, flat vector 느낌은 쓰지 않는다.
- 하지만 Dowin은 팀 실행 운영 도구이므로 너무 귀엽거나 게임처럼 보이면 안 된다.
- `primary #3A64C7`, `black/zinc-950`, `white`, `zinc-100~300`을 중심으로 만든다.
- 에셋은 페이지 위에 떠 있는 독립 요소처럼 보여야 한다.
- 대부분은 순백색 배경 위에 단독 오브젝트로 생성한다.
- 작은 크기에서도 실루엣이 읽혀야 한다.
- 텍스트는 생성하지 않는다. 필요하면 선, 점, 칩, 막대, 체크만 사용한다.

### 4.2. 공통 베이스 프롬프트

아래 문장은 모든 생성 프롬프트에 반드시 중복 포함한다. 담당자가 별도 조합을 하지 않아도 되도록, 실제 복붙용 문서의 각 개별 프롬프트는 이 문장을 자체 포함해야 한다.

```text
Create a small isolated 3D landing-page asset for Dowin. Match this exact visual style: matte soft clay/plastic product render, pure white background, centered object, generous empty margin, rounded bevels, thick extruded shapes, raised UI details, soft ambient occlusion, and subtle gray edge shading. Do not include any shadow under or around the object. Use only Dowin primary indigo blue #3A64C7, white, light cool gray, medium zinc gray, and deep black #09090B. The asset should look like the same set as a 3D target icon, stacked checklist cards, chat bubbles, and a small dashboard card. Use a shallow front 3/4 product-render angle, not a steep isometric angle. No readable text.
```

### 4.3. 공통 네거티브 프롬프트

아래 네거티브도 각 개별 프롬프트에 중복 포함한다.

```text
No shadow, no drop shadow, no contact shadow, no cast shadow, no realistic photo, no flat vector, no glass, no metal, no glossy toy, no coin, no reward badge, no mascot, no human, no hand, no neon, no purple, no pink, no colorful gradient, no transparent background, no busy background, no readable text, no fake letters, no app screenshot, no thin flat icon, no steep isometric perspective.
```

### 4.4. 권장 생성 옵션

- 배경: pure white background
- 그림자: 생성 이미지 자체에는 넣지 않음
- 비율: 1:1 우선, 말풍선/카드는 4:3 또는 3:2
- 해상도: 1024x1024 또는 1536x1536
- 스타일: soft 3D object, product UI miniature, clean landing graphic
- 사용 위치: Hero 주변, Problem 카드 사이, Solution 카드 내부, Loop 노드 주변, Pricing 카드 주변, Final CTA 배경

## 6. 순차 작업 가이드

담당자는 아래 순서대로 진행한다.

### 6.1. 0단계: 준비

1. `public/assets/landing/` 폴더가 있는지 확인한다.
2. 기존 대표 이미지 파일은 삭제하거나 덮어쓰지 않는다.
3. 새로 만드는 소형 에셋은 반드시 `element-` prefix로 저장한다.
4. 생성 도구에서 pure white background 옵션을 켠다.
5. 이 문서의 공통 베이스 프롬프트와 공통 네거티브 프롬프트를 항상 함께 사용한다.

### 6.2. 1단계: 기본 톤 4종 생성

먼저 아래 4개를 만든다.

1. `element-square-check-cell.png`
2. `element-mini-scoreboard-card.png`
3. `element-action-item-chip-stack.png`
4. `element-goal-target-orb.png`

작업 방식:

- 각 에셋당 3안을 생성한다.
- 가장 좋은 1안을 고른다.
- 4개 에셋이 같은 조명, 같은 그림자, 같은 재질감으로 보이는지 확인한다.
- 4개 중 하나라도 너무 캐릭터/게임/장난감처럼 보이면 같은 톤으로 재생성한다.

1단계 통과 기준:

- 흰 배경 위에서 잘 보인다.
- 검은 배경 위에서도 테두리/그림자가 묻히지 않는다.
- primary blue와 black이 둘 다 느껴진다.
- 텍스트가 전혀 없다.

### 6.3. 2단계: MVP 5종 추가 생성

1단계가 통과되면 아래 5개를 만든다.

1. `element-problem-chat-bubbles.png`
2. `element-weekly-check-cells.png`
3. `element-progress-ring-object.png`
4. `element-team-member-tokens.png`
5. `element-mobile-checkin-mini.png`

작업 방식:

- 각 에셋당 2안을 생성한다.
- 1단계에서 고른 4개와 나란히 놓고 톤을 비교한다.
- 색감이 튀면 primary/black/white/zinc 중심으로 다시 생성한다.

2단계 통과 기준:

- Problem, Solution, Loop, Pricing 중 최소 한 섹션에 바로 배치할 수 있다.
- 작은 크기에서도 실루엣이 읽힌다.
- 기능을 과장하지 않는다.

### 6.4. 3단계: 2차 후보 생성

시간이 있거나 섹션 배치 중 빈 공간이 남으면 아래를 만든다.

1. `element-scattered-task-cards.png`
2. `element-unclear-question-chip.png`
3. `element-weekly-review-stack.png`
4. `element-review-note-cards.png`
5. `element-basic-plan-badge.png`
6. `element-seat-counter-tokens.png`
7. `element-focus-black-panel.png`
8. `element-primary-blue-block.png`
9. `element-floating-primary-rings.png`
10. `element-notification-dot-card.png`

작업 방식:

- 각 에셋당 1~2안만 생성한다.
- MVP 9개보다 품질이 낮으면 저장하지 않는다.
- 장식용으로만 보이고 제품 의미가 약하면 제외한다.

### 6.5. 4단계: 저장

선택된 파일만 아래에 저장한다.

```text
public/assets/landing/
```

저장 후 확인:

- 파일명이 체크 표와 일치하는가?
- 확장자가 `.png`인가?
- 배경이 순백색인가?
- 파일 용량이 과도하지 않은가?
- 기존 대표 이미지 파일을 덮어쓰지 않았는가?

### 6.6. 5단계: 최종 검수

각 파일을 흰 배경과 검은 배경 위에 각각 올려 확인한다.

검수 항목:

- 160px 너비에서도 의미가 읽힌다.
- 320px 너비에서도 깨져 보이지 않는다.
- 흐릿한 가짜 텍스트가 실제 문자처럼 보이지 않는다.
- Dowin primary `#3A64C7` 계열이 중심이다.
- black/zinc-950 대비가 적절히 들어 있다.
- Toss처럼 부드럽지만 Dowin답게 프로덕트 도구 느낌이 난다.

재생성해야 하는 경우:

- 한글/영어처럼 보이는 글자가 생성된 경우
- 캐릭터, 마스코트, 사람 얼굴 중심으로 나온 경우
- 다색 네온/보라색/분홍색이 주색이 된 경우
- 배경이 순백색이 아니거나 복잡해서 바로 쓰기 어려운 경우
- 작은 크기에서 무엇인지 전혀 안 읽히는 경우
- 실제 앱 스크린샷처럼 오해될 정도로 UI가 구체적인 경우

### 6.7. 6단계: 체크 표 업데이트

파일 저장이 끝나면 이 문서의 `에셋 생성 체크 표`에서 해당 행의 상태를 바꾼다.

- 생성 완료: `생성` 칸을 `[x]`로 변경
- 검수 완료: `검수` 칸을 `[x]`로 변경
- 랜딩 코드나 디자인 시안에 실제 반영: `적용` 칸을 `[x]`로 변경

이미지만 저장하고 코드에 아직 넣지 않았다면 `적용`은 비워둔다.

## 9. 생성 품질 체크리스트

- 소형 에셋 단독으로 봐도 무엇을 상징하는지 읽히는가?
- 순백색 배경이 랜딩의 흰 섹션과 자연스럽게 이어지는가?
- 검은 섹션에 올릴 후보는 흰 배경 박스째로 배치해도 어색하지 않은가?
- primary blue와 black이 함께 쓰였는가?
- 텍스트가 생성되어 깨져 보이지 않는가?
- 너무 귀엽거나 게임 보상처럼 보이지 않는가?
- 섹션 전체 이미지처럼 너무 커 보이지 않는가?
- 같은 오브젝트를 2~3개 반복 배치해도 어색하지 않은가?
