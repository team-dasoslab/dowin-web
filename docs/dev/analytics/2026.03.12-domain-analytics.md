# Analytics 도메인 설계 (Post-MVP)

> ⚠️ 이 도메인은 MVP 이후 구현 대상입니다. 현재는 설계안으로만 관리합니다.

## 1. 개요

**Analytics 도메인**은 누적된 Daily Log 데이터를 다양한 기간 단위로 집계하고,  
시각화 차트, AI 기반 실행계획 리뷰, 엑셀 다운로드를 통해 사용자의 자기 인식과 실행 품질을 강화한다.

AI 기능의 중심은 "결과를 평가하는 AI"가 아니라 "좋은 계획을 세우게 돕는 AI"다.  
즉, 점수판 생성/수정 시점에 Dowin, 후행지표, 선행지표가 실제로 서로 잘 연결되어 있는지 짧고 명확하게 짚어주는 보조 리뷰 기능을 제공한다.

---

## 2. 기능 목록

| 기능 | 설명 | 우선순위 |
|------|------|----------|
| 기간별 달성률 | 이번 주 / 이번 달 / 전체 누적 달성률 | Post-MVP 1순위 |
| 주간 비교 | 지난 주 대비 이번 주 달성률 변화 | Post-MVP 1순위 |
| 월간 비교 | 지난 달 대비 이번 달 달성률 변화 | Post-MVP 2순위 |
| 동적 차트 | 선, 막대, 파이 차트 유형 전환 | Post-MVP 2순위 |
| AI 실행계획 리뷰 | 점수판 생성/수정 시 4DX 기준 적합성 검토 및 보완 제안 | Post-MVP 2순위 |
| 엑셀 다운로드 | 집계 데이터 엑셀(xlsx) 내보내기 | Post-MVP 3순위 |

---

## 3. 집계 단위 정의

| 단위 | 기간 | 설명 |
|------|------|------|
| `weekly` | 해당 주 月~日 | 주간 달성 수 / 목표 수 |
| `monthly` | 해당 月 1일~말일 | 월간 달성 수 / 목표 수 |
| `all-time` | 점수판 시작일~오늘 | 전체 누적 달성률 |

---

## 4. 핵심 규칙 (Business Rules)

| # | 규칙 |
|---|------|
| BR-1 | 집계는 ACTIVE 선행지표만 포함 (ARCHIVED 지표는 집계에서 제외) |
| BR-2 | 목표 수는 `target_value × 집계 기간 내 일수`로 산정 |
| BR-3 | AI 실행계획 리뷰는 저장 전 초안에도 호출 가능하지만, 저장을 막는 하드 검증이 아니라 보조 피드백으로 동작 |
| BR-4 | AI 실행계획 리뷰는 최소 `goalName`, `lagMeasure`, `leadMeasures[]`를 입력받아 4DX 적합성, 누락 항목, 표현 개선안을 반환 |
| BR-5 | AI 결과는 사용자가 명시적으로 요청한 경우에만 생성하여 비용을 통제 |
| BR-6 | AI 실행계획 리뷰 결과는 짧은 TTL 캐시(예: 10분)를 사용 가능 |
| BR-7 | 엑셀 다운로드는 화면용 장식 데이터가 아니라 재활용 가능한 원시 집계 + 요약 지표를 함께 포함 |

## 5. AI 기능 상세 정의

### 5.1. AI 실행계획 리뷰

점수판 생성/수정 화면에서 사용자가 입력한 계획의 질을 점검하는 기능이다.

- **진입 위치**
  - `setup` 점수판 생성/수정 화면의 Dowin/후행지표/선행지표 입력 영역 하단
  - 버튼 예시: `AI로 계획 점검하기`
- **호출 시점**
  - 사용자가 버튼을 눌렀을 때만 호출
  - 자동 호출은 기본 비활성. 비용과 지연을 줄이기 위함
- **입력 컨텍스트**
  - Dowin 목표명
  - 후행지표 문구
  - 시작일/마감일
  - 선행지표 목록, 목표 빈도, 주기
- **출력 목표**
  - 4DX 관점에서 계획이 왜 좋은지/약한지 설명
  - 불명확한 표현, 통제 불가능한 항목, 수치화 부족, DOWIN와의 정렬 부족을 짚음
  - 수정 예시를 바로 제안
- **피드백 형식**
  - `overallAssessment`: 한 줄 총평
  - `alignmentReview`: Dowin-후행지표-선행지표 연결성 점검
  - `strengths[]`: 잘된 점 1~3개
  - `risks[]`: 약한 점 1~3개
  - `suggestions[]`: 문구 개선안 또는 선행지표 보완안
  - `exampleRewrite`: 더 나은 예시 문안

### 5.2. 리뷰 톤과 출력 원칙

- 결과를 길게 분석하지 않는다
- 한 번에 1~3개의 핵심 지적만 전달한다
- "왜 이게 약한지"와 "어떻게 바꾸면 나아지는지"를 함께 말한다
- 저장을 막는 검증기가 아니라, 사용자가 더 좋은 문구를 만들도록 돕는 코치로 동작한다
- 출력은 짧고 바로 수정 가능한 문장 중심으로 유지한다

예시:

- "목표는 분명하지만 선행지표가 DOWIN와 직접 연결된다는 점이 문장에서 잘 드러나지 않습니다."
- "후행지표는 있지만 완료 기준 시점이 빠져 있어 성공 여부를 판단하기 어렵습니다."
- "선행지표가 행동보다는 상태 설명에 가까워 실제 실행 점검에 불리합니다."

## 6. UX 시나리오

### 6.1. 점수판 생성/수정 시 AI 실행계획 리뷰

```text
1. 사용자가 Dowin, 후행지표, 선행지표를 입력
2. "AI로 계획 점검하기" 버튼 클릭
3. 로딩 상태 후 리뷰 카드/시트 노출
4. 사용자는 제안 내용을 보고 문구를 직접 수정
5. 저장 또는 수정 없이 닫기
```

UX 원칙:

- AI 제안은 자동 반영하지 않는다
- 사용자가 계획의 주도권을 가진다
- 버튼은 저장 CTA보다 보조적이어야 한다
- 리뷰 결과에서 "이 제안을 적용" 같은 빠른 반영 보조 액션은 Post-MVP 후보로 둔다

### 6.2. 엑셀 다운로드

```text
1. 사용자가 analytics 화면에서 기간을 선택
2. "엑셀 다운로드" 클릭
3. 서버에서 현재 필터 기준 export 데이터를 조회
4. 프론트에서 csv/xlsx 파일 생성
5. 요약 시트 + 상세 시트가 포함된 파일 다운로드
```

## 7. 엑셀 다운로드 상세 정의

### 7.1. 목적

- 개인 회고 자료로 보관
- 오프라인 회의 공유
- 추가 분석을 위해 스프레드시트에서 재가공

### 7.2. 다운로드 범위

초기 Post-MVP 범위는 개인 기준으로 시작한다.

- `weekly`
- `monthly`
- 사용자 지정 기간(`from` ~ `to`, 최대 92일)
- 선행지표 선택(`leadMeasureIds[]`)

팀 단위 일괄 내보내기는 개인정보/공개 범위 정책이 정해진 뒤 확장한다.

### 7.3. 파일 구성

- **Sheet 1. 요약**
  - 기간 라벨
  - Dowin
  - 후행지표
  - 전체 달성 수 / 목표 수 / 달성률
  - 지난 기간 대비 증감
- **Sheet 2. 선행지표별 집계**
  - 선행지표명
  - 주기
  - 목표 수
  - 달성 수
  - 달성률
  - 메모용 빈 컬럼(optional)
- **Sheet 3. 일자별 상세**
  - 날짜
  - 요일
  - 선행지표명
  - 달성 여부

### 7.4. 비포함 범위

- AI 피드백 원문 포함 여부는 초기에는 제외 권장
- 팀원 전체 상세 로그 일괄 다운로드는 제외
- 시각화 차트 이미지를 엑셀에 임베딩하는 기능은 제외
- 서버에서 바이너리 `xlsx`를 직접 생성하는 방식은 Cloudflare Free 제약상 MVP에서 제외

## 8. 성공 기준

### 8.1. AI 실행계획 리뷰

- 사용자가 저장 전에 계획의 부족한 점을 바로 인지할 수 있다
- 리뷰 결과가 4DX 언어로 해석 가능하다
- 선행지표를 "더 통제 가능하고 더 측정 가능하게" 바꾸는 데 직접 도움이 된다

### 8.2. 엑셀 다운로드

- 사용자가 화면에서 보던 기간 기준 데이터를 그대로 외부에서 재활용할 수 있다
- 회의 자료나 개인 회고 자료로 추가 정리 없이 바로 사용할 수 있다

## 9. API 명세 (설계안)

### GET `/api/analytics/summary`

**Query Params**
- `period`: `weekly` | `monthly` | `all-time`
- `weekStart` / `month`: 기준 날짜 (기본값: 현재)

**Response (200 OK)**
```json
{
  "period": "weekly",
  "periodLabel": "2026년 3월 2주차",
  "achieved": 14,
  "total": 21,
  "achievementRate": 66.7,
  "comparedToPrevious": 5.2,
  "leadMeasureBreakdown": [
    {
      "name": "매일 물 2L 마시기",
      "achieved": 5,
      "total": 7,
      "rate": 71.4
    }
  ]
}
```

---

### GET `/api/analytics/chart`

**Query Params**
- `granularity`: `daily` | `weekly`
- `from` / `to`: 기간 범위 (ISO date)

**Response (200 OK)**
```json
{
  "granularity": "daily",
  "series": [
    { "date": "2026-03-09", "achievementRate": 85.7 },
    { "date": "2026-03-10", "achievementRate": 42.9 }
  ]
}
```

---

### POST `/api/analytics/plan-review`

> 점수판 생성/수정 초안을 기반으로 AI 실행계획 리뷰 생성

**Request**
```json
{
  "goalName": "체중을 감량한다",
  "lagMeasure": "80kg에서 75kg까지 12주 안에 달성",
  "startDate": "2026-03-16",
  "endDate": "2026-06-08",
  "leadMeasures": [
    {
      "name": "주 3회 30분 유산소",
      "targetValue": 3,
      "period": "WEEKLY"
    },
    {
      "name": "매일 물 2L 마시기",
      "targetValue": 7,
      "period": "DAILY"
    }
  ]
}
```

**Response (200 OK)**
```json
{
  "overallAssessment": "전반적으로 DOWIN와 선행지표의 연결은 좋지만, 후행지표의 완료 기준을 조금 더 명확히 쓰면 좋습니다.",
  "alignmentReview": {
    "score": 82,
    "summary": "선행지표가 대부분 직접 통제 가능한 행동으로 정의되어 있습니다."
  },
  "strengths": [
    "운동과 수분 섭취처럼 행동 중심 지표가 포함되어 있습니다."
  ],
  "risks": [
    "후행지표에 측정 시점과 체크 방식이 드러나지 않습니다."
  ],
  "suggestions": [
    "후행지표에 측정 시점과 단위를 명시해보세요.",
    "식단 관련 선행지표가 필요하다면 주 5회 기록처럼 통제 가능한 형태로 추가해보세요."
  ],
  "exampleRewrite": {
    "lagMeasure": "매주 월요일 아침 체중 기준으로 80kg에서 75kg까지 12주 안에 감량"
  },
  "generatedAt": "2026-03-16T00:00:00Z"
}
```

### GET `/api/analytics/export-data`

**Query Params**
- `from`: `YYYY-MM-DD`
- `to`: `YYYY-MM-DD`
- `leadMeasureIds`: comma-separated ids (optional, 미입력 시 전체)
- `view`: `week` | `month` (optional, 기간 기본값 추론/라벨링용)
- `from`~`to` 기간은 최대 92일까지만 허용 (초과 시 `400 Bad Request`)

**Response (200 OK)**
```json
{
  "periodMeta": {
    "from": "2026-03-01",
    "to": "2026-03-31",
    "dayCount": 31
  },
  "summary": {
    "achieved": 42,
    "total": 70,
    "achievementRate": 60
  },
  "leadMeasureBreakdown": [
    {
      "leadMeasureId": "lm_1",
      "name": "매일 물 2L 마시기",
      "period": "DAILY",
      "achieved": 20,
      "total": 31,
      "achievementRate": 64.5
    }
  ],
  "dailyRows": [
    {
      "date": "2026-03-01",
      "leadMeasureId": "lm_1",
      "leadMeasureName": "매일 물 2L 마시기",
      "status": "ACHIEVED"
    }
  ]
}
```

## 10. 기술 고려사항

- **LLM 연동**: Cloudflare Workers AI 또는 OpenAI API
- **프롬프트 설계**
  - 실행계획 리뷰 전용 프롬프트를 사용
  - 4DX 원칙(예측성, 통제 가능성, 측정 가능성, Dowin 정렬)을 명시적으로 넣음
  - 출력 길이와 톤을 강하게 제한해 "짧은 리뷰"로 유지
  ```
  [시스템 프롬프트]
  당신은 4DX 방법론 기반의 목표 코치입니다.
  사용자가 작성한 점수판 계획을 검토하고, 모호한 표현 대신 구체적이고 실행 가능한 개선 방향을 짧게 제안하세요.
  결과 평가는 하지 말고, Dowin-후행지표-선행지표의 연결성과 실행 가능성만 리뷰하세요.
  출력은 짧은 총평 1개와 개선 포인트 1~3개만 반환하세요.

  [데이터]
  Dowin: 체중을 감량한다
  후행지표: 80kg에서 75kg까지 12주 안에 달성
  선행지표1: 주 3회 30분 유산소
  선행지표2: 매일 물 2L 마시기
  ```
- **비용 최적화**
  - 리뷰는 사용자 명시 액션 기반으로만 생성
  - 같은 초안 payload에 대한 plan review는 짧은 TTL 캐시 검토
- **엑셀 생성**
  - Free 플랜 MVP에서는 서버가 JSON export 데이터를 반환하고, 프론트에서 파일을 생성
  - 기본 포맷은 csv 우선, 필요 시 프론트 `xlsx` 생성으로 확장
  - 데이터량이 커지면 서버 비동기 export job + 객체 저장소(R2)로 확장 가능

## 11. 오픈 이슈

- AI 실행계획 리뷰를 `analytics` 도메인으로 둘지, `scoreboard/setup` 보조 기능으로 분리할지 최종 경계 결정 필요
- 팀 단위 엑셀 다운로드 허용 범위와 공개 정책 정리 필요
- AI 제안의 "빠른 반영" UX를 제공할지 여부는 실제 사용성 검증 후 결정
