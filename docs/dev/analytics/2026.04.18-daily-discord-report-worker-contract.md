# Daily Discord Report Worker Contract

## 1. 배경

- Dowin 앱은 제품 이벤트를 GA4로 전송한다.
- 일일 운영 리포트는 `GA4 Data API -> 집계 -> Discord 웹훅 발송` 흐름으로 생성한다.
- 이 실행 경로는 앱 Worker 내부 cron이 아니라 `별도 스케줄러 워커`가 담당한다.

즉 현재 경계는 아래와 같다.

- Dowin 앱
  - 이벤트 정의와 발생
  - Discord 리포트 payload 포맷 계약 제공
- 스케줄러 워커
  - GA4 Data API 호출
  - 날짜/기간 집계
  - 이상 징후 판단
  - Discord 웹훅 발송

## 2. 책임 분리 원칙

### 2.1. Dowin 앱이 책임지는 것

- GA4 이벤트명과 파라미터 안정성 유지
- `push_notification_shown`, `push_notification_clicked`, `push_notification_opened_target` 등 이벤트 발생
- Discord 리포트 포맷 계약 유지
  - `src/domain/analytics/services/daily-discord-report.service.ts`

### 2.2. 스케줄러 워커가 책임지는 것

- GA4 Property 조회 인증
- 집계 기준 날짜 계산
- metric/filter 조합 생성
- 어제 값, 최근 7일/14일 값 집계
- 임계값 기반 `alerts`, `notes` 생성
- Discord 웹훅 POST
- 실패 로그와 재시도

### 2.3. 이 저장소가 직접 하지 않는 것

- GA4 Admin/Data API 인증 시크릿 관리
- Discord 발송 스케줄링
- 리포트 재시도 큐
- 장기 보관용 analytics warehouse

## 3. 스케줄러 워커 입력

필수 시크릿/환경 변수 예시:

- `GA4_PROPERTY_ID`
- `GA4_CLIENT_EMAIL`
- `GA4_PRIVATE_KEY`
- `DISCORD_WEBHOOK_URL`
- `REPORT_TIMEZONE`
  - 기본값: `Asia/Seoul`

선택:

- `ALERT_DROP_RATIO`
  - 기본값 예: `0.3`
- `ALERT_STREAK_DAYS`
  - 기본값 예: `3`

## 4. 스케줄러 워커 출력

### 4.1. Discord 웹훅 payload

스케줄러 워커는 최종적으로 아래 형식의 입력을 앱 저장소 계약에 맞게 만든 뒤 전송한다.

- 타입 원천:
  - `DailyDiscordReportInput`
- 포맷터:
  - `DailyDiscordReportService`

필수 집계 항목:

- `acquisition`
  - `signUps`
  - `workspaceActivatedUsers`
  - `scoreboardCreatedUsers`
  - `firstLogUsers`
  - `activationRateFromSignup`
  - `firstLogRateFromScoreboard`
- `engagement`
  - `activeUsers`
  - `loggingUsers`
  - `streakCandidateUsers`
- `retention`
  - `returningUsers7d`
  - `retainedUsers14d`
- `intervention`
  - `pushClicks`
  - `pushFollowupConversions24h`
  - 추후 `leaderReportClicks`
  - 추후 `leaderReminderConversions24h`
- `alerts`
- `notes`

### 4.2. 운영 로그

스케줄러 워커는 최소 아래 로그를 남긴다.

- `report_date`
- `report_timezone`
- `ga4_query_started_at`
- `ga4_query_finished_at`
- `discord_sent_at`
- `report_version`
- `status`
  - `success` | `failed`
- 실패 시 `error_message`

## 5. GA4 집계 기준

### 5.1. 이벤트 원천

우선 집계 대상 이벤트:

- `sign_up_completed`
- `workspace_created`
- `workspace_joined`
- `scoreboard_created`
- `daily_log_checked`
- `dashboard_my_viewed`
- `push_notification_shown`
- `push_notification_clicked`
- `push_notification_opened_target`

추후 포함:

- `lead_measure_created`
- `weekly_review_started`
- `weekly_review_completed`
- `leader_report_clicked`
- `leader_reminder_sent`
- `push_followup_log_checked` (BigQuery 대체용 클라이언트 이벤트)

### 5.2. 기본 해석 규칙

- `workspaceActivatedUsers`
  - `workspace_created`와 `workspace_joined`의 user 기준 union

- `firstLogUsers`
  - **정의**: 어제 날짜에 역대 처음으로 `daily_log_checked`를 발생시킨 user 수
  - **GA4 Data API 근사 처리**: `newVsReturning = new` 필터로 신규 세션 user 중 어제 `daily_log_checked`가 있는 user 수로 대체 허용
  - 정확도보다 추세 파악이 목적이므로 근사 처리로 충분

- `activeUsers`
  - 어제 `dashboard_my_viewed` 또는 `daily_log_checked`가 있는 user 수

- `streakCandidateUsers`
  - **정의**: 최근 14일 내 서로 다른 날짜에 2일 이상 `daily_log_checked`가 있는 user 수
  - GA4 Data API로 구현 시: 최근 14일 기간의 `daily_log_checked` unique user를 날짜 dimension으로 group by 후, 2일 이상 등장한 user 수를 집계

- `pushFollowupConversions24h`
  - **BigQuery 없이 정확한 24시간 window 집계는 불가**하므로, 앱 클라이언트에서 별도 이벤트를 발생시키는 방식을 채택한다.
  - **구현 방식**:
    - **서비스 워커 유입**: 푸시 클릭 시 `push_notification_opened_target` 발생과 함께 컨텍스트 저장.
    - **외부 링크 유입 (Discord 등)**: URL에 `?campaign_id=...&push_type=...` 포함 시 `useCampaignAttribution` 훅이 이를 감지하여 `push_notification_opened_target` 발생 및 컨텍스트 저장.
    - **후속 행동**: 이후 24시간 이내에 `daily_log_checked`가 발생하면 `push_followup_log_checked` 이벤트를 GA4에 전송.
  - **집계**: `pushFollowupConversions24h` = 어제 `push_followup_log_checked`가 있는 unique user 수
  - **파라미터**: `push_type`, `campaign_id`
  - 이 방식은 GA4 Data API만으로 단순 집계가 가능하며, BigQuery 없이도 정확한 의미의 전환을 측정할 수 있다.

중요:

- 이 값은 단순 이벤트 count가 아니라 가능하면 `user 기준 dedupe`를 우선한다.
- count와 unique user를 섞어 쓰면 리포트 해석이 틀어진다.

## 6. 권장 집계 절차

```text
1. report_date 결정
   - 기본: KST 기준 어제
2. GA4 Data API에서 일 단위/최근 7일/최근 14일 데이터 조회
3. 이벤트별 unique user 집계
4. activation/engagement/retention/intervention 구조로 변환
5. 7일 평균 및 직전 3일 추세와 비교
6. alerts, notes 생성
7. DailyDiscordReportService.buildPayload() 형식과 맞는 입력 생성
8. Discord webhook 발송
9. 결과 로그 저장
```

## 7. 이상 징후 판단 기본값

- 핵심 전환율이 3일 연속 하락
  - `watch`
- 7일 평균 대비 30% 이상 급락
  - `alert`
- 클릭은 유지되는데 후속 기록이 하락
  - `message/landing mismatch`
- 가입은 유지되는데 첫 기록이 급락
  - `activation gap`

이 값은 워커 설정값으로 override 가능하게 두는 것이 좋다.

## 8. 재시도와 실패 처리

- Discord 발송 실패는 최소 1회 재시도
- GA4 조회 실패와 Discord 발송 실패는 별도 에러 코드로 구분
- 같은 날짜 리포트를 중복 발송하지 않도록 idempotency key를 둔다
  - 예: `dowin-daily-report:${report_date}:${timezone}`

## 9. 보안 원칙

- GA4 서비스 계정 키와 Discord 웹훅 URL은 앱 저장소에 두지 않는다.
- 스케줄러 워커에서만 secret binding 또는 별도 secret store로 관리한다.
- GA4에서 내려온 raw row를 그대로 Discord에 보내지 않는다.
- Discord에는 PII와 자유 텍스트 본문을 보내지 않는다.
