# 선행지표 태그 백엔드 설계

## 1. 범위

이 문서는 `docs/planning/2026.04.10-lead-measure-tag-plan.md`를 구현 가능한 백엔드 설계로 구체화한다.

이번 설계의 범위는 아래에 한정한다.

- 태그를 워크스페이스 공용 데이터로 둘지 결정
- DB 저장 모델 확정
- 중복 방지와 최대 개수 제약 정의
- Lead Measure 조회/생성/수정 시 `tags` shape 정의
- 태그 rename/delete 정책과 인가 기준 정의

이번 문서는 UI 세부 인터랙션이나 태그 필터/집계 UX를 다루지 않는다.

## 2. 핵심 결정

### 2.1. 태그 소유권

- 태그는 개인 메타데이터가 아니라 `워크스페이스 공용 분류 체계`로 둔다.
- 같은 워크스페이스 안에서는 여러 사용자가 동일 태그를 재사용한다.
- 이후 태그 기반 필터, 집계, export, 추천 기능은 워크스페이스 공용 태그를 기준으로 확장한다.

### 2.2. 저장 모델

- `lead_measures` 테이블에 CSV나 JSON 문자열 컬럼을 추가하지 않는다.
- 태그는 `태그 사전 테이블 + 조인 테이블` 구조로 저장한다.

여기서 `태그 사전`은 "이 워크스페이스에서 사용할 수 있는 태그 목록"을 뜻한다.

권장 테이블은 아래 두 개다.

1. `workspace_tags`
2. `lead_measure_tags`

이 구조를 선택하는 이유:

- 태그 이름 수정(rename)을 한 row 업데이트로 처리할 수 있다.
- 태그 삭제 시 연결 관계를 cascade로 정리할 수 있다.
- 중복 태그와 동의어 난립을 DB 제약으로 일부 방지할 수 있다.
- 이후 색상, 정렬 순서, 설명 같은 메타데이터를 붙이기 쉽다.

## 3. DB 설계

### 3.1. `workspace_tags`

워크스페이스 공용 태그 사전 테이블이다.

예상 컬럼:

| 컬럼 | 설명 |
| --- | --- |
| `id` | 태그 PK |
| `workspace_id` | 소속 워크스페이스 ID |
| `name` | 사용자 표시용 태그명 |
| `normalized_name` | 중복 판정용 정규화 값 |
| `created_by_user_id` | 최초 생성 사용자 |
| `created_at` | 생성 시각 |

권장 제약:

- `workspace_id -> workspaces.id` cascade
- `created_by_user_id -> users.id` set null 또는 cascade
- `(workspace_id, normalized_name)` unique

### 3.2. `lead_measure_tags`

선행지표와 태그의 N:M 연결 테이블이다.

예상 컬럼:

| 컬럼 | 설명 |
| --- | --- |
| `id` | 연결 PK |
| `lead_measure_id` | 선행지표 ID |
| `tag_id` | 태그 ID |
| `created_at` | 연결 생성 시각 |

권장 제약:

- `lead_measure_id -> lead_measures.id` cascade
- `tag_id -> workspace_tags.id` cascade
- `(lead_measure_id, tag_id)` unique

### 3.3. 왜 문자열 배열/CSV를 쓰지 않는가

`lead_measures.tags` 같은 text/json 컬럼 방식은 초기 구현은 단순하지만 아래 문제가 있다.

- 태그 rename 시 여러 선행지표 row를 모두 수정해야 한다.
- 중복 방지 기준을 애플리케이션 로직에만 의존하게 된다.
- 태그 삭제, 병합, 공용 태그 목록 조회가 불편하다.
- 이후 태그별 집계나 필터 쿼리가 약해진다.

따라서 MVP 시점에도 조인 테이블 구조가 더 안전하다.

## 4. 정규화와 제약

### 4.1. 최대 개수

- 선행지표 1개당 태그는 최대 3개까지 허용한다.
- 이 제한은 validation 레벨과 서비스 레벨에서 모두 강제한다.
- DB는 개수 제한을 직접 강제하기 어렵기 때문에 저장 전 검증으로 막는다.

### 4.2. 중복 금지

- 같은 선행지표에 동일 태그를 중복 연결할 수 없다.
- 같은 워크스페이스 안에서 같은 의미의 태그가 중복 생성되지 않도록 `normalized_name` 기반 unique를 둔다.

### 4.3. normalize 규칙

`normalized_name`은 중복 판정용 표준화 문자열이다.

MVP 기본 규칙:

- 앞뒤 공백 제거
- 내부 연속 공백은 한 칸으로 축약
- 영문 대소문자는 소문자로 통일

예:

- `" 운동 "` -> `"운동"`
- `"Deep   Work"` -> `"deep work"`

이 값은 표시용이 아니라 중복 판정과 조회 정합성 유지에만 사용한다.

## 5. 인가와 소유권 검증

태그는 워크스페이스 공용이므로, 모든 쓰기 요청은 `workspace_id` 기준으로 검증한다.

핵심 규칙:

- 요청자는 해당 워크스페이스의 현재 참여자여야 한다.
- 연결하려는 `tag_id`는 수정 대상 `lead_measure`와 같은 워크스페이스 소속이어야 한다.
- 다른 워크스페이스에서 만든 태그를 현재 선행지표에 주입할 수 없어야 한다.

검증 기준은 항상 아래 3개를 함께 본다.

1. 요청자 `workspace_members`
2. 대상 `lead_measure`가 속한 `scoreboard.workspace_id`
3. 대상 `workspace_tag.workspace_id`

추후 한 사용자가 여러 워크스페이스에 동시에 속할 수 있게 바뀌더라도, 이 규칙은 그대로 유지 가능해야 한다.

## 6. rename / delete 정책

사용자 요청 기준으로 아래 정책을 채택한다.

- 태그 rename은 워크스페이스 권한과 무관하게 현재 참여자라면 가능하다.
- 태그 delete도 워크스페이스 권한과 무관하게 현재 참여자라면 가능하다.

단, 이 정책은 공용 분류 체계가 모든 참여자에게 열려 있다는 뜻이므로 아래 동작은 명확해야 한다.

### 6.1. rename

- rename은 `workspace_tags.name`, `workspace_tags.normalized_name`을 갱신한다.
- rename 즉시 해당 태그를 참조하던 모든 선행지표의 조회 결과에 새 이름이 반영된다.
- rename 후 `(workspace_id, normalized_name)` unique 충돌이 나면 실패해야 한다.

### 6.2. delete

- delete는 `workspace_tags` row 삭제로 처리한다.
- 연결된 `lead_measure_tags`는 cascade delete로 정리한다.
- 삭제 후 해당 태그를 쓰던 선행지표들은 단순히 그 태그가 빠진 상태가 된다.

이 정책은 운영상 변동성이 크므로, 실제 구현 시 UI에는 "공용 태그 전체에 영향" 경고가 필요하다.

## 7. API 응답 shape

Lead Measure 응답에는 태그를 선행지표별 배열로 포함한다.

권장 shape:

```json
{
  "id": 12,
  "name": "주 3회 운동",
  "targetValue": 3,
  "period": "WEEKLY",
  "status": "ACTIVE",
  "tags": [
    { "id": 1, "name": "운동" },
    { "id": 4, "name": "건강" }
  ]
}
```

원칙:

- `tags`는 항상 배열로 반환한다.
- 태그가 없으면 `[]`를 반환한다.
- 문자열 배열보다 객체 배열을 우선한다.
  - 이유: 이후 색상, 순서, 설명, preset 여부 같은 메타데이터를 추가하기 쉽다.

## 8. API 변경 방향

구현 단계에서 최소 변경이 필요한 계약은 아래와 같다.

### 8.1. Lead Measure 생성/수정

- `POST /api/scoreboards/:id/lead-measures`
- `PUT /api/lead-measures/:id`

추가 필드:

- `tags: number[]` 또는 `tagIds: number[]`

권장:

- 요청 필드는 `tagIds`로 둔다.
- 응답 필드는 `tags` 객체 배열로 둔다.

이유:

- 요청은 식별자 기반이 단순하다.
- 응답은 렌더링 친화적인 shape가 낫다.

### 8.2. Lead Measure 조회

- `GET /api/scoreboards/:id/lead-measures`
- 필요 시 단건 조회가 생기면 동일한 `tags` shape를 재사용한다.

### 8.3. 태그 사전 API

MVP에서 아래 API가 필요할 가능성이 높다.

- `GET /api/workspaces/:id/tags`
- `POST /api/workspaces/:id/tags`
- `PUT /api/workspaces/:id/tags/:tagId`
- `DELETE /api/workspaces/:id/tags/:tagId`

이 API는 모두 "현재 워크스페이스 참여자" 기준으로 열 수 있다.

## 9. 서비스/스토리지 레이어 반영 포인트

### 9.1. storage

추가가 필요한 저장소 책임:

- 워크스페이스 태그 목록 조회
- normalized 기준 태그 생성/중복 확인
- 태그 rename/delete
- Lead Measure 생성/수정 시 tag 연결 upsert
- Lead Measure 조회 시 tag 조인 로드

### 9.2. service

추가가 필요한 서비스 책임:

- 최대 3개 검증
- 동일 워크스페이스 태그만 연결되도록 검증
- rename 충돌 처리
- delete 영향 범위 검증

## 11. 남은 주의사항

- rename/delete를 모든 참여자에게 열면 공용 분류 체계가 흔들릴 수 있다.
- 따라서 권한 모델은 요청대로 열되, 충돌/삭제 영향에 대한 서버 에러와 프론트 경고는 반드시 명확해야 한다.
- 태그 사용량이 늘면 `workspace_tags(workspace_id, normalized_name)`와 `lead_measure_tags(lead_measure_id, tag_id)` 인덱스가 핵심이 된다.
