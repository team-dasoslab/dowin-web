# Basic seat 결제 구매 기록 보존 개발 설계

## 1. 한 줄 결론

현재 Dowin billing 구현은 Basic 권한을 반영하는 데는 충분해지고 있지만, 세무/분쟁/환불 대응에 필요한 구매 기록 보존 구조로는 부족하다.

개선 방향은 명확하다.

- 카드 원문, CVC/CVV, 결제수단 민감값은 Dowin DB, 로그, 에러 리포트에 저장하지 않는다.
- Polar webhook 원문 payload를 그대로 저장하지 않고, allowlist 기반 normalized snapshot만 저장한다.
- 권한 projection용 billing ledger와 법정 보존용 billing archive를 분리한다.
- 계정/워크스페이스 삭제가 발생해도 법정 보존 대상 구매 기록은 `legal_retention_until` 전까지 유지한다.
- 개인정보처리방침과 checkout 고지는 실제 보존 기간, 자동갱신, 환불 조건, 정책 버전과 일치해야 한다.

## 2. 배경

Dowin의 신규 결제 정책은 Basic-only seat 결제다.

사용자는 워크스페이스 생성 단계에서 Basic seat checkout을 진행하고, 결제 성공 후 워크스페이스와 seat entitlement가 활성화된다. 이 흐름이 실제 판매 구조가 되면 Dowin는 단순히 `현재 구독이 활성인가`만 저장해서는 부족하다.

판매 이후에는 아래 질문에 답할 수 있어야 한다.

- 누가 결제했는가?
- 어떤 워크스페이스에 몇 seat를 샀는가?
- 사용자가 어떤 가격/주기/환불/자동갱신 조건을 보고 결제했는가?
- 결제, 환불, 해지, 차지백, 세금 처리는 어떻게 발생했는가?
- 계정 삭제 후에도 세무/분쟁 대응에 필요한 구매 기록이 남아 있는가?
- 보존 기간이 지나면 어떤 값이 삭제 또는 익명화되는가?

기준 문서의 결론은 `구매 기록 10년 보존`이다. 한국만 먼저 판매해도 EU/영국 OSS 가능성과 해외 세무 확장을 고려하면 10년을 기본 설계값으로 두는 편이 단순하다.

## 3. 현재 문제

현재 구현은 결제 권한 운영에는 가까워졌지만, 구매 기록 보존 요구사항은 아직 충족하지 못한다.

| 문제                      | 현재 상태                                                                           | 리스크                                                         |
| ------------------------- | ----------------------------------------------------------------------------------- | -------------------------------------------------------------- |
| webhook 원문 저장         | `billing_events.payload_json`에 request body가 그대로 저장된다                      | 불필요한 개인정보나 provider 내부 민감값이 장기 보관될 수 있다 |
| 법정 보존 archive 부재    | `billing_events`, `workspace_billing_state`, `workspace_seat_entitlements` 중심이다 | 권한 projection 테이블을 세무/분쟁 archive로 오용하게 된다     |
| 삭제와 보존 분리 부족     | 계정 삭제는 user hard delete, 일부 billing FK는 cascade 또는 set null이다           | 구매자/동의/거래 입증력이 약해질 수 있다                       |
| checkout 증거 부족        | 정책 버전, 자동갱신 동의, displayed price, country/tax evidence가 없다              | 사용자가 어떤 조건을 보고 결제했는지 사후 입증하기 어렵다      |
| invoice/tax snapshot 부족 | invoice/order/payment/tax amount/rate/jurisdiction 저장 위치가 없다                 | provider dashboard에 과도하게 의존한다                         |
| 법률 문구 불일치          | 개인정보처리방침은 `필요한 기간` 수준이다                                           | 실제 10년 보존 정책과 사용자 고지가 어긋난다                   |

## 4. 설계 목표

### 4.1. MVP 목표

- Polar webhook 저장 전에 sanitizer를 적용한다.
- `billing_events.payload_json`에는 원문이 아니라 normalized snapshot만 저장한다.
- `billing_retention_records` 또는 동등한 별도 archive를 추가한다.
- checkout, subscription, order, invoice, refund, revoke 이벤트에서 archive snapshot을 남긴다.
- `legal_retention_until = occurred_at + 10 years`를 기본값으로 둔다.
- 계정/워크스페이스 삭제와 archive 보존을 분리한다.
- 개인정보처리방침에 결제/세무/분쟁 보존 항목과 기간을 반영한다.
- checkout 화면, metadata, archive가 같은 policy version 상수를 사용한다.

### 4.2. 비목표

- Polar를 대체하는 결제사 추상화 전면 재설계
- 국가별 세금 계산 엔진 자체 구현
- EU/미국/캐나다 판매 확정
- 보존기간 만료 자동 삭제 job 구현
- 운영자용 archive export UI 구현

보존기간 만료 삭제 job, legal hold 관리 UI, archive export는 Post-MVP로 둔다. 다만 MVP 스키마에는 나중에 붙일 수 있도록 `legal_hold`, `legal_retention_until`을 포함한다.

## 5. 목표 구조

Billing 데이터는 역할별로 나눈다.

| 계층                   | 역할                                                | 대표 테이블                                              |
| ---------------------- | --------------------------------------------------- | -------------------------------------------------------- |
| checkout intent        | 결제 전 워크스페이스 생성 의도와 checkout 생성 증거 | `pending_workspace_checkouts`, `billing_checkout_events` |
| billing ledger         | provider 이벤트 처리 이력과 projection 입력         | `billing_events`                                         |
| entitlement projection | 현재 제품 권한과 seat 수                            | `workspace_billing_state`, `workspace_seat_entitlements` |
| retention archive      | 세무/분쟁/환불/삭제 후 보존 구매 기록               | `billing_retention_records`                              |

핵심 원칙:

- `billing_events`는 이벤트 처리 원장이지 법정 보존 archive가 아니다.
- `workspace_billing_state`는 현재 상태 projection이지 증빙 원본이 아니다.
- `billing_retention_records`는 user/workspace 삭제와 독립적으로 유지한다.
- 외부 provider 원문 payload는 저장하지 않는다. 저장 대상은 Dowin가 정의한 normalized snapshot이다.

FK 전략:

- `workspace_id_snapshot`, `billing_owner_user_id_snapshot`에는 cascade FK를 걸지 않는다.

- 현재 운영 데이터와 연결 조회가 필요하면 nullable soft reference로만 다룬다.

개인정보 접근:

- `billing_email`, `billing_name`, `tax_id`, 상세 주소가 추가될 경우 강한 접근통제 대상이다.
- D1에서 필드 단위 암호화를 바로 적용하지 않는다면 최소한 운영자 UI/API 노출을 제한하고, export/debug log에 포함하지 않는다.

### 6.2. `billing_events.payload_json`

현재 필드명은 유지할 수 있지만 의미를 바꾼다.

- 변경 전: webhook body 원문에 가까운 JSON
- 변경 후: `normalizePolarBillingEventPayload(payload)` 결과 JSON

원문 보관이 꼭 필요하다는 요구가 생기면 일반 DB가 아니라 별도 암호화 저장소, 별도 접근권한, 짧은 retention을 전제로 다시 설계한다. MVP에서는 원문을 저장하지 않는다.

### 6.3. checkout evidence

checkout 생성 시점에는 아래 값을 내부 pending record 또는 `billing_checkout_events`에 남긴다.

- `targetPlanCode = BASIC`
- `requestedSeatCount`
- `displayedPrice`
- `currency`
- `billingInterval`
- `termsVersion`
- `privacyPolicyVersion`
- `billingPolicyVersion`
- `checkoutNoticeVersion`
- `autoRenewalNoticeAcceptedAt`
- `locale`
- `ipCountry`
- `billingCountry`
- `taxEvidenceSource`

Polar metadata에는 provider 확인에 필요한 최소값만 넣는다. Dowin 내부 증거는 내부 DB에 저장하는 쪽을 우선한다.

## 7. sanitizer 설계

### 7.1. 위치

권장 파일:

- `src/domain/billing/polar-payload-normalizer.ts`
- 테스트: `src/domain/billing/polar-payload-normalizer.test.ts`

사용 지점:

- `src/domain/billing/services/polar-webhook.service.ts`
- `src/domain/billing/polar.ts`의 Polar API error log 정규화

### 7.2. allowlist

저장 허용 값:

- event ID, event type, occurred at
- customer/subscription/checkout/order/invoice/payment ID
- status
- amount, tax amount, currency
- product ID, price ID, plan code, seat count
- current period start/end
- refund/dispute/chargeback status
- receipt/invoice URL 또는 key
- Dowin metadata 중 `workspaceId`, `workspaceCheckoutId`, `requestedByUserId`, `targetPlanCode`, `requestedSeatCount`, policy version

### 7.3. denylist

아래 키는 깊이에 상관없이 제거한다.

- `card`
- `cards`
- `cvc`
- `cvv`
- `cid`
- `cav2`
- `pan`
- `number`
- `track`
- `pin`
- `pin_block`
- `payment_method`
- `payment_method_details`
- `billing_address` 중 보존 목적에 필요 없는 상세
- provider 내부 token 또는 secret으로 보이는 값

주의: denylist만으로 충분하지 않다. 기본은 allowlist이며, denylist는 방어층으로만 둔다.

## 8. 처리 흐름

### 8.1. Workspace setup checkout

1. 사용자가 워크스페이스 이름과 seat count를 입력한다.
2. 서버가 pending checkout intent를 만든다.
3. 서버가 checkout evidence를 저장한다.
4. 서버가 Polar checkout을 생성한다.
5. 사용자는 Polar hosted checkout으로 이동한다.
6. 성공 redirect는 화면 전환 보조로만 사용한다.
7. 실제 권한 반영과 archive 저장은 webhook 또는 checkout 검증 결과를 기준으로 한다.

### 8.2. Polar webhook

1. route handler가 signature 검증을 수행한다.
2. raw body를 서비스로 넘기기 전에 parsed payload를 확보한다.
3. `normalizePolarBillingEventPayload`를 호출한다.
4. `billing_events`에는 normalized snapshot만 append한다.
5. 이벤트 유형별 projection을 갱신한다.
6. archive 대상 이벤트면 `billing_retention_records`에 append 또는 upsert한다.
7. archive row에는 checkout evidence와 provider payment/invoice/tax snapshot을 함께 넣는다.

### 8.3. 계정/워크스페이스 삭제

계정 삭제 또는 워크스페이스 삭제는 운영 데이터 삭제와 구매 기록 보존을 분리해야 한다.

정책:

- active Polar subscription이 있으면 기존처럼 workspace 삭제를 막는다.
- 삭제 전 archive에 필요한 구매자/워크스페이스 snapshot이 있는지 확인한다.
- archive가 없고 billing state 또는 provider key가 있으면 최소 archive row를 생성한다.
- 운영 데이터 삭제 후에도 archive row는 `legal_retention_until` 전까지 유지한다.
- archive에는 화면 표시용 닉네임 같은 비필수 개인정보를 새로 추가하지 않는다.

현재 구현 상태:

- 계정 탈퇴는 `users` row hard delete다. user FK cascade 대상 row는 삭제될 수 있고, `workspace_billing_state.billing_owner_user_id`는 set null 된다.
- 워크스페이스 삭제는 `workspaces.deleted_at`을 설정하는 soft delete다. active Polar subscription이 있으면 삭제를 막는다.
- `billing_events`, `billing_checkout_events`, `workspace_billing_state`, `workspace_seat_entitlements`는 workspace FK를 가지고 있으므로, 현재 soft delete에서는 남지만 향후 hard purge가 추가되면 cascade 삭제될 수 있다.
- `billing_retention_records`는 user/workspace FK를 걸지 않는 snapshot archive다. 따라서 신규 webhook에서 생성된 archive row는 계정/워크스페이스 삭제와 독립적으로 `legal_retention_until`까지 보존할 수 있다.
- 단, 탈퇴/삭제 직전에 archive 누락을 감지해 최소 snapshot을 생성하는 로직은 아직 없다. 이 로직이 붙기 전에는 `archive row가 이미 생성된 결제 건`만 삭제와 독립적으로 보존된다고 봐야 한다.

보존기간:

- MVP 기본 보존기간은 `event_occurred_at + 10 years`다.
- 현재 구현은 webhook archive 생성 시 `legal_retention_until`에 이 값을 저장한다.
- 보존기간 만료 삭제/익명화 job은 아직 Post-MVP이며, `legal_hold`는 스키마에만 준비되어 있다.

원자성/idempotency 요구:

- webhook 처리에서 `billing_events`, `billing_retention_records`, `workspace_billing_state`, `workspace_seat_entitlements`, `workspaces` projection 갱신은 하나의 처리 단위다.
- 현재 구현은 이 쓰기 묶음을 `BillingStorage.recordPolarWebhookBillingEvent` transaction으로 처리한다.
- 중간 실패 시 event insert도 rollback되어 같은 `provider_event_id` provider retry가 다시 처리될 수 있어야 한다.
- transaction 내부 unique conflict로 event가 생성되지 않으면 concurrent duplicate로 보고 ignored 처리한다.

## 9. 법률 문구와 UI 정렬

`src/content/legal-documents.ts`는 구현과 함께 갱신해야 한다.

개인정보처리방침에 필요한 항목:

- 계약/청약철회/계약해제 기록: 5년
- 대금결제/재화 또는 서비스 공급 기록: 5년
- 소비자 불만/분쟁 처리 기록: 3년
- 세무 증빙 및 해외 판매 대응 구매 기록: 최대 10년
- 법정 보존 대상 기록은 운영 계정정보와 분리 보관
- 보존기간 만료 후 삭제 또는 익명화

checkout/billing 화면에 필요한 항목:

- Basic seat 상품명
- seat 수
- 가격과 통화
- 결제 주기
- 자동갱신 여부
- 다음 결제일 또는 계산 기준
- 해지 경로
- 환불/청약철회 조건
- 즉시 제공 여부
- 정책 버전과 실제 문구의 일치

## 10. 구현 순서

### Phase 0. 즉시 방어

1. `normalizePolarBillingEventPayload`를 추가한다.
2. webhook 저장 경로에서 원문 대신 normalized snapshot을 저장한다.
3. Polar API error log에도 sanitizer를 적용한다.
4. 민감 키가 DB/log snapshot에 남지 않는 테스트를 추가한다.

완료 기준:

- `card`, `cvc`, `cvv`, `pan`, `number`, `payment_method_details`, 불필요한 `billing_address`가 저장 snapshot에서 제거된다.
- 기존 webhook projection 동작은 유지된다.

### Phase 1. 보존 archive

1. `billing_retention_records` 스키마를 추가한다.
2. migration은 직접 작성하지 않고 저장소 규칙에 따라 생성/적용한다.
3. `BillingStorage`에 archive append/upsert 메서드를 추가한다.
4. checkout completion, subscription update/cancel/end/revoke, order paid/refund 이벤트에서 archive 기록을 남긴다.
5. `legal_retention_until`과 `legal_hold`를 저장한다.

완료 기준:

- archive row는 user/workspace 삭제와 독립적으로 유지된다.
- 같은 provider event 중복 수신 시 중복 archive row가 생기지 않는다.

### Phase 2. checkout evidence

1. billing policy version 상수를 추가한다.
2. workspace setup checkout과 existing workspace checkout 생성 시 evidence를 저장한다.
3. archive 생성 시 checkout evidence를 복사한다.
4. checkout UI와 legal document version이 같은 상수를 참조하게 한다.

완료 기준:

- 특정 결제 건에서 사용자가 본 가격/주기/정책 버전을 archive로 추적할 수 있다.

### Phase 3. 삭제 플로우 정렬

1. 계정 삭제 전에 billing archive 보존 상태를 확인한다.
2. 워크스페이스 삭제 전에 billing archive 보존 상태를 확인한다.
3. 필요한 경우 최소 archive snapshot을 생성한다.
4. 삭제 API 테스트에 archive 보존 회귀 테스트를 추가한다.

완료 기준:

- 사용자 삭제 후에도 법정 보존 대상 구매 기록이 유지된다.
- workspace soft delete 또는 향후 hard purge와 archive가 연결되어 삭제되지 않는다.

### Phase 4. 법률 문구 정렬

1. 개인정보처리방침의 결제/세무/분쟁 보존 기간을 항목별로 구체화한다.
2. 결제 및 환불 정책과 checkout 화면 문구를 Basic-only seat 결제 기준으로 맞춘다.
3. 해외 판매 전 merchant of record, VAT/OSS, sales tax/GST/HST 책임 범위를 별도 검토 항목으로 남긴다.

완료 기준:

- 실제 DB 보존 정책과 사용자 고지가 충돌하지 않는다.

## 11. 테스트 기준

필수 테스트:

- webhook normalizer가 민감 키를 제거한다.
- webhook normalizer가 필요한 결제/구독/환불/세금 필드를 유지한다.
- `PolarWebhookService`가 `billing_events.payload_json`에 normalized snapshot만 저장한다.
- 같은 provider event 재수신 시 billing event와 archive가 멱등 처리된다.
- checkout evidence가 pending record 또는 checkout event에 저장된다.
- archive row의 `legal_retention_until`이 기본 10년으로 계산된다.
- user deletion 후 archive row가 유지된다.
- workspace deletion 후 archive row가 유지된다.
