# Production Deployment Flow

## 1. 목적

이 문서는 DOWIN의 현재 프로덕션 배포 흐름을 설명한다.

- 왜 `main`이 아니라 `production` 브랜치에서 배포하는지
- Cloudflare Git integration이 어떻게 연결되어 있는지
- GitHub Actions가 D1 migration 검증과 실제 적용을 어떻게 실행하는지
- 어떤 secret과 Cloudflare 권한이 필요한지

## 2. 현재 운영 모델

- 개발 통합 브랜치: `main`
- 프로덕션 배포 브랜치: `production`
- Cloudflare Workers Builds(Git integration)는 `production` 브랜치 push만 자동 배포한다
- `production` 브랜치는 직접 push하지 않고 PR 머지로만 반영한다

### 2.1. 배포 PR 명명 규칙

`main -> production` 배포 PR을 생성할 때는 히스토리 추적을 위해 아래와 같은 제목 형식을 사용한다.

```text
release: YYYY-MM-DD vX.Y.Z
```

예시: `release: 2026-07-11 v1.10.1`

### 2.2. 배포 브랜치 분리 목적

이러한 배포 구조를 운영하는 핵심 목적은 다음과 같다.

- 문서 수정, 내부 정리, 실험용 커밋이 `main`에 들어가도 즉시 프로덕션 배포되지 않게 한다
- 실제 운영 반영 시점을 `main -> production` PR 머지로 제한한다
- 프로덕션 반영 전 리뷰와 확인 지점을 하나 더 둔다

## 3. 왜 `production` 브랜치로 배포하는가

기존에는 `main`에 push되면 바로 Cloudflare에서 자동 배포됐다.

이 방식은 작은 팀에서는 빠르지만, 아래 문제가 있다.

- 문서-only 변경도 프로덕션 배포를 트리거한다
- 운영 반영 타이밍과 개발 통합 타이밍이 섞인다
- 스키마 변경이 드물더라도, 배포 시점 통제가 약하다

그래서 현재는 배포 기준 브랜치를 `production`으로 분리했다.

운영 규칙은 아래처럼 본다.

1. 기능 개발과 문서 작업은 `main`에서 통합한다
2. 실제 프로덕션 반영이 필요할 때만 `main -> production` PR을 연다
3. `production` PR 단계에서 migration 검증을 통과한 뒤 머지한다
4. `production` PR 머지 후 Cloudflare가 자동 배포한다
5. `production` PR이 merge되면 production DB migration 워크플로가 자동 실행된다

## 4. 현재 GitHub Actions 구성

현재 관련 워크플로 파일:

- [production-db-migration-check.yml](.github/workflows/production-db-migration-check.yml:1)
- [production-db-migrate.yml](.github/workflows/production-db-migrate.yml:1)

### 4.1. `production` PR 단계 migration 검증

현재 동작은 아래와 같다.

- 트리거
  - `production` 대상 PR 생성/업데이트
  - `workflow_dispatch`
- 대상 DB
  - production DB가 아니라 검증용 D1 DB
  - `wrangler.jsonc`에 검증용 D1 DB 바인딩을 추가해야 함
  - GitHub Actions variable `D1_MIGRATION_CHECK_DATABASE`에 Wrangler 바인딩명 저장
- 실제 실행 순서
  - checkout
  - Node 설정
  - Corepack 활성화
  - `yarn install --immutable`
  - Cloudflare credentials와 검증 DB 이름 확인
  - `yarn wrangler d1 migrations apply "$D1_MIGRATION_CHECK_DATABASE" --remote`

이 워크플로 목적은 `production`에 머지되기 전에 migration이 깨지는지 검증하는 것이다.

### 4.2. `production` PR merge 후 production DB migration 적용

현재 동작은 아래와 같다.

- 트리거
  - `production` 대상 PR이 merge되어 closed 되는 경우
  - `workflow_dispatch`
- 런타임
  - `ubuntu-latest`
  - Node.js 20
  - Yarn 4.10.0 via Corepack
- 실제 실행 순서
  - checkout
  - Node 설정
  - Corepack 활성화
- `yarn install --immutable`
- Cloudflare credentials 존재 확인
- `yarn mig:remote`

production DB 적용 워크플로 핵심은 아래 한 줄이다.

```yaml
- name: Apply remote D1 migrations
  run: yarn mig:remote
```

실패 알림은 기존 Discord 실패 알림 워크플로에서 함께 처리한다.

- [ci-failure-discord.yml](.github/workflows/ci-failure-discord.yml:1)
- 감시 대상 워크플로:
  - `PR CI`
  - `Production DB Migration Check`
  - `Production DB Migration`
- 위 워크플로들 중 하나라도 실패하면 `DISCORD_WEBHOOK_URL` secret으로 Discord 웹훅 알림을 전송한다

## 5. 필요한 GitHub Secrets

GitHub repository secrets에 아래 값을 넣어야 한다.

- `CLOUDFLARE_API_TOKEN`
- `CLOUDFLARE_ACCOUNT_ID`

GitHub Actions variable로 아래 값을 추가해야 한다.

- `D1_MIGRATION_CHECK_DATABASE`

### 5.1. `CLOUDFLARE_API_TOKEN`

용도:

- GitHub Actions에서 Wrangler가 Cloudflare API를 호출할 때 인증에 사용

권장 권한:

- `D1 database - Edit`

현재 목적이 `wrangler d1 migrations apply --remote` 실행뿐이라면, D1 migration 전용 최소 권한으로 시작하는 것이 맞다.

Cloudflare 공식 참고:

- GitHub Actions with Wrangler:
  - https://developers.cloudflare.com/workers/ci-cd/external-cicd/github-actions/
- D1 Wrangler commands:
  - https://developers.cloudflare.com/d1/wrangler-commands/
- D1 migrations reference:
  - https://developers.cloudflare.com/d1/reference/migrations/

### 5.2. `CLOUDFLARE_ACCOUNT_ID`

용도:

- CI 환경에서 Wrangler가 어떤 Cloudflare account를 대상으로 작업할지 지정

이 값은 이메일이 아니다. Cloudflare account의 고유 ID 문자열이다.

Cloudflare 공식 참고:

- GitHub Actions 문서의 Account ID 섹션:
  - https://developers.cloudflare.com/workers/ci-cd/external-cicd/github-actions/#cloudflare-account-id
- Wrangler system environment variables:
  - https://developers.cloudflare.com/workers/wrangler/system-environment-variables/
- Account ID 찾는 방법:
  - https://developers.cloudflare.com/fundamentals/account/find-account-and-zone-ids/

조회 위치는 보통 아래 둘 중 하나다.

1. Cloudflare Dashboard -> `Workers & Pages` -> `Account details` -> `Account ID`
2. Cloudflare Dashboard -> `Account home` -> 계정 메뉴 -> `Copy account ID`

### 5.3. `D1_MIGRATION_CHECK_DATABASE`

용도:

- `production` PR 단계 migration 검증에 사용할 별도 D1 데이터베이스의 Wrangler 바인딩명

주의:

- production DB 바인딩 `DB`를 넣으면 안 된다
- PR 검증용 또는 staging 성격의 별도 D1 DB를 `wrangler.jsonc`에 추가한 뒤 그 바인딩명을 사용해야 한다

## 6. Cloudflare 쪽 구성

현재 전제는 아래와 같다.

- Cloudflare Workers Builds의 Git integration 배포 브랜치를 `production`으로 설정
- `main`은 자동 배포 대상이 아님
- 프로덕션 배포는 `production` 브랜치 업데이트로만 발생

공식 참고:

- Cloudflare GitHub integration:
  - https://developers.cloudflare.com/workers/ci-cd/builds/git-integration/github-integration/

## 7. 현재 구조의 중요한 한계

지금 구조에서 GitHub Actions migration과 Cloudflare Git integration 배포는 서로 다른 시스템이다.

즉, 현재는 아래를 보장하지 못한다.

- migration 성공 후에만 배포
- migration 실패 시 배포 자동 중단

정확히 말하면 현재 구성은:

1. `production` PR에서 검증용 D1 DB에 migration check 실행
2. 머지 후 `production`에 push 발생
3. Cloudflare Git integration이 배포를 시작
4. `production` PR merge를 트리거로 GitHub Actions가 production DB에 `yarn mig:remote`를 실행

따라서 production DB migration은 PR 게이트는 아니지만, `production` merge 직후 자동 실행되는 후속 단계다.

이 점 때문에 아래 원칙이 필요하다.

- 파괴적 schema 변경은 여전히 수동 판단이 필요하다
- 가능하면 migration은 expand/contract 방식으로 유지한다
- migration이 포함된 배포는 PR에서 한 번 더 검토한다

## 8. 현재 운영 규칙

- `main`에서 개발과 문서 작업을 진행한다
- 프로덕션 반영은 `main -> production` PR로만 진행한다
- `production` PR은 `PR CI`와 `Production DB Migration Check`를 통과해야 머지한다
- `production` 브랜치에는 직접 push하지 않는다
- 검증용 D1 DB migration check는 GitHub Actions `Production DB Migration Check`가 수행한다
- production DB migration 적용은 `production` PR merge 후 GitHub Actions `Production DB Migration`이 자동 수행한다
- 파괴적 schema 변경은 자동화에 전적으로 맡기지 않는다

## 9. 다음 개선 후보

현재보다 더 강한 배포 보장을 원하면 아래 중 하나가 필요하다.

1. Cloudflare Git integration을 끄고 GitHub Actions에서 migration + deploy를 한 파이프라인으로 통합
2. 파괴적 migration만 별도 수동 워크플로로 분리
3. production merge 전 smoke check와 migration 영향 체크리스트를 추가
