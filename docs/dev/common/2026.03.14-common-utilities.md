# 공통 유틸리티 및 에러 처리 규격 (2026.03.14)

## 1. 개요
Dowin 서비스의 백엔드 개발 시 공통적으로 사용되는 응답 규격, 에러 처리 체계 및 세션 관리 유틸리티에 대해 정의합니다. 모든 API 핸들러와 비즈니스 로직은 본 규격을 준수해야 합니다.

## 2. API 응답 (src/lib/api-response.ts)
표준화된 HTTP 응답을 생성하기 위한 헬퍼 함수입니다.

### 2.1. apiSuccess
- **용도**: 성공적인 요청 처리 결과 반환.
- **사용 예시**: `return apiSuccess({ user }, 201);`
- **특징**: `NextResponse.json`을 래핑하며, 리소스 객체나 배열을 직접 반환합니다.

### 2.2. apiError
- **용도**: 에러 발생 시 표준 응답 구조 반환.
- **구조**: `{ error: { code, message, details? } }`
- **사용 예시**: `return apiError("INVALID_CREDENTIALS");`

## 3. 에러 클래스 (src/lib/errors.ts)
비즈니스 로직 내부에서 명확한 의도를 가지고 에러를 발생시키기 위한 커스텀 에러 클래스입니다.

- **PlatformError (Abstract)**: 모든 커스텀 에러의 부모 클래스.
- **BadRequestError (400)**: 요청 파라미터가 잘못된 경우.
- **UnauthorizedError (401)**: 인증 실패 또는 세션 만료.
- **ForbiddenError (403)**: 권한 부족.
- **NotFoundError (404)**: 리소스가 존재하지 않는 경우.
- **ConflictError (409)**: 비즈니스 상태 충돌 (예: 중복 데이터).
- **ValidationError (422)**: 유효성 검사 실패.

## 4. 에러 핸들링 래퍼 (src/lib/with-error-handler.ts)
Route Handler의 최상위를 감싸는 고차 함수입니다.

- **역할**: 핸들러 내부에서 발생한 `PlatformError`를 포착하여 `apiError`로 자동 변환합니다. 정의되지 않은 예기치 못한 에러는 `INTERNAL_ERROR`로 처리하고 서버 로그에 기록합니다.
- **사용 예시**: 
  ```typescript
  export const POST = withErrorHandler(async (req, { env }) => { ... });
  ```

## 5. 인증 및 세션 (src/lib/auth.ts)
사용자 세션 정보를 조회하기 위한 유틸리티입니다.

- **getSession**: 쿠키에서 `dowin_sid`를 읽어 DB(D1)에서 유효한 세션인지 검증하고 사용자 ID(`userId`)를 반환합니다.

## 6. 개발 규정
1. 모든 API Route Handler는 `withErrorHandler`로 감싸야 합니다.
2. 비즈니스 로직(Service)에서 발생한 예외 상황은 적절한 `PlatformError` 하위 클래스를 `throw` 합니다.
3. 직접 `NextResponse.json`을 구성하기보다 `apiSuccess`, `apiError` 유틸리티 사용을 권장합니다.
