# 데이터베이스 스키마 설계

이 문서는 현재 구현 기준의 Dowin DB 스키마를 설명한다. 기준 파일은 [`src/db/schema.ts`](/src/db/schema.ts)다. 문서와 코드가 다르면 코드를 우선한다.

## 1. 현재 스키마의 역할

DOWIN의 DB는 크게 아래 책임을 가진다.

- 계정과 세션 관리
- 워크스페이스와 멤버십 관리
- 점수판, 선행지표, 일일 기록 저장
- 워크스페이스 공용 태그와 선행지표-태그 연결 저장
- 워크스페이스 초대코드 관리
- 팀 대시보드 메모 저장
- 푸시 구독 저장

현재 분석 export는 기존 도메인 데이터를 조합해 생성하며, 별도 analytics 전용 테이블은 없다.

## 2. 설계 원칙

- DB 타깃은 Cloudflare D1(SQLite)이다.
- 실제 스키마 정의는 Drizzle의 [`src/db/schema.ts`](/src/db/schema.ts)에 있다.
- 시간값은 대체로 Unix timestamp 정수(`integer(..., { mode: "timestamp" })`)로 저장한다.
- 날짜 키는 `start_date`, `end_date`, `log_date`처럼 텍스트 기반 canonical date 문자열로 저장한다.
- 부모 삭제 시 자식이 정리되어야 하는 경로는 `onDelete: "cascade"`를 사용한다.
- 소유권/중복 방지와 직접 연결되는 제약은 DB 레벨에서도 최대한 잡는다.

| 영역         | 테이블                       | 역할                                                              |
| ------------ | ---------------------------- | ----------------------------------------------------------------- |

| 인증/계정    | `sessions`                   | `dowin_sid` 세션 쿠키와 연결되는 서버 세션을 저장                   |

| 부가 기능    | `user_notification_settings` | 사용자별 일일 리마인드 시간/활성 상태를 저장                      |

| 결제/청구    | `billing_provider_products`  | 플랜 코드와 외부 결제 provider 상품 ID 매핑을 저장                |
| 결제/청구    | `billing_plan_limits`        | 플랜별 제품 제한 정책을 저장                                      |

현재 구현에 아직 없는 후속 후보 테이블:

- `workspace_notification_settings`
- `leader_reminder_deliveries`

운영 콘솔(`dowin-admin.dasoslab.com`) 관련 테이블은 현재 기획 확정 단계이며, 구현 시 이 문서와 `src/db/schema.ts`를 함께 갱신해야 한다.

## 4. 현재 테이블 정리

### 4.1. `users`

| 컬럼             | 설명                 |
| ---------------- | -------------------- |
| `id`             | 사용자 PK            |
| `custom_id`      | 로그인용 사용자 ID   |
| `password_hash`  | bcrypt 비밀번호 해시 |
| `nickname`       | 표시용 닉네임        |
| `avatar_key`     | preset avatar 식별자 |
| `is_first_login` | 최초 로그인 여부     |
| `locale`         | 사용자 locale        |
| `created_at`     | 생성 시각            |

제약 / 특성:

- `custom_id` unique
- `avatar_key` nullable
- `locale` 기본값 `ko`

### 4.2. `auth_recovery_codes`

| 컬럼         | 설명                    |
| ------------ | ----------------------- |
| `id`         | 복원코드 PK             |
| `user_id`    | 복원코드 소유 사용자 ID |
| `code_hash`  | 복원코드 해시           |
| `used_at`    | 복원코드 사용 시각      |
| `created_at` | 생성 시각               |

제약 / 특성:

- 복원코드 원문 미저장
- `code_hash` unique
- `user_id -> users.id` cascade

### 4.3. `sessions`

| 컬럼         | 설명                |
| ------------ | ------------------- |
| `id`         | 세션 ID             |
| `user_id`    | 세션 소유 사용자 ID |
| `expires_at` | 세션 만료 시각      |
| `created_at` | 생성 시각           |

제약 / 특성:

- `id` text PK
- `user_id -> users.id` cascade
- `dowin_sid` 세션과 연결

### 4.3.a. `auth_login_attempts`

| 컬럼             | 설명                                 |
| ---------------- | ------------------------------------ |
| `id`             | 로그인 시도 레코드 PK                |
| `custom_id`      | 로그인 시도 대상 아이디              |
| `ip_address`     | 로그인 시도 IP                       |
| `failure_count`  | 현재 윈도우 내 누적 실패 횟수        |
| `first_failed_at`| 현재 윈도우의 첫 실패 시각           |
| `last_failed_at` | 마지막 실패 시각                     |
| `blocked_until`  | 차단 해제 시각                       |
| `created_at`     | 생성 시각                            |
| `updated_at`     | 갱신 시각                            |

제약 / 특성:

- `(custom_id, ip_address)` unique
- 일반 사용자 로그인 rate limit 상태 저장용
- 현재 구현은 같은 아이디 + 같은 IP 기준 15분 동안 10회 실패 시 15분 차단

### 4.3.a. `admin_users` / `admin_sessions` / `admin_role_grants` (기획 확정, 구현 예정)

운영 콘솔은 서비스 사용자 계정과 분리된 인증 체계를 사용한다.

핵심 원칙:

- 서비스 도메인 `dowin.dasoslab.com`과 운영 도메인 `dowin-admin.dasoslab.com`은 세션을 공유하지 않는다.
- 운영 계정은 `users`를 재사용하지 않고 별도 `admin_users`를 둔다.
- 운영 권한은 `admin_users.role` 단일 컬럼 대신 `admin_role_grants`로 관리한다.
- 운영 세션은 원문 토큰 직접 저장 대신 `session_token_hash` 저장을 기본안으로 한다.

예정 테이블 요약:

- `admin_users`
  - `login_id`, `password_hash`, `display_name`, `status`, `mfa_enabled`, `last_login_at`, `created_at`, `updated_at`
- `admin_sessions`
  - `id`, `admin_user_id`, `session_token_hash`, `expires_at`, `created_at`, `last_accessed_at`, `ip_address`, `user_agent`
- `admin_role_grants`
  - `admin_user_id`, `role`, `granted_by_admin_user_id`, `grant_reason`, `granted_at`, `revoked_at`, `revoked_by_admin_user_id`, `revoke_reason`

자세한 초안은 아래 문서를 본다.

- `docs/planning/2026.04.30-admin-dashboard-operation-tool-plan.md`
- `docs/planning/2026.04.20-audit-logs-mvp-schema-plan.md`

### 4.4. `device_push_tokens`

| 컬럼         | 설명                    |
| ------------ | ----------------------- |
| `id`         | 디바이스 토큰 PK        |
| `user_id`    | 토큰 소유 사용자 ID     |
| `provider`   | 푸시 provider (`FCM`)   |
| `platform`   | 플랫폼 (`IOS/ANDROID`)  |
| `token`      | FCM 디바이스 토큰       |
| `notification_enabled` | 현재 토큰 활성 여부 |
| `disabled_at` | 비활성화 시각          |
| `created_at` | 생성 시각               |

제약 / 특성:

- `token` unique
- `user_id -> users.id` cascade
- 현재 기기 로그아웃 또는 invalid token 응답 시 soft disable 처리
- 이유:
  - 워크스페이스 ADMIN 대상 리포트 발송과 리더 개입 이력 추적은 사용자/멤버십 조인이 안전해야 하기 때문
  - 현재처럼 text 식별자를 유지하면 구독 정합성과 소유권 검증을 서비스 코드에 과도하게 의존하게 된다

### 4.5. `workspaces`

| 컬럼                            | 설명                                             |
| ------------------------------- | ------------------------------------------------ |
| `id`                            | 워크스페이스 PK                                  |
| `name`                          | 워크스페이스 이름                                |
| `plan_code`                     | 현재 플랜 코드                                   |
| `billing_customer_external_ref` | provider customer와 workspace 매핑용 외부 참조값 |
| `billing_owner_user_id`         | 현재 billing 관리 책임자 사용자 ID               |
| `created_at`                    | 생성 시각                                        |

제약 / 특성:

- 워크스페이스 기본 정보 테이블
- 기본값은 `FREE`
- 현재 유료 플랜 코드는 `STANDARD`까지 스키마에 반영돼 있다
- append-only billing 도입 이후에도 현재 billing 상태 필드는 `workspaces`에 직접 두지 않는다
- `plan_code`는 기존 플랜 게이트 호환을 위한 캐시 성격을 가진다
- 현재 billing 진실 원천은 `billing_events`, 조회 기준은 `workspace_billing_state`다
- 플랜 게이트는 우선 워크스페이스 단위로만 적용한다
- 이 컬럼을 추가하는 배치에서는 `Workspace` API 응답과 OpenAPI 스키마에도 `planCode`를 같은 배치로 반영한다

### 4.5.a. `billing_events`

| 컬럼                | 설명                                             |
| ------------------- | ------------------------------------------------ |
| `id`                | billing event PK                                 |
| `workspace_id`      | 대상 워크스페이스 ID                             |
| `provider`          | 현재는 `POLAR`                                   |
| `provider_event_id` | provider event id, 없으면 nullable               |
| `event_type`        | 내부 표준 이벤트명 또는 provider 이벤트명        |
| `subscription_key`  | provider subscription 식별자                     |
| `customer_key`      | provider customer 식별자                         |
| `occurred_at`       | provider 기준 이벤트 발생 시각                   |
| `recorded_at`       | DOWIN가 이벤트를 append한 시각                     |
| `payload_json`      | 원본 payload 또는 내부 보정 payload              |
| `status`            | `ACCEPTED`, `IGNORED`, `FAILED`                  |
| `failure_reason`    | 실패 사유 메모                                   |
| `source`            | `WEBHOOK`, `RECONCILIATION`, `MANUAL_CORRECTION` |

제약 / 특성:

- billing 사실 원장 테이블
- append-only로 유지하며 update/delete를 진실 원천으로 사용하지 않는다
- 중복 webhook도 필요 시 `IGNORED` 상태 이벤트로 남길 수 있다
- projection 재생성은 이 테이블 replay만으로 가능해야 한다
- 스키마에는 이미 존재하지만, 실제 Polar webhook 연동/API 연결은 후속 구현 범위다

### 4.5.b. `billing_checkout_events`

| 컬럼                   | 설명                                 |
| ---------------------- | ------------------------------------ |
| `id`                   | checkout event PK                    |
| `workspace_id`         | 대상 워크스페이스 ID                 |
| `requested_by_user_id` | checkout 요청 관리자 사용자 ID       |
| `request_id`           | 클라이언트/서버가 생성한 멱등 키     |
| `target_plan_code`     | 예: `STANDARD`                       |
| `provider`             | 현재는 `POLAR`                       |
| `provider_checkout_id` | provider checkout/session 식별자     |
| `event_type`           | checkout 관련 내부 이벤트명          |
| `occurred_at`          | 이벤트 발생 시각                     |
| `recorded_at`          | DOWIN가 append한 시각                  |
| `payload_json`         | provider 응답 또는 내부 메타 payload |

제약 / 특성:

- checkout 요청/재시도/실패 로그 테이블
- 이 테이블도 append-only로 유지한다
- `status` row를 갱신하는 방식 대신 `CHECKOUT_REQUESTED`, `CHECKOUT_SESSION_CREATED`, `CHECKOUT_FAILED` 같은 후속 이벤트 append로 상태 변화를 표현한다
- request 단위 현재 상태는 projection 또는 `request_id` 기준 최신 이벤트 조회로 계산한다
- 현재 유니크 인덱스는 `(request_id, event_type, recorded_at)`이며, 최종 멱등 제약은 실제 billing 구현 시 재검토가 필요하다

### 4.5.c. `workspace_billing_state`

| 컬럼                     | 설명                                               |
| ------------------------ | -------------------------------------------------- |
| `workspace_id`           | 대상 워크스페이스 ID, PK                           |
| `provider`               | 현재는 `POLAR`                                     |
| `billing_status`         | `NONE`, `ACTIVE`, `CANCELED`, `EXPIRED`, `REVOKED` |
| `plan_code`              | projection 기준 현재 entitlement 플랜 코드         |
| `entitlement_source`     | entitlement의 출처 (`POLAR`, `MANUAL_GRANT` 등)   |
| `customer_key`           | provider customer 식별자                           |
| `subscription_key`       | provider subscription 식별자                       |
| `current_period_end`     | 현재 결제 주기 종료 시각                           |
| `cancel_at_period_end`   | 기간 말 해지 여부                                  |
| `billing_owner_user_id`  | billing 관리 책임자 사용자 ID                      |
| `last_event_id`          | 마지막으로 반영한 billing event ID                 |
| `last_event_occurred_at` | 마지막 반영 이벤트 발생 시각                       |
| `updated_at`             | projection 갱신 시각                               |

제약 / 특성:

- billing 조회용 projection/read model
- 진실 원천이 아니다
- `billing_events` replay로 재생성 가능해야 한다
- `billing_status`와 별도로 entitlement가 왜 열려 있는지(`POLAR`, 수동 grant 등)를 추적한다
- entitlement 판단과 `GET /api/billing/me` 응답은 이 테이블 기준으로 계산하는 편이 적절하다
- 스키마에는 이미 존재하지만, 현재 공개 API는 아직 이 projection을 사용자-facing billing 응답으로 노출하지 않는다

### 4.5.d. `billing_provider_products`

| 컬럼                  | 설명                                      |
| --------------------- | ----------------------------------------- |
| `id`                  | provider product mapping PK               |
| `provider`            | 현재는 `POLAR`                            |
| `environment`         | `sandbox`, `production`                   |
| `plan_code`           | Dowin 플랜 코드, 현재 checkout 대상은 `STANDARD` |
| `provider_product_id` | Polar product ID                          |
| `is_active`           | checkout 생성에 사용할 활성 매핑 여부     |
| `created_at`          | 생성 시각                                 |
| `updated_at`          | 수정 시각                                 |

제약 / 특성:

- 가격/멤버 상한 같은 제품 정책은 이 테이블에 저장하지 않는다.
- checkout은 서버가 결정한 `plan_code` 기준으로 active provider product id를 조회한다.
- 사용자 요청에서 provider product id를 직접 받지 않는다.
- `provider`, `environment`, `plan_code` 조합은 하나의 매핑만 유지한다.

### 4.5.e. `billing_plan_limits`

| 컬럼           | 설명                              |

## 5. 관계 요약

핵심 관계는 아래와 같다.

- `users -> sessions` : 1:N
- `users -> auth_recovery_codes` : 1:N
- `users -> workspace_members` : 1:N
- `workspaces -> workspace_members` : 1:N
- `workspaces -> workspace_invites` : 1:N
- `workspaces -> workspace_tags` : 1:N
- `users -> workspace_invites(created_by)` : 1:N
- `users -> workspace_tags(created_by)` : 1:N
- `users -> scoreboards` : 1:N
- `workspaces -> scoreboards` : 1:N
- `scoreboards -> lead_measures` : 1:N
- `lead_measures -> lead_measure_tags` : 1:N
- `workspace_tags -> lead_measure_tags` : 1:N
- `lead_measures -> daily_logs` : 1:N
- `workspaces -> team_memos` : 1:N
- `users -> team_memos(author/target/resolver)` : 1:N
- `users -> user_notification_settings` : 1:N 관계 정의가 있으나, 현재 unique 제약으로 실질 사용은 사용자당 1개다
- `workspaces -> billing_events` : 1:N
- `workspaces -> billing_checkout_events` : 1:N
- `workspaces -> workspace_billing_state` : 1:1
- `users -> workspace_billing_state(billing_owner)` : 1:N

## 6. 현재 스키마에서 중요한 제약

### 6.1. 단일 워크스페이스 소속

현재 구현은 `workspace_members.user_id` 단독 unique 인덱스를 둔다. 즉 사용자 1명이 동시에 여러 워크스페이스에 속하는 구조가 아니다.

이 제약은 문서 초기 버전의 “M:N 관계 지원” 설명과 다르다. 현재 활성 코드 기준으로는 단일 워크스페이스 모델이 맞다.

### 6.2. 단일 활성 점수판

`scoreboards`는 `(user_id, workspace_id)`에 대해 `status = 'ACTIVE'`인 경우만 unique 인덱스를 강제한다.

즉:

- 활성 점수판은 사용자당 1개
- 보관 점수판은 여러 개 가능

### 6.3. 일일 기록 Upsert 기반 중복 방지

`daily_logs`는 `(lead_measure_id, log_date)` unique 인덱스를 사용한다. 같은 선행지표/날짜에 중복 row를 허용하지 않는다.

### 6.4. 복원코드 원문 미저장

복원코드는 `auth_recovery_codes.code_hash`로만 저장된다. 원문 코드는 가입 시 1회 노출 후 서버에 평문으로 남기지 않는다.

## 7. 신규 기능에서 스키마 설계 시 체크할 것

신규 기능에서 persisted data가 필요하면 구현 전에 아래를 먼저 결정한다.

- 새 테이블이 필요한지, 기존 테이블 확장으로 충분한지
- 소유권 경계가 `user_id` 기준인지 `workspace_id` 기준인지
- nullable 컬럼인지 필수 컬럼인지
- unique 인덱스가 필요한지
- 조회 경로를 위한 인덱스가 필요한지
- 부모 삭제 시 cascade / set null / restrict 중 무엇이 맞는지
- API 응답 구조보다 먼저 DB에서 어떤 상태 전이를 보장해야 하는지

기준 파일:

- [`src/db/schema.ts`](/src/db/schema.ts)
- [`drizzle/`](/drizzle)
- 관련 storage/service 코드
