# 데이터베이스 스키마 설계

이 문서는 현재 구현 기준의 WIG DB 스키마를 설명한다. 기준 파일은 [`src/db/schema.ts`](/src/db/schema.ts)다. 문서와 코드가 다르면 코드를 우선한다.

## 1. 현재 스키마의 역할

WIG의 DB는 크게 아래 책임을 가진다.

- 계정과 세션 관리
- 워크스페이스와 멤버십 관리
- 점수판, 선행지표, 일일 기록 저장
- 워크스페이스 공용 태그와 선행지표-태그 연결 저장
- 워크스페이스 초대코드 관리
- 팀 대시보드 메모 저장
- 푸시 구독 저장

현재 분석 export는 기존 도메인 데이터를 조합해 생성하며, 별도 analytics 전용 테이블은 없다.

## 2. 설계 원칙

- DB 타깃은 Cloudflare D1(SQLite)이다.
- 실제 스키마 정의는 Drizzle의 [`src/db/schema.ts`](/src/db/schema.ts)에 있다.
- 시간값은 대체로 Unix timestamp 정수(`integer(..., { mode: "timestamp" })`)로 저장한다.
- 날짜 키는 `start_date`, `end_date`, `log_date`처럼 텍스트 기반 canonical date 문자열로 저장한다.
- 부모 삭제 시 자식이 정리되어야 하는 경로는 `onDelete: "cascade"`를 사용한다.
- 소유권/중복 방지와 직접 연결되는 제약은 DB 레벨에서도 최대한 잡는다.

## 3. 엔티티 개요

| 영역         | 테이블                            | 역할                                                              |
| ------------ | --------------------------------- | ----------------------------------------------------------------- |
| 인증/계정    | `users`                           | 로그인 ID, 비밀번호 해시, 닉네임, avatar, 최초 로그인 여부를 저장 |
| 인증/계정    | `sessions`                        | `wig_sid` 세션 쿠키와 연결되는 서버 세션을 저장                   |

| 워크스페이스 | `workspaces`                      | 팀/모임 단위 공간을 저장                                          |
| 워크스페이스 | `workspace_members`               | 사용자와 워크스페이스의 소속 및 역할을 저장                       |
| 워크스페이스 | `workspace_invites`               | 초대코드, 사용 횟수, 활성 상태를 저장                             |
| 워크스페이스 | `workspace_tags`                  | 워크스페이스 공용 태그 사전을 저장                                |

| 핵심 4DX     | `lead_measures`                   | 점수판 하위 실행 지표를 저장                                      |
| 핵심 4DX     | `lead_measure_tags`               | 선행지표와 태그의 연결을 저장                                     |
| 핵심 4DX     | `daily_logs`                      | 선행지표별 날짜 기록을 저장                                       |
| 부가 기능    | `team_memos`                      | 팀 대시보드 메모를 저장                                           |
| 부가 기능    | `push_subscriptions`              | 웹 푸시 구독 endpoint와 키를 저장                                 |
| 부가 기능    | `user_notification_settings`      | 사용자별 일일 리마인드 시간/활성 상태를 저장                      |
| 부가 기능    | `workspace_notification_settings` | 워크스페이스별 리더 주간 리포트 설정을 저장                       |
| 부가 기능    | `leader_reminder_deliveries`      | 리더가 보낸 팀원 리마인드 이력을 저장                             |

## 4. 현재 테이블 정리

### 4.1. `users`

| 컬럼             | 설명                 |
| ---------------- | -------------------- |
| `id`             | 사용자 PK            |
| `custom_id`      | 로그인용 사용자 ID   |
| `password_hash`  | bcrypt 비밀번호 해시 |
| `nickname`       | 표시용 닉네임        |
| `avatar_key`     | preset avatar 식별자 |
| `is_first_login` | 최초 로그인 여부     |
| `created_at`     | 생성 시각            |

제약 / 특성:

- `custom_id` unique
- `avatar_key` nullable

### 4.2. `auth_recovery_codes`

| 컬럼         | 설명                    |
| ------------ | ----------------------- |
| `id`         | 복원코드 PK             |
| `user_id`    | 복원코드 소유 사용자 ID |
| `code_hash`  | 복원코드 해시           |
| `used_at`    | 복원코드 사용 시각      |
| `created_at` | 생성 시각               |

제약 / 특성:

- 복원코드 원문 미저장
- `code_hash` unique
- `user_id -> users.id` cascade

### 4.3. `sessions`

| 컬럼         | 설명                |
| ------------ | ------------------- |
| `id`         | 세션 ID             |
| `user_id`    | 세션 소유 사용자 ID |
| `expires_at` | 세션 만료 시각      |
| `created_at` | 생성 시각           |

제약 / 특성:

- `id` text PK
- `user_id -> users.id` cascade
- `wig_sid` 세션과 연결

### 4.4. `push_subscriptions`

| 컬럼         | 설명                    |
| ------------ | ----------------------- |
| `id`         | 푸시 구독 PK            |
| `user_id`    | 구독 소유 사용자 식별값 |
| `endpoint`   | 웹 푸시 endpoint        |
| `p256dh`     | 웹 푸시 공개키          |
| `auth`       | 웹 푸시 auth 키         |
| `created_at` | 생성 시각               |

제약 / 특성:

- `endpoint` unique
- 현재 `user_id`는 text이며 FK 없음
- 다음 리더 리포트/리더 리마인드 단계에 들어가기 전, `user_id`를 `integer` FK(`users.id`)로 정리하는 마이그레이션이 선행되어야 한다
- 이유:
  - 워크스페이스 ADMIN 대상 리포트 발송과 리더 개입 이력 추적은 사용자/멤버십 조인이 안전해야 하기 때문
  - 현재처럼 text 식별자를 유지하면 구독 정합성과 소유권 검증을 서비스 코드에 과도하게 의존하게 된다

### 4.5. `workspaces`

| 컬럼         | 설명              |
| ------------ | ----------------- |
| `id`         | 워크스페이스 PK   |
| `name`       | 워크스페이스 이름 |
| `plan_code`  | 현재 플랜 코드    |
| `created_at` | 생성 시각         |

제약 / 특성:

- 워크스페이스 기본 정보 테이블
- MVP에서는 별도 `workspace_plans` 테이블보다 `workspaces.plan_code` 단일 컬럼으로 시작한다
- 기본값은 `FREE`
- 후속 유료 플랜 코드는 `STANDARD` 기준으로 검토 중이다
- 결제/청구 상태가 생기기 전까지는 `plan_status`, `expires_at` 같은 필드를 추가하지 않는다
- 플랜 게이트는 우선 워크스페이스 단위로만 적용한다
- 이 컬럼을 추가하는 배치에서는 `Workspace` API 응답과 OpenAPI 스키마에도 `planCode`를 같은 배치로 반영한다

### 4.5.a. `user_notification_settings`

| 컬럼                     | 설명                    |
| ------------------------ | ----------------------- |
| `id`                     | 설정 PK                 |
| `user_id`                | 대상 사용자 ID          |
| `daily_reminder_enabled` | 일일 리마인드 활성 여부 |
| `daily_reminder_hour`    | 발송 시(hour)           |
| `daily_reminder_minute`  | 발송 시(minute)         |
| `timezone`               | 시간대 문자열           |
| `created_at`             | 생성 시각               |
| `updated_at`             | 수정 시각               |

제약 / 특성:

- `user_id -> users.id` cascade
- 사용자당 1개 설정 row를 unique로 유지
- 현재 시간대는 `Asia/Seoul`만 사용

### 4.5.b. `workspace_notification_settings` (planned)

| 컬럼                               | 설명                       |
| ---------------------------------- | -------------------------- |
| `id`                               | 설정 PK                    |
| `workspace_id`                     | 대상 워크스페이스 ID       |
| `leader_weekly_report_enabled`     | 리더 주간 리포트 활성 여부 |
| `leader_weekly_report_day_of_week` | 발행 요일(1=월, 7=일)      |
| `leader_weekly_report_hour`        | 발행 시(hour)              |
| `leader_weekly_report_minute`      | 발행 시(minute)            |
| `timezone`                         | 시간대 문자열              |
| `created_at`                       | 생성 시각                  |
| `updated_at`                       | 수정 시각                  |

제약 / 특성:

- `workspace_id -> workspaces.id` cascade
- 워크스페이스당 1개 설정 row를 unique로 유지
- ADMIN이 여러 명이어도 동일한 요일/시간 설정을 공유
- 실제 발송 대상은 해당 워크스페이스의 웹 푸시 구독 ADMIN 전체
- MVP에서는 `timezone`을 `Asia/Seoul` 기본값으로 저장하되, 외부 입력으로는 아직 열지 않는다
- 리더 리포트 설정 자체는 `FREE`에서도 허용한다
- 선행 조건:
  - `push_subscriptions.user_id`가 `users.id` FK로 정리되어 있어야 한다

### 4.5.c. `leader_reminder_deliveries` (planned)

| 컬럼              | 설명                       |
| ----------------- | -------------------------- |
| `id`              | 발송 이력 PK               |
| `sender_user_id`  | 발송한 리더 사용자 ID      |
| `scoreboard_id`   | 대상 팀원의 활성 점수판 ID |
| `lead_measure_id` | 대상 선행지표 ID           |
| `template_key`    | 사용한 기본 템플릿 키      |
| `custom_message`  | 사용자 입력 메시지         |
| `delivered_count` | 실제 발송 성공 건수        |
| `created_at`      | 발송 시각                  |

제약 / 특성:

- `sender_user_id -> users.id` cascade
- `scoreboard_id -> scoreboards.id` cascade
- `lead_measure_id -> lead_measures.id` cascade
- `STANDARD` 플랜 리더 리마인드 사용 이력 및 향후 quota 계산의 기준 테이블
- MVP에서는 발송 성공 건수와 최소 발송 메타데이터만 저장하고, 상세 열람/분석 필드는 후속 확장으로 둔다
- `workspace_id`, `target_user_id`는 `scoreboard_id`로부터 유도 가능하므로 중복 저장하지 않는다
- 서비스 계층 필수 검증:
  - 발송자(`sender_user_id`)는 해당 점수판의 워크스페이스 ADMIN이어야 한다
  - `lead_measure_id`는 `scoreboard_id` 소속 선행지표여야 한다
  - 점수판 상태는 `ACTIVE`여야 한다

### 4.5.d. 출시 순서 기준 스키마 변경안

1. `workspaces.plan_code` 추가

## 5. 관계 요약

핵심 관계는 아래와 같다.

- `users -> sessions` : 1:N
- `users -> auth_recovery_codes` : 1:N
- `users -> workspace_members` : 1:N
- `workspaces -> workspace_members` : 1:N
- `workspaces -> workspace_invites` : 1:N
- `workspaces -> workspace_tags` : 1:N
- `users -> workspace_invites(created_by)` : 1:N
- `users -> workspace_tags(created_by)` : 1:N
- `users -> scoreboards` : 1:N
- `workspaces -> scoreboards` : 1:N
- `scoreboards -> lead_measures` : 1:N
- `lead_measures -> lead_measure_tags` : 1:N
- `workspace_tags -> lead_measure_tags` : 1:N
- `lead_measures -> daily_logs` : 1:N
- `workspaces -> team_memos` : 1:N
- `users -> team_memos(author/target/resolver)` : 1:N

## 6. 현재 스키마에서 중요한 제약

### 6.1. 단일 워크스페이스 소속

현재 구현은 `workspace_members.user_id` 단독 unique 인덱스를 둔다. 즉 사용자 1명이 동시에 여러 워크스페이스에 속하는 구조가 아니다.

이 제약은 문서 초기 버전의 “M:N 관계 지원” 설명과 다르다. 현재 활성 코드 기준으로는 단일 워크스페이스 모델이 맞다.

### 6.2. 단일 활성 점수판

`scoreboards`는 `(user_id, workspace_id)`에 대해 `status = 'ACTIVE'`인 경우만 unique 인덱스를 강제한다.

즉:

- 활성 점수판은 사용자당 1개
- 보관 점수판은 여러 개 가능

### 6.3. 일일 기록 Upsert 기반 중복 방지

`daily_logs`는 `(lead_measure_id, log_date)` unique 인덱스를 사용한다. 같은 선행지표/날짜에 중복 row를 허용하지 않는다.

### 6.4. 복원코드 원문 미저장

복원코드는 `auth_recovery_codes.code_hash`로만 저장된다. 원문 코드는 가입 시 1회 노출 후 서버에 평문으로 남기지 않는다.

## 7. 신규 기능에서 스키마 설계 시 체크할 것

신규 기능에서 persisted data가 필요하면 구현 전에 아래를 먼저 결정한다.

- 새 테이블이 필요한지, 기존 테이블 확장으로 충분한지
- 소유권 경계가 `user_id` 기준인지 `workspace_id` 기준인지
- nullable 컬럼인지 필수 컬럼인지
- unique 인덱스가 필요한지
- 조회 경로를 위한 인덱스가 필요한지
- 부모 삭제 시 cascade / set null / restrict 중 무엇이 맞는지
- API 응답 구조보다 먼저 DB에서 어떤 상태 전이를 보장해야 하는지

- [`src/db/schema.ts`](/src/db/schema.ts)
- [`drizzle/`](/drizzle)
- 관련 storage/service 코드
