# 프론트엔드 테스트 전략

## 1. 역할

이 문서는 Dowin 프론트엔드 변경 시 어떤 테스트를 추가해야 하는지 판단하는 기준이다.  
목표는 모든 DOM 세부사항을 고정하는 것이 아니라, 사용자가 실제로 거치는 핵심 흐름과 회귀 위험이 큰 분기를 안정적으로 보호하는 것이다.

프론트엔드 테스트는 아래 상황을 우선 보호한다.

- 로그인, 회원가입, 계정 복구처럼 진입 실패가 치명적인 흐름
- 워크스페이스 생성, 참가, 결제 완료, 구독 필요 화면처럼 라우팅과 권한 분기가 있는 흐름
- 개인/팀 대시보드처럼 서버 상태와 낙관적 업데이트가 결합된 화면
- 점수판 생성/수정/보관처럼 제품 데이터 구조를 바꾸는 화면
- 프로필, 결제, 알림처럼 사용자의 계정/워크스페이스 상태를 바꾸는 화면

정적 약관, 단순 문서 페이지, 링크 모음처럼 상태 변화가 거의 없는 화면은 핵심 흐름 테스트 이후에 보강한다.

## 2. 테스트 단위

### 2.1. Page 테스트

화면 단위 테스트는 사용자가 보는 결과와 주요 상호작용을 검증한다.

우선 검증할 것:

- loading, empty, error, no-workspace 상태
- 권한별 CTA 노출 여부
- submit, toggle, archive, delete, redirect 같은 주요 액션
- query string이나 route param에 따른 화면 분기
- mutation pending 중 버튼/입력 비활성화

피할 것:

- Tailwind class 전체 스냅샷
- 아이콘 SVG 내부 구조
- 구현상 우연히 생긴 wrapper DOM 깊이
- 사용자에게 의미 없는 내부 상태명

### 2.2. Hook 테스트

hook 테스트는 화면보다 상태 전이와 side effect가 핵심일 때 작성한다.

우선 검증할 것:

- Zod 또는 로컬 validation 결과
- TanStack Query invalidate/clear 호출
- toast 메시지
- router push/replace
- API 성공/실패 분기
- 낙관적 업데이트와 실패 롤백

페이지가 로컬 form state, mutation, cache invalidation, navigation side effect를 모두 갖고 있으면 `_hooks`로 분리하고 hook 테스트를 먼저 둔다.

### 2.3. Component 테스트

공통 UI나 도메인 컴포넌트는 재사용되거나 접근성 계약이 있을 때 테스트한다.

우선 검증할 것:

- `label`과 input 연결
- disabled 상태
- role, aria-label, accessible name
- 반복 아이템의 핵심 텍스트/버튼
- callback 호출 계약

공통 UI는 화면 문맥을 몰라도 동작해야 한다. 도메인 컴포넌트는 해당 도메인의 사용자 행동을 기준으로 검증한다.

## 3. 우선순위

테스트를 한 번에 모두 작성하지 못할 때는 아래 순서로 진행한다.

1. Auth 진입: 로그인, 회원가입, 계정 복구, 세션 redirect
2. Workspace 진입: 생성, 참가, 생성 완료 콜백, 워크스페이스 없음 상태
3. Billing/entitlement gate: Basic 미활성 접근 차단, 구독 필요 화면, billing/pricing 상태
4. Dashboard: 개인 기록, 팀 뷰, 기간 이동, 낙관적 업데이트
5. Scoreboard/setup: 생성, 수정, 보관, 재활성화, 액션 아이템 입력 제한
6. Workspace admin: 멤버, 초대, 설정, 권한별 액션
7. Profile: 메인, 언어, 알림, 아바타, 비밀번호, 문의, 탈퇴
8. 저위험 public/static 화면

CSV export와 리더 리포트는 현재 비활성 범위로 두며, 다시 활성화할 때 별도 테스트 계획을 잡는다.

## 4. Mock 기준

### 4.1. API hook

Orval 생성 hook은 테스트에서 직접 네트워크를 타지 않도록 mock한다.

- query hook은 `data`, `isLoading`, `error`, `refetch`처럼 화면이 쓰는 필드만 제공한다.
- mutation hook은 `mutateAsync` 또는 `mutate`와 pending 상태만 제공한다.
- 응답은 실제 API status 분기를 반영해 `{ status, data }` 형태로 둔다.

### 4.2. Router와 navigation

라우팅 side effect는 실제 navigation 대신 mock으로 검증한다.

- `router.push`, `router.replace`
- `useParams`
- `useSearchParams`
- `window.location.href` 또는 `window.location.assign`
- `window.history.replaceState`

redirect 자체가 제품 흐름의 핵심이면 반드시 assertion을 둔다.

### 4.3. Toast와 storage

사용자 피드백은 회귀가 잦으므로 toast 호출을 검증한다.

- 성공/실패 toast type
- API 에러 메시지 fallback
- `sessionStorage` flash toast
- 로그인 직후 intent storage

### 4.4. WebView와 native app

앱 전용 기능은 브라우저 fallback과 앱 환경을 분리해서 테스트한다.

- native app에서는 결제/구독 화면이 차단되는지
- 알림 토글은 native bridge availability, permission, push token 단계를 구분하는지
- 브라우저에서는 앱 전용 기능이 비활성화되거나 안내 문구를 노출하는지

브리지 계약 자체를 바꾸는 경우에는 `frontend-webview` 지침과 app-webview 문서를 함께 갱신한다.

## 5. 작성 패턴

### 5.1. Rendering

공통 provider가 필요하면 `src/test/render.tsx`의 `renderWithProviders`를 사용한다.

```tsx
renderWithProviders(<TargetPage />);
```

TanStack Query cache 검증이 필요하면 `createTestQueryClient()`를 직접 넘긴다.

```tsx
const queryClient = createTestQueryClient();
renderWithProviders(<TargetPage />, { queryClient });
```

### 5.2. 사용자 관점 쿼리

우선순위는 아래 순서를 따른다.

1. `getByRole`
2. `getByLabelText`
3. `getByText`
4. `getByPlaceholderText`
5. `data-testid`

`data-testid`는 실제 accessible name이 없고 테스트 대상이 사용자 텍스트가 아닐 때만 사용한다.

### 5.3. 비동기 검증

API 성공/실패, effect, redirect는 `findBy...` 또는 `waitFor`로 검증한다.

```tsx
await waitFor(() => {
  expect(router.replace).toHaveBeenCalledWith("/workspace-1/dashboard/my");
});
```

단순히 timeout을 기다리는 방식은 사용하지 않는다.

## 6. 검증 명령

작업 중에는 가장 작은 범위부터 실행한다.

```bash
yarn test --run <changed-test-files>
yarn eslint <changed-files>
yarn tsc --noEmit
```

여러 화면 흐름을 건드렸다면 관련 테스트 묶음을 함께 실행한다.

```bash
yarn test --run \
  src/app/_components/LoginPageClient.test.tsx \
  'src/app/[locale]/(protected)/workspace/billing/ProfileBillingPageClient.test.tsx'
```

마지막 전체 확인이 필요한 시점에만 broader gate를 실행한다.

```bash
yarn test --run
yarn lint
yarn tsc --noEmit
```

## 7. 현재 핵심 커버리지 기준

프론트엔드 핵심 흐름은 다음 축을 기본 커버리지로 본다.

- Auth: 로그인, 회원가입, 복원코드 저장, 계정 복구, 세션 redirect
- Workspace: 생성, 참가, 생성 완료 콜백, 워크스페이스 없음 상태
- Billing gate: 운영 화면 접근 차단, 구독 필요 화면, billing/pricing 상태 분기
- Dashboard: 개인/팀 화면, 기간 이동, 기록 상태, 팀 메모
- Scoreboard/setup: 점수판 목록, 생성/수정, 보관, 액션 아이템 입력
- Workspace admin: 멤버, 초대, 설정
- Profile: 메인, 언어, 아바타, 비밀번호, 문의, 탈퇴

새 화면을 추가하거나 위 흐름 중 하나를 바꿀 때는 같은 축에 맞는 page/hook/component 테스트를 함께 추가한다.

## 8. 리뷰 체크리스트

- 사용자가 실제로 보는 텍스트, 버튼, 링크, redirect를 검증했는가?
- loading, empty, error, pending 중 하나라도 빠진 중요한 상태가 있는가?
- 권한 또는 billing 상태에 따른 CTA 노출이 고정되어 있는가?
- mutation 성공 후 cache invalidation 또는 navigation side effect를 확인했는가?
- API 실패 시 사용자 피드백이 검증되어 있는가?
- 접근성 query로 테스트할 수 없는 공통 UI 결함이 숨어 있지 않은가?
- disabled/비활성 범위인 CSV export와 리더 리포트를 건드리지 않았는가?
