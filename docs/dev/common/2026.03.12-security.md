# DOWIN 보안 정책 문서

## 1. 인증 & 세션 (Authentication & Session)

### 세션 쿠키 방식

- **방식**: 서버 사이드 세션 + HttpOnly 쿠키
- **세션 저장소**: Cloudflare D1 — `sessions` 테이블
- **쿠키 속성**:

| 속성       | 값               | 이유                                                   |
| ---------- | ---------------- | ------------------------------------------------------ |
| `HttpOnly` | `true`           | XSS 공격으로 JS에서 쿠키 접근 불가                     |
| `Secure`   | `true`           | HTTPS 전송만 허용                                      |
| `SameSite` | `Lax`            | CSRF 방어. `Strict`는 외부 링크 진입 시 세션 끊김 유발 |
| `Path`     | `/`              | 전체 경로 유효                                         |
| `Max-Age`  | `2592000` (30일) | 세션 유효 기간                                         |

```typescript
// 세션 쿠키 생성 예시
const SESSION_COOKIE = "dowin_sid";

export const setSessionCookie = (response: Response, sessionId: string) => {
  response.headers.append(
    "Set-Cookie",
    `${SESSION_COOKIE}=${sessionId}; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=2592000`,
  );
};
```

### 세션 테이블 (DB 스키마 추가 필요)

```sql
CREATE TABLE sessions (
  id TEXT PRIMARY KEY,           -- nanoid 기반 세션 ID
  user_id TEXT REFERENCES users(id) ON DELETE CASCADE,
  expires_at DATETIME NOT NULL,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_sessions_user ON sessions(user_id);
```

### 세션 생명주기

- **생성**: 로그인 성공 시 `nanoid()` 세션 ID 발급 → D1 저장 → 쿠키 설정
- **검증**: 모든 인증 필요 API에서 쿠키 세션 ID → D1 조회 → 만료 여부 확인
- **재발급**: 잔여 만료 기간이 3일 이하일 때 만료를 30일로 연장(슬라이딩 세션)
- **만료**: `expires_at < NOW()` → 401 반환 + 세션 레코드 삭제
- **로그아웃**: D1에서 세션 레코드 삭제 + 쿠키 만료 처리 (`Max-Age=0`)

## 2. 인가 (Authorization)

### 권한 레벨

| 레벨   | 대상                    | 접근 가능 API             |
| ------ | ----------------------- | ------------------------- |
| Public | 미인증                  | `POST /api/auth/login` 만 |
| Member | 인증됨                  | 본인 리소스 전체          |
| Admin  | 워크스페이스 ADMIN 역할 | 멤버 관리, 초대 코드      |

### 소유권 검증 (Ownership Check)

모든 리소스 접근 시, 해당 리소스가 **요청한 사용자의 것인지** 반드시 검증.

```typescript
// ✅ 올바른 예: user_id 조건을 쿼리에 포함
const scoreboard = await db
  .select()
  .from(scoreboards)
  .where(and(eq(scoreboards.id, id), eq(scoreboards.userId, session.userId)))
  .get();
if (!scoreboard) return error(403);

// ❌ 잘못된 예: ID만으로 조회 후 사후 검증 (IDOR 취약점)
const scoreboard = await db
  .select()
  .from(scoreboards)
  .where(eq(scoreboards.id, id))
  .get();
if (scoreboard.userId !== session.userId) return error(403);
```

> "쿼리 레벨에서 소유권 필터링"이 "가져온 뒤 비교"보다 안전하고 효율적.

## 3. 입력 검증 (Input Validation)

### Zod 스키마 기반 검증

모든 API 요청 바디는 **Zod 스키마**로 파싱. 검증 실패 시 `422 Unprocessable Entity` 반환.

```typescript
import { z } from "zod";

export const LoginSchema = z.object({
  customId: z.string().regex(/^[a-zA-Z0-9]{3,20}$/, "ID는 영문/숫자 3~20자"),
  password: z.string().min(8, "비밀번호는 8자 이상"),
});

// API 라우트에서
const body = await request.json();
const parsed = LoginSchema.safeParse(body);
if (!parsed.success) return error(422, parsed.error.flatten());
```

### ID / PW 유효성 규칙

`docs/dev/auth/2026.03.12-domain-auth.md` 섹션 2-1 참조.

### SQL Injection 방어

- **Drizzle ORM** 사용 시 기본적으로 Prepared Statement 적용됨
- Raw 쿼리(`db.run()`) 사용 시 반드시 `?` 바인딩 파라미터 사용
- 사용자 입력을 SQL 문자열에 직접 삽입하는 행위 **절대 금지**

### XSS 방어

- API는 JSON만 반환 (HTML 렌더링 없음)
- Next.js의 기본 이스케이프 처리에 의존
- `Content-Type: application/json` 헤더 명시

## 4. 비밀번호 보안

| 정책          | 내용                                       |
| ------------- | ------------------------------------------ |
| 해시 알고리즘 | **bcrypt** (cost factor 12)                |
| 평문 저장     | **절대 금지**                              |
| 전송          | HTTPS를 통해서만 전송                      |
| 비교          | `bcrypt.compare()` 사용 (타이밍 공격 방어) |

```typescript
import bcrypt from "bcryptjs";

export const hashPassword = (pw: string) => bcrypt.hash(pw, 12);
export const verifyPassword = (pw: string, hash: string) =>
  bcrypt.compare(pw, hash);
```

## 5. CSRF 방어

세션 쿠키 기반이므로 CSRF 위험이 있으나, 아래 방어 조합으로 MVP 수준 방어:

- `SameSite=Lax` — 외부 사이트에서의 쿠키 전송 차단
- `Content-Type: application/json` 요청만 처리 — 브라우저 폼 기반 CSRF 차단
- Post-MVP: CSRF 토큰 도입 고려

## 6. 환경 변수 & 시크릿 관리

| 변수                           | 설명                                  | 저장소                             |
| ------------------------------ | ------------------------------------- | ---------------------------------- |
| `SESSION_SECRET`               | 세션 ID 서명용 시크릿 (선택적)        | Cloudflare Secret                  |
| `NEXT_PUBLIC_VAPID_PUBLIC_KEY` | Web Push 공개키                       | `.env`, `.dev.vars`, CF Secret     |
| `VAPID_PRIVATE_KEY`            | Web Push 비밀키                       | Cloudflare Secret (절대 커밋 금지) |
| `CRON_SECRET`                  | push internal/manual 호출 보호용 토큰 | Cloudflare Secret / `.dev.vars`    |

- `.env`, `.dev.vars`는 `.gitignore`에 포함 확인 필수
- Cloudflare 프로덕션 배포 시 `wrangler secret put` 명령으로 주입

## 7. 인프라 보안 (Cloudflare)

- **HTTPS only**: Cloudflare에서 자동으로 TLS 처리
- **DDoS 방어**: Cloudflare 기본 제공
- **Rate Limiting**: 로그인 API에 대한 요청 제한 (Post-MVP, Cloudflare Rate Limiting 규칙)
- **Cron 실행 주체**: 일일 푸시는 Cloudflare Cron Trigger가 Worker `scheduled()`에서 직접 실행
- **Internal API 보호**: 수동 재실행 또는 내부 호출용 push 배치 라우트는 `Authorization: Bearer <CRON_SECRET>` 검증으로 보호

```typescript
// /api/internal/* 보호 예시
export const isInternalRequest = (request: Request) => {
  // Cloudflare Cron은 CF-Worker 헤더를 포함
  return request.headers.get("CF-Worker") !== null;
};
```
