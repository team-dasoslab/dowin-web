# Dowin 품질 검증 전략

## 1. 개요

소프트웨어 품질 검증은 단순한 "버그 찾기"가 아니라, **제품이 의도한 대로 동작하고 있음을 지속적으로 보증**하는 과정이다.  
DOWIN는 소규모 서비스이지만, 핵심 비즈니스 규칙(단일 점수판, 미래 기록 금지 등)이 破 될 경우 데이터 신뢰를 잃으므로 품질 검증이 중요하다.

---

## 2. 품질 속성 (Quality Attributes)

ISO/IEC 25010 기반으로, DOWIN에서 중점 관리할 품질 속성:

| 속성 | 의미 | DOWIN에서의 중요도 |
|------|------|----------------|
| **정확성 (Correctness)** | 비즈니스 규칙대로 동작하는가 | ⭐⭐⭐ 최우선 |
| **신뢰성 (Reliability)** | 오류 없이 일관되게 동작하는가 | ⭐⭐⭐ |
| **사용성 (Usability)** | 사용자가 쉽게 사용할 수 있는가 | ⭐⭐⭐ |
| **보안 (Security)** | 인증/인가가 올바르게 동작하는가 | ⭐⭐⭐ |
| **성능 (Performance)** | 대시보드 조회가 빠른가 | ⭐⭐ (소규모이므로 MVP에서는 부차적) |
| **유지보수성 (Maintainability)** | 코드 변경이 쉬운가 | ⭐⭐ |

---

## 3. 테스트 전략 피라미드

```
        ┌─────────┐
        │  E2E    │  ← 소수의 핵심 사용자 시나리오 (Playwright)
        ├─────────┤
        │통합테스트│  ← API 엔드포인트 + DB 레이어 검증 (Vitest + D1)
        ├─────────┤
        │단위테스트│  ← 비즈니스 로직, 유틸 함수 (Vitest)
        └─────────┘
```

**실용적 관점**: 소규모 팀(1인)이므로 과도한 테스트 코드보다 **핵심 비즈니스 규칙**에 집중.

---

## 4. 테스트 유형별 전략

### 4.1. 단위 테스트 (Unit Test)

**대상**: 비즈니스 로직 함수, 유틸리티  
**도구**: Vitest  
**커버리지 목표**: 핵심 도메인 로직 80% 이상

**우선 테스트 대상**

| 모듈 | 테스트 항목 |
|------|------------|
| Auth | 비밀번호 해시/검증, 세션 생성/검증, 비밀번호 유효성 검사 |
| Daily Log | 지난주 이전 수정 거부 로직, 미래 날짜 거부 로직, 주간 달성률 계산, Win 판정 로직 |
| Scoreboard | ACTIVE 단일성 체크, 상태 전이 유효성 |
| LeadMeasure | ARCHIVED 상태의 DailyLog 추가 거부 로직 |
| Workspace | 초대 코드 만료 검증, 중복 가입 방지 |

```typescript
// 예시: Daily Log 미래 날짜 유효성 테스트
describe('validateLogDate', () => {
  it('오늘 날짜는 기록 가능', () => {
    expect(validateLogDate('2026-03-12')).toBe(true);
  });
  it('미래 날짜는 기록 불가', () => {
    expect(validateLogDate('2026-03-13')).toBe(false);
  });
  it('지난주 이전 날짜는 수정 불가', () => {
    expect(validateLogDate('2026-03-01')).toBe(false);
  });
});
```

### 4.2. 통합 테스트 (Integration Test)

**대상**: API 라우트 핸들러 + DB(Cloudflare D1)  
**도구**: Vitest + `@cloudflare/vitest-pool-workers`  
**커버리지 목표**: 모든 API 엔드포인트의 Happy Path + 주요 Error Case

**도메인별 핵심 테스트 케이스**

#### Auth
| 시나리오 | 검증 포인트 |
|----------|------------|
| 유효한 ID/PW로 로그인 | 세션 쿠키 발급, `isFirstLogin` 플래그 |
| 잘못된 PW로 로그인 | 401 반환, 구체적 사유 미노출 |
| 비밀번호 변경 | 변경 후 기존 PW로 로그인 불가 |
| 만료된 세션으로 API 호출 | 401 반환 |

#### Workspace
| 시나리오 | 검증 포인트 |
|----------|------------|
| 초대 코드로 참가 | 멤버십 생성 |
| 만료된 초대 코드로 참가 | 410 반환 |
| 이미 소속된 유저의 재참가 시도 | 409 반환 |

#### Scoreboard
| 시나리오 | 검증 포인트 |
|----------|------------|
| 활성 점수판이 있는데 새 점수판 생성 | 409 반환 |
| 점수판 보관 후 새 점수판 생성 | 성공 |
| ARCHIVED 점수판 수정 시도 | 403 반환 |

#### Daily Log
| 시나리오 | 검증 포인트 |
|----------|------------|
| O/X 토글 Upsert | 동일 날짜 중복 없이 업데이트 |
| 미래 날짜 기록 | 400 반환 |
| ARCHIVED 지표에 기록 시도 | 403 반환 |

---

### 4.3. E2E 테스트 (End-to-End Test)

**대상**: 핵심 사용자 시나리오  
**도구**: Playwright  
**커버리지 목표**: 주요 Happy Path 시나리오 커버

**핵심 시나리오**

| # | 시나리오 | 설명 |
|---|----------|------|
| S1 | 신규 멤버 온보딩 | 로그인 → 초대코드 참가 → 점수판 생성 → 선행지표 추가 |
| S2 | 매일 기록 플로우 | 대시보드 접속 → O/X 토글 → Confetti 확인 |
| S3 | 팀 뷰 확인 | My View → Team View 전환 → 팀원 카드 표시 확인 |
| S4 | 점수판 보관 → 재시작 | Archive → 새 점수판 생성 |
| S5 | 계정 탈퇴 | 비밀번호 확인 → 삭제 → 로그인 불가 확인 |

## 5. 도메인별 검증 체크리스트

### 5.1. 인증 (Auth)
- [ ] 로그인 성공/실패 케이스
- [ ] 최초 로그인 플래그 토스트 노출
- [ ] 비밀번호 변경 유효성 (최소 8자, 영문+숫자)
- [ ] 세션 만료 시 재로그인 유도

### 5.2. 워크스페이스 (Workspace)
- [ ] 초대 코드 생성 및 만료
- [ ] 중복 참가 방지
- [ ] ADMIN 전용 기능 접근 제어

### 5.3. 점수판 (Scoreboard)
- [ ] 단일 ACTIVE 제약 (DB Unique Index + API 레벨 이중 검증)
- [ ] 상태 전이 (Active → Archived만 허용)
- [ ] ARCHIVED 상태의 수정 금지

### 5.4. 선행지표 (Lead Measure)
- [ ] ARCHIVED 지표의 Daily Log 추가 금지
- [ ] 물리 삭제 시 Daily Log Cascade 확인

### 5.5. 일일 기록 (Daily Log)
- [ ] 미래 날짜 기록 금지
- [ ] Upsert 동작 (중복 없음)
- [ ] 낙관적 업데이트 실패 시 롤백

### 5.6. 대시보드 (Dashboard)
- [ ] 주간 집계 정확성 (Mon~Sun)
- [ ] Win 판정 기준 (80% 이상)
- [ ] 빈 상태 분기 처리

### 5.7. 프로필 (Profile)
- [ ] 닉네임 유효성 (2~20자, 특수문자 금지)
- [ ] 계정 탈퇴 시 Cascade Delete 확인
- [ ] ADMIN 유일 시 탈퇴 금지

### 5.8. 알림 (Notification)
- [ ] 구독/해제 토글
- [ ] 만료된 구독 자동 삭제 (410 처리)
- [ ] 오늘 기록 완료 유저 발송 스킵

## 6. 비기능 품질 검증

### 6.1. 보안
- [ ] 타인의 Scoreboard/DailyLog 접근 시 403 반환 (소유권 검증)
- [ ] ADMIN 전용 API에 MEMBER 접근 시 403 반환
- [ ] SQL Injection 방어 (Prepared Statement 사용 확인)
- [ ] 세션 쿠키 및 세션 조회 로직 검증

### 6.2. 성능 (MVP 기준 가벼운 검증)
- [ ] 대시보드 API 응답시간 < 500ms (소규모 팀 기준)
- [ ] Cron 알림 발송 처리 시간 < 30초

### 6.3. 모바일 사용성 (수동 검증)
- [ ] 앱 WebView 환경에서 대시보드 정상 렌더링
- [ ] 주간 점수판 테이블 세로형 레이아웃
- [ ] O/X 토글 탭 영역 충분 (최소 44px × 44px, WCAG 기준)
