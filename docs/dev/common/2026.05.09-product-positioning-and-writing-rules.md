# 제품 포지셔닝 및 문서 작성 규칙

## 1. 이 문서의 역할

- DOWIN를 외부 방법론 소개물이나 도서 기반 파생 제품처럼 설명하지 않기 위한 공통 기준 문서다.
- planning, dev, onboarding, README, agent skill 문서가 같은 방향으로 쓰이도록 기준을 고정한다.

## 2. 제품 정체성

- DOWIN는 팀의 목표 실행과 주간 운영을 관리하는 독립적인 제품이다.
- 핵심 가치는 `기록` 자체가 아니라 `상태 파악 -> 개입 -> 후속 확인`이 이어지는 운영 루프다.
- 제품의 핵심 사용자는 팀 리더이며, 최종 효용은 `감시`나 `평가`가 아니라 `팀 실행력 향상`이다.
- DOWIN는 외부 책이나 방법론에서 영감을 받았을 수는 있지만, 현재 제품 판단 기준과 메커니즘은 DOWIN 고유의 것으로 다룬다.

## 3. 금지 원칙

- 제품을 `4DX 기반`, `책 기반`, `책의 원칙을 옮긴 서비스`처럼 현재형으로 설명하지 않는다.
- 새 문서에서 외부 도서/방법론을 제품 정당화의 1차 근거로 사용하지 않는다.
- 사용자-facing 개념 설명에 `가중목`, `선행지표`, `후행지표`, `책무` 같은 내부 초기 용어를 기본 언어로 사용하지 않는다.
- `책에 따르면`, `원래 방법론상`, `4DX적으로` 같은 서술로 제품 결정을 설명하지 않는다.

## 4. 허용 원칙

- 내부 구현/히스토리 맥락 설명에서 기존 모델명(`scoreboard`, `lead_measure`)을 언급하는 것은 허용한다.
- 과거 문서나 초기 설계 맥락을 설명할 때만 `초기 참고`, `히스토리`, `이전 용어` 수준으로 제한적으로 언급할 수 있다.
- 외부 레퍼런스는 `문제 정의 방식`, `실험 설계`, `운영 원칙` 보강용 참고 자료로만 사용한다.

## 5. 문서 작성 규칙

- 현재형 설명은 항상 DOWIN 고유 언어로 쓴다.
  - 권장: `핵심 목표 / 성공 기준 / 액션 아이템 / 점수판 / 주간 점검`
- planning 문서는 외부 방법론 정렬도가 아니라 아래 기준으로 판단한다.
  - 리더가 더 빨리 상태를 파악하는가
  - 개입 시작 비용이 줄어드는가
  - 후속 변화 확인이 쉬워지는가
  - 제품이 더 독립적인 운영 도구로 보이는가
- dev 문서는 도메인 책임, 데이터 구조, API 계약, UX 규칙을 제품 고유 메커니즘 기준으로 설명한다.
- onboarding/README는 신규 참여자가 `어떤 제품인지`를 바로 이해하도록 써야 하며, 외부 참고 맥락을 먼저 노출하지 않는다.

## 6. 기존 문서 정리 원칙

- 모든 과거 문서를 한 번에 전면 재작성할 필요는 없다.
- 다만 아래 문서는 항상 최신 포지셔닝 기준을 반영해야 한다.
  - `README.md`
  - `docs/onboarding.md`
  - `docs/dev/README.md`
  - 관련 planning 시작 문서
  - 관련 agent skill / planning rules
- 과거 문서에 남아 있는 책/방법론 언급은 히스토리 맥락이면 유지 가능하지만, 새 문서나 갱신 문서에서는 점진적으로 제거한다.

## 7. 레퍼런스 문서 처리

- `docs/ref/book.md` 같은 자료는 제품 정의 문서가 아니라 참고 보관 자료로만 취급한다.
- `docs/ref/*` 문서는 제품 방향을 직접 결정하지 않는다.
- 제품 결정은 현재 구현, 현재 사용자 장면, 현재 우선순위 문서에서 내린다.

## 8. 문서 점검 체크리스트

- 이 문서가 제품을 외부 도서/방법론의 파생물처럼 설명하고 있지 않은가?
- 현재형 제품 설명이 DOWIN 고유 언어로 쓰였는가?
- 외부 레퍼런스가 있다면 참고 자료로만 쓰였는가?
- 사용자 가치가 `기록`이 아니라 `운영 루프` 중심으로 설명되는가?
- 팀 리더가 왜 이 제품을 다시 열어야 하는지가 드러나는가?
