# Profile 백엔드 구현 결과

## 1. 구현 개요

Profile 도메인에서 현재 활성화된 서버 기능은 내 프로필 조회, 닉네임/avatar 변경, 계정 탈퇴다.

## 2. 현재 구현 범위

### 2.1. GET `/api/users/me`
- 세션 쿠키 `dowin_sid` 기반 인증 후 현재 사용자 프로필을 조회한다.
- 응답에는 `id`, `customId`, `nickname`, `avatarKey`, `role`, `workspaceId`, `workspaceName`, `createdAt`이 포함된다.
- 워크스페이스 멤버십이 없으면 역할과 워크스페이스 정보는 `null`로 내려간다.
- `role`은 프로필 응답의 보조 membership 정보이며, active workspace 기준 권한 판정용 계약이 아니다.
  - 워크스페이스별 관리자/멤버 판정은 `GET /api/workspaces/me.role` 또는 `GET /api/workspaces`의 workspace item `role`을 사용한다.
  - 다중 워크스페이스 사용자는 workspace마다 role이 다를 수 있으므로, 프론트에서 `users/me.role`로 현재 워크스페이스 관리 메뉴 노출을 결정하지 않는다.

### 2.2. PUT `/api/users/me`
- 요청 본문은 `profileUpdateSchema`로 검증한다.
- 닉네임은 trim 후 2자 이상 20자 이하, 영문/숫자/한글/공백만 허용한다.
- `avatarKey`는 허용된 preset 목록 또는 `null`만 허용한다.
- 변경 결과는 갱신된 프로필 전체 객체로 반환한다.

### 2.3. DELETE `/api/users/me`
- 요청 본문은 `profileDeleteSchema`로 검증한다.
- 현재 비밀번호가 일치해야만 계정 삭제를 진행한다.
- 사용자가 워크스페이스의 마지막 ADMIN이면 탈퇴를 막는다.
- 삭제 성공 시 세션 쿠키를 제거하고 `204 No Content`를 반환한다.

## 3. 구현 방식

- Route Handler: `src/app/api/users/me/route.ts`
- Service: `src/domain/profile/services/profile.service.ts`
- Storage: `src/domain/profile/storage/profile.storage.ts`
- Validation: `src/domain/profile/validation.ts`

기존 저장소 패턴에 맞춰 `Route -> Service -> Storage` 구조를 유지했다.

## 4. 반영된 변경점

- 닉네임과 avatar 선택이 더 이상 프론트 임시 상태에만 머물지 않고 DB에 저장된다.
- 조회 시 워크스페이스 멤버십과 조합해 프로필 화면에 필요한 최소 정보를 함께 내려준다.

## 5. 현재 제약

- 닉네임 중복 정책은 현재 워크스페이스 내 허용이다.
- 프로필 수정 범위는 현재 닉네임과 avatarKey를 지원한다.
