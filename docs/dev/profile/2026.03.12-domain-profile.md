# Profile 도메인 설계

## 1. 개요

**프로필/설정 도메인**은 사용자 자신의 개인 정보를 관리하는 영역이다.  
`User` 엔티티를 직접 읽고 쓰는 도메인이며, Auth 도메인과 엔티티를 공유한다.

---

## 2. 핵심 규칙 (Business Rules)

| # | 규칙 |
|---|------|
| BR-1 | 닉네임은 2자 이상 20자 이내, 특수문자 없음 |
| BR-2 | 닉네임은 워크스페이스 내 중복 허용 (식별은 user_id로) |
| BR-3 | 비밀번호 변경 시 현재 비밀번호 확인 필수 |
| BR-4 | 프로필 이미지는 사용자 업로드를 받지 않고, 서비스 preset avatar 중 하나를 선택한다 |
| BR-5 | 프로필 이미지 저장값은 URL이 아닌 `avatarKey`이며, 허용된 preset 목록 또는 `null`만 허용한다 |
| BR-6 | 계정 탈퇴 시 현재 비밀번호 확인이 필요하다 |
| BR-7 | 활성 워크스페이스 기준 유일한 `ADMIN`은 계정을 탈퇴할 수 없다 |
| BR-8 | soft delete 된 워크스페이스의 멤버십은 계정 탈퇴 차단 판정에서 제외한다 |
| BR-9 | 계정 탈퇴 전 사용자를 nullable billing owner로 참조하는 워크스페이스/청구 상태는 `null`로 정리한 뒤 user row를 hard delete한다 |

## 3. 마이페이지 구성 요소

| 섹션 | 내용 |
|------|------|
| 프로필 | 아바타, 닉네임, 아이디(custom_id, 읽기 전용), 역할 뱃지(ADMIN/MEMBER) |
| 보안 설정 | 비밀번호 변경 |
| 알림 설정 | 앱 전용 푸시 알림 On/Off (Notification 도메인 연동) |
| 워크스페이스(ADMIN) | 워크스페이스 이름 변경, 멤버 관리, 초대코드 관리 |
| 데이터 | CSV 다운로드 진입 |
| 앱 둘러보기 | 새 기능 모아보기, iPhone 설치 가이드 |
| 계정 관리 | 로그아웃, 서비스 탈퇴 |

## 4. 사용자 흐름 (User Flow)

### 4.1. 닉네임 변경
```
[마이페이지]
  └─ "닉네임 변경" 메뉴 선택
  └─ 현재 구현은 prompt 기반 입력
  └─ 성공 → "닉네임이 변경되었습니다" 토스트
```

### 4.1.a. Preset Avatar 변경
```
[마이페이지]
  └─ "/profile/avatar" 전용 화면 이동
  └─ preset avatar grid에서 썸네일 선택
  └─ 저장 → "프로필 이미지가 변경되었습니다" 토스트
```

### 4.2. 비밀번호 변경
```
[마이페이지 > 보안 설정]
  └─ "비밀번호 변경" 메뉴 선택
  └─ 현재 구현은 prompt 기반 입력
  └─ 저장 → 성공 토스트
```

### 4.3. 계정 탈퇴
```
[마이페이지 > 계정 관리]
  └─ "서비스 탈퇴" 화면 이동
  └─ 현재 비밀번호 입력
  └─ 활성 워크스페이스의 유일한 ADMIN이면 권한 이전 또는 워크스페이스 삭제 안내
  └─ 탈퇴 가능하면 세션 쿠키 삭제 및 계정 hard delete
```

## 5. 데이터 모델

> Profile 도메인은 별도 테이블 없이 `users` 테이블을 직접 사용.

```typescript
// User 테이블 (읽기/쓰기 대상 필드)
interface UserProfile {
  id: string;           // 읽기 전용
  custom_id: string;    // 읽기 전용 (로그인 ID)
  nickname: string;     // 수정 가능
  avatar_key: string | null;   // preset avatar key, null이면 미선택
  // password_hash는 API를 통해서만 변경
}
```

## 6. API 명세

### GET `/api/users/me`

`role`, `workspaceId`, `workspaceName`은 프로필 화면의 보조 정보다.
다중 워크스페이스에서는 사용자가 워크스페이스 A에서는 `ADMIN`, 워크스페이스 B에서는 `MEMBER`일 수 있으므로, 이 응답의 `role`을 현재 active workspace 권한 판정에 사용하지 않는다.
현재 워크스페이스 기준 권한이 필요하면 `GET /api/workspaces/me`의 `role` 또는 `GET /api/workspaces`의 해당 workspace item `role`을 사용한다.

**Response (200 OK)**
```json
{
  "id": "uuid",
  "customId": "john123",
  "nickname": "존",
  "avatarKey": null,
  "role": "ADMIN",
  "workspaceId": "ws_uuid",
  "workspaceName": "우리 모임",
  "createdAt": "2026-03-01T00:00:00Z"
}
```

---

### PUT `/api/users/me`

**Request** — 수정할 필드만 포함
```json
{ "nickname": "새닉네임", "avatarKey": "avatar-03" }
```

**Response (200 OK)**
```json
{
  "nickname": "새닉네임",
  "avatarKey": "avatar-03"
}
```

**Error Cases**
| 상태 코드 | 사유 |
|-----------|------|
| 422 | 닉네임 유효성 규칙 미충족 (2자 미만, 20자 초과, 특수문자 포함) |
| 422 | 허용되지 않은 `avatarKey` 입력 (`null` 또는 preset 목록 외 값) |

---

### DELETE `/api/users/me`

계정 탈퇴 API다. 현재 비밀번호 확인 후 탈퇴를 진행한다.

탈퇴 가능 여부는 활성 워크스페이스 멤버십 기준으로 판단한다.
워크스페이스가 soft delete 된 경우 해당 멤버십은 탈퇴 차단 판정에서 제외한다.

탈퇴 직전에는 user hard delete를 막을 수 있는 nullable 참조를 먼저 정리한다.
현재 구현은 `workspaces.billing_owner_user_id`와 `workspace_billing_state.billing_owner_user_id`를 `null`로 갱신한 뒤 `users` row를 삭제한다.

**Request**
```json
{ "currentPassword": "password123" }
```

**Response**: `204 No Content`

**Error Cases**
| 상태 코드 | 사유 |
|-----------|------|
| 400 | 현재 비밀번호가 일치하지 않음 |
| 403 | 활성 워크스페이스의 유일한 `ADMIN`이라 탈퇴 불가 |
| 422 | `currentPassword` 누락 또는 입력 형식 오류 |

## 7. 엣지 케이스

| 케이스 | 처리 방식 |
|--------|----------|
| 닉네임에 욕설/부적절 단어 포함 | Post-MVP에서 필터링 고려 (MVP는 단순 길이/특수문자만 검증) |
| `avatarKey`가 `null` | 기존 fallback 프로필 아이콘 렌더링 |
| 허용 목록에 없는 avatarKey 입력 | `422` + 유효하지 않은 preset avatar 오류 |
| soft delete 된 워크스페이스에서만 ADMIN인 사용자 | 해당 멤버십을 무시하고 계정 탈퇴 가능 여부를 판단 |
| 활성 워크스페이스의 유일한 ADMIN인 사용자 | 권한 이전 또는 워크스페이스 삭제 전까지 계정 탈퇴 불가 |
| billing owner로 참조 중인 사용자 | 탈퇴 직전 nullable 참조를 `null`로 정리한 뒤 user hard delete |
