# Dowin Partial Data Recovery Guide

## 1. 이 문서가 다루는 상황

아래 조건이 동시에 성립하는 사고를 다룬다.

- 시점 `A`에 데이터가 삭제되거나 손상됐다
- 시점 `B`에 그 사실을 알게 됐다
- `A~B` 사이에 정상 신규 데이터가 많이 쌓였다
- 따라서 운영 DB를 `A`로 통째로 restore하면 안 된다

이 경우의 목표는 간단하다.

- `A~B` 신규 데이터는 보존
- `A`에서 사라진 데이터만 복구

즉, `full restore`가 아니라 `partial recovery` 문제다.

## 2. 일반적인 서비스에서의 표준 방식

보통 서비스는 운영 DB를 바로 되돌리지 않고 아래 순서로 간다.

1. `A` 직전 상태를 별도 DB에 복원
2. 현재 운영 DB와 비교
3. 사라지거나 깨진 row만 추출
4. 충돌 규칙을 적용
5. 복구 SQL 또는 스크립트로 선별 반영

실무에서는 이를 보통 아래처럼 부른다.

- surgical recovery
- selective restore
- point-in-time restore to side database + diff

핵심은 “백업에서 운영 DB를 덮어쓴다”가 아니라 “백업에서 필요한 데이터만 꺼낸다”다.

## 3. 전체 restore를 하면 안 되는 이유

운영 DB를 `A` 시점으로 통째로 restore하면 아래 문제가 생긴다.

- `A~B` 사이 정상 생성된 기록이 사라짐
- 사용자 활동이 많을수록 손실 범위가 커짐
- 사고 복구가 또 다른 데이터 손실 사고가 됨

따라서 `A~B` 데이터가 중요하면 전체 restore는 최후 수단이다.

## 4. 실무 절차

### 4.1. 임시 복원본 만들기

- `A` 직전 상태를 임시 DB 또는 비운영 환경에 복원한다.
- 운영 DB에는 손대지 않는다.

이 복원본의 역할:

- 삭제 전 데이터 원본
- diff 기준점
- 복구 SQL 생성용 소스

### 4.2. 현재 운영 DB와 비교

보통 아래를 비교한다.

- PK 기준으로 없는 row
- 컬럼 값이 달라진 row
- 부모/자식 관계가 같이 깨진 row
- `updated_at`, `deleted_at`, 상태 컬럼 차이

대표 예:

- `users` row는 남아 있는데 관련 `memberships`만 사라짐
- `daily_logs` 일부만 사라지고 `scoreboards`는 남아 있음
- 특정 row는 살아 있지만 컬럼 값이 잘못 바뀜

### 4.3. 복구 후보 추출

보통 아래 방식으로 추출한다.

- 운영 DB에 없는 row만 `INSERT` 후보로 분리
- 값이 잘못 바뀐 row는 `UPDATE` 후보로 분리
- PK 충돌, unique 충돌, 상태 충돌이 있는 row는 수동 검토 목록으로 분리

### 4.4. 충돌 규칙 적용

이 단계가 가장 중요하다.

예:

- `A`에서 사라진 유저를 복구하려는데 `B` 이후 같은 이메일로 새 유저가 생성됨
- `A`에서 삭제된 daily log를 복구하려는데 `B` 이후 사용자가 같은 날짜에 새 기록을 남김
- `A`에서 사라진 workspace membership을 복구하려는데 이후 role 변경이 있었음

따라서 복구 전 아래 규칙이 필요하다.

- PK 충돌 시 수동 검토
- unique key 충돌 시 현재 운영 상태 우선
- `B` 이후 수정 이력이 더 최신이면 자동 덮어쓰기 금지
- 자식 row 복구 전 부모 row 존재 여부 확인

### 4.5. 선별 반영

- 운영 DB에 바로 대량 반영하지 않는다.
- 작은 배치로 넣고 검증한다.
- 반영 후 row count, 핵심 조회, 앱 동작을 확인한다.

## 5. Dowin 기준으로 생각해야 할 복구 단위

DOWIN에서는 아래 단위로 생각하는 편이 현실적이다.

### 사용자 단위

- `users`
- `sessions`
- `notification_subscriptions`

### 워크스페이스 단위

- `workspaces`
- `workspace_memberships`
- 초대 코드 관련 테이블

### 점수판 단위

- `scoreboards`
- `lead_measures`
- `daily_logs`

DOWIN는 관계형 연결이 강하므로, 단일 테이블만 보고 복구하면 정합성이 깨질 수 있다.  
예를 들어 `daily_logs`를 넣기 전에 해당 `lead_measure`와 `scoreboard`가 살아 있는지 먼저 확인해야 한다.

## 6. DOWIN에서 지금 부족한 것

현재 Dowin 운영 문서 기준으로는 아래가 아직 약하다.

### 6.1. soft delete 기준 부재

하드 delete만 있으면 “무엇이 언제 사라졌는지” 찾기 어려워진다.

효과:

- 실수 삭제 복구가 쉬워짐
- 부분 복구 범위를 줄일 수 있음

### 6.2. audit log 부재

최소한 아래 정도는 남길 가치가 크다.

- 누가
- 언제
- 어떤 엔티티를
- 생성/수정/삭제했는지

이게 없으면 A에서 무엇이 사라졌는지 찾는 비용이 커진다.

### 6.3. 장기 보관 export 정책 미정

Time Travel만으로는 보관 기간 밖 사고나 비교 작업이 어렵다.

부분 복구에는 아래가 도움이 된다.

- 일 단위 export 보관
- 위험 작업 전 수동 export
- 복원 리허설

### 6.4. 복구용 검증 쿼리 미정

실제 사고 때는 즉석에서 SQL을 만들기보다, 미리 “무엇을 확인할지” 정해둔 편이 낫다.

예:

- 최근 24시간 생성 row 수
- 테이블별 row count
- 특정 workspace 기준 연결 데이터 개수
- 특정 user 기준 기록 수

## 7. Dowin 권장 운영 방안

### 지금 당장

- 전체 restore와 부분 복구를 명확히 구분한다.
- 위험 작업 전 bookmark 확보 또는 export를 습관화한다.
- 특정 엔티티 단위 검증 쿼리를 미리 정리한다.

### 출시 직후

- D1 export 장기 보관 정책 추가
- 복구용 검증 체크리스트 추가
- workspace, scoreboard, daily log 기준 기본 복구 시나리오 정의

### 다음 단계

- soft delete 후보 테이블 선정
- 최소 audit log 설계
- 부분 복구 리허설 1회

## 8. 사고 시 의사결정 기준

### 전체 restore 쪽이 맞을 때

- 전체 DB 손상
- 사고 후 신규 정상 데이터가 거의 없음
- 빠른 전체 서비스 복구가 최우선

### 부분 복구 쪽이 맞을 때

- 일부 엔티티 또는 일부 기간 데이터만 손상
- 사고 후 정상 데이터가 많이 쌓임
- 특정 사용자, 특정 workspace, 특정 scoreboard 단위로 복구 가능

## 9. 최소 실행 순서

1. 사고 시각 `A`와 인지 시각 `B` 기록
2. 운영 DB 변경 일시 중지 여부 결정
3. `A` 직전 상태를 임시 DB에 복원
4. 운영 DB와 복원본 diff
5. `INSERT`, `UPDATE`, 수동 검토 목록 분리
6. 작은 배치로 반영
7. 핵심 API와 화면 검증
8. 복구 범위와 잔여 리스크 공지
