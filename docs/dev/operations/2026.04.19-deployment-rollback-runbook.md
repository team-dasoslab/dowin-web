# WIG Deployment Rollback Runbook

## 1. 이 문서가 다루는 사고

- 최근 배포 직후 5xx 급증
- 로그인/대시보드/API가 새 버전에서만 실패
- Worker 코드 오류로 서비스가 깨짐
- 배포를 이전 버전으로 되돌리는 것이 가장 빠른 복구책인 상황

이 문서는 DB restore 문서가 아니다.  
Cloudflare Worker rollback은 연결된 D1 리소스 상태를 함께 되돌리지 않는다.

## 2. 선행 원칙

- 배포 rollback은 “코드/설정” 복구다.
- DB schema/data 손상이 있으면 rollback만으로 해결되지 않을 수 있다.
- rollback 전 현재 장애가 code-only인지, schema/data까지 깨졌는지 먼저 본다.

## 3. 즉시 체크리스트

1. 장애 시작 시각 확인
2. 최근 배포 시각 확인
3. 장애가 특정 배포 직후 시작했는지 확인
4. `no such column`, `no such table`, migration 관련 에러가 있는지 확인
5. code rollback만으로 복구 가능한지 판단

## 4. rollback을 우선 검토할 때

- 최근 배포 직후부터 장애 시작
- DB migration은 없었거나, 있어도 이전 코드와 호환 가능
- 에러가 JS/TS 런타임 또는 API 로직 중심임
- 이전 버전이 안정적이었던 것이 명확함

## 5. rollback만 하면 안 될 때

- migration으로 컬럼/테이블 삭제가 이미 반영됨
- 이전 코드가 현재 DB 스키마를 이해하지 못함
- 사고 후 수동 데이터 수정이 들어감
- 배포가 아니라 Cloudflare 플랫폼 장애임

## 6. 사전 확인

### 6.1. Workers Logs 확인

확인할 것:

- uncaught exception
- 5xx 급증 시점
- 특정 API만 실패하는지
- 에러가 schema/data 문제인지

### 6.2. Cloudflare status 확인

- Cloudflare 자체 incident가 있으면 공급자 장애 가능성을 먼저 본다.

### 6.3. DB 호환성 확인

아래 질문에 `yes`여야 rollback 우선이 맞다.

- 이전 Worker 버전이 현재 D1 schema에서도 동작하는가?
- 배포 이후 실행된 migration이 이전 코드와 충돌하지 않는가?

하나라도 애매하면 먼저 DB 사고 여부를 확인한다.

## 7. 실행 절차

### 7.1. 대상 버전 확인

Cloudflare Dashboard 또는 Wrangler에서 이전 안정 버전을 확인한다.

### 7.2. rollback 실행

Cloudflare 공식 절차 기준:

- Dashboard에서 Worker > Deployments > Rollback
- 또는 Wrangler의 `wrangler rollback`

예시:

```bash
yarn wrangler rollback
```

특정 버전 ID를 알고 있으면 그 버전으로 지정한다.

## 8. rollback 직후 검증

1. 로그인 가능
2. `GET /api/workspaces/me` 성공
3. 내 대시보드 조회 성공
4. daily log 쓰기 성공
5. 주요 5xx 감소 확인

복구 안 되면:

- DB 스키마 문제인지 다시 확인
- [2026.04.19-db-restore-runbook.md](/docs/dev/operations/2026.04.19-db-restore-runbook.md) 기준으로 복구 판단

## 9. 기록할 것

- 장애 시작 시각
- rollback 실행 시각
- rollback 대상 버전
- rollback 실행자
- rollback 후 smoke check 결과
- DB 호환성 확인 여부

## 10. 금지 사항

- DB 호환성 확인 없이 기계적으로 rollback
- 공급자 장애 상황에서 무의미한 반복 rollback
- rollback 직후 smoke check 없이 복구 완료 선언
