# Dashboard Worker CPU Limit Incident Note

## 1. Summary

- Cloudflare Workers에서 `Worker exceeded CPU time limit` 오류가 발생했다.
- 영향 경로는 대시보드 조회 계열이다.
  - `GET /api/workspaces/{workspaceId}/dashboard/my`
  - `GET /api/workspaces/{workspaceId}/dashboard/team`
  - `GET /api/workspaces/{workspaceId}/reports/team-weekly`
- 직접 원인은 dashboard API가 필요한 기간보다 넓은 daily logs를 읽고, 서비스 레이어에서 같은 로그를 여러 집계 계산에 반복 스캔하던 구조였다.
- 대응은 D1 schema나 API contract 변경 없이 코드 hot path를 줄이는 방식으로 처리했다.

## 3. Impact

- 사용자가 `/dashboard` 또는 `/dashboard/my`에 진입할 때 5xx, 빈 화면, 반복 retry를 겪을 수 있었다.
- daily log 쓰기 API 자체는 별도 경로라 살아 있을 수 있지만, 대시보드 재조회 실패로 기록이 반영되지 않은 것처럼 보일 수 있었다.
- 오래된 scoreboard, 많은 daily logs, 많은 workspace members, 많은 active lead measures가 있는 계정에서 더 쉽게 재현될 수 있는 형태였다.

## 4. Root Cause

### 4.1. 넓은 로그 조회 범위

- `currentStreak` 계산 때문에 scoreboard 시작일부터 현재 주까지의 과거 로그를 가져올 수 있었다.
- 오래 사용한 계정일수록 조회 범위가 계속 커지는 구조였다.
- Cloudflare Workers CPU budget 안에서 대시보드 집계가 끝나야 하는 현재 환경과 맞지 않았다.

### 4.2. 반복 집계 스캔

- 가져온 daily logs를 주간 달성률, 월간 달성률, 주간 추세, streak 계산에서 반복 `filter()`했다.
- 팀 대시보드에서는 `members x leadMeasures x logs` 형태로 비용이 커질 수 있었다.

## 5. Resolution

### 5.1. 로그 조회 범위 축소

커밋: `3076c661 fix: 대시보드 로그 조회 범위 축소`

- `dashboard/team`
  - 조회 시작일을 `지난주 시작일`과 현재 dashboard 월간/주간 표시 범위 중 더 이른 날짜로 제한했다.
  - `scoreboard.startDate`를 전체 로그 조회 시작일로 쓰지 않게 했다.
- `dashboard/my`
  - 조회 시작일을 `선택 주차 기준 최근 4주`와 `선택 월간 표시 범위` 중 더 이른 날짜로 제한했다.
  - `currentStreak`는 요청에서 가져온 bounded window 안에서만 계산하게 했다.

### 5.2. 집계 반복 스캔 축소

커밋: `76d444cf fix: 대시보드 집계 반복 스캔 축소`

- 요청에서 받은 logs를 한 번 순회해서 지표별 통계 맵을 만들었다.
- 통계 맵은 아래 값을 가진다.
  - 날짜별 cell: 대시보드 표 렌더링용
  - 달성 날짜 목록: 월간 범위 집계용
  - weekStart별 달성 수: 주간/추세/streak 집계용
- `dashboard/team`, `dashboard/my`, `team-weekly-report`, `team-trend`가 같은 request 안에서 이 통계 맵을 재사용하게 했다.

## 7. Tradeoffs

- `currentStreak`가 전체 과거 기준의 완전한 장기 streak가 아닐 수 있다.
- 대신 dashboard 조회 비용이 scoreboard 사용 기간에 비례해서 계속 증가하지 않는다.
- 전체 과거 기준 streak가 제품적으로 중요해지면 request-time 전체 스캔이 아니라 persisted aggregate로 분리해야 한다.

## 9. Recurrence Runbook

1. 최근 배포 이후 시작됐는지 확인한다.
   - `production` 머지 시각
   - Cloudflare Workers 배포 버전
   - 최근 dashboard 관련 커밋
2. Workers Logs에서 실패 경로를 분리한다.
   - `/dashboard/my`만 실패하는지
   - `/dashboard/team`도 실패하는지
   - 특정 workspace 또는 오래된 scoreboard에 집중되는지
3. DB 손상 사고로 오판하지 않는다.
   - CPU limit은 우선 code hot path 문제로 본다.
   - D1 restore는 데이터 손상 증거가 있을 때만 검토한다.
4. 최근 배포가 명확한 원인이면 Worker rollback을 검토한다.
   - rollback은 D1 schema/data를 되돌리지 않는다.
   - 이번 유형처럼 schema 변경 없이 dashboard service 로직 변경이면 code rollback만으로 복구 가능한 편이다.
5. 운영 배포 후 아래를 확인한다.
   - 오래된 scoreboard를 가진 계정으로 `/dashboard/my` 진입
   - 팀원이 여러 명인 workspace에서 `/dashboard` 진입
   - 주간 보기와 월간 보기 전환
   - daily log 토글 후 대시보드 재조회
   - Workers Logs에서 CPU limit 재발 여부

## 10. Prevention Rules

- dashboard, report, weekly/monthly summary 경로는 전체 과거 로그 조회를 기본값으로 두지 않는다.
- `scoreboard.startDate`를 로그 조회 시작일로 직접 쓰는 변경은 performance review 대상이다.
- `members x leadMeasures x logs` 형태의 nested scan은 지표별 map 또는 aggregate로 먼저 줄인다.
- 새 dashboard 응답 필드가 장기 히스토리를 요구하면 아래 중 하나를 먼저 설계한다.
  - 별도 persisted aggregate
  - bounded window 정책
  - 페이지네이션 또는 lazy fetch
  - batch query with grouped result
