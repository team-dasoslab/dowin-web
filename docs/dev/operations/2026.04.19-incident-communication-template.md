# WIG Incident Communication Template

## 1. 원칙

- 기술 원인을 확신하기 전에는 단정하지 않는다.
- 사용자가 알아야 할 것은 “현재 영향”, “우리가 하고 있는 조치”, “다음 업데이트 시각”이다.
- 내부 공유와 외부 공지를 분리한다.
- 복구 전에는 장문의 원인 분석보다 짧고 자주 업데이트하는 편이 낫다.

## 2. 내부 사고 채널 템플릿

```text
[Incident Open]
시각:
등급: P1 | P2 | P3
현재 영향:
처음 감지 경로:
최근 배포 여부:
Cloudflare status 확인 결과:
현재 담당:
- IC:
- Ops:
- Comms:
다음 업데이트 예정 시각:
```

## 3. 첫 사용자 공지 템플릿

```text
현재 WIG 일부/전체 기능에 장애가 발생해 확인 중입니다.

- 영향 범위:
- 확인 시작 시각:
- 현재 조치:
- 다음 안내 예정 시각:

불편을 드려 죄송합니다. 확인되는 내용은 이 공지에 계속 업데이트하겠습니다.
```

## 4. 읽기 전용 또는 일부 기능 제한 공지 템플릿

```text
현재 장애 대응 과정에서 일부 기능을 일시 제한하고 있습니다.

- 정상 사용 가능:
- 일시 제한 기능:
- 제한 시작 시각:
- 목적: 추가 오류 확산 방지 및 데이터 보호
- 다음 안내 예정 시각:
```

예:

- 조회는 가능하나 기록 저장은 일시 중단
- 로그인은 가능하나 팀 대시보드는 지연

## 5. 공급자 장애 공지 템플릿

```text
현재 WIG가 사용 중인 외부 인프라 제공자 측 장애 영향으로 일부 기능이 불안정할 수 있습니다.

- 영향 범위:
- 외부 상태 확인 시각:
- 현재 WIG 측 조치:
- 다음 안내 예정 시각:

제공자 복구 상황을 계속 확인하며 업데이트하겠습니다.
```

주의:

- Cloudflare 장애가 의심돼도 내부 배포 문제를 배제하기 전에는 단정 문구를 쓰지 않는다.

## 6. 복구 공지 템플릿

```text
장애 대응이 완료되어 현재 WIG 주요 기능이 다시 정상 동작하고 있습니다.

- 영향 시간:
- 영향 범위:
- 완료 시각:
- 확인한 정상 기능:

이번 장애 원인과 후속 조치는 정리 후 별도로 공유하겠습니다.
```

## 7. 사후 공유 템플릿

```text
[Incident Review]
사고 일시:
영향 범위:
사용자 영향:
즉시 원인:
근본 원인:
대응 타임라인:
- 시각 / 조치
- 시각 / 조치
잘 된 점:
아쉬운 점:
재발 방지 액션:
- 담당자 / 기한 / 항목
```

## 8. 운영 중 문구 가이드

### 써야 하는 표현

- 확인 중입니다
- 영향 범위를 파악하고 있습니다
- 일부 기능을 일시 제한했습니다
- 다음 업데이트는 XX:XX에 드리겠습니다

### 피해야 하는 표현

- 곧 해결됩니다
- 원인은 확실히 XX입니다
- 데이터 손실은 없을 것입니다
- 완전히 정상화됐습니다

확인 전 확정 문구는 이후 신뢰를 크게 깎는다.

## 9. 다음 업데이트 주기 기준

- P1: 15~30분마다
- P2: 30~60분마다
- P3: 의미 있는 변경이 있을 때

변화가 없어도 “아직 확인 중이며 다음 업데이트 시각은 그대로다”를 남기는 편이 낫다.
