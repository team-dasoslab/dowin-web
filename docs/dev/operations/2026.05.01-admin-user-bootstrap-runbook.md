# DOWIN Admin User Bootstrap Runbook

이 문서는 현재 구현 기준에서 첫 운영자 계정을 제품 UI가 아니라 수동으로 생성해야 할 때 사용하는 실행 문서다.

현재 전제:

- 운영자 계정은 서비스 사용자 `users`와 별도인 `admin_users`에 저장된다.
- 운영 권한은 `admin_role_grants`로 별도 부여한다.
- 로그인 세션은 `dowin_admin_sid` 쿠키와 `admin_sessions`를 사용한다.
- 첫 `SYSTEM_ADMIN` 생성용 제품 UI나 bootstrap API는 아직 없다.

관련 기준 문서:

- [어드민 대시보드(운영 도구) 기획안](/docs/planning/2026.04.30-admin-dashboard-operation-tool-plan.md)
- [데이터베이스 스키마 설계](/docs/dev/common/2026.03.09-database-schema.md)

## 1. 언제 이 문서를 쓰는가

- 로컬 개발 환경에서 첫 운영자 계정을 만들어야 할 때
- 운영 콘솔 로그인 API를 직접 검증해야 할 때
- 아직 운영자 계정 생성/권한 부여 UI가 구현되지 않은 시점에 수동 bootstrap이 필요할 때

이 문서는 로컬 D1 기준이다. 원격 DB에는 그대로 복붙하지 말고, 별도 승인과 운영 절차를 거쳐야 한다.

## 2. 사전 조건

- 로컬 마이그레이션이 적용되어 있어야 한다.
- `wrangler.jsonc`의 로컬 D1 대상은 `wig-db`다.
- 비밀번호는 평문 저장이 아니라 `bcrypt` 해시로 저장해야 한다.

필요 명령:

```bash
yarn mig:local
```

## 3. 로컬 첫 운영자 계정 bootstrap

예시 로그인 ID:

```text
dowin.admin@dasoslab.com
```

### 3.1. 비밀번호 bcrypt 해시 생성

아래 명령에서 `YOUR_ADMIN_PASSWORD`를 실제 비밀번호로 바꾼다.

```bash
yarn node -e "const bcrypt=require('bcryptjs'); bcrypt.hash(process.argv[1], 10).then(v=>console.log(v))" 'YOUR_ADMIN_PASSWORD'
```

출력된 값을 아래 단계의 `<BCRYPT_HASH>` 자리에 넣는다.

### 3.2. `admin_users` row 생성

```bash
yarn wrangler d1 execute wig-db --local --command "
INSERT INTO admin_users (login_id, password_hash, display_name, status, mfa_enabled)
VALUES ('dowin.admin@dasoslab.com', '<BCRYPT_HASH>', 'Dowin Admin', 'ACTIVE', 0);
"
```

설명:

- `login_id`: 운영자 로그인 ID
- `password_hash`: bcrypt 해시
- `display_name`: 운영 콘솔 표시 이름
- `status`: 로그인 가능 상태여야 하므로 `ACTIVE`
- `mfa_enabled`: 현재 MVP에서는 `0`

### 3.3. `SYSTEM_ADMIN` 권한 부여

```bash
yarn wrangler d1 execute wig-db --local --command "
INSERT INTO admin_role_grants (admin_user_id, role, grant_reason)
SELECT id, 'SYSTEM_ADMIN', 'local bootstrap'
FROM admin_users
WHERE login_id = 'dowin.admin@dasoslab.com';
"
```

중요:

- 운영자 계정 row만 있고 role grant가 없으면 로그인은 될 수 있어도 실제 운영 API 접근은 막힌다.
- 첫 bootstrap 계정은 최소 `SYSTEM_ADMIN`을 부여해야 운영 콘솔 전 영역 접근이 가능하다.

### 3.4. 생성 결과 확인

```bash
yarn wrangler d1 execute wig-db --local --command "
SELECT
  u.id,
  u.login_id,
  u.status,
  u.mfa_enabled,
  g.role,
  g.revoked_at
FROM admin_users u
LEFT JOIN admin_role_grants g
  ON g.admin_user_id = u.id
WHERE u.login_id = 'dowin.admin@dasoslab.com';
"
```

정상 기대값:

- `login_id = dowin.admin@dasoslab.com`
- `status = ACTIVE`
- `role = SYSTEM_ADMIN`
- `revoked_at = NULL`

## 4. 로그인 확인

현재는 운영자 로그인 전용 UI가 아직 없을 수 있으므로 API로 직접 확인하는 편이 빠르다.

로컬 개발 서버는 `package.json` 기준 `4000` 포트를 사용한다.

```bash
curl -i -X POST http://localhost:4000/api/admin/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"loginId":"dowin.admin@dasoslab.com","password":"YOUR_ADMIN_PASSWORD"}'
```

정상 기대값:

- HTTP `200`
- 응답 body에 `adminUser`, `roles` 포함
- `Set-Cookie`에 `dowin_admin_sid` 발급

## 5. 실패 시 점검 순서

### 5.1. `INVALID_CREDENTIALS`

가능 원인:

- `login_id` 오타
- bcrypt 해시가 아닌 평문을 저장
- 다른 비밀번호로 해시를 생성했음

### 5.2. `FORBIDDEN`

가능 원인:

- `admin_users.status`가 `ACTIVE`가 아님

### 5.3. 로그인은 되는데 운영 API가 403

가능 원인:

- `admin_role_grants`에 필요한 role이 없음
- grant는 있지만 `revoked_at`이 채워져 있음

## 6. 보안 메모

- 로컬에서도 비밀번호를 문서나 커밋에 평문으로 남기지 않는다.
- 원격 DB에는 임시 테스트 운영자 계정을 오래 남기지 않는다.
- bootstrap 이후 운영자 생성/정지/권한 부여는 장기적으로 운영 콘솔 내부 기능으로 이관해야 한다.
- MFA, 운영자 비밀번호 재설정, 초대 토큰 플로우는 아직 MVP 범위 밖이다.

## 7. 후속 구현 과제

- 운영자 계정 bootstrap 스크립트 추가
- 운영자 생성/정지/권한 부여 API
- 운영자 로그인 UI
- MFA 및 재인증 정책
