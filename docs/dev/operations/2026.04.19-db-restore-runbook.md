# DOWIN DB Restore Runbook

## 1. 이 문서가 다루는 사고

- 대량 `DELETE` 또는 잘못된 `UPDATE`
- 잘못된 migration으로 인한 데이터 손상
- 특정 시점 이후 데이터 정합성 붕괴
- 운영자가 “DB를 특정 시점으로 되돌려야 한다”고 판단한 사고

이 문서는 “앱 코드 rollback” 문서가 아니다.  
DB restore는 데이터 상태를 되돌리는 작업이고, 앱 배포 rollback과는 별도로 판단해야 한다.

## 2. 선행 원칙

- restore는 destructive operation이다.
- restore 전에 추가 배포와 추가 migration을 중단한다.
- restore 시 in-flight query와 transaction이 취소될 수 있다.
- 사고 직후 이미 쌓인 정상 데이터가 많으면 전체 restore보다 수동 보정이 나을 수 있다.

## 3. 역할

- Incident Commander
  - restore 여부 최종 결정
  - 기준 시점 승인
- Ops Lead
  - 실제 D1 확인, export, restore 수행
- Communications Lead
  - 내부/외부 상태 공유

## 4. 15분 체크리스트

1. 추가 배포 중단
2. `yarn mig:remote` 같은 원격 migration 작업 중단
3. 최근 사고 시각 추정
4. 영향 범위 확인
   - 전체 DB인지
   - 특정 테이블인지
   - 특정 워크스페이스/유저만인지
5. 현재 bookmark 확보
6. restore vs 수동 보정 결정
7. restore 시각과 검증 담당자 지정

## 5. 판단 기준

### 전체 restore를 우선 검토할 때

- 전체 테이블 또는 핵심 테이블이 광범위하게 손상됨
- 사고 시점이 명확함
- 사고 이후 정상 write가 거의 없음
- 빠른 서비스 정상화가 최우선임

### 전체 restore를 피할 때

- 일부 사용자 데이터만 손상됨
- 사고 이후 정상 데이터가 많이 쌓임
- 특정 테이블만 보정하면 됨
- restore로 잃는 정상 데이터가 너무 큼

## 6. 실행 전 확인

### 6.1. D1 상태 확인

```bash
yarn wrangler d1 info wig-db
```

확인할 것:

- `version: production` 여부
- 대상 DB가 맞는지

### 6.2. 현재 bookmark 확보

```bash
yarn wrangler d1 time-travel info wig-db
```

기록할 것:

- 현재 bookmark
- 실행 시각
- 사고 추정 시각

### 6.3. 가능하면 현재 상태 export

사고 상태라도 나중에 비교와 부분 복구에 쓸 수 있으므로, 가능하면 export를 남긴다.

```bash
yarn wrangler d1 export wig-db --remote --output=./tmp/wig-db-pre-restore.sql
```

주의:

- export는 DB 요청을 막을 수 있으므로 장애 상황에서 실행 여부를 Incident Commander가 판단한다.

## 7. Restore 절차

### 7.1. 복구 기준 시점 정하기

- 사고 직전 정상 시각을 RFC3339 또는 Unix timestamp로 정한다.
- 예: `2026-04-19T14:25:00+09:00`

### 7.2. restore 실행

```bash
yarn wrangler d1 time-travel restore wig-db --timestamp="2026-04-19T14:25:00+09:00"
```

또는 bookmark가 명확하면:

```bash
yarn wrangler d1 time-travel restore wig-db --bookmark="<BOOKMARK>"
```

기록할 것:

- restore 기준 시점
- restore 시작/종료 시각
- Cloudflare가 반환한 이전 bookmark

그 이전 bookmark는 “복구를 되돌리는 용도”라서 반드시 남긴다.

## 8. Restore 직후 검증

아래는 최소 smoke check다.

1. 로그인 가능
2. `GET /api/workspaces/me` 성공
3. 활성 점수판 조회 성공
4. 최근 daily log 조회 성공
5. 신규 daily log 1건 쓰기 성공
6. 팀 대시보드 조회 성공

가능하면 아래도 확인한다.

- 주요 테이블 row count 급감 여부
- 특정 장애 제보 계정 데이터 복구 여부
- 5xx 급증이 멈췄는지

## 9. Restore 후 추가 판단

### restore 후 정상화되면

- 서비스 재개 공지
- 중단한 배포/migration 재개 여부 별도 판단

### restore 후에도 앱이 깨지면

- DB는 복구됐지만 앱 코드가 현재 스키마와 안 맞는지 확인
- 이 경우 [2026.04.19-deployment-rollback-runbook.md](/docs/dev/operations/2026.04.19-deployment-rollback-runbook.md) 기준으로 code rollback을 검토한다

- 원인 불명 상태에서 반복 restore
- 현재 bookmark 기록 없이 restore
- 앱 호환성 확인 없이 “restore만 하면 끝”이라고 판단
- 사고 중 추가 migration 실행
