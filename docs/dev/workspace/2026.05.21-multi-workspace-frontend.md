# 다중 워크스페이스 프론트엔드 라우트 및 내비게이션 설계

## 개요
기존 단일 워크스페이스 기반이었던 Dowin에서 다중 워크스페이스를 지원함에 따라, 사용자 프로필과 워크스페이스 관리 화면 간의 책임(Responsibility)을 명확히 분리하기 위한 프론트엔드 변경 사항을 정의한다.

## 1. 기존 구현의 문제점
- `src/app/[locale]/(protected)/profile/page.tsx` 한 곳에서 "유저 설정(비밀번호, 닉네임, 알림)"과 "워크스페이스 설정(멤버 관리, 결제, 이름 변경, 삭제)"이 혼재되어 있었다.
- 다중 워크스페이스 체제가 되면, 사용자가 어느 워크스페이스의 설정을 변경하는 것인지 직관적이지 않으며, 워크스페이스 자체의 관리 화면으로서의 응집도가 떨어진다.

## 2. 결정 사항 (2026.05.21)

### 2.1 프로필과 워크스페이스 관리의 물리적 라우트 분리
- **유저 프로필 (`/profile`)**:
  - 오직 '사용자 계정'과 관련된 설정만 담당한다.
  - 닉네임 변경, 비밀번호 변경, 언어 설정, 알림(푸시 리마인더) 설정, 계정 삭제 등.
- **워크스페이스 관리 (`/workspace/*`)**:
  - 기존 `profile/page.tsx`에 있던 워크스페이스 레벨의 모든 관리 기능을 `/workspace/*` 하위 라우트로 이관한다.
  - `/workspace/settings`: 기본 설정(이름 변경) 및 워크스페이스 삭제
  - `/workspace/members`: 멤버 관리
  - `/workspace/invites`: 초대 관리
  - `/workspace/billing`: 결제(플랜) 정보

### 2.2 새로운 워크스페이스 추가 진입점
다중 워크스페이스 환경을 지원하기 위해, 새 워크스페이스를 만들 수 있는 진입점을 다음 두 곳에 배치한다.
1. **사이드바 드롭다운 (`WorkspaceSwitcher.tsx`)**: 워크스페이스 전환 목록 하단에 `+ 워크스페이스 추가` 버튼 배치. (접근성 극대화)
2. **워크스페이스 관리 화면 (`/workspace/settings`)**: "내 워크스페이스 목록"을 두어, 해당 화면에서도 손쉽게 추가할 수 있도록 구성.

### 2.3 Active workspace 동기화 규칙

프론트엔드는 URL의 `[workspaceId]`와 서버의 `dowin_workspace_id` 쿠키가 같은 워크스페이스 public uid를 가리킨다는 전제에서 동작한다.

- `WorkspaceSwitcher`에서 워크스페이스를 선택하면 `PUT /api/workspaces/current`를 호출하고, 성공 후 URL을 대상 워크스페이스 경로로 이동한다.
- API 전환 실패 시 낙관적으로 바꾼 선택 상태를 원복해야 한다.
- `POST /api/workspaces/join-by-invite`와 `POST /api/workspaces/join`은 성공 시 서버에서 `dowin_workspace_id` 쿠키를 참가한 워크스페이스로 갱신한다.
- 조인 성공 후 프론트는 관련 workspace/profile/dashboard query를 무효화하고 새 active workspace 기준으로 다시 진입해야 한다.

이 규칙이 깨지면 사용자가 워크스페이스 B에 `MEMBER`로 참가했더라도, 이전 active workspace A의 `ADMIN` 컨텍스트가 계속 표시될 수 있다. role 판정은 항상 active workspace 기준이므로, 조인/전환 경로는 쿠키 갱신을 기능 완료 조건으로 봐야 한다.

## 3. 후속 작업(Next Steps)
- `profile/page.tsx`에서 워크스페이스 관련 메뉴(members, invites, billing, name, delete) 제거.
- `/workspace/settings/page.tsx` 등 관련 라우트 및 컴포넌트 신규 작성.
- `WorkspaceSwitcher` 드롭다운 UI 내에 `+ 워크스페이스 추가` 링크(`href="/workspace/new"`) 추가.
