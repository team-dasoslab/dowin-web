# Workspace 초대코드 백엔드 구현

## 1. 구현 범위

- 초대코드 기반 워크스페이스 참가 API 추가
- 워크스페이스 ADMIN 초대코드 발급/조회/상태변경 API 추가
- 참가 로직의 존재 검증/오류 코드 정합성 보강
- OpenAPI/생성 코드/DB 마이그레이션 반영

## 2. 정책 반영 (2026-03-24 확정)

- 초대코드는 `사용 횟수 제한(maxUses)` 기반으로 운영
- 만료 시간은 현재 미적용
- 코드 상태는 `ACTIVE` / `INACTIVE` 토글 지원
- 만료 시간 기반 정책은 Post-MVP 확장 항목으로 유지

## 3. API 변경 사항

### 3.1 신규 API

- `POST /api/workspaces/join-by-invite`
  - 인증 필요
  - 입력: `{ code }`
  - 성공: `{ message: "워크스페이스에 참가했습니다." }`
- `GET /api/workspaces/{id}/invites`
  - 인증 + 워크스페이스 ADMIN 필요
  - 해당 워크스페이스 초대코드 목록 조회
- `POST /api/workspaces/{id}/invites`
  - 인증 + 워크스페이스 ADMIN 필요
  - 입력: `{ maxUses }`
  - 초대코드 발급 (기본 상태 `ACTIVE`, `usedCount=0`)
- `PATCH /api/workspaces/{id}/invites/{inviteId}/status`
  - 인증 + 워크스페이스 ADMIN 필요
  - 입력: `{ status: "ACTIVE" | "INACTIVE" }`

### 3.2 기존 API 보강

- `POST /api/workspaces/join`
  - 서비스 레이어에서 대상 워크스페이스 존재 확인 후 `NOT_FOUND` 처리

## 4. DB/도메인 변경 사항

- `src/db/schema.ts`
  - `workspace_invites` 테이블 추가
  - `workspaces` relation에 `invites` 추가
- `drizzle/0005_omniscient_lilith.sql`
  - `workspace_invites` 생성 마이그레이션

`workspace_invites` 주요 필드:

- `workspaceId`
- `code` (unique)
- `maxUses`
- `usedCount`
- `status` (`ACTIVE`/`INACTIVE`)
- `createdByUserId`

## 5. 서비스/스토리지 구현 포인트

- `WorkspaceService`
  - `createInvite`, `listInvites`, `updateInviteStatus`, `joinWorkspaceByInvite` 추가
  - invite 참가 시 상태/사용횟수 선검증 + 트랜잭션 결과 재검증
- `WorkspaceStorage`
  - invite CRUD성 조회/변경 메서드 추가
  - `addMemberByInvite`에서
    - `ACTIVE` + `usedCount < maxUses` 조건으로 원자적 증가
    - 성공 시 멤버십 추가

## 6. 오류 코드 추가

- `INVITE_CODE_INACTIVE` (409)
- `INVITE_CODE_USAGE_LIMIT_REACHED` (409)

## 7. 계약/생성 코드 반영
