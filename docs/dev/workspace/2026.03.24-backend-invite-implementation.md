# Workspace 초대코드 백엔드 구현

## 1. 구현 범위

- 초대코드 기반 워크스페이스 참가 API 추가
- 워크스페이스 ADMIN 초대코드 발급/조회/상태변경 API 추가
- 참가 로직의 존재 검증/오류 코드 정합성 보강
- OpenAPI/생성 코드/DB 마이그레이션 반영

## 2. 정책 반영 (2026-03-24 확정)

- 초대코드는 `사용 횟수 제한(maxUses)` 기반으로 운영
- 만료 시간은 현재 미적용
- 코드 상태는 `ACTIVE` / `INACTIVE` 토글 지원
- 만료 시간 기반 정책은 Post-MVP 확장 항목으로 유지

## 3. API 변경 사항

### 3.1 신규 API

- `POST /api/workspaces/join-by-invite`
  - 인증 필요
  - 입력: `{ code }`
  - 성공: `{ message: "워크스페이스에 참가했습니다." }`
  - 성공 시 참가한 워크스페이스의 public uid를 `dowin_workspace_id` 쿠키에 설정한다
- `GET /api/workspaces/{id}/invites`
  - 인증 + 워크스페이스 ADMIN 필요
  - 해당 워크스페이스 초대코드 목록 조회
- `POST /api/workspaces/{id}/invites`
  - 인증 + 워크스페이스 ADMIN 필요
  - 입력: `{ maxUses }`
  - 초대코드 발급 (기본 상태 `ACTIVE`, `usedCount=0`)
- `PATCH /api/workspaces/{id}/invites/{inviteId}/status`
  - 인증 + 워크스페이스 ADMIN 필요
  - 입력: `{ status: "ACTIVE" | "INACTIVE" }`

### 3.2 기존 API 보강

- `POST /api/workspaces/join`
  - 서비스 레이어에서 대상 워크스페이스 존재 확인 후 `NOT_FOUND` 처리
  - 참가 성공 시 대상 워크스페이스를 active workspace로 전환하도록 `dowin_workspace_id` 쿠키 갱신

## 5.1 Active workspace 쿠키 회귀 방지 (2026-06-19)

초대코드 참가 성공 후 멤버십은 정상적으로 `MEMBER`로 생성되더라도, `dowin_workspace_id` 쿠키가 이전 워크스페이스를 계속 가리키면 `/api/workspaces/me`와 레이아웃이 이전 워크스페이스 컨텍스트를 표시할 수 있다.

대표 사례:

- 사용자가 기존 워크스페이스 A에서는 `ADMIN`
- 초대코드로 워크스페이스 B에 `MEMBER`로 참가
- 참가 성공 후 active workspace 쿠키가 B로 갱신되지 않음
- 화면은 여전히 A의 `ADMIN` 컨텍스트를 표시

따라서 초대코드 참가 route는 성공 응답 전에 반드시 다음을 수행한다.

1. `WorkspaceService.joinWorkspaceByInvite`에서 조인된 워크스페이스 public uid를 반환받는다.
2. `dowin_workspace_id` 쿠키를 해당 uid로 설정한다.
3. 테스트에서 `Set-Cookie` 동작을 검증한다.

이 규칙은 `POST /api/workspaces/join` 직접 참가 경로에도 동일하게 적용한다.
