# WebView Bridge Specification

이 문서는 Dowin App(Native Shell)과 WebView(Web Product) 간의 통신 규격을 정의합니다. `gronxb/webview-bridge` 라이브러리를 기반으로 하며, 타입 안전성과 반응형 상태 동기화를 지원합니다.

현재 구현은 `src/bridge/index.ts`, `src/hooks/use-bridge-sync.ts`, `src/components/main-webview.tsx`를 기준으로 정리되어 있습니다. 문서 예시는 실제 `@webview-bridge/react-native@1.7.x` API에 맞춰 업데이트합니다.

## 1. 개요

- **라이브러리**: `@webview-bridge/react-native`
- **목적**: 네이티브 기능 제어 및 시스템 상태(Safe Area, App State)의 실시간 공유.
- **연동 방식**:
  - **HTTP Headers**: 초기 요청 시 커스텀 헤더를 통해 네이티브 클라이언트임을 식별.
  - **Methods**: 웹에서 네이티브 함수를 호출 (Promise 기반).
  - **Store**: 네이티브 상태를 웹에서 반응형(Store)으로 구독.

## 2. 클라이언트 식별 (HTTP Headers)

웹뷰에서 서버로 요청을 보낼 때 다음 헤더가 자동으로 포함됩니다. 서버 및 클라이언트 초기 진입 시 네이티브 앱 여부를 판단하는 데 사용합니다.

| 헤더명           | 값    | 설명                              |
| :--------------- | :---- | :-------------------------------- |
| `X-DOWIN-Client` | `app` | 도윈 네이티브 클라이언트임을 명시 |

## 3. Native Store (App -> Web)

네이티브에서 관리하며 웹으로 실시간 동기화되는 상태입니다. 웹 개발자는 이 스토어를 구독하여 UI를 조정합니다.

| 속성명                   | 타입                                        | 설명                                                       |
| :----------------------- | :------------------------------------------ | :--------------------------------------------------------- |
| `isNative`               | `boolean`                                   | 현재 네이티브 앱 환경인지 여부 (항상 `true`)               |
| `appState`               | `'active' \| 'background' \| 'inactive'`    | 앱의 활성화 상태                                           |
| `safeAreaInsets`         | `object`                                    | `{ top, bottom, left, right }` 기기별 세이프 에어리어 수치 |
| `statusBarHeight`        | `number`                                    | 상태바의 높이 (px)                                         |
| `platform`               | `'ios' \| 'android'`                        | 실행 플랫폼                                                |
| `notificationPermission` | `'granted' \| 'denied' \| 'not-determined'` | 알림 권한 상태                                             |
| `lastNotification`       | `object \| null`                            | 최근 수신한 푸시 알림의 페이로드 (JSON)                    |
| `lastDeepLink`           | `string \| null`                            | 최근 진입한 딥링크 URL                                     |

동기화 원칙:

- `notificationPermission`은 `granted`, `denied`, `not-determined`를 그대로 유지합니다.
- `lastNotification`은 포그라운드 수신 알림과 사용자가 탭한 마지막 알림 응답 모두에서 갱신됩니다.
- `lastDeepLink`는 초기 실행 URL과 런타임 URL 이벤트 모두에서 갱신됩니다.
- `appState`는 React Native의 `'unknown'` 값을 직접 노출하지 않고 `'inactive'`로 정규화합니다.

## 3.1 문법 규칙

이 섹션은 권장사항이 아니라 브리지 계약 규칙입니다. 새 필드나 메서드를 추가할 때 반드시 지켜야 합니다.

- Native Store 필드는 반드시 JSON-serializable 이어야 합니다.
- 허용 값은 `string`, `number`, `boolean`, `null`, plain object, array 뿐입니다.
- `Date`, `Map`, `Set`, `Error`, `Function`, class instance, `undefined`, symbol, BigInt는 Store에 넣지 않습니다.
- Expo 또는 React Native 원본 객체를 Store에 그대로 넣지 않습니다. 브리지 전송 전 plain JSON payload로 투영합니다.
- `lastNotification` 같은 payload 필드는 원본 SDK 타입이 아니라 브리지용 JSON projection 으로 저장합니다.
- 메서드 시그니처는 명시 타입으로 유지합니다. `any` 사용 금지입니다.
- 브리지 메서드는 반드시 `Promise`를 반환합니다.
- 브리지 메서드의 인자는 additive 하게 확장합니다. 기존 positional argument 의미 변경은 금지합니다.
- 실패는 조용히 삼키지 않고 설명 가능한 `Error`로 반환합니다.
- 웹이 직접 변경하면 안 되는 상태는 메서드로 우회 노출하지 않습니다. Native Store는 네이티브만 갱신합니다.
- 상태 필드명과 메서드명은 충돌하면 안 됩니다.
- 한 번 공개한 필드명과 메서드명은 rename 대신 additive migration을 우선합니다.
- 웹 복사용 `src/bridge/types.ts`는 다른 로컬 파일 import 없이 단독 복사 가능해야 합니다.
- 웹 복사용 타입 파일에는 Expo, React Native 런타임 의존 타입을 넣지 않습니다.
- 문서 예시와 실제 구현의 문법이 다르면 문서를 즉시 업데이트합니다.

## 4. Native Methods (Web -> App)

웹에서 네이티브의 기능을 실행하기 위해 호출하는 메서드입니다.

### 4.1 UI & UX

- `setStatusBar(style: 'light' | 'dark', animated?: boolean): Promise<void>`
  - 상태바 아이콘 색상 변경.
- `vibrate(type: 'impactLight' | 'impactMedium' | 'impactHeavy' | 'notificationSuccess' | 'notificationWarning' | 'notificationError' | 'selection'): Promise<void>`
  - 상황에 맞는 햅틱 피드백(진동) 실행.
- `share(options: { title?: string, message?: string, url?: string }): Promise<void>`
  - 네이티브 공유 시트 호출.
- `requestAppReview(): Promise<void>`
  - 인앱 리뷰 팝업(별점) 호출.

### 4.2 System

- `openInBrowser(url: string): Promise<void>`
  - 인앱 브라우저가 아닌 시스템 기본 브라우저로 URL 실행.
  - 현재 구현은 `http(s)`만 허용합니다.
- `requestNotificationPermission(): Promise<'granted' | 'denied'>`
  - 네이티브 알림 권한 팝업을 실행하고 결과를 반환.
  - 반환값과 별도로 Native Store의 `notificationPermission`도 함께 갱신합니다.
- `getPushToken(): Promise<string>`
  - 푸시 알림 수신을 위한 기기 고유 토큰(FCM/APNS) 반환.
  - 현재 구현은 physical device와 알림 권한 허용이 선행되어야 합니다.
  - 조건이 맞지 않으면 설명 가능한 에러를 throw 합니다.
- `getAppVersion(): Promise<string>`
  - 현재 설치된 앱의 버전 정보 반환.

주의:

- 웹은 Native Store를 직접 변경하지 않습니다. 상태는 네이티브 소스 오브 트루스입니다.
- 브리지 메서드는 additive 원칙으로 추가합니다. 기존 메서드 시그니처 변경은 구버전 웹/앱 호환성 검토가 필요합니다.

## 5. 구현 가이드

### 5.1 Native (App) 구현

`@webview-bridge/react-native`의 `bridge()`는 메서드와 상태를 함께 담는 `BridgeStore`를 반환합니다. 별도 `appBridge.store(...)` API는 없습니다.

```typescript
// src/bridge/index.ts
const initialAppBridgeState = {
  isNative: true,
  platform: Platform.OS as "ios" | "android",
  appState: "active" as const,
  safeAreaInsets: { top: 0, bottom: 0, left: 0, right: 0 },
  statusBarHeight: 0,
  notificationPermission: "not-determined" as const,
  lastNotification: null,
  lastDeepLink: null,
};

export const appBridge = bridge({
  ...initialAppBridgeState,
  async vibrate(type) {
    /* ... */
  },
  async share(options) {
    /* ... */
  },
  // ... 메서드 구현
});

export const appStore = appBridge;
```

이 구조에서:

- 함수가 아닌 필드는 웹에 동기화되는 shared state입니다.
- 함수 필드는 웹이 호출하는 native method입니다.
- 네이티브 내부 상태 변경은 `appStore.setState(...)`로 수행합니다.

### 5.2 Web (Client) 구현

제공된 `AppBridge` 타입을 사용하여 브릿지를 연결합니다.

```typescript
import { linkBridge } from "@webview-bridge/web";
import type { AppBridge } from "./types";

const bridge = linkBridge<AppBridge>();

// 스토어 구독 예시
bridge.subscribe((state) => {
  if (state.appState === "active") {
    /* ... */
  }
});
```

## 6. 타입 공유 방식

모노레포가 아니므로, 네이티브 프로젝트에서 생성된 `src/bridge/types.ts` 파일을 웹 프로젝트의 `src/types/bridge.ts`로 수동 복사하여 사용합니다.

타입 파일 규칙:

- `src/bridge/types.ts`는 다른 앱 로컬 파일 import 없이 단독으로 복사 가능해야 합니다.
- 웹 프로젝트에서 그대로 붙여 넣을 수 있도록 React Native 또는 Expo 런타임 타입 의존성을 넣지 않습니다.
- 브리지 시그니처를 바꾸면 이 파일을 웹 프로젝트에도 즉시 동기화합니다.
- `src/bridge/index.ts`의 상태 타입과 `src/bridge/types.ts`의 공개 타입은 의미상 동일해야 합니다.
- 앱 내부 편의를 위해 타입을 넓히는 변경은 금지합니다. 먼저 브리지 JSON 제약을 만족하는지 확인합니다.

## 7. WebView 보안 규칙

- WebView `source.uri`는 `EXPO_PUBLIC_SERVICE_URL` 기준 단일 origin만 허용합니다.
- `originWhitelist`는 신뢰 origin 하나로 제한합니다.
- 신뢰 origin 외 탐색 요청은 WebView 내부 로딩을 막고 OS 외부 앱으로 위임합니다.
- 브리지 신뢰 범위 역시 동일 origin 전제를 따릅니다.

## 8. WebView 셸 동작 규칙

- 앱 시작 시 native splash는 기본 자동 숨김을 막고 primary WebView 초기 로드 완료 시점까지 유지합니다.
- 초기 WebView 로드가 timeout, network error, HTTP error로 실패하면 splash를 숨기고 로컬 fallback 화면을 표시합니다.
- 로컬 fallback 화면은 최소한 retry 액션을 제공해야 합니다.
- Android 하드웨어 뒤로가기는 우선 WebView 브라우징 히스토리에 연결합니다.
- WebView가 `canGoBack === true`이면 네이티브가 뒤로가기를 소비하고 `webView.goBack()`을 호출합니다.
- WebView가 더 이상 뒤로갈 수 없으면 하드웨어 뒤로가기는 시스템 기본 동작으로 넘깁니다.
- 다음 경로에서는 Android 하드웨어 뒤로가기를 WebView 뒤로가기로 소비하지 않습니다:
- `/dashboard`
- `/dashboard/my`
- `/report`
- `/scoreboards`
- `/setup?mode=update`
- `/profile`
- 위 예외 경로에서는 WebView 히스토리가 있어도 시스템 기본 뒤로가기 동작으로 넘깁니다.
- 이 동작은 네이티브 라우트를 추가하지 않는 현재 단일 WebView 셸 전제에서 유지합니다.

## 9. 현재 구현 상태 메모

- 구현됨: safe area, app state, notification permission, last notification, deep link, status bar, haptics, share, 외부 브라우저 열기, 앱 버전 조회
- 구현됨: Android 하드웨어 뒤로가기를 WebView 뒤로가기에 연결
- 구현됨: 초기 WebView 로드 timeout/network/HTTP failure 시 로컬 fallback과 retry 제공
- 구현 세부: `getPushToken()`은 device token을 반환하며, simulator/emulator 또는 권한 미허용 상태에서는 명시적 에러를 반환함
- 미구현 또는 placeholder: `requestAppReview()`는 현재 로그만 남기며, 현재 프로젝트 범위에서는 `expo-store-review` 같은 추가 구현을 진행하지 않음
- 타입 공유 파일은 수동 복사 방식이므로 브리지 변경 시 웹 타입 동기화 절차가 필요함

현재 결정:

- `requestAppReview()`는 브리지 시그니처만 남겨두고 실제 스토어 리뷰 연동은 하지 않습니다.
- 따라서 `expo-store-review`는 현재 앱 범위에서 필수 의존성이 아닙니다.
- 이후 실제 인앱 리뷰 UX가 필요해질 때만 별도 도입 여부를 검토합니다.
