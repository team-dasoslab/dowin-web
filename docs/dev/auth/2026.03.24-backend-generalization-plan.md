# Auth 범용화 전환 문서 (백엔드)

## 2. 현재 구현 사실 (코드 기준)

### 2.1. 인증 흐름

- 로그인: `POST /api/auth/login`
  - `customId/password` 검증 후 `dowin_sid` 쿠키 설정
  - 실패 시 `INVALID_CREDENTIALS`
- 로그아웃: `POST /api/auth/logout`
  - 세션이 있으면 삭제, 없어도 `204` 반환
- 비밀번호 변경: `PUT /api/auth/password`
  - 세션 필수, 현재 비밀번호 확인 후 변경
- 관리자 사용자 생성: `POST /api/admin/users`
  - 호출자 워크스페이스 ADMIN 검증 후 계정 생성
  - 생성 사용자 자동으로 같은 워크스페이스 `MEMBER` 가입

### 2.2. 세션/보안

- 세션 쿠키 이름: `dowin_sid`
- 세션 TTL: 30일 (`SESSION_TTL_SECONDS`)
- 세션 재발급: 3일 경과 시 만료 연장(sliding)
- 비밀번호 해시: bcrypt (`hash(..., 10)`)
- 인증 필요 라우트는 `getSession(db)` 사용
- 프로덕션 테스트 계정(`test2/3/4`) 쓰기 제한 가드 적용

### 2.3. 현재 미구현

- Self-signup API 없음
- 계정 복구(비밀번호 재설정) API 없음
- 이메일/OTP 기반 본인확인 플로우 없음
- 회원가입 rate limiting 없음

## 3. 문서/스펙과의 차이

- 도메인 문서의 장기 세션(10년) 설명과 달리, 실제 코드는 30일 슬라이딩 세션이다.
- 보안 문서의 bcrypt cost 12 설명과 달리, 실제 코드는 cost 10이다.
- 현재 상태가 "폐쇄형(관리자 생성 계정)"인데, 범용 전환 우선순위 문서와 맞추려면 self-signup이 선행되어야 한다.

## 4. 변경 목표 (범용 서비스 + 무료 가치 우선)

### 4.0. 확정 결정 (2026-03-24)

- 범용 서비스 전환 시점에 공개 테스트 계정(`test2`, `test3`, `test4`)은 전부 제거한다.
- 테스트 계정 전용 쓰기 제한 로직(`restricted-test-account`)은 운영 경로에서 제거한다.
- 가입은 공개(`self-signup`)로 전환하고, 워크스페이스 합류는 초대코드 기반으로 분리한다.
- 계정 복구는 `복구코드(기본)` + `워크스페이스 관리자 수동 복구(보조)` 정책으로 간다.

### 4.1. 목표

- 관리자 개입 없이 신규 사용자가 바로 가입/로그인 가능해야 한다.
- 무료 사용자가 첫 진입에서 막히지 않도록 최소 온보딩 인증 흐름을 완성해야 한다.
- 유료화 기능 없이도 안전하게 운영 가능한 복구/보안 정책을 갖춰야 한다.

### 4.2. 비목표

- 결제/청구 연동
- 엔터프라이즈 SSO
- 고도화된 권한 시스템 전면 개편

## 5. 변경안 (백엔드)

### 5.1. Self-signup 도입

- 신규 API(초안): `POST /api/auth/signup`
- 입력(초안): `customId`, `nickname`, `password`
- 동작:
  - 중복 `customId` 검증 (`CUSTOM_ID_ALREADY_EXISTS`)
  - 사용자 생성 + 세션 즉시 발급(로그인 상태로 시작)
- 레이어 분리:
  - validation: signup schema
  - service: `signup` 유스케이스
  - storage: 기존 createUser/createSession 재사용

### 5.2. 계정 정책 정리

- `customId`를 로그인 식별자로 계속 유지할지, email 기반으로 확장할지 결정 필요
- MVP 제안:
  - 당장 `customId` 유지(변경 비용 최소화)
  - 추후 `email` 컬럼 추가 가능한 구조로 문서화
- 비밀번호 규칙은 현재 regex 유지하되, 정책 문서와 실제 구현을 일치시킨다.

### 5.3. 계정 복구/보안 정책 최소 도입

- 1차(기본): 복구코드 기반 복구
  - 가입 시 복구코드 N개(예: 8개) 1회 발급
  - 서버는 복구코드 원문이 아닌 해시만 저장
  - 사용자가 코드 1개를 제출하면 1회 사용 처리 후 비밀번호 재설정 허용
- 2차(보조): 워크스페이스 관리자 수동 복구
  - 같은 워크스페이스 ADMIN만 멤버의 비밀번호 초기화 가능
  - 초기화 후 대상 사용자의 `isFirstLogin=true`로 강제 변경 유도
- 보안 보강:
- 로그인/가입 시도 rate limit
  - 실패 횟수 기반 임시 잠금(선택)

### 5.5. 복구 API 초안

- 사용자 복구코드:
  - `POST /api/auth/recovery-codes/verify`
  - `PUT /api/auth/password/by-recovery-code`
- 관리자 수동 복구:
  - `POST /api/workspaces/:id/members/:memberId/password-reset` (ADMIN)

### 5.4. 테스트 계정 제거 전환 작업

- 제거 대상:
  - 공개 테스트 계정 데이터(`test2`, `test3`, `test4`)
  - 테스트 계정 쓰기 제한 가드 코드 (`src/lib/server/restricted-test-account.ts`)
- 정리 항목:
  - 해당 가드를 호출하는 라우트에서 분기 제거
  - README의 공개 테스트 계정 안내 제거
  - 운영 DB/시드에서 테스트 계정 삭제 마이그레이션 또는 스크립트 추가

## 6. 구현 순서 제안

1. OpenAPI에 `POST /auth/signup` 계약 추가
2. auth validation/service/storage/route 구현
3. signup + 기존 login 충돌 케이스 테스트 추가
4. 보안 정책 문서와 실제 상수(세션 TTL, bcrypt cost) 동기화
5. 테스트 계정 제거(가드/문서/데이터) 적용
6. 복구코드/관리자 수동 복구 API 도입
7. 가입 rate limit/추가 보안 강화는 후속 마일스톤으로 분리

## 9. 후속 반영 상태 메모 (2026-05-03)

- 일반 사용자 로그인에는 같은 아이디 + 같은 IP 기준 15분 동안 10회 실패 시 15분 차단이 적용되었다.
- 비밀번호 변경 및 복원코드 기반 비밀번호 재설정 시 해당 사용자 세션은 전체 무효화된다.
- 회원가입 rate limit은 아직 미구현이다.
