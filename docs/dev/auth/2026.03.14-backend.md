# Auth 도메인 작업 결과 (2026.03.14)

## 1. 개요
인증(Auth) 도메인의 핵심 비즈니스 로직, 레이어드 아키텍처(Controller-Service-Storage) 및 표준화된 API 인프라를 구축하였습니다.

## 2. 작업 내용

### 2.1. 아키텍처 및 시스템 인프라
- **레이어드 아키텍처**: 전체 백엔드 구조를 Controller(Route), Service(Logic), Storage(DB)로 분리하고 이를 현재 백엔드 스킬/규칙 문서 기준으로 정리.
- **표준 API 유틸리티**: `src/lib/`에 `api-response.ts`(성공/에러 응답), `errors.ts`(커스텀 HTTP 에러), `with-error-handler.ts`(에러 래퍼), `auth.ts`(세션 유틸) 구축.
- **Cloudflare 환경 지원**: 로컬 커맨드 및 `next dev` 환경에서 D1 바인딩에 안정적으로 접근하기 위해 `@opennextjs/cloudflare`의 `getCloudflareContext()` 적용. (동기식 호출 준수)
- **DB 마이그레이션 및 관계 설정**: Drizzle Kit 및 Wrangler를 연동하여 `wig-db`의 스키마 동기화 완료. 특히 `db.query` 사용을 위해 `schema.ts`에 `relations` 정의를 추가하여 Relational Query 안정성 확보.

### 2.2. 인증 도메인 구현 (Auth Domain)
- **Controller (Next.js Routes)**:
  - `src/app/api/auth/login/route.ts`: 로그인 및 세션 쿠키(`dowin_sid`) 발급.
  - `src/app/api/auth/logout/route.ts`: 세션 종료 및 쿠키 삭제.
  - `src/app/api/auth/password/route.ts`: 인증된 사용자의 비밀번호 변경.
  - `src/app/api/admin/users/route.ts`: 관리자용 신규 멤버 생성. 요청 본문으로 비밀번호를 직접 받고, 응답에 초기 비밀번호는 포함하지 않음.
- **Service & Storage**:
  - `AuthService`: `bcryptjs`를 이용한 해싱/비교, `nanoid` 기반 세션 ID 생성 등 핵심 로직 담당.
  - `AuthStorage`: Drizzle ORM을 활용한 `users`, `sessions` 테이블 접근 로직 캡슐화.

### 2.3. 입력값 검증 및 보안
- **Zod Schema**: `loginSchema`, `passwordChangeSchema`, `adminCreateUserSchema`를 통한 엄격한 요청 검증.
- **보안**: `bcryptjs` 활용(Workers CPU 제한 주의사항 주석 포함), `httpOnly`/`secure` 세션 쿠키 설정.

## 3. 결과 확인
- **단위 테스트**: `yarn test`를 통해 Validation(18건) 및 AuthService(7건) 전체 성공 확인.
- **구조적 이점**: HTTP 처리, 비즈니스 로직, 데이터 접근의 책임을 분리하여 유지보수성 및 테스트 용이성 확보.

## 4. 향후 계획
- 관리자 권한 체크 미들웨어 또는 로직 보완.
- Orval을 이용한 클라이언트 타입 생성 및 프론트엔드 연동.
- Workspace 등 타 도메인으로 동일한 아키텍처 패턴 확장.

## 5. 추가 반영 (2026.03.24)

- `POST /api/auth/signup` 공개 회원가입 API를 추가했다.
  - 요청값: `customId`, `nickname`, `password`
  - 성공 시 사용자 생성 후 `dowin_sid` 세션 쿠키를 즉시 발급하고 `201`을 반환한다.
  - 중복 ID는 `CUSTOM_ID_ALREADY_EXISTS(409)`로 처리한다.
- AuthService에 `signup` 유스케이스를 추가해 계정 생성과 세션 발급을 일관된 서비스 레이어에서 처리한다.
- 회원가입 시 복원코드 8개를 생성해 1회 발급한다.
  - 서버 저장은 원문이 아닌 SHA-256 해시(`auth_recovery_codes`)로 저장한다.
  - `POST /api/auth/signup`의 `201` 응답에 `recoveryCodes: string[]`를 포함한다.
- 복원코드 기반 복구 API를 추가했다.
  - `POST /api/auth/recovery-codes/verify`: 복원코드로 계정(`customId`, `nickname`) 조회
  - `PUT /api/auth/password/by-recovery-code`: 복원코드 1회 소진 후 비밀번호 재설정
