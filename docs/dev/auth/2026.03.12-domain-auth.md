# Auth 도메인 설계

## 1. 개요

DOWIN는 ID/Password 기반 인증을 사용하며, **공개 회원가입(Self-signup)**과 **관리자 멤버 생성**을 함께 지원한다.  
즉, 개인 계정 생성(Auth)과 팀 합류(Workspace)는 분리된 흐름으로 동작한다.

**세션 방식**: 서버사이드 세션 + HttpOnly 쿠키  
→ 보안 상세는 `docs/dev/common/2026.03.12-security.md` 참조

## 2. 핵심 규칙 (Business Rules)

| # | 규칙 |
|---|------|
| BR-1 | 누구나 `signup`으로 계정을 생성할 수 있다 |
| BR-2 | ADMIN은 워크스페이스 멤버 추가를 위해 별도 계정 생성 API를 사용할 수 있다 |
| BR-3 | 최초 로그인 시, 비밀번호 변경 권장 토스트 1회 노출 |
| BR-4 | 비밀번호 변경은 강제가 아닌 권장 (강제 이탈 없음) |
| BR-5 | 세션 만료 시 재로그인 유도 |
| BR-6 | 로그아웃 시 클라이언트 토큰 삭제 |
| BR-7 | ID/PW 유효성 검사는 **클라이언트 + 서버 양쪽**에서 수행 (Defense in Depth) |

## 2-1. ID / PW 유효성 규칙 (Validation Rules)

### ID (`custom_id`)

| 항목 | 규칙 |
|------|------|
| 허용 문자 | 영문 소문자/대문자, 숫자 (`a-z`, `A-Z`, `0-9`) |
| 금지 문자 | 특수문자 전체, 공백 |
| 길이 | 3자 이상 ~ 20자 이하 |
| 정규식 | `^[a-zA-Z0-9]{3,20}$` |

```typescript
const ID_REGEX = /^[a-zA-Z0-9]{3,20}$/;
export const validateCustomId = (id: string): boolean => ID_REGEX.test(id);
```

---

### PW (`password`)

| 항목 | 규칙 |
|------|------|
| 최소 길이 | 8자 이상 |
| 허용 문자 | 영문, 숫자, 안전 특수문자 |
| **허용 특수문자** | `! @ # $ % ^ & * ( ) - _ = + [ ] { } \| : < > ? , . / ~` |
| **금지 특수문자** | `'` `"` `` ` `` `;` `\` — SQL Injection / 파싱 취약점 유발 가능 |
| 정규식 | `^[a-zA-Z0-9!@#$%^&*()\-_=+\[\]{}|:<>?,./~]{8,}$` |

```typescript
// 허용: 영문, 숫자, 안전 특수문자
// 금지: ' " ` ; \ (SQL injection / 파싱 취약점 유발 문자)
const PW_REGEX = /^[a-zA-Z0-9!@#$%^&()\-_=+\[\]{}|:<>?,./~]{8,}$/;
export const validatePassword = (pw: string): boolean => PW_REGEX.test(pw);
```

> **중요**: SQL Injection의 근본적 방어는 **Prepared Statement(파라미터 바인딩)**로 처리한다. 위 문자 제한은 추가 방어 레이어(Defense in Depth)이며, 문자 제한만으로 SQL Injection을 막는다고 가정하지 않는다.

> **초기 비밀번호 예외**: 기본값 `user`는 위 규칙을 충족하지 않는다. 비밀번호 **변경 시에만** 유효성 검사를 적용하고, 로그인 시에는 해시 비교만 수행한다.

## 3. 사용자 흐름 (User Flow)

### 3.1. 일반 로그인
```
[로그인 화면]
  └─ ID / PW 입력
  └─ 로그인 버튼 클릭
      ├─ 성공 → 세션 쿠키 발급 → 대시보드 이동
      │         └─ (최초 로그인 여부 확인)
      │             └─ 최초 O → 비밀번호 변경 권장 토스트 노출
      └─ 실패 → "아이디 또는 비밀번호가 올바르지 않습니다" 에러 메시지
```

### 3.2. 관리자 계정 생성 (Admin Only)
```
[설정 > 멤버 관리] (ADMIN 한정)
  └─ "새 멤버 추가" 버튼
  └─ ID / 닉네임 / 비밀번호 입력
  └─ 저장 → 전달한 비밀번호로 계정 생성
  └─ 생성된 ID/PW를 팀원에게 전달 (초대 코드 별도 아님)
```

### 3.3. 공개 회원가입 (Self-signup)
```
[회원가입 화면]
  └─ customId / 닉네임 / 비밀번호 입력
  └─ 회원가입 버튼 클릭
      ├─ 성공 → 사용자 생성 + 세션 쿠키 발급 → 온보딩(워크스페이스 생성/참가)
      └─ 실패(중복 ID) → "이미 사용 중인 아이디입니다." 메시지
```

## 4. 데이터 모델

```typescript
// users 테이블 연관 필드
interface User {
  id: string;              // UUID (PK)
  custom_id: string;       // 로그인용 ID (예: 'john123') — UNIQUE
  password_hash: string;   // bcrypt hash
  nickname: string;
  is_first_login: boolean; // 최초 로그인 여부 추적 → 토스트 제어
  created_at: string;
}
```

> **참고**: `is_first_login` 필드와 `sessions` 테이블은 현재 구현에 포함되어 있다.

```typescript
// sessions 테이블
interface Session {
  id: string;          // nanoid — 쿠키에 저장되는 값
  user_id: string;
  expires_at: string;  // 만료 시각 (현재 구현 기준 30일, 슬라이딩 재발급)
  created_at: string;
}
```

## 5. API 명세

### POST `/api/auth/signup`

**Request**
```json
{ "customId": "john123", "nickname": "존", "password": "newSecurePass1!" }
```

**Response (201 Created)**
```json
{
  "user": {
    "id": "uuid",
    "nickname": "존",
    "isFirstLogin": true
  },
  "recoveryCodes": [
    "6JYSB959TK",
    "Q2M7K8N4PD",
    "W8R3T6Y2PL",
    "A7S4D9F1GH",
    "Z5X2C8V6BN",
    "L4K9J3H7GF",
    "P8N2M6Q4WR",
    "T3Y7U1I5OP"
  ]
}
```

**Set-Cookie Header (응답 헤더)**
```
Set-Cookie: dowin_sid=<sessionId>; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=2592000
```

**Error Cases**
| 상태 코드 | 사유 |
|-----------|------|
| 409 | 이미 사용 중인 ID |
| 422 | 입력값 유효성 오류 |

### PUT `/api/auth/password`

> 인증 필요 (세션 쿠키)

**Request**
```json
{ "currentPassword": "user", "newPassword": "newSecurePass1!" }
```

**비밀번호 유효성 규칙** — 섹션 2-1 참조
- 최소 8자 이상
- 허용 특수문자: `! @ # $ % ^ & * ( ) - _ = + [ ] { } | : < > ? , . / ~`
- **금지 특수문자**: `'` `"` `` ` `` `;` `\` (SQL Injection 취약점 유발 가능)

**Response (200 OK)**
```json
{ "message": "비밀번호가 변경되었습니다." }
```

**Error Cases**
| 상태 코드 | 사유 |

| 422 | 비밀번호 유효성 규칙 미충족 |

## 6. 엣지 케이스

| 케이스 | 처리 방식 |
|--------|----------|
| 이미 사용 중인 ID로 계정 생성 시도 | `409 Conflict` 반환 |
| 만료된 세션으로 API 호출 | `401 Unauthorized` → 클라이언트 로그인 화면으로 리다이렉트 |
| 비밀번호 변경 후 기존 세션 계속 사용 | 세션은 D1 기준으로 유효 유지. 무효화 기능은 Post-MVP (전체 세션 삭제 방식) |

## 7. 보안 고려사항

→ 보안 정책 전체는 `docs/dev/common/2026.03.12-security.md` 참조.

**Auth 도메인 핵심 요약:**
- 비밀번호는 **bcrypt** (cost factor 12) 해시 후 저장
- 세션 쿠키: `HttpOnly; Secure; SameSite=Lax; Max-Age=2592000` (30일)
- 현재 구현은 30일 TTL을 사용하며, 세션 잔여 기간이 3일 이하가 되면 만료를 연장하는 슬라이딩 재발급 정책을 적용한다.
- 세션 만료 시 D1 레코드 삭제 + 쿠키 `Max-Age=0` 처리
- 로그인 실패 시 `INVALID_CREDENTIALS` 코드로 단일 응답 (어느 쪽인지 구분하지 않음)
- **SQL Injection 방어**: Drizzle ORM Prepared Statement 사용
- **ID/PW 입력값 검사**: 클라이언트 + 서버(Zod) 이중 검증
