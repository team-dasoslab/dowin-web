# Notification 도메인 설계

## 1. 개요

**알림 도메인**은 사용자가 매일 바쁜 일상 속에서도 선행지표를 기록할 수 있도록 **능동적으로 행동을 촉진**하는 역할을 한다.  
현재는 PWA(Web Push + VAPID) 기반으로 아래 알림을 제공한다.

- 사용자별 시간 설정이 가능한 개인 기록 리마인드
- 기존 주간 집중 리마인드

---

## 2. 핵심 규칙 (Business Rules)

| # | 규칙 |
|---|------|
| BR-1 | 알림은 사용자가 명시적으로 구독(Subscribe)해야 발송됨 |
| BR-2 | 개인 기록 리마인드는 사용자 설정 시각(KST) 기준, ACTIVE 점수판이 있는 구독 유저에게만 발송 |
| BR-3 | 오늘 모든 지표를 이미 기록한 사용자에게는 알림 발송 생략 |
| BR-4 | 알림 클릭 시 대시보드로 딥링크 |
| BR-5 | 사용자가 알림을 off하면 해당 구독 정보 서버에서 삭제 |
| BR-6 | 푸시 구독의 사용자 식별자는 클라이언트 입력이 아니라 서버 세션 기준으로 저장 |
| BR-7 | 개인 기록 리마인드 시간은 사용자별로 저장되며 현재 UI에서는 06:00~23:00 사이 1시간 단위로 선택 가능 |
| BR-8 | 주간 집중 리마인드는 기존 목요일 고정 정책을 유지하며, 최저 실행률 동률일 때만 AI tie-break를 수행 |

---

## 3. 기술 구조

```
[GitHub Actions Cron] (매시간)
  └─ /api/push/send-daily (Authorization 헤더 기반 내부 호출)
  └─ 사용자별 설정 시각과 현재 KST를 비교
  └─ ACTIVE 점수판 + 오늘 미기록 유저 필터링
  └─ PushSubscription 목록 조회
  └─ Web Push API (VAPID) 발송
```

```
[GitHub Actions Cron] (매주 목요일 15:00 KST)
  └─ /api/push/send-weekly-focus (Authorization 헤더 기반 내부 호출)
  └─ 구독 유저별 ACTIVE 점수판 조회
  └─ 이번 주/이번 달 진행률 기준으로 최저 실행률 선행지표 선택
  └─ 동률일 때만 AI SDK + Gemini 2.5 Flash tie-break
  └─ Web Push API (VAPID) 발송
```

```
[클라이언트 - PWA]
  └─ Service Worker 등록
  └─ PushManager.subscribe() → endpoint + keys 취득
  └─ POST /api/push/subscribe 서버 저장
```

## 4. 사용자 흐름 (User Flow)

### 4.1. 알림 구독
```
[마이페이지 > 알림 설정]
  └─ 푸시 토글 ON
  └─ 브라우저 알림 권한 요청 팝업
  └─ 허용 → POST /api/push/subscribe
  └─ 서버는 현재 세션의 user_id로 구독 저장
  └─ 성공 → 기본 일일 리마인드 설정(21:00 KST) 생성 가능
  └─ 브라우저가 이미 거부한 경우 → "브라우저 설정에서 알림을 허용해주세요" 안내
```

### 4.2. 알림 해제
```
[마이페이지 > 알림 설정]
  └─ 토글 OFF
  └─ DELETE /api/push/subscribe
  └─ Service Worker PushManager.unsubscribe()
```

### 4.3. 개인 리마인드 시간 변경
```
[마이페이지 > 알림 설정]
  └─ 시간 선택 (현재 UI는 06:00~23:00, 1시간 단위)
  └─ PUT /api/push/settings
  └─ 서버는 user_notification_settings를 upsert
```

### 4.4. 유료 자동 리마인드 확장안 (Post-MVP / Paid)
```
[프로필 > 알림 설정 > 유료 기능]
  └─ 자동 리마인드 활성화
  └─ 임계 달성률(예: 40% 이하, 60% 이하) 선택
  └─ 발송 주기(매일, 주 2회, 주 1회 등) 선택
  └─ 발송 시간 선택
  └─ 시스템이 활성 선행지표 중 임계치 이하 항목을 계산
  └─ 조건을 만족하는 선행지표만 자동 푸시 발송
```

## 5. 데이터 모델

```typescript
interface PushSubscription {
  id: string;
  user_id: string;
  endpoint: string;         // Web Push endpoint URL
  p256dh: string;           // 암호화 키
  auth: string;             // 인증 키
  created_at: string;
}
```

추가 persisted model:

- `user_notification_settings`

## 6. API 명세

### POST `/api/push/subscribe`

**Request**
```json
{
  "endpoint": "https://fcm.googleapis.com/...",
  "keys": {
    "p256dh": "...",
    "auth": "..."
  }
}
```

**Response (201 Created)**
```json
{ "subscribed": true }
```

---

### DELETE `/api/push/subscribe`

**Response**: `204 No Content`

---

### GET `/api/push/settings`

- 현재 사용자의 일일 리마인드 시간/활성 상태 조회

### PUT `/api/push/settings`

- 현재 사용자의 일일 리마인드 시간 저장

### Paid 확장 예정 API 방향

- 달성률 임계치 기반 자동 리마인드 설정 조회/저장 API 추가 가능
- 예시 설정 항목:
  - `autoReminderEnabled`
  - `triggerBelowPercent`
  - `sendFrequency`
  - `sendTime`
  - `timezone`

---

### GET `/api/push/send-daily` (Internal — Cron Only)

> GitHub Actions 또는 내부 호출에서 `Authorization: Bearer <CRON_SECRET>` 헤더를 붙여 호출한다.

**동작**
1. `ACTIVE` 점수판 보유 + Push 구독 유저 목록 조회
2. 오늘 `ACTIVE` 선행지표 중 미기록 지표(`null`) 있는 유저 필터링
3. Web Push 발송

**알림 메시지 예시**
```
제목: 🎯 오늘 기록이 남아있어요
본문: 매일 물 2L 마시기 외 2개 지표를 아직 체크하지 않았어요.
링크: /dashboard/my
```

---

### GET `/api/push/send-weekly-focus` (Internal — Cron Only)

> GitHub Actions 또는 내부 호출에서 `Authorization: Bearer <CRON_SECRET>` 헤더를 붙여 호출한다.

**동작**
1. Push 구독 유저별 ACTIVE 점수판 조회
2. 이번 주 기준 실행률이 가장 낮은 선행지표 후보 계산
3. `MONTHLY` 선행지표는 이번 달 누적 기준으로 정규화하여 비교
4. 최저 실행률 동률이면 AI SDK + Gemini 2.5 Flash로 1개 선택
5. Web Push 발송

**알림 메시지 예시**
```
제목: 리마인드
본문: 오늘은 주 3회 운동 해볼까요?
링크: /dashboard/my
```

## 7. 엣지 케이스

| 케이스 | 처리 방식 |
|--------|----------|
| 브라우저 알림 권한 거부 | 서버 구독 저장 없음 + 클라이언트에서 설정 안내 토스트 |
| Push 발송 실패 (410 Gone) | 해당 구독 정보 자동 삭제 (만료된 구독) |
| 이미 오늘 전부 기록 완료한 유저 | 발송 스킵 (불필요한 알림 방지) |
| iOS PWA에서 알림 미지원 시 | 인앱 토스트로 fallback: "어제 기록이 빠져있네요!" |
| 최저 실행률 후보가 없음 | 주간 집중 푸시 발송 스킵 |
| 최저 실행률 후보가 동률이며 AI 실패 | 첫 후보 fallback으로 발송 |

## 8. Post-MVP / Paid 확장 메모

- 유료 기능 후보로 `달성률이 n% 이하인 선행지표 자동 리마인드`를 검토한다.
- 이 기능은 수동 리마인드가 아니라 시스템 자동 발송이며, 사용자가 아래를 직접 설정하는 구조를 전제로 한다.
  - 임계 달성률
  - 발송 주기
  - 발송 시간
- 자동 리마인드 대상은 `활성 점수판`의 `활성 선행지표`로 제한하고, 동일 지표에 과도한 중복 발송이 일어나지 않도록 cooldown 규칙이 필요하다.
- 무료 플랜 기본 알림과 섞이지 않게 `Paid 전용 자동화`로 구분하는 것이 현재 제품 방향에 더 맞다.
