# Scoreboard 백엔드 구현 결과

## 1. 구현 개요

Scoreboard 도메인은 이제 활성 점수판 조회, 생성, 수정, 보관, 보관 목록 조회, 재활성화까지 한 사이클의 상태 전환을 모두 지원한다.  
활성 점수판은 사용자 기준 하나만 허용되며, 보관 시 종료일을 자동으로 기록하고 재활성화 시 종료일을 다시 비운다.

## 2. 현재 구현 범위

### 2.1. GET `/api/scoreboards/active`
- 현재 사용자의 활성 점수판 1개를 조회한다.
- 점수판이 없으면 404를 반환한다.
- 응답에는 점수판 본문과 연결된 선행지표 목록이 포함된다.

### 2.2. POST `/api/scoreboards`
- 새 점수판을 생성한다.
- 같은 사용자/워크스페이스에 이미 ACTIVE 점수판이 있으면 `ACTIVE_SCOREBOARD_EXISTS` 409를 반환한다.

### 2.3. PUT `/api/scoreboards/:id`
- ACTIVE 상태의 점수판만 수정할 수 있다.
- ARCHIVED 점수판 수정 시 `SCOREBOARD_ARCHIVED` 403을 반환한다.

### 2.4. POST `/api/scoreboards/:id/archive`
- ACTIVE 점수판을 ARCHIVED로 전환한다.
- 보관 시 `endDate`를 현재 날짜로 자동 저장한다.
- 이미 보관된 점수판이면 `SCOREBOARD_ARCHIVED` 403을 반환한다.

### 2.5. GET `/api/scoreboards`
- 현재 사용자의 보관된 점수판 목록을 최신 생성 순으로 조회한다.
- 점수판 보관함 화면에서 이 목록을 그대로 사용한다.

### 2.6. POST `/api/scoreboards/:id/reactivate`
- 보관된 점수판을 다시 ACTIVE로 전환한다.
- 재활성화 시 `endDate`를 `null`로 되돌린다.
- 이미 ACTIVE 상태면 `SCOREBOARD_ALREADY_ACTIVE` 400을 반환한다.
- 다른 ACTIVE 점수판이 있으면 `ACTIVE_SCOREBOARD_EXISTS` 409를 반환한다.

## 3. 구현 방식

- Route Handler: `src/app/api/scoreboards/**`
- Service: `src/domain/scoreboard/services/scoreboard.service.ts`
- Storage: `src/domain/scoreboard/storage/scoreboard.storage.ts`
- Validation: `src/domain/scoreboard/validation.ts`
- Error mapping: `src/lib/server/api-response.ts`

기존 저장소 규칙에 맞춰 `Route -> Service -> Storage` 구조를 유지했다.

## 4. 반영된 변경점

- 점수판 상태 전환이 `ACTIVE -> ARCHIVED -> ACTIVE` 흐름까지 닫혔다.
- 점수판 보관 시 활성 기간 표시에 필요한 `endDate`가 자동으로 남는다.
- 프론트는 `GET /api/scoreboards`와 `POST /api/scoreboards/:id/reactivate`를 통해 보관함 UI를 구성할 수 있다.
- 선행지표 생성/수정 시 횟수 상한 검증을 서비스 레이어에서 추가로 강제한다.
  - `WEEKLY`: `targetValue <= 7`
  - `MONTHLY`: 점수판 시작월의 최대 일수 이내
  - 위반 시 `VALIDATION_ERROR`로 422 응답
- 선행지표는 `trackingMode`와 `dailyTargetCount`를 가진다.
  - `BOOLEAN`: 기존 O 기록, `dailyTargetCount=1`
  - `COUNT`: 날짜별 count가 `dailyTargetCount` 이상일 때 해당 날짜를 달성으로 집계
  - 주간/월간/팀 대시보드 로그 셀은 `{ value, count, achieved } | null` 형태로 반환

## 5. 현재 제약

- 점수판 상태 히스토리는 `startDate`/`endDate`만 저장하며 별도 상태 변경 로그 테이블은 없다.
- 보관 시점의 시간까지는 저장하지 않고 날짜 단위만 저장한다.
- 보관 목록은 현재 사용자 자신의 점수판만 제공하며 팀원 점수판 히스토리는 제공하지 않는다.
