# Daily Log 백엔드 구현 결과

## 1. 구현 개요

Daily Log 도메인의 핵심 API와 주간/월간 집계 조회 흐름을 실제 서비스 코드에 반영했습니다.
추가로 Next App Router에서 발생하던 동적 세그먼트 충돌을 정리하고, 인증 세션 유틸의 타입을 실제 Drizzle DB 타입 기준으로 보정했습니다.

## 2. 주요 작업 내역

### 2.1. Daily Log API 라우트 정리
- `PUT /api/lead-measures/:id/logs/:date`
  - 특정 날짜의 기록을 생성 또는 수정하는 Upsert 흐름 유지.
- `DELETE /api/lead-measures/:id/logs/:date`
  - 특정 날짜 기록을 삭제하여 미기록 상태로 되돌리는 흐름 유지.
- `GET /api/scoreboards/:id/logs/weekly`
  - 대시보드용 주간 집계 조회 API 유지.
- `GET /api/scoreboards/:id/logs/monthly`
  - 월간 화면용 집계 조회 API 추가.

### 2.2. 동적 세그먼트 이름 충돌 정리
- `lead-measures/[leadMeasureId]` 계열 라우트를 `lead-measures/[id]`로 통일.
- `scoreboards/[scoreboardId]` 계열 라우트를 `scoreboards/[id]`로 통일.
- 내부 Zod 검증은 기존 스키마를 유지하고, 라우트 레벨에서 `id -> leadMeasureId`, `id -> scoreboardId`로 매핑하도록 정리.

### 2.3. 세션 인증 유틸 타입 보정
- `src/lib/server/auth.ts`의 `getSession(db)`가 더 이상 `unknown` 기반 임시 타입을 사용하지 않도록 수정.
- `ReturnType<typeof getDb>`와 `sessions.$inferSelect`를 사용해 실제 Drizzle 쿼리 결과 타입을 반환하도록 변경.
- 그 결과 API 라우트에서 `session.userId` 접근 시 발생하던 타입 오류를 제거.

### 2.4. 월간 집계 서비스 추가
- `DailyLogService.getMonthlyLogs`를 추가하여 `monthStart ~ monthEnd` 구간 집계를 지원.
- 월간 응답은 `WEEKLY + MONTHLY` 활성 선행지표를 함께 반환한다.
- 각 항목에는 `period`를 포함하여 프론트가 주간 목표와 월간 목표를 구분해 렌더링할 수 있도록 했다.
- `monthStart` 쿼리를 생략하면 서버 기준 이번 달 1일을 기본값으로 사용.
- 월간 응답에 `monthLabel(YYYY.MM)`과 `summary(achieved/total/achievementRate/isWinning)`를 포함해 프론트 계산 의존을 줄임.
- `summary` 계산 시 `MONTHLY` 지표는 월 누적 기준, `WEEKLY` 지표는 달 안의 각 주차 목표 상한 합산 기준으로 처리한다.

### 2.5. 기록 입력 규칙 단순화
- 현재 `dailyLogUpsertSchema`는 `value: true`만 허용한다.
- 미기록 복귀는 `DELETE`로 처리한다.
- 즉, 현재 활성 제품 흐름은 `미기록 <-> O` 토글이며 별도 `X(false)` 입력 UI는 없다.
- 기록 수정은 KST 기준 이번 주(월~일) 범위에서만 허용한다.
- 지난주부터의 과거 날짜는 `PUT`, `DELETE` 모두 403으로 차단한다.
- **FREE 플랜 히스토리 제한 (2026-04-16)**:
  - 로그 조회 시 `FREE` 플랜 워크스페이스는 최근 6개월 데이터만 접근 가능하다. (`assertHistoryLimit`)
  - 제한일 이전 데이터 접근 시 `403 FREE_PLAN_HISTORY_LIMIT_REACHED` 에러를 반환한다.
  - **주간 보기 예외**: 사용자가 선택한 주(Week)의 마지막 날(일요일)이 제한일 이후라면, 해당 주 전체의 조회를 허용한다. 이는 6개월 경계에 걸친 주차의 데이터가 부분적으로 누락되어 사용자에게 혼란을 주는 것을 방지하기 위함이다.

## 3. 사용된 기술 및 패턴
- **Layered Architecture**: Route -> Service -> Storage 구조 유지.
- **Zod Validation**: `dailyLogDateParamSchema`, `weeklyLogsQuerySchema` 재사용.
- **공통 API 패턴**: `apiSuccess`, `apiError`, `withErrorHandler` 유지.
- **세션 인증**: `dowin_sid` 쿠키 기반 `getSession(db)` 사용.

## 4. 검증 결과

### 4.1. 확인 완료
- Daily Log 관련 라우트의 dynamic slug 충돌 제거 확인.
- 월간 집계 API 계약(OpenAPI)과 Orval 생성 코드 갱신 확인.
- 변경 파일 기준 ESLint 통과 확인.

### 4.2. 남은 이슈
- 현재 저장소 기준 `yarn tsc --noEmit`, `yarn lint`, `yarn test --run`을 기본 검증 게이트로 사용할 수 있다.
- Storybook browser 테스트는 Daily Log 백엔드 작업의 기본 검증 범위에는 포함되지 않는다.

## 5. 향후 과제
- Daily Log 라우트/서비스 단위 테스트 보강.
