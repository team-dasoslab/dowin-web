# Dowin 성능 최적화 포인트 (1차)

## 1. 배경

Cloudflare 대시보드 스냅샷 기준 현재 지표는 아래와 같다.

| 메트릭 | 현재 | 목표 가이드 | 상태 |
|---|---:|---:|---|
| CPU Time | 86ms | 20ms 이하(1차) | 병목 가능성 높음 |
| Wall Time | 306ms | 150ms 이하(1차) | 느림 |
| Errors | 0 | 0 | 정상 |
| Duration | 305ms | 200ms 이하(1차) | 사용자 지연 체감 가능 |

해석:
- 오류율은 안정적이지만, CPU/Duration이 높아 응답 지연 체감이 발생할 수 있다.
- Wall Time 대비 CPU 비중이 높아 I/O 단독 병목보다 서버 계산/가공 비용도 함께 의심된다.

## 2. 현재 코드 기준 병목 후보

### 2.1. Team Dashboard 집계 계산량 과다 가능성
- 위치: `src/domain/dashboard/services/dashboard.service.ts`
- 징후:
  - 멤버 x 지표 단위로 `filter`, `reduce`, `Object.fromEntries` 반복 수행
  - 월간 계산에서 주차별 재필터링(`truthyLogs.filter`, `getWeekStart`)이 중첩됨
  - 로그 데이터를 `Map<number, logs[]>`로 만든 뒤에도 하위 반복이 많아 입력 규모 증가 시 CPU 비용이 빠르게 증가

### 2.2. Weekly/Monthly Log 서비스의 반복 스캔
- 위치: `src/domain/daily-log/services/daily-log.service.ts`
- 징후:
  - `measures.map` 내부에서 `logs.filter` 반복
  - 월간 요약 계산에서도 동일 데이터셋을 다시 `filter`/`reduce`
- 영향:
  - 요청 1회당 같은 로그 배열을 여러 번 스캔하여 CPU Time 증가

### 2.3. 조회 범위 과대 가능성
- 위치:
  - `src/domain/scoreboard/storage/scoreboard.storage.ts`
  - `src/domain/daily-log/storage/daily-log.storage.ts`
- 징후:
  - `findActiveScoreboardsByWorkspace(..., with: { leadMeasures: true })`로 연관 데이터 일괄 로드
  - 월간/주간을 함께 커버하는 넓은 날짜 범위 조회 후 애플리케이션 레벨 집계
- 영향:
  - 전송/파싱/가공 비용 증가로 Wall/CPU 동시 악화 가능

## 3. 최적화 우선순위

### P0 (즉시)
- Dashboard/Daily Log 서비스의 반복 스캔 제거
  - 로그를 1회 순회로 인덱싱: `leadMeasureId + date` 혹은 `leadMeasureId + weekStart`
  - `map/filter` 중첩을 줄이고 사전 집계 구조 재사용
- 요청 경로별 성능 계측 추가
  - `/api/dashboard/team`
  - `/api/scoreboards/:id/logs/weekly`
  - `/api/scoreboards/:id/logs/monthly`
- 목표:
  - CPU Time 86ms -> 40ms 이하
  - Duration 305ms -> 220ms 이하

### P1 (단기)
- DB 조회량 축소
  - 필요한 컬럼만 조회(가능한 범위에서 select 축소)
  - 로그 조회 범위를 실제 표시 구간과 맞춰 축소
- 병렬화 가능한 I/O 재점검
  - 의존 없는 조회는 `Promise.all` 적용 검토
- 목표:
  - CPU Time 40ms -> 25ms 이하
  - Duration 220ms -> 180ms 이하

### P2 (중기)
- 읽기 모델(집계 전용 응답) 또는 캐시 전략 도입
  - 팀 대시보드 요약 결과 캐시 TTL 도입 검토
  - 변동이 적은 집계는 사전 계산 구조 고려

## 4. 측정/검증 계획

### 4.1. 기준선(Baseline) 고정
- 같은 엔드포인트에 대해 최소 30회 샘플 수집
- p50/p95로 CPU Time, Duration 비교
- 워크스페이스 규모(멤버 수/선행지표 수)에 따라 샘플을 분리

### 4.2. 변경 단위 검증
- 최적화는 1개 묶음씩 적용하고 지표 재측정
- 각 변경에서 "CPU 개선", "Duration 개선"을 분리 기록

### 4.3. 성공 기준
- 에러 0 유지
- `/api/dashboard/team` p95 Duration < 200ms
- `/api/dashboard/team` p95 CPU Time < 30ms

## 5. 실행 액션 아이템

1. Dashboard/Daily Log 서비스에 공통 로그 인덱싱 유틸 초안 작성
2. Team Dashboard 월간 집계 루프를 단일 패스 중심으로 리팩터링
3. Weekly/Monthly 로그 계산에서 중복 `filter` 제거
4. 엔드포인트별 성능 로그 포맷 표준화(시작시간, DB쿼리수, 계산시간)
5. 변경 전/후 결과를 `docs/dev/performance`에 후속 문서로 누적
