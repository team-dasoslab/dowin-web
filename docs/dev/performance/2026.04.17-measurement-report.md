# WIG 성능 측정 리포트 (Run #2)

## 1. 측정 요약

- 측정일: 2026-04-17
- 측정 환경: 로컬 `next dev -p 4000`
- 측정 도구:
  - `yarn perf:api`
  - `autocannon`
  - `tmp/perf-api.ndjson`
- 주의:
  - 이번 측정은 개발 서버 기준이므로 절대 성능 수치보다 "경로 간 상대 비교"와 "변경 전/후 비교"에 의미를 둔다.
  - `GET /api/push/send-weekly-focus`는 실제 운영 우선순위가 낮아 후속 최적화 대상에서 제외했다.

## 2. 측정 대상

1. `GET /api/users/me`
2. `GET /api/workspaces/:id`
3. `GET /api/push/send-weekly-focus`

## 3. 1차 측정 결과 (`autocannon`)

### 3.1. `GET /api/users/me`

- 평균 latency: `123.02ms`
- p99 latency: `326ms`
- req/sec avg: `80.41`
- 1차 해석:
  - 세 라우트 중 가장 꾸준히 느렸다.
  - 기능 테스트 로그에서 느려 보이던 현상이 실제 요청에서도 일부 확인됐다.

### 3.2. `GET /api/workspaces/:id`

- 평균 latency: `28.01ms`
- p99 latency: `96ms`
- req/sec avg: `350.4`
- 1차 해석:
  - 상대적으로 충분히 빠른 편이며 우선순위가 낮다.

### 3.3. `GET /api/push/send-weekly-focus`

- 평균 latency: `80.63ms`
- p99 latency: `569ms`
- req/sec avg: `122.9`
- 1차 해석:
  - 평균은 나쁘지 않지만 tail latency가 튀었다.
  - 다만 이 경로는 스케줄러 off로 운영 영향 제어가 가능해 우선순위를 낮췄다.

## 5. `/api/users/me` 병목 분석

### 5.1. 최적화 전

주요 샘플에서 두 구간이 함께 컸다.

- `session`: 약 `25ms ~ 108ms`
- `get-profile`: 약 `32ms ~ 131ms`

대표 샘플:

| totalMs  | session | get-profile |
| -------- | ------: | ----------: |
| 168.93ms | 35.73ms |    131.89ms |
| 162.55ms | 97.55ms |     64.61ms |
| 146.71ms | 38.05ms |    108.51ms |

해석:

- `/api/users/me`는 한 지점만 느린 게 아니라 세션 조회와 프로필 조회가 모두 비용을 만들고 있었다.
- 특히 프로필 조회는 사용자 조회와 워크스페이스 membership 조회가 분리돼 있어 불필요한 왕복이 있었다.

### 5.2. 적용한 최적화

#### A. GET `/api/users/me`에서 세션 재발급 제거

- 변경 파일: `src/app/api/users/me/route.ts`
- 변경 내용:
  - `getSessionWithRefresh` 대신 `getSession` 사용
- 이유:
  - 읽기 전용이면서 자주 호출되는 경로에서 세션 재발급까지 수행하는 것은 비용 대비 이득이 작다.

#### B. 프로필 조회를 2쿼리에서 1조인으로 축소

- 변경 파일: `src/domain/profile/storage/profile.storage.ts`
- 변경 내용:
  - `users`
  - `workspace_members`
  - `workspaces`
    를 `leftJoin` 1회 조회로 통합
- 이유:
  - 기존에는 user 조회 후 membership/workspace 조회를 다시 수행했다.
  - 프로필 응답 shape는 유지하면서 DB round-trip을 줄일 수 있었다.

## 6. `/api/users/me` 최적화 후 결과

### 6.1. 전/후 비교

`get-profile` 구간이 눈에 띄게 감소했다.

| 구간        |      최적화 전 |                               최적화 후 |
| ----------- | -------------: | --------------------------------------: |
| session     | `25ms ~ 108ms` | `25ms ~ 35ms`가 주 구간, 일부 tail 존재 |
| get-profile | `32ms ~ 131ms` | `14ms ~ 30ms`가 주 구간, 일부 tail 존재 |

대표 샘플:

| totalMs | session | get-profile |
| ------- | ------: | ----------: |
| 45.12ms | 25.18ms |     19.83ms |
| 50.52ms | 30.16ms |     20.23ms |
| 57.39ms | 27.76ms |     29.49ms |
| 68.92ms | 34.15ms |     34.64ms |

### 6.2. 해석

- `get-profile` 최적화는 성공으로 판단한다.
- 일반 구간에서 `/api/users/me`는 대체로 `45ms ~ 60ms` 수준으로 내려왔다.
- 다만 tail latency는 아직 남아 있다.
  - `session`이 `80ms ~ 180ms`까지 튀는 샘플이 존재
  - 일부 샘플은 `get-profile`도 `60ms+`, 드물게 `100ms+`까지 상승

## 7. session 경로 추가 최적화

### 7.1. 적용한 변경

- 변경 파일: `src/lib/server/auth.ts`
- 변경 내용:
  - 세션 조회 반환 타입을 `id`, `userId`, `expiresAt`만 가지는 최소 shape로 축소
  - `findValidSession`에서 필요한 칼럼만 명시적으로 조회
  - 읽기 전용 `getSession`은 세션 자동 연장 로직 없이 조회만 수행
- 이유:
  - `GET /api/users/me`는 읽기 요청이므로 session reissue가 필요 없고, session row 전체를 읽을 이유도 없다.

### 7.2. 측정 결과

추가 최적화 후 `session` 주 구간은 아래처럼 내려왔다.

| 구간    | 1차 최적화 후 |          auth 최적화 후 |
| ------- | ------------: | ----------------------: |
| session | `25ms ~ 35ms` | `24ms ~ 31ms`가 주 구간 |

대표 샘플:

| totalMs | session | get-profile |
| ------- | ------: | ----------: |
| 42.11ms | 24.55ms |     17.42ms |
| 46.37ms | 29.06ms |     17.16ms |
| 49.84ms | 30.78ms |     18.92ms |

해석:

- `session` 평균과 일반 구간은 분명히 개선됐다.
- 다만 tail은 완전히 사라지지 않았다.
- 이 시점부터 `/api/users/me`의 tail은 `session`과 `get-profile`이 번갈아 기여하는 형태로 바뀌었다.

## 8. get-profile tail 추가 분석

### 8.1. storage 내부 계측

- 변경 파일: `src/domain/profile/storage/profile.storage.ts`
- 추가 로그:
  - `ProfileStorage.findProfileRowByUserId`
  - 초기에는 단일 join query 기준 `query` step
  - 이후 2쿼리 재실험에서는 `user-query`, `membership-query` step

로그 비교 결과 `GET /api/users/me`의 `get-profile` 시간은 storage query 시간과 거의 1:1로 대응했다.

대표 샘플:

| route get-profile | storage query |
| ----------------: | ------------: |
|         `45.41ms` |     `43.42ms` |
|        `105.38ms` |    `105.22ms` |
|        `155.62ms` |    `155.39ms` |

해석:

- `get-profile` tail은 service나 response serialization이 아니라 DB query 자체에서 발생했다.

### 8.2. 2쿼리 재실험

join query tail이 실제 원인인지 확인하기 위해 프로필 조회를 다시 2단계 조회로 되돌려 비교했다.

- `users` 단건 조회
- `workspace_members + workspace` 조회

결과:

- 극단적인 tail은 일부 완화됐다.
- 하지만 일반 구간이 더 나빠졌다.

2쿼리 재실험 샘플:

|    totalMs |   session | get-profile | storage breakdown                   |
| ---------: | --------: | ----------: | ----------------------------------- |
|  `66.61ms` | `26.49ms` |   `40.01ms` | `user 15.98ms + membership 23.81ms` |
|  `84.93ms` | `29.99ms` |   `54.82ms` | `user 24.90ms + membership 29.70ms` |
| `108.64ms` | `30.67ms` |   `77.83ms` | `user 30.69ms + membership 46.90ms` |

재실험 해석:

- 2쿼리는 join 1회보다 round-trip 비용이 커서 평시 latency가 상승했다.
- `membership-query`가 자주 큰 비중을 차지했지만, `user-query`도 함께 튀는 샘플이 있었다.
- 따라서 문제를 "join 하나"로 단정하기는 어렵고, 현재 dev 환경에서는 2쿼리로 되돌리는 것이 더 좋은 선택이 아니었다.
- 최종적으로 2쿼리 실험안은 채택하지 않고, 1조인 접근을 유지하는 판단이 타당하다.

## 9. 현재 결론

### 9.1. 확정된 내용

- `/api/workspaces/:id`는 이번 조사 범위에서 병목이 아니다.
- `/api/push/send-weekly-focus`는 현재 기준 운영 우선순위가 낮고, 스케줄러 제어가 가능해 이번 최적화 범위에서 제외한다.
- `/api/users/me`는 실제로 개선 여지가 있었고, `get-profile` 경로는 의미 있게 줄였다.
- `src/lib/server/auth.ts` 최적화로 `session` 주 구간도 추가로 낮출 수 있었다.
- `profile.storage`를 2쿼리로 되돌리는 실험은 tail 억제 효과보다 평시 성능 악화가 더 커서 기각했다.

### 9.2. 남은 병목

현재 `/api/users/me`는 단일 고정 병목보다 tail latency 관리 이슈에 가깝다.

- `session`은 평균/주 구간은 줄었지만 여전히 일부 tail이 남아 있다.
- `get-profile`은 1조인 기준 평소 성능은 좋지만 드문 query tail이 존재한다.
- 이번 재실험 기준으로는 2쿼리 분해가 해결책이 아니었다.
