# Polar workspace checkout webhook race 버그 리포트

## 1. 현상

Basic seat checkout 완료 후 Polar subscription은 정상 생성됐지만 Dowin의 `workspace_billing_state.subscription_key`가 비어 있는 사례가 확인됐다.

이 상태에서는 아래 문제가 발생한다.

- `PATCH /api/workspaces/{workspaceId}/billing/seats`가 `BILLING_NOT_READY`를 반환한다.
- Polar 대시보드에서 seat 수를 정상 변경해도 Dowin의 `workspace_seat_entitlements.purchasedSeatCount`가 자동 갱신되지 않는다.
- 결제 관리/구독 관리 흐름이 workspace billing projection에 의존하는 경우 정상 진입 또는 후속 변경이 불안정해진다.

## 2. 확인된 사실

- Polar webhook endpoint가 켜져 있고 signature 검증이 통과하면 webhook route는 `200`을 반환한다.
- 실패했던 webhook을 workspace 생성 이후 redelivery하면 `billing_events` row가 생성되고 `workspace_billing_state.subscription_key`가 채워진다.
- Polar webhook payload에는 subscription id가 정상 포함되어 있다.
  - `subscription.updated` payload: `data.id`
  - `order.paid` payload: `data.subscription_id`
- workspace setup checkout payload에는 `metadata.workspaceId`가 없고 `metadata.workspaceCheckoutId`만 있다.
- customer external id도 workspace 생성 전에는 `workspace-checkout:{pendingWorkspaceCheckoutUid}` 형태다.
- `billing_events`가 비어 있던 사례는 webhook route 실패가 아니라 webhook handler가 workspace를 찾지 못해 append 전에 `ignored`로 종료한 것으로 확인됐다.

## 3. 원인

결제 직후 Polar webhook이 Dowin workspace 생성 완료보다 먼저 도착한다.

현재 webhook handler는 workspace를 아래 기준으로 찾는다.

- `metadata.workspaceId`
- `customer.external_id = workspace:{workspaceId}`
- `workspaces.billingCustomerExternalRef`

하지만 workspace setup checkout의 초기 webhook은 아직 생성된 workspace row가 없고, payload도 `workspaceCheckoutId`만 갖고 있다. 따라서 handler가 workspace를 찾지 못하고 `ignored`로 종료한다. 이 경우 Polar에는 `200`을 반환하지만 `billing_events` append와 billing projection apply는 수행되지 않는다.

이후 checkout complete에서 workspace가 생성되더라도, checkout detail 조회 결과에 `subscription_id`가 없거나 아직 반영 전이면 `workspace_billing_state.subscription_key`가 `null`로 저장된다. 이미 무시된 webhook은 자동 재처리되지 않으므로 구독키 누락 상태가 남는다.

## 4. 임시 복구

- Polar 대시보드에서 해당 webhook delivery를 redelivery한다.
- 또는 Polar 대시보드에서 subscription id를 확인해 admin billing override로 `subscriptionKey`를 수동 보정한다.

단, 수동 보정은 sandbox/운영 디버깅용 임시 조치로만 사용한다.

## 5. 수정 방향

우선순위 1:

- workspace checkout complete에서 `GET /checkouts/{checkoutId}` 응답의 `subscription_id`가 없으면 짧게 재조회한다.
- 재조회 후에도 subscription id가 없으면 `ACTIVE` provision을 완료하지 않고 `WORKSPACE_CHECKOUT_NOT_READY`로 둔다.

우선순위 2:

- webhook handler가 `metadata.workspaceCheckoutId` 또는 `customer.external_id = workspace-checkout:{uid}`를 해석하도록 확장한다.
- pending checkout이 이미 completed workspace와 연결되어 있으면 해당 workspace에 projection을 적용한다.
- 아직 workspace가 생성 전이면 이벤트를 완전히 버리지 않고, 나중에 complete 단계에서 보정할 수 있는 reconciliation 경로를 둔다.

우선순위 3:

- `order.paid` / `order.created` payload의 `subscription_id`, `subscription.id`, `seats`, `metadata.workspaceCheckoutId`를 workspace setup checkout 보정 source로 사용할지 결정한다.
- 이 경로를 사용한다면 `billing_events` append 시점과 projection apply 시점의 멱등성 기준을 명확히 해야 한다.

## 6. 검증 기준

- workspace 생성 전 `subscription.updated` webhook이 먼저 도착해도 최종 workspace에는 `subscriptionKey`가 채워져야 한다.
- `billing_events`에는 결제 관련 provider event가 append되어야 한다.
- `workspace_seat_entitlements.purchasedSeatCount`는 Polar payload의 실제 `seats`를 반영해야 한다.
- redelivery 없이도 checkout 완료 후 결제 관리와 seat 변경 API가 동작해야 한다.

## 7. 1차 수정

2026-06-04 1차 수정은 workspace checkout complete 단계에서 `GET /checkouts/{checkoutId}` 응답의 `subscription_id`가 늦게 채워지는 경우를 보완한다.

- checkout 상태와 metadata 검증이 통과했지만 `subscriptionKey`가 없으면 checkout detail을 짧게 재조회한다.
- 재조회 중 `subscriptionKey`가 채워지면 그 값을 `workspace_billing_state.subscription_key`로 저장해 provision한다.
- 재조회 후에도 `subscriptionKey`가 없으면 workspace를 `ACTIVE`로 provision하지 않고 `WORKSPACE_CHECKOUT_NOT_READY`를 반환한다.

남은 후속:

- workspace 생성 전 도착한 webhook 자체를 보관하거나 pending checkout 기준으로 재처리하는 reconciliation 경로는 아직 필요하다.
