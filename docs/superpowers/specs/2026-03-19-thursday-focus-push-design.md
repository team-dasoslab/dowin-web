# Thursday Focus Push Design

작성일: 2026-03-19
상태: Draft Approved
주제: 매주 목요일 오후 3시(KST) 추가 푸시로 이번 주 실행도가 가장 낮은 선행지표를 다시 행동으로 연결한다.

## 1. 문제 정의

현재 DOWIN는 목표, 선행지표, 일일 기록, 점수판을 통해 "무엇을 관리해야 하는지"는 잘 보여준다. 반면 사용자가 실제 행동으로 옮기도록 돕는 장치는 약하다.

특히 다음 특성이 있다.

- 사용자는 해야 할 일을 모르는 상태가 아니라, 알아도 움직이지 못하는 상태에 자주 머문다.
- 매일 밤 9시 리마인드 푸시는 존재하지만, "오늘 무엇을 하면 좋을지"까지는 좁혀주지 못한다.
- 책임감 자극이나 평가성 메시지는 오히려 사용자의 기운을 떨어뜨릴 수 있다.

이번 기능의 목적은 사용자를 강하게 밀어붙이는 것이 아니라, 이번 주 안에 한 번 더 움직일 수 있도록 가장 부담이 적은 행동 트리거를 추가하는 것이다.

## 2. 목표

- 매주 목요일 오후 3시(KST)에 추가 푸시를 발송한다.
- 이번 주 기준으로 가장 실행률이 낮은 선행지표 1개를 골라 사용자에게 제시한다.
- 추천 로직은 가능한 한 규칙 기반으로 유지하되, 동률일 때만 AI를 보조적으로 사용한다.
- 첫 버전의 문구는 단순한 고정 템플릿으로 둔다.

비목표:

- AI가 푸시 문구를 생성하지 않는다.
- 장기 추세 분석이나 복잡한 행동 코칭을 도입하지 않는다.
- 리드미저 기록 의미를 바꾸거나 O/X 모델을 확장하지 않는다.

## 3. 사용자 경험

발송 시점에 사용자는 짧고 판단 피로가 없는 문구를 받는다.

- 제목: 기존 푸시 시스템 톤에 맞춘 간단한 제목 사용
- 본문: `오늘은 [선행지표명] 해볼까요?`
- 클릭 딥링크: `/dashboard/my`

이 푸시는 분석 리포트가 아니라 행동 유도용이다. 따라서 설명을 길게 붙이지 않고, 지금 당장 떠올릴 수 있는 행동 하나만 전면에 둔다.

## 4. 선택 로직

### 4.1 대상 사용자

아래 조건을 모두 만족하는 사용자만 대상이 된다.

- 푸시 구독 정보가 존재한다.
- 활성 점수판이 있다.
- 이번 주 기준으로 평가 가능한 선행지표가 있다.

### 4.2 기본 후보 계산

후보는 활성 점수판에 속한 선행지표에서만 계산한다.

각 선행지표에 대해 이번 주 기준 실행률을 계산한다.

- 분자: 이번 주 실제 기록 횟수
- 분모: 이번 주 시점에서 기대되는 누적 실행 횟수

첫 버전에서는 "이번 주에 가장 밀린 것"을 찾는 것이 목적이므로 장기 추세는 쓰지 않는다.

### 4.3 최종 선택

1. 이번 주 기준 실행률이 가장 낮은 선행지표를 찾는다.
2. 최저 실행률 선행지표가 0개면 발송하지 않는다.
3. 최저 실행률 선행지표가 1개면 바로 선택한다.
4. 최저 실행률 선행지표가 2개 이상이면 그 후보들만 AI에 전달해 1개를 고르게 한다.

핵심 원칙은 `규칙 우선, AI 보조`다. AI는 전면 판단기가 아니라 동률 해소기다.

## 5. AI 역할

AI는 아래 상황에서만 호출한다.

- 이번 주 최저 실행률 선행지표가 2개 이상인 경우

AI 입력에는 아래 정보만 넣는다.

- 가중목 제목
- 동률 후보 선행지표 이름
- 동률 후보 선행지표 설명
- 각 후보의 이번 주 실행 정보

AI 출력은 "선택된 선행지표 식별자 1개"만 받는다.

AI는 다음 일을 하지 않는다.

- 푸시 문구 생성
- 사용자 심리 분석
- 자유형 장문 응답 반환

### 5.1 실패 시 fallback

AI 호출이 실패하거나, 응답이 비어 있거나, 후보 밖의 값을 반환하면 규칙 기반 fallback으로 대체한다.

fallback 우선순위:

1. 후보 중 랜덤 1개 선택
2. 랜덤 구현이 부담되면 정렬 기준 첫 항목 선택

## 6. 스케줄링 구조

현재 `/api/push/send-daily` 내부 cron 라우트가 존재하므로, 이번 기능은 같은 내부 푸시 발송 구조를 재사용하는 방향으로 설계한다.

선택지는 두 가지다.

- 기존 라우트에 목요일 오후 3시 분기를 추가
- 주간 실행 유도 전용 내부 라우트를 별도로 추가

권장안은 전용 내부 라우트를 별도로 두는 것이다. 이유는 다음과 같다.

- 매일 밤 9시 리마인드와 목적이 다르다.
- 필터링 규칙과 메시지 로직이 다르다.
- 운영 중 로그, 실패 분석, 향후 실험을 분리하기 쉽다.

따라서 구현 단계에서는 예를 들어 `/api/push/send-weekly-focus` 같은 전용 내부 엔드포인트를 두고, Cloudflare Cron이 매주 목요일 15:00 KST에 호출하도록 구성하는 방향을 우선 검토한다.

## 7. 에러 처리와 예외 조건

- 평가 가능한 선행지표가 0개면 발송하지 않는다.
- 활성 점수판이 없으면 발송하지 않는다.
- 푸시 구독이 없으면 발송하지 않는다.
- 최저 실행률 항목이 1개면 AI를 호출하지 않는다.
- AI tie-break가 실패하면 규칙 기반 fallback으로 발송한다.
- 푸시 전송 실패 처리와 만료 구독 정리는 기존 푸시 발송 정책을 따른다.

## 8. 테스트 전략

### 8.1 단위 테스트

- 이번 주 실행률 계산
- 최저 실행률 0개 케이스
- 최저 실행률 1개 케이스
- 최저 실행률 동률 여러 개 케이스
- AI fallback 선택 케이스

### 8.2 통합 테스트

- 푸시 대상 사용자 필터링
- 전용 cron 라우트 인증
- AI 비호출 경로와 호출 경로 분기
- 최종 푸시 payload 검증

### 8.3 회귀 체크

- 기존 매일 9시 리마인드 푸시가 영향받지 않는지 확인
- 딥링크가 `/dashboard/my`로 유지되는지 확인

## 9. 단계적 출시

1차 출시 범위:

- 매주 목요일 오후 3시(KST) 발송
- 이번 주 최저 실행률 선행지표 선택
- 동률일 때만 AI tie-break
- 고정 본문 `오늘은 [선행지표명] 해볼까요?`

후속 확장 후보:

- 제목 문구 A/B 테스트
- tie-break 품질 로그 수집
- 사용자가 푸시 후 실제 기록까지 이어졌는지 전환율 측정

## 10. 결정 사항 요약

- 발송 시간: 매주 목요일 오후 3시 KST
- 기준 기간: 이번 주
- 선택 방식: 최저 실행률 우선, 동률일 때만 AI
- 문구 생성: 템플릿 고정
- 후보 0개: 발송 안 함
- 딥링크: `/dashboard/my`
