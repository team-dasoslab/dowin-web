# DOWIN Planning Rules

## Read Order

1. `docs/onboarding.md`
2. the relevant docs in `docs/planning/`
3. the relevant docs in `docs/dev/`
4. current implementation only when feasibility or status matters

## Repository Conventions

- Prefer extending an existing planning thread over creating fragmented docs.
- Keep YAML frontmatter on planning docs.
- Use the established planning sections when applicable.
- Produce concrete action items, not only abstract discussion.
- Keep 4DX terminology aligned with the rest of the repository.
- Update `docs/onboarding.md` when planning changes alter the project's current state or working model.

## Common Outputs

- feature plan
- milestone breakdown
- implementation-ready action list
- documentation refresh plan
- conflict or inconsistency summary across docs
